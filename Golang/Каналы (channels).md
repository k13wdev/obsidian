## Суть и контекст

**Канал (channel)** — механизм связи, позволяющий горутинам обмениваться данными путём отправки и получения значений определённого типа. Канал обеспечивает одновременно **синхронизацию** и **передачу данных**.

Концептуально канал включает: разделение ролей (читатель/писатель), механизмы блокировки и очереди ожидания. Это более высокоуровневый примитив, чем мьютексы или атомики.

---

## Ключевые идеи

- Канал всегда аллоцируется в куче; переменная канала — это указатель на структуру `hchan`.
- Небуферизованный канал: передача только когда **оба** (отправитель и получатель) готовы одновременно.
- Буферизованный канал: отправитель не блокируется пока буфер не полон; данные копируются.
- В небуферизованных каналах данные передаются **напрямую** из стека отправителя в стек получателя.
- `select` при нескольких готовых каналах выбирает **случайно** — это честный выбор.
- Канал должен закрывать **отправитель**, когда данных больше не будет.
- Запись в закрытый канал, повторное закрытие или закрытие nil-канала → **паника**.
- Чтение из закрытого канала: сначала выдаёт оставшиеся данные, потом zero value бесконечно.
- `chan<-` — только запись; `<-chan` — только чтение; компилятор проверяет на этапе сборки.

## Определения и термины

**Channel (канал)** — типизированный механизм передачи данных между горутинами. Создаётся через `make(chan T)` или `make(chan T, cap)`.

**hchan** — внутренняя структура канала в рантайме Go. Всегда выделяется в куче. Переменная типа `chan T` — это указатель на `hchan`.

**sudog** — элемент двусвязного списка в очередях `sendq` / `recvq`. Представляет одну заблокированную горутину. Ключевые поля: `g` — ссылка на саму горутину; `elem` — данные для отправки (у sender) или ячейка памяти для записи результата (у receiver). Содержит и другие поля, но концептуально центральными являются `g` и `elem`.

**waitq** — внутренняя структура в `hchan`, хранящая голову (`head`) и хвост (`tail`) двусвязного списка `sudog`. Поля `sendq` и `recvq` в `hchan` имеют тип `waitq`.

**sendDirect** — внутренняя оптимизация рантайма: если в момент отправки в очереди `recvq` уже ждёт получатель, данные копируются напрямую из стека отправителя в стек получателя, минуя `hchan` и буфер.

**gopark** — внутренняя функция рантайма: переводит горутину в состояние ожидания (`Gwaiting`) и освобождает поток ОС для других задач.

**goready** — внутренняя функция рантайма: переводит горутину из состояния ожидания обратно в `Grunnable`.

**Небуферизованный канал** — канал с нулевой ёмкостью (`make(chan T)`). Отправка и получение возможны только одновременно.

**Буферизованный канал** — канал с заданной ёмкостью (`make(chan T, n)`). Отправитель не блокируется до заполнения буфера. Работает как кольцевая очередь (FIFO).

**sendq / recvq** — двусвязные списки структур `sudog` с горутинами, заблокированными в ожидании отправки или получения.

**select** — оператор, позволяющий горутине ожидать готовности одного из нескольких каналов.

**Однонаправленный канал** — канал с ограниченным направлением: `chan<- T` (только запись) или `<-chan T` (только чтение).

**Ownership (Владение каналом)** — паттерн: горутина-«владелец» инициализирует канал (`make`), выполняет в него запись, закрывает его (`close`) и возвращает потребителям как `<-chan T`. Исключает риск записи в nil или закрытый канал, а также риск повторного закрытия.

**Done channel** — паттерн: канал типа `chan struct{}` для сигнализации о завершении работы.

**Fan-in** — паттерн: слияние нескольких входных каналов в один выходной.

**Fan-out** — паттерн: один источник читается несколькими горутинами параллельно.

**Pipeline** — паттерн: цепочка стадий обработки, соединённых каналами.

**Worker Pool** — паттерн: группа горутин, читающих задания из общего канала.

**Semaphore / Rate Limiter** — паттерн: буферизованный канал используется как хранилище «токенов» для ограничения количества одновременных операций.

## Как устроено

### Структура hchan

```
hchan
├── buf        → указатель на кольцевой буфер данных
├── dataqsiz   → ёмкость буфера (cap)
├── qcount     → текущее кол-во элементов в буфере
├── sendx      → индекс следующей записи в буфер
├── recvx      → индекс следующего чтения из буфера
├── sendq      → двусвязный список sudog (заблокированные отправители)
├── recvq      → двусвязный список sudog (заблокированные получатели)
├── lock       → мьютекс (потокобезопасность всех операций)
└── closed     → uint32 (не bool — атомарные операции ЦП не поддерживают bool)
```

Кольцевой буфер организован через два скользящих индекса `sendx` и `recvx`: `sendx` — куда пишется следующий элемент; `recvx` — откуда читается.

### Структура sudog

```
sudog
├── g     → указатель на заблокированную горутину
└── elem  → (sender) данные, которые нужно отправить
            (receiver) ячейка памяти, куда записать полученные данные
```

`sendq` и `recvq` внутри `hchan` имеют тип `waitq` — структура с двумя указателями: `head` и `tail` на начало и конец списка `sudog`.

**Как формируется sudog при блокировке на запись (sendq):**
1. Горутина-отправитель не может записать данные (буфер полон или получателя нет).
2. Создаётся `sudog`: в `g` помещается сама горутина, в `elem` — передаваемые данные.
3. `sudog` добавляется в хвост `sendq`.
4. Вызывается `gopark` — горутина засыпает.

**Как формируется sudog при блокировке на чтение (recvq):**
1. Горутина-получатель обращается к пустому каналу.
2. Создаётся `sudog`: в `g` помещается горутина, в `elem` — указатель на ячейку памяти, куда нужно записать данные.
3. `sudog` добавляется в хвост `recvq`.
4. Вызывается `gopark` — горутина засыпает.

**Кто пробуждает:** Пробуждением занимается та горутина, которая пришла взаимодействовать с каналом. Например, если пришёл отправитель и нашёл `sudog` в `recvq` — он сам забирает `sudog` из очереди и вызывает `goready` для спящей горутины.

### Механизм блокировки горутин (gopark / goready)

- **`gopark`**: меняет состояние горутины с `running` → `waiting`; разрывает связь горутины с потоком ОС; освободившийся поток передаётся следующей горутине из `runq`.
- **`goready`**: меняет состояние горутины `waiting` → `runnable`; помещает горутину обратно в локальную очередь планировщика `runq`; при наличии свободного потока выполнение возобновляется с места остановки.

### Небуферизованный канал: передача данных (sendDirect)

Когда нет буфера и в `recvq` уже ждёт получатель, рантайм применяет оптимизацию `sendDirect`: данные копируются **напрямую** из стека отправляющей горутины в стек принимающей, минуя `hchan` и не требуя дополнительных блокировок.

```
Отправитель         Получатель
  stack     ──────►   stack
           sendDirect
```

Если получателя нет — отправитель блокируется (`gopark`) и попадает в `sendq`. Когда появится получатель, рантайм разбудит отправителя (`goready`).

### Буферизованный канал: поток данных

```
Отправитель → [slot₀][slot₁]...[slotN] → Получатель
               ↑sendx              ↑recvx
```

- Отправитель блокируется только если `qcount == dataqsiz`.
- Получатель блокируется только если `qcount == 0`.

### Поведение при закрытии

| Операция | Результат |
|---|---|
| `close(ch)` — закрыть открытый | Успешно |
| `ch <- x` — писать в закрытый | **Паника** |
| `close(ch)` — закрыть уже закрытый | **Паника** |
| `close(nil)` — закрыть nil-канал | **Паника** |
| `<-ch` — читать из закрытого (буфер не пуст) | Данные из буфера |
| `<-ch` — читать из закрытого (буфер пуст) | Zero value, немедленно |
| `v, ok := <-ch` — закрытый и пустой | `ok = false` |
| `range ch` — закрытый и пустой | Завершается автоматически |

### Оператор select

```go
select {
case v := <-ch1:
    // ch1 готов
case ch2 <- x:
    // ch2 готов
default:
    // ни один не готов — выполнить немедленно
}
```

**Алгоритм:**
1. Все выражения каналов и значений для отправки вычисляются **один раз** в порядке следования.
2. Если несколько `case` готовы — выбирается **случайный** (равномерное псевдослучайное распределение). Предотвращает «голодание» горутин.
3. Если ни один не готов — горутина блокируется (или выполняется `default`).
4. `default` делает операцию **неблокирующей**.

### Однонаправленные каналы

| Тип | Направление | Ограничения |
|---|---|---|
| `chan T` | Двунаправленный | Нет |
| `chan<- T` | Только запись | Чтение → ошибка компиляции |
| `<-chan T` | Только чтение | `close()` → ошибка компиляции |

Двунаправленный канал автоматически конвертируется в однонаправленный при передаче в функцию или возврате из неё.

## Детали и нюансы

### 9 аксиом каналов

| Операция | Nil-канал | Открытый (пустой) | Открытый (не пустой) | Закрытый |
|---|---|---|---|---|
| Чтение | Deadlock | Блокировка | Значение | Zero value |
| Запись | Deadlock | Блокировка | Запись в буфер | **Паника** |
| Закрытие | **Паника** | Успешно | Успешно | **Паника** |

Полный список (9 правил):
1. Запись в nil-канал — блокирует горутину навсегда.
2. Чтение из nil-канала — блокирует горутину навсегда.
3. Закрытие nil-канала — `panic: close of nil channel`.
4. Запись в открытый канал — блокируется, если нет получателя или буфер полон.
5. Чтение из открытого канала — блокируется, если канал пуст.
6. Закрытие открытого канала — успешно; все заблокированные получатели просыпаются.
7. Запись в закрытый канал — `panic: send on closed channel`.
8. Чтение из закрытого канала — никогда не блокируется; сначала возвращаются данные из буфера, затем zero value. `v, ok := <-ch` → `ok == false` если закрыт и пуст.
9. Повторное закрытие закрытого канала — `panic: close of closed channel`.

### Nil-канал в select

Nil-канал в `select` **никогда не будет выбран** — он навсегда отключает соответствующую ветку `case`. Используется для динамического включения/выключения веток.

### Пустой select

```go
select{}
```

Блокирует горутину навсегда. Эквивалентно бесконечному ожиданию без каналов.

### Буферизованный канал как семафор / Rate Limiter

Буферизованный канал с ёмкостью N используется как хранилище «токенов» для ограничения количества одновременно выполняемых операций:

```go
sem := make(chan struct{}, N) // N — максимум параллельных задач

go func() {
    sem <- struct{}{} // захват токена (блокировка при N занятых)
    defer func() { <-sem }() // освобождение токена
    doWork()
}()
```

Идеальное применение: rate limiting, ограничение числа одновременных HTTP-запросов, пулы соединений.

## Паттерны использования

### Generator (генератор)

Функция-«владелец» создаёт канал, запускает горутину-писателя и возвращает `<-chan T`. Владелец закрывает канал через `defer close`:

```go
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out) // владелец всегда закрывает свой канал
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}
```

**Generator с done-каналом** (для предотвращения утечек):

```go
generator := func(done <-chan interface{}, integers ...int) <-chan int {
    intStream := make(chan int)
    go func() {
        defer close(intStream)
        for _, i := range integers {
            select {
            case <-done:
                return
            case intStream <- i:
            }
        }
    }()
    return intStream
}
```

### Fan-in (слияние)

Несколько источников → один выходной канал. Для каждого входного канала запускается горутина-пересылальщик:

```go
func fanIn(channels ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c {
                out <- v
            }
        }(ch)
    }
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}
```

> **Критика Fan-in:** Классический Fan-in создаёт сильный оверхед на планировщик — лишние горутины и каналы, занятые только перекладыванием данных. Часто эффективнее просто увеличить количество воркеров, пишущих в единый выходной канал (защищённый мьютексом канала под капотом).

### Fan-out (распределение)

Один источник → несколько горутин-воркеров читают параллельно (по сути, Worker Pool):

```go
jobs := make(chan int, 100)
for w := 0; w < numWorkers; w++ {
    go func() {
        for job := range jobs {
            process(job)
        }
    }()
}
```

### Pipeline (конвейер)

Цепочка стадий: выход одной горутины → вход следующей. Экономит память — данные передаются потоком:

```go
func producer() <-chan int {
    out := make(chan int)
    go func() {
        for i := 0; i < 5; i++ {
            out <- i
        }
        close(out)
    }()
    return out
}

func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for v := range in {
            out <- v * v
        }
        close(out)
    }()
    return out
}
```

### Worker Pool с context

```go
func workerPool(ctx context.Context, jobs <-chan int, results chan<- int, numWorkers int) {
    var wg sync.WaitGroup
    for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for {
                select {
                case <-ctx.Done():
                    return
                case j, ok := <-jobs:
                    if !ok {
                        return
                    }
                    results <- j * 2
                }
            }
        }()
    }
    wg.Wait()
    close(results)
}
```

### Done channel (сигнал отмены)

```go
done := make(chan struct{})

go func() {
    for {
        select {
        case <-done:
            return // завершить горутину
        case v := <-work:
            process(v)
        }
    }
}()

close(done) // сигнал всем горутинам
```

### Or-done channel

Паттерн-обёртка для чтения из канала с одновременной проверкой done-сигнала:

```go
orDone := func(done, c <-chan interface{}) <-chan interface{} {
    valStream := make(chan interface{})
    go func() {
        defer close(valStream)
        for {
            select {
            case <-done:
                return
            case v, ok := <-c:
                if ok == false {
                    return
                }
                select {
                case valStream <- v:
                case <-done:
                }
            }
        }
    }()
    return valStream
}
```

### select с таймаутом

```go
select {
case res := <-c1:
    fmt.Println(res)
case <-time.After(3 * time.Second):
    fmt.Println("timeout")
}
```

### Неблокирующая операция с default

```go
select {
case v := <-ch:
    fmt.Println("got", v)
default:
    fmt.Println("no data ready")
}
```

## Сравнения и trade-offs

| | Небуферизованный | Буферизованный |
|---|---|---|
| Синхронизация | Строгая (рандеву) | Асинхронная |
| Блокировка отправителя | Всегда без получателя | Только при полном буфере |
| Блокировка получателя | Всегда без отправителя | Только при пустом буфере |
| Гарантия доставки | Немедленная | До переполнения |
| Копий данных | Одна (прямое копирование) | Одна (через буфер) |

| Примитив | Когда использовать |
|---|---|
| Канал | Передача данных + синхронизация между горутинами |
| sync.Mutex | Защита общего состояния (non-channel) |
| sync.WaitGroup | Ожидание завершения группы горутин |
| context | Таймауты, отмена, иерархия завершения |

## Важно / подводные камни / best practices

**Подводные камни:**

- **Запись в закрытый канал → паника.** Никогда не пишите в канал после `close`. Закрывает канал только отправитель.
- **Повторное или nil-закрытие → паника.** Используй флаг или `sync.Once` для защиты от двойного закрытия. Если отправителей несколько — синхронизируй через `sync.WaitGroup` и закрывай только когда все завершили работу.
- **Deadlock при nil-канале.** Чтение и запись в nil-канал блокируются навсегда.
- **Утечки горутин.** Горутина заблокирована на чтении из канала, в который больше никто не пишет — висит вечно. Используй `done` channel или `context`. Если получатель выходит раньше времени (ошибка, таймаут), а генератор не использует done/context — генератор навсегда заблокируется на записи.
- **Data race.** Общие переменные без синхронизации при конкурентном доступе. Проверяй через `-race`.
- **Чтение из закрытого пустого канала всегда успешно.** `v, ok := <-ch` возвращает `ok=false` и zero value — не паника.
- **`for range` по каналу — забыть `close` → deadlock.** Цикл `for v := range ch` завершается только когда канал закрыт. Если генератор не вызовет `close(out)`, получатель заблокируется навсегда.
- **Shadowing (затенение) каналов — антипаттерн.** Обнуление канала (`out1 = nil`) внутри цикла для «отключения» ветки делает код нечитаемым. Лучше выделить отправку в отдельные горутины.
- **Closure в go func() в цикле (Go < 1.22).** Если запускать горутины в цикле без передачи переменной как аргумента — все горутины могут захватить последнее значение:

```go
// ОПАСНО (Go < 1.22): все горутины видят одно и то же i
for i := 0; i < N; i++ {
    go func() { use(i) }()
}

// Безопасно: передавать как аргумент
for i := 0; i < N; i++ {
    go func(val int) { use(val) }(i)
}
```

**Best practices:**

- Канал закрывает **отправитель** (горутина-«владелец») — только тот, кто знает, что данных больше не будет.
- Используй `v, ok := <-ch` для проверки, закрыт ли канал.
- Используй паттерн **Ownership**: функция создаёт канал, запускает горутину-писателя с `defer close(out)` и возвращает `<-chan T`.
- Применяй `context.WithTimeout` / `WithCancel` для управления жизненным циклом горутин.
- Используй `sync.WaitGroup` для ожидания завершения группы горутин.
- Всегда запускай тесты с флагом `-race` для обнаружения гонок данных.
- `select` с `default` — неблокирующая операция. Используй, если не хочешь блокировать горутину.
- Буферизованный канал со `sync.WaitGroup` — основа worker pool.
- Буферизованный канал как семафор — для ограничения одновременных операций (rate limiter).

## Карточки для повторения

#flashcards/go/channels

Что такое канал в Go?::Механизм связи между горутинами. Обеспечивает одновременно синхронизацию и передачу данных определённого типа.

Что хранится внутри структуры hchan?
?
- buf — кольцевой буфер данных
- dataqsiz — ёмкость буфера
- qcount — текущее кол-во элементов
- sendx/recvx — индексы записи/чтения в буфере
- sendq/recvq — очереди заблокированных горутин
- lock — мьютекс
- closed — флаг закрытия

Где аллоцируется hchan (структура канала)?::В куче (heap). Переменная типа `chan T` — это указатель на hchan.

Чем отличается небуферизованный канал от буферизованного?
?
- **Небуферизованный** (`make(chan T)`): передача только когда оба — отправитель и получатель — готовы одновременно
- **Буферизованный** (`make(chan T, n)`): отправитель не блокируется пока буфер не полон; получатель не блокируется пока буфер не пуст

Что такое прямое копирование в небуферизованных каналах?::Оптимизация рантайма: если получатель уже ждёт, данные копируются напрямую из стека отправителя в стек получателя, минуя hchan.

Что произойдёт при записи в закрытый канал?::**Паника**. Никогда не пишите в канал после `close`.

Что произойдёт при чтении из закрытого и пустого канала?::Немедленно вернётся zero value для типа канала. `v, ok := <-ch` вернёт `ok = false`. Паники нет.

Что произойдёт при закрытии nil-канала?::**Паника**.

Что произойдёт при чтении/записи в nil-канал?::**Deadlock** — горутина заблокируется навсегда.

Как работает select при нескольких готовых каналах?::Если несколько `case` готовы одновременно — выбирается **случайный** (равномерное псевдослучайное распределение). Предотвращает «голодание» горутин.

Что делает `default` в операторе select?::Выполняется немедленно, если все остальные `case` заблокированы. Делает операцию с каналом **неблокирующей**.

Как вести себя nil-канал в операторе select?::Nil-канал **никогда не будет выбран** в select — навсегда отключает соответствующую ветку `case`.

Что означает `chan<- T` и `<-chan T`?
?
- `chan<- T` — однонаправленный канал **только для записи**. Чтение → ошибка компиляции.
- `<-chan T` — однонаправленный канал **только для чтения**. `close()` → ошибка компиляции.

Кто должен закрывать канал?::**Отправитель** — тот, кто знает, что данных больше не будет. Получатель не закрывает канал.

Как обнаружить, что канал закрыт при чтении?::Использовать comma ok idiom: `v, ok := <-ch`. Если `ok == false` — канал закрыт и пуст.

Что такое паттерн Fan-in?::Слияние нескольких входных каналов в один выходной. Для каждого входа запускается горутина-пересылальщик; выходной канал закрывается через sync.WaitGroup.

Что такое паттерн Fan-out?::Один канал-источник читается несколькими горутинами-воркерами параллельно. Распределяет нагрузку.

Что такое паттерн Pipeline?::Цепочка стадий обработки, соединённых каналами: выход одной горутины — вход следующей. Экономит память — данные передаются потоком.

Что такое Done channel?::Паттерн с `chan struct{}` для сигнализации горутинам о необходимости завершить работу. `close(done)` — широковещательный сигнал для всех подписанных горутин.

Как сделать операцию с каналом неблокирующей?
?
Использовать `select` с веткой `default`:
```go
select {
case v := <-ch:
    use(v)
default:
    // нет данных — продолжить
}
```

Как реализовать таймаут при ожидании из канала?
?
```go
select {
case res := <-ch:
    use(res)
case <-time.After(3 * time.Second):
    // таймаут
}
```

Какие причины утечки горутин через каналы?
?
- Горутина ждёт из канала, в который больше никто не пишет
- Горутина пытается писать в канал, который никто не читает
- Нет механизма отмены (done channel, context)

Как проверить код на гонки данных?::Запускать тесты с флагом: `go test -race ./...` или `go run -race main.go`.

Что такое sudog?::Элемент двусвязного списка в очередях `sendq` / `recvq`. Представляет одну заблокированную горутину. Ключевые поля: `g` — ссылка на горутину; `elem` — данные для отправки (sender) или ячейка памяти для записи результата (receiver).

Что хранится в поле `elem` структуры sudog у отправителя?::Сами данные, которые горутина-отправитель пытается передать в канал.

Что хранится в поле `elem` структуры sudog у получателя?::Указатель на ячейку памяти (в стеке получателя), куда отправитель должен скопировать данные через `sendDirect`.

Что такое waitq в hchan?::Структура с двумя указателями — `head` и `tail` на начало и конец двусвязного списка `sudog`. Поля `sendq` и `recvq` в `hchan` имеют тип `waitq`.

Какие шаги выполняются при блокировке горутины на запись в канал?
?
1. Создаётся `sudog`: `g` = горутина, `elem` = данные для отправки.
2. `sudog` добавляется в хвост `sendq`.
3. Вызывается `gopark` — горутина переходит в `waiting`, поток ОС освобождается.

Какие шаги выполняются при блокировке горутины на чтение из канала?
?
1. Создаётся `sudog`: `g` = горутина, `elem` = указатель на ячейку памяти для результата.
2. `sudog` добавляется в хвост `recvq`.
3. Вызывается `gopark` — горутина переходит в `waiting`, поток ОС освобождается.

Кто пробуждает заблокированную горутину в канале?::Та горутина, которая пришла взаимодействовать с каналом. Например, пришедший отправитель сам забирает `sudog` из `recvq` и вызывает `goready` для спящего получателя.

Почему поле `closed` в hchan — `uint32`, а не `bool`?::Атомарные операции ЦП не поддерживают булевы типы — только целочисленные. `uint32` позволяет использовать атомарные операции для потокобезопасного изменения флага.

Что такое sendDirect?::Оптимизация рантайма: если в момент отправки в `recvq` уже ждёт получатель, данные копируются напрямую из стека отправителя в стек получателя, минуя `hchan`. Снижает число копий и блокировок.

Что делают gopark и goready?
?
- `gopark` — переводит горутину в состояние `Gwaiting` и освобождает поток ОС для других задач.
- `goready` — переводит горутину из ожидания обратно в `Grunnable`.

Что такое паттерн Ownership (владение каналом)?::Горутина-«владелец» инициализирует канал (`make`), выполняет запись, закрывает его (`defer close`) и возвращает потребителям как `<-chan T`. Исключает паники от записи в nil/закрытый канал и повторного закрытия.

Как использовать буферизованный канал как семафор?
?
```go
sem := make(chan struct{}, N)
go func() {
    sem <- struct{}{} // захват токена
    defer func() { <-sem }() // освобождение
    doWork()
}()
```
Буфер ёмкостью N разрешает не более N одновременных операций.

В чём критика классического паттерна Fan-in?::Создаёт оверхед на планировщик: лишние горутины-посредники и каналы, занятые только перекладыванием данных. Часто эффективнее увеличить число воркеров, пишущих в единый общий выходной канал.

Edge case: что произойдёт при `for v := range ch`, если канал никогда не закроется?::Цикл заблокируется навсегда — `range` завершается только когда канал закрыт. Генератор должен вызывать `close(out)`.

Антипаттерн Shadowing: в чём проблема обнуления канала внутри цикла?::Обнуление (`out1 = nil`) внутри `for`/`select` ради «отключения» ветки делает код нечитаемым и считается антипаттерном. Лучше выделить отправку в отдельные горутины.

Edge case: closure в go func() в цикле (Go < 1.22)?::Все горутины могут захватить последнее значение переменной цикла. Решение — передавать переменную как аргумент: `go func(val int) { use(val) }(i)`.

Что делает пустой `select{}`?::Блокирует горутину навсегда — эквивалент бесконечного ожидания без каналов.
