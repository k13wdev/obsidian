## Суть и контекст

Пакет `context` — инструмент для управления временем жизни горутин и операций: передачи сигналов отмены, дедлайнов, таймаутов и данных, привязанных к области видимости запроса. Используется повсеместно: HTTP-обработчики, gRPC, работа с БД, long-running горутины.

Ключевая идея: контекст распространяется **сверху вниз** через весь стек вызовов — от точки входа (main, HTTP handler) до самой глубокой операции (SQL-запрос, сетевой вызов).

---

## Ключевые идеи

- `Context` — интерфейс из 4 методов: `Deadline`, `Done`, `Err`, `Value`.
- Контексты образуют **дерево**: отмена родителя автоматически отменяет всех потомков.
- `context.Background()` — корень дерева; `context.TODO()` — заглушка при рефакторинге.
- `WithCancel` / `WithTimeout` / `WithDeadline` — создают отменяемых потомков.
- `WithValue` — хранит данные, специфичные для запроса (traceID, токен авторизации).
- Контекст **никогда не хранится в структурах** — только передаётся первым аргументом.
- Никогда не передавай `nil` вместо контекста — используй `context.TODO()`.
- Всегда вызывай `defer cancel()` — иначе утечка памяти.
- Горутина реагирует на отмену через `select { case <-ctx.Done(): }`.

---

## Определения и термины

**Context** — интерфейс из пакета `context`. Несёт сигнал отмены, дедлайн и пары ключ-значение. Передаётся через весь стек вызовов.

**CancelFunc** — функция типа `func()`, возвращаемая `WithCancel`, `WithTimeout`, `WithDeadline`. При вызове отменяет контекст и освобождает ресурсы.

**Done channel** — канал `<-chan struct{}`, который возвращает метод `Done()`. Закрывается при отмене контекста. Закрытый канал разблокирует все `select`, ожидающие на нём.

**context.Background()** — пустой ненулевой корневой контекст. Никогда не отменяется, не имеет дедлайна и значений. Используется в `main`, `init`, тестах.

**context.TODO()** — пустой ненулевой контекст-заглушка. Семантически означает «контекст будет добавлен позже». Статические анализаторы распознают его. Передаётся вместо `nil` при рефакторинге.

**context.WithCancel** — создаёт дочерний контекст с ручной отменой через `CancelFunc`.

**context.WithTimeout** — создаёт дочерний контекст, автоматически отменяемый через `time.Duration`.

**context.WithDeadline** — создаёт дочерний контекст, автоматически отменяемый в момент `time.Time`.

**context.WithValue** — создаёт дочерний контекст с парой ключ-значение.

**context.WithoutCancel** *(Go 1.21)* — создаёт дочерний контекст, который наследует значения, но игнорирует сигнал отмены родителя.

**Goroutine leak (утечка горутин)** — ситуация, когда горутина «зависает» навечно (ожидая канал, с которым больше никто не работает), продолжая потреблять ресурсы памяти и планировщика.

## Как устроено

### Интерфейс Context

```go
type Context interface {
    Deadline() (deadline time.Time, ok bool)
    Done() <-chan struct{}
    Err() error
    Value(key any) any
}
```

| Метод | Возвращает | Описание |
|---|---|---|
| `Deadline()` | `(time.Time, bool)` | Момент автоматической отмены; `ok=false` если дедлайна нет |
| `Done()` | `<-chan struct{}` | Канал, закрывающийся при отмене; `nil` если контекст не отменяем |
| `Err()` | `error` | `nil` пока активен; `context.Canceled` или `context.DeadlineExceeded` после отмены |
| `Value(key)` | `any` | Значение по ключу; `nil` если не найдено |

### Дерево контекстов

Контексты организованы в дерево. Каждый вызов `With*` создаёт нового потомка:

```
context.Background()
└── WithCancel → ctxA, cancelA
    ├── WithTimeout(5s) → ctxB, cancelB
    │   └── WithValue("userID", 42) → ctxC
    └── WithDeadline(t) → ctxD, cancelD
```

**Распространение отмены:** при вызове `cancelA()` — автоматически отменяются `ctxB`, `ctxD` и все их потомки. Сигнал идёт только **вниз** по дереву — никогда вверх.

**Сборка мусора:** `cancel()` удаляет ссылку родителя на дочерний контекст. Без вызова `cancel()` родитель хранит ссылку до своей собственной отмены → **утечка памяти**.

### Внутренние типы *(из исходного кода Go)*

| Тип | Создаётся через | Ключевые поля |
|---|---|---|
| `emptyCtx` | `Background()`, `TODO()` | — нет полей, никогда не отменяется |
| `cancelCtx` | `WithCancel` | `mu Mutex`, `done chan struct{}` (ленивая инициализация), `err error`, `children map[canceler]struct{}` |
| `timerCtx` | `WithTimeout`, `WithDeadline` | встраивает `cancelCtx` + `timer *time.Timer` + `deadline time.Time` |
| `valueCtx` | `WithValue` | `parent Context`, `key`, `val any` |

**cancelCtx при отмене:**
1. Закрывает канал `done`.
2. Рекурсивно обходит `children` и отменяет каждого потомка.
3. Отвязывает себя от родителя (удаляет из `children` родителя).

**valueCtx.Value(key):**
- Проверяет свой ключ.
- Если не совпадает → делегирует родителю вверх по дереву.
- Поиск — обход дерева вверх: O(глубина дерева).

### Конструкторы

#### context.Background() / context.TODO()

```go
ctx := context.Background() // корень: main, init, тесты
ctx := context.TODO()       // заглушка при рефакторинге
```

Обе возвращают пустой ненулевой контекст без отмены, дедлайна и значений. Разница — только семантическая: `TODO` — явный сигнал «сюда вернуться».

#### context.WithCancel

```go
ctx, cancel := context.WithCancel(parent)
defer cancel() // ВСЕГДА
```

- Создаёт дочерний `cancelCtx`.
- `cancel()` → закрывает `Done()`, отменяет всех потомков, отвязывает от родителя.
- `defer cancel()` сразу после создания — гарантирует очистку ресурсов даже при панике или раннем `return`.

#### context.WithTimeout / context.WithDeadline

```go
ctx, cancel := context.WithTimeout(parent, 5*time.Second)
defer cancel()

ctx, cancel := context.WithDeadline(parent, time.Now().Add(5*time.Second))
defer cancel()
```

| | `WithTimeout` | `WithDeadline` |
|---|---|---|
| Аргумент времени | `time.Duration` (интервал) | `time.Time` (момент) |
| Когда отменяется | Через N секунд от сейчас | В конкретный момент времени |
| Внутренняя реализация | Вызывает `WithDeadline(now + duration)` |  |

`timerCtx` при срабатывании таймера вызывает внутренний `cancel()`. `CancelFunc` всё равно нужно вызвать через `defer` — для случая досрочного выхода из функции.

#### context.WithValue

```go
type contextKey string // unexported тип — защита от коллизий
const userIDKey contextKey = "userID"

ctx = context.WithValue(parent, userIDKey, 42)

// Извлечение
if v, ok := ctx.Value(userIDKey).(int); ok {
    fmt.Println(v)
}
```

**Почему ключ должен быть unexported типом:** если бы два пакета использовали строку `"userID"` как ключ, они могли бы случайно читать/перезаписывать данные друг друга. Уникальный тип (`type contextKey string`) из пакета A никогда не совпадёт с `type contextKey string` из пакета B — даже если базовые значения одинаковы.

#### context.WithoutCancel *(Go 1.21)*

```go
bgCtx := context.WithoutCancel(r.Context())
go doBackgroundWork(bgCtx)
```

- Копирует значения (`WithValue`) из родителя.
- **Игнорирует сигнал отмены** родителя.
- Используется для фоновых задач, которые должны завершиться даже если HTTP-запрос отменён (логирование аудита, отправка метрик).

## Детали и нюансы

### Связь с select и каналами

`ctx.Done()` возвращает `<-chan struct{}`. Горутина ожидает на нём через `select`:

```go
select {
case data := <-workCh:
    process(data)
case <-ctx.Done():
    fmt.Println("отменено:", ctx.Err())
    return
}
```

**Механизм:** при отмене контекста канал `Done` **закрывается**. Закрытый канал немедленно разблокирует любой `select`, ожидающий на нём — горутина получает возможность чисто завершиться.

### Context в HTTP сервере

```go
func handler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()
    result, err := db.QueryContext(ctx, "SELECT ...")
    // если клиент разорвал соединение — ctx отменяется автоматически
    // db.QueryContext получает сигнал и прерывает запрос
}
```

Каждый входящий HTTP-запрос привязан к своему контексту `r.Context()`. При разрыве TCP-соединения со стороны клиента HTTP-сервер Go автоматически отменяет этот контекст → все операции, принявшие его, получают сигнал остановки.

### Предотвращение утечек горутин

**Проблема:** горутина зависла в ожидании канала — никто больше не пишет туда и не закрывает его.

**Решение с context:**

```go
func worker(ctx context.Context, jobs <-chan Job) {
    for {
        select {
        case job := <-jobs:
            process(job)
        case <-ctx.Done():
            fmt.Println("worker stopped:", ctx.Err())
            return // горутина завершается чисто
        }
    }
}
```

## Сравнения и trade-offs

| Функция | Кто отменяет | Аргумент | Когда использовать |
|---|---|---|---|
| `WithCancel` | Вручную через `CancelFunc` | — | Отмена по бизнес-логике |
| `WithTimeout` | Автоматически + вручную | `time.Duration` | «Не дольше N секунд» |
| `WithDeadline` | Автоматически + вручную | `time.Time` | «До конкретного момента» |
| `WithValue` | Никогда | ключ + значение | Данные запроса (traceID, auth) |
| `WithoutCancel` | Никогда (игнорирует родителя) | — | Фоновые задачи пережившие запрос |

| | `context.Background()` | `context.TODO()` |
|---|---|---|
| Отмена | Нет | Нет |
| Дедлайн | Нет | Нет |
| Значения | Нет | Нет |
| Назначение | Корень дерева в продакшн-коде | Заглушка при рефакторинге |

## Примеры

### 1. WithCancel — ручная отмена

```go
ctx, cancel := context.WithCancel(context.Background())
defer cancel()

go func(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            fmt.Println("горутина завершена:", ctx.Err())
            return
        default:
            doWork()
        }
    }
}(ctx)

time.Sleep(2 * time.Second)
cancel() // сигнал горутине остановиться
```

### 2. WithTimeout — таймаут операции

```go
ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()

req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := http.DefaultClient.Do(req)
if err != nil {
    if errors.Is(err, context.DeadlineExceeded) {
        fmt.Println("таймаут истёк")
    }
}
```

### 3. WithValue — передача traceID

```go
type traceKey struct{}

func withTraceID(ctx context.Context, id string) context.Context {
    return context.WithValue(ctx, traceKey{}, id)
}

func getTraceID(ctx context.Context) string {
    if v, ok := ctx.Value(traceKey{}).(string); ok {
        return v
    }
    return ""
}
```

### 4. select с ctx.Done()

```go
func process(ctx context.Context, jobs <-chan string) error {
    for {
        select {
        case job, ok := <-jobs:
            if !ok {
                return nil // канал закрыт
            }
            handleJob(job)
        case <-ctx.Done():
            return ctx.Err() // отмена или таймаут
        }
    }
}
```

### 5. HTTP handler с контекстом

```go
func handler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()

    result, err := db.QueryContext(ctx, "SELECT * FROM users")
    if err != nil {
        if errors.Is(err, context.Canceled) {
            // клиент отключился
            return
        }
        http.Error(w, err.Error(), 500)
        return
    }
    json.NewEncoder(w).Encode(result)
}
```

### 6. Полный пример WithCancel с гонкой таймера

```go
func f1(t int) {
    c1 := context.Background()
    c1, cancel := context.WithCancel(c1)
    defer cancel()

    go func() {
        time.Sleep(4 * time.Second)
        cancel()
    }()

    select {
    case <-c1.Done():
        fmt.Println("f1():", c1.Err())
        return
    case r := <-time.After(time.Duration(t) * time.Second):
        fmt.Println("f1():", r)
    }
}
```

### 7. WithoutCancel — фоновая задача (Go 1.21)

```go
func handler(w http.ResponseWriter, r *http.Request) {
    // Фоновый лог аудита должен завершиться даже при отмене запроса
    auditCtx := context.WithoutCancel(r.Context())
    go logAudit(auditCtx, r.URL.Path)

    // Основная работа с отменяемым контекстом
    doWork(r.Context())
}
```

## Важно / подводные камни / best practices

**Подводные камни:**

- **Забытый `cancel()` → утечка памяти.** Родительский контекст держит ссылку на потомка в `children` до своей отмены. Всегда: `defer cancel()` сразу после создания.
- **Передача `nil` вместо контекста → паника.** Вызов `context.WithValue(nil, ...)` или `http.NewRequestWithContext(nil, ...)` паникует в рантайме. Использовать `context.TODO()`.
- **Хранение контекста в структуре — антипаттерн.** Контекст отражает жизненный цикл конкретной операции. В структуре он устаревает. Передавать только как первый аргумент функции.
- **`WithValue` не для бизнес-данных.** Значения из контекста не типизированы — нужно кастить через type assertion. Потеря читаемости и типобезопасности. `WithValue` только для request-scoped метаданных: traceID, auth-token, IP.
- **Ключ без unexported типа → коллизии между пакетами.** Строки и встроенные типы как ключи конфликтуют между пакетами. Всегда создавай `type myKey struct{}`.
- **Горутина без `ctx.Done()` → goroutine leak.** Долгоживущие горутины без обработки `ctx.Done()` не имеют механизма остановки.
- **Глубокое дерево WithValue замедляет поиск.** `ctx.Value(key)` идёт вверх по дереву. При большой глубине — линейный обход.

**Best practices:**

- Первый аргумент всегда `ctx context.Context` — стандарт в промышленном Go-коде.
- `defer cancel()` — сразу после создания контекста, без исключений.
- Проверяй `ctx.Err()` после блокирующих операций для точной диагностики (`Canceled` vs `DeadlineExceeded`).
- Проверяй контекст в начале дорогой операции: `if err := ctx.Err(); err != nil { return err }`.
- Используй `context.WithoutCancel` (Go 1.21) для фоновых задач, которые переживают родительский запрос.
- Никогда не передавай глобальный или сохранённый контекст в горутины — он может быть уже отменён.

## Карточки для повторения

#flashcards/go/context

Что такое `context.Context`?::Интерфейс из 4 методов: `Deadline()`, `Done()`, `Err()`, `Value()`. Несёт сигнал отмены, дедлайн и request-scoped данные. Передаётся первым аргументом через весь стек вызовов.

Что возвращает `ctx.Done()`?::Канал `<-chan struct{}`, закрывающийся при отмене контекста. Закрытый канал немедленно разблокирует любой `select`, ожидающий на нём.

Что возвращает `ctx.Err()` после отмены?
?
- `context.Canceled` — контекст отменён вручную через `CancelFunc`
- `context.DeadlineExceeded` — истёк таймаут или дедлайн
- `nil` — контекст ещё активен

Чем отличаются `context.Background()` и `context.TODO()`?::Оба — пустые ненулевые корневые контексты без отмены и значений. Background() — для продакшн-кода (main, init, тесты). TODO() — семантическая заглушка при рефакторинге; статические анализаторы его распознают.

Почему нельзя передавать `nil` вместо контекста?::Вызов `context.WithValue(nil, ...)` или передача `nil` в функции, принимающие Context, вызывает панику в рантайме. Использовать `context.TODO()`.

Что делает `cancel()` в `WithCancel`?
?
1. Закрывает канал `Done`
2. Рекурсивно отменяет всех потомков в дереве
3. Удаляет ссылку родителя на дочерний контекст → GC может очистить память

Что происходит если не вызвать `cancel()`?::Родительский контекст хранит ссылку на дочерний в `children` до своей собственной отмены → утечка памяти. Всегда: `defer cancel()` сразу после создания.

Чем отличается `WithTimeout` от `WithDeadline`?::WithTimeout принимает `time.Duration` (интервал от сейчас). WithDeadline принимает `time.Time` (конкретный момент). Под капотом WithTimeout вызывает WithDeadline(now + duration).

Как горутина реагирует на отмену контекста?
?
```go
select {
case data := <-jobs:
    process(data)
case <-ctx.Done():
    return ctx.Err() // чистое завершение
}
```
`ctx.Done()` закрывается при отмене → `select` немедленно разблокируется.

Почему ключ в `context.WithValue` должен быть unexported типом?::Два пакета с одинаковой строкой `"userID"` как ключом будут конфликтовать. Уникальный тип `type myKey struct{}` из пакета A не совпадёт с таким же типом из пакета B — даже при одинаковом базовом значении.

Что нельзя хранить в `context.WithValue`?::Бизнес-данные, зависимости (соединения с БД), опциональные параметры функций. Только request-scoped метаданные: traceID, токен авторизации, IP клиента.

Почему контекст нельзя хранить в структурах?::Контекст отражает жизненный цикл конкретной операции. В структуре он устаревает — операция завершилась, контекст отменён, но структура продолжает его держать. Передавать только как первый аргумент `ctx context.Context`.

Что такое `context.WithoutCancel` (Go 1.21)?::Создаёт дочерний контекст, который наследует значения WithValue, но игнорирует сигнал отмены родителя. Используется для фоновых задач, которые должны завершиться даже при отмене родительского запроса.

Как context помогает предотвратить goroutine leak?::Каждая блокирующая операция в горутине оборачивается в `select` с веткой `case <-ctx.Done(): return`. При отмене контекста канал закрывается → горутина немедленно разблокируется и завершается.

Как работает `r.Context()` в HTTP сервере?::Каждый входящий HTTP-запрос автоматически привязан к контексту. При разрыве TCP-соединения клиентом сервер Go автоматически отменяет этот контекст → все операции (SQL, RPC), принявшие его, получают сигнал остановки.

Что внутри `cancelCtx`?::Поля: `mu Mutex`, `done chan struct{}` (ленивая инициализация), `err error`, `children map[canceler]struct{}` (дочерние контексты). При отмене закрывает `done`, обходит `children` и отменяет каждого.

Как `ctx.Value(key)` ищет значение?::Проверяет свой ключ. Если не совпадает — делегирует поиск родительскому контексту. Обход идёт вверх по дереву: O(глубина дерева). Глубокое дерево WithValue замедляет поиск.

Что возвращает `ctx.Done()` для неотменяемого контекста?::Возвращает `nil`. Чтение из `nil`-канала в `select` блокируется навсегда — эта ветка никогда не выбирается.

Каков правильный порядок аргументов функции с контекстом?::`ctx context.Context` — всегда **первый** аргумент. Стандарт Go: `func DoSomething(ctx context.Context, arg1 T1, ...) error`.

Когда использовать `WithCancel` vs `WithTimeout`?::WithCancel — отмена по бизнес-логике (пользователь нажал стоп, первый из воркеров завершился). WithTimeout — операция не должна длиться дольше N секунд (HTTP-запрос, SQL-запрос, RPC-вызов).
