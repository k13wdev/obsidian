## Суть и контекст

Пакет `sync` и `sync/atomic` предоставляют низкоуровневые примитивы синхронизации для защиты общего состояния при конкурентном доступе из нескольких горутин.

- **sync.Mutex** — взаимное исключение: только одна горутина выполняет критическую секцию.
- **sync.RWMutex** — разделённый доступ: несколько читателей одновременно, запись монопольна.
- **sync.WaitGroup** — ожидание завершения группы горутин («точка джойна»).
- **sync.Once** — гарантирует однократное выполнение функции при любом числе горутин.
- **sync.Cond** — рассылка сигналов между горутинами (wait/signal/broadcast).
- **sync.Map** — потокобезопасная map, оптимизированная для read-heavy и disjoint-ключей.
- **sync/atomic** — атомарные операции на уровне CPU; быстрее мьютекса, но применимы только к одиночным переменным.

---

## Ключевые идеи

- `sync.Mutex` — универсальная защита любых структур и групп переменных.
- `sync.RWMutex` — оптимизация для «чтение чаще записи»; несколько читателей не блокируют друг друга.
- `atomic` — самый низкоуровневый примитив: неделимые операции на уровне CPU + барьеры памяти.
- `atomic` не заменяет мьютекс когда нужна согласованность нескольких полей одновременно.
- Mutex нельзя копировать после первого использования.
- Всегда используй `defer m.Unlock()` — гарантирует освобождение даже при панике.
- `CompareAndSwap (CAS)` — строительный блок, на котором построен сам `sync.Mutex`.
- Deadlock, Livelock, Starvation — три класса проблем при неправильной синхронизации.
- `sync.WaitGroup` и `sync.Once` нельзя копировать; передавать только по указателю.

---

## Определения и термины

**sync.Mutex** — примитив взаимного исключения. Содержит поля `state int32` (битовые флаги состояния) и `sema uint32` (семафор для очереди ожидания).

**sync.RWMutex** — расширенный мьютекс для разделённого доступа. Несколько горутин могут держать **read-lock** одновременно; write-lock монопольный.

**Критическая секция** — фрагмент кода, который в конкретный момент времени должна выполнять только одна горутина.

**sync.WaitGroup** — счётчик горутин для синхронизации точки завершения. `Add` увеличивает, `Done` уменьшает, `Wait` блокирует до нуля.

**sync.Once** — гарантирует, что переданная функция будет вызвана ровно один раз, даже при конкурентных вызовах из многих горутин.

**sync.Cond** — условная переменная. Позволяет горутинам ждать наступления события и получать уведомление через `Signal` (одна горутина) или `Broadcast` (все).

**sync.Map** — потокобезопасная map с оптимизацией для read-heavy нагрузки и разных ключей у разных горутин (disjoint access).

**sync/atomic** — пакет атомарных операций. Реализует неделимые операции над примитивными типами на уровне инструкций процессора.

**Барьер памяти (Memory Barrier)** — инструкция процессора, гарантирующая порядок видимости изменений памяти между горутинами.

**Acquire (барьер получения)** — гарантирует, что текущая горутина видит самые свежие данные от всех остальных.

**Release (барьер выпуска)** — гарантирует, что все другие горутины увидят изменения, сделанные текущей горутиной.

**CompareAndSwap (CAS)** — атомарная операция: замена значения происходит только если текущее значение совпадает с ожидаемым. Возвращает `bool`. На CAS построен `sync.Mutex`.

**atomic.Value** — тип для атомарного хранения значений произвольных типов (например, структур конфигурации).

**Deadlock (взаимная блокировка)** — ситуация, когда горутины бесконечно ждут друг друга и ни одна не может продолжить выполнение.

**Starvation (голодание)** — горутина не может получить доступ к ресурсу, так как планировщик или другие горутины постоянно отдают приоритет другим задачам.

**Livelock (живая блокировка)** — горутины постоянно меняют состояния в ответ на действия друг друга, но не продвигаются к результату.

## Как устроено

### sync.Mutex

#### Внутренняя структура

```go
type Mutex struct {
    state int32
    sema  uint32
}
```

`sema` — семафор для управления очередью ожидающих горутин.

`state` — 32-битное поле, разбитое на флаги *(точная побитовая раскладка не раскрыта в источниках; ниже — из исходного кода Go)*:

| Биты | Константа | Смысл |
|---|---|---|
| 0 | `mutexLocked` | 1 = мьютекс захвачен, 0 = свободен |
| 1 | `mutexWoken` | Горутина пробуждена и пытается захватить лок |
| 2 | `mutexStarving` | Включён режим голодания (starvation mode) |
| 3–31 | счётчик waiters | Количество горутин в очереди ожидания |

#### Алгоритм Lock / Unlock

**Fast path:**
1. Один атомарный CAS: попытка изменить `state` с `0` на `1` (`mutexLocked`).
2. Успех → мьютекс захвачен немедленно, без обращения к планировщику.

**Slow path** (вызывается если fast path не сработал):
1. **Spinning:** если система многоядерная, горутина выполняет несколько пустых процессорных циклов (spin) — в расчёте, что мьютекс вот-вот освободится.
2. **Очередь:** если spinning не помог — инкрементировать счётчик waiters в `state`, вызвать `runtime_SemacquireMutex` → горутина засыпает на семафоре `sema`.

**Unlock:**
1. Атомарно сбросить бит `mutexLocked`.
2. Если `waiters > 0` → вызвать `runtime_Semrelease`, пробудить первую горутину из очереди.

#### Starvation mode

*(Не описан в источниках явно; из исходного кода Go, на который ссылаются видеоматериалы)*

**Проблема нормального режима:** разбуженная из очереди горутина соревнуется с новыми горутинами, которые уже крутятся на CPU. Она часто проигрывает и снова уходит в хвост очереди.

**Включение:** если горутина ждёт мьютекс **более 1 миллисекунды** — устанавливается бит `mutexStarving`.

**Режим голодания:**
- `Unlock()` передаёт мьютекс строго первой горутине в FIFO-очереди.
- Новые горутины не крутятся в spin и не пытаются захватить лок — сразу встают в хвост очереди.

**Выключение:** когда получившая мьютекс горутина видит, что ждала меньше 1 мс, либо была последней в очереди.

#### Паники

| Ситуация | Сообщение |
|---|---|
| Все горутины спят — нет прогресса | `fatal error: all goroutines are asleep - deadlock!` |
| `Unlock()` незаблокированного мьютекса | `fatal error: sync: unlock of unlocked mutex` |

### sync.RWMutex

#### Внутренняя структура

```go
type RWMutex struct {
    w           Mutex   // held if there are pending writers
    writerSem   uint32  // semaphore for writers to wait for completing readers
    readerSem   uint32  // semaphore for readers to wait for completing writers
    readerCount int32   // number of pending readers
    readerWait  int32   // number of departing readers
}
```

#### Алгоритм

**RLock():**
1. Атомарно инкрементировать `readerCount`.
2. Если `readerCount` стал отрицательным (значит активен write-lock) → заблокироваться на `readerSem`.

**RUnlock():**
1. Атомарно декрементировать `readerCount`.
2. Если `readerCount` стал 0 и есть ожидающий писатель → пробудить его через `writerSem`.

**Lock() (write):**
1. Захватить внутренний `w Mutex` — запрещает новые читатели и других писателей.
2. Атомарно вычесть из `readerCount` большое число (`rwmutexMaxReaders`) → переводит в отрицательное значение (флаг «есть писатель»).
3. Сохранить в `readerWait` количество уже работающих читателей.
4. Заблокироваться на `writerSem`, пока последний из работающих читателей не вызовет `RUnlock()`.

**Unlock() (write):**
1. Восстановить `readerCount` (убрать отрицательное смещение) → пробуждает всех ожидающих читателей через `readerSem`.
2. Освободить внутренний `w Mutex`.

**TryLock() / TryRLock()** *(добавлены в Go 1.18):*
- Один атомарный CAS без засыпания. Возвращает `false` немедленно если лок занят.
- По документации Go: «правильное использование TryLock существует, но встречается редко и часто является признаком более глубокой проблемы в архитектуре».

### sync.WaitGroup

#### Внутренняя структура

```go
type WaitGroup struct {
    noCopy noCopy
    state1 [4]byte  // упакованные счётчики
    sema   uint32
}
```

- `noCopy` — запрещает копирование; `go vet` выдаст ошибку.
- `state1` — 12 байт, в которых упакованы два 32-битных счётчика: **goroutine counter** (кол-во работающих горутин) и **waiter counter** (кол-во ожидающих `Wait()`). Упаковка обеспечивает 64-битное выравнивание даже на 32-битных архитектурах.
- `sema` — семафор для блокировки `Wait()`.

#### Алгоритм Add / Done / Wait

**Add(n):**
1. Атомарно прибавить `n` к goroutine counter.
2. **Важно:** вызывать `Add(1)` строго **до** запуска горутины (`go func()`), иначе `Wait()` может сработать раньше, чем горутина успеет зарегистрироваться.

**Done():**
1. Внутри это просто `Add(-1)`.
2. Рекомендуется вызывать через `defer wg.Done()`.

**Wait():**
1. Проверить goroutine counter. Если `> 0` → инкрементировать waiter counter, заснуть на `sema`.
2. Когда goroutine counter достигает `0` — пробудить все горутины, ожидающие на `sema`.

#### Паники и гонки

| Ситуация | Результат |
|---|---|
| `Done()` вызван больше раз, чем `Add()` | `panic: sync: negative WaitGroup counter` |
| WaitGroup передана по значению (не указателю) | `Done()` работает с копией → оригинальный `Wait()` никогда не разблокируется → deadlock |

### sync.Once

#### Назначение

Гарантирует, что переданная функция будет вызвана ровно **один раз**, даже если `Do()` вызывают конкурентно из множества горутин. Пример использования: ленивая инициализация, однократное закрытие при первой ошибке.

#### Внутренняя структура *(из исходного кода Go)*

```go
type Once struct {
    done uint32
    m    Mutex
}
```

#### Алгоритм Do(f)

**Fast path:**
1. `atomic.LoadUint32(&done)` — атомарно проверить флаг.
2. Если `done == 1` → немедленно вернуться. Никаких блокировок.

**Slow path** (если `done == 0`):
1. `m.Lock()` — захватить мьютекс.
2. **Double-check:** снова проверить `done == 0` — другая горутина могла успеть выполнить `f()` пока мы ждали лок.
3. Если всё ещё `0` → вызвать `f()`.
4. `defer atomic.StoreUint32(&done, 1)` — установить флаг и освободить мьютекс.

**Паника внутри f:** если `f()` вызовет панику, `defer` всё равно установит `done = 1`. Последующие вызовы `Do()` не будут повторять `f()`.

**Edge case — счётчик вызовов, не функций:** `sync.Once` считает вызовы метода `Do`, а не уникальность переданных функций. Если передать разные функции — выполнится только первая.

**Edge case — циклическая ссылка:**

```go
var once sync.Once
once.Do(func() {
    once.Do(func() { /* ... */ }) // deadlock: внутренний мьютекс уже захвачен
})
```

Вызов `Do()` внутри самого `Do()` для той же переменной `Once` приведёт к мертвой блокировке (deadlock).

### sync.Cond

#### Назначение

Условная переменная. Используется для рассылки сигналов между горутинами: одна горутина ждёт наступления условия, другая сигнализирует о его изменении. В большинстве случаев заменяется каналами — с `sync.Cond` мало кто работает.

#### Внутренняя структура *(из исходного кода Go)*

```go
type Cond struct {
    noCopy  noCopy
    L       Locker     // обычно *sync.Mutex
    notify  notifyList // внутренняя очередь горутин
}
```

#### Алгоритм

**Wait():**
1. Добавить горутину во внутреннюю очередь `notifyList`.
2. Разблокировать `c.L` (мьютекс).
3. Приостановить горутину.
4. При пробуждении — снова захватить `c.L` перед продолжением.

**Signal():**
- Пробудить **первую** горутину из очереди `notifyList`.

**Broadcast():**
- Пробудить **все** горутины из очереди `notifyList`.

#### Паттерн с for-loop (обязателен)

```go
mu.Lock()
for !condition() {   // именно for, не if
    cond.Wait()
}
// ... работа с данными
mu.Unlock()
```

**Почему `for`, а не `if`:**
- Пока горутина спала в `Wait()`, мьютекс был отпущен. При пробуждении она снова захватывает лок — но за это время другая горутина могла вклиниться и изменить условие.
- Цикл заставляет перепроверить условие после пробуждения.
- Защита от spurious wakeups (ложных пробуждений планировщика).

### sync.Map

#### Когда использовать вместо `map + Mutex`

- **Read-heavy:** большинство операций — чтения, запись редка. Реальное преимущество по производительности начинается от **десятков и сотен** одновременных читателей.
- **Disjoint access:** разные горутины работают с разными ключами (непересекающийся доступ).

В других сценариях обычная `map + sync.Mutex` быстрее из-за накладных расходов sync.Map.

#### Внутренняя структура *(из исходного кода Go)*

```go
type Map struct {
    mu     Mutex
    read   atomic.Pointer[readOnly]  // неизменяемая копия для быстрого чтения
    dirty  map[any]*entry            // пишется только под mu
    misses int                       // счётчик промахов мимо read
}
```

- `read` — атомарный указатель на неизменяемую map. Чтение без лока (fast path).
- `dirty` — обычная map с новыми ключами. Доступ защищён `mu`.
- `misses` — при накоплении промахов `dirty` «продвигается» в `read`.

#### Алгоритм

**Load:**
1. Атомарно читать из `read`. Если ключ есть → вернуть значение (без мьютекса).
2. Если нет → `mu.Lock()`, перепроверить `read`, искать в `dirty`. Инкрементировать `misses`.
3. Если `misses >= len(dirty)` → `dirty` атомарно становится новым `read`, `dirty = nil`.

**Store:**
1. Если ключ есть в `read` → атомарно заменить значение (без мьютекса).
2. Если нет → `mu.Lock()`, записать в `dirty`.

**Delete:**
1. Если ключ в `read` → атомарно пометить как `expunged` (логическое удаление без мьютекса).
2. Физически удалится при следующей пересборке `dirty`.

#### API

| Метод | Описание |
|---|---|
| `Store(key, value)` | Записать пару |
| `Load(key)` | Читать; возвращает `(value, ok)` |
| `LoadOrStore(key, value)` | Атомарно: читать или записать если нет |
| `Delete(key)` | Удалить ключ |
| `Range(func(k, v any) bool)` | Обход всех пар |

### sync/atomic

#### Барьеры памяти

Атомарность обеспечивается специальными инструкциями CPU. В многопроцессорных средах дополняется барьерами памяти для корректной видимости изменений между ядрами:
- **Acquire** — горутина видит самые свежие данные от всех остальных.
- **Release** — все другие горутины увидят изменения текущей горутины.

**Цена:** частое использование атомиков вызывает инвалидацию кэшей ядер процессора — при каждом обновлении ядра вынуждены пересинхронизировать кэши, что замедляет программу при высокой конкуренции.

**Паттерн Monitor Goroutine:** при очень высокой частоте обновления (высококонкурентный atomic) эффективнее собрать всю логику управления состоянием в одной горутине-«мониторе», которая получает команды через каналы — вместо того чтобы все ядра CPU конкурировали за один atomic.

#### Классический API (указатели)

| Функция | Описание |
|---|---|
| `atomic.AddInt64(&v, delta)` | Атомарно прибавить `delta`; вернуть новое значение |
| `atomic.LoadInt64(&v)` | Атомарно читать |
| `atomic.StoreInt64(&v, val)` | Атомарно записать |
| `atomic.SwapInt64(&v, new)` | Заменить, вернуть старое |
| `atomic.CompareAndSwapInt64(&v, old, new)` | CAS: заменить если `v == old`; вернуть `bool` |

#### Типизированный API (Go 1.19+)

*(Новые типы из Go 1.19 не описаны в источниках явно; из исходного кода Go)*

До Go 1.19: переменная `var count int64`, передаётся по указателю `&count` — легко случайно прочитать напрямую без atomic и получить data race.

С Go 1.19 — типизированные обёртки: `atomic.Int64`, `atomic.Int32`, `atomic.Uint64`, `atomic.Bool`, `atomic.Pointer[T]`.

**Преимущества:**
- Скрывают значение в неэкспортируемом поле — прямой доступ без методов невозможен.
- Содержат `noCopy` — `go vet` предупреждает при копировании.
- Автоматическое 64-битное выравнивание — нет паник на 32-битных архитектурах.

```go
var counter atomic.Int64

counter.Add(1)
counter.Store(42)
v := counter.Load()
counter.CompareAndSwap(old, new)
```

#### atomic.Value

Тип для атомарного хранения значений произвольных типов.

| Метод | Описание |
|---|---|
| `Load()` | Атомарно читает; возвращает `any` |
| `Store(val)` | Атомарно записывает; `val` не должен быть `nil` |
| `Swap(val)` | Атомарно заменяет; возвращает старое |
| `CompareAndSwap(old, new)` | CAS для произвольного типа |

**Ограничение:** все вызовы `Store` должны передавать значения одного и того же конкретного типа. Иначе — паника.

## Детали и нюансы

### Мьютекс нельзя копировать

После первого использования копирование мьютекса копирует внутреннее состояние (включая флаги блокировки). Это приводит к непредсказуемому поведению — копия может оказаться «уже заблокированной». Передавать только по указателю или инкапсулировать в структуру. `go vet` обнаруживает копирование `sync.Mutex`, `sync.RWMutex`, `sync.WaitGroup`, `sync.Once`.

### WaitGroup: Add до старта горутины

```go
// ✅ Правильно
wg.Add(1)
go func() {
    defer wg.Done()
    // ...
}()

// ❌ Неправильно — гонка: Wait() может сработать раньше Add()
go func() {
    wg.Add(1)
    defer wg.Done()
    // ...
}()
```

### WaitGroup: передача по указателю

```go
// ❌ Передача по значению — Done() отработает для копии, Wait() не разблокируется
func process(wg sync.WaitGroup) { ... }

// ✅ По указателю
func process(wg *sync.WaitGroup) { ... }
```

### Deadlock — причины и профилактика

| Причина | Профилактика |
|---|---|
| Горутина читает из канала, в который никто не пишет | Закрывать каналы после записи |
| Повторный захват мьютекса без Unlock (нерекурсивный) | `defer m.Unlock()` сразу после `Lock()` |
| WaitGroup передана по значению | Передавать только по указателю |
| `select` без готовых веток и без `default` | Проектировать граф операций заранее |

### Starvation (голодание)

В Go `select` выбирает готовую ветку **случайно** — встроенная защита от голодания каналов. Планировщик использует Work Stealing для равномерного распределения нагрузки.

### Livelock (живая блокировка)

Горутины «вежливо» уступают друг другу, постоянно откатывая состояние — но не продвигаясь вперёд. Аналогия: два человека в узком коридоре бесконечно уступают дорогу в одну сторону.

## Сравнения и trade-offs

| | sync.Mutex | sync.RWMutex | sync.Map | sync/atomic |
|---|---|---|---|---|
| Уровень | Высокий | Высокий | Высокий | Низкий (CPU) |
| Скорость | Средняя | Высокая при read-heavy | Высокая при read-heavy | Самая высокая |
| Применимость | Универсально | Read >> Write | Read-heavy / disjoint keys | Одна переменная |
| Сложность | Низкая | Средняя | Средняя | Высокая |
| «Транзакционность» | ✅ | ✅ | ✗ | ✗ |
| Копирование | ✗ | ✗ | ✗ | ✅ |

| Сценарий | Atomic | Mutex |
|---|---|---|
| Один счётчик, один флаг | ✅ Быстрее | ✓ |
| Группа связанных полей | ✗ Нет согласованности | ✅ |
| Сложные структуры (map, slice) | ✗ | ✅ |
| Read-heavy кэш | ✓ atomic.Value | ✅ RWMutex |

## Примеры

### 1. sync.Mutex — защита map

```go
type SafeCache struct {
    m     sync.Mutex
    items map[string]string
}

func (c *SafeCache) Set(key, value string) {
    c.m.Lock()
    defer c.m.Unlock()
    c.items[key] = value
}
```

### 2. sync.RWMutex — read-heavy кэш

```go
type SafeCache struct {
    m     sync.RWMutex
    items map[string]string
}

func (c *SafeCache) Get(key string) string {
    c.m.RLock()
    defer c.m.RUnlock()
    return c.items[key]
}

func (c *SafeCache) Set(key, value string) {
    c.m.Lock()
    defer c.m.Unlock()
    c.items[key] = value
}
```

### 3. sync.WaitGroup — ожидание горутин

```go
var wg sync.WaitGroup

for i := 0; i < 5; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        fmt.Println("goroutine", id)
    }(i)
}

wg.Wait()
fmt.Println("all done")
```

### 4. sync.Once — ленивая инициализация

```go
var (
    instance *DB
    once     sync.Once
)

func GetDB() *DB {
    once.Do(func() {
        instance = &DB{} // выполнится ровно один раз
    })
    return instance
}
```

### 5. sync.Cond — паттерн ожидания условия

```go
var mu sync.Mutex
var cond = sync.NewCond(&mu)
var ready bool

// Ожидающая горутина
go func() {
    mu.Lock()
    for !ready {        // for, не if
        cond.Wait()
    }
    fmt.Println("условие выполнено")
    mu.Unlock()
}()

// Сигнализирующая горутина
time.Sleep(time.Second)
mu.Lock()
ready = true
cond.Signal()
mu.Unlock()
```

### 6. sync.Map — конкурентный кэш

```go
var m sync.Map

m.Store("key", "value")

if v, ok := m.Load("key"); ok {
    fmt.Println(v)
}

m.Range(func(k, v any) bool {
    fmt.Println(k, v)
    return true // продолжить обход
})
```

### 7. Атомарный счётчик (классический API)

```go
var counter int64
var wg sync.WaitGroup

for i := 0; i < 1000; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        atomic.AddInt64(&counter, 1)
    }()
}
wg.Wait()
fmt.Println(atomic.LoadInt64(&counter))
```

### 8. atomic.Int64 (Go 1.19+)

```go
var counter atomic.Int64

counter.Add(1)
fmt.Println(counter.Load())
```

### 9. CompareAndSwap — оптимистичная блокировка

```go
var state int64
swapped := atomic.CompareAndSwapInt64(&state, 0, 1)
if swapped {
    fmt.Println("state изменён с 0 на 1")
}
```

### 10. atomic.Value — атомарное обновление конфигурации

```go
var config atomic.Value

config.Store(MyConfig{Timeout: 30})

cfg := config.Load().(MyConfig)
fmt.Println(cfg.Timeout)
```

## Важно / подводные камни / best practices

**Подводные камни:**

- **Забытый `Unlock()` → deadlock.** Всегда: `defer m.Unlock()` сразу после `Lock()`.
- **Копирование мьютекса.** Нельзя копировать `sync.Mutex`, `sync.RWMutex`, `sync.WaitGroup`, `sync.Once`. Передавать только по указателю. `go vet` обнаруживает.
- **Atomic не заменяет мьютекс для группы полей.** Два отдельных `atomic.Store` не атомарны вместе — между ними может вклиниться другая горутина.
- **Частые атомарные записи → инвалидация кэша.** При высокой конкуренции атомики могут быть медленнее мьютекса.
- **Повторный Lock без Unlock → deadlock.** Mutex в Go не рекурсивный.
- **`wg.Add(1)` внутри горутины — гонка.** `Add` должен вызываться **до** `go`.
- **`sync.Once` с паникой.** Если `f()` паникует — `done` всё равно выставляется в `1`. Повторный `Do()` не запустит `f` снова.
- **`sync.Map` не для всего.** Быстрее обычной map+Mutex только при read-heavy или disjoint ключах. В write-heavy сценариях — медленнее.
- **`sync.Cond.Wait()` только в цикле `for`.** Условие нужно перепроверять после пробуждения.

**Best practices:**

- `defer m.Unlock()` — сразу после `Lock()`, всегда.
- Держи критическую секцию минимальной: не выполняй внутри сетевые запросы или тяжёлые вычисления.
- Инкапсулируй мьютекс и защищаемые данные в одну структуру (**паттерн Confinement**): мьютекс и защищаемые поля — в одном типе, доступ к данным только через методы структуры.
- `sync.RWMutex` только если **реально** чтений значительно больше записей (десятки и сотни читателей).
- Используй `atomic` для простых счётчиков и флагов; для всего остального — мьютекс.
- При очень высокой конкуренции за атомик — рассмотри **паттерн Monitor Goroutine**: одна горутина управляет состоянием через каналы.
- Запускай тесты с флагом `-race` для обнаружения гонок данных.
- Предпочитай типизированный API (`atomic.Int64`, `atomic.Bool`) классическому — безопаснее и без адресов.
- **Правило горутин:** если горутина порождает другую горутину — она обязана иметь возможность её остановить (через `done`-канал или `context`). Иначе — утечки горутин.

## Карточки для повторения

#flashcards/go/sync

Что такое sync.Mutex?::Примитив взаимного исключения. Гарантирует, что критическую секцию в конкретный момент выполняет только одна горутина. Содержит поля `state int32` и `sema uint32`.

Какие биты хранит поле `state` в sync.Mutex?
?
- bit 0 `mutexLocked` — мьютекс захвачен
- bit 1 `mutexWoken` — горутина пробуждена
- bit 2 `mutexStarving` — режим голодания
- bits 3–31 — счётчик ожидающих горутин

Что такое fast path и slow path в Lock()?
?
- **Fast path:** один CAS из 0 в 1 — если мьютекс свободен, захват без планировщика
- **Slow path:** spinning (несколько spin-циклов на многоядерных), затем засыпание на семафоре `sema`

Что такое starvation mode в sync.Mutex?::Режим, включающийся если горутина ждёт > 1 мс. В этом режиме Unlock() передаёт мьютекс строго первой горутине в FIFO-очереди; новые горутины не спинят и сразу встают в хвост.

Что такое sync.RWMutex?::Мьютекс с разделённым доступом. Несколько горутин могут держать **read-lock** одновременно. Write-lock монопольный — блокирует всех читателей и писателей.

Какие внутренние поля у sync.RWMutex?
?
- `w Mutex` — внутренний мьютекс для писателей
- `writerSem` — семафор: писатель ждёт завершения читателей
- `readerSem` — семафор: читатели ждут завершения писателя
- `readerCount` — кол-во активных читателей
- `readerWait` — кол-во читателей при захвате write-lock

Что происходит при вызове Lock() в RWMutex?
?
1. Захватить внутренний `w Mutex`
2. Атомарно вычесть `rwmutexMaxReaders` из `readerCount` (переводит в отрицательное — флаг «есть писатель»)
3. Сохранить в `readerWait` кол-во работающих читателей
4. Заблокироваться на `writerSem` до завершения последнего читателя

Что такое sync.WaitGroup?::Счётчик горутин для синхронизации точки завершения. `Add(n)` увеличивает, `Done()` уменьшает (= `Add(-1)`), `Wait()` блокируется до нуля.

Почему `wg.Add(1)` нужно вызывать ДО запуска горутины?::Гонка: `Wait()` может отработать до того, как горутина вызовет `Add(1)`, и не дождётся её завершения.

Какие паники может вызвать sync.WaitGroup?
?
- `panic: sync: negative WaitGroup counter` — вызовов Done() больше, чем Add()
- Передача по значению → Done() работает с копией → оригинальный Wait() не разблокируется → deadlock

Что такое sync.Once?::Гарантирует, что переданная в Do() функция выполнится ровно один раз, даже при конкурентных вызовах. Реализована через atomic флаг + двойную проверку с мьютексом.

Что происходит если функция внутри sync.Once паникует?::Флаг `done` всё равно устанавливается в 1. Повторные вызовы Do() не будут запускать функцию снова.

Что такое sync.Cond?::Условная переменная. Wait() — ждать сигнала (освобождает мьютекс до сигнала). Signal() — разбудить одну горутину. Broadcast() — разбудить все.

Почему в паттерне sync.Cond нужен `for`, а не `if`?::Условие нужно перепроверять: пока горутина спала, другая могла вклиниться и изменить условие. Также защита от spurious wakeups.

Когда использовать sync.Map вместо map + Mutex?::При read-heavy нагрузке или disjoint access (разные горутины работают с разными ключами). В write-heavy сценариях обычная map+Mutex быстрее.

Какие две внутренние map у sync.Map?::1. `read` — атомарный указатель на неизменяемую map, чтение без мьютекса. 2. `dirty` — обычная map с новыми ключами, защищена мьютексом `mu`.

Что такое CompareAndSwap (CAS)?::Атомарная операция: замена значения происходит только если текущее значение равно ожидаемому. Возвращает `bool`. На CAS построен сам `sync.Mutex`.

Что такое барьеры памяти Acquire и Release?
?
- **Acquire** — горутина видит самые свежие данные от всех остальных
- **Release** — все другие горутины увидят изменения текущей горутины

Перечисли основные функции классического API sync/atomic.
?
- `Add` — атомарная арифметика
- `Load` — атомарное чтение
- `Store` — атомарная запись
- `Swap` — замена с возвратом старого
- `CompareAndSwap` — замена если равно ожидаемому

Что добавили в пакет atomic в Go 1.19?::Типизированные обёртки: `atomic.Int64`, `atomic.Int32`, `atomic.Bool`, `atomic.Pointer[T]`. Скрывают значение внутри, запрещают прямой доступ, содержат noCopy, гарантируют 64-битное выравнивание.

Что такое atomic.Value и какое у него ограничение?::Тип для атомарного хранения произвольных значений (Store/Load/Swap/CAS). Ограничение: все вызовы Store должны передавать один и тот же конкретный тип — иначе паника.

Когда atomic может быть медленнее мьютекса?::При высокой конкуренции: каждое атомарное обновление вызывает инвалидацию кэшей ядер процессора — они вынуждены пересинхронизироваться.

Что такое Deadlock?::Ситуация, когда горутины бесконечно ждут друг друга и ни одна не может продолжить выполнение. Go выбрасывает: `fatal error: all goroutines are asleep - deadlock!`

Что такое Starvation (голодание)?::Горутина не может получить доступ к ресурсу — планировщик или другие горутины постоянно отдают приоритет другим задачам.

Что такое Livelock?::Горутины постоянно меняют состояния в ответ на действия друг друга, но не продвигаются к результату. Аналогия: два человека в узком коридоре бесконечно уступают дорогу в одну сторону.

Какие паники вызывает sync.Mutex?
?
- `fatal error: all goroutines are asleep - deadlock!` — все горутины заблокированы
- `fatal error: sync: unlock of unlocked mutex` — Unlock() у незаблокированного мьютекса

Можно ли вложить Lock внутри Lock для одного мьютекса?::Нет. Mutex в Go **не рекурсивный**. Повторный Lock() без Unlock() → deadlock.

Когда использовать RWMutex вместо Mutex?::Когда операций чтения **значительно больше**, чем записей (read-heavy). Реальный выигрыш производительности начинается от **десятков и сотен** одновременных читателей. При равном соотношении накладные расходы RWMutex не оправданы.

Edge case — sync.Once: что происходит если внутри Do() вызвать Do() для той же переменной?::Deadlock. Внутренний мьютекс уже захвачен первым вызовом, второй будет ждать его освобождения вечно.

Edge case — sync.Once считает что?::Вызовы метода `Do`, а не уникальность переданных функций. Если вызвать `Do` с разными функциями, выполнится только первая — последующие вызовы игнорируются.

Что такое паттерн Monitor Goroutine?::Альтернатива высококонкурентному atomic: одна горутина-«монитор» управляет всем состоянием и получает команды через каналы. Устраняет постоянную инвалидацию кэшей CPU при высокой конкуренции за атомик.

Что такое паттерн Confinement применительно к мьютексу?::Инкапсуляция мьютекса и защищаемых данных в одну структуру: мьютекс и поля — в одном типе, доступ к данным только через методы структуры. Предотвращает случайный доступ к данным в обход мьютекса.

Правило горутин в Go?::Если горутина порождает другую горутину — она обязана иметь возможность её остановить (через `done`-канал или `context`). Горутина без механизма остановки приводит к утечкам.
