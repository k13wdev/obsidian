## Суть и контекст

Стек (stack) и куча (heap) — два основных сегмента памяти, используемых процессом для хранения данных во время выполнения программы. Каждая запущенная программа получает оба этих сегмента.

В языке Go управление памятью частично автоматизировано: компилятор самостоятельно решает, куда поместить переменную — на стек или в кучу. Это решение принимается на этапе компиляции через механизм escape analysis.

---

## Ключевые идеи

- Стек организован по принципу LIFO и работает быстрее кучи.
- Куча предназначена для объектов с динамическим размером и управляется сборщиком мусора (GC).
- В Go размещение переменных определяет компилятор через Escape Analysis — программист не контролирует это напрямую.
- Переменная «утекает» в кучу, если компилятор не может доказать, что она не выйдет за пределы своей функции.
- Стек горутины в Go может динамически расти — при нехватке места данные переаллоцируются в новом месте памяти.
- Память стека не очищается сразу после завершения функции — она помечается как невалидная и перетирается позже.
- Выделение памяти в куче медленнее и создаёт нагрузку на GC.
- Приоритет — держать переменные на стеке, но без преждевременной оптимизации.

## Определения и термины

**Стек (Stack)** — область памяти, организованная как абстрактный тип данных по принципу LIFO (Last-In, First-Out). Аналог: стопка тарелок — положить новую или взять можно только сверху.

**Куча (Heap)** — сегмент данных памяти, предназначенный для динамического распределения объектов, размер которых может изменяться во время выполнения. В отличие от стека, куча обычно растёт «вверх» — к старшим адресам.

**Stack Pointer (указатель стека)** — специальный регистр процессора, который в каждый момент времени ссылается на текущую вершину стека.

**Stack Frame (стековый фрейм)** — область данных, создаваемая в стеке для каждой вызванной функции. Содержит: входные параметры, локальные и временные переменные, адрес возврата.

**LIFO (Last-In, First-Out)** — принцип организации данных: последний добавленный элемент извлекается первым.

**Escape Analysis (анализ утечки)** — механизм компилятора Go, который на этапе компиляции определяет, где будет размещена переменная: на стеке или в куче.

**Dangling pointer (висячий указатель)** — указатель, ссылающийся на уже невалидную область памяти (например, на очищенный фрейм стека).

**Invalid memory** — память стека, помеченная как недействительная после завершения функции. Данные физически остаются там до перетирания фреймом новой функции.

**GC (Garbage Collector, сборщик мусора)** — компонент Go-рантайма, отвечающий за освобождение памяти в куче. Идентифицирует объекты, на которые больше нет ссылок, и освобождает их.

## Как устроено

### Стек

1. При вызове функции в стеке создаётся **stack frame** с параметрами, локальными переменными и адресом возврата.
2. Stack Pointer сдвигается вперёд, указывая на вершину нового фрейма.
3. При завершении функции Stack Pointer сдвигается назад — память фрейма помечается как **invalid memory**.
4. Данные физически не удаляются — они остаются в памяти до тех пор, пока следующая функция не займёт это место и не перетрёт их своими значениями.

**Динамический рост стека в Go (Stack copying):**
- Стек горутины не имеет фиксированного размера.
- Механизм роста называется **Stack copying**: при нехватке места стек аллоцируется в новом месте памяти с бо́льшим размером, и все переменные со старого стека (включая фреймы всех функций в стеке вызовов) копируются на новые адреса.
- Старый сегментированный стек подхода (segmented stacks) — в источниках не раскрыт.
- При этом физические адреса переменных на стеке могут измениться.

### Куча

1. Объекты помещаются в кучу в момент аллокации.
2. Куча растёт к старшим адресам.
3. За освобождение памяти отвечает **сборщик мусора (GC)**: он находит объекты, на которые больше нет ссылок, и освобождает занимаемую ими память.
4. Аллокация в куче медленнее аллокации в стеке.

### Escape Analysis

Компилятор Go строит **взвешенный граф зависимостей** и анализирует область видимости каждой переменной:

- **Стек** — если компилятор может доказать, что переменная не используется за пределами функции, в которой объявлена.
- **Куча** — если компилятор не может доказать это, переменная «утекает» в кучу (`escapes to heap`). Go делает это автоматически, чтобы избежать dangling pointers.

**Условия утечки переменной в кучу:**

| Условие | Пример |
|---|---|
| Функция возвращает ссылку на локальную переменную | `return &b` |
| Объект слишком велик для стека (лимит ~10 МБ) | большой массив |
| Размер слайса или мапы неизвестен на этапе компиляции | `make([]int, n)` |
| Переменная передаётся в функцию, принимающую интерфейс | `fmt.Println(v)` |
| Литералы функций | замыкания |
| Внутренние структуры коллекций | бакеты map, базовый массив slice, массив байт string, кольцевой буфер channel |

## Детали и нюансы

**Динамическая переаллокация стека:**
Когда Go переаллоцирует стек горутины, данные копируются в новое место памяти. Это означает, что **адреса переменных на стеке могут измениться в процессе выполнения программы**.

**Invalid Memory:**
После завершения функции её стековый фрейм не очищается немедленно. Данные остаются как невалидная память и перетираются лишь при вызове новой функции, которая займёт это место.

**Интерфейсная ловушка:**
Даже если логически утечка не нужна, передача переменной в функцию с параметром `interface{}` (например, `fmt.Println(v)`) часто «выталкивает» `v` в кучу — потому что тип и размер могут быть неочевидны компилятору.

**Гарантии на собеседовании:**
Нельзя гарантировать, что конкретная переменная попадёт на стек. Решение всегда принимает компилятор на основе сложного алгоритма (взвешенного графа). Безопаснее формулировать так: «Если переменная не возвращается по ссылке, она, скорее всего, будет на стеке».

**Директива `//go:noinline`:**
Используется, чтобы компилятор не встраивал (inlining) код функции в место вызова — это могло бы скрыть факт утечки переменной при анализе.

## Сравнения и trade-offs

| Критерий | Стек | Куча |
|---|---|---|
| Скорость аллокации | Быстро (смещение указателя) | Медленнее |
| Управление памятью | Автоматически (по завершении функции) | Сборщик мусора (GC) |
| Размер объектов | Ограничен (~10 МБ) | Не ограничен (в рамках RAM) |
| Время жизни | Ограничено областью видимости функции | Пока есть ссылки |
| Нагрузка на GC | Нет | Есть |
| Адреса переменных | Могут меняться (при росте стека) | Стабильны |
| Направление роста | «Вниз» | «Вверх» (к старшим адресам) |

**Трейд-офф: копирование vs передача по ссылке:**

Передача большого объекта (5–9 МБ) по значению создаёт его копию на стеке — это может быть **медленнее**, чем аллокация в куче и передача по ссылке. Выбор требует проверки через профилирование.

## Примеры

### 1. Возврат ссылки — переменная утекает в кучу

```go
//go:noinline
func bar() *int {
    b := 10
    return &b // b утечет в кучу
}
```

Директива `//go:noinline` запрещает инлайнинг, чтобы факт утечки был виден в отчёте компилятора.

### 2. Команда для просмотра escape analysis

```bash
go build -gcflags="-m" main.go
```

Для более детального вывода:

```bash
go build -gcflags="-m -m" main.go
```

Компилятор выводит строки вида: `move to heap: v` — это означает, что переменная `v` была перемещена в кучу.

## Важно / подводные камни / best practices

**Best practices:**
- Держи переменные на стеке, когда это возможно — это снижает нагрузку на GC.
- Используй `go build -gcflags="-m"` для проверки, куда попадают переменные.
- Применяй EA-оптимизации только после того, как профилирование (`pprof`) показало, что аллокации в куче являются узким местом. Не занимайся преждевременной оптимизацией, если программа работает достаточно быстро.

**Подводные камни:**
- Динамический рост стека горутины меняет физические адреса переменных — не полагайся на стабильность адресов стека.
- Использование `interface{}` в качестве типа параметра часто провоцирует утечку — даже там, где это не очевидно (например, `fmt.Println`).
- После возврата из функции данные её фрейма физически не исчезают — не полагайся на немедленную очистку.
- Нельзя гарантировать размещение переменной на стеке — всегда решает компилятор.
