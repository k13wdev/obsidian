# Интерфейсы (interfaces)

---

## Level 1 — Raw Knowledge

Интерфейс в Go — это программная синтаксическая структура (контракт или спецификация), которая описывает набор методов, обязательных для реализации конкретным типом.

[Источник: NotebookLM — Интерфейсы]

Интерфейсы определяют поведение объектов, а не их данные или внутреннее устройство. В отличие от абстрактных классов в других языках, не могут содержать никакой реализации.

[Источник: NotebookLM — Интерфейсы]

Главная легитимная цель применения интерфейсов в Go — уменьшение связанности кода (decoupling) и разрыв связей между использованием и реализацией.

> «Единственная легитимная цель применения интерфейсов — это уменьшение степени связанности»

[Источник: Данил Подольский, видео «Интерфейсы в Go»]

Интерфейс — это **рубеж обороны против распространения сложности** по программе.

> «Путь современного программиста — это самоограничение и прямота»

[Источник: Данил Подольский, видео «Интерфейсы в Go»]

Разработка интерфейса тождествена разработке **спецификации**.

[Источник: Серия видеолекций skill issue]

## Level 2 — Important Points

- Интерфейс — контракт поведения, а не структура данных
- Реализация неявная (implicit) — нет ключевого слова `implements`
- Duck Typing: если тип реализует все методы — компилятор автоматически считает его реализующим интерфейс
- Внутри интерфейсная переменная — структура из двух указателей (16 байт на 64-бит)
- Пустой интерфейс `interface{}` (`any`) удовлетворяется любым типом, но устроен иначе
- Интерфейс `nil` только когда нет ни типа, ни значения — главная ловушка
- Type assertion и type switch работают через рефлексию → ошибки только в рантайме
- Интерфейс объявляется **там, где используется**, а не там, где реализуется
- `[]MyStruct` нельзя напрямую передать как `[]MyInterface` — требует цикл

---

## Level 3 — Core Insight

Интерфейс в Go — это механизм уменьшения связанности через неявный контракт поведения. Тип реализует интерфейс автоматически, если у него есть все нужные методы. Мощь интерфейса — в том, что производитель и потребитель не знают друг о друге.

---

## Level 4 — Ultra Summary

Интерфейс = контракт поведения с неявной реализацией; единственная цель — уменьшить связанность кода.

## Как работает

### Синтаксис объявления

```go
package myInterface

type Shape interface {
    Area() float64
    Perimeter() float64
}
```

[Источник: Михалис Цукалос, «Golang для профи»]

### Неявная реализация

Связывание происходит только в момент присваивания конкретного типа интерфейсной переменной. Компилятор автоматически проверяет соответствие.

[Источник: NotebookLM — Интерфейсы]

```go
type square struct{ X float64 }
type circle struct{ R float64 }

func (s square) Area() float64      { return s.X * s.X }
func (s square) Perimeter() float64 { return 4 * s.X }

func (c circle) Area() float64      { return c.R * c.R * math.Pi }
func (c circle) Perimeter() float64 { return 2 * c.R * math.Pi }
```

[Источник: Михалис Цукалос, «Golang для профи»]

Оба типа автоматически удовлетворяют `Shape` — без единого ключевого слова.

### Внутреннее устройство: iface, eface, itab

Интерфейсная переменная занимает в памяти размер двух ссылок: **16 байт** на 64-бит, **8 байт** на 32-бит.

[Источник: NotebookLM — Интерфейсы]

**Обычный интерфейс — `iface`:**

Состоит из двух указателей:
1. Указатель на **itab** — таблицу, содержащую описание интерфейса и информацию о конкретном типе
2. Указатель на **данные** — само конкретное значение

[Источник: NotebookLM — Интерфейсы]

**Пустой интерфейс — `eface`:**

Устроен иначе: не требует методов, поэтому нет ссылки на описание интерфейса. Хранит только:
1. Указатель на **тип** данных
2. Указатель на **данные**

[Источник: NotebookLM — Интерфейсы]

### Пустой интерфейс

```go
// fmt.Println внутри стандартной библиотеки
func Println(a ...interface{}) (n int, err error)
```

[Источник: Михалис Цукалос, «Golang для профи»]

Любой тип удовлетворяет пустому интерфейсу. Но с переменной `interface{}` невозможно сделать ничего полезного без приведения типов — проверка совместимости переносится из компиляции в рантайм.

[Источник: NotebookLM — Интерфейсы]

> «Пустой интерфейс — универсальная отмычка»

[Источник: Серия видеолекций skill issue]

### Type Assertion

Синтаксис: `x.(T)`

**Опасная форма** — panic при несовпадении:
```go
i := myInt.(int) // паника если тип не int
```

**Безопасная форма с `ok`:**
```go
v, ok := myInt.(float64)
if ok {
    fmt.Println(v)
} else {
    fmt.Println("Failed without panicking!")
}
```

[Источник: Михалис Цукалос, «Golang для профи»]

### Type Switch

```go
func tellInterface(x interface{}) {
    switch v := x.(type) {
    case square:
        fmt.Println("This is a square!")
    case circle:
        fmt.Printf("%v is a circle!\n", v)
    case rectangle:
        fmt.Println("This is a rectangle!")
    default:
        fmt.Printf("Unknown type %T!\n", v)
    }
}
```

[Источник: Михалис Цукалос, «Golang для профи»]

Ключевое слово `type` используется только в этом контексте. Если информация о типе известна (даже если значение `nil`), switch выберет соответствующий кейс.

### Embedding (встраивание)

```go
type ReadWriter interface {
    Reader  // встраиваем интерфейс Reader
    Writer  // встраиваем интерфейс Writer
}
```

[Источник: Михалис Цукалос, «Golang для профи»]

Тип, удовлетворяющий `ReadWriter`, автоматически удовлетворяет и `Reader`, и `Writer`.

[Источник: NotebookLM — Интерфейсы]

## Детали и ограничения

### Место объявления

Интерфейс должен объявляться **в месте использования**, а не рядом с реализующим типом. Если интерфейс нужен в 100 местах — лучше скопировать 100 раз, чем выносить в общий пакет.

> «Маленькое копирование лучше маленькой связанности»

[Источник: Данил Подольский, видео «Интерфейсы в Go»]

### Минимализм методов

Интерфейс должен содержать только те методы, которые необходимы в данный момент. Если структуре нужны три метода — не включай все двадцать.

[Источник: Серия видеолекций skill issue]

### Интерфейс vs абстрактный класс

Интерфейс в Go **не может содержать поведения** — в отличие от абстрактных классов в других языках.

[Источник: NotebookLM — Интерфейсы]

### «Прибитые гвоздями» типы

Если метод интерфейса принимает или возвращает тип из другого пакета — интерфейс жёстко привязывается к этому пакету. Решение: использовать базовые типы или выносить типы в отдельные пакеты.

[Источник: Серия видеолекций skill issue]

### Производительность

Type assertion и type switch используют рефлексию — ошибки обнаруживаются только в рантайме. Замедляют программу.

[Источник: Данил Подольский, видео «Интерфейсы в Go»]

### Стандартная библиотека — не пример для подражания

> Стандартная библиотека развивалась хаотично и представляет собой «один большой развесистый модуль» — она не является примером хорошего проектирования интерфейсов.

[Источник: Данил Подольский, видео «Интерфейсы в Go»]

### Ad-hoc полиморфизм

Интерфейсы обеспечивают только ad-hoc полиморфизм (полиморфизм «по месту»). Истинный параметрический полиморфизм — это генерики (Go 1.18+).

[Источник: Серия видеолекций skill issue]

## Примеры

### Полный цикл: объявление → реализация → использование

```go
// 1. Объявление интерфейса (в месте использования)
type Shape interface {
    Area() float64
    Perimeter() float64
}

// 2. Реализация (неявная — нет implements)
type square struct{ X float64 }
func (s square) Area() float64      { return s.X * s.X }
func (s square) Perimeter() float64 { return 4 * s.X }

// 3. Использование
func Calculate(x Shape) {
    fmt.Println(x.Area())
    fmt.Println(x.Perimeter())
}
```

[Источник: Михалис Цукалос, «Golang для профи»]

### Стандартные интерфейсы библиотеки

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}
type Writer interface {
    Write(p []byte) (n int, err error)
}
type error interface {
    Error() string
}
```

[Источник: Михалис Цукалос, «Golang для профи»]

### Ловушка nil-интерфейса

```go
func returnError() error {
    var s *someStruct = nil // значение nil, но тип *someStruct известен
    return s               // интерфейс получает информацию о типе!
}

func main() {
    err := returnError()
    if err != nil {
        // Выполнится! В интерфейсе сохранён тип *someStruct
        fmt.Println("Произошла ошибка, хотя данные nil!")
    }
}
```

**Исправление — явно возвращать `nil`:**

```go
func returnInterface() interface{} {
    return nil // чистый nil: нет ни типа, ни значения
}
```

[Источник: Данил Подольский, видео «Интерфейсы в Go»; NotebookLM — Интерфейсы]

## Подводные камни

1. **Ловушка nil-интерфейса** — переменная конкретного `nil`-типа, присвоенная интерфейсу, не равна `nil`. Особенно опасно с `if err != nil`. Попытка вызвать метод приведёт к панике. [Источник: NotebookLM — Интерфейсы]

2. **Panic при type assertion** — опасная форма `x.(T)` вызывает panic в рантайме, если тип не совпадает. Всегда используй форму с `ok`. [Источник: Михалис Цукалос, «Golang для профи»]

3. **Конвертация слайсов** — нельзя напрямую передать `[]MyStruct` туда, где ожидается `[]MyInterface`. Требуется цикл с копированием элементов по одному.
   > «Самый большой косяк системы типов Go» [Источник: Серия видеолекций skill issue]

4. **Пустой интерфейс в продакшене** — приведение типов через `interface{}` переносит проверку ошибок на рантайм. Источник частых сбоев в продакшене. [Источник: NotebookLM — Интерфейсы]

5. **«Прибитые гвоздями» типы** — если метод интерфейса принимает/возвращает структуру из другого пакета, интерфейс жёстко связывается с этим пакетом. [Источник: Серия видеолекций skill issue]

6. **Производительность** — type assertion и type switch используют рефлексию, замедляют программу, ошибки только в рантайме. [Источник: Данил Подольский, видео «Интерфейсы в Go»]

7. **Утечки памяти через Context** — если горутина создана с производным контекстом и завершается без вызова `cancel()`, ссылка сохраняется в родительском контексте и блокирует GC. [Источник: NotebookLM — Интерфейсы]

## Сравнения

### Go vs Java/C#

| Аспект | Go | Java / C# |
|---|---|---|
| Ключевое слово реализации | Нет (implicit) | `implements` / `: IInterface` |
| Место объявления интерфейса | Там, где используется | Там, где описывается тип |
| Иерархия наследования | Нет, только композиция | Классическое наследование |
| Переопределение методов | «Родительский» метод не знает об изменении | Истинный override через virtual/abstract |
| Проверка соответствия | Компилятор автоматически | Явное указание |

[Источник: Серия видеолекций skill issue]

### Trade-offs

| Подход | Плюсы | Минусы |
|---|---|---|
| Type assertion опасная форма | Лаконично | Паника в рантайме при несовпадении типа |
| Type assertion с `ok` | Безопасно | Чуть многословнее |
| Type switch | Удобен для множества типов | Рефлексия, замедляет, ошибки в рантайме |
| Пустой интерфейс `interface{}` | Принимает любой тип | Теряется type safety, ошибки в рантайме |
| Минимальный интерфейс | Низкая связанность, легко мокать | Требует дисциплины проектирования |
| Интерфейс в общем пакете | Один источник | Увеличивает связанность |
| Интерфейсы вместо генериков | Работало до Go 1.18 | Только ad-hoc полиморфизм |

[Источник: Серия видеолекций skill issue; Данил Подольский]

# Flashcards

#flashcards/go/interfaces

Что такое интерфейс в Go?::Программная синтаксическая структура (контракт/спецификация), описывающая набор методов, которые должен реализовать конкретный тип. Определяет поведение, не данные.
Difficulty: Easy

Как в Go тип реализует интерфейс?::Неявно (implicit): нет ключевого слова implements. Если тип реализует все методы интерфейса — компилятор автоматически считает его реализующим. Duck Typing.
Difficulty: Easy

Что такое Duck Typing в Go?::"Если крякает как утка — значит утка". Тип не объявляет соответствие интерфейсу явно — компилятор проверяет по наличию методов автоматически.
Difficulty: Easy

Из чего состоит обычная интерфейсная переменная (iface) внутри?::Два указателя: 1) itab — описание интерфейса + информация о типе; 2) указатель на конкретные данные. Размер: 16 байт на 64-бит.
Difficulty: Medium

Чем eface (пустой интерфейс) отличается от iface внутри?::eface не хранит описание интерфейса (itab). Хранит только: указатель на тип + указатель на данные. iface хранит itab + данные.
Difficulty: Hard

Когда интерфейсная переменная равна nil?::Только когда в ней нет ни информации о типе, ни значения. Если присвоить nil-указатель конкретного типа — интерфейс получит тип и перестанет быть nil.
Difficulty: Medium

В чём ловушка nil-интерфейса?
?
Присвоение `nil`-указателя конкретного типа интерфейсу → интерфейс получает информацию о типе → `if err != nil` возвращает true, хотя данных нет → panic при вызове метода. Исправление: явно возвращать `nil`.
Difficulty: Hard

Что такое type assertion? Синтаксис?::Операция для проверки и извлечения конкретного типа из интерфейсной переменной. Синтаксис: `x.(T)`. При несовпадении — panic.
Difficulty: Easy

Как избежать panic при type assertion?::Безопасная форма с двумя значениями: `v, ok := x.(T)`. При несовпадении ok=false, паники нет, v получает нулевое значение.
Difficulty: Easy

Что такое type switch? Синтаксис?::Форма switch для выбора ветки по динамическому типу интерфейса. Синтаксис: `switch v := x.(type) { case T: ... }`.
Difficulty: Easy

Что такое embedding интерфейсов?::Встраивание одного интерфейса в другой. Синтаксический сахар для объединения методов. Тип, удовлетворяющий составному интерфейсу, автоматически удовлетворяет всем встроенным.
Difficulty: Easy

Где должен объявляться интерфейс в Go?::В месте его использования, а не там, где описывается реализующий тип. Если нужен в 100 местах — лучше скопировать 100 раз: «маленькое копирование лучше маленькой связанности».
Difficulty: Medium

Можно ли передать `[]MyStruct` туда, где ожидается `[]MyInterface`?::Нет. Нельзя напрямую. Нужно написать цикл и скопировать элементы по одному. «Самый большой косяк системы типов Go».
Difficulty: Medium

Чем опасен `interface{}` в продакшене?::Приведение типов через пустой интерфейс переносит проверку ошибок на рантайм — источник частых сбоев. Называют «универсальной отмычкой».
Difficulty: Medium

Что такое itab в интерфейсе Go?::Структура данных в iface, хранящая описание интерфейса и информацию о конкретном типе. На неё указывает первый указатель обычного (не пустого) интерфейса.
Difficulty: Hard

Что такое ad-hoc полиморфизм и чем отличается от параметрического?::Ad-hoc — полиморфизм «по месту», который дают интерфейсы в Go. Параметрический — истинный полиморфизм через генерики (Go 1.18+).
Difficulty: Medium

Может ли интерфейс в Go содержать реализацию?::Нет. Только перечень методов (контракт). Поведение содержат абстрактные классы — концепция, отсутствующая в Go.
Difficulty: Easy

Что такое «прибитые гвоздями» типы в интерфейсе?::Если метод интерфейса принимает/возвращает тип из другого пакета — интерфейс жёстко связывается с этим пакетом. Решение: базовые типы или вынос в отдельный пакет.
Difficulty: Hard

Почему type assertion и type switch замедляют программу?::Используют рефлексию в рантайме. Ошибки обнаруживаются только во время выполнения, а не при компиляции.
Difficulty: Medium

Чем Go-интерфейсы отличаются от Java/C#?
?
1. Нет ключевого слова implements — реализация неявная. 2. Нет иерархии наследования — только композиция. 3. Интерфейс объявляется в месте использования. 4. Нет истинного override — «родительский» метод не знает об изменении.
Difficulty: Medium

## Learning Path

1. **Базовые концепции**
   - Что такое интерфейс и зачем нужен
   - Синтаксис объявления
   - Неявная реализация (Duck Typing)

2. **Внутреннее устройство**
   - iface: itab + данные
   - eface: тип + данные
   - Размер в памяти (16 байт)

3. **Работа с типами**
   - Type assertion: опасная и безопасная форма
   - Type switch
   - Interface embedding (композиция)

4. **Подводные камни**
   - Nil interface ловушка
   - Конвертация слайсов
   - «Прибитые гвоздями» типы

5. **Философия и best practices**
   - Место объявления
   - Минимализм методов
   - Decoupling как единственная цель
   - Интерфейсы vs генерики
