## Суть и контекст

**Goroutine** — минимальная единица выполнения в Go: легковесный поток управления, работающий в адресном пространстве процесса. Горутины управляются рантаймом языка, а не операционной системой.

**Планировщик Go** — компонент рантайма, распределяющий готовые к выполнению горутины по потокам операционной системы. В основе лежит модель **GMP**: Goroutine, Machine, Processor.

---

## Ключевые идеи

- Горутина весит ~2 КБ стека против 1–8 МБ у потока ОС — это на порядки легче.
- Планировщик Go использует модель **GMP**: G — горутина, M — поток ОС, P — логический процессор.
- Каждый P имеет свою локальную очередь (LRQ, до 256 горутин) без блокировок.
- При пустой LRQ P крадёт половину задач у соседнего P — **Work Stealing**.
- Блокирующий syscall: M блокируется, P передаётся другому потоку — **Handoff**.
- Сетевые операции обрабатывает **Netpoller**: горутина снимается с выполнения, не блокируя M.
- С Go 1.14 введено **вытеснение (preemption)**: горутина, работающая >10 мс, принудительно вытесняется.
- `GOMAXPROCS` = количество P = по умолчанию равно числу логических ядер CPU.
- Стек горутины динамически растёт: от 2 КБ до 1 ГБ (64-bit) / 250 МБ (32-bit).
- Утечка горутины — горутина заблокирована навсегда; обнаружить через `runtime.NumGoroutine()`.

## Определения и термины

**Goroutine (G)** — единица конкурентного выполнения в Go. Содержит состояние кода (стек, счётчик команд). Начальный стек — 2 КБ.

**Machine (M)** — обёртка над потоком операционной системы, непосредственно исполняющего код. Чтобы выполнять код Go, M обязан быть связан с P.

**Processor (P)** — логический процессор: ресурс, необходимый для выполнения кода. Содержит локальную очередь горутин (LRQ) и кэши. Количество P задаётся через `GOMAXPROCS`.

**LRQ (Local Run Queue)** — локальная очередь горутин у каждого P, вмещает до 256 горутин. Не требует блокировок — с ней работает только один M.

**GRQ (Global Run Queue)** — общая очередь горутин для всех P. Защищена мьютексом, доступ медленнее.

**Work Stealing** — механизм балансировки нагрузки: P с пустой LRQ крадёт половину задач из хвоста LRQ другого P.

**Handoff** — передача P от заблокированного M к другому (новому или из пула) при выполнении блокирующего системного вызова.

**Netpoller** — асинхронный механизм обработки сетевых операций. Горутина, ожидающая сети, снимается с потока и помещается в Netpoller, освобождая M для других задач.

**sysmon** — фоновый системный поток рантайма. Следит за горутинами и принудительно вытесняет те, что выполняются более 10 мс.

**Preemption (вытеснение)** — принудительное снятие горутины с выполнения по сигналу от sysmon. Введено в Go 1.14. До этого планировщик был кооперативным.

**GOMAXPROCS** — параметр рантайма, задающий количество P (логических процессоров). По умолчанию равен числу логических ядер CPU.

**runtime.Gosched()** — добровольная уступка CPU планировщику. Горутина не блокируется — она ставится обратно в очередь и возобновляется автоматически.

**Утечка горутины** — горутина, запущенная, но никогда не завершающаяся. Остаётся в памяти, потребляя ресурсы.

## Как устроено

### Модель GMP

```
┌────────────────────────────────────────┐
│              Go Runtime                │
│                                        │
│  P1 [LRQ: G4, G5]    P2 [LRQ: G6]    │
│   │                    │               │
│   M1 ─── G1 (run)     M2 ─── G3 (run)│
│                                        │
│         GRQ: [G7, G8, G9...]          │
│         Netpoller: [G2 (wait net)]    │
└────────────────────────────────────────┘
```

- Каждый M должен быть привязан к P для выполнения кода.
- У каждого P — своя LRQ и кэши.
- GRQ — общий пул, защищён мьютексом.

### Порядок поиска задач

Когда M освобождается, он ищет следующую горутину в порядке:

1. **LRQ** своего P
2. **GRQ** (глобальная очередь)
3. **Netpoller** (завершившиеся сетевые операции)
4. **Work Stealing** — крадёт половину задач из хвоста LRQ другого P

Планировщик принудительно проверяет GRQ каждый **61-й тик**, чтобы горутины в ней не простаивали вечно.

### Work Stealing

Если LRQ пуста, P не простаивает — он идёт к соседнему P и забирает **половину** горутин из хвоста его LRQ. Это обеспечивает равномерную загрузку всех ядер.

### Обработка блокирующих syscall (Handoff)

```
Горутина вызывает блокирующий syscall
         │
   M блокируется
         │
Планировщик: отсоединить P от M
         │
P передаётся другому M (из пула или новый)
         │
Остальные горутины продолжают работу
         │
По завершении syscall: заблокированный M
ищет свободный P или уходит в пул
```

### Обработка сетевых операций (Netpoller)

Горутина, ожидающая сетевого ответа, снимается с M и помещается в Netpoller (на базе `epoll`/`kqueue`). M остаётся свободным. Когда данные готовы, Netpoller возвращает горутину в очередь на выполнение.

### Вытеснение (Preemption)

До Go 1.14 планировщик был **кооперативным**: горутины сами решали, когда уступить место. С Go 1.14 введено вытеснение через `sysmon`:

- sysmon — отдельный поток, работающий в фоне
- Если горутина выполняется более **10 мс** — ей посылается сигнал
- Горутина принудительно переносится в конец GRQ

### Жизненный цикл горутины

```
Gdead/Gidle
    │  go func()
    ▼
Grunnable ──── в LRQ или GRQ
    │  M+P свободны
    ▼
Grunning ──── выполняется на M
    │  ┌──── блокировка (канал, mutex)
    │  └──► Gwaiting
    │  ┌──── syscall
    │  └──► Gsyscall
    │  функция завершилась
    ▼
Gdead ──── G готова к переиспользованию
```

### Стек горутины

| Параметр | Значение |
|---|---|
| Начальный размер | 2 КБ |
| Максимум (64-bit) | 1 ГБ |
| Максимум (32-bit) | 250 МБ |
| Поток ОС (для сравнения) | 1–8 МБ (фиксированный) |

При нехватке стека рантайм выделяет новый, больший участок памяти и переносит туда данные. Адреса переменных на стеке могут измениться.

## Детали и нюансы

### GOMAXPROCS=1

Делает приложение фактически **однопоточным** в контексте выполнения горутин. Увеличение GOMAXPROCS сверх числа ядер не даёт прироста производительности — реальный параллелизм ограничен аппаратно.

### Кооперативная vs вытесняющая многозадачность

| | Кооперативная (до Go 1.14) | Вытесняющая (с Go 1.14) |
|---|---|---|
| Кто решает переключение | Горутина сама | sysmon по таймеру |
| Риск | Одна горутина может захватить поток | Нет |
| Точки переключения | Вызовы функций, каналы | + По сигналу sysmon |

### Ловушка замыканий в цикле

Горутины внутри `for` захватывают переменную итерации **по ссылке**. К моменту их запуска цикл может завершиться, и все горутины увидят только **последнее** значение.

**Неправильно:**
```go
for i := 0; i < 5; i++ {
    go func() {
        fmt.Println(i) // все напечатают 5
    }()
}
```

**Правильно:**
```go
for i := 0; i < 5; i++ {
    i := i // локальная копия
    go func() {
        fmt.Println(i)
    }()
}
```

### Concurrent map writes

Одновременная запись в `map` из разных горутин без синхронизации вызывает **фатальную ошибку** `concurrent map writes`. Защита: `sync.Mutex` или `sync.RWMutex`.

### Утечки горутин

**Основные причины:**
- Горутина пытается отправить в канал, который никто не читает
- Горутина ждёт из пустого канала, в который никто не пишет
- Бесконечное ожидание сетевого соединения без таймаута

**Обнаружение:** `runtime.NumGoroutine()` — если число постоянно растёт при стабильной нагрузке, есть утечка.

## Сравнения и trade-offs

| | Поток ОС | Горутина |
|---|---|---|
| Стек | 1–8 МБ (фиксированный) | 2 КБ (динамический, до 1 ГБ) |
| Создание | Дорого (~мкс) | Дёшево (~нс) |
| Переключение контекста | Дорого (syscall) | Дёшево (в user space) |
| Управление | ОС | Рантайм Go |
| Количество | ~тысячи | ~миллионы |

| Очередь | Блокировка | Скорость доступа |
|---|---|---|
| LRQ (локальная) | Нет | Быстро |
| GRQ (глобальная) | Мьютекс | Медленнее |

## Примеры

### 1. Создание горутины

```go
go functionName()           // обычная функция
go func() { /* ... */ }()  // анонимная функция
```

### 2. Правильный WaitGroup + безопасное замыкание

```go
var wg sync.WaitGroup
for i := 0; i < 5; i++ {
    wg.Add(1)
    go func(n int) {
        defer wg.Done()
        fmt.Printf("%d ", n)
    }(i) // передаём i как аргумент
}
wg.Wait()
```

### 3. Защита map через Mutex

```go
var m sync.Mutex
var data = make(map[int]int)

func safeWrite(key, val int) {
    m.Lock()
    data[key] = val
    m.Unlock()
}
```

### 4. Таймаут через context

```go
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()

select {
case <-doWork():
    fmt.Println("Done")
case <-ctx.Done():
    fmt.Println("Timeout:", ctx.Err())
}
```

### 5. Добровольная уступка CPU

```go
for {
    doHeavyWork()
    runtime.Gosched() // уступить другим горутинам
}
```

### 6. Проверка количества горутин

```go
fmt.Println(runtime.NumGoroutine()) // текущее количество активных горутин
```

### 7. Настройка GOMAXPROCS

```go
runtime.GOMAXPROCS(4) // установить 4 логических процессора
```

## Важно / подводные камни / best practices

**Подводные камни:**

- **Ловушка замыканий в цикле.** Горутины захватывают переменную итерации по ссылке. Решение: `i := i` или передать как аргумент.
- **Concurrent map writes.** Параллельная запись в `map` без синхронизации = фатальная ошибка. Всегда защищать мьютексом.
- **`time.Sleep` для ожидания горутин.** Ненадёжно. Использовать `sync.WaitGroup`.
- **Горутина без таймаута.** Ожидание сетевого соединения или канала без `context` может зависнуть навсегда.
- **Увеличение GOMAXPROCS сверх ядер.** Не даёт прироста производительности.

**Best practices:**

- Используй `sync.WaitGroup`: `Add(1)` перед запуском, `defer Done()` внутри.
- Используй `context.WithTimeout` / `WithDeadline` для всех долгих операций.
- Защищай конкурентный доступ к `map`: `sync.Mutex` (равное чтение/запись) или `sync.RWMutex` (чтение чаще записи).
- Передавай переменные цикла в горутину аргументом, не через замыкание.
- Следи за утечками через `runtime.NumGoroutine()` в тестах и мониторинге.
- `runtime.Gosched()` применяй в плотных вычислительных циклах, чтобы дать поработать другим горутинам и GC.

Из каких трёх сущностей состоит модель GMP?

?
- **LRQ** (Local Run Queue) — локальная очередь P, до 256 горутин, без блокировок, быстрый доступ
- **GRQ** (Global Run Queue) — общая очередь, защищена мьютексом, доступ медленнее

В каком порядке M ищет следующую задачу?
?
1. LRQ своего P
2. GRQ (глобальная очередь)
3. Netpoller
4. Work Stealing у соседнего P

Что происходит при блокирующем syscall в модели GMP (Handoff)?::M блокируется вместе с горутиной. Планировщик отсоединяет P от заблокированного M и передаёт P другому M, чтобы остальные горутины продолжали работу.

Что такое GOMAXPROCS?::Параметр, задающий количество логических процессоров P. По умолчанию равен числу логических ядер CPU. Определяет степень реального параллелизма.

?

- Горутина ждёт из пустого канала, в который никто не пишет
- Бесконечное ожидание сетевого соединения без таймаута

Почему concurrent map writes — фатальная ошибка?::Go детектирует одновременную запись в map и завершает программу паникой `concurrent map writes`. Защита: `sync.Mutex` или `sync.RWMutex`.

Когда использовать sync.Mutex vs sync.RWMutex?::Mutex — когда чтение и запись примерно одинаковы. RWMutex — когда чтений значительно больше, чем записей: несколько горутин могут читать одновременно, запись блокирует всех.

Почему нельзя использовать `time.Sleep` для ожидания горутин?::Ненадёжно: не гарантирует, что горутина успеет завершиться. Правильно: `sync.WaitGroup`.

Как правильно использовать sync.WaitGroup?::Вызвать `wg.Add(1)` **до** запуска горутины. Внутри горутины — `defer wg.Done()`. В вызывающем коде — `wg.Wait()`.

- **Grunning** — выполняется на M
- **Gwaiting** — заблокирована (канал, mutex)
- **Gsyscall** — выполняет системный вызов
- **Gdead** — завершена, готова к переиспользованию

| Создание | Дорого | Дёшево |
| Переключение | Syscall | User space |
| Управление | ОС | Рантайм Go |
