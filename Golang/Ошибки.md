## Суть и контекст

В Go ошибки — это обычные значения, которые возвращаются из функций и методов наравне с другими результатами. В языке нет механизма исключений (try-catch) — вместо этого программа явно получает и обрабатывает ошибку как значение. Язык предоставляет инструменты для создания, оборачивания, разворачивания и проверки ошибок, а также механизмы `defer`, `panic`, `recover` для управления потоком при исключительных ситуациях.

---

## Ключевые идеи

- `error` — встроенный интерфейс; `nil` означает отсутствие ошибки
- Ошибки — это значения: с ними программируют логику, а не пишут try-catch
- Go 1.13: `%w` для оборачивания; `errors.Is` и `errors.As` проверяют всю цепочку (error chain)
- `defer` — планирует выполнение функции перед завершением текущей; порядок LIFO
- Аргументы `defer` вычисляются в момент объявления, а не при вызове
- `panic` — для неисправимых ситуаций; `recover` — перехват паники, только внутри `defer`
- Внешний API пакета всегда должен возвращать `error`, даже если внутри используется `panic`

---

## Определения и термины

**error** — встроенный интерфейс: `type error interface { Error() string }`. Любой тип, реализующий метод `Error() string`, удовлетворяет этому интерфейсу.

**nil error** — значение `nil` означает отсутствие ошибки.

**Sentinel error** — заранее известное, фиксированное значение ошибки (например, `io.EOF`, `sql.ErrNoRows`). Код сравнивает полученную ошибку с таким значением, чтобы определить конкретную ситуацию.

**Custom error type** — пользовательская структура, реализующая интерфейс `error`; позволяет хранить дополнительный контекст в полях.

**Wrapping (оборачивание)** — передача ошибки выше по стеку с добавлением контекста через `fmt.Errorf` с глаголом `%w`.

**Unwrap chain (цепочка ошибок)** — последовательность ошибок, получаемая путём многократного вызова `Unwrap()`.

**defer** — инструкция, планирующая выполнение функции непосредственно перед завершением текущей функции, независимо от способа завершения.

**panic** — встроенная функция, останавливающая нормальный ход выполнения программы и начинающая процесс «раскручивания стека» (stack unwinding).

**recover** — встроенная функция, позволяющая перехватить управление у паникующей горутины и остановить раскручивание стека. Полезна только внутри отложенных функций (`defer`).

## Как устроено

### Интерфейс error

```go
type error interface {
    Error() string
}
```

Значение `nil` означает отсутствие ошибки. Значение не-nil означает наличие проблемы.

### Создание ошибок: errors.New и fmt.Errorf

```go
err1 := errors.New("произошла ошибка")
err2 := fmt.Errorf("ошибка при обработке элемента %d", 42)
```

- `errors.New` — простое создание из строки
- `fmt.Errorf` — создание с форматированием (аналог `fmt.Sprintf`)

### Sentinel errors

```go
if err == sql.ErrNoRows {
    // обработка отсутствия записей
}
```

Заранее известные фиксированные значения ошибок. Прямое сравнение `==` работает только для самой верхней ошибки (без учёта цепочки).

### Custom error types

```go
type QueryError struct {
    Query string
    Err   error
}

func (e *QueryError) Error() string { return e.Query + ": " + e.Err.Error() }
```

Для извлечения данных — type assertion или type switch:

```go
if e, ok := err.(*os.PathError); ok {
    fmt.Println("Проблема с файлом:", e.Path)
}
```

Пример из стандартной библиотеки: `os.PathError` хранит имя файла, операцию и системную ошибку.

### Wrapping (%w) — Go 1.13

```go
if err != nil {
    return fmt.Errorf("ошибка доступа к БД: %w", err)
}
```

- `fmt.Errorf` с `%w` создаёт ошибку, реализующую метод `Unwrap() error`
- Если `e1.Unwrap()` возвращает `e2` — говорят, что `e1` оборачивает `e2`
- Последовательность таких вызовов — **error chain (цепочка ошибок)**

### errors.Is и errors.As

Вместо прямых сравнений `==` и type assertion — проверяют **всю цепочку ошибок**:

```go
// errors.Is — проверяет совпадение на любом уровне цепочки
if errors.Is(err, sql.ErrNoRows) {
    // ...
}

// errors.As — проверяет тип на любом уровне, присваивает значение
var e *QueryError
// Передача указателя на указатель — обязательна
if errors.As(err, &e) {
    fmt.Println(e.Query)
}
```

Типы могут кастомизировать поведение `errors.Is`, реализовав собственный метод `Is()`.

### defer

Планирует выполнение функции перед завершением текущей функции (в т.ч. при ошибке). Несколько `defer` выполняются в порядке **LIFO** («последним пришёл — первым ушёл»):

```go
for i := 0; i < 5; i++ {
    defer fmt.Println(i) // выведет: 4 3 2 1 0
}
```

**Аргументы вычисляются в момент объявления defer, а не при выполнении:**

```go
defer untrace(trace("a"))
// trace("a") выполнится НЕМЕДЛЕННО (вычисление аргумента)
// untrace будет вызвана при return
```

**defer + named return values:**

Defer-замыкание имеет доступ к именованным возвращаемым значениям и может изменять их до финального return:

```go
func doSomething() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("паника: %v", r)
        }
    }()
    // ...
}
```

Применение: закрытие файлов (`defer f.Close()`), освобождение мьютексов.

### panic

```go
panic("Что-то пошло не так!")
```

- Останавливает нормальное выполнение текущей функции
- Начинает раскручивание стека (stack unwinding)
- По пути выполняются все `defer` в паникующей горутине
- После этого программа аварийно завершается

**Runtime panic (неявная):**
- Выход за пределы массива или среза (index out of bounds)
- Ошибки приведения типов (type assertion)
- Обращение через nil pointer или nil interface

Применение явной `panic`: неисправимые ситуации, невозможные состояния, невозможность инициализации библиотеки.

### recover

```go
func safelyDo() {
    defer func() {
        if err := recover(); err != nil {
            log.Println("Перехвачена паника:", err)
        }
    }()
    panic("Что-то пошло не так!")
}
```

- Полезна только внутри отложенных функций (`defer`)
- При нормальном выполнении возвращает `nil`
- При панике — захватывает значение, переданное в `panic`, возобновляет нормальное выполнение

## Детали и нюансы

### Паттерн bufio.Scanner

Вместо проверки ошибки на каждой итерации чтения метод `Scan()` возвращает только `bool`. Ошибка сохраняется внутри объекта. Клиентский код проверяет ошибку один раз в конце через `Err()`:

```
// Scan() → true/false
// при сбое сканер останавливается и сохраняет ошибку
// проверяем один раз: scanner.Err()
```

### Паттерн errWriter

Структура-обёртка, хранящая оригинальный `Writer` и поле для первой ошибки:

- Метод записи: если ошибка уже сохранена — **no-op** (ничего не делает)
- Если нет — выполняет запись, сохраняет возможную ошибку
- Вызывающий код проверяет ошибку **один раз в конце**

Используется в `bufio.Writer`, `archive/zip`.

> В предоставленных источниках не раскрыто: `errors.Join` (Go 1.20).

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| `errors.New` / `fmt.Errorf` без `%w` | Просто; не раскрывает детали реализации | Вызывающий не может проверить тип ошибки через `errors.Is`/`errors.As` | Когда детали реализации не должны утекать в публичный API |
| `fmt.Errorf` с `%w` | Цепочка сохраняется; `errors.Is`/`errors.As` работают | Раскрывает детали реализации как часть публичного API | Когда нужно передать контекст и дать вызывающему возможность проверить вложенную ошибку |
| Custom error type | Богатый контекст в полях; можно извлекать данные | Более многословно | Когда нужно передать структурированные данные об ошибке |
| Sentinel error | Лёгкое сравнение | Прямое `==` не работает в цепочке (нужен `errors.Is`) | Заранее известные, фиксированные ситуации (EOF, ErrNotFound) |
| `panic` / `recover` | Прерывает выполнение при критических ошибках | Не для обычной обработки ошибок; сложнее контролировать | Неисправимые ситуации внутри пакета; изолируется через `recover` на границе API |

## Примеры

```go
// Интерфейс error
type error interface {
    Error() string
}

// Создание ошибок
err1 := errors.New("произошла ошибка")
err2 := fmt.Errorf("ошибка при обработке элемента %d", 42)

// Custom error type
type QueryError struct {
    Query string
    Err   error
}
func (e *QueryError) Error() string { return e.Query + ": " + e.Err.Error() }

// os.PathError — пример из стандартной библиотеки
if e, ok := err.(*os.PathError); ok {
    fmt.Println("Проблема с файлом:", e.Path)
}

// Wrapping
if err != nil {
    return fmt.Errorf("ошибка доступа к БД: %w", err)
}

// errors.Is
if errors.Is(err, sql.ErrNoRows) {
    // ...
}

// errors.As — передача указателя на указатель обязательна
var e *QueryError
if errors.As(err, &e) {
    fmt.Println(e.Query)
}

// defer + recover
func safelyDo() {
    defer func() {
        if err := recover(); err != nil {
            log.Println("Перехвачена паника:", err)
        }
    }()
    panic("Что-то пошло не так!")
}

// defer + named return values
func doSomething() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("паника: %v", r)
        }
    }()
    // ...
}
```

## Важно / подводные камни / best practices

1. **Ошибки — это значения.** Не воспринимай их как исключения (try-catch); программируй логику работы с ними как с любыми другими значениями.

2. **Всегда проверяй ошибки.** Игнорировать возвращаемую ошибку через `_` — ужасная практика.

3. **Контекст в сообщениях.** Строки ошибок должны указывать на источник происхождения: префикс с именем операции или пакета (например, `"image: unknown format"`).

4. **Когда оборачивать (`%w`):** когда нужно раскрыть детали вызывающему коду для проверки через `errors.Is`/`errors.As`.

5. **Когда НЕ оборачивать:** если это нарушает абстракцию — например, если скрытая реализация использует SQL, возврат обёрнутой `sql.ErrNoRows` делает эту деталь частью публичного API. В таких случаях лучше возвращать собственный тип ошибки.

6. **Изоляция panic:** если пакет использует `panic` для внутренней обработки исключительных ситуаций, его внешний API всё равно должен возвращать обычный `error`.

7. **Абстракция повторяющихся проверок:** используй паттерны errWriter / bufio.Scanner вместо `if err != nil` после каждого вызова.

8. **`recover` работает только внутри `defer`.** Вызов `recover()` вне `defer` всегда возвращает `nil`.

9. **Аргументы `defer` вычисляются в момент объявления** — не при вызове. Это ключевой источник ошибок при трассировке и отладке.
