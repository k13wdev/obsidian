# Паттерны конкурентного программирования в Go

## Суть и контекст

Конкурентное программирование в Go строится на базовых примитивах — горутинах, каналах и конструкции `select`. Из этих примитивов формируются устойчивые и масштабируемые **паттерны**, каждый из которых решает конкретную задачу: организация потоков данных, управление жизненным циклом горутин, ограничение ресурсов, обработка ошибок и автоматическое восстановление. Паттерны применяются при построении конвейеров обработки данных, сетевых серверов, микросервисов и любых систем, требующих параллельного выполнения.

---

## Ключевые идеи

- Конвейер (Pipeline) позволяет обрабатывать потоки данных по частям, снижая потребление памяти и позволяя стадиям работать параллельно.
- Fan-Out / Fan-In — распараллеливание тяжёлых стадий конвейера и слияние результатов обратно в один поток.
- Worker Pool ограничивает количество одновременно работающих горутин, защищая систему от исчерпания ресурсов.
- Семафор реализуется через буферизованный канал для ограничения конкурентного доступа к ресурсу.
- Done Channel — главное правило Go: тот, кто создает горутину, обязан обеспечить способ её завершения.
- Context инкапсулирует done channel, дополняя его таймаутами, дедлайнами и передачей данных.
- Горутины не подвергаются сборке мусора (not garbage collected) — они обязаны завершиться самостоятельно.
- Добавление очередей (Queuing) не ускоряет общее время выполнения конвейера (закон Литтла), а лишь изменяет его поведение.
- Паттерн Steward + Heartbeat обеспечивает автоматическое восстановление неисправных горутин.
- Replicated Requests — трата дополнительных ресурсов ради максимальной скорости ответа.

## Определения и термины

**Pipeline (Конвейер)** — серия стадий, соединённых каналами, где каждая стадия представляет собой группу горутин, выполняющих одну и ту же функцию. Первая стадия называется **генератором** (producer), последняя — **потребителем** (sink).

**Generator (Генератор)** — функция, конвертирующая набор дискретных значений в поток значений в канале. Запись данных происходит в отдельной горутине, что исключает блокировку основного потока выполнения.

**Fan-Out (Разветвление)** — запуск нескольких горутин для чтения из одного входящего канала, пока этот канал не будет закрыт. Позволяет распределить тяжёлую вычислительную работу (CPU-intensive) или задачи ввода-вывода между группой воркеров.

**Fan-In (Слияние)** — мультиплексирование нескольких входных каналов в один выходной канал, который закрывается, когда закрыты все входные каналы.

**Worker Pool (Пул обработчиков)** — шаблон, ограничивающий количество одновременно работающих горутин для обработки задач из общей очереди или слайса. Предотвращает неконтролируемое создание десятков тысяч горутин и истощение ресурсов.

**Semaphore (Семафор)** — механизм ограничения количества одновременных доступов к ресурсу или параллельных горутин, реализуемый в Go через буферизованный канал.

**Done Channel** — канал только для чтения (`<-chan interface{}` или `<-chan struct{}`), который передаётся родительской горутиной дочерним для сигнала об отмене. Закрытие канала `done` работает как широковещательный сигнал.

**Or-Channel** — паттерн объединения одного или нескольких каналов `done` в единый канал `done`, который закроется, если закроется любой из его компонентов.

**Or-Done-Channel** — обёртка, инкапсулирующая операцию чтения из канала вместе с чтением из сигнального канала `done`, возвращающая новый канал. Избавляет от необходимости писать громоздкие for-select циклы в потребителе.

**Tee Channel** — паттерн, разделяющий значения из одного канала на два (или более) независимых канала (аналог команды `tee` в UNIX).

**Bridge Channel (Связующий канал)** — функция, деструктурирующая канал каналов (`<-chan <-chan interface{}`) в плоский канал (`<-chan interface{}`). Позволяет потребителю считывать значения из множества последовательно пересоздаваемых каналов в одном цикле.

**Confinement (Изоляция)** — метод обеспечения потокобезопасности за счёт лексической области видимости, ограничивающий доступ к данным или правам на канал (только чтение / только запись).

**Heartbeat (Сердцебиение)** — сигнал, испускаемый горутиной сторонним наблюдателям, подтверждающий, что она жива и функционирует. Существует в двух вариантах: интервальные (interval-based) тики и перед каждой единицей работы (per-unit-of-work).

**Steward (Стюард)** — горутина, логика которой контролирует состояние здоровья другой горутины (ward) и перезапускает её в случае сбоя. Опирается на паттерн Heartbeat.

**Ward (Опекаемый)** — долгоживущая горутина, которая мониторится стюардом. Передаёт сигналы heartbeat стюарду; если сигнал отсутствует дольше таймаута, ward отменяется и перезапускается.

**Replicated Requests (Реплицированные запросы)** — отправка одного и того же запроса нескольким дублирующим обработчикам и использование результата от того, который ответит первым.

**Rate Limiting (Ограничение скорости)** — ограничение частоты событий, например количества запросов в секунду (RPS).

**MultiLimiter** — комбинированный рейт-лимитер, инкапсулирующий несколько объектов `RateLimiter` и сортирующий их от самого строгого к наименее строгому. Блокирует запрос до тех пор, пока не будут получены токены от всех дочерних лимитеров.

**errgroup** — механизм, оборачивающий `sync.WaitGroup`, общий канал ошибок и контекст отмены для группы горутин. Позволяет ограничить количество конкурентных горутин (`SetLimit`) и автоматически отменяет контекст для всех горутин, если хотя бы одна возвращает ошибку.

**Утечка горутины (Goroutine Leak)** — ситуация, когда результат работы горутины никому не нужен, но горутина продолжает работать (или заблокирована навечно) и потреблять ресурсы памяти и процессора. Горутины не очищаются сборщиком мусора — они обязаны завершиться самостоятельно.

**Queuing (Теория очередей)** — внедрение буферизации (очередей) в стадии конвейера. По закону Литтла (Little's Law) добавление очередей не ускоряет общее время выполнения конвейера, а лишь изменяет его поведение.

## Как устроено

### 1. Pipeline (Конвейер)

Стадия получает значения из входящих (inbound) каналов, выполняет над ними преобразования и отправляет результаты в исходящие (outbound) каналы. Каждая стадия — это функция, принимающая `done` канал и входной канал, возвращающая выходной канал. Стадии можно комбинировать цепочкой.

```go
// Генератор переводит дискретные значения в канал
generator := func(done <-chan interface{}, integers ...int) <-chan int {
    intStream := make(chan int)
    go func() {
        defer close(intStream)
        for _, i := range integers {
            select {
            case <-done:
                return
            case intStream <- i:
            }
        }
    }()
    return intStream
}

// Стадия конвейера (умножение)
multiply := func(done <-chan interface{}, intStream <-chan int, multiplier int) <-chan int {
    multipliedStream := make(chan int)
    go func() {
        defer close(multipliedStream)
        for i := range intStream {
            select {
            case <-done:
                return
            case multipliedStream <- i * multiplier:
            }
        }
    }()
    return multipliedStream
}
```

### 2. Fan-Out (Разветвление)

Если стадия конвейера выполняется долго (например, `primeFinder`), можно запустить несколько её копий для параллельной обработки:

```go
numFinders := runtime.NumCPU()
finders := make([]<-chan int, numFinders)
for i := 0; i < numFinders; i++ {
    finders[i] = primeFinder(done, randIntStream) // Fan-out
}
```

### 3. Fan-In (Слияние)

Для каждого входного канала создаётся горутина, которая перекладывает данные в общий `multiplexedStream`. Завершение координируется через `sync.WaitGroup`.

```go
fanIn := func(done <-chan interface{}, channels ...<-chan interface{}) <-chan interface{} {
    var wg sync.WaitGroup
    multiplexedStream := make(chan interface{})

    multiplex := func(c <-chan interface{}) {
        defer wg.Done()
        for i := range c {
            select {
            case <-done:
                return
            case multiplexedStream <- i:
            }
        }
    }

    wg.Add(len(channels))
    for _, c := range channels {
        go multiplex(c)
    }

    go func() {
        wg.Wait()
        close(multiplexedStream)
    }()
    return multiplexedStream
}
```

**Антипаттерн Fan-In:** В классическом виде (создание отдельной горутины-посредника на каждый канал) Fan-In может быть избыточным и перегружать планировщик. На практике часто эффективнее использовать подход Worker Pool, где все воркеры напрямую пишут в один выходной канал, синхронизируясь мьютексом самого канала.

### 4. Worker Pool (Пул обработчиков)

#### Классический Worker Pool

Создаётся фиксированное число воркеров (горутин). Задачи перекладываются в канал, из которого воркеры конкурентно их вычитывают:

```go
func makeWP(nWorkers int) {
    var w sync.WaitGroup
    for i := 0; i < nWorkers; i++ {
        w.Add(1)
        go worker(&w)
    }
    w.Wait()
    close(data) // data — выходной канал
}

func worker(w *sync.WaitGroup) {
    defer w.Done()
    for c := range clients { // clients — входной канал с задачами
        square := c.integer * c.integer
        output := Data{c, square}
        data <- output
    }
}
```

#### Инвертированный Worker Pool (хранение воркеров в буфере)

Буферизованный канал хранит пул воркеров. Метод `Handle` извлекает свободного воркера (блокируясь, если все заняты), запускает работу и возвращает воркера обратно:

```go
type Pool struct {
    workers chan *Worker
    handler func(workerID int, data interface{})
}

// 1. CREATE
func (p *Pool) Create(numWorkers int) {
    p.workers = make(chan *Worker, numWorkers)
    for i := 0; i < numWorkers; i++ {
        p.workers <- &Worker{ID: i}
    }
}

// 2. HANDLE (PROCESS)
func (p *Pool) Handle(data interface{}) {
    w := <-p.workers // Берём свободного воркера (или блокируемся)
    go func() {
        p.handler(w.ID, data)
        p.workers <- w // Возвращаем воркера в пул
    }()
}

// 3. WAIT
func (p *Pool) Wait(numWorkers int) {
    for i := 0; i < numWorkers; i++ {
        <-p.workers // Ожидаем возврата всех воркеров
    }
}
```

### 5. Semaphore (Семафор)

В Go семафор реализуется через буферизованный канал. Прежде чем начать работу, горутина должна положить токен (пустую структуру) в канал. Если канал полон — горутина блокируется или отбрасывает задачу (через `select` и `default`). После выполнения токен вычитывается, освобождая место:

```go
semaphore := make(chan struct{}, 10) // Максимум 10 одновременных горутин

// Заполнение семафора перед запуском логики
select {
case semaphore <- struct{}{}:
    go func() {
        defer func() { <-semaphore }() // Освобождаем место по завершении
        // Выполнение задачи
    }()
default:
    // Если семафор полон, прерываем выполнение (non-blocking)
    return
}
```

#### Semaphore с неблокирующим пропуском задач (Skip if full)

Использование `select` + `default` позволяет отбрасывать некритичные фоновые задачи (например, фоновую запись в кэш), чтобы не замедлять основной handler:

```go
func withSemaphore(fn func()) {
    select {
    case semaphore <- struct{}{}: // Захват токена
        go func() {
            defer func() { <-semaphore }()
            fn()
        }()
    default:
        // Если семафор полон (10 из 10), просто выходим без выполнения
        return
    }
}
```

### 6. Done Channel (Предотвращение утечек горутин)

Закрытие канала `done` работает как широковещательный сигнал. Дочерние горутины используют `select`, чтобы проверять состояние `done`. Чтение из закрытого канала происходит мгновенно, что позволяет горутине выйти и освободить ресурсы.

```go
doWork := func(done <-chan interface{}, strings <-chan string) <-chan interface{} {
    terminated := make(chan interface{})
    go func() {
        defer fmt.Println("doWork exited.")
        defer close(terminated)
        for {
            select {
            case s := <-strings:
                fmt.Println(s)
            case <-done:
                return // Выход из горутины при сигнале отмены
            }
        }
    }()
    return terminated
}

done := make(chan interface{})
terminated := doWork(done, nil)
go func() {
    time.Sleep(1 * time.Second)
    close(done) // Родитель сигнализирует дочерней горутине о завершении
}()
<-terminated
fmt.Println("Done.")
```

### 7. Or-Channel

Рекурсивно объединяет произвольное количество `done`-каналов в один. Удобен на пересечении модулей, когда количество условий отмены заранее неизвестно:

```go
var or func(channels ...<-chan interface{}) <-chan interface{}
or = func(channels ...<-chan interface{}) <-chan interface{} {
    if len(channels) == 0 { return nil }
    if len(channels) == 1 { return channels[0] }

    orDone := make(chan interface{})
    go func() {
        defer close(orDone)
        switch len(channels) {
        case 2:
            select {
            case <-channels[0]:
            case <-channels[1]:
            }
        default:
            select {
            case <-channels[0]:
            case <-channels[1]:
            case <-channels[2]:
            case <-or(append(channels[3:], orDone)...): // Рекурсия
            }
        }
    }()
    return orDone
}
```

### 8. Or-Done-Channel

Обёртка для безопасного чтения из канала с учётом `done`. Избавляет от громоздких for-select циклов в потребителе и предотвращает утечки горутин, когда неизвестно, будет ли исходный канал когда-либо закрыт при отмене `done`:

```go
orDone := func(done, c <-chan interface{}) <-chan interface{} {
    valStream := make(chan interface{})
    go func() {
        defer close(valStream)
        for {
            select {
            case <-done:
                return
            case v, ok := <-c:
                if ok == false {
                    return
                }
                select {
                case valStream <- v:
                case <-done:
                }
            }
        }
    }()
    return valStream
}
```

### 9. Tee Channel

Разделяет значения из одного канала на несколько независимых выходных каналов.

**Антипаттерн классической реализации:** Использование затенения переменных и `nil`-каналов в `select` критикуется как сложный и трудночитаемый антипаттерн. Современная и более масштабируемая практика — создавать под каждый выходной канал отдельную горутину, синхронизируя их через `sync.WaitGroup`:

```go
func tee(in <-chan interface{}, numChannels int) []<-chan interface{} {
    channels := make([]chan interface{}, numChannels)
    for i := range channels {
        channels[i] = make(chan interface{})
    }

    go func() {
        for val := range in {
            var wg sync.WaitGroup
            wg.Add(numChannels)
            for _, ch := range channels {
                go func(c chan interface{}) {
                    defer wg.Done()
                    c <- val
                }(ch)
            }
            wg.Wait()
        }
        for _, ch := range channels {
            close(ch)
        }
    }()

    // Преобразование []chan interface{} в []<-chan interface{} для возврата
    result := make([]<-chan interface{}, numChannels)
    for i, ch := range channels {
        result[i] = ch
    }
    return result
}
```

### 10. Bridge Channel (Связующий канал)

Деструктурирует канал каналов в единый плоский канал. Потребитель видит непрерывный поток данных, а `bridge` внутри переключается между поступающими каналами, исчерпывая их по очереди:

```go
bridge := func(
    done <-chan interface{},
    chanStream <-chan <-chan interface{},
) <-chan interface{} {
    valStream := make(chan interface{})
    go func() {
        defer close(valStream)
        for {
            var stream <-chan interface{}
            select {
            case maybeStream, ok := <-chanStream:
                if ok == false {
                    return
                }
                stream = maybeStream
            case <-done:
                return
            }
            for val := range orDone(done, stream) {
                select {
                case valStream <- val:
                case <-done:
                }
            }
        }
    }()
    return valStream
}

// Пример использования:
genVals := func() <-chan <-chan interface{} {
    chanStream := make(chan (<-chan interface{}))
    go func() {
        defer close(chanStream)
        for i := 0; i < 10; i++ {
            stream := make(chan interface{}, 1)
            stream <- i
            close(stream)
            chanStream <- stream
        }
    }()
    return chanStream
}

for v := range bridge(nil, genVals()) {
    fmt.Printf("%v ", v)
}
```

### 11. Confinement (Изоляция)

Вместо использования мьютексов, создатель канала инициализирует канал внутри функции, запускает пишущую горутину и возвращает вызывающему коду канал только для чтения (`<-chan`). Это исключает риск того, что потребитель закроет канал или запишет в него данные. Компилятор на этапе сборки запрещает несанкционированную запись:

```go
chanOwner := func() <-chan int {
    results := make(chan int, 5)
    go func() {
        defer close(results)
        for i := 0; i <= 5; i++ {
            results <- i
        }
    }()
    return results // Неявно приводится к <-chan int
}
```

### 12. Heartbeat (Сердцебиение)

#### Интервальный Heartbeat

Горутина регулярно отправляет тики через `time.Tick`. Наблюдатель использует `select` с таймаутом — если не получает пульс в срок, признаёт горутину неисправной:

```go
doWork := func(
    done <-chan interface{},
    pulseInterval time.Duration,
) (<-chan interface{}, <-chan time.Time) {
    heartbeat := make(chan interface{})
    results := make(chan time.Time)
    go func() {
        defer close(heartbeat)
        defer close(results)

        pulse := time.Tick(pulseInterval)
        workGen := time.Tick(2 * pulseInterval)

        sendPulse := func() {
            select {
            case heartbeat <- struct{}{}:
            default: // Важно: не блокируемся, если пульс никто не читает
            }
        }
        sendResult := func(r time.Time) {
            for {
                select {
                case <-done:
                    return
                case <-pulse:
                    sendPulse()
                case results <- r:
                    return
                }
            }
        }

        for {
            select {
            case <-done:
                return
            case <-pulse:
                sendPulse()
            case r := <-workGen:
                sendResult(r)
            }
        }
    }()
    return heartbeat, results
}
```

#### Per-Unit-of-Work Heartbeat

Канал `heartbeat` создаётся с буфером 1. Перед обработкой или отправкой реальных данных горутина пытается записать пустую структуру в `heartbeat` через неблокирующий `select` с `default`. Удобен для **детерминированного тестирования** — позволяет безопасно писать юнит-тесты без необходимости устанавливать фиксированные (`time.Sleep` / timeout) задержки, которые делают тесты хрупкими:

```go
doWork := func(done <-chan interface{}) (<-chan interface{}, <-chan int) {
    heartbeatStream := make(chan interface{}, 1)
    workStream := make(chan int)
    go func() {
        defer close(heartbeatStream)
        defer close(workStream)

        for i := 0; i < 10; i++ {
            select {
            case heartbeatStream <- struct{}{}:
            default:
            }

            select {
            case <-done:
                return
            case workStream <- rand.Intn(10):
            }
        }
    }()

    return heartbeatStream, workStream
}
```

### 13. Steward (Автоисцеление горутин)

Стюард контролирует горутину-ward через heartbeat. Если пульс не приходит дольше таймаута — ward убивается (через закрытие `wardDone`) и перезапускается:

```go
newSteward := func(timeout time.Duration, startGoroutine startGoroutineFn) startGoroutineFn {
    return func(done <-chan interface{}, pulseInterval time.Duration) <-chan interface{} {
        heartbeat := make(chan interface{})
        go func() {
            defer close(heartbeat)
            var wardDone chan interface{}
            var wardHeartbeat <-chan interface{}
            startWard := func() {
                wardDone = make(chan interface{})
                wardHeartbeat = startGoroutine(or(wardDone, done), timeout/2)
            }
            startWard()
            pulse := time.Tick(pulseInterval)

        monitorLoop:
            for {
                timeoutSignal := time.After(timeout)
                for {
                    select {
                    case <-pulse:
                        select {
                        case heartbeat <- struct{}{}:
                        default:
                        }
                    case <-wardHeartbeat:
                        continue monitorLoop // Ward жива
                    case <-timeoutSignal:
                        log.Println("steward: ward unhealthy; restarting")
                        close(wardDone) // Убийство зависшего ward
                        startWard()     // Перезапуск
                        continue monitorLoop
                    case <-done:
                        return
                    }
                }
            }
        }()
        return heartbeat
    }
}
```

### 14. Replicated Requests (Реплицированные запросы)

Запускается несколько идентичных горутин. Первая, вернувшая результат, считается победителем. После этого `done` закрывается, чтобы отменить работу остальных:

```go
doWork := func(
    done <-chan interface{},
    id int,
    wg *sync.WaitGroup,
    result chan<- int,
) {
    started := time.Now()
    defer wg.Done()

    // Simulate random load
    simulatedLoadTime := time.Duration(1+rand.Intn(5)) * time.Second
    select {
    case <-done:
    case <-time.After(simulatedLoadTime):
    }

    select {
    case <-done:
    case result <- id:
    }

    took := time.Since(started)
    if took < simulatedLoadTime {
        took = simulatedLoadTime
    }
    fmt.Printf("%v took %v\n", id, took)
}

done := make(chan interface{})
result := make(chan int)

var wg sync.WaitGroup
wg.Add(10)

for i := 0; i < 10; i++ {
    go doWork(done, i, &wg, result)
}

firstReturned := <-result // Блокируемся до первого ответа
close(done)               // Отменяем остальные 9 запросов
wg.Wait()

fmt.Printf("Received an answer from #%v\n", firstReturned)
```

### 15. Error Handling в конкурентном коде

Ошибки должны рассматриваться как объекты первого класса. Так как горутина не может вернуть ошибку через `return` напрямую, создаётся структура `Result`. Горутина передаёт этот объект в канал, а вызывающая сторона решает, что делать с ошибкой:

```go
type Result struct {
    Error    error
    Response *http.Response
}

// Внутри горутины:
resp, err := http.Get(url)
result := Result{Error: err, Response: resp}
select {
case <-done:
    return
case results <- result:
}
```

#### errgroup

Оборачивает `sync.WaitGroup`, общий канал ошибок и контекст отмены. Автоматически каскадно отменяет контекст для всех горутин, если хотя бы одна возвращает ошибку:

```go
g.SetLimit(100)

g.Go(func() error {
    if err != nil {
        return err
    }
    return nil
})

if err := g.Wait(); err != nil {
    return err
}
```

`SetLimit(100)` ограничивает пул до 100 одновременно выполняемых горутин. Если переданная функция возвращает `error != nil`, внутренний механизм отменяет общий контекст `ctx`.

### 16. Rate Limiting (Ограничение скорости)

Используется стандартный пакет `golang.org/x/time/rate`, реализующий алгоритм **«маркерной корзины» (token bucket)**, где параметр `r` — скорость пополнения маркеров (limit), а `b` — размер корзины (burst). Либо используется `time.Ticker` для ручного ограничения RPS.

#### Хелпер Per

```go
func Per(eventCount int, duration time.Duration) rate.Limit {
    return rate.Every(duration / time.Duration(eventCount))
}
```

#### MultiLimiter

Лимитеры сортируются от самого ограничивающего к самому свободному. Метод `Wait` последовательно вызывает `Wait(ctx)` у каждого дочернего лимитера:

```go
type RateLimiter interface {
    Wait(context.Context) error
    Limit() rate.Limit
}

func MultiLimiter(limiters ...RateLimiter) *multiLimiter {
    byLimit := func(i, j int) bool {
        return limiters[i].Limit() < limiters[j].Limit()
    }
    sort.Slice(limiters, byLimit)
    return &multiLimiter{limiters: limiters}
}

type multiLimiter struct {
    limiters []RateLimiter
}

func (l *multiLimiter) Wait(ctx context.Context) error {
    for _, l := range l.limiters {
        if err := l.Wait(ctx); err != nil {
            return err
        }
    }
    return nil
}

func (l *multiLimiter) Limit() rate.Limit {
    return l.limiters[0].Limit()
}
```

#### Применение MultiLimiter к API

```go
func Open() *APIConnection {
    return &APIConnection{
        apiLimit: MultiLimiter(
            rate.NewLimiter(Per(2, time.Second), 2),
            rate.NewLimiter(Per(10, time.Minute), 10),
        ),
        diskLimit: MultiLimiter(
            rate.NewLimiter(rate.Limit(1), 1),
        ),
        networkLimit: MultiLimiter(
            rate.NewLimiter(Per(3, time.Second), 3),
        ),
    }
}

type APIConnection struct {
    networkLimit,
    diskLimit,
    apiLimit RateLimiter
}

func (a *APIConnection) ReadFile(ctx context.Context) error {
    err := MultiLimiter(a.apiLimit, a.diskLimit).Wait(ctx)
    if err != nil {
        return err
    }
    // Полезная работа
    return nil
}

func (a *APIConnection) ResolveAddress(ctx context.Context) error {
    err := MultiLimiter(a.apiLimit, a.networkLimit).Wait(ctx)
    if err != nil {
        return err
    }
    // Полезная работа
    return nil
}
```

### 17. Context Cancellation (Отмена через контекст)

Интерфейс `context.Context` с методами `Deadline()`, `Done()`, `Err()` и `Value()` обеспечивает API для отмены ветвей графа вызовов и контейнер для передачи данных:

- `context.Background()` — пустой корневой контекст.
- `context.WithCancel(parent)` — дочерний контекст и функция `cancel`. Закрывает канал `Done()` при вызове `cancel()`.
- `context.WithTimeout(parent, duration)` — автоматически вызывает `cancel()` по истечении `duration`.
- `context.WithDeadline(parent, time)` — срабатывает в конкретную точку времени (timestamp).
- `context.WithValue(parent, key, val)` — сохраняет пару «ключ-значение», поиск рекурсивно поднимается вверх по дереву контекстов.

#### Пример WithTimeout

```go
func genGreeting(ctx context.Context) (string, error) {
    ctx, cancel := context.WithTimeout(ctx, 1*time.Second)
    defer cancel()

    switch locale, err := locale(ctx); {
    case err != nil:
        return "", err
    case locale == "EN/US":
        return "hello", nil
    }
    return "", fmt.Errorf("unsupported locale")
}

func locale(ctx context.Context) (string, error) {
    select {
    case <-ctx.Done():
        return "", ctx.Err()
    case <-time.After(1 * time.Minute):
    }
    return "EN/US", nil
}
```

`WithTimeout` устанавливает лимит в 1 секунду. Функция `locale` долго спит (1 * `time.Minute`). В результате `ctx.Done()` в `locale` сработает из-за таймаута предка, вернув `context.DeadlineExceeded`.

#### Пример WithValue

```go
type ctxKey int
const (
    ctxUserID ctxKey = iota
    ctxAuthToken
)

func ProcessRequest(userID, authToken string) {
    ctx := context.WithValue(context.Background(), ctxUserID, userID)
    ctx = context.WithValue(ctx, ctxAuthToken, authToken)
    HandleResponse(ctx)
}

func HandleResponse(ctx context.Context) {
    fmt.Printf("handling response for %v (%v)",
        ctx.Value(ctxUserID),
        ctx.Value(ctxAuthToken),
    )
}
```

Использование неэкспортируемого пользовательского типа `ctxKey int` предотвращает коллизии ключей между разными пакетами.

### 18. Select Patterns (Паттерны select)

**Случайный выбор (Pseudo-random selection):** если готовы несколько каналов, `select` выбирает один из них с равномерным случайным распределением, чтобы предотвратить «голодание» одних каналов за счёт других. Разрешено писать одинаковые кейсы для регулировки вероятности.

**Неблокирующий выбор (Default case):** ветка `default` выполняется мгновенно, если ни один из каналов не готов. Часто применяется с бесконечным циклом `for` для фоновой работы (for-select loop).

**For-select loop:**

```go
for {
    select {
    case <-done:
        return
    default:
    }
    // Выполнение невытесняемой (non-preemptable) работы
}
```

**Таймаут в select:** использование `time.After(duration)` или `time.NewTimer().C` как одного из кейсов позволяет разблокировать `select` по истечении времени.

**Чтение из nil-канала:** блокируется навсегда. Этот механизм используется для «отключения» ветки `case` прямо во время работы — присвоив переменной канала значение `nil`, когда данные из него больше не нужны.

**Пустой `select {}`:** блокирует горутину навсегда (иногда используется для предотвращения выхода из `main`).

### 19. Queuing (Теория очередей)

Очереди внедряются только на входе в конвейер или в стадиях, где группировка в пакеты (batching/chunking) повышает эффективность системы (например, запись на диск). Не используются для обхода медленных вычислительных (CPU-bound) стадий — это не уменьшит общее время прохождения всех элементов.

## Детали и нюансы

- **Утечки горутин:** горутины не очищаются GC. Стек каждой горутины занимает от 2 КБ. Если горутина заблокирована навечно (забыли `done`-канал, никто не читает/пишет в канал) — это ведёт к постепенному исчерпанию памяти (memory creep). Для обнаружения утечек существует библиотека `goleak`.
- **Data Race при записи в map:** `fatal error: concurrent map writes` — мгновенная паника. Мапы в Go внутренне не потокобезопасны из-за структуры бакетов и механизма эвакуации данных. Обнаруживается с флагом `-race`.
- **Deadlock:** циклическое ожидание мьютексов — горутина А берёт блокировку 1 и ждёт блокировку 2, горутина Б берёт блокировку 2 и ждёт блокировку 1. Результат: `fatal error: all goroutines are asleep - deadlock!`.
- **Блокировка на nil-канале:** чтение или запись в nil-канал блокируется навсегда. Если забыли инициализировать канал через `make` — результатом будет deadlock.
- **Забытый `<-semaphore`:** если не вернуть токен в буферизованный канал (забыть `defer func() { <-semaphore }()`), новые задачи заблокируются на записи `semaphore <- struct{}{}` — логический deadlock обработки.
- **Неудачная запись Heartbeat:** горутина, генерирующая heartbeat, не должна блокироваться, если наблюдатель в данный момент не вычитывает сигнал. Решение — `select` с веткой `default`.
- **Коллизии ключей Context:** считывание значения контекста с некорректным типом или коллизия ключей при использовании `context.WithValue` могут привести к перезаписи данных другого пакета или панике при приведении типов. Ключи `context` должны быть сопоставимыми (comparable), значения должны быть потокобезопасными.
- **MultiLimiter Wait:** функция `Wait` поочерёдно вызывает `Wait` дочерних лимитеров. Гарантируется ожидание ровно в течение времени работы самого строгого лимитера.
- **Очереди скрывают дедлоки:** внедрение буферизации может скрыть логические ошибки в синхронизации (дедлоки и лайвлоки) и усложнить чтение кода, не увеличивая общую пропускную способность конвейера.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Fan-In (классический) | Простая абстракция, каждый канал — отдельная горутина | Избыточен, перегружает планировщик горутинами-посредниками | Небольшое число каналов, разнородные источники |
| Worker Pool (все пишут в один канал) | Эффективнее, мьютекс канала синхронизирует сам | Менее гибко при разнородных источниках | CPU/IO-bound задачи, однородные воркеры |
| Done Channel (ручной) | Полный контроль, минимальный overhead | Бойлерплейт, ручное управление | Простые сценарии без дерева вызовов |
| Context | Каскадная отмена, таймауты, передача данных, стандарт Go | Больше абстракций | Сетевые запросы, деревья вызовов, middleware |
| errgroup | Минимум бойлерплейта, автоотмена, лимит горутин | Дополнительная зависимость | Группы параллельных задач с общей обработкой ошибок |
| sync.WaitGroup + ручная отмена | Полный контроль | Много ручного кода, легко допустить ошибку | Простейшие случаи без ошибок |
| Heartbeat (интервальный) | Подходит для долгих операций | Недетерминированность в тестах | Production мониторинг |
| Heartbeat (per-unit-of-work) | Детерминированное тестирование | Не подходит для долгих пауз между задачами | Юнит-тесты конкурентного кода |
| Queuing (буферизация) | Сглаживание всплесков, batching | Не ускоряет CPU-bound стадии, скрывает дедлоки | Вход в конвейер, запись на диск |
| Replicated Requests | Минимальная латентность ответа | Избыточное потребление ресурсов | Критичная скорость ответа, доступные реплики |

### Worker Pool vs Semaphore — критерии выбора

| Критерий | Worker Pool | Semaphore |
|----------|-------------|-----------|
| Жизненный цикл горутин | Создаются один раз, живут до закрытия канала задач | Создаются динамически под каждую задачу, уничтожаются после выполнения |
| Потребление памяти в простое | Постоянное: N воркеров × 2 КБ стека, даже если задач нет | Нулевое: горутины существуют только при наличии работы |
| Overhead на создание горутин | Минимальный: горутины переиспользуются | Есть: каждая задача = аллокация стека + уничтожение GC |
| CPU-intensive задачи | **Предпочтителен:** строго ограничивает число ядер (например, 5 из 10), не парализуя систему | Подходит, но без переиспользования горутин |
| Синхронные syscalls (файловый I/O) | **Предпочтителен:** каждый syscall блокирует OS-тред; 100 000 горутин с I/O → 100 000 OS-тредов → OOM / Thread Exhaustion | Опасен: динамическое создание горутин с syscalls может истощить OS-треды |
| Сетевой I/O (async, netpoller) | Подходит | **Предпочтителен:** netpoller не плодит OS-треды; горутины дёшевы; On-Demand экономит память при спайках |
| Спорадическая нагрузка (0 → N) | Неэффективен: 100 000 спящих воркеров = ~200 МБ впустую | **Предпочтителен:** горутина «умирает» после выполнения, освобождая стек |
| Непрерывный поток задач | **Предпочтителен:** экономия на аллокации, горутины переиспользуются | Дополнительный overhead на создание/уничтожение горутин |
| Non-blocking отбрасывание | Требует дополнительной логики | Встроено: `select` + `default` позволяет отбросить задачу, если семафор полон |

**Trade-off по CPU vs памяти:** Worker Pool экономит CPU (горутины созданы заранее), но резервирует память на весь цикл работы программы. Semaphore экономит память (горутины On-Demand), но тратит CPU на постоянное создание и уничтожение горутин.

## Важно / подводные камни / best practices

**Владение каналом:** владелец канала (функция, создающая его) должен: инстанцировать канал (`make`), выполнять запись, закрывать канал (`close`), возвращать канал только на чтение (`<-chan`). Это исключает паники от записи в закрытый или nil-канал и предотвращает закрытие канала более одного раза.

**Правила Context:**
- Context должен быть передан первым параметром функции.
- Для значений ключей `WithValue` создавайте пользовательские неэкспортируемые типы (например, `type ctxKey int`).
- Вызовы `WithCancel`, `WithTimeout` всегда должны сопровождаться `defer cancel()`.
- **Не хранить Context внутри структуры** — контекст привязан к жизненному циклу конкретного графа вызовов (request-scoped), а структура может жить дольше запроса.

**Запуск горутин:** если горутина создаёт дочернюю горутину, она обязана иметь способ её остановить. Запуск долгоиграющих горутин (`go func() {...}`) без передачи `done`-канала или контекста ведёт к утечкам.

**errgroup:** переносите логику управления `sync.WaitGroup`, каналами ошибок и контекстами внутрь `errgroup`. Код, запускаемый параллельно, оборачивайте в функцию, возвращающую `error`.

**Queuing:** не добавляйте буферизованные каналы везде, где система работает медленно — это скрывает дедлоки и лайвлоки, не увеличивая пропускную способность.

**Per-unit-of-work Heartbeat в тестах:** для недетерминированных функций, старт которых зависит от загрузки CPU/диска/сети, используйте heartbeat с буфером 1 перед каждой итерацией — это позволяет писать юнит-тесты без хрупких `time.Sleep`.

Generator (Генератор):::Функция, конвертирующая набор дискретных значений в поток значений в канале. Запись данных происходит в отдельной горутине

Fan-Out (Разветвление):::Запуск нескольких горутин для чтения из одного входящего канала. Распределяет тяжёлую вычислительную работу между группой воркеров

Fan-In (Слияние):::Мультиплексирование нескольких входных каналов в один выходной канал, который закрывается, когда закрыты все входные каналы

Worker Pool (Пул обработчиков):::Шаблон, ограничивающий количество одновременно работающих горутин для обработки задач из общей очереди. Защищает от исчерпания памяти

Semaphore в Go:::Буферизованный канал, ограничивающий количество одновременных доступов к ресурсу. Горутина кладёт токен (struct{}) перед работой и вычитывает после

Done Channel:::Канал только для чтения (<-chan interface{} или <-chan struct{}), передаваемый родительской горутиной дочерним. Закрытие работает как широковещательный сигнал отмены

Or-Channel:::Паттерн рекурсивного объединения нескольких done-каналов в один, который закроется при закрытии любого компонента

Or-Done-Channel:::Обёртка для безопасного чтения из канала с учётом done-канала. Избавляет от громоздких for-select циклов в потребителе

Tee Channel:::Паттерн разделения значений из одного канала на два (или более) независимых канала (аналог tee в UNIX)

Bridge Channel:::Функция, деструктурирующая канал каналов (<-chan <-chan interface{}) в плоский канал (<-chan interface{})

Confinement (Изоляция):::Метод потокобезопасности через лексическую область видимости — ограничение доступа к каналу (только чтение / только запись) вместо мьютексов

Heartbeat (Сердцебиение):::Сигнал, испускаемый горутиной наблюдателю для подтверждения работоспособности. Бывает интервальным и per-unit-of-work

Steward (Стюард):::Горутина-наблюдатель, контролирующая здоровье ward-горутины через heartbeat и перезапускающая её при сбое

Ward (Опекаемый):::Долгоживущая горутина, мониторимая стюардом. Передаёт heartbeat-сигналы; при отсутствии — отменяется и перезапускается

Replicated Requests:::Отправка одного запроса нескольким обработчикам и использование самого быстрого ответа. Трата ресурсов ради минимальной латентности

Rate Limiting:::Ограничение частоты событий (RPS). В Go реализуется через golang.org/x/time/rate (алгоритм token bucket) или time.Ticker

MultiLimiter:::Комбинированный рейт-лимитер, сортирующий дочерние лимитеры от строгого к свободному и блокирующий до получения токенов от всех

errgroup:::Обёртка над sync.WaitGroup с контекстом отмены. Автоматически отменяет контекст всех горутин, если одна вернула ошибку. Поддерживает SetLimit

Утечка горутины (Goroutine Leak):::Горутина продолжает работать/блокироваться, хотя её результат никому не нужен. Не очищается GC, потребляет от 2 КБ памяти

Queuing (Теория очередей):::Буферизация в стадиях конвейера. По закону Литтла не ускоряет общее время, а лишь изменяет поведение системы

Когда использовать Worker Pool, а не Semaphore?
?
1. CPU-intensive задачи — строго ограничивает число ядер
2. Синхронные syscalls (файловый I/O) — предотвращает создание тысяч OS-тредов
3. Непрерывный поток задач — экономия на аллокации горутин

Когда использовать Semaphore, а не Worker Pool?
?
1. Сетевой I/O (netpoller не плодит OS-треды)
2. Спорадическая нагрузка (0 → N) — горутины создаются On-Demand, не висят в памяти
3. Когда нужен non-blocking отброс задач (select + default)

В чём ключевой trade-off Worker Pool vs Semaphore?::Worker Pool экономит CPU (горутины переиспользуются), но резервирует память на весь цикл работы. Semaphore экономит память (горутины On-Demand), но тратит CPU на создание/уничтожение горутин.

Почему Worker Pool опасен при спорадической нагрузке от 0 до 100 000?::100 000 спящих горутин × 2 КБ = ~200 МБ впустую в простое. Семафор создаёт горутины только при наличии задач.

Почему Semaphore опасен при синхронных syscalls?::Каждый синхронный syscall блокирует OS-тред. Динамическое создание 100 000 горутин с I/O → 100 000 OS-тредов → OOM / Thread Exhaustion.

Что делает Semaphore с `select + default`?::Если семафор полон, задача отбрасывается без блокировки. Применяется для некритичных фоновых задач (запись в кэш), чтобы не замедлять основной handler.

Что происходит при конкурентной записи в map без sync.Mutex?::fatal error: concurrent map writes — мгновенная паника. Обнаруживается флагом -race

Что происходит при чтении/записи в nil-канал?::Блокировка навсегда. Если забыли make — результатом будет deadlock

Как реализуется семафор в Go?::Через буферизованный канал: `semaphore := make(chan struct{}, N)`. Запись токена — захват, чтение — освобождение

Как предотвратить утечку горутины?::Передать done-канал или context. Правило: создатель горутины обязан обеспечить способ её завершения

Почему нельзя хранить Context в структуре?::Context привязан к жизненному циклу запроса (request-scoped), а структура может жить дольше запроса

Зачем нужен default в select при записи heartbeat?::Чтобы горутина не блокировалась, если наблюдатель в данный момент не вычитывает сигнал

Когда применять Queuing (буферизацию) в конвейере?
?
1. На входе в конвейер для сглаживания всплесков
2. В стадиях, где batching/chunking повышает эффективность (запись на диск)
3. НЕ для обхода медленных CPU-bound стадий

Как работает Steward pattern?
?
1. Стюард запускает ward-горутину
2. Устанавливает таймер (timeoutSignal)
3. Ожидает heartbeat от ward
4. Если таймаут — закрывает wardDone (убивает ward)
5. Вызывает startWard() для перезапуска

Правила владения каналом (Channel Ownership)
?
1. Владелец инстанцирует канал (make)
2. Владелец выполняет запись
3. Владелец закрывает канал (close)
4. Возвращает канал только на чтение (<-chan)

Правила использования Context
?
1. Передавать первым параметром функции
2. Для ключей WithValue — неэкспортируемые пользовательские типы
3. WithCancel/WithTimeout — всегда defer cancel()
4. Не хранить в структуре

В чём антипаттерн классического Fan-In?::Создание отдельной горутины-посредника на каждый канал перегружает планировщик. Эффективнее Worker Pool, где воркеры пишут напрямую в один канал

В чём антипаттерн классического Tee Channel?::Использование затенения переменных и nil-каналов в select — сложно и трудночитаемо. Лучше отдельная горутина на каждый выходной канал с sync.WaitGroup

Зачем нужен per-unit-of-work heartbeat в тестах?::Позволяет писать детерминированные юнит-тесты без хрупких time.Sleep/timeout задержек

Как select выбирает при нескольких готовых каналах?::Pseudo-random selection — равномерное случайное распределение для предотвращения «голодания» каналов

Что делает пустой `select {}`?::Блокирует горутину навсегда. Иногда используется для предотвращения выхода из main

Как «отключить» ветку case в select во время работы?::Присвоить переменной канала значение nil — чтение из nil-канала блокируется навсегда

Алгоритм token bucket в rate limiting имеет параметр r — ==скорость пополнения маркеров (limit)==, а b — ==размер корзины (burst)==

Горутины потребляют от ==2 КБ== памяти на стек и ==не подвергаются сборке мусора==
