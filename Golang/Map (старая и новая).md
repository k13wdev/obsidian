## Суть и контекст

**Map** в Go — встроенная структура данных, представляющая неупорядоченную коллекцию пар «ключ — значение», реализованная на основе хэш-таблицы. Операции вставки, поиска и удаления выполняются за константное время **O(1)**.

В Go 1.24 классическая реализация на основе бакетов была заменена на **Swiss Tables** — новый дизайн с открытой адресацией, метаданными и SIMD-параллелизмом.

---

## Ключевые идеи

- Map = хэш-таблица; операции insert/get/delete — O(1).
- Ключи должны поддерживать оператор `==` (быть comparable).
- Порядок итерации **не гарантирован** и случаен — намеренно.
- Конкурентная запись из нескольких горутин без синхронизации → **паника**.
- Нельзя брать адрес элемента map `&m["key"]` — данные перемещаются при росте.
- Чтение из nil-map безопасно; запись → паника.
- **Classic (до 1.24):** бакеты по 8 элементов, load factor 6.5, инкрементальная эвакуация.
- **Swiss Tables (1.24+):** группы по 8 слотов, SIMD-поиск, независимые таблицы ≤ 1024 записей.

---

## Определения и термины

**hmap** — заголовок (header) map: основная структура с метаданными. Содержит: количество элементов, количество бакетов (как степень двойки), хэш-зерно (seed), указатель на массив бакетов.

**bmap (bucket)** — структура одного бакета в классической реализации. Вмещает до 8 пар ключ-значение.

**tophash** — массив из 8 байтов внутри bmap, хранящий старшие биты хэша каждого ключа. Используется для быстрой предварительной проверки перед дорогим сравнением ключей.

**Overflow bucket** — дополнительный бакет, создаваемый при переполнении основного (>8 элементов). Связывается через указатель, образуя связный список.

**Load factor** — коэффициент заполненности: среднее количество элементов на бакет. В классической реализации порог — **6.5** (в одном из источников упоминается значение **6.2**); превышение запускает расширение. В Swiss Tables порог — **87.5% (7/8)** — до 7 занятых слотов из 8 в группе.

**Evacuation (эвакуация)** — процесс переноса данных из старых бакетов в новые при росте map. В классической реализации — инкрементальный.

**oldbuckets** — поле hmap, хранящее ссылку на старый массив бакетов во время эвакуации.

**Swiss Tables** — новая реализация map в Go 1.24+, основанная на открытой адресации, контрольных словах (metadata) и SIMD-инструкциях.

**Control word** — 64-битное слово метаданных группы в Swiss Tables. Каждый байт соответствует слоту и хранит либо 7 нижних бит хэша ключа (h2), либо статус (пустой, tombstone).

**H1** — верхние 57 бит 64-битного хэша ключа. Используется для выбора таблицы и конкретной группы слотов.

**H2** — нижние 7 бит 64-битного хэша ключа. Сохраняется в control word группы и используется для быстрой проверки через SIMD.

**Tombstone** — специальная метка в слоте Swiss Tables при удалении элемента. Не оставляет слот пустым, чтобы не разорвать цепочку пробирования при поиске.

**SIMD (Single Instruction, Multiple Data)** — процессорные инструкции, позволяющие сравнивать хэш сразу со всеми 8 байтами control word за одну операцию.

**Comma ok idiom** — идиома `val, ok := m[key]`: безопасное чтение с проверкой наличия ключа.

## Как устроено

### Классическая Map (до Go 1.24)

#### Структура hmap

```
hmap
├── count      — общее кол-во элементов (len(m))
├── B          — log₂ кол-ва бакетов (2^B = кол-во бакетов)
├── hash0      — случайный seed для хэш-функции (защита от DDoS)
├── noverflow  — приблизительный счётчик overflow buckets
├── buckets    → [bmap₀][bmap₁]...[bmap_{2^B - 1}]
├── oldbuckets → старый массив бакетов во время эвакуации
└── nevacuate  — индекс следующего бакета для эвакуации
```

**Поле B:** хранит не само количество бакетов, а логарифм. Например, B=2 → 4 бакета (2²). Это экономит память и ускоряет побитовые операции для вычисления индекса бакета.

**Поле hash0 и защита от DDoS:** случайный seed генерируется при запуске приложения. Без него злоумышленник мог бы подобрать ключи, все хэши которых попадают в один бакет — map деградировала бы до O(n), что приводит к исчерпанию CPU (Hash-DoS атака).

**Поле nevacuate:** все бакеты с индексом меньше nevacuate считаются уже эвакуированными. Инкрементальная эвакуация продвигает этот счётчик вперёд.

#### Структура bmap (bucket)

```
bmap
├── tophash[8]     — 8 байтов: старший байт хэша каждого ключа (или статус слота)
├── keys[8]        — 8 ключей подряд (не перемежаются со значениями)
├── values[8]      — 8 значений подряд
└── overflow *bmap — указатель на следующий бакет переполнения
```

Данные хранятся не парами `(key, val)`, а блоками: сначала все 8 ключей, затем все 8 значений. Это устраняет лишний padding при выравнивании типов разного размера.

**Коды состояний tophash** *(точные константы не раскрыты в источниках; ниже — описание из исходного кода Go, на которое ссылаются видеоматериалы блокнота):*

| Константа | Значение | Смысл |
|---|---|---|
| `emptyRest` | 0 | Слот пуст, и все последующие слоты в бакете тоже пусты → поиск можно немедленно прервать |
| `emptyOne` | 1 | Слот пуст (ключ удалён), но дальше могут быть заполненные слоты |
| `evacuatedX` | 2 | Ключ эвакуирован в первую половину (X) нового массива |
| `evacuatedY` | 3 | Ключ эвакуирован во вторую половину (Y) нового массива |
| `evacuatedEmpty` | 4 | Слот был пуст до эвакуации |
| `minTopHash` | 5 | Минимальное значение для реального хэша; если хэш < 5, к нему прибавляется minTopHash, чтобы не пересекаться со статусными флагами |

#### Алгоритм Lookup (Поиск)

1. Вычислить 64-битный хэш ключа с использованием hash0.
2. Взять **младшие B бит** → индекс бакета в массиве.
3. Взять **старший байт** хэша → tophash для быстрой проверки.
4. Если идёт эвакуация: проверить nevacuate. Если бакет ещё не эвакуирован → читать из `oldbuckets`.
5. Перебрать tophash[8] в бакете (сравнение одного байта — очень быстро).
   - `emptyRest` → ключа нет, прервать поиск немедленно.
   - Несовпадение → пропустить слот.
   - Совпадение tophash → выполнить полное сравнение ключа в памяти.
6. Если совпал → вернуть значение.
7. Если нет → перейти по `overflow *bmap` к следующему бакету в цепочке.
8. Если цепочка закончилась → вернуть zero value.

#### Алгоритм Insert (Вставка)

1. Вычислить хэш, найти бакет (как в Lookup).
2. Если ключ уже существует → обновить значение.
3. Если ключа нет — найти первый свободный слот (tophash = `emptyOne` или `emptyRest`).
4. Если все 8 слотов заняты → создать новый overflow bucket, связать через `overflow *bmap`, записать туда. Инкрементировать `noverflow`.
5. Записать tophash, ключ, значение в слот.
6. Увеличить `count`.

#### Алгоритм Delete (Удаление)

1. Найти слот через алгоритм Lookup.
2. Очистить ключ и значение. Если значения — указатели, обнулить их (nil) → GC может освободить память.
3. Пометить tophash слота:
   - Если это последний заполненный слот в бакете → `emptyRest` (ускоряет будущие поиски).
   - Иначе → `emptyOne` (дальше могут быть занятые слоты).
4. Уменьшить `count`.

#### Два триггера роста и два типа эвакуации

Рост запускается при **двух независимых условиях:**

**Триггер 1: Load factor > 6.5**
- Среднее количество элементов на бакет превысило 6.5.
- Тип роста: **double-size rehash** — новый массив в 2 раза больше (B увеличивается на 1).
- Цель: снизить плотность, «размазать» элементы по большему числу бакетов.

**Триггер 2: Слишком много overflow buckets**
- `noverflow` превышает порог (зависит от текущего B).
- Тип роста: **same-size rehash** — новый массив того же размера.
- Цель: «уплотнить» данные. Map пережила много удалений/вставок — элементов мало, но они размазаны по длинным цепочкам overflow. Эвакуация перепакует их в основные бакеты без цепочек.

#### Инкрементальная эвакуация (шаги)

1. Выделить новый массив бакетов (×2 или same-size).
2. Сохранить старый массив в `oldbuckets`. Установить `nevacuate = 0`.
3. При **каждой** операции записи или удаления — перенести **1–2 бакета:**
   - Взять бакет с индексом `nevacuate` из `oldbuckets`.
   - Перераспределить его элементы в новые бакеты (при double-size элемент может попасть в бакет X или X + 2^(B-1) — зависит от одного дополнительного бита хэша).
   - Пометить tophash слотов как `evacuatedX` / `evacuatedY` / `evacuatedEmpty`.
   - Увеличить `nevacuate`.
4. Чтение во время эвакуации:
   - Проверить: `nevacuate > индекс бакета`? → бакет эвакуирован → читать из новых бакетов.
   - Иначе → читать из `oldbuckets`.
5. После переноса всех бакетов → `oldbuckets = nil`, GC освобождает память.

#### Итерация (`for range`)

За рандомизацию отвечает функция `mapiterinit` из runtime. Она вызывает `fastrand` и вычисляет:
- **startBucket** — случайный индекс бакета, с которого начнётся обход.
- **offset** — случайное смещение (0–7) внутри каждого бакета, с которого начинается чтение слотов.

Функция `mapiternext` обходит все бакеты по кругу, начиная со startBucket, пропуская пустые слоты и выдавая валидные пары. При незавершённой эвакуации итератор запоминает ссылку на `oldbuckets` и корректно обходит обе версии.

### Swiss Tables (Go 1.24+)

Новая реализация использует открытую адресацию и метаданные для параллельного поиска.

**Архитектура:**
```
Map
├── Table₁ (≤1024 записей)
│   ├── Group₀: [slot₀..slot₇] + control_word (64 bit)
│   ├── Group₁: [slot₀..slot₇] + control_word
│   └── ...
├── Table₂ (≤1024 записей)
└── ...
```

**Control word (64 бита = 8 байтов):**

| Байт | Значение |
|---|---|
| `0b0xxxxxxx` | 7 бит хэша ключа (h2) — слот занят |
| `0b10000000` | Слот пуст |
| `0b11111110` | Tombstone — слот удалён |

**Алгоритм Lookup (Поиск) в Swiss Tables:**
1. Вычислить 64-битный хэш ключа. Разделить на **H1** (верхние 57 бит) и **H2** (нижние 7 бит).
2. По H1 найти стартовую группу.
3. SIMD-инструкцией сравнить H2 **одновременно** со всеми 8 байтами control word → битовая маска совпадений.
4. Для каждого совпавшего слота — выполнить полное сравнение ключа в памяти.
5. Если совпадений нет, но есть **пустой слот** → поиск немедленно прекращается (ключа в маp нет).
6. Если группа заполнена (или содержит tombstone) → перейти к следующей группе по треугольному пробированию.

**Алгоритм Insert (Вставка) в Swiss Tables:**
1. Выполнить Lookup — убедиться, что ключа нет. Если есть → обновить значение.
2. Найти первый свободный слот: пустой или tombstone.
3. Если группа полностью занята → треугольное пробирование к следующей группе.
4. Записать ключ и значение в найденный слот.
5. Обновить соответствующий байт control word значением H2.

**Алгоритм Delete (Удаление) в Swiss Tables:**
1. Выполнить Lookup — найти слот с ключом.
2. Очистить содержимое слота (ключ и значение). Если значения — указатели, обнулить их (nil) для GC.
3. Если в группе есть **пустые слоты** → пометить слот как пустой.
4. Если группа была **полностью заполнена** → пометить слот как **tombstone**. Это сохраняет цепочку пробирования для других ключей, смещённых из-за коллизий.

**Треугольное пробирование:**
При открытой адресации если нужная группа занята, следующая ищется по формуле: `i+1, i+1+2, i+1+2+3, ...`. Это эффективно решает проблему кластеризации и гарантирует обход всех слотов таблицы за O(1) при размере, равном степени двойки.

**Независимые таблицы:**
- Каждая таблица хранит ≤ 1024 групп.
- Load factor таблицы: **87.5% (7/8)** — при заполнении 7 из 8 слотов в среднем таблица делится.
- При переполнении — только эта таблица делится на две, не затрагивая остальные.
- Максимальная задержка эвакуации ограничена константой (копируется ≤ 1024 элементов).
- Если `hint ≤ 8` при `make` → создаётся упрощённая однотабличная map без структуры директорий.

## Детали и нюансы

### Почему порядок итерации не гарантирован

Два намеренных механизма:

1. **Случайный seed (`hash0`)** — генерируется при запуске программы. Разные запуски дают разное распределение ключей по бакетам.
2. **Случайное начало итерации** — итератор специально начинает обход со случайного бакета и случайного смещения внутри него.

### Модификация map во время `range`

Go разрешает изменять map во время `range`:
- Элемент **удалён до** того, как итератор до него дошёл → **не будет выведен**.
- Элемент **добавлен во время** итерации → **может быть как выведен, так и пропущен** (неопределённо).

### Нельзя брать адрес элемента

`&m["key"]` — **ошибка компиляции**. Даже если бы это было возможно: при эвакуации (classic) или делении таблиц (Swiss Tables) данные перемещаются в памяти, и указатель стал бы невалидным.

### nil map

| Операция | nil map | Результат |
|---|---|---|
| Чтение `val := m[key]` | nil | Безопасно, вернёт zero value |
| `val, ok := m[key]` | nil | Безопасно, `ok = false` |
| `m[key] = val` | nil | **Паника** |
| `delete(m, key)` | nil | Безопасно |
| `len(m)` | nil | 0 |

## Сравнения и trade-offs

| | Classic Map (до 1.24) | Swiss Tables (1.24+) |
|---|---|---|
| Адресация | Цепочки (linked list) | Открытая адресация |
| Поиск в группе | Последовательный | SIMD-параллельный |
| Единица группировки | Бакет (8 элементов) | Группа слотов (8) |
| Эвакуация | Инкрементальная | Деление таблиц |
| Предел роста | Один массив бакетов | Независимые таблицы ≤ 1024 |
| Удаление | Помечает слот пустым | Tombstone |

| Параметр | Значение |
|---|---|
| Load factor (classic) | 6.5 (в одном источнике — 6.2) |
| Load factor (Swiss Tables) | 87.5% (7/8) |
| Элементов в бакете/группе | 8 |
| Макс. групп в таблице (Swiss) | 1024 |
| Рост при эвакуации (classic) | ×2 бакетов |
| Макс. элементов при сплите (Swiss) | ≤ 1024 (константа) |
| Ускорение в микробенчмарках (Swiss vs Classic) | до 60% |
| Снижение CPU в реальных приложениях (Swiss vs Classic) | ~1.5% |

## Примеры

### 1. Создание с пре-аллокацией

```go
m := make(map[string]int, 100) // hint = 100
```

### 2. Comma ok idiom — безопасное чтение

```go
if val, ok := m["apple"]; ok {
    fmt.Println("Value:", val)
} else {
    fmt.Println("Key not found")
}
```

### 3. Удаление и полная очистка

```go
delete(m, "apple")  // безопасно даже если ключа нет
clear(m)            // полная очистка (Go 1.21+)
```

### 4. Защита мьютексом при конкурентной работе

```go
var mu sync.Mutex
var m = make(map[string]int)

func safeWrite(key string, val int) {
    mu.Lock()
    m[key] = val
    mu.Unlock()
}

func safeRead(key string) (int, bool) {
    mu.Lock()
    val, ok := m[key]
    mu.Unlock()
    return val, ok
}
```

### 5. Итерация

```go
for key, val := range m {
    fmt.Println(key, val) // порядок случаен при каждом запуске
}
```

## Связи с другими темами

- [[Сборщик мусора (GC)]] — большие map с указателями в значениях замедляют GC: он сканирует их полностью при каждом цикле.
- [[Nil values]] — nil-map безопасна для чтения, но не для записи.
- [[Планировщик и Goroutine]] — конкурентная запись в map без синхронизации вызывает панику `concurrent map writes`.
- [[Массивы и Срезы]] — в отличие от среза, map не имеет упорядоченного доступа по индексу.
- [[Стек и куча]] — map аллоцируется в куче; внутренние структуры перемещаются при эвакуации.

## Важно / подводные камни / best practices

**Подводные камни:**

- **Concurrent map writes.** Параллельная запись из горутин без синхронизации → паника с сообщением `fatal error: concurrent map writes`. Использовать `sync.Mutex`, `sync.RWMutex` или `sync.Map`.
- **Нельзя брать адрес элемента.** `&m["key"]` — ошибка компиляции. Даже если бы компилятор позволил: при эвакуации/делении таблиц данные перемещаются.
- **nil map при записи.** Чтение из nil-map безопасно. Запись → паника.
- **Случайный порядок итерации.** Не полагайся на порядок `range` по map. Используй отсортированный срез ключей, если нужен детерминированный обход.
- **Большие map с указателями.** GC сканирует всю map при каждом цикле — значительный overhead при миллионах элементов.

**Best practices:**

- Указывай `hint` при создании через `make(map[K]V, size)`, если примерный размер известен — предотвращает лишние эвакуации.
- Используй comma ok idiom `val, ok := m[key]` для безопасного чтения.
- Для конкурентного чтения чаще записи используй `sync.RWMutex`: несколько горутин могут читать одновременно.
- Если ключи читаются из множества горутин, а запись редка, используй `sync.Map` из стандартной библиотеки — он спроектирован для безопасной работы с ключами из разных горутин.
- Для полной очистки map используй `clear(m)` (Go 1.21+) вместо пересоздания.

Каково внутреннее устройство классической map в Go (до 1.24)?
?
- **hmap** — заголовок с метаданными (count, B, seed, указатель на бакеты)
- **bmap** — бакет на 8 пар ключ-значение: tophash[8] + keys[8] + values[8] + overflow pointer
- Бакеты хранятся в массиве, размер кратен степени двойки
<!--SR:!2026-03-12,1,210-->

Как устроен bmap: почему ключи и значения хранятся раздельными блоками?::Сначала все 8 ключей, затем все 8 значений — чтобы устранить лишний padding при выравнивании типов данных разного размера.
<!--SR:!2026-03-12,1,210-->

Что такое tophash?::Массив из 8 байтов в bmap, хранящий старшие биты хэша каждого ключа. Используется для быстрой предварительной проверки перед полным сравнением ключей.
<!--SR:!2026-03-12,1,210-->

Как происходит поиск ключа в классической map?
?
1. Вычислить 64-битный хэш ключа
2. Младшие биты → индекс бакета
3. Старшие биты (tophash) → сравнить с tophash[8]
4. При совпадении → полное сравнение ключа
5. Нет → overflow bucket
<!--SR:!2026-03-12,1,210-->

Что такое load factor в map и каков его порог в Go?::Среднее количество элементов на бакет. Порог в классической реализации — **6.5**. При превышении запускается расширение (эвакуация).
<!--SR:!2026-03-12,1,210-->

Как происходит эвакуация данных при росте классической map?
?
1. Load factor > 6.5 → создаётся новый массив бакетов ×2
2. Старые бакеты сохраняются в oldbuckets
3. При каждой операции записи/удаления переносятся 1–2 бакета
4. Флаг `evacuated` отслеживает статус
5. Чтение проверяет обе версии до завершения эвакуации
<!--SR:!2026-03-12,1,210-->

Что такое Swiss Tables и с какой версии Go они появились?::Новая реализация map с **Go 1.24**. Использует открытую адресацию, метаданные (control word) и SIMD-инструкции для параллельного поиска внутри групп по 8 слотов.
<!--SR:!2026-03-12,1,210-->

Как устроен control word в Swiss Tables?::64-битное слово (8 байтов), по одному байту на слот. Каждый байт — либо 7 нижних бит хэша ключа (h2, слот занят), либо статус: пустой (`10000000`) или tombstone (`11111110`).
<!--SR:!2026-03-12,1,210-->

Что такое SIMD в контексте Swiss Tables?::Процессорные инструкции, позволяющие за одну операцию сравнить h2 искомого ключа со всеми 8 байтами control word группы. Позволяет найти все потенциальные совпадения в группе параллельно.
<!--SR:!2026-03-12,1,210-->

Почему при удалении элемента из Swiss Tables ставится tombstone, а не просто пустой слот?::При открытой адресации пустой слот разрывает цепочку пробирования — элементы, сдвинувшиеся из-за коллизий, станут недостижимы. Tombstone говорит поиску продолжать идти дальше.
<!--SR:!2026-03-12,1,210-->

Почему порядок итерации по map в Go случаен?
?
1. Случайный seed (`hash0`) при запуске → разное распределение ключей по бакетам
2. Итератор специально начинает со случайного бакета и случайного смещения внутри него
<!--SR:!2026-03-12,1,210-->

Что произойдёт при изменении map во время итерации `range`?::Разрешено. Удалённый до прохода элемент — не выведется. Добавленный во время итерации — может быть как выведен, так и пропущен (неопределённо).
<!--SR:!2026-03-12,1,210-->

Почему нельзя взять адрес элемента map `&m["key"]`?::Это ошибка компиляции. При эвакуации (classic) или делении таблиц (Swiss Tables) данные физически перемещаются в памяти — любой указатель стал бы невалидным.
<!--SR:!2026-03-12,1,210-->

Что безопасно делать с nil-map, а что вызовет панику?::Чтение (`m[key]`, `len(m)`, `delete`) — безопасно. Запись (`m[key] = val`) — **паника**.
<!--SR:!2026-03-12,1,210-->

Как защитить map при конкурентной работе горутин?::Использовать `sync.Mutex` (полная блокировка) или `sync.RWMutex` (несколько горутин могут читать одновременно, запись блокирует всех).
<!--SR:!2026-03-12,1,210-->

Зачем передавать hint в `make(map[K]V, size)`?::Для пре-аллокации памяти под ожидаемое количество элементов. Предотвращает лишние эвакуации (classic) или деления таблиц (Swiss Tables) и повышает производительность.
<!--SR:!2026-03-12,1,210-->

Как полностью очистить map в Go 1.21+?::`clear(m)` — очищает все элементы, сохраняя аллоцированную память.
<!--SR:!2026-03-12,1,210-->

Чем Swiss Tables отличаются от классических map по структуре роста?::Classic: один массив бакетов, эвакуация всей таблицы инкрементально. Swiss Tables: несколько независимых таблиц ≤ 1024 записей; переполнение → только эта таблица делится на две.
<!--SR:!2026-03-12,1,210-->

Каковы требования к типам ключей map?::Ключи должны быть **comparable** — поддерживать оператор `==`. Срезы, map, функции в качестве ключей — запрещены.
<!--SR:!2026-03-12,1,215-->

Для чего в hmap поле B вместо прямого количества бакетов?::B — логарифм по основанию 2. Пример: B=3 → 8 бакетов (2³). Экономит память и ускоряет побитовые операции при вычислении индекса бакета.
<!--SR:!2026-03-12,1,215-->

Что такое hash0 в hmap и зачем он нужен?::Случайный seed для хэш-функции, генерируется при запуске. Защищает от Hash-DoS атаки: без seed злоумышленник может подобрать ключи, все попадающие в один бакет, деградируя map до O(n).
<!--SR:!2026-03-12,1,215-->

Что означает константа tophash `emptyRest` (0)?::Слот пуст, и ВСЕ последующие слоты в этом бакете (и overflow цепочке) тоже пусты. Позволяет немедленно прервать поиск, не обходя остаток бакета.
<!--SR:!2026-03-12,1,215-->

Что означает константа tophash `emptyOne` (1)?::Слот пуст (ключ удалён), но дальше в бакете могут быть заполненные слоты — поиск продолжается.
<!--SR:!2026-03-12,1,215-->

Что означают константы `evacuatedX` (2) и `evacuatedY` (3) в tophash?::Ключ уже эвакуирован: X — в первую половину нового массива, Y — во вторую. Используется при double-size rehash для определения направления переноса.
<!--SR:!2026-03-12,1,215-->

Что такое `minTopHash` (5) в tophash?::Минимальное значение для реального хэша ключа. Если старший байт хэша < 5, к нему прибавляют minTopHash — чтобы избежать коллизии с системными статусными флагами (0–4).
<!--SR:!2026-03-12,1,215-->

Каков пошаговый алгоритм Lookup в классической map?
?
1. Вычислить 64-битный хэш с hash0
2. Младшие B бит → индекс бакета
3. Старший байт → tophash для быстрой проверки
4. Если идёт эвакуация: проверить nevacuate → читать из oldbuckets если не эвакуирован
5. Перебрать tophash[8]: emptyRest → стоп; несовпадение → пропустить; совпадение → полное сравнение ключа
6. Не найдено → перейти по overflow *bmap
7. Цепочка закончилась → вернуть zero value
<!--SR:!2026-03-12,1,215-->

Каков пошаговый алгоритм Insert в классической map?
?
1. Найти бакет по хэшу
2. Если ключ существует → обновить значение
3. Найти свободный слот (emptyOne или emptyRest)
4. Если все 8 слотов заняты → создать overflow bucket, связать через pointer, инкрементировать noverflow
5. Записать tophash, ключ, значение
<!--SR:!2026-03-12,1,215-->

Каков пошаговый алгоритм Delete в классической map?
?
1. Найти слот через Lookup
2. Очистить ключ и значение (обнулить указатели для GC)
3. Если последний занятый слот → tophash = emptyRest
4. Иначе → tophash = emptyOne
5. Уменьшить count
<!--SR:!2026-03-12,1,215-->

Какие два триггера запускают рост классической map?
?
1. **Load factor > 6.5** → double-size rehash: новый массив ×2, элементы размазываются
2. **Слишком много overflow buckets** (noverflow > порог) → same-size rehash: уплотнение данных без увеличения размера
<!--SR:!2026-03-12,1,215-->

В чём разница между double-size и same-size rehash?::Double-size: B+1, новый массив ×2 — снижает плотность при большом load factor. Same-size: B не меняется — уплотняет данные при избытке overflow buckets после многих удалений/вставок.
<!--SR:!2026-03-12,1,215-->

Как работает nevacuate при инкрементальной эвакуации?::Счётчик индекса следующего бакета для эвакуации. Все бакеты с индексом < nevacuate считаются уже перенесёнными. При каждой записи/удалении продвигается вперёд на 1–2 бакета.
<!--SR:!2026-03-12,1,215-->

Как map читает данные во время незавершённой эвакуации?::Проверяет nevacuate: если индекс бакета < nevacuate → бакет эвакуирован → читать из нового массива. Иначе → читать из oldbuckets.
<!--SR:!2026-03-12,1,215-->

Как рандомизируется итерация в классической map?::mapiterinit вызывает fastrand и вычисляет startBucket (случайный бакет старта) и offset (случайное смещение 0–7 внутри бакета). mapiternext обходит все бакеты по кругу с этой точки.
<!--SR:!2026-03-12,1,215-->

Каков load factor в Swiss Tables (Go 1.24+)?::87.5% (7/8) — до 7 занятых слотов из 8 в группе. Это выше, чем в классической реализации (6.5), что снижает потребление памяти.
<!--SR:!2026-03-12,1,215-->

Как разбивается 64-битный хэш ключа в Swiss Tables?::H1 (верхние 57 бит) — выбор таблицы и группы. H2 (нижние 7 бит) — сохраняется в control word, используется для SIMD-сравнения.
<!--SR:!2026-03-12,1,215-->

Каков алгоритм Insert в Swiss Tables?
?
1. Выполнить Lookup — если ключ есть, обновить значение
2. Найти первый свободный слот (пустой или tombstone)
3. Если группа полностью занята → треугольное пробирование
4. Записать ключ и значение в слот
5. Обновить байт control word значением H2
<!--SR:!2026-03-12,1,215-->

Каков алгоритм Delete в Swiss Tables?
?
1. Найти слот через Lookup
2. Очистить ключ и значение (обнулить указатели для GC)
3. Если в группе есть пустые слоты → пометить как пустой
4. Если группа была полностью заполнена → пометить как **tombstone** (иначе разорвётся цепочка пробирования)
<!--SR:!2026-03-12,1,215-->

Что такое треугольное пробирование?::Стратегия поиска следующей группы при коллизии: `i+1, i+1+2, i+1+2+3, ...`. Решает проблему кластеризации и гарантирует обход всех слотов за O(1) при размере таблицы, равном степени двойки.
<!--SR:!2026-03-12,1,215-->

Какой прирост производительности даёт Swiss Tables по сравнению с классической map?::В микробенчмарках до **60% быстрее**. В реальных крупных приложениях снижение CPU-времени ~**1.5%**. Повышенный load factor (87.5%) снижает потребление памяти.
<!--SR:!2026-03-12,1,215-->

Какая паника возникает при конкурентной записи в map?::`fatal error: concurrent map writes` — среда выполнения Go намеренно вызывает панику при обнаружении data race в map.
<!--SR:!2026-03-12,1,215-->

Что такое `sync.Map` и когда его использовать?::Специализированный тип из стандартной библиотеки для безопасной конкурентной работы с ключами из разных горутин. Оптимален при частых чтениях и редких записях из множества горутин.
<!--SR:!2026-03-12,1,215-->

Что происходит при `make(map[K]V, hint)` если hint ≤ 8?::Создаётся упрощённая однотабличная map без сложной структуры директорий — одна группа слотов без дополнительных уровней абстракции.
<!--SR:!2026-03-12,1,210-->
