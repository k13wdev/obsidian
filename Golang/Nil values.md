## Суть и контекст

`nil` в Go — это нулевое значение (zero value) для определённых типов данных, указывающее на отсутствие значения или на то, что переменная не была инициализирована. По своей сути `nil` является идентификатором, который используется для обозначения пустого состояния указателей, интерфейсов и ссылочных типов.

Основной контекст использования `nil` — обработка ошибок. Большинство функций в Go возвращают значение ошибки или `nil`, если операция прошла успешно. Также `nil` применяется для проверки, был ли объект инициализирован, перед началом работы с ним.

---

## Ключевые идеи

- `nil` — нулевое значение для указателей, срезов, мап, каналов, функций и интерфейсов.
- Поведение программы при взаимодействии с `nil` принципиально различается в зависимости от типа.
- Интерфейс равен `nil` только тогда, когда не заданы **ни тип, ни значение** — это главный источник неожиданных багов.
- Чтение из `nil`-мапы безопасно; запись — паника.
- Операции с `nil`-каналом (чтение/запись) вызывают вечную блокировку; в `select` — отключают ветку.
- `nil`-срезы безопасны для `append`; чтение по индексу — паника.
- Разыменование `nil`-указателя или вызов `nil`-функции — немедленная паника.
- `if err != nil` — фундаментальный паттерн обработки ошибок в Go.

---

## Определения и термины

**nil** — идентификатор в Go, представляющий нулевое значение для типов: указатели, срезы, мапы, каналы, функции, интерфейсы. Означает отсутствие значения или неинициализированное состояние.

**Zero value** — значение по умолчанию для переменной в Go, если она не была явно инициализирована. Для ссылочных типов zero value равно `nil`.

**Nil pointer dereference** — паника, возникающая при попытке разыменовать указатель, равный `nil`. Одна из наиболее частых runtime-ошибок.

**Deadlock** — ситуация вечной блокировки при чтении или записи в `nil`-канал: операция никогда не завершается.

**Nil interface** — интерфейс, у которого не заданы **ни конкретный тип, ни значение**. Только в этом случае `i == nil` возвращает `true`.

## Как устроено

### Типы, которые могут быть nil

| Тип | nil по умолчанию? | Описание |
|---|---|---|
| Указатель (pointer) | Да | Не ссылается ни на какой адрес в памяти |
| Срез (slice) | Да | Длина и ёмкость равны 0 |
| Мапа (map) | Да | Объявлена без инициализации через `make` |
| Канал (channel) | Да | Неинициализированный канал |
| Функция (func) | Да | Переменная типа функции без присвоенного значения |
| Интерфейс (interface) | Да | Не содержит ни типа, ни значения |

### Устройство nil-интерфейса

Интерфейс в Go под капотом состоит из двух компонентов:
1. **Конкретный тип** — динамический тип хранимого значения.
2. **Значение** — само хранимое значение.

Интерфейсная переменная считается равной `nil` **только тогда, когда оба компонента — и тип, и значение — не заданы**.

**Главный подводный камень:** если интерфейсу присвоить указатель, который сам по себе равен `nil`, интерфейс **перестанет быть равным nil**. Поле «тип» в структуре интерфейса будет заполнено (например, типом `*int`), хотя значение остаётся `nil`.

## Детали и нюансы

### Поведение nil по типам

#### Мапы (maps)

| Операция | Результат |
|---|---|
| Чтение из nil-мапы | Безопасно — вернёт zero value соответствующего типа |
| Запись в nil-мапу | Паника: `assignment to entry in nil map` |
| `delete` с nil-мапой | Безопасно, ошибок нет |
| `len` с nil-мапой | Безопасно, возвращает 0 |
| `range` с nil-мапой | Безопасно, цикл не выполняется |

#### Каналы (channels)

| Операция | Результат |
|---|---|
| Чтение из nil-канала | Вечная блокировка (deadlock) |
| Запись в nil-канал | Вечная блокировка (deadlock) |
| Закрытие nil-канала | Паника |
| Nil-канал в `select` | Отключает соответствующую ветку `case` |

Поведение nil-канала в `select` используется намеренно — для динамического управления логикой выбора: «выключенный» канал никогда не будет выбран планировщиком.

#### Срезы (slices)

| Операция | Результат |
|---|---|
| `append` к nil-срезу | Безопасно — срез инициализируется при добавлении первого элемента |
| Чтение по индексу из nil-среза | Паника |
| `len` nil-среза | 0 |
| `cap` nil-среза | 0 |

Срезы считаются относительно «NIL-безопасными» благодаря поведению `append`.

#### Указатели и функции

| Операция | Результат |
|---|---|
| Разыменование nil-указателя | Немедленная паника: `nil pointer dereference` |
| Вызов nil-функции | Немедленная паника |

#### Интерфейсы

- `if i == nil` — стандартная проверка: истинна только если и тип, и значение не заданы.
- Присвоение интерфейсу nil-указателя делает интерфейс **не равным nil** — это ловушка при возврате ошибок.

|---|---|---|---|---|
| Указатель | — | — | Паника (dereference) | — |
| Срез | Паника (по индексу) | append — безопасно | — | — |
| Мапа | Безопасно (zero value) | Паника | — | — |
| Канал | Deadlock | Deadlock | Паника | Отключает ветку |
| Функция | — | — | Паника (вызов) | — |
| Интерфейс | Зависит от реализации | — | — | — |

## Примеры

### 1. Nil-интерфейс и ловушка с nil-указателем

```go
var aType error = nil
switch aType.(type) {
case nil:
    fmt.Println("Это nil интерфейс!") // Сработает, если и тип, и значение пусты
default:
    fmt.Println("Интерфейс не nil!")
}
```

### 2. Ловушка при возврате ошибки через интерфейс

Функция возвращает `error` как интерфейс. Внутри создаётся указатель на пользовательскую структуру ошибки, который равен `nil`. Если функция вернёт этот указатель, проверка `if err != nil` во внешнем коде окажется **истинной**, хотя фактической ошибки нет — потому что поле «тип» интерфейса заполнено.

### 3. Nil-канал в select для управления логикой

Ниловый канал в операторе `select` отключает соответствующую ветку `case` — она никогда не будет выбрана. Это используется для динамического включения/выключения веток выбора.

## Связи с другими темами

- [[Интерфейсы (interfaces)]] — nil-интерфейс имеет двухкомпонентную структуру (тип + значение); возврат nil-ошибки через интерфейс — главная ловушка `if err != nil`.
- [[Каналы (channels)]] — nil-канал имеет специфическое поведение в `select`, что используется как паттерн управления.
- [[Массивы и Срезы]] — nil-срез безопасен для `append`, но не для индексирования.
- [[Map (старая и новая)]] — nil-мапа доступна для чтения, но запись вызывает панику.
- [[Указатели (ссылки)]] — разыменование nil-указателя вызывает панику; всегда проверяй `if p != nil`.

## Важно / подводные камни / best practices

**Подводные камни:**

- **Nil-интерфейс с непустым типом.** Если функция возвращает интерфейс (например, `error`), а внутри создаётся конкретный nil-указатель и возвращается как интерфейс — проверка `if err != nil` даст `true`. Всегда возвращай чистый `nil`, а не типизированный nil-указатель.
- **Nil-канал — дедлок, не ошибка.** Чтение/запись в nil-канал не вызывают паники — программа просто зависнет навсегда.
- **Nil-срез — не всегда безопасен.** `append` работает корректно, но доступ по индексу — паника.

**Best practices:**

- Всегда инициализируй мапу через `make` перед записью, чтобы избежать паники.
- Никогда не передавай `nil` вместо контекста в функции. Если контекст ещё не определён — используй `context.TODO()`.
- Перед разыменованием указателя, если нет уверенности в его инициализации, делай проверку `if p != nil`.
- Используй `if err != nil` как фундаментальный паттерн обработки ошибок.
- Инициализируй каналы через `make` перед использованием, если не задумана логика с nil-каналом в `select`.
