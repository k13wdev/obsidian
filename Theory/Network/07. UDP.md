## Суть и контекст

UDP — простейший транспортный протокол без установления соединения и без гарантий надёжности. Не выполняет рукопожатие, не контролирует перегрузки, не гарантирует порядок доставки. Применяется там, где важнее скорость, чем полнота данных.

---

## Ключевые идеи

- Нет установления соединения — передача начинается мгновенно
- Нет контроля перегрузок — отправитель шлёт на любой скорости
- Заголовок — всего 8 байт (против 20+ у TCP)
- Нет состояния соединения — сервер поддерживает больше клиентов одновременно
- Демультиплексирование по порту получателя + IP-адресу назначения
- Контрольная сумма — единственный механизм обнаружения ошибок

---

## Определения и термины

**UDP** — транспортный протокол без соединения; передаёт данные как независимые дейтаграммы без гарантий доставки, порядка и целостности.

**Дейтаграмма** — единица данных UDP; каждая дейтаграмма независима и маршрутизируется отдельно.

**Мультиплексирование** — сбор сообщений из разных сокетов, добавление заголовков с номерами портов и передача на сетевой уровень.

**Демультиплексирование** — при получении пакета — определение нужного сокета по порту получателя и IP-адресу назначения.

**Checksum (контрольная сумма)** — вычисляется как дополнение до единицы суммы всех 16-битных слов сегмента и псевдозаголовка (с IP-адресами).

**Псевдозаголовок UDP** — структура из 12 байт (Source IP, Destination IP, Protocol=17, UDP length), включаемая в вычисление контрольной суммы для обнаружения неверно доставленных пакетов. Нарушает строгую иерархию протоколов, используя IP-адреса сетевого уровня.

## Как устроено

### Структура UDP-дейтаграммы (заголовок — 8 байт)

| Поле | Размер | Описание |
|---|---|---|
| Порт отправителя | 16 бит | Идентифицирует процесс-источник для обратной связи |
| Порт получателя | 16 бит | Указывает приложение-адресат на хосте |
| Длина | 16 бит | Полная длина сегмента (заголовок + данные) в байтах |
| Контрольная сумма | 16 бит | Обнаружение повреждений битов при передаче |

### Демультиплексирование в UDP

В отличие от TCP (где сокет определяется 4-tuple: src IP, src port, dst IP, dst port), UDP-демультиплексирование использует только **2 параметра**: порт получателя + IP-адрес назначения.

Все UDP-сегменты с одинаковым портом назначения и IP-адресом направляются в **один и тот же сокет** — независимо от IP-адреса и порта источника.

### Контрольная сумма

- Вычисляется по всем 16-битным словам сегмента + псевдозаголовок (IP-адреса)
- При несовпадении на стороне получателя — сегмент отбрасывается или передаётся приложению с предупреждением

### Алгоритм вычисления контрольной суммы
1. Поле Checksum устанавливается в 0
2. Данные дополняются нулевым байтом, если длина нечётная
3. Суммируются все 16-битные слова псевдозаголовка, заголовка и данных (с переносом переполнения)
4. Вычисляется дополнение до единицы (инверсия всех битов)
5. Если результат = 0, записывается `0xFFFF` (ноль зарезервирован как «не вычислялась»)

### Псевдозаголовок IPv4 (12 байт)
| Поле | Размер |
|---|---|
| Source IP address | 32 бита |
| Destination IP address | 32 бита |
| Нули | 8 бит |
| Protocol ID | 8 бит (= 17) |
| UDP length | 16 бит |

## Детали и нюансы

### Максимальный размер дейтаграммы
Теоретический предел — **65 515 байт** (максимум IP-пакета 65 535 − 20 байт IP-заголовка).

### ICMP Port Unreachable
Если UDP-пакет приходит на порт, не прослушиваемый ни одним сокетом — хост отбрасывает пакет и отправляет ICMP «Port unreachable» (тип 3, код 3).

### Переполнение буфера
Если приложение считывает данные медленнее, чем они поступают — буфер переполняется и последующие дейтаграммы **безвозвратно теряются**.

### Принцип сквозной связи (End-to-End Principle)
Контрольная сумма UDP — воплощение End-to-End Principle: критичные проверки целостности реализуются на конечных хостах, так как промежуточные узлы (канальные CRC) защищают только hop-by-hop, а не весь путь.

## Сравнения и trade-offs

### UDP vs TCP

| Характеристика | UDP | TCP |
|---|---|---|
| Соединение | Без установления | С handshake |
| Надёжность | Ненадёжный | Гарантированная доставка |
| Порядок | Не гарантирован | Упорядоченный поток |
| Контроль перегрузок | Нет | Есть (slow start, AIMD) |
| Контроль потока | Нет | Есть (rwnd) |
| Заголовок | 8 байт | 20+ байт |
| Задержка | Минимальная | Выше (handshake, ACK) |
| Состояние | Нет | Stateful |

### Когда выбирать UDP

| Причина | Объяснение |
|---|---|
| Нет задержки на handshake | DNS: быстрая трансляция имён без лишних пауз |
| Скорость важнее полноты | VoIP, видео: потеря кадра лучше, чем задержка |
| Малый заголовок | Снижает накладные расходы |
| Больше клиентов | Нет состояния → больше одновременных соединений на сервере |

### Применение UDP

| Протокол/сервис | Причина выбора UDP |
|---|---|
| DNS | Минимальная задержка, короткие запросы |
| VoIP (Skype и др.) | Задержки хуже потерь; реалтайм |
| Видеоконференции | Аналогично VoIP |
| Потоковое видео (live) | Реальное время важнее полноты |
| Сетевые игры | Минимальный RTT |

## Важно / подводные камни / best practices

1. **UDP — «неуправляемый трафик»** — при перегрузке сети UDP-источник не замедляется; это может вытеснять TCP-потоки
2. **Потери нормальны** — приложения поверх UDP должны сами решать, что делать с потерянными дейтаграммами
3. **DNS по TCP** — DNS переходит на TCP при зонных передачах и ответах >512 байт
4. **Контрольная сумма обязательна в IPv6** — в отличие от IPv4, где она опциональна
5. **Отключение контрольной суммы** — допустимо только для трафика, невосприимчивого к битовым искажениям (оцифрованная речь). Искажения могут возникать в памяти маршрутизаторов, а не только в линии связи
6. **UDP без контроля перегрузки** — при масштабном использовании в интернете приводит к congestion collapse и вытеснению TCP-трафика. Необходимо реализовывать контроль перегрузки на прикладном уровне
7. **DDoS Amplification** — открытые UDP-сервисы (DNS, NTP) используются для DDoS-атак отражения/усиления. Коэффициент усиления: DNS — 54.6×, NTP — 556.9×

Что такое UDP?::Транспортный протокол без установления соединения. Не гарантирует доставку, порядок и целостность данных. Быстрый, с минимальным заголовком (8 байт).

Из каких полей состоит заголовок UDP?
?
4 поля по 2 байта (итого 8 байт):
1. Порт отправителя
2. Порт получателя
3. Длина (заголовок + данные)
4. Контрольная сумма

Почему выбирают UDP вместо TCP?
?
- Нет задержки на handshake (DNS)
- Нет контроля перегрузок (мультимедиа реального времени)
- Малый заголовок (меньше накладных расходов)
- Нет состояния → больше клиентов на сервере

Как работает демультиплексирование в UDP?::По двум параметрам: порт получателя + IP-адрес назначения. Все UDP-сегменты с одинаковым dst-портом и IP идут в один сокет, независимо от источника.

Чем демультиплексирование UDP отличается от TCP?::TCP: сокет определяется 4-tuple (src IP, src port, dst IP, dst port). UDP: только 2 параметра (dst IP, dst port).

Где применяется UDP?::DNS (быстрые запросы), VoIP, видеоконференции, потоковое видео в реальном времени, сетевые игры.

?
1. Checksum = 0
2. Дополнение данных нулевым байтом (если нечётная длина)
3. Сумма 16-битных слов псевдозаголовка + заголовка + данных
4. Инверсия битов (дополнение до единицы)
5. Если результат = 0, записывается 0xFFFF

Что такое псевдозаголовок UDP?::12-байтная структура (Source IP, Dest IP, Protocol=17, UDP length), включаемая в вычисление контрольной суммы для обнаружения неверно доставленных пакетов. Нарушает иерархию протоколов.

Каков максимальный размер UDP-дейтаграммы?::65 515 байт (65 535 макс. IP-пакет − 20 байт IP-заголовка).

Почему UDP опасен для интернета без контроля перегрузки?::Отправители не снижают скорость при потерях → congestion collapse маршрутизаторов + вытеснение TCP-трафика, который корректно замедляется.
