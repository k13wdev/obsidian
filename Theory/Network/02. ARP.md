## Суть и контекст

ARP — протокол преобразования IP-адресов (сетевой уровень L3) в MAC-адреса (канальный уровень L2) в локальных сетях. Маршрутизаторы и хосты используют IP для логической адресации, но для физической передачи кадра по Ethernet необходимо знать аппаратный MAC-адрес узла. ARP является **граничным протоколом** между L2 и L3.

---

## Ключевые идеи

- ARP работает только в пределах одной локальной подсети (broadcast domain)
- Маршрутизаторы **не пересылают** широковещательные ARP-запросы
- Записи в ARP-таблице временные: типичный TTL — **20 минут**
- Стандартный ARP — **stateless** и не проверяет подлинность ответов → уязвим к spoofing
- В IPv6 ARP заменён протоколом **NDP** (Neighbor Discovery Protocol)

---

## Определения и термины

- **ARP-таблица (ARP-кэш)** — локальная таблица соответствий IP ↔ MAC на каждом хосте
- **Gratuitous ARP** — широковещательный ARP, отправляемый хостом *без* входящего запроса: используется для обнаружения IP-конфликтов и обновления кэшей других узлов
- **Proxy ARP** — роутер отвечает на ARP-запрос *от имени* хоста в другой сети, подставляя свой MAC
- **RARP (Reverse ARP)** — обратный ARP: хост знает свой MAC, но не знает IP; устарел, заменён DHCP
- **ARP Spoofing / Poisoning** — атака: злоумышленник рассылает поддельные ARP-ответы, связывая свой MAC с чужим IP

---

## Как устроено

### Пошаговый процесс ARP-запроса и ответа

```
Хост A хочет отправить пакет на IP 192.168.1.5
```

1. **Проверка маршрутизации**: ОС определяет — целевой IP в локальной подсети или нужен default gateway? Это определяет, чей MAC запрашивать.

2. **Поиск в ARP-кэше**: если запись есть → сразу формируется кадр.

3. **Формирование ARP-запроса** (если кэш пуст):
   - Поля: IP отправителя, MAC отправителя, IP цели
   - Поле `Target MAC` = `00:00:00:00:00:00` (нули)

4. **Широковещательная рассылка**:
   - ARP-пакет инкапсулируется в Ethernet-кадр
   - MAC назначения = `FF:FF:FF:FF:FF:FF`
   - Получают **все** узлы сегмента

5. **Обработка узлами**:
   - Все адаптеры принимают кадр
   - Только узел с совпадающим IP готовит ответ

6. **ARP-ответ (unicast)**:
   - Целевой узел отправляет ответ **напрямую** (unicast) запрашивающему хосту
   - В ответе указан его MAC-адрес

7. **Обновление таблиц**:
   - Отправитель сохраняет MAC в кэш
   - Целевой узел и другие «подслушивающие» узлы также могут обновить свой кэш данными об отправителе из запроса

8. **Передача данных**:
   - IP-пакет упаковывается в кадр с корректным MAC назначения и отправляется

### ARP в многосетевом окружении (через маршрутизатор)

- Роутеры **не форвардят** широковещательные ARP-кадры
- Если цель в другой подсети: хост запрашивает MAC **шлюза по умолчанию**, а не финального получателя
- Далее пакет движется **hop-by-hop**: каждый роутер делает ARP-запрос для следующего узла в пути

## Формат ARP-пакета

Стандартный ARP-пакет для IPv4 over Ethernet — **28 байт**:

| Поле | Размер | Значение |
|------|--------|----------|
| Hardware Type | 2 байта | `1` = Ethernet |
| Protocol Type | 2 байта | `0x0800` = IPv4 |
| Hardware Address Length | 1 байт | `6` (длина MAC для Ethernet) |
| Protocol Address Length | 1 байт | `4` (длина IP для IPv4) |
| Operation | 2 байта | `1` = Request, `2` = Reply |
| Sender Hardware Address | 6 байт | MAC отправителя |
| Sender Protocol Address | 4 байта | IP отправителя |
| Target Hardware Address | 6 байт | MAC цели (нули в запросе) |
| Target Protocol Address | 4 байта | IP цели |

ARP-пакет инкапсулируется непосредственно в **Ethernet-кадр**.

| Параметр | Значение |
|----------|----------|
| Типичный TTL записи | ~20 минут |
| Условие удаления | Запись не обновлялась (узел молчал) |
| Обновление | При получении нового ARP-ответа или «подслушивании» чужих запросов |

## Детали и нюансы

### Варианты ARP

| Тип | Описание | Применение |
|-----|----------|------------|
| **ARP** | Стандартный запрос IP → MAC | Обычная передача в LAN |
| **Gratuitous ARP** | Хост анонсирует свой IP→MAC без запроса | Обнаружение IP-конфликтов, обновление кэшей при смене MAC |
| **Proxy ARP** | Роутер отвечает за хост в другой сети, подставляя свой MAC | Удалённые узлы, dial-up соединения |
| **RARP** | MAC → IP (обратный) | Устарел, заменён DHCP |

### Где ARP не используется

- **Point-to-point соединения** (VPN-туннели) — нет multi-access среды
- **4G/5G мобильные сети** — другая адресация
- **IPv6** — заменён NDP (Neighbor Discovery Protocol, использует multicast вместо broadcast)

### NDP в IPv6 (детально)

1. Узел формирует ICMPv6 **Neighbor Solicitation (NS)**
2. Адрес назначения — **Solicited-Node Multicast Address**, синтезированный из целевого IP
3. Пакет упаковывается в Ethernet-кадр с групповым MAC-адресом
4. Только узлы, подписанные на эту multicast-группу (скорее всего, только искомый узел), обрабатывают запрос
5. Целевой узел отвечает **Neighbor Advertisement (NA)** с MAC-адресом

NDP также обеспечивает **DAD (Duplicate Address Detection)** — обнаружение дубликатов адресов.

Защита: проверка **TTL=255** (пакет пришёл из локального сегмента, а не маршрутизировался).

---

## Сравнения и trade-offs

| | ARP (IPv4) | NDP (IPv6) |
|-|------------|------------|
| Метод обнаружения | Broadcast | Multicast |
| Протокол | Отдельный | Часть ICMPv6 |
| Безопасность | Stateless, нет проверки | Поддерживает SEcure ND (SEND) |
| Overhead | Выше (broadcast все узлы) | Ниже (multicast только заинтересованные) |
| DAD | Нет | Есть (Duplicate Address Detection) |
| Защита от подмены | Нет | Проверка TTL=255 |

## Важно / подводные камни / best practices

### ARP Spoofing / Poisoning

Поскольку ARP stateless и не проверяет подлинность ответов:

1. **ARP Spoofing**: злоумышленник рассылает поддельные ARP-ответы → связывает свой MAC с легитимным IP
2. **MITM-атака**: «отравляя» ARP-таблицы двух сторон, атакующий встаёт между ними — перехватывает, модифицирует или записывает весь трафик

### Broadcast Storms

Неисправные устройства могут наводнить сеть ARP-трафиком → паралич LAN.

### Роутер как барьер

Роутер ограничивает broadcast domain — ARP-запросы не выходят за его пределы, что естественно изолирует подсети.
