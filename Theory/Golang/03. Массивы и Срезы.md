## Суть и контекст

**Массив (array)** — структура данных фиксированного размера, хранящая последовательность элементов одного типа в непрерывной области памяти.

**Срез (slice)** — динамическая реализация массива, позволяющая расширять или сужать последовательность элементов. Срез — один из ключевых типов Go, используется повсеместно вместо массивов.

---

## Ключевые идеи

- Массив имеет фиксированную длину; длина является частью его типа — `[3]int` и `[4]int` это **разные типы**.
- Длина массива — всегда **константа**, известная на этапе компиляции. Задать через переменную нельзя.
- Срез — это дескриптор из трёх полей: указатель на базовый массив + длина (`len`) + ёмкость (`cap`).
- Внутри среза **всегда лежит базовый массив**.
- Массив передаётся по значению (копируются все элементы); срез — ссылочный тип (копируется только дескриптор).
- `len()` — количество используемых элементов; `cap()` — сколько можно добавить без перевыделения памяти.
- При нехватке `cap` функция `append` выделяет **новый** массив большего размера и копирует данные.
- Срезы от одного базового массива разделяют данные — изменение одного меняет другой.
- Маленький срез может удерживать в памяти огромный базовый массив — источник утечек.
- Нулевое значение массива — массив с нулями; нулевое значение среза — `nil`.
- Трёхиндексный срез `s[a:b:c]` позволяет ограничить ёмкость и защитить данные.
- При росте стека массив может **перемещаться** в памяти (менять адрес).

## Определения и термины

**Массив (array)** — последовательность элементов одного типа фиксированной длины. Длина — часть типа. Хранится как непрерывная область памяти. Не является указателем.

**Срез (slice)** — динамическая структура-дескриптор из трёх полей: указатель на базовый массив, длина (`len`), ёмкость (`cap`). Тип среза не включает размер: `[]int`.

**Базовый массив (underlying array)** — непрерывный массив в памяти, на который ссылается срез через внутренний указатель.

**Длина (`len`)** — количество элементов, фактически содержащихся в массиве или срезе. Возвращается функцией `len()`.

**Ёмкость (`cap`)** — максимальное количество элементов, которое может вместить базовый массив, начиная с позиции указателя среза. Возвращается функцией `cap()`. Для массива всегда равна длине.

**Дескриптор среза** — небольшая структура (указатель + len + cap), которая копируется при передаче среза в функцию.

**Релокация (relocation)** — процесс выделения нового базового массива при переполнении `cap` во время `append`. Старые данные копируются в новый массив, указатель в дескрипторе обновляется.

**Трёхиндексный срез (`s[a:b:c]`)** — синтаксис явного задания ёмкости нового среза: `len = b - a`, `cap = c - a`.

**Утечка памяти через срезы** — ситуация, когда маленький срез удерживает ссылку на огромный базовый массив, не давая GC его освободить.

**Утечка через срез указателей** — при под-срезе от массива указателей: элементы вне нового среза не освобождаются GC, т.к. базовый массив всё ещё удерживается. Решение: занулить отсечённые элементы через `clear()` или `s[i] = nil` до взятия под-среза.

## Как устроено

### Массив в памяти

Массив — непрерывная область памяти. Не является указателем; это структура с адресом начала, длиной и размером элементов. Все элементы хранятся последовательно.

### Срез в памяти

Срез — это дескриптор из трёх полей:

```
┌─────────────┐
│  ptr        │──► [базовый массив в памяти]
│  len        │
│  cap        │
└─────────────┘
```

При передаче среза в функцию копируется этот дескриптор (24 байта), но не данные. Обе копии дескриптора указывают на один и тот же базовый массив.

### Аллокация: стек или куча

| Тип | Условие попадания в стек |
|---|---|
| Массив | Размер < 10 МБ |
| Базовый массив среза | Размер < 64 КБ |

Финальное решение принимает [[01. Стек и куча]] компилятора.

### Алгоритм роста `append`

1. Если `len < cap` — записывает элемент в следующую свободную ячейку, увеличивает `len` на 1. Аллокаций нет.
2. Если `cap` исчерпан — выделяет новый, более вместительный базовый массив, копирует старые данные, добавляет новый элемент, обновляет указатель в дескрипторе. Старый массив удаляется GC, если на него больше нет ссылок.

**Коэффициент роста ёмкости:**
- **До Go 1.18**: удвоение при `cap < 1024`, затем ~25%.
- **С Go 1.22**: порог удвоения снижен до **256** элементов; после применяется более плавная математическая формула роста (не просто 25%).

**Edge case с литералами:** Если инициализировать срез литералом с нечётным числом элементов начиная с 5 (например, `[]int{1,2,3,4,5}`), компилятор может выделить `cap = 6` — специфика внутреннего аллокатора.

### Функция `copy`

`copy(dst, src)` — поэлементное копирование. Копирует `min(len(dst), len(src))` элементов.

Важно: если `dst` имеет `len == 0`, `copy` не скопирует ничего. Для полной независимой копии нужно предварительно создать `dst` нужной длины через `make`, или использовать `slices.Clone`.

### Трёхиндексный срез `s[a:b:c]`

- `len = b - a`
- `cap = c - a`

Если задать `b == c`, то `append` внутри функции, получившей такой срез, немедленно вызовет аллокацию нового массива и **не перезапишет** данные за границами под-среза в родительском массиве.

### Двумерные срезы

Двумерный срез — срез, элементами которого являются другие срезы (`[][]int`). При создании через `make([][]int, n)` внутренние срезы равны `nil`. Для добавления элементов используется `append` для каждого внутреннего среза отдельно.

## Детали и нюансы

### Разделяемый базовый массив

`s2 := s1[1:3]` — создаёт новый дескриптор, указывающий на **тот же** базовый массив. Любое изменение элементов `s2` изменит данные в `s1`. Однако если `append` к `s2` превысит его ёмкость, `s2` создаст **новый** базовый массив и «отвяжется» от оригинала.

### Утечка памяти

Маленький срез, ссылающийся на огромный базовый массив, не даёт GC освободить этот массив. Решение — скопировать нужные данные в новый срез с минимальной ёмкостью:

```go
largeData := make([]byte, 1024*1024) // 1 MB
smallCopy := make([]byte, 2)
copy(smallCopy, largeData[0:2])
// теперь largeData может быть собрана GC
```

### Пустой срез vs nil-срез

| | nil-срез (`var s []int`) | Пустой срез (`[]int{}` / `make([]int, 0)`) |
|---|---|---|
| `len` | 0 | 0 |
| `cap` | 0 | 0 |
| Указатель | `nil` | Специальный разделяемый адрес (не `nil`) |
| JSON | `null` | `[]` |

Проверяй через `len(s) == 0`, а не `s == nil` — пустой срез при `== nil` вернёт `false`.
<!--SR:!2026-03-12,1,209-->

### Передача среза в функцию и `append`

При передаче среза по значению и вызове `append` внутри функции:
- Если хватает `cap` — данные в базовом массиве изменяются (и видны снаружи).
- Если `cap` исчерпан — создаётся новый массив; изменения **не видны** снаружи.
- `len` оригинального дескриптора в вызывающей функции **никогда не изменится**.
- **Best practice**: если функция добавляет элементы, она должна возвращать срез: `s = modify(s)`.

### `range` и срезы

Выражение справа от `range` **вычисляется один раз** до начала цикла и копируется. Поэтому `append` к тому же срезу внутри цикла не создаст бесконечный цикл — итерация пройдёт только по первоначальной длине.

```go
s := []int{1, 2, 3}
for _, v := range s {
    s = append(s, v*10) // не бесконечный цикл — range видит len=3
}
// s = [1 2 3 10 20 30]
```

**Переменная в `range` (Go 1.22):** До Go 1.22 переменная значения (`value` в `for _, value := range`) переиспользовала один адрес памяти — взятие `&value` приводило к багам. С Go 1.22 для каждой итерации создаётся уникальный экземпляр.

### Конвертация среза в массив

```go
s := []int{1, 2, 3, 4}

// Указатель на массив — shared память, изменения в arr меняют s
arr := (*[4]int)(s)
arr[0] = 99 // s[0] тоже станет 99

// Копия массива — независимая
arrCopy := [4]int(s)
arrCopy[0] = 99 // s не изменится
```

### Unsafe-оптимизации: `[]byte` ↔ `string` без копирования

При стандартном преобразовании `[]byte` → `string` происходит выделение памяти и копирование. Через `unsafe` можно шарить один буфер:

```go
// []byte → string без копирования (Go 1.20+)
// unsafe.SliceData возвращает указатель на базовый массив среза
s := unsafe.String(unsafe.SliceData(b), len(b))

// string → []byte без копирования
b := unsafe.Slice(unsafe.StringData(s), len(s))
```

> **Риск:** изменение байтов в исходном срезе изменит содержимое строки, нарушая её контракт иммутабельности.

### Утечка памяти через срез указателей

При под-срезе от массива указателей (`ptrs[a:b]`) элементы вне нового среза (`ptrs[0:a]` и `ptrs[b:]`) **не освобождаются GC** — базовый массив всё ещё удерживается. Решение: занулить отсекаемые элементы перед взятием под-среза:

```go
// Занулить «хвост» перед отсечением
for i := b; i < len(ptrs); i++ {
    ptrs[i] = nil
}
result := ptrs[a:b]
// или использовать clear():
clear(ptrs[b:])
result := ptrs[a:b]
```

## Сравнения и trade-offs

| Критерий | Массив | Срез |
|---|---|---|
| Размер | Фиксированный | Динамический |
| Длина как часть типа | Да (`[3]int` ≠ `[4]int`) | Нет (`[]int`) |
| Передача в функцию | По значению (копия всех элементов) | Ссылочный тип (копия дескриптора) |
| Нулевое значение | Массив с нулями | `nil` |
| `cap` | Всегда равна `len` | Может превышать `len` |
| Аллокация (стек) | < 10 МБ | Базовый массив < 64 КБ |

| Способ создания среза | Синтаксис | Особенность |
|---|---|---|
| Литерал | `[]int{1, 2, 3}` | `len == cap == 3` |
| `make` с len | `make([]int, 5)` | `len == cap == 5` |
| `make` с len и cap | `make([]int, 0, 10)` | `len=0`, `cap=10` |
| Срез из массива | `a[1:3]` | Разделяет базовый массив |
| Трёхиндексный | `a[1:3:3]` | Ограничивает `cap` |

## Примеры

### 1. Объявление массивов

```go
var a [5]int              // массив из 5 нулей
a := [3]int{1, 2, 3}     // литерал массива
a := [...]int{1, 2}      // компилятор определит длину
```

### 2. Создание срезов

```go
s := []int{1, 2, 3}          // литерал среза
s := make([]int, 5)           // len=5, cap=5
s := make([]int, 0, 10)       // len=0, cap=10 — пре-аллокация
s := a[1:3]                   // срез из массива
```

### 3. Трёхиндексный срез — защита базового массива

```go
parent := []int{1, 2, 3, 4, 5}
child := parent[1:3:3]  // len=2, cap=2 — append не затронет '4' и '5'
```

### 4. Правильное копирование для предотвращения утечки

```go
largeData := make([]byte, 1024*1024)
smallCopy := make([]byte, 2)
copy(smallCopy, largeData[0:2])
// largeData теперь может быть собрана GC
```

### 5. `append` — всегда присваивать той же переменной

```go
s = append(s, 42)       // правильно
_ = append(s, 42)       // неправильно — результат потерян
```

### 6. Двумерный срез

```go
matrix := make([][]int, 3)   // 3 nil-среза внутри
for i := range matrix {
    matrix[i] = append(matrix[i], 1, 2, 3)
}
```

- [[02. Указатели (ссылки)]] — дескриптор среза содержит указатель на базовый массив; изменения через срез — мутация по адресу.
- [[15. Сборщик мусора (GC)]] — GC не освобождает базовый массив, пока на него ссылается хотя бы один срез — источник утечек памяти.
- [[06. Строки]] — строка внутри реализована как read-only `[]byte`; срезы строк разделяют базовый массив аналогично срезам.

## Важно / подводные камни / best practices

**Подводные камни:**

- **Разделяемый базовый массив.** `s2 := s1[1:3]` — оба среза указывают на один массив. Изменение элементов `s2` меняет данные в `s1`.
- **`append` и отвязка.** Если `append` к под-срезу исчерпает его `cap`, он создаст новый массив и изменения перестанут отражаться в оригинале — неожиданное поведение.
- **Утечка памяти.** Маленький срез от огромного массива удерживает весь массив в памяти. Решение — копировать нужные байты в новый срез.
- **`copy` с пустым `dst`.** Если `len(dst) == 0`, `copy` не скопирует ничего.
- **`len` не изменится снаружи.** Передача среза в функцию, где делается `append`, не изменит `len` оригинального дескриптора в вызывающей функции.
- **nil vs пустой срез.** При JSON-сериализации `nil` → `null`, `[]int{}` → `[]`. Не путать при проектировании API.

**Best practices:**

- Функция `clear(s)` для обнуления всех элементов среза более эффективна, чем ручная итерация, так как под капотом использует `memclr` — оптимизированную инструкцию очистки памяти.
- Пре-аллоцируй срез через `make([]T, 0, cap)`, если заранее известно примерное количество элементов — избегай лишних релокаций.
- Всегда присваивай результат `append` обратно той же переменной: `s = append(s, ...)`.
- Проверяй пустоту через `len(s) == 0`, а не через `s == nil`.
- Для защиты базового массива при передаче в функцию используй трёхиндексный срез `s[a:b:b]` или передавай копию.
- Для полной независимой копии среза: `dst := make([]T, len(src)); copy(dst, src)` или `slices.Clone(src)`.

Когда под-срез «отвязывается» от оригинала?
?
Когда `append` к под-срезу исчерпывает его `cap`. Создаётся новый базовый массив — с этого момента изменения не отражаются в оригинальном срезе.
<!--SR:!2026-03-12,1,202-->

Что такое утечка памяти через срезы?
?
Маленький срез удерживает ссылку на огромный базовый массив. GC не может освободить массив, пока на него есть хотя бы одна ссылка.
<!--SR:!2026-03-12,1,202-->

Как предотвратить утечку памяти через срез?
?
Скопировать нужные данные в новый срез с минимальной ёмкостью через `copy`, чтобы большой массив мог быть очищен GC.
<!--SR:!2026-03-12,1,202-->

В чём разница между nil-срезом и пустым срезом?
?
nil-срез (`var s []int`) при JSON-сериализации → `null`. Пустой срез (`s := []int{}`) → `[]`. Обе имеют `len == 0`, но это разные вещи.
<!--SR:!2026-03-12,1,202-->

Как правильно проверять срез на пустоту?
?
Через `len(s) == 0`, а не `s == nil` — пустой и nil-срез оба дают `len == 0`, но `s == nil` для пустого среза вернёт `false`.
<!--SR:!2026-03-12,1,202-->

Почему результат `append` нужно присваивать обратно той же переменной?
?
Если `append` вызвал релокацию, он возвращает новый дескриптор с новым указателем. Без присваивания (`s = append(s, ...)`) старый дескриптор устареет.
<!--SR:!2026-03-12,1,202-->

Что произойдёт с `len` оригинального среза, если функция делает `append` к принятому по значению срезу?
?
Ничего. `len` оригинального дескриптора в вызывающей функции не изменится — функция работает с копией дескриптора.
<!--SR:!2026-03-12,1,202-->

Где хранится базовый массив среза: в стеке или куче?
?
В стеке, если его размер < 64 КБ. В куче — если больше или если Escape Analysis определит, что данные «уходят» за пределы функции.
<!--SR:!2026-03-12,1,202-->

Где хранится массив: в стеке или куче?
?
В стеке, если размер < 10 МБ. В куче — если больше или при escape analysis.
<!--SR:!2026-03-12,1,190-->

Каково нулевое значение массива и среза?
?
Массив: заполнен нулевыми значениями его типа (например, `0` для `int`). Срез: `nil`.
<!--SR:!2026-03-12,1,202-->

Что делает `make([]int, 0, 10)`?
?
Создаёт срез с `len=0` и `cap=10`. Базовый массив на 10 элементов уже выделен — первые 10 вызовов `append` не вызовут релокации.
<!--SR:!2026-03-12,1,208-->

Может ли длина массива задаваться переменной?::Нет. Длина массива — всегда константа, известная на этапе компиляции. `var n = 5; var a [n]int` — ошибка компиляции.

Что происходит с массивом при росте стека горутины?::Массив может перемещаться в памяти — менять свой адрес. Это особенность работы стека Go.

Чем отличается указатель в nil-срезе от указателя в пустом срезе?::nil-срез: указатель == `nil`. Пустой срез (`[]int{}` / `make([]int, 0)`): указатель указывает на специальный разделяемый адрес для пустых структур (не `nil`).

Как изменился алгоритм роста ёмкости append в Go 1.22?::До Go 1.22: удвоение при `cap < 1024`, затем ~25%. С Go 1.22: порог удвоения снижен до 256; после 256 применяется плавная математическая формула (не просто 25%).

Edge case: какой `cap` будет у `[]int{1,2,3,4,5}`?::Компилятор может выделить `cap = 6` — специфика внутреннего аллокатора для литералов с нечётным числом элементов (≥5).

Что происходит с выражением справа от `range` в Go?::Вычисляется и копируется **один раз** до начала цикла. `append` к тому же срезу внутри цикла не создаёт бесконечный цикл — итерация идёт по первоначальной длине.

Edge case: `range` и переменная значения до Go 1.22?::До Go 1.22: переменная значения (`value` в `for _, value := range`) переиспользовала один адрес памяти. `&value` в разных итерациях указывал на одно место. С Go 1.22 — уникальный экземпляр на каждую итерацию.

В чём разница между `(*[4]int)(s)` и `[4]int(s)`?
?
- `(*[4]int)(s)` — указатель на массив в той же памяти, что и срез. Изменения через массив изменяют срез.
- `[4]int(s)` — копирование: независимый массив, изменения не влияют на срез.

Как преобразовать `[]byte` в `string` без копирования памяти?::Через пакет `unsafe`: `s := unsafe.String(&b[0], len(b))`. Риск: изменение байтов среза изменит строку, нарушая её иммутабельность.

Почему под-срез от слайса указателей вызывает утечку памяти?::Базовый массив удерживается целиком, даже если новый срез покрывает только часть. Элементы за границами нового среза не освобождаются GC. Решение: занулить их через `clear()` или `s[i] = nil` до взятия под-среза.
