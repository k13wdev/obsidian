## Суть и контекст

Конспект покрывает работу с удалёнными репозиториями (`remote`, `fetch`, `pull`, `push`), транспортные протоколы Git, настройку SSH, credential storage для HTTP-доступа, продвинутые инструменты: `stash`, `bisect`, `submodule`, `subtree`, `clean`, `blame`, `grep`, интерактивное индексирование (`git add -i/-p`), упаковку и передачу без сети (`git archive`, `git bundle`), а также email-workflow для публичных проектов (`format-patch`, `am`, `send-email`, `imap-send`).

---

## Ключевые идеи

- Remote-репозитории — версии проекта в сети; локально для них хранится URL и short name (обычно `origin`).
- `git fetch` не сливает автоматически — только скачивает данные; `git pull` = `fetch` + `merge` (или `rebase` с `--rebase`).
- Git поддерживает четыре транспортных протокола: Local, HTTP/HTTPS, SSH, Git (`git://`).
- **Credential storage** — механизм кэширования учётных данных для HTTP: `cache`, `store`, `osxkeychain`, `wincred`.
- **Интерактивное индексирование** (`git add -i`, `git add -p`) позволяет выбирать куски (hunks), а не файлы целиком.
- `git stash` — стек временных сохранений для грязного рабочего каталога.
- `git bisect` — бинарный поиск коммита с багом (может быть автоматизирован через `bisect run`).
- Submodule хранит только ссылку на конкретный коммит внешнего репозитория (режим `160000`).
- `git clean` удаляет untracked файлы; `-n` — «сухой прогон».
- `git archive` упаковывает снимок кода (tar/zip) без истории; `git bundle` — бинарный пакет коммитов и ссылок для передачи без сети.
- Email-workflow: `git format-patch` → `git imap-send`/`git send-email` → `git am` у сопровождающего.

## Определения и термины

**Remote repository** — версия проекта, размещённая в интернете или сети.

**Short name (shortname)** — короткое имя удалённого репозитория (по умолчанию `origin`).

**Refspec** — строка формата `+<src>:<dst>`, правило сопоставления ссылок между локальным и удалённым репозиториями (например, `+refs/heads/*:refs/remotes/origin/*`).

**Dumb protocol** — простой HTTP, работающий через GET-запросы к статическим файлам.

**Smart protocol** — протокол с процессами `upload-pack` (отдача) и `receive-pack` (приём) на сервере; договаривается о передаче только нужных объектов.

**`upload-pack`** — серверный процесс, отдающий данные клиенту.

**`receive-pack`** — серверный процесс, принимающий данные от клиента.

**Credential helper** — внешняя программа, которая отвечает на запросы Git `get`/`store`/`erase` для HTTP-учётных данных.

**Stash** — стек временных сохранений незафиксированных изменений.

**Bisect** — бинарный поиск по истории коммитов для нахождения того, что внёс баг.

**Submodule** — способ держать один Git-репозиторий как подкаталог другого; хранится только ссылка на конкретный коммит внешнего проекта (режим `160000`).

**Subtree** — альтернатива submodule: влить содержимое другого репозитория в подкаталог, сохранив возможность подтягивать обновления, без механизма submodule.

**Archive** — tar/zip-снимок версии кода без истории (`git archive`).

**Bundle** — бинарный файл с объектами и ссылками Git; может использоваться как URL для clone/fetch (`git bundle`).

## Как устроено

### Работа с remote-репозиториями

- `git remote -v` — список удалённых серверов (short name + URL для чтения и записи).
- `git remote add [name] [url]` — добавляет новый remote.
- `git remote show [name]` — подробная информация: URL, отслеживаемые ветки, какие ветки сольются при `git pull`.
- `git remote rename [old] [new]` — переименовывает remote (и все связанные remote-tracking ветки).
- `git remote rm [name]` — удаляет ссылку на remote.

### Синхронизация

- `git fetch [remote]` — скачивает все данные, которых ещё нет; **не** объединяет автоматически.
- `git pull` — `fetch` + `merge` текущей ветки с её upstream.
- `git pull --rebase` — интеграция через rebase вместо merge (линейная история, без merge-коммитов).
- `git push [remote] [branch]` — отправляет коммиты на remote.
  - `-u` / `--set-upstream` — связывает локальную ветку с удалённой (tracking).
  - `--force` (`-f`) — принудительная перезапись истории на сервере. **Опасно**: может уничтожить работу коллег.
  - `--force-with-lease` — **в предоставленных источниках не упоминается**.
- **Удаление удалённой ветки**: `git push [remote] --delete [branch]` — удаляет указатель на сервере (данные могут храниться до сборки мусора).

### Транспортные протоколы

| Протокол | Описание | Плюсы | Минусы |
|----------|----------|-------|--------|
| **Local** | Репозиторий в другой папке или NFS | Простота; использование разрешений ФС | Трудно с внешним доступом; медленно через сетевые диски; нет защиты от случайной порчи |
| **HTTP/HTTPS** | Smart HTTP с версии 1.6.6, работает как SSH, но через стандартные порты | Один URL для всех видов доступа; проходит корпоративные firewalls; простая аутентификация логином/паролем | Иногда сложнее настроить на сервере; ввод пароля (решается кэшем учётных данных) |
| **SSH** | Самый распространённый для записи при самохостинге | Безопасность (шифрование + аутентификация); высокая эффективность | Нет анонимного доступа на чтение — всем нужны ключи |
| **Git (`git://`)** | Специальный демон на порту 9418 | Самый быстрый | Нет аутентификации (обычно public read-only); сложность настройки; нужен открытый порт в firewall |

**Настройка SSH-ключей (в общих чертах)**: пользователь генерирует пару ключей (открытый и закрытый); открытый ключ (`id_rsa.pub`) отправляется администратору, который добавляет его в `~/.ssh/authorized_keys` специального пользователя (обычно `git`) на сервере.

### Credential Storage (HTTP)

Механизм кэширования учётных данных для HTTP/HTTPS.

Установка: `git config --global credential.helper <mode>`.

**Режимы:**
- **`cache`** — хранит пароли в памяти временно (по умолчанию 15 минут, настраивается через `--timeout`).
- **`store`** — сохраняет учётные данные в **открытом виде** в текстовом файле на диске (по умолчанию `~/.git-credentials`).
- **`osxkeychain`** — использует системную связку ключей macOS (шифрованное хранилище на диске).
- **`wincred`** — использует Windows Credential Store для безопасного хранения.

**Внутренняя механика**: команда `git credential` обрабатывает три действия:
- `get` — запрос данных;
- `store` — сохранение;
- `erase` — удаление.

### Интерактивное индексирование

Позволяет гибко формировать коммит, выбирая отдельные файлы или их куски.

**`git add -i`** — текстовое меню:
- `status` — статус;
- `update` — индексация изменённых файлов;
- `revert` — исключение из индекса;
- `add untracked` — добавление новых файлов;
- `patch` — поблочная работа (равносильно `git add -p`);
- `diff` — показать дельту.

**`git add -p`** (patch mode) — построчный разбор изменений (hunks):
- `y` — индексировать кусок;
- `n` — не индексировать;
- `s` — разбить на более мелкие куски;
- `e` — редактировать кусок вручную.

Флаг `--patch` также поддерживают `git reset`, `git checkout`, `git stash`.

### Git Stash

Временное сохранение изменений в «грязном» рабочем каталоге.

- `git stash` (или `git stash push`) — сохранить в стек.
- `git stash list` — список сохранённых состояний.
- `git stash apply [stash@{n}]` — применить, но не удалить из стека.
- `git stash pop` — применить последнее и удалить из стека.
- `git stash drop [name]` — удалить запись из стека.
- `git stash branch [name]` — создаёт новую ветку, переключается на неё и применяет stash там (удобно при конфликтах в исходной ветке).
- Флаги:
  - `-u` / `--include-untracked` — сохранять также untracked файлы.
  - `--keep-index` — оставить в рабочем каталоге то, что уже в индексе.
- **Именованный stash**: `git stash save "message"` — добавить описание.

### Git Bisect

Бинарный поиск коммита, внёсшего баг.

- `git bisect start` — начало процесса.
- `git bisect bad` — пометить текущий коммит как сломанный.
- `git bisect good [commit]` — последний известный рабочий коммит.
- `git bisect reset` — возврат к HEAD до начала бисекции.
- `git bisect run [script]` — автоматизация: запускает скрипт на каждом шаге. Код возврата `0` — коммит хороший, иначе — плохой.

### Submodules

- `git submodule add [url]` — добавляет внешний проект как подкаталог.
- `git clone --recurse-submodules` — клонирует основной проект вместе со всеми submodule.
- `git submodule update --init --recursive` — инициализация и обновление submodule в уже клонированном проекте.
- Git хранит только ссылку на конкретный коммит подмодуля (режим `160000`), а не его содержимое.
- **Submodule merge**: Git не выполняет автоматическое слияние, если указатели на коммиты в подмодулях разошлись; требуется ручное разрешение конфликта в директории подмодуля.

### Subtree

Альтернатива submodule: влить содержимое другого репозитория в подкаталог текущего, сохраняя возможность подтягивать обновления или отправлять правки обратно, без механизма submodule.

### Clean, blame, grep

- `git clean -n` — «сухой прогон», показывает, что будет удалено.
- `git clean -f` — принудительное удаление untracked-файлов.
- `git clean -d` — удаление также пустых директорий.
- `git blame -L [start],[end] [file]` — кто и когда изменил каждую строку в диапазоне.
- `git grep` — быстрый поиск строк/regex в файлах рабочего каталога или любых коммитах истории.

> `git worktree` в предоставленных выдержках **не описывается**.

### `git archive` — упаковка снимка

- Снимок версии кода без истории Git (tar/zip).
- `--prefix=<dir>/` — добавляет вложенную папку внутри архива.
- Атрибут `export-ignore` в `.gitattributes` исключает файлы/папки из архива.
- Атрибут `export-subst` подставляет метаданные (например, SHA-1) в файлы внутри архива.

### `git bundle` — передача без сети

Упаковывает объекты и ссылки Git в один бинарный файл.

- `git bundle create <file> <range>` — создать пакет (например, `git bundle create repo.bundle HEAD master`).
- Полученный файл используется как URL для `git clone` или `git fetch`.
- `git bundle verify <file>` — проверяет целостность и наличие необходимых родительских коммитов в целевом репозитории.

### Email workflow — вклад через почту

Для открытых проектов без GitHub/GitLab доступа.

- `git format-patch -M origin/master` — генерирует mbox-файлы (один на коммит) с патчами и метаданными.
- `git imap-send` — отправляет набор патчей в папку черновиков через IMAP.
- `git send-email` — отправляет патчи напрямую по SMTP.
- Сопровождающий применяет патчи через `git am <file>.patch` (корректно сохраняет авторство, метаданные и сообщения).

## Детали и нюансы

- `git fetch` скачивает данные, но локальные ветки не двигает — нужно вручную `merge` или `rebase`.
- `git push --force` уничтожает чужие коммиты на сервере, если на них уже строилась работа.
- После `git push --delete` данные могут ещё храниться на сервере до следующей сборки мусора.
- Smart HTTP появился с Git 1.6.6.
- Submodule «отстаёт», если вы не обновили его в основном репозитории после изменений во внешнем.
- `credential.helper store` — **небезопасно в shared-системах**: пароли лежат в открытом виде в `~/.git-credentials`.
- `credential.helper cache` по умолчанию хранит 15 минут, время настраивается.
- `git am` корректно работает с патчами от `format-patch` — сохраняется автор, дата, сообщение.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| `git pull` | Удобно: одна команда | Может создать merge-коммит, если истории разошлись | Быстрая интеграция |
| `git pull --rebase` | Линейная история | Переписывает локальные коммиты | Личная ветка, linear log |
| `git push --force` | Перезапись истории | Уничтожает работу коллег | Только в личной ветке |
| Submodule | Чистое разделение репозиториев; контроль версии внешнего проекта | Сложность обслуживания; отдельный цикл обновления | Разные релизные циклы, чёткие границы |
| Subtree | Содержимое в репо; не нужны дополнительные команды при clone | История мешается; сложнее тянуть обновления | Разделяемый код, единый релизный цикл |
| `credential.helper cache` | Ничего не пишется на диск | Ограниченное время | Короткие сессии |
| `credential.helper store` | Не требует ввода пароля | Пароли в открытом виде | Личные машины с полным доверием |
| `credential.helper osxkeychain/wincred` | Шифрование системной связкой ключей | ОС-специфично | macOS / Windows |
| `git add -p` | Атомарные коммиты, «одна идея — один commit» | Дольше, требует дисциплины | Полировка перед review |
| `git archive` | Снимок без истории; можно отдавать наружу | Без истории — нет возможности обновить | Релизные tarball'ы |
| `git bundle` | Передача через любой носитель; полноценный репо | Бинарный файл, нужно выбирать правильный range | Offline-передача, airgap |

## Примеры

### Добавление и просмотр remote

```bash
git remote add upstream https://github.com/original/repo.git
git remote -v
git remote show upstream
```

### Push новой ветки с tracking

```bash
git push -u origin feature/login
```

### Удаление удалённой ветки

```bash
git push origin --delete feature/old
```

### Настройка credential helper

```bash
git config --global credential.helper cache
git config --global credential.helper 'cache --timeout=3600'
git config --global credential.helper osxkeychain      # macOS
git config --global credential.helper wincred          # Windows
git config --global credential.helper store            # ~/.git-credentials
```

### Интерактивное добавление

```bash
git add -i
# меню: status / update / revert / add untracked / patch / diff / quit
git add -p src/auth.go
# для каждого hunk: y/n/s/e
```

### Stash с untracked файлами

```bash
git stash push -u -m "WIP: refactor auth"
git stash list
git stash pop
```

### Автоматический bisect

```bash
git bisect start
git bisect bad
git bisect good v1.0
git bisect run ./run_tests.sh
git bisect reset
```

### Добавление submodule

```bash
git submodule add https://github.com/lib/dep.git vendor/dep
git commit -m "Add dep submodule"
# в другой клонированной копии:
git submodule update --init --recursive
```

### Чистка рабочего каталога

```bash
git clean -n           # что будет удалено
git clean -fd          # удалить untracked + пустые директории
```

### Архив без истории

```bash
git archive --format=tar --prefix=project-1.4/ HEAD | gzip > project-1.4.tar.gz
```

### Bundle для передачи без сети

```bash
git bundle create repo.bundle HEAD master
# на другой машине:
git bundle verify repo.bundle
git clone repo.bundle my-clone
```

### Email-вклад в open source проект

```bash
# у контрибьютора
git format-patch -M origin/master
git send-email 0001-fix-bug.patch
# или
git imap-send < 0001-fix-bug.patch

# у сопровождающего
git am 0001-fix-bug.patch
```

## Важно / подводные камни / best practices

- `git fetch` — безопасная привычка: скачать и посмотреть, что произошло, прежде чем мержить.
- `--force` — только в личных ветках; в общих предпочитайте `revert`.
- Для submodule всегда используйте `--recurse-submodules` при clone.
- `git clean -n` перед `-f` — чтобы не удалить случайно важное.
- SSH не поддерживает анонимный read-access — для публичного чтения используйте HTTPS или Git-протокол.
- При именованном stash (`git stash save "message"`) проще найти нужное состояние в `stash list`.
- **Никогда** не используйте `credential.helper store` на shared-машинах — пароли в plain text.
- `git add -p` помогает делать атомарные коммиты «одна идея — один commit» даже если изменения накопились в разных местах одного файла.
- Для бэкапа/airgap-передачи Git `bundle` предпочтительнее `archive` — сохраняет историю.

3. SSH — безопасность, эффективность; нужны ключи
4. Git (`git://`) — самый быстрый, порт 9418, без аутентификации

Плюсы SSH-протокола::Безопасность (шифрование + аутентификация); высокая эффективность передачи.

Минус SSH-протокола::Нет анонимного доступа на чтение — всем пользователям нужны ключи.

Плюсы HTTP/HTTPS::Один URL для всех видов доступа; проходит корпоративные firewalls; простая аутентификация.

Плюс Git-протокола::Самый быстрый.

Минус Git-протокола::Нет аутентификации; обычно только public read-only.

Настройка SSH-ключей (схема)
?
1. Пользователь генерирует пару ключей (открытый и закрытый)
2. Открытый ключ (`id_rsa.pub`) отправляется администратору
3. Администратор добавляет его в `~/.ssh/authorized_keys` пользователя `git` на сервере

Режимы `credential.helper`
?
1. cache — в памяти, по умолчанию 15 минут
2. store — в `~/.git-credentials`, в открытом виде
3. osxkeychain — связка ключей macOS
4. wincred — Windows Credential Store

Edge case: `credential.helper store` на shared-машине?::Опасно — пароли лежат в открытом виде в `~/.git-credentials`.

Действия `git credential`:: ==`get` / `store` / `erase`==

Что делает `git add -i`?::Запускает интерактивное меню: status/update/revert/add untracked/patch/diff.

Что делает `git add -p`?::Patch mode — построчный разбор изменений по hunks: y/n/s/e.

Ответы в `git add -p`
?
y — индексировать кусок
n — пропустить
s — разбить на более мелкие куски
e — редактировать вручную

Что делает `git stash`?::Сохраняет незафиксированные изменения в стек и очищает рабочий каталог.

Разница `git stash apply` и `git stash pop`
?
`apply` — применяет, но оставляет в стеке.
`pop` — применяет и удаляет из стека.

Что делает `git stash branch [name]`?::Создаёт новую ветку, переключается на неё и применяет stash там; удобно при конфликтах.

Флаг `git stash -u`:: ==включить в stash также untracked файлы==

Флаг `git stash --keep-index`:: ==оставить в рабочем каталоге то, что уже в индексе==

Именованный stash::`git stash save "message"` — добавить описание.

Bisect: основные команды
?
1. `git bisect start`
2. `git bisect bad` / `git bisect good [commit]`
3. Git автоматически выбирает средний коммит и переключается
4. `git bisect reset` — возврат в исходное состояние

Что делает `git bisect run [script]`?::Автоматически запускает скрипт на каждом шаге: код 0 — коммит хороший, иначе — плохой.

Что делает `git submodule add [url]`?::Добавляет внешний проект как подкаталог текущего репозитория.

Клонирование с submodule::`git clone --recurse-submodules [url]`

Инициализация submodule в уже клонированном репо::`git submodule update --init --recursive`

Что Git хранит для submodule?::Только ссылку на конкретный коммит внешнего репозитория (режим `160000`), а не его содержимое.

Edge case: submodule merge с разошедшимися указателями?::Автоматического слияния нет — требуется ручное разрешение конфликта в директории подмодуля.

Что делает `git clean -n`?::Сухой прогон: показывает, какие untracked-файлы будут удалены.

Что делает `git clean -fd`?::Удаляет untracked-файлы и пустые директории.

Что делает `git blame -L 10,20 file.c`?::Показывает, кто и когда изменил каждую строку в диапазоне 10–20.

Что делает `git grep`?::Быстрый поиск строк/regex в файлах рабочего каталога или любых коммитах истории.

Атрибут `.gitattributes` `export-ignore`:: ==исключить файлы/папки из `git archive`==

Атрибут `.gitattributes` `export-subst`:: ==подставлять метаданные (SHA-1 и т.п.) в файлы архива==

Что делает `git bundle create repo.bundle HEAD master`?::Упаковывает коммиты в бинарный файл, который можно использовать как URL для clone/fetch.

Что делает `git bundle verify`?::Проверяет целостность файла и наличие необходимых родительских коммитов в целевом репозитории.

Email workflow — ключевые команды
?
1. `git format-patch -M origin/master` — сгенерировать mbox-файлы с патчами
2. `git send-email` или `git imap-send` — отправить
3. У сопровождающего: `git am <patch>` — применить, сохраняя автора и метаданные

Что делает `git format-patch`?::Генерирует mbox-файлы с патчами и метаданными коммитов для отправки по email.

Что делает `git am`?::Применяет патч из mbox-файла, сохраняя автора, дату и сообщение коммита.

Edge case: `git push --delete` и сборка мусора?::Указатель удалён, но данные могут храниться до следующей сборки мусора на сервере.
