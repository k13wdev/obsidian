## Суть и контекст

Git — контентно-адресуемая файловая система с интерфейсом VCS поверх неё. Понимание внутренностей помогает объяснять, что именно происходит при коммите, merge, push и как Git гарантирует целостность данных. Конспект покрывает: директорию `.git`, четыре типа объектов, plumbing-команды, хранение (loose/packfiles), ссылки (refs), транспортные протоколы, `git replace`, `filter-branch`, обслуживание базы (`gc`, `fsck`), переменные окружения, распределённые workflow-модели, GitHub и GitLab (включая Pull/Merge Requests, API, webhooks, уровни доступа, Markdown), а также Git hooks — клиентские и серверные.

---

## Ключевые идеи

- В основе Git — хранилище «ключ-значение», где ключом является SHA-1 от содержимого объекта.
- Четыре типа объектов: **blob**, **tree**, **commit**, **tag**.
- Директория `.git` — место, где Git хранит всё: конфигурацию, объекты, ссылки, hooks.
- Ссылки (`refs/`) — это просто файлы с SHA-1 коммитов.
- Объекты хранятся как **loose** или упакованы в **packfiles** с дельта-сжатием.
- `git gc` запускается автоматически при ~7000 loose-объектов или >50 pack-файлов.
- Транспортные протоколы: **dumb** (тупой HTTP) и **smart** (через `upload-pack` / `receive-pack`).
- **`filter-branch`** переписывает историю (удаление файла из всей истории, изменение email, выделение поддиректории).
- **`git replace`** подставляет один объект вместо другого без переписывания истории.
- Workflow-модели: centralized, integration-manager, dictator-and-lieutenants.
- **GitHub**: PR, Review, @mentions, #issue-refs, Markdown, webhooks, REST API, Commit Status API.
- **GitLab**: самохостный полнофункциональный Git-сервер, Namespace, Groups, уровни доступа Guest/Reporter/Developer/Master/Owner, Merge Requests.
- Hooks — скрипты в `.git/hooks/`; бывают клиентские (легко обойти через `--no-verify`) и серверные (надёжны для policy).
- SHA-1 collision: если совпадение — Git проигнорирует запись нового объекта и всегда отдаст первый.

## Определения и термины

**Content-addressable filesystem** — файловая система, в которой ключом к содержимому служит хеш этого содержимого.

**SHA-1** — 40-символьная строка, вычисляемая на основе содержимого объекта и заголовка.

**Bare repository (голый репозиторий)** — репозиторий Git без рабочей директории; содержит только содержимое папки `.git`. Используется как серверный.

**Blob** — объект, хранящий только содержимое файла без метаданных.

**Tree** — объект, соответствующий UNIX-директории; хранит имена файлов и ссылки на blob или другие деревья вместе с режимами доступа.

**Commit** — объект, содержащий ссылку на корневое дерево снимка, автора, коммиттера, дату, сообщение и ссылки на родительские коммиты.

**Tag (object)** — объект, похожий на commit, содержащий информацию о теггере, дате и ссылающийся обычно на коммит.

**Loose object** — объект, хранимый в одном файле на диске.

**Packfile** — упакованный файл с набором объектов и дельта-сжатием.

**Dangling object** — объект, не достижимый ни из одной ссылки в репозитории.

**Mark** — идентификатор коммита, используемый в процессе `fast-import` для связывания объектов.

**Dumb protocol** — простой HTTP через GET-запросы к статическим файлам.

**Smart protocol** — протокол с `upload-pack` (отдача) и `receive-pack` (приём); интеллектуальное согласование передачи только нужных объектов.

**Refspec** — строка формата `+<src>:<dst>`, правило сопоставления ссылок между локальным и удалённым репозиториями (например, `+refs/heads/*:refs/remotes/origin/*`).

**Hook** — скрипт, запускаемый Git при определённом событии.

**Topic (feature) branch** — короткоживущая ветка для конкретной задачи.

**Long-running branch** — постоянная ветка (например, `master`, `develop`).

**Pull Request (PR)** — механизм GitHub для предложения слияния форка/ветки в канонический репозиторий; реализация integration-manager workflow.

**Merge Request (MR)** — аналог PR в GitLab.

**Namespace** — пространство имён в GitLab: пользователь или группа (`http://server/username/project`).

**Blessed repository** — канонический репозиторий сопровождающего в integration-manager workflow.

## Как устроено

### Директория `.git`

- `HEAD` — файл, указывающий на текущую ветку (символьная ссылка).
- `config` — настройки репозитория.
- `description` — используется только GitWeb, не важен.
- `hooks/` — сценарии перехвата.
- `info/` — файл исключений для шаблонов, которые не должны отслеживаться и не добавлены в `.gitignore`.
- `objects/` — база данных объектов.
- `refs/` — указатели на объекты коммитов (ветки, теги, remote-tracking).
- `index` — информация об области подготовленных изменений.

### Объекты Git

Все объекты хешируются SHA-1 (40 символов) от содержимого + заголовка.

**Blob** — содержимое файла без метаданных.

**Tree** — соответствует директории:
- имена файлов + ссылки на blob / поддеревья;
- режимы доступа: `100644` (обычный файл), `100755` (исполняемый), `120000` (символическая ссылка), `160000` (submodule).

**Commit** — ссылка на корневое tree + автор + коммиттер + дата + сообщение + ссылки на родителей.

**Tag** (annotated) — объект с теггером, датой, сообщением, ссылающийся обычно на коммит.

### Низкоуровневые (plumbing) команды

- `git hash-object -w --stdin` — вычислить SHA-1 и сохранить объект в базу.
- `git cat-file -p [hash]` — показать содержимое; `-t` — показать тип.
- `git update-index --add --cacheinfo [mode] [hash] [file]` — искусственно добавить файл в индекс.
- `git write-tree` — записать состояние индекса в объект дерева.
- `git read-tree --prefix=[dir] [hash]` — считать дерево в индекс, позволяя создавать поддиректории.
- `git commit-tree [tree_hash] -p [parent_hash]` — создать объект коммита на основе дерева.
- `git update-ref` — безопасно обновлять указатели (ветки).
- `git symbolic-ref` — прочитать/изменить HEAD как символическую ссылку.

### Ссылки (refs)

- Ветки — файлы в `refs/heads/[name]` с SHA-1 последнего коммита.
- Теги — в `refs/tags/`.
- Remote-tracking — в `refs/remotes/`.
- HEAD — символьная ссылка на текущую ветку.
- После упаковки — `.git/packed-refs` содержит ссылки в одном файле.

### Хранение: loose vs packfiles

- Изначально Git хранит объекты как **loose** — один файл на объект.
- `git gc` упаковывает их в **packfiles** с дельта-сжатием: одна версия целиком, остальные — как разница.

### Транспортные протоколы

- **Dumb protocol** — простой HTTP через GET к статическим файлам.
- **Smart protocol** — использует `upload-pack` (сервер → клиент) и `receive-pack` (клиент → сервер), интеллектуально согласовывая передачу только нужных объектов.

**Refspec**: `+refs/heads/*:refs/remotes/origin/*` — отображение веток сервера в локальные remote-tracking ветки.

### Обслуживание (maintenance)

- `git gc` — упаковывает объекты в дельта-сжатые pack-файлы и ссылки в `.git/packed-refs`. Автоматически запускается при **~7000 loose-объектов** или **>50 pack-файлов**.
- `git fsck --full` — проверяет целостность базы, находит dangling-объекты (не достижимые ни из одной ссылки).
- `git prune` — удаляет объекты, на которые нет ссылок.
- `git reflog` — журнал перемещений HEAD, хранится локально; позволяет восстанавливать потерянные данные.

### `filter-branch` — массовая перезапись истории

Переписывает всю историю; **каждое применение меняет SHA-1** всех затронутых коммитов.

- **Удаление файла из всей истории:**
  - `git filter-branch --tree-filter 'rm -f passwords.txt' HEAD` — медленно (checkout каждого коммита);
  - `git filter-branch --index-filter 'git rm --cached --ignore-unmatch filename' HEAD` — быстрее, работает с индексом.
- **Глобальное изменение email**: `--commit-filter` с проверкой `$GIT_AUTHOR_EMAIL` и вызовом `git commit-tree "$@"`.
- **Выделение поддиректории в корень**: `git filter-branch --subdirectory-filter trunk HEAD` — делает содержимое папки новым корнем, удаляя коммиты, её не затрагивающие.
- **Очистка после работы**: Git сохраняет старые ссылки в `refs/original/`; для полного удаления — удалить эти ссылки, reflog и запустить `git gc`.

### `git replace` — подмена объектов

`git replace [A] [B]` — при обращении к объекту `A` Git подставляет содержимое объекта `B`. Используется для **объединения историй** (например, склейка старой и новой истории проекта) без реального переписывания SHA-1.

### Переменные окружения

- `GIT_DIR` — путь к папке `.git`. Если не задан, Git ищет её вверх по дереву до корня.
- `GIT_WORK_TREE` — путь к корню рабочей директории.
- `GIT_INDEX_FILE` — путь к файлу индекса.
- `GIT_AUTHOR_NAME` / `GIT_COMMITTER_NAME` — переопределяют имя автора и коммиттера.
- `GIT_AUTHOR_EMAIL` — используется, например, в `filter-branch --commit-filter`.
- `GIT_TRACE` — включает общую трассировку выполнения команд.
- `GIT_ASKPASS` — переопределяет программу для запроса пароля у пользователя.

### Распределённые Workflow-модели

#### Centralized (Централизованный)

- Один центральный узел (hub) принимает код; все разработчики — потребители.
- При одновременных изменениях второй не сможет запушить («non-fast-forward»), пока не выполнит `fetch` и `merge` локально.

#### Integration-Manager

- Основа GitHub / GitLab.
- Каждый разработчик имеет запись в свой публичный репозиторий и чтение во все остальные.
- **Схема:**
  1. Сопровождающий пушит в «blessed» (канонический) репозиторий.
  2. Контрибьютор делает fork и вносит изменения.
  3. Контрибьютор пушит в свой форк.
  4. Сопровождающий добавляет форк как remote, тестирует, сливает и пушит в blessed.
- Pull Request — концептуально реализация этой модели.

#### Dictator and Lieutenants

- Иерархическая модель для сверхкрупных проектов (ядро Linux).
- **Схема:**
  1. Разработчики rebase работу на master «диктатора».
  2. «Лейтенанты» сливают ветки разработчиков.
  3. «Диктатор» сливает ветки лейтенантов и пушит в эталонный репозиторий.

### Contributing — вклад в проекты

- **Public Small Team** — закрытые проекты, все участники имеют write-доступ; слияния происходят на стороне клиента перед отправкой на сервер.
- **Private Managed Team** — работа через PR внутри организации с review.
- **Forked Public Project** — форк + PR (см. GitHub Flow).
- **Email Workflow** — `format-patch` → `send-email`/`imap-send` → `am` (см. [[03. Удалённые репозитории и продвинутые инструменты]]).

### Типы веток

- **Long-running branches** — постоянные: `master` (стабильная), `develop` (интеграция функций).
- **Topic (feature) branches** — короткоживущие, под задачу; удаляются после слияния.

### GitHub

#### Аккаунт и безопасность

- Бесплатный аккаунт — неограниченное число публичных проектов.
- GitHub сопоставляет коммиты с пользователем **по email** — поэтому email нужно подтвердить.
- Рекомендуется 2FA (мобильное приложение или SMS).
- SSH-ключи: содержимое `~/.ssh/id_rsa.pub` добавляется в «Settings» → «SSH keys».

#### Совместная работа

- **Fork** — если нет прав на запись, делается копия проекта в своём namespace.
- **Collaborators** — прямой write-доступ другим пользователям через «Settings» → «Collaborators».
- **Organizations + Teams** — управление группами людей и проектами со сложными правами.

#### GitHub Flow и Pull Request

Рабочий процесс на ветках и PR:

1. Создать **topic branch** от master.
2. Внести изменения и коммиты.
3. Push ветки в свой проект на GitHub.
4. Открыть Pull Request.
5. Обсуждение и возможная доработка (новые коммиты в ту же ветку **автоматически обновляют PR**).
6. Merge или закрытие PR владельцем.

**Особенности PR:**
- **Review**: комментарии к конкретным строкам кода.
- **Merge**: кнопка «Merge» на сайте делает **non-fast-forward** слияние — создаётся merge commit, даже если возможен fast-forward.
- **`@username`** — уведомляет пользователя и подписывает его на обновления.
- **`#номер`** — ссылка на PR или Issue.

#### Markdown в PR и emoji

GitHub Flavored Markdown:

- **Task Lists**: `- [X] Сделано`
- **Code Snippets**: тройные обратные кавычки с указанием языка для подсветки.
- **Emoji**: `:name:` (автодополнение при вводе `:`).

#### Webhooks и API

- **Webhooks**: уведомление внешних сервисов о событиях (push и т. п.). GitHub отправляет HTTP POST с JSON.
- **REST API**:
  - Лимит: 60 запросов/час без аутентификации, **до 5000** — с Personal Access Token.
- **Примеры эндпоинтов:**
  - `GET /users/schacon` — информация о пользователе;
  - `POST /repos/<user>/<repo>/issues/<num>/comments` — добавить комментарий к Issue;
  - **Commit Status API** — сервисы CI/CD ставят «галочки»/«крестики» (success/failure) рядом с коммитами.

> **GitHub Actions** в данном издании Pro Git (2014 г.) не упоминаются.

### GitLab

Описывается как полнофункциональный Git-сервер с веб-интерфейсом и БД, ориентированный на **самохостинг** внутри файрвола.

#### Архитектура и установка

- Веб-приложение с базой данных (обычно PostgreSQL).
- Установка: образ ВМ (Bitnami), пакеты (RPM/DEB), скрипты (Chef).
- По умолчанию: админ `admin@local.host`, пароль `5iveL!fe`.
- Большинство административных задач — через браузер («Admin area»), без правки конфигов и SSH.
- Включает встроенные вики-страницы и системы обслуживания (issue tracking и т. д.).

#### Namespace, проекты и права доступа

- **Namespace**: `http://server/username/project` — пространство имён пользователя или группы.
- **Groups** — объединение проектов; пользователям назначаются уровни.
- **Уровни доступа**: Guest (только задачи и чат) → Reporter → Developer → Master → Owner (полный контроль).
- **Видимость проекта**:
  - **Private** — доступ по приглашению;
  - **Internal** — любой авторизованный пользователь;
  - **Public** — виден всем.

#### Merge Requests и хуки

- **MR** — аналог Pull Request: построчный review, назначение ответственных, milestones.
- Пользователи с правами **Developer** и выше могут пушить напрямую; без прав — fork + MR.
- **Хуки**: на уровне проекта и всей системы. При наступлении события GitLab делает HTTP POST с описанием в JSON.

### Признаки хорошего коммита

- Логически разделённые изменения (один коммит — одна задача).
- Отсутствие ошибок пробельных символов (проверка `git diff --check`).
- **Формат сообщения**:
  - краткий заголовок до 50 символов;
  - пустая строка;
  - детальное описание до 72 символов в строке, с мотивацией изменений;
  - повелительное наклонение: «Add tests», а не «I added tests».

### Git Hooks

Хуки — скрипты в `.git/hooks/`, поставляются с расширением `.sample`. Для активации: убрать `.sample` и сделать исполняемыми (`chmod +x`).

**Клиентские хуки:**
- **`pre-commit`** — перед вводом сообщения; проверка кода (линтеры, тесты). Пропуск: `--no-verify`.
- **`prepare-commit-msg`** — динамическое редактирование сообщения (вставка ID тикета из шаблона).
- **`commit-msg`** — проверка соответствия сообщения формату.
- **`post-commit`** — после завершения коммита; уведомления.
- **`pre-rebase`** — можно запретить rebase коммитов, уже отправленных на сервер.
- **`post-checkout`** / **`post-merge`** — настройка рабочего окружения или восстановление данных, которые Git не отслеживает (например, прав доступа).
- **`pre-push`** — выполняется при `git push`; может отклонить пуш по политике.

**Серверные хуки (enforce policy):**
- **`pre-receive`** — принимает список обновляемых ссылок из stdin; если скрипт завершается с ошибкой, весь push отклоняется. Для ACL и защиты от force-push.
- **`update`** — аналогичен `pre-receive`, но запускается **отдельно для каждой ветки**; позволяет отклонять изменения в конкретных ветках.
- **`post-receive`** — после успешного приёма; рассылки, обновления тикетов, запуск CI.

## Детали и нюансы

- Git — именно **content-addressable** FS: один и тот же контент = один и тот же SHA-1, что и обеспечивает дедупликацию.
- Tree хранит **режимы доступа** — это важно для прав на исполняемые файлы и submodules (`160000`).
- Commit ссылается на **корневое tree**, а не на список файлов напрямую.
- Packfile дельта-сжатие: хранится одна полная версия, остальные — дельта от неё.
- `reflog` не передаётся при clone — локальный журнал, живёт несколько месяцев.
- `git gc` автоматически: порог **~7000 loose-объектов** или **>50 pack-файлов**.
- Pull Request — концептуально реализация integration-manager workflow.
- GitHub кнопка «Merge» создаёт **merge commit** даже при возможности fast-forward — это отличается от дефолтного поведения CLI-merge.
- GitLab делает ставку на self-hosted внутри корпоративного периметра.
- **SHA-1 collision**: если объект с идентичным хешем уже в базе, Git проигнорирует запись нового и всегда вернёт первый записанный.
- `filter-branch` оставляет старые ссылки в `refs/original/` — для реального удаления данных нужно их стереть, плюс reflog и `gc`.
- `git replace` не переписывает SHA-1 — это локальная подмена при чтении; удобно для склейки историй без поломки ссылок.

## Сравнения и trade-offs

| Подход/Workflow | Плюсы | Минусы | Когда использовать |
|------------------|-------|--------|--------------------|
| Centralized | Простота ментальной модели | Единая точка отказа; нет изоляции | Небольшие закрытые команды |
| Integration-Manager | Чёткое разделение ролей; основа GitHub PR | Нужен интегратор | Open source, команды с review |
| Dictator & Lieutenants | Масштабирование на огромные проекты | Сложная иерархия | Проекты масштаба ядра Linux |
| Long-running branches | Стабильные «уровни зрелости» кода | Риск сложных слияний | Релизная разработка |
| Topic branches | Изоляция задач, чистая история | Требует дисциплины merge/delete | Практически везде |
| Loose objects | Простота создания/доступа | Много мелких файлов, неэффективно | Сразу после коммита |
| Packfiles | Компактное хранение; быстрая передача | Нужно вовремя запускать `gc` | После набора истории |
| GitHub | Массовая экосистема; PR, Issues, API, Actions | Проприетарный сервис, публичный по умолчанию | Open source, стартапы |
| GitLab | Self-hosted, всё через браузер, MR | Тяжеловесный стек (БД, веб-приложение) | Корпоративные инсталляции за файрволом |
| `filter-branch` | Массовая перезапись истории | Меняет SHA-1 всех коммитов; медленно | Удаление секретов, split репозитория |
| `git replace` | Без переписывания истории; склейка | Локально; не синхронизируется естественно | Объединение старой и новой истории |
| Клиентские хуки | Быстрая обратная связь разработчику | Легко обходятся `--no-verify` | Удобство, не policy |
| Серверные хуки | Нельзя обойти | Требуют серверной настройки | ACL, защита от force-push, CI-триггеры |

## Примеры

### Ручное создание объектов (plumbing)

```bash
echo 'hello' | git hash-object -w --stdin
# → SHA-1 сохранённого blob
git cat-file -p <hash>     # hello
git cat-file -t <hash>     # blob
```

### Создание коммита вручную

```bash
tree=$(git write-tree)
commit=$(echo "msg" | git commit-tree $tree -p $parent)
git update-ref refs/heads/master $commit
```

### Поиск висячих объектов и очистка

```bash
git fsck --full
git prune
git gc
```

### Удаление файла из всей истории

```bash
git filter-branch --index-filter \
  'git rm --cached --ignore-unmatch passwords.txt' HEAD
# очистка бэкап-ссылок
rm -rf .git/refs/original
git reflog expire --expire=now --all
git gc --prune=now
```

### Выделение поддиректории в корень

```bash
git filter-branch --subdirectory-filter trunk HEAD
```

### Подмена объекта через `git replace`

```bash
git replace old-commit-sha new-commit-sha
```

### Активация клиентского хука

```bash
cd .git/hooks
mv pre-commit.sample pre-commit
chmod +x pre-commit
```

### Пример серверной политики через `update`

Скрипт `update` может проверять каждое сообщение коммита на соответствие регулярному выражению (например, наличие `ref: 1234`) и отклонять push при несовпадении.

### Пример user-based ACL

Hook считывает файл ACL и разрешает push в определённые директории только конкретным пользователям. В Pro Git приведён пример реализации на Ruby.

### Использование GitHub API

```bash
# Информация о пользователе
curl https://api.github.com/users/schacon

# Комментарий к issue (требует токен)
curl -X POST -H "Authorization: token $TOKEN" \
  -d '{"body":"thanks!"}' \
  https://api.github.com/repos/user/repo/issues/42/comments
```

  Private  — по приглашению

## Важно / подводные камни / best practices

- `git diff --check` перед коммитом — ловит проблемы с whitespace.
- Сообщения коммитов: заголовок ≤50, тело ≤72, повелительное наклонение.
- Активация hook: убрать `.sample` + `chmod +x`.
- Клиентские hooks легко пропустить (`--no-verify`) — для политик безопасности используйте серверные (`pre-receive`, `update`).
- `pre-rebase` защищает от rebase уже опубликованных коммитов.
- `git gc` вовремя, иначе разрастание loose-объектов.
- `filter-branch` меняет SHA-1 всех коммитов: предупредите команду, пересоздайте PR, перевыпустите теги.
- После `filter-branch` не забудьте удалить `refs/original/`, expire reflog и запустить `gc --prune=now` — иначе старые объекты останутся.
- GitHub-кнопка «Merge» всегда создаёт merge commit — для линейной истории используйте rebase+force-push локально или «Rebase and merge» (не упоминается в Pro Git явно, но отсутствие — это сигнал).
- GitHub API: без токена 60 req/час — закончится быстро; создайте Personal Access Token.
- Email для коммитов должен совпадать с подтверждённым в GitHub, иначе коммит не будет привязан к профилю.

## Карточки для повторения

#flashcards/git/internals

Content-addressable filesystem:::Файловая система, где ключом к содержимому служит хеш этого содержимого.

Git SHA-1:::40-символьная строка, вычисляемая на основе содержимого объекта и заголовка.

Bare repository:::Репозиторий без рабочей директории, содержит только содержимое `.git`; используется как серверный.

Blob:::Объект Git, хранящий только содержимое файла без метаданных.

Tree:::Объект, соответствующий UNIX-директории; хранит имена файлов, ссылки на blob или поддеревья и режимы доступа.

Commit object:::Объект со ссылкой на корневое дерево, автором, коммиттером, датой, сообщением и ссылками на родителей.

Tag object:::Объект с теггером, датой, сообщением, ссылающийся обычно на коммит.

Mark (fast-import):::Идентификатор коммита в процессе fast-import для связывания объектов.

Режим `100644`:: ==обычный файл==

Режим `100755`:: ==исполняемый файл==

Режим `120000`:: ==символьная ссылка==

Режим `160000`:: ==submodule==

Loose object:::Объект Git, хранимый в одном файле на диске.

Packfile:::Упакованный файл с объектами и дельта-сжатием (одна версия целиком, остальные — как разница).

Dangling object:::Объект, не достижимый ни из одной ссылки в репозитории.

Dumb protocol:::Простой HTTP через GET-запросы к статическим файлам.

Smart protocol:::Протокол с `upload-pack` / `receive-pack`, передающий только нужные объекты.

Refspec:::Строка `+<src>:<dst>` — правило сопоставления ссылок локального и удалённого репо.

Содержимое `.git/HEAD`:::Указатель на текущую ветку (символьная ссылка).

Назначение `.git/objects/`:::База данных объектов (blob, tree, commit, tag).

Назначение `.git/refs/`:::Указатели на объекты коммитов (ветки, теги, remote-tracking).

Назначение `.git/index`:::Информация об области подготовленных изменений (staging area).

Назначение `.git/description`:::Используется только GitWeb; не важен для работы Git.

Назначение `.git/packed-refs`:::Все ссылки в одном файле после упаковки (`git gc`).

Что делает `git hash-object -w --stdin`?::Вычисляет SHA-1 и сохраняет объект в базу.

Что делает `git cat-file -p`?::Показывает содержимое объекта (pretty-print).

Что делает `git cat-file -t`?::Показывает тип объекта.

Что делает `git write-tree`?::Записывает состояние индекса в объект дерева, возвращает его SHA-1.

Что делает `git commit-tree`?::Создаёт объект коммита на основе дерева и родительских коммитов.

Что делает `git update-ref`?::Безопасно обновляет указатели (ветки, теги).

Что делает `git symbolic-ref`?::Читает/изменяет HEAD как символическую ссылку.

Что делает `git fsck --full`?::Проверяет целостность базы, находит dangling-объекты.

Что делает `git prune`?::Удаляет объекты, на которые нет ссылок.

Назначение `git gc`:::Упаковывает loose-объекты в packfiles с дельта-сжатием и ссылки в `.git/packed-refs`.

Пороги автоматического `git gc`:: ==~7000 loose-объектов или >50 pack-файлов==

`git filter-branch --tree-filter` vs `--index-filter`
?
`--tree-filter` — checkout каждого коммита, медленно
`--index-filter` — работает напрямую с индексом, быстрее

Удаление файла из всей истории::`git filter-branch --index-filter 'git rm --cached --ignore-unmatch FILE' HEAD`

Выделение поддиректории в корень::`git filter-branch --subdirectory-filter DIR HEAD`

Глобальное изменение email::`git filter-branch --commit-filter` с проверкой `$GIT_AUTHOR_EMAIL`

Что нужно сделать после `filter-branch` для реального удаления данных?
?
1. Удалить `.git/refs/original/`
2. Expire reflog: `git reflog expire --expire=now --all`
3. `git gc --prune=now`

Что делает `git replace A B`?::При обращении к объекту A Git подставляет содержимое объекта B; используется для объединения историй без реального переписывания SHA-1.

Edge case: SHA-1 collision?::Git проигнорирует запись нового объекта и всегда будет отдавать первый записанный.

Переменная `GIT_DIR`:::Путь к папке `.git`; если не задана, Git ищет её вверх по дереву.

Переменная `GIT_WORK_TREE`:::Путь к корню рабочей директории.

Переменная `GIT_INDEX_FILE`:::Путь к файлу индекса.

Переменная `GIT_AUTHOR_EMAIL`:::Используется в `filter-branch --commit-filter` для изменения email автора.

Переменная `GIT_TRACE`:::Включает общую трассировку выполнения команд.

Переменная `GIT_ASKPASS`:::Переопределяет программу запроса пароля у пользователя.

Где лежат hooks?:: ==`.git/hooks/`==

Как активировать hook?::Убрать расширение `.sample` и сделать файл исполняемым (`chmod +x`).

Как пропустить клиентский hook?::Флаг `--no-verify` (например, `git commit --no-verify`).

`pre-commit` hook:::Перед вводом сообщения; используется для линтеров/тестов.

`prepare-commit-msg` hook:::Динамическое редактирование сообщения коммита (вставка ID тикета).

`commit-msg` hook:::Проверка соответствия сообщения коммита формату.

`post-commit` hook:::После завершения коммита; уведомления.

`pre-rebase` hook:::Можно запретить rebase коммитов, уже отправленных на сервер.

`post-checkout` / `post-merge` hook:::Настройка рабочего окружения или восстановление данных, которые Git не отслеживает (права доступа и т. п.).

`pre-push` hook:::Выполняется при `git push`; может отклонить пуш по политике.

`pre-receive` (серверный):::Принимает список обновляемых ссылок; если возвращает ошибку — весь push отклоняется. Для ACL и защиты от force-push.

`update` (серверный):::Аналогичен `pre-receive`, но запускается отдельно для каждой ветки; отклоняет изменения в конкретных ветках.

`post-receive` (серверный):::После успешного приёма; рассылки, обновления тикетов, запуск CI.

Почему клиентские hooks ненадёжны для policy?::Их легко пропустить через `--no-verify`; для безопасности нужны серверные.

Пример server policy через `update`::Проверка регулярным выражением наличия `ref: <num>` в сообщении; отклонение push при несовпадении.

Пример user-based ACL::Hook считывает файл ACL и разрешает push в конкретные директории только определённым пользователям (в Pro Git — пример на Ruby).
