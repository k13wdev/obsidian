## Суть и контекст

Git — распределённая система управления версиями (DVCS). В отличие от централизованных систем (CVS, SVN, Perforce) клиенты не просто скачивают последний снимок файлов, а полностью копируют репозиторий. Если сервер выйдет из строя, любой клиентский репозиторий можно использовать для восстановления. Конспект собирает основы: модель данных, три состояния, три области, базовые команды, настройку, `.gitignore`, теги (включая подписанные GPG), `git describe` и типичные подводные камни.

---

## Ключевые идеи

- Git хранит данные как серию **snapshot**'ов миниатюрной файловой системы, а не дельт файлов. Если файл не менялся, Git сохраняет только ссылку на предыдущий идентичный файл.
- Три состояния файлов: **modified → staged → committed** (плюс исходное **untracked**).
- Три области: **working directory**, **staging area (index)**, **`.git` directory**.
- `.git` — самая важная часть репозитория, копируется при клонировании.
- Теги бывают **lightweight** и **annotated**; аннотированные — полноценные объекты в базе Git; могут быть подписаны GPG.
- `git push` по умолчанию не передаёт теги; их нужно отправлять явно.
- `git describe` формирует человекочитаемое имя коммита на основе ближайшего аннотированного тега.
- GPG-подпись: `git tag -s`, `git tag -v`, `git commit -S`, проверка в `git log --show-signature`, защита слияний через `--verify-signatures`.

## Определения и термины

**Git** — распределённая система управления версиями, хранящая данные как серию снимков.

**Distributed VCS (DVCS)** — система, где клиенты полностью зеркалируют репозиторий, включая историю всех версий.

**Working Directory** — «песочница» для правок; извлечённая из базы данных версия проекта, с которой вы работаете на диске.

**Staging Area (Index)** — плоский манифест, представляющий предлагаемый снимок для следующего коммита; файл в `.git`.

**`.git` directory** — место, где Git хранит метаданные и базу данных объектов проекта.

**Untracked** — файл, о котором Git ещё не знает.

**Modified** — файл изменён, но не зафиксирован в базе данных.

**Staged** — файл помечен в текущей версии для включения в следующий коммит.

**Committed** — данные надёжно сохранены в локальной базе данных.

**Snapshot** — полный слепок всех файлов проекта на момент коммита.

**HEAD** — указатель на текущую ссылку ветки, которая указывает на последний коммит этой ветки.

**Tag** — неизменяемая «закладка» на важный момент истории (обычно релиз).

**Lightweight tag** — простая ссылка на конкретный коммит, без метаданных.

**Annotated tag** — полноценный объект в базе Git, содержащий автора, email, дату, сообщение; может быть подписан.

**Signed tag** — аннотированный тег с GPG-подписью автора (`git tag -s`).

**Detached HEAD** — состояние, когда HEAD указывает напрямую на коммит или тег, а не на имя ветки.

**GPG key (user.signingkey)** — идентификатор GPG-ключа, используемого Git по умолчанию для подписи.

## Как устроено

### Три состояния и три области

```
Working Directory ──(git add)──▶ Staging Area ──(git commit)──▶ .git repository
      modified                       staged                        committed
```

### Базовые команды

**`git init`** — создаёт в текущем каталоге подкаталог `.git` со всеми необходимыми файлами репозитория (скелет проекта).

**`git clone [url]`** — получает полную копию данных с сервера, включая историю всех версий каждого файла.

**`git status`** — основной инструмент для определения состояния файлов.
- `-s` / `--short` — компактный вывод (`??` — untracked, `A` — добавлен в индекс, `M` — изменён).

**`git add [file]`** — многоцелевая команда:
- начинает отслеживание новых файлов;
- подготавливает (stage) изменённые файлы;
- помечает файлы как разрешённые после конфликта слияния.

**`git commit`** — создаёт снимок состояния индекса.
- `-m "сообщение"` — задать комментарий прямо в командной строке.
- `-a` — автоматически добавить в коммит все уже отслеживаемые изменённые файлы (пропуск `git add`).
- `--amend` — заменяет последний коммит новым, объединяя текущую область подготовки с данными предыдущего коммита.
- `-S` — подписать коммит GPG-ключом (Git 1.7.9+).

**`git log`** — просмотр истории.
- `--oneline` — каждый коммит в одну строку.
- `-p` — показывает дельту, внесённую в каждом коммите.
- `--stat` — краткая статистика изменений по каждому файлу.
- `--graph` — ASCII-график ветвлений и слияний.
- `--since="2.weeks"`, `--until` — фильтрация по времени.
- `--author="name"` — поиск по автору.
- `--grep="pattern"` — поиск по ключевым словам в сообщениях.
- `--show-signature` — отображает и проверяет GPG-подписи коммитов.
- `--pretty=format:"%G?"` — статус подписи: `G` (хорошая), `B` (плохая), `U` (неизвестный ключ), `N` (нет).

**`git diff`** — показывает, что изменено, но ещё не staged.
- `--staged` (синоним `--cached`) — изменения в индексе относительно последнего коммита.

**`git show [object]`** — информация о конкретном объекте (теге, коммите).

**`git rm [file]`** — удаляет файл из индекса и рабочего каталога. Если файл был изменён и уже в индексе — требуется `-f`.

**`git rm --cached [file]`** — удаляет файл из индекса, но оставляет на диске. Применяется, если файл случайно закоммитили и он должен быть в `.gitignore`.

**`git mv [from] [to]`** — переименование. На самом деле это сокращение трёх команд: `mv` в системе, `git rm` старого имени, `git add` нового. Git понимает переименование неявно.

### `.gitignore`

Синтаксис:
- пустые строки и строки, начинающиеся с `#`, игнорируются;
- стандартные glob-шаблоны: `*`, `[abc]`, `?`, диапазоны;
- `**/` — соответствие вложенным каталогам (`a/**/z`);
- `/` в начале — отмена рекурсивности (только в текущем каталоге);
- `/` в конце — указывает на каталог;
- `!` — negation: инвертирует шаблон.

```
*.a       # игнорировать все файлы .a
!lib.a    # НО отслеживать lib.a
/TODO     # игнорировать TODO только в корне
build/    # игнорировать всё в папке build/
doc/*.txt # игнорировать txt в doc/, но не в doc/server/
```

### `git config` и уровни настроек

- `--system` — `/etc/gitconfig`, для всех пользователей системы.
- `--global` — `~/.gitconfig`, для текущего пользователя.
- `--local` (по умолчанию) — `.git/config` конкретного репозитория. Перекрывает предыдущие уровни.

Основные настройки:
- `user.name`, `user.email` — обязательны перед первым коммитом.
- `user.signingkey` — идентификатор GPG-ключа для подписи тегов/коммитов.
- `core.editor` — текстовый редактор по умолчанию (emacs, vim).
- **Alias** — `git config --global alias.st status` → теперь можно писать `git st`. Если нужно выполнить внешнюю команду, псевдоним начинается с `!`.

### Теги

- **Lightweight**: `git tag [tagname]` — просто ссылка на коммит.
- **Annotated**: `git tag -a v1.4 -m "my version 1.4"` — полноценный объект с автором, датой, сообщением; рекомендуется для официальных релизов.
- **Signed** (подписанный): `git tag -s v1.5 -m 'my signed tag'` — аннотированный тег с GPG-подписью. При `git show` отображается блок PGP-подписи.
- `git tag -v [tagname]` — проверка GPG-подписи; требует наличия открытого ключа подписанта в keyring.
- `git tag -l "v1.8.5*"` — список тегов по шаблону.
- `git show [tagname]` — информация о теге и коммите под ним.
- **Push тегов**: по умолчанию `git push` их не передаёт.
  - `git push origin [tagname]` — отправить конкретный тег.
  - `git push origin --tags` — отправить все отсутствующие теги.

### Подписанные коммиты

- `git commit -S -m 'signed commit'` — подпись коммита GPG (Git ≥ 1.7.9).
- `git merge -S` — подписать создаваемый коммит слияния.
- `git merge --verify-signatures` / `git pull --verify-signatures` — отклонить слияние, если подпись отсутствует или невалидна; защита от вливания непроверенного кода.
- `git log --show-signature` — вывод с проверкой подписей.

### `git describe`

Формирует человекочитаемое уникальное имя коммита на основе ближайшего аннотированного тега.

Формат: `[тег]-[количество_коммитов_после_тега]-g[SHA-1_коммита]`

- По умолчанию учитывает только аннотированные теги (`-a` или `-s`) — гарантирует корректные имена сборок.
- Применение: номера версий, имена архивных файлов при релизах.

## Детали и нюансы

- `--amend` нельзя применять к коммитам, которые уже были отправлены (push) в общий репозиторий.
- **Detached HEAD**: новые коммиты в этом состоянии трудно найти после переключения на другую ветку, если не создать новую ветку.
- На Windows Git может не заметить изменение регистра букв в имени файла (`README` → `readme`) — лечится настройкой `core.ignorecase`.
- GPG-подпись тега верифицируется только при наличии открытого ключа подписанта в локальном keyring.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| CVCS (SVN, CVS) | Простая ментальная модель; единый сервер — единая истина | Сервер — точка отказа; ограничен offline-режим | Legacy, централизованные команды |
| DVCS (Git) | Полная копия репозитория у каждого; работа offline; любой клиент может восстановить сервер | Больше концепций для освоения | Современная разработка |
| Lightweight tag | Простая ссылка | Без метаданных | Временные закладки |
| Annotated tag | Автор/дата/сообщение; может быть подписан | Объект в базе | Официальные релизы |
| Signed tag/commit | Подтверждение авторства и целостности | Требует GPG-ключа и доверия | Релизы в open source, security-чувствительные проекты |

## Примеры

### Минимальный цикл работы

```bash
git init
git add README.md
git commit -m "Initial commit"
git status -s
git log --oneline --graph
```

### Отмена отслеживания случайно закоммиченного файла

```bash
echo "secrets.env" >> .gitignore
git rm --cached secrets.env
git commit -m "Stop tracking secrets.env"
```

### Исправление последнего коммита

```bash
git add forgotten.txt
git commit --amend -m "Fix: include forgotten file"
```

### Создание и отправка annotated-тега

```bash
git tag -a v1.4 -m "my version 1.4"
git push origin v1.4
```

### Подписанный тег и его проверка

```bash
git config --global user.signingkey <gpg-key-id>
git tag -s v1.5 -m 'my signed tag'
git tag -v v1.5
```

### Подписанный коммит

```bash
git commit -S -m 'signed commit'
git log --show-signature
git merge --verify-signatures feature
```

### `git describe`

```bash
git describe                 # → v1.4-5-g2414721
git describe --always HEAD   # fallback на голый SHA, если тегов нет
```

## Важно / подводные камни / best practices

- **Опасно**: `git checkout -- [file]` безвозвратно перезаписывает изменения в рабочем каталоге версией из последнего коммита.
- **Опасно**: `git reset --hard` перезаписывает и индекс, и рабочий каталог — уничтожает незафиксированные изменения.
- **Никогда** не делайте `rebase` или `commit --amend` для коммитов, уже отправленных в общий репозиторий.
- Detached HEAD — всегда создавайте ветку для новых коммитов.
- Для релизов предпочитайте annotated или signed теги — lightweight не несут метаданных.
- `--verify-signatures` — простой способ защититься от вливания непроверенного кода из форка.
