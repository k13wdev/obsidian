## Суть и контекст

Ветвление — главное преимущество Git. Ветки — легковесные подвижные указатели на коммиты. Конспект описывает внутреннее устройство ветки, fast-forward vs 3-way merge, разрешение конфликтов, продвинутые стратегии слияния (resolve/recursive/ours/theirs/octopus, `-X` опции), rebase (включая интерактивный), cherry-pick, reset demystified (через модель трёх деревьев), revert, reflog, rerere и tracking-branches.

---

## Ключевые идеи

- Ветка — это файл в 41 байт: 40 символов SHA-1 + `\n`. Никаких копий файлов.
- HEAD — указатель на текущую локальную ветку.
- Fast-forward merge просто двигает указатель; 3-way merge создаёт merge commit с двумя родителями.
- Rebase = «пересаживание патчей» для создания линейной истории.
- **Золотое правило rebase**: не ребейзить коммиты, уже отправленные в публичный репозиторий.
- `git reset` манипулирует тремя деревьями: HEAD, Index, Working Directory.
- `git revert` — безопасная отмена в публичной истории (новый коммит с обратными изменениями).
- Reflog локален, не передаётся при clone, хранится несколько месяцев → спасательный круг при потерях.
- **rerere** (reuse recorded resolution) — Git запоминает ручное разрешение конфликта и автоматически применяет его при повторе.
- **Advanced merging**: можно выбрать стратегию (`-s`), модификатор (`-X`), отменить (`--abort`), посмотреть стадии через `git show :1:/:2:/:3:file`.

## Определения и термины

**Ветка (branch)** — легковесный подвижный указатель на один из коммитов; технически файл в 41 байт с SHA-1 коммита.

**HEAD** — указатель на текущую ссылку ветки, которая указывает на последний коммит этой ветки.

**Detached HEAD** — состояние, когда HEAD указывает напрямую на конкретный коммит/тег.

**Fast-forward merge** — слияние, когда вливаемая ветка — прямой потомок текущей; Git просто двигает указатель вперёд.

**3-way merge** — слияние, когда истории разошлись; использует верхушки двух веток и их общего предка (merge base); создаёт коммит слияния с более чем одним родителем.

**Merge commit** — коммит, у которого более одного родителя.

**Merge base** — общий предок двух веток при трёхстороннем слиянии.

**Rebase** — процесс перекладывания патчей изменений с одной ветки на верхушку другой.

**Cherry-pick** — копирование одного конкретного коммита в текущую ветку (как rebase для одного коммита).

**Revert** — отмена изменений коммита путём создания нового коммита с противоположными изменениями.

**Reflog** — журнал перемещений HEAD и веток, локальный, хранится несколько месяцев.

**rerere (reuse recorded resolution)** — механизм, при котором Git сохраняет ручное разрешение конфликта и повторно применяет его, если такой же конфликт возникнет снова.

**Remote-tracking branch** — локальная ссылка на состояние ветки в удалённом репозитории (например, `origin/master`); доступна только для чтения.

**Tracking branch** — локальная ветка, имеющая прямую связь с удалённой (upstream branch).

**Merge strategy** — алгоритм слияния (`recursive`, `resolve`, `ours`, `octopus` и т. д.), задаваемый через `-s`.

**Merge option (`-X`)** — модификатор поведения стратегии (например, `-Xignore-all-space`, `-Xours`).

## Как устроено

### Ветка и HEAD

- Ветка — файл в `refs/heads/[name]` с SHA-1 коммита.
- `HEAD` — символьная ссылка на текущую ветку.
- Переключение ветки перемещает HEAD.

### Fast-forward vs 3-way merge

```
Fast-forward (ветка — прямой потомок):
A---B---C (master)
         \
          D---E (feature)   →   A---B---C---D---E (master, feature)

3-way merge (истории разошлись):
A---B---C---F (master)
     \
      D---E (feature)       →   A---B---C---F---M (master)
                                          \   /
                                           D-E
```

### Разрешение конфликтов

Конфликт возникает, если в двух ветках изменена одна и та же часть одного файла.

Маркеры:
```
<<<<<<< HEAD
изменения текущей ветки
=======
изменения вливаемой ветки
>>>>>>> [имя_ветки]
```

Процесс:
1. Отредактировать файл вручную.
2. `git add` — пометить конфликт как разрешённый (добавить в индекс).
3. `git commit` — завершить merge commit.

Инструменты:
- `git status` — показывает файлы с «unmerged paths».
- `git mergetool` — визуальный инструмент.
- `git log --merge` — показывает только коммиты (в обеих ветках), которые затронули конфликтующие файлы.
- `git show :1:file` / `:2:file` / `:3:file` — содержимое стадий индекса: `:1` общий предок, `:2` наша версия, `:3` их версия.
- `git merge --abort` — откатывает попытку слияния к состоянию до запуска команды.

### Управление ветками

- `git branch [name]` — создать ветку на текущем коммите.
- `git checkout [branch]` — переключиться (обновляет HEAD и файлы).
- `git branch -d [name]` — удалить (только если ветка уже слита).
- `git branch -D [name]` — принудительное удаление неслитой ветки.
- `git branch -v` — список веток с последними коммитами.
- `git branch --merged` / `--no-merged` — фильтр по состоянию слияния.
- `git branch -vv` — детальный список локальных веток с информацией, насколько они опережают (ahead) или отстают (behind) от удалённых (на основе закэшированных данных последнего запроса к серверу).
- `git switch` — в современных версиях Git заменяет функционал `checkout` по переключению веток, отделяя его от восстановления файлов.

### Advanced merging — стратегии и опции

**Стратегии слияния (`-s`):**
- `recursive` (по умолчанию для двух веток) — трёхсторонний merge.
- `resolve` — альтернативный трёхсторонний алгоритм.
- `ours` — «фейковое» слияние: результат идентичен текущей ветке, код вливаемой ветки полностью игнорируется. Полезно, чтобы «искусственно» пометить ветку как слитую.
- `octopus` — слияние более двух веток одновременно; выбирается автоматически.

**Модификаторы (`-X`):**
- `-Xignore-all-space` — полностью игнорирует пробелы при сравнении строк.
- `-Xignore-space-change` — считает последовательности пробелов эквивалентными.
- `-Xours` — при конфликте предпочитает локальную версию, но неконфликтующие части объединяет автоматически. Отличается от `-s ours`: та просто отбрасывает вливаемую ветку.
- `-Xtheirs` — зеркально, предпочитает версию вливаемой ветки.

**Отмена слияния:**
```bash
git merge --abort        # до коммита
git reset --hard HEAD    # если merge уже закоммичен и не отправлен
```

**Стадии индекса при конфликте:**
```bash
git show :1:path/to/file  # общий предок
git show :2:path/to/file  # наша версия (HEAD)
git show :3:path/to/file  # их версия (вливаемая)
```

### Rebase

Механика: Git идёт к общему предку, берёт дифф каждого коммита текущей ветки, сохраняет их во временные файлы, сбрасывает текущую ветку на целевую и по очереди применяет сохранённые изменения.

**Интерактивный rebase** (`git rebase -i`):
- `pick` — использовать коммит как есть;
- `reword` — использовать коммит, но изменить сообщение;
- `edit` — остановиться для изменения файлов (с `git commit --amend`);
- `squash` — объединить с предыдущим, объединяя сообщения;
- `fixup` — как `squash`, но сообщение отбрасывается;
- `drop` — удалить коммит (удаление строки в редакторе).

**`git rebase --onto master server client`** — взять изменения ветки `client`, которых нет в `server`, и переложить их на `master`.

**Золотое правило rebase**: никогда не делайте rebase коммитов, уже отправленных в публичный репозиторий. Иначе коллегам придётся переподтягивать ваши изменения — путаница и дублирование.

### Cherry-pick и revert

- `git cherry-pick [hash]` — копирует один коммит в текущую ветку.
- `git revert [hash]` — создаёт новый коммит с противоположными изменениями; безопасный способ отмены в публичной истории.

### Reset Demystified — три дерева

`git reset` работает с моделью трёх деревьев:
- **HEAD** — снимок последнего коммита; родитель следующего коммита.
- **Index (staging area)** — предлагаемый следующий коммит.
- **Working Directory** — «песочница» для правок.

**Сценарии:**

| Флаг | HEAD | Index | Working Dir | Когда использовать |
|------|------|-------|-------------|--------------------|
| `--soft` | ✅ | ❌ | ❌ | Объединить коммиты / переделать сообщение; новая работа остаётся в stage |
| `--mixed` (default) | ✅ | ✅ | ❌ | Пересобрать stage; рабочая директория сохранена |
| `--hard` | ✅ | ✅ | ✅ | ⚠️ Уничтожает незафиксированные изменения |

**Reset с путями** (`git reset [commit] -- path`):
- HEAD не двигается.
- Обновляется только указанный путь в индексе.
- Рабочая директория не затрагивается.
- Применение: «анstage» файла (`git reset HEAD file`).

**Squash через reset:** `git reset --soft HEAD~3 && git commit -m "..."` — объединяет три последних коммита в один.

**Отличие `reset` от `checkout`:**
- `checkout` безопасен для рабочей директории (проверяет наличие изменений); `reset --hard` заменяет всё без проверки.
- `reset` перемещает ссылку ветки, на которую указывает HEAD; `checkout` перемещает сам HEAD на другую ссылку.

### Reflog и восстановление

`git reflog` — журнал перемещений HEAD и веток; локальный, хранится несколько месяцев.

Сценарий восстановления после `reset --hard` или удаления ветки:
```bash
git reflog
# находим SHA потерянного коммита
git branch recovery-branch [SHA]
```

Синтаксис времени: `master@{yesterday}`, `HEAD@{2.hours.ago}` — но работает только в пределах срока хранения reflog.

### rerere — reuse recorded resolution

Git может запомнить ручное разрешение конфликта и автоматически применить его при повторе.

- Включение: `git config --global rerere.enabled true` (или создание `.git/rr-cache/`).
- Механика: при конфликте Git сохраняет состояние «до» и «после» ручного исправления; при повторе того же конфликта — автоматически применяет сохранённое решение.
- `git rerere status` — файлы с записанными состояниями.
- `git rerere diff` — текущая стадия разрешения.
- Применение: частые rebase длинных веток; «пробные» слияния, которые потом откатывают.

### Tracking branches

- `git branch -u origin/feature` / `--set-upstream-to` — связывает локальную ветку с удалённой.

## Детали и нюансы

- `git checkout -- [file]` — старый способ отмены изменений в рабочем каталоге (копирует файл из индекса в рабочую область). В современном Git функционал разделён между `git switch` (переключение веток) и `git restore` (восстановление файлов).
- Remote-tracking branches (`origin/master`) перемещаются автоматически при сетевых операциях и доступны только для чтения.
- В detached HEAD новые коммиты не принадлежат ни одной ветке и могут быть потеряны при переключении, если не создать новую ветку.
- **Reflog lifetime**: данные хранятся всего несколько месяцев, поэтому команды вида `master@{yesterday}` не сработают для очень старых состояний.
- **Octopus strategy** — выбирается автоматически, когда сливается больше двух веток; используется, если конфликтов не возникает (иначе нужен `recursive` по парам).

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Merge | Сохраняет реальную хронологию; не деструктивен | История запутана обилием merge commits | Публичные ветки, командная работа |
| Rebase | Чистая линейная история | Переписывает историю (меняет SHA-1); опасен для публичных веток | Локальная полировка перед push |
| `git revert` | Безопасно для публичной истории | Оставляет «отменяющий» коммит | Откат уже отправленного коммита |
| `git reset --hard` | Быстро убирает изменения | Уничтожает незафиксированную работу | Только локально |
| `-s ours` | Помечает ветку как слитую без применения кода | Код вливаемой ветки игнорируется | Устаревшие ветки, которые «закрываем» |
| `-Xours` / `-Xtheirs` | Автоматическое разрешение конфликта по приоритету | Теряется часть изменений другой стороны | Простые конфликты, где сторона уже проверена |
| rerere | Экономит время на повторяющихся конфликтах | Не рассчитано на изменение логики разрешения | Длинные feature-ветки, частые rebase |

## Примеры

### Создание ветки и merge

```bash
git checkout -b feature
# ... работа ...
git commit -am "add feature"
git checkout master
git merge feature
git branch -d feature
```

### Разрешение конфликта

```bash
git merge feature
# CONFLICT in app.js
git status               # список unmerged paths
git show :1:app.js       # версия общего предка
git show :2:app.js       # наша (HEAD)
git show :3:app.js       # их (feature)
$EDITOR app.js           # вручную убрать маркеры
git add app.js
git commit
```

### Отмена начатого merge

```bash
git merge feature
# CONFLICT, решили передумать
git merge --abort
```

### Merge с игнорированием пробелов

```bash
git merge -Xignore-all-space feature
```

### Фейковое слияние через `-s ours`

```bash
git checkout master
git merge -s ours old-feature   # помечаем old-feature как слитую, код не тянем
```

### Интерактивный rebase

```bash
git rebase -i HEAD~5
# в редакторе: pick/reword/squash/fixup/drop
```

### Squash через soft reset

```bash
git reset --soft HEAD~3
git commit -m "Squashed last 3 commits"
```

### Восстановление из reflog

```bash
git reflog
# HEAD@{2}: reset: moving to HEAD~3
git branch recovery-branch HEAD@{2}
```

### Включение rerere

```bash
git config --global rerere.enabled true
git rerere status
git rerere diff
```

## Важно / подводные камни / best practices

- **Золотое правило rebase**: никогда не ребейзьте публичные коммиты.
- `reset --hard` уничтожает незафиксированные изменения.
- `revert` — предпочтительный способ отмены в публичной истории.
- `reflog` локален — не рассчитывайте на него после clone; хранится лишь несколько месяцев.
- В detached HEAD создавайте ветку, прежде чем уходить с коммита.
- Для хорошей истории перед push — интерактивный rebase с `squash`/`fixup`.
- Различайте `-s ours` (отбрасывает код) и `-Xours` (предпочитает при конфликте) — легко перепутать.
- Включённый rerere особенно полезен для длинных feature-веток, где rebase приходится делать многократно.

Интерактивный rebase: `reword`:: ==использовать коммит, изменить сообщение==

Интерактивный rebase: `edit`:: ==остановиться для изменения файлов коммита==

Интерактивный rebase: `squash`:: ==объединить с предыдущим, объединяя сообщения==

Интерактивный rebase: `fixup`:: ==как squash, но сообщение отбрасывается==

Интерактивный rebase: `drop`:: ==удалить коммит==

Что делает `git rebase --onto master server client`?::Берёт изменения `client`, которых нет в `server`, и переносит их на `master`.

Золотое правило rebase::Никогда не делайте rebase коммитов, уже отправленных в публичный репозиторий.

В чём разница merge и rebase?
?
Merge: сохраняет реальную хронологию, не деструктивен, но история запутана.
Rebase: линейная чистая история, но переписывает SHA-1 и опасен для публичных веток.

Три дерева Git для reset::HEAD, Index, Working Directory.

`git reset --soft`:: ==двигает только HEAD; Index и Working Directory не трогает==

`git reset --mixed` (default):: ==двигает HEAD + обновляет Index; Working Directory не трогает==

`git reset --hard`:: ==двигает HEAD + обновляет Index + перезаписывает Working Directory (опасно)==

Что делает `git reset HEAD file`?::«Анstage» файла: обновляет только индекс для пути, HEAD и рабочая директория не меняются.

Squash через reset::`git reset --soft HEAD~N && git commit -m "..."` — объединяет N последних коммитов в один.

В чём отличие reset и checkout?
?
`reset` перемещает ссылку ветки и может затереть рабочую директорию без проверки (`--hard`).
`checkout` перемещает HEAD на другую ссылку и безопасен для рабочей директории.

В чём разница revert и reset?
?
Revert: создаёт новый коммит с обратными изменениями — безопасно для публичной истории.
Reset: сдвигает указатели назад — переписывает историю.

Сценарий восстановления через reflog
?
1. `git reflog` — найти SHA потерянного коммита
2. `git branch recovery-branch [SHA]` — создать ветку-закладку
3. Продолжить работу с восстановленной ветки

Edge case: reflog lifetime?::Хранится всего несколько месяцев; команды `master@{yesterday}` не сработают для очень старых состояний.

Что делает `git branch -u origin/feature`?::Устанавливает upstream для локальной ветки (tracking).

Cherry-pick это по сути:: ==rebase для одного коммита==

Включение rerere:: ==`git config --global rerere.enabled true`==

Зачем нужен rerere?::Автоматически применяет ранее записанное разрешение конфликта при повторе; полезно для частых rebase длинных веток.

Что делает `git rerere status`?::Показывает файлы, для которых записаны состояния конфликта.

Edge case: rebase уже опубликованных коммитов?::Коллегам придётся переподтягивать историю; дубликаты коммитов, путаница.

Edge case: reset --hard с незакоммиченной работой?::Работа уничтожается — восстановить можно только из reflog, если коммиты были.
