## Суть и контекст

Scope (область видимости) — фундаментальный механизм JavaScript, определяющий доступность переменных и функций для текущего выполняемого кода. Scope — это набор правил для хранения переменных в некотором месте и для обнаружения этих переменных позднее по имени идентификатора. Понимание scope необходимо для работы с замыканиями, модульным паттерном, инкапсуляцией и предотвращением утечек в глобальное пространство.

---

## Ключевые идеи

- JavaScript использует **лексическую** (статическую) область видимости — scope определяется на этапе написания кода, а не во время выполнения
- Существует три типа scope: глобальный (Global Scope), функциональный (Function Scope) и блочный (Block Scope)
- Поиск переменной идёт по цепочке областей видимости (Scope Chain) — от текущего контекста вверх до глобального
- `var` имеет только функциональную область видимости; `let`/`const` — блочную
- Механизмы `eval` и `with` способны изменять лексическую область видимости в рантайме, но это антипаттерн
- Объявления переменных и функций «поднимаются» (hoisting) в верх своей области видимости на этапе компиляции
- Функции, созданные через `new Function`, имеют Scope, ссылающийся на глобальный объект `window`
- `var` на глобальном уровне создаёт свойство `window`, а `let`/`const` — нет

## Определения и термины

**Область видимости (Scope)** — набор правил для хранения переменных в некотором месте и для обнаружения этих переменных позднее по имени идентификатора. Определяет доступность переменных для текущего выполняемого кода.

**Лексическая область видимости (Lexical Scope)** — область видимости, которая определена во время разбора на лексемы. Основана на том, где переменные и блоки области видимости были созданы во время написания кода. Позиция функции или блока жёстко фиксируется на этапе компиляции и не зависит от того, откуда функция вызывается.

**Динамическая область видимости (Dynamic Scope)** — модель, при которой область видимости определяется динамически во время выполнения, основываясь на стеке вызовов (откуда была вызвана функция), а не на вложенности областей видимости в коде. В JavaScript нет динамической области видимости, но механизм привязки `this` функционально на неё похож.

**Цепочка областей видимости (Scope Chain)** — вложенные зоны областей видимости. Поиск переменной по идентификатору начинается с самой внутренней области видимости и работает по направлению наружу/вверх, пока не найдётся первое совпадение, после чего останавливается. Если совпадений несколько, внутренний идентификатор «затеняет» внешний (Shadowing).

**LexicalEnvironment (лексическое окружение)** — скрытый внутренний объект, который создаётся при запуске функции и в который записываются её локальные переменные, аргументы и вложенные функции. На верхнем уровне (в браузере) LexicalEnvironment совпадает с глобальным объектом `window`.

**Scope** — скрытое внутреннее системное свойство функции, которое ссылается на LexicalEnvironment, в контексте которого эта функция была создана. Привязывает функцию к месту её «рождения». Свойство никогда не меняется.

**Затенение (Shadowing)** — ситуация, когда одно и то же имя идентификатора указано в нескольких слоях вложенных областей видимости, при этом внутренний идентификатор перекрывает (затеняет) доступ к внешнему. Поиск в лексической области видимости останавливается при первом совпадении.

**Temporal Dead Zone (TDZ)** — период от начала выполнения блока до того момента, когда выполнение кода достигает места объявления и инициализации переменной (`let`, `const`, `class`). Любая попытка доступа к переменной в состоянии TDZ приводит к `ReferenceError`. Зона зависит от порядка выполнения (времени), а не от физического положения в коде.

**Hoisting (поднятие)** — процесс, при котором интерпретатор, по-видимому, перемещает объявления функций, переменных, классов или импортов в верхнюю часть их области видимости до выполнения кода. Не является нормативным термином спецификации ECMAScript — это концептуальная метафора работы компилятора.

**Принцип наименьших привилегий (Principle of Least Privilege)** — принцип дизайна ПО, который заявляет, что следует выставлять наружу только то, что минимально необходимо, и «прятать» всё остальное. В JavaScript применяется к выбору области видимости: идентификаторы ограничиваются наименьшей возможной областью (функцией или блоком).

**IIFE (Immediately Invoked Function Expression)** — немедленно вызываемое функциональное выражение. Оборачивание функции в скобки с последующим вызовом: `(function() { ... })();`. Используется для создания изолированной области видимости и сокрытия локальных переменных от глобального пространства.

## Как устроено

### Три участника обработки кода

Обработкой JavaScript-кода занимаются три компонента:
1. **Engine (Движок)** — отвечает за компиляцию и выполнение
2. **Compiler (Компилятор)** — разбивает код на лексемы, строит AST-дерево, генерирует машинный код
3. **Scope (Область видимости)** — хранит объявленные идентификаторы и обеспечивает правила доступа

### Обработка объявления `var a = 2;`

1. Компилятор разбивает код на лексемы и строит AST-дерево
2. Компилятор встречает `var a` — запрашивает у Области видимости наличие переменной `a`. Если её нет, просит объявить новую переменную `a`
3. Компилятор генерирует машинный код для Движка, оставляя инструкцию присваивания `a = 2;` на фазу выполнения
4. При выполнении Движок запрашивает у Области видимости **LHS-поиск** (Left-Hand Side) переменной `a` — ищет сам контейнер
5. Область видимости ищет `a` в текущем контексте. Если не находит — переходит во внешнюю область по цепочке (Scope Chain)
6. После нахождения контейнера Движок присваивает значение `2`

### Алгоритм поиска переменной (Scope Chain)

1. Проверить текущий `LexicalEnvironment`
2. Если переменная не найдена — перейти по Scope во внешнее окружение
3. Повторять до глобального объекта (`window` в браузере)
4. Поиск останавливается при первом совпадении (Shadowing)

> Аналогия из источника: вложенные области видимости — это многоэтажное здание. Текущая область видимости — первый этаж. Если переменная не найдена, движок «поднимается на лифте» на этаж выше (наружу), пока не достигнет крыши — глобальной области видимости.

### Работа LexicalEnvironment

1. До выполнения первой строчки кода функции создаётся пустой объект `LexicalEnvironment`
2. В него записываются аргументы функции и локальные переменные (для `var` — со значением `undefined`)
3. При запуске функции `LexicalEnvironment` получает ссылку на внешнее лексическое окружение из Scope функции
4. Во время выполнения происходит присвоение значений переменным
5. В конце выполнения объект `LexicalEnvironment` удаляется сборщиком мусора, если на него нет активных ссылок Scope от вложенных функций (замыканий)

### Типы областей видимости

**Global Scope** — самая внешняя область видимости. Переменные, объявленные через `var`, создают свойства на глобальном объекте `window`. Переменные `let`/`const` не создают свойств на `window`.

**Function Scope** — каждая функция создаёт новый `LexicalEnvironment`. Все объявления внутри функции ограничены её областью видимости и недоступны снаружи. `var` привязывается к ближайшей функции (или глобальному scope), блоки `if`, `for`, `while` для `var` не создают новой области.

**Block Scope** — `let` и `const` привязываются к ближайшему блоку `{ ... }`. Блок `catch(err)` создаёт блочную область видимости для переменной ошибки (даже в ES3). В `for (let i ...)` — для каждой итерации создаётся своя независимая переменная `i`.

### Механика Hoisting

Движок JS компилирует код перед выполнением. Во время компиляции все объявления переменных и функций извлекаются и привязываются к текущей области видимости.

**Приоритет:** объявления функций поднимаются **первыми**, затем — переменные.

| Тип объявления | Что поднимается | Инициализация |
|---|---|---|
| `function declaration` | Целиком (тело + имя) | Полная — можно вызвать до строки объявления |
| `var` | Только объявление | `undefined` — до присваивания |
| `let`, `const`, `class` | Поднимаются в scope | Не инициализируются — попадают в TDZ |
| `function expression` | Как `var` (только имя) | `undefined` — вызов до присваивания → `TypeError` |

Повторные объявления `var` с тем же именем — игнорируются компилятором. Последовательные объявления `function` с одинаковым именем — перекрывают предыдущие.

### Изменение Lexical Scope в рантайме

**`eval(str)`** — принимает строку кода и выполняет её в текущей лексической области видимости, загрязняя её. В Strict Mode `eval` создаёт свой собственный изолированный `LexicalEnvironment`.

**`with(obj)`** — принимает объект и создаёт из него абсолютно новую лексическую область видимости, где свойства объекта становятся локальными идентификаторами. Обычное объявление `var` внутри `with` не входит в его область видимости, а привязывается к окружающей функции. Запрещён в Strict Mode.

### `new Function` и Scope

Функции, созданные через `new Function('', 'code')`, имеют Scope, который **всегда** ссылается на глобальный объект `window`, игнорируя текущий `LexicalEnvironment`. Такие функции не имеют доступа к замыканиям локальных областей видимости — только к глобальным переменным.

### `let` в цикле `for` — перепривязка

При использовании `let` в заголовке `for`:
1. Движок привязывает переменную к телу цикла
2. На каждой итерации создаётся **новая копия** переменной
3. Новая переменная инициализируется значением, полученным на момент окончания предыдущей итерации
4. Любая коллбэк-функция внутри цикла замыкается на `LexicalEnvironment` конкретной итерации с собственной переменной

## Детали и нюансы

### LHS vs RHS поиск

- **LHS-поиск** (Left-Hand Side) — ищет сам контейнер переменной (для присваивания)
- **RHS-поиск** (Right-Hand Side) — ищет значение переменной (для чтения)

В нестрогом режиме, если LHS-поиск не находит переменную вплоть до глобальной области, глобальная область **неявно создаёт** новую глобальную переменную. В строгом режиме — выбрасывается `ReferenceError`.

RHS-поиск несуществующей переменной **всегда** выбрасывает `ReferenceError` (в любом режиме).

### Strict Mode и scope

- Запрещает неявное создание глобальных переменных (невыполненный LHS-поиск → `ReferenceError`)
- `eval` в strict mode создаёт собственную изолированную лексическую область видимости — объявления внутри `eval` не загрязняют внешний scope
- Конструкция `with` полностью запрещена на уровне синтаксиса

### Дублирование объявлений

- Повторные `var` с тем же именем — игнорируются компилятором (значение не сбрасывается), но если есть инициализатор (`var a = 3`), он переопределит значение
- Последовательные `function` с одинаковым именем — последнее перекрывает предыдущие
- `function declaration` внутри блоков `if`/`for` — исторически «вырывались» из блока, с ES6 стандартизированы как Block Scope, но реализация может отличаться в старых браузерах

### `catch` и блочная область видимости

Блок `catch(err) { ... }` создаёт блочную область видимости для переменной ошибки. Обращение к `err` вне блока `catch` → `ReferenceError`. В старых браузерах (IE8-) этот механизм реализован с ошибкой — `err` может перезаписать глобальную переменную.

### `typeof` и TDZ

`typeof undeclaredVar` — безопасно вернёт строку `"undefined"`. Но `typeof varInTDZ` — выбросит `ReferenceError`.

### V8 оптимизация

V8 (Chrome, Node.js) агрессивно оптимизирует использование памяти. Если переменная из внешнего `LexicalEnvironment` не используется ни в одном вложенном замыкании, V8 удаляет её. При попытке обращения через консоль отладки — переменная не определена.

### `const` — что можно и нельзя

`const` требует обязательной инициализации и запрещает перезапись **ссылки** (попытка → `TypeError`). Но если в `const` хранится объект, его внутренние свойства можно свободно изменять (мутировать).

### `eval`/`with` и производительность

Движок JavaScript, встречая `eval` или `with`, **отключает** большинство статических оптимизаций поиска идентификаторов, так как не может заранее предсказать состав Scope. Минификаторы также не могут безопасно переименовать локальные переменные в присутствии `eval`/`with`.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|---|---|---|---|
| `var` (Function Scope) | Широко поддерживается, hoisting | Нет блочной области, утечка из циклов, classic loop bug | Legacy-код |
| `let` (Block Scope) | Блочная область, перепривязка в циклах, TDZ ловит ошибки | — | Переменные, которые будут переприсваиваться |
| `const` (Block Scope) | Блочная область, явное намерение неизменности ссылки | Объекты по-прежнему мутабельны | Значения по умолчанию, ссылки, не меняющиеся |
| IIFE | Изоляция scope в ES5 | Громоздкий синтаксис | Legacy-код, полифиллинг блочной области |
| `new Function` | Scope = `window`, безопасно для минификации | Нет доступа к замыканиям | Динамический код вместо `eval` |
| `eval` | Выполнение динамического кода | Загрязнение scope, отключение оптимизаций | Избегать; в крайнем случае — strict mode |
| `with` | Сокращает обращения к свойствам объекта | Утечки в глобальный scope, запрещён в strict mode | Не использовать |

| | Лексическая область видимости | Динамическая область видимости |
|---|---|---|
| Определяется | На этапе написания кода (компиляции) | Во время выполнения (runtime) |
| Зависит от | Где функция **объявлена** | Откуда функция **вызвана** (call stack) |
| Используется в JS | Да — основной механизм scope | Нет, но `this` функционально похож |
| Языки | JavaScript, большинство языков | Bash, некоторые режимы Perl |

## Примеры

### Лексическая область видимости и поиск по Scope Chain

```javascript
function foo(a) {
  var b = a * 2;
  function bar(c) {
    console.log( a, b, c );
  }
  bar(b * 3);
}
foo( 2 ); // 2, 4, 12
```

Функция `bar` не имеет локальных `a` и `b`. Сначала она ищет их в своём Scope. Не найдя, идёт в Scope (в `foo`), где находит значения `2` и `4`.

### Hoisting: функции поднимаются раньше переменных

```javascript
foo(); // 1

var foo;

function foo() {
  console.log( 1 );
}

foo = function() {
  console.log( 2 );
};
```

Движок сначала компилирует `function foo() { ... }`. Затем видит `var foo;`, но игнорирует его, так как идентификатор `foo` уже занят функцией.

### Hoisting: `var` поднимается с `undefined`

```javascript
console.log(a); // undefined
var a = 2;
```

### Hoisting: Function Expression не поднимается

```javascript
foo(); // TypeError
var foo = function bar() {
  // ...
};
```

Поднимается только `var foo;` (равен `undefined`). Вызов `undefined` как функции → `TypeError`.

### `var` не имеет блочной области видимости

```javascript
for (var i = 0; i < 10; i++) {
  console.log( i );
}
console.log( i ); // 10
```

Переменная `i`, объявленная через `var`, привязывается к окружающей функции (или глобальному Scope), а не к блоку цикла.

### `let` создаёт блочную область видимости

```javascript
var foo = true;
if (foo) {
  let bar = foo * 2;
  bar = something( bar );
  console.log( bar );
}
console.log( bar ); // ReferenceError
```

`let` привязывает `bar` к блоку `{ ... }` оператора `if`. Снаружи она недоступна.

### TDZ — Temporal Dead Zone

```javascript
let x = 1;
if (true) {
  let foo = x + 55; // ReferenceError
  let x = 2;
}
```

Внутри блока `if` переменная `x` перекрывает внешнюю `x`. Но выражение `x + 55` вычисляется до инициализации локальной `let x = 2`, находясь в TDZ.

### Затенение и коллизия идентификаторов (антипаттерн)

```javascript
function foo() {
  function bar(a) {
    i = 3; // неожиданно перезаписывает i из цикла for!
    console.log( a + i );
  }
  for (var i = 0; i < 10; i++) {
    bar( i * 2 ); // бесконечный цикл!
  }
}
foo();
```

Внутри `bar(a)` присваивание `i = 3;` — LHS-поиск. Так как в `bar` нет локальной `i`, поиск находит `i` из цикла `for` и сбрасывает счётчик. Решение: использовать `var i = 3;` внутри `bar` (затенение).

### Сокрытие данных (принцип наименьших привилегий)

```javascript
function doSomething(a) {
  function doSomethingElse(a) {
    return a - 1;
  }
  var b;
  b = a + doSomethingElse( a * 2 );
  console.log( b * 3 );
}
doSomething( 2 ); // 15
```

`b` и `doSomethingElse` скрыты внутри `doSomething` и защищены от случайного вызова/изменения извне.

### IIFE для изоляции области видимости

```javascript
(function() {
  var message = "Привет";
  function showMessage() {
    alert( message );
  }
  showMessage(); // Привет
})();
// alert(message); // Ошибка, переменная недоступна снаружи
```

### IIFE в цикле (до появления `let`)

```javascript
for (var i = 1; i <= 5; i++) {
  (function(j) {
    setTimeout( function timer() {
      console.log( j );
    }, j * 1000 );
  })( i );
}
```

IIFE вызывается на каждой итерации, копируя текущее значение `i` в локальный параметр `j`.

### `let` в цикле — перепривязка

```javascript
function makeArmy() {
  let shooters = [];
  for (let i = 0; i < 10; i++) {
    shooters.push(function() {
      alert( i );
    });
  }
  return shooters;
}
var army = makeArmy();
army[0](); // 0
army[5](); // 5
```

`let` создаёт отдельную независимую переменную `i` для каждого шага цикла. Каждая анонимная функция замыкается на свою собственную `i`.

### `var` vs `let` на глобальном уровне

```javascript
var x = 'global';
let y = 'global';
console.log(this.x); // "global" (создано свойство window.x)
console.log(this.y); // undefined (свойство не создано)
```

### `eval` без strict mode (загрязнение Scope)

```javascript
function foo(str, a) {
  eval( str ); // модифицирует лексическую область видимости!
  console.log( a, b );
}
var b = 2;
foo( "var b = 3;", 1 ); // 1, 3
```

Строка `"var b = 3;"` динамически создаёт локальную переменную `b`, которая затеняет внешнюю `b = 2`.

### `eval` в strict mode (безопасно)

```javascript
"use strict";
eval("var a = 5; function f() { }");
alert( a ); // ошибка, переменная не определена
```

В строгом режиме `eval` создаёт свой собственный `LexicalEnvironment`. Переменные уничтожаются после завершения `eval`.

### `with` — утечка в глобальный scope

```javascript
function foo(obj) {
  with (obj) {
    a = 2;
  }
}
var o1 = { a: 3 };
var o2 = { b: 3 };

foo( o1 );
console.log( o1.a ); // 2 (найдено свойство o1.a, значение изменено)

foo( o2 );
console.log( o2.a ); // undefined (в o2 нет свойства a)
console.log( a );    // 2 (утечка: создана глобальная переменная window.a)
```

При `foo(o2)` Движок ищет `a` в объекте `o2`. Не найдя, идёт вверх по Scope Chain. В нестрогом режиме глобальная область неявно создаёт переменную `a`.

### `new Function` — Scope = `window`

```javascript
var a = 1;
function getFunc() {
  var a = 2;
  var func = new Function('', 'alert(a)');
  return func;
}
getFunc()(); // 1, из window
```

Если бы функция была создана как `function() { alert(a); }`, она бы вывела `2`. `new Function` игнорирует локальный `LexicalEnvironment` функции `getFunc`.

### `catch` — блочная область видимости

```javascript
try {
  undefined(); // незаконная операция!
} catch (err) {
  console.log( err ); // работает! выведет ошибку TypeError
}
console.log( err ); // ReferenceError: err is not defined
```

Идентификатор `err` существует только внутри блока `catch`.

## Важно / подводные камни / best practices

- **`var` не имеет блочной области видимости** — привязывается к ближайшей функции или глобальному scope. Использовать `let`/`const` для блочной области.
- **Неявное создание глобальных переменных** — присваивание без `var`/`let` в нестрогом режиме создаёт глобальную переменную. Решение: `"use strict"`.
- **`eval` и `with` — антипаттерны.** Загрязняют scope, отключают оптимизации движка, ломают минификацию. `with` запрещён в strict mode. Вместо `eval` — `new Function`.
- **Приоритет hoisting:** функции поднимаются раньше переменных. Смешивание `var` и `function` с одинаковыми именами ведёт к неожиданному поведению.
- **TDZ опасна:** `typeof` для переменной в TDZ выбросит `ReferenceError` (в отличие от необъявленной переменной).
- **`function declaration` внутри блоков `if`/`for`** — поведение нестандартизировано в pre-ES6 окружениях. Избегать.
- **Принцип наименьших привилегий** — идентификаторы следует ограничивать наименьшей возможной областью видимости для предотвращения коллизий и несанкционированного доступа.
- **`catch(err)` в IE8-** — механизм реализован с ошибкой, `err` может перезаписать глобальную переменную.
- **V8 оптимизация** — переменные замыкания, не используемые вложенными функциями, удаляются из памяти. При отладке могут быть недоступны.
- **`const` запрещает перезапись ссылки**, но свойства объекта можно мутировать.

## Карточки для повторения

#flashcards/javascript/scope

**Определения**

Что такое Scope (область видимости)?:::Набор правил для хранения переменных в некотором месте и для обнаружения этих переменных позднее по имени идентификатора

Что такое лексическая область видимости (Lexical Scope)?:::Область видимости, определённая во время разбора на лексемы; основана на том, где переменные и блоки были созданы в коде; фиксируется на этапе компиляции и не зависит от места вызова

Что такое динамическая область видимости (Dynamic Scope)?:::Модель, при которой scope определяется во время выполнения, основываясь на стеке вызовов (откуда функция вызвана), а не на вложенности в коде. В JS нет Dynamic Scope, но `this` функционально похож

Что такое Scope Chain?:::Цепочка вложенных LexicalEnvironment-объектов от текущего контекста до глобального; поиск переменной идёт снизу вверх и останавливается при первом совпадении

Что такое Shadowing (затенение)?:::Ситуация, когда внутренний идентификатор перекрывает одноимённый внешний; поиск в Scope Chain останавливается при первом совпадении

Что такое LexicalEnvironment?:::Скрытый внутренний объект, создаваемый при запуске функции; хранит аргументы, локальные переменные и вложенные функции

Что такое Scope?:::Скрытое системное свойство функции, ссылающееся на LexicalEnvironment места создания функции; неизменно

Что такое TDZ (Temporal Dead Zone)?:::Период от начала выполнения блока до строки инициализации `let`/`const`/`class`; обращение к переменной в TDZ → `ReferenceError`

Что такое Hoisting (поднятие)?:::Концептуальный процесс, при котором объявления переменных и функций перемещаются в верх области видимости на этапе компиляции, до выполнения кода

Что такое принцип наименьших привилегий (Principle of Least Privilege)?:::Принцип дизайна: выставлять наружу только минимально необходимое, прятать всё остальное; в JS — ограничивать идентификаторы наименьшей возможной областью видимости

Что такое IIFE?:::Immediately Invoked Function Expression — функция, оборачиваемая в скобки и вызываемая немедленно `(function() { ... })();`; создаёт изолированную область видимости

**Типы scope**

Какие три типа области видимости существуют в JavaScript?
?
1. Global Scope — самая внешняя область
2. Function Scope — создаётся каждой функцией
3. Block Scope — `let`/`const` привязываются к блоку `{ ... }`

К какому scope привязывается `var`?::К ближайшей функции (Function Scope) или к глобальному scope; блоки `if`, `for`, `while` для `var` не создают новой области

К какому scope привязываются `let` и `const`?::К ближайшему блоку `{ ... }` (Block Scope)

**Механизм**

Три компонента обработки JS-кода
?
1. Engine (Движок) — компиляция и выполнение
2. Compiler (Компилятор) — лексемы, AST, машинный код
3. Scope (Область видимости) — хранение и поиск идентификаторов

Алгоритм поиска переменной по Scope Chain
?
1. Проверить текущий LexicalEnvironment
2. Если не найдена — перейти по Scope во внешнее окружение
3. Повторять до глобального объекта (window)
4. Остановиться при первом совпадении (Shadowing)

Что такое LHS-поиск и RHS-поиск?
?
LHS (Left-Hand Side) — ищет контейнер переменной для присваивания
RHS (Right-Hand Side) — ищет значение переменной для чтения

Что происходит, если LHS-поиск не находит переменную в нестрогом режиме?::Глобальная область видимости неявно создаёт новую глобальную переменную (утечка)

Что происходит, если LHS-поиск не находит переменную в строгом режиме?::Выбрасывается `ReferenceError`

**Hoisting**

Какой приоритет у hoisting функций vs переменных?::Объявления функций поднимаются первыми, затем переменные. Повторные `var` игнорируются

Как поднимается `function declaration`?::Целиком (имя + тело) — можно вызвать до строки объявления в коде (Value hoisting)

Как поднимается `var`?::Только объявление с инициализацией `undefined` — до присваивания вернёт `undefined` (Declaration hoisting)

Как поднимаются `let` и `const`?::Поднимаются в scope, но не инициализируются — попадают в TDZ до строки объявления

Как поднимается Function Expression?::Как `var` — поднимается только имя со значением `undefined`; вызов до присваивания → `TypeError`

Что происходит при повторных объявлениях `var` с одинаковым именем?::Дублирующие объявления без инициализатора игнорируются; с инициализатором — значение переопределяется

Что происходит при повторных объявлениях `function` с одинаковым именем?::Последнее объявление перекрывает предыдущие

**eval, with, new Function**

Как `eval` влияет на лексическую область видимости?::В нестрогом режиме `eval` выполняет код в текущей области видимости, загрязняя её. В strict mode — создаёт свой изолированный LexicalEnvironment

Почему `eval` и `with` снижают производительность?::Движок отключает статические оптимизации поиска идентификаторов, так как не может предсказать состав Scope; минификаторы не могут безопасно переименовать переменные

Почему `with` опасен для scope?::При присваивании несуществующему свойству объекта LHS-поиск уходит по Scope Chain и создаёт глобальную переменную (утечка)

Куда ссылается Scope функции, созданной через `new Function`?::Всегда на глобальный объект `window`, игнорируя текущий LexicalEnvironment; доступ только к глобальным переменным

**let в цикле**

Как `let` решает проблему замыканий в цикле `for`?
?
1. Движок перепривязывает переменную — создаёт новую копию для каждой итерации
2. Новая переменная инициализируется значением с конца предыдущей итерации
3. Каждая коллбэк-функция замыкается на свой LexicalEnvironment со своей переменной

**Глобальный объект**

В чём разница `var` и `let` на глобальном уровне?
?
`var x = 'global'` → создаёт свойство `window.x`
`let y = 'global'` → НЕ создаёт свойства на `window`; `this.y === undefined`

**Edge cases**

Безопасен ли `typeof` для переменных в TDZ?::Нет! `typeof varInTDZ` → `ReferenceError`. Обычно `typeof` безопасен для необъявленных переменных, но не для тех, что в TDZ

Edge case: `catch(err)` в IE8-::Механизм реализован с ошибкой — `err` может перезаписать одноимённую глобальную переменную вместо создания блочной области

Edge case: `function declaration` внутри блоков `if`/`for`::Исторически «вырывались» из блока; с ES6 стандартизированы как Block Scope, но поведение может отличаться в старых браузерах

Edge case: `const` и мутабельность::Запрещает перезапись ссылки (`TypeError`), но свойства объекта или массива можно свободно изменять

Edge case: V8 оптимизация и отладка замыканий::Если переменная из внешнего LexicalEnvironment не используется вложенными функциями, V8 удаляет её; при отладке — переменная не определена

**Best practices**

Какой механизм заменяет `eval` для безопасного выполнения динамического кода?::Использовать `new Function('args', 'code')` — Scope = `window`, нет доступа к локальным переменным, безопасно для минификации
