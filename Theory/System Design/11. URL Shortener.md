## Суть и контекст

URL Shortener — сервис сокращения длинных URL в короткие уникальные ссылки. При переходе по короткой ссылке происходит redirect на оригинальный URL. Требует генерации уникальных идентификаторов, хранения маппинга в базе данных и выбора стратегии перенаправления (301 vs 302).

---

## Ключевые идеи

- Back-of-the-envelope: 100 млн URL/день, 1160 RPS запись, 11 600 RPS чтение, 365 TB за 10 лет
- Два подхода генерации shortURL: Hash + Collision resolution и Base 62 Conversion
- Base 62 использует 62 символа (0-9, a-z, A-Z); 62^7 ≈ 3.5 триллиона — достаточно для 365 млрд записей
- 301 Redirect — permanent, кэшируется браузером (снижение нагрузки). 302 Redirect — temporary, каждый клик проходит через сервис (аналитика)
- Архитектура: Load Balancer → Web Servers → Cache → Database
- Хранение в реляционной БД, не в хэш-таблице в памяти (365 TB не поместятся в RAM)

---

## Определения и термины

**URL Shortener** — сервис, преобразующий длинный URL в короткую уникальную ссылку. При переходе по shortURL выполняется HTTP redirect на оригинальный longURL.

**301 Redirect (Permanent Redirect)** — HTTP ответ, указывающий, что запрошенный URL был окончательно перемещен. Браузер кэширует этот ответ и в дальнейшем направляет запросы сразу на новый URL.

**302 Redirect (Temporary Redirect)** — HTTP ответ, указывающий, что URL временно перемещен. Последующие запросы по-прежнему будут сначала отправляться на сервис сокращения URL.

**Base 62 Conversion** — метод преобразования уникального числового ID в строку, использующую 62 символа (0-9, a-z, A-Z).

**Hash + Collision Resolution** — метод генерации shortURL с использованием хэш-функции (MD5/SHA-1). Берутся первые 7 символов хэша. При коллизии к строке добавляется предопределённая последовательность до достижения уникальности.

## Как устроено

### Back-of-the-envelope расчёт

1. Создание: 100 млн URL в день
2. Запись (RPS): 100 000 000 / 24 / 3600 = ~1160 RPS
3. Чтение (RPS, соотношение 10:1): 1160 × 10 = 11 600 RPS
4. Записей за 10 лет: 100 млн × 365 × 10 = 365 млрд записей
5. Память (при 100 байтах на URL): 365 млрд × 100 байт = 365 TB

### API

```
POST /api/v1/data/shorten   — принимает longURL, возвращает shortURL
GET  /api/v1/shortUrl/{shortUrl} — возвращает longURL или выполняет 301/302 Redirect
```

### Создание shortURL (Base 62)

1. Сервер получает longURL
2. Проверяет БД: если longURL уже есть, возвращает существующий shortURL
3. Если нет — генератор выдаёт новый глобально уникальный числовой ID (например, 2009215674938)
4. ID переводится в Base 62 (значение zn9edcu)
5. Пара ID, shortURL, longURL сохраняется в РСУБД

### Архитектура URL Shortener (redirect flow)

1. Пользователь переходит по tinyurl.com/zn9edcu
2. Запрос попадает в Load Balancer
3. Load Balancer направляет на Web Servers
4. Веб-сервер делает запрос в Cache
5. Если промах (miss) — запрос в Database
6. Найденный longURL возвращается пользователю через redirect

---

## Детали и нюансы

- **Длина shortURL**: для поддержки 365 млрд URL требуется минимум 7 символов, так как 62^7 ≈ 3.5 триллиона > 365 млрд
- **Hash + Collision resolution**: MD5/SHA-1 дают длинную строку; берутся первые 7 символов. При коллизии добавляется предопределённая последовательность до уникальности — это вызывает затратные запросы в БД при проверке
- **Предсказуемость ID**: если ID инкрементируется на 1, злоумышленнику легко предсказать следующий shortURL — угроза безопасности
- **Зависимость от ID генератора**: Base 62 требует надёжного Distributed Unique ID Generator для глобальной уникальности

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|-------------------|
| Hash + Collision resolution | Фиксированная длина, не требует ID генератора | Коллизии, затратные проверки в БД | Когда нет распределённого ID генератора |
| Base 62 Conversion | Нет коллизий, прямая конвертация | Зависит от ID генератора, длина пропорциональна размеру ID | При наличии надёжного Distributed Unique ID Generator |
| 301 Redirect (Permanent) | Снижение нагрузки на серверы (кэш в браузере) | Нет аналитики — повторные клики не проходят через сервис | Когда нагрузка критичнее аналитики |
| 302 Redirect (Temporary) | Каждый клик проходит через сервис — аналитика | Нагрузка на серверы при каждом переходе | Когда критически важна аналитика (сбор каждого клика) |

```
POST /api/v1/data/shorten

GET /api/v1/shortUrl/zn9edcu
  Response: HTTP 301/302 → Location: https://example.com/very/long/path
```

## Важно / подводные камни / best practices

- **301 vs 302**: 301 для снижения нагрузки (кэшируется), 302 для аналитики (каждый клик проходит через сервис)
- **Хранение в реляционной БД**, не в хэш-таблице в памяти — ОЗУ ограничена и слишком дорога для 365 TB данных
- **Предсказуемость ID** при инкрементальной генерации — угроза безопасности; необходима защита
- **Base 62 зависит от Distributed Unique ID Generator** — надёжность генератора критична для всей системы
