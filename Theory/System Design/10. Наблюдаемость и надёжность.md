## Суть и контекст

Наблюдаемость (Observability) — свойство системы, позволяющее отслеживать её внутреннее состояние по внешним выходным данным (метрикам, логам, трейсам) для предотвращения ситуации "чёрного ящика" при инцидентах. Надёжность обеспечивается через circuit breaker, retry, graceful degradation и health checks.

---

## Ключевые идеи

- Три столпа наблюдаемости: метрики, логи, трейсы — связаны через trace_id
- Измерять процентили (p50, p95, p99), а не среднее — среднее скрывает outliers
- Метрики (SLI) управляют алертингом: алерт при пересечении SLO
- Structured logging: JSON с обязательными полями (request_id, trace_id, user_id)
- Distributed tracing: trace_id проходит через всю цепочку микросервисов
- Circuit breaker — "предохранитель" от каскадных сбоев
- Graceful degradation — система работает в ограниченном режиме вместо полного падения
- Retries опасны без лимита и идемпотентности — failure amplification

---

## Определения и термины

**Метрики (Metrics)** — количественные показатели состояния системы. Уровни: Host level (CPU, RAM, disk I/O), Aggregated level (производительность БД/кэша), Key business metrics (DAU, revenue). Технические: RPS, доля ошибок, задержка (p50/p95/p99), насыщение (saturation), глубина очередей, cache hit rate, пул соединений БД.

**Логи (Logs)** — подробные записи о событиях. Требуют структурированного формата (structured logging) с полями: request_id, trace_id, user_id, order_id, понятные коды ошибок.

**Трейсинг (Distributed Tracing)** — отслеживание пути одного запроса через цепочку микросервисов, очередей и БД. Показывает, где потрачено время или возникла ошибка.

**Trace ID** — уникальный идентификатор, проходящий через весь путь запроса. **Span** — отдельный логический блок работы внутри одного сервиса, привязанный к trace_id.

**Задержка p50, p95, p99** — время обработки в процентилях. p95 = 200 мс означает: 95% запросов быстрее 200 мс. Используются вместо среднего, которое скрывает "длинный хвост" (long tail).

**Saturation (Насыщение)** — уровень исчерпания ресурсов (CPU, RAM, disk, network).

**Health checks** — балансировщик опрашивает серверы и исключает unhealthy из пула.

**Circuit Breaker (Предохранитель)** — при массовой деградации нижестоящего сервиса временно прекращает отправку запросов и перенаправляет на fallback-логику. Как электрический предохранитель: при коротком замыкании разрывает цепь.

**Graceful degradation (Мягкая деградация)** — система работает в ограниченном режиме вместо полного падения при отказе некритичных подсистем.

**Retries (Повторные попытки)** — повторная отправка при transient failures. Допустимы ТОЛЬКО с лимитом попыток и ТОЛЬКО для идемпотентных операций.

**Инструменты:**
- **Prometheus, Grafana** — агрегация и визуализация метрик в дашборды
- **ELK (Elasticsearch + Logstash + Kibana)** — полнотекстовый поиск и визуализация логов
- **OpenTelemetry, Jaeger** — распределённый трейсинг

## Как устроено

### Распределённый трейсинг — процесс

1. API-шлюз при входящем запросе генерирует глобальный trace_id
2. Передаёт trace_id в заголовке первому микросервису
3. Сервис выполняет работу, создаёт локальный span (отметки старта и завершения), логирует с trace_id
4. При вызове БД или Kafka — trace_id передаётся в метаданных
5. Jaeger собирает все span по trace_id и визуализирует timeline

### Sidecar для сбора телеметрии (Service Mesh)

1. К каждому микросервису прикрепляется sidecar-контейнер
2. Sidecar перехватывает весь сетевой трафик
3. Автоматически извлекает метрики, логи, управляет Circuit Breaker
4. Данные отправляются в Service Mesh control plane

### Алертинг

1. Prometheus собирает метрики (SLI) со всех узлов
2. SLI (например, p95 latency) выходит за порог SLO
3. Система фильтрует случайный шум
4. Генерируется алерт с runbook-инструкцией и описанием user impact

## Детали и нюансы

- **Среднее (mean) маскирует проблемы**: 1% запросов может занимать 10x больше, но среднее нормальное. Нужны процентили. Алгоритмы T-digest или Forward decay вычисляют процентили эффективно
- **Каскадный сбой**: бесконечное ожидание ответа от зависшего сервиса блокирует потоки и память. Обязательны **жёсткие таймауты** на все внешние вызовы
- **Blind retries** — антипаттерн: если сервис упал из-за перегрузки, дублирование запросов умножает нагрузку (failure amplification). Ретраи только для transient, с лимитом, только идемпотентные
- **SPOF**: один балансировщик, одна БД или один регион — отказ = полная недоступность

---

## Сравнения и trade-offs

| Столп | Что даёт | Инструменты |
|-------|---------|------------|
| Метрики | Количественное состояние (RPS, latency, errors) | Prometheus, Grafana |
| Логи | Детализация событий, расследование ошибок | ELK (Elasticsearch + Kibana) |
| Трейсы | Топология вызовов, где потрачено время | Jaeger, OpenTelemetry |

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| p50/p95/p99 | Показывает реальное распределение задержек | Дороже вычисления, чем mean |
| Mean latency | Просто вычислить | Маскирует long tail, бесполезно для outliers |
| Circuit breaker | Защита от каскадных сбоев | Сложность настройки порогов |
| Blind retries | Просто | Failure amplification при перегрузке |

## Примеры

### Structured logging (JSON)

```json
{
  "level": "error",
  "trace_id": "abc-123-def",
  "request_id": "req-456",
  "user_id": "user-789",
  "service": "order-service",
  "message": "insufficient stock",
  "error_code": "STOCK_001",
  "timestamp": "2026-03-19T10:00:00Z"
}
```

### Graceful degradation — примеры

- Сервис рекомендаций упал → отдавать базовую (fallback) ленту
- Поиск недоступен → показывать старый снимок кэша
- Push-уведомление не работает → не блокировать покупку, сохранить событие в очередь

## Важно / подводные камни / best practices

- **Не использовать mean** — только процентили (p50, p95, p99) для latency
- **Наблюдаемость с первого дня** — не afterthought. Без trace_id найти причину в цепочке из 5 сервисов невозможно
- **Таймауты обязательны** на все внешние сетевые вызовы
- **Retries только**: для transient, с лимитом, для идемпотентных операций
- **Алерты = user impact** — не реагировать на каждый чих, иначе "усталость от алертов"
- **Graceful degradation** — fallback вместо полного падения
- **Профилирование в production** — для расследования деградации без очевидных причин
