## Суть и контекст

Репликация и партиционирование — два фундаментальных механизма распределённых баз данных, почти всегда применяемых вместе (Shared-nothing архитектура). Репликация хранит копии данных на нескольких машинах для доступности и масштабирования чтения. Партиционирование разбивает большую БД на части для масштабирования объёма и вычислений. Узел может быть одновременно лидером для партиции A и фолловером для партиции B.

---

## Ключевые идеи

- Репликация: снижение задержки (географическая близость), high availability, масштабирование чтения
- Три модели: single-leader, multi-leader, leaderless (Dynamo-style)
- Синхронная репликация надёжнее, но блокирует запись; на практике используется semi-synchronous
- Replication lag приводит к проблемам: read-after-write, monotonic reads, consistent prefix reads
- Failover опасен: split brain, потеря нереплицированных записей, ложные срабатывания
- Партиционирование: по диапазону ключей (range) или по хэшу (hash)
- Кворумы (w + r > n) гарантируют пересечение наборов записи и чтения
- ZooKeeper используется для маршрутизации запросов к нужной партиции

## Определения и термины

**Репликация (Replication)** — хранение копии одних и тех же данных на нескольких машинах, объединённых по сети. Применяется для снижения задержки, повышения доступности и масштабирования чтения.

**Single-leader (Master-Slave / Active-Passive)** — один узел (лидер) принимает все записи, сохраняет их и пересылает фолловерам в виде журнала репликации (replication log).

**Multi-leader (Master-Master / Active-Active)** — более одного узла могут принимать записи. Каждый лидер одновременно фолловер по отношению к другим.

**Leaderless (Dynamo-style)** — отсутствует концепция лидера: любой реплике разрешено принимать записи от клиентов.

**Synchronous replication** — лидер дожидается подтверждения от фолловера перед сообщением об успехе клиенту.

**Asynchronous replication** — лидер отправляет изменения, но не дожидается ответа.

**Replication lag** — задержка между записью на лидере и её отражением на фолловере. Приводит к eventual consistency.

**Read-after-write consistency (Read-your-writes)** — гарантия: пользователь всегда видит обновления, которые сам только что отправил.

**Monotonic reads** — гарантия: пользователь не увидит перемещения времени "вспять" при последовательных чтениях.

**Consistent prefix reads** — гарантия: последовательность записей читается в том же порядке, в каком была выполнена.

**Failover** — процесс обработки сбоя лидера: фолловер становится новым лидером, клиенты перенастраиваются.

**Split brain** — два узла одновременно считают себя лидером. Данные портятся. Защита: STONITH (Shoot The Other Node In The Head).

**Партиционирование (Sharding)** — разделение большой БД на меньшие подмножества (партиции/шарды), назначаемые разным узлам. Цель: распределить данные по дискам и нагрузку по процессорам.

**Key range partitioning** — ключи сортируются, каждая партиция владеет диапазоном от min до max.

**Hash partitioning** — к ключу применяется хэш-функция, партиция владеет диапазоном хэшей.

**Scatter/gather** — запрос отправляется во все партиции параллельно, результаты объединяются (при локальных вторичных индексах).

**Rebalancing** — перемещение данных с одного узла на другой при добавлении/удалении узлов.

**Request Routing (Service Discovery)** — механизм определения, к какому IP и порту подключиться для чтения/записи конкретного ключа.

## Как устроено

### Подключение нового фолловера

1. Создаётся консистентный снимок (snapshot) БД лидера
2. Снимок копируется на узел фолловера
3. Фолловер подключается к лидеру и запрашивает все изменения с момента снимка (по log sequence number или binlog coordinates)
4. Фолловер применяет отставание (catch up) и переходит в потоковый режим

### Автоматический Failover

1. Определение сбоя лидера по таймауту (например, 30 секунд без ответов)
2. Выбор нового лидера через выборы кворумом (реплика с самыми свежими данными)
3. Перенастройка: клиенты перенаправляются на нового лидера, старый лидер при возвращении понижается до фолловера

### Чтение/Запись в Leaderless с кворумом

1. Клиент параллельно отправляет запись на n реплик
2. Запись успешна при подтверждении от **w** узлов
3. Клиент параллельно читает с **r** реплик
4. Сравниваются версии (version numbers); используется самая новая, устаревшие обновляются (Read repair)

**Условие кворума:**
```
w + r > n
```
Где n — число реплик, w — кворум записи, r — кворум чтения. Гарантирует пересечение хотя бы в одном узле со свежими данными.

### Dynamic Partition Rebalancing

1. БД стартует с одной партиции
2. При достижении максимального размера партиция делится (split) на две половины
3. Каждая забирает 50% ключей
4. При уменьшении данных партиции сливаются (merge)

### Request Routing через ZooKeeper

1. ZooKeeper хранит маппинг партиций → узлы
2. Routing tier подписывается на ZooKeeper
3. При изменении узлов ZooKeeper уведомляет routing tier
4. Маршрутизатор перенаправляет запрос по актуальному адресу

## Детали и нюансы

- **Лог репликации** может быть: statement-based (SQL), write-ahead log (WAL), logical/row-based, trigger-based
- **Сбой + асинхронная репликация**: новый лидер может не иметь всех записей старого. Нереплицированные записи **отбрасываются** (discarded). Инцидент GitHub: повторное использование auto-increment ключей
- **Короткий таймаут failover**: ложные срабатывания при пиках нагрузки или глитчах сети, усугубляя нестабильность
- **Concurrent writes** в multi-leader / leaderless: нет единого порядка → конфликты. Last Write Wins (LWW) небезопасен из-за clock skew — данные молчаливо отбрасываются. Нужны version vectors
- **Sloppy quorum**: при недоступности узлов записи принимаются "чужими" узлами. Кворум r не гарантирует свежие данные, пока не выполнится hinted handoff
- **Hash partitioning** разрушает порядок ключей — невозможны эффективные range scans
- **Горячие ключи (Celebrity problem)**: хэш не помогает, если миллионы запросов к одному ID. Решение: добавление случайного числа (0–99) к ключу на уровне приложения
- **Semi-synchronous**: один фолловер синхронный, остальные асинхронные — практичный компромисс

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Single-leader | Простота, строгий порядок записей | Единая точка записи, уязвимость failover | Большинство систем, OLTP |
| Multi-leader | Быстрая запись локально, устойчивость к отказу дата-центра | Write conflicts, сложность разрешения | Несколько дата-центров |
| Leaderless | Нет единой точки отказа, высокая доступность | Сложность кворумов, eventual consistency | Dynamo, Cassandra, Riak |
| Синхронная репликация | Гарантия свежести на фолловере | Отказ фолловера блокирует всю БД | Один узел (semi-synchronous) |
| Асинхронная репликация | Не блокирует запись | Потеря данных при сбое лидера | По умолчанию |
| Key range partitioning | Эффективные range scans | Горячие точки (hot spots) | Сортированные данные, range queries |
| Hash partitioning | Равномерное распределение | Нет range scans | Точечные запросы по ключу |
| Local secondary index | Быстрая запись (один узел) | Scatter/gather при чтении (tail latency) | Write-heavy |
| Global secondary index | Быстрое чтение (один узел) | Запись требует распределённых транзакций | Read-heavy |
| w=1, r=1 | Сверхнизкая задержка | Низкая консистентность | Когда допустим stale |
| w+r > n | Гарантия свежести | Задержка = самая медленная реплика | Строгая консистентность |

## Примеры

### Комбинация репликации и партиционирования

```
Node 1: [Partition 1 Leader] [Partition 2 Follower] [Partition 3 Follower]
Node 2: [Partition 2 Follower] [Partition 3 Leader]  [Partition 4 Follower]
Node 3: [Partition 1 Follower] [Partition 4 Leader]  [Partition 2 Leader]
Node 4: [Partition 3 Follower] [Partition 1 Follower] [Partition 4 Follower]
```

Узел одновременно лидер для одной партиции и фолловер для других.

### Аналогия Sloppy Quorum

"Если вы заперли дверь своего дома (узел недоступен), вы стучите к соседу и просите переждать на диване (соседний узел принимает запись). Когда находите ключи, сосед просит вас уйти домой и отдаёт вещи (hinted handoff возвращает данные на правильный узел)."

## Важно / подводные камни / best practices

- **Read-after-write**: читать свои данные с лидера; чужие — с фолловеров. Или читать с лидера 1 минуту после обновления
- **Monotonic reads**: маршрутизировать все запросы одного клиента на одну реплику (по хэшу user_id)
- **Multi-leader**: избегать конфликтов — все записи одного пользователя в один дата-центр
- **hash mod N** — антипаттерн: изменение N перераспределяет все ключи. Использовать consistent hashing или динамическое число шардов
- **Semi-synchronous** — один синхронный фолловер + остальные асинхронные (компромисс)
- **LWW (Last Write Wins)** — небезопасен из-за clock skew; нужны version vectors
- **Горячие ключи**: добавление случайного суффикса (0–99) к ключу на уровне приложения
- **STONITH** — защита от split brain, но при ошибке может выключить оба узла

Репликация (Replication):::Хранение копии одних и тех же данных на нескольких машинах по сети. Для снижения задержки, high availability и масштабирования чтения

Single-leader (Master-Slave):::Один лидер принимает все записи, пересылает фолловерам через replication log. Просто, строгий порядок, но единая точка записи

Multi-leader (Active-Active):::Несколько узлов принимают записи. Каждый лидер — фолловер для других. Быстрая запись локально, но write conflicts

Leaderless (Dynamo-style):::Любая реплика принимает записи. Нет единой точки отказа, но сложность кворумов и eventual consistency

Replication lag:::Задержка между записью на лидере и отражением на фолловере. Приводит к eventual consistency

Split brain:::Два узла одновременно считают себя лидером. Данные портятся. Защита: STONITH

Условие кворума для Leaderless::w + r > n, где n — реплики, w — кворум записи, r — кворум чтения. Гарантирует пересечение хотя бы в одном узле со свежими данными

Три проблемы replication lag
?
1. **Read-after-write** — пользователь не видит свои изменения
2. **Monotonic reads** — данные "перемещаются вспять" при чтении с разных реплик
3. **Consistent prefix reads** — нарушение порядка записей при чтении

Шаги автоматического Failover
?
1. Определение сбоя по таймауту (30 сек)
2. Выборы нового лидера кворумом (реплика с самыми свежими данными)
3. Перенастройка: клиенты → новый лидер, старый → фолловер

Как обеспечить Read-after-write consistency?::Читать данные, которые пользователь мог редактировать (свой профиль), строго с лидера. Остальные — с фолловеров. Или читать с лидера 1 минуту после обновления

Как обеспечить Monotonic reads?::Маршрутизировать все запросы одного клиента на одну реплику (по хэшу user_id)

Почему LWW (Last Write Wins) небезопасен?::Часы рассинхронизированы (clock skew), данные молчаливо отбрасываются. Нужны version vectors

Key range vs Hash partitioning
?
- **Range** — ключи сортируются, партиция владеет диапазоном. Эффективные range scans, но hot spots
- **Hash** — хэш-функция на ключ, партиция владеет диапазоном хэшей. Равномерно, но нет range scans

Local vs Global secondary index
?
- **Local** — индекс на том же узле, что и документ. Быстрая запись, scatter/gather при чтении
- **Global** — индекс сам шардирован по термину. Быстрое чтение, запись требует распределённых транзакций

Scatter/gather:::Запрос к локальным вторичным индексам: отправляется во все партиции параллельно, результаты объединяются. Риск tail latency

Что такое Sloppy quorum?::При недоступности узлов записи принимаются "чужими" узлами. Кворум r не гарантирует свежие данные, пока не выполнится hinted handoff

Как решить проблему горячих ключей при шардировании?::Добавление случайного суффикса (0–99) к ключу на уровне приложения. Хэш-функция не помогает, если миллионы запросов к одному ID

Semi-synchronous репликация — это ==один== синхронный фолловер + остальные ==асинхронные==

При сбое лидера + асинхронная репликация, нереплицированные записи ==отбрасываются (discarded)==

ZooKeeper в Request Routing хранит ==маппинг партиций на узлы== и уведомляет routing tier при изменениях
