# SRP и OCP

## Суть и контекст

SRP (Single Responsibility Principle) и OCP (Open-Closed Principle) — первые два принципа из набора SOLID. SRP определяет, как проводить границы между модулями: у каждого модуля должна быть ровно одна причина для изменения. OCP определяет, как расширять систему: новая функциональность добавляется через создание новых сущностей и их композицию со старыми, а не через изменение существующего кода. Оба принципа направлены на снижение стоимости изменений при эволюции бизнес-требований.

---

## Ключевые идеи

- У модуля должна быть только одна причина для изменения; весь код, который меняется по этой причине, должен быть собран в этом модуле.
- Принцип SRP предлагает проводить границы между модулями так, чтобы изменение в бизнес-правилах затрагивало один модуль.
- Чем выше связность (cohesion), тем строже модули следуют SRP и тем выше сфокусирован модуль на конкретной задаче.
- Модули должны быть открыты для расширения, но закрыты для изменения.
- Расширение функциональности должно происходить с помощью создания новых сущностей и композиции их со старыми, а не через изменение существующего кода.
- Нарушение OCP приводит к нарушению SRP.
- Следование SRP ведёт к максимальному повышению связности (cohesion).
- Замена прямого наследования на полиморфизм и композицию ослабляет зацепление модулей.
- Схема OCP: Client -> Abstract Server -> Server (вместо прямой связи Client -> Server).
- Just-in-time design — сущности и абстракции добавляются по мере необходимости, но не раньше.

## Определения и термины

**SRP (Single Responsibility Principle, Принцип единственной ответственности)** — у модуля должна быть только одна причина для изменения. Весь код, который меняется по этой причине, должен быть собран в этом модуле. Принцип предлагает проводить границы между модулями так, чтобы изменение в бизнес-правилах затрагивало один модуль.

**Причина изменения (Reason to change)** — обновление бизнес-правил, которое вынуждает менять код какого-либо модуля. Разные модули должны иметь разные причины для изменения.

**Связность (Cohesion)** — степень, в которой задачи некоторого модуля связаны друг с другом. Чем выше связность, тем строже модули следуют SRP, и тем выше сфокусирован модуль на конкретной задаче.

**OCP (Open-Closed Principle, Принцип открытости-закрытости)** — модули должны быть открыты для расширения, но закрыты для изменения. Расширение функциональности должно происходить с помощью создания новых сущностей и композиции их со старыми, а не через изменение существующего кода.

**Открытость для расширения** — функциональность модуля может быть дополнена с помощью других модулей при изменении бизнес-требований.

**Закрытость для изменения** — расширение функциональности модуля не должно приводить к изменениям в модулях, которые его используют.

**Божественный объект (God Object)** — сущность, которая отвечает за слишком много. Внутри скапливается неоправданно большое количество данных, затрудняется рефакторинг и изолированное тестирование. Является прямым нарушением SRP.

## Как устроено

### Соблюдение SRP: рефакторинг "Выделение класса"

Если класс содержит несколько причин для изменения, ответственности разделяются путём выделения отдельных классов. Зависимость между ними устанавливается через интерфейс и инъекцию в конструктор. После рефакторинга каждый класс имеет только одну причину для изменения.

### Соблюдение OCP через полиморфизм (замена instanceof)

1. Определить модуль, в котором появляется множество проверок типов `instanceof`.
2. Создать интерфейс-абстракцию, задающий контракт поведения (например, метод `areaOf()` или `sendMessage()`).
3. Реализовать интерфейс во всех дочерних сущностях (`Rectangle`, `Circle` или `SmsSender`, `PushSender`).
4. В вызывающем коде заменить зависимость от конкретных типов на массив объектов абстрактного типа (например, `AreaCalculatable[]`).
5. Заменить `if (shape instanceof Rectangle)` на полиморфный вызов `shape.areaOf()`.

После этого добавление новой сущности не потребует изменений в вызывающем коде.

### Паттерн "Стратегия" для соблюдения OCP

1. Вынести выбор формата/алгоритма из основного класса.
2. Создать словарь `formatters`, где ключом является тип конфигурации (`ReportTypes.Html`), а значением — абстрактная фабрика или класс.
3. Написать функцию маршрутизации `formatterStrategy(reportType)`, которая динамически возвращает нужную сущность из словаря.
4. При добавлении нового алгоритма пополнить словарь новой строкой без изменения управляющей конструкции.

### Схема зависимости по OCP

Вместо прямой связи `Client -> Server` вводится промежуточная абстракция:

```
Client -> Abstract Server -> Server
```

Клиент зависит от абстракции, а конкретная реализация подставляется через композицию.

## Детали и нюансы

- **Преждевременная оптимизация по SRP:** образование лишних абстракций, усложнение программы. Выделить зоны ответственности сущностей на старте бывает трудно. Применять SRP стоит тогда, когда вносить изменения в нарушающий класс становится дорого.
- **Попытка закрыть систему на 100% от любых изменений:** люди плохо умеют прогнозировать. Всегда есть изменения, которые невозможно внедрить без изменения существующего кода. Использование OCP ведёт к появлению большего количества сущностей и интерфейсов. В маленьких приложениях это превращается в генерацию бесполезного бойлерплейт-кода.
- **Нарушение OCP приводит к нарушению SRP:** если при расширении приходится менять существующий модуль, у этого модуля появляется дополнительная причина для изменения.
- **Смешение архитектурных слоёв (MVC)** нарушает SRP: приводит к появлению "fat controllers" и к попаданию бизнес-логики в модель.

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Строгое следование SRP с ранним выделением классов | Модули сфокусированы, легко тестировать изолированно | Образование лишних абстракций, усложнение программы на старте | Когда вносить изменения в нарушающий класс становится дорого |
| Полиморфизм (замена instanceof) для OCP | Добавление новых сущностей не требует изменения вызывающего кода, ослабляет зацепление | Больше интерфейсов и сущностей | Когда часто добавляются новые типы/варианты поведения |
| Паттерн "Стратегия" для OCP | Динамический выбор алгоритма через словарь, заменяет if/else | Дополнительный уровень абстракции | Когда алгоритм/формат выбирается в рантайме и набор вариантов расширяется |
| Божественный объект (God Object) | Всё в одном месте на начальном этапе | Затруднён рефакторинг и изолированное тестирование, неоправданно большое количество данных | Никогда (антипаттерн) |

## Примеры

### Пример 1 — Нарушение SRP и исправление через "Выделение класса"

**До рефакторинга (нарушение SRP):**

```typescript
class Person {
  name: string
  phone: string
  officeCode: string

  constructor(name: string, phone: string, officeCode: string) {
    this.name = name; this.phone = phone; this.officeCode = officeCode
  }

  phoneNumberOf(): string {
    return `${this.phone} доб. ${this.officeCode}`
  }
}
```

**После рефакторинга:**

```typescript
interface IPhoneNumber {
  phone: string; officeCode: string; valueOf(): string
}

class PhoneNumber implements IPhoneNumber {
  phone: string; officeCode: string
  constructor(phone: string, officeCode: string) {
    this.phone = phone; this.officeCode = officeCode
  }
  valueOf(): string {
    return `${this.phone} доб. ${this.officeCode}`
  }
}

class Person {
  name: string
  phoneNumber: IPhoneNumber
  constructor(name: string, phoneNumber: IPhoneNumber) {
    this.name = name; this.phoneNumber = phoneNumber
  }
  phoneNumberOf(): string {
    return this.phoneNumber.valueOf()
  }
}
```

Форматирование номера делегируется экземпляру `PhoneNumber` через зависимость в конструкторе. Класс `Person` теперь имеет только одну причину для изменения.

### Пример 2 — Нарушение OCP через instanceof

```typescript
class Notifier {
  constructor(private api: SmsSender | PushSender | EmailSender) {}

  notify(): void {
    const message = 'Some user notification';

    if (this.api instanceof SmsSender) {
      this.api.sendSms(message)
    } else if (this.api instanceof PushSender) {
      this.api.sendPush(message)
    } else if (this.api instanceof EmailSender) {
      this.api.sendEmail(message)
    }
  }
}
```

Жёсткая привязка к конкретным классам. При добавлении нового отправителя (`DoveSender`) потребуется изменять код метода `notify`.

### Пример 3 — Исправление OCP через полиморфизм

```typescript
interface Sender {
  sendMessage(message: MessageText): void
}

class SmsSender implements Sender {
  sendMessage(message: MessageText) { /* ... */ }
}

class Notifier {
  constructor(private api: Sender) {}

  notify(): void {
    const message = 'Some user notification';
    this.api.sendMessage(message);
  }
}
```

Вызывающий класс зависит от абстрактного интерфейса `Sender`. Добавление новой сущности `DoveSender implements Sender` не потребует изменений в `Notifier`.

### Пример 4 — Шаблон "Стратегия"

```typescript
function formatterStrategy(reportType: ReportTypes) {
  const formatters = {
    [ReportTypes.Html]: HtmlFormatterFactory,
    [ReportTypes.Txt]: TxtFormatterFactory,
    default: TxtFormatterFactory
  }
  return formatters[reportType] || formatters.default
}
```

Динамический выбор алгоритма через словарь. Заменяет if/else конструкции.

## Важно / подводные камни / best practices

- **Just-in-time design** — сущности и абстракции добавляются по мере необходимости, но не раньше. Аналогия с Бритвой Оккама: хороший дизайн — простой дизайн.
- **Антипаттерн: instanceof-проверки (switch over types)** — прямое нарушение OCP. При появлении нового типа требуется менять существующий код.
- **Антипаттерн: смешение архитектурных слоёв (MVC)** — нарушает SRP. Приводит к появлению "fat controllers" и к попаданию бизнес-логики в модель.
- **Антипаттерн: Божественный объект** — аккумулирует множество ответственностей, нарушая SRP. Затрудняет рефакторинг и изолированное тестирование.
- **Преждевременная оптимизация по SRP** — применять SRP стоит тогда, когда вносить изменения в нарушающий класс становится дорого, а не заранее.
- **Невозможно закрыть систему от всех изменений** — всегда есть изменения, которые невозможно внедрить без изменения существующего кода. В маленьких приложениях чрезмерное следование OCP превращается в генерацию бесполезного бойлерплейт-кода.
