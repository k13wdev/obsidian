# DIP, Dependency Injection и IoC

## Суть и контекст

Принцип инверсии зависимостей (DIP) — последний из пяти принципов SOLID, направленный на снижение зацепления (coupling) между модулями. DIP вводит абстракции между высокоуровневыми и низкоуровневыми модулями. Dependency Injection (DI) — конкретный паттерн реализации DIP, а Inversion of Control (IoC) — более широкий подход, частным случаем которого является DI.

---

## Ключевые идеи

- Высокоуровневые модули не должны зависеть от низкоуровневых; оба типа должны зависеть от абстракций
- Абстракции не должны зависеть от деталей, детали должны зависеть от абстракций
- DI реализуется тремя способами: через конструктор, через сеттер, через интерфейс
- DI-контейнер автоматизирует создание экземпляров и управление жизненным циклом зависимостей
- IoC — подход, при котором контроль за выполнением программы отдаётся фреймворку
- Антипаттерны "Контрол-фрик" и "Локатор служб" — основные нарушения DIP
- Раздувшийся конструктор сигнализирует о нарушении SRP

## Определения и термины

**Принцип инверсии зависимостей (DIP)** — высокоуровневые модули не должны зависеть от низкоуровневых; оба типа должны зависеть от абстракций. Абстракции не должны зависеть от деталей, детали должны зависеть от абстракций. Снижает зацепление модулей (coupling), делая систему менее хрупкой.

**Низкоуровневые модули** — модули, содержащие утилитарную функциональность (обращение к базе данных, запросы к серверу, рендеринг DOM-элементов). Изменение низкоуровневых модулей не должно требовать обновления высокоуровневых.

**Высокоуровневые модули** — модули, содержащие сложную, более абстрактную бизнес-логику (авторизация пользователей, валидация форм, отправка уведомлений). Достаточно абстрактны для переиспользования в разных проектах.

**Инверсия управления (Inversion of Control, IoC)** — подход, при котором контроль за выполнением программы отдаётся фреймворку, который знает, в какой момент и какую функцию надо вызвать. Цель IoC — сделать систему расширяемой. DI является частным случаем IoC.

**Инъекция зависимостей (Dependency Injection, DI)** — паттерн привязки зависимостей не через конкретный объект, а через интерфейс с возможностью подменять их на лету. Реализуется через конструктор, сеттеры или интерфейсно.

**DI-контейнер** — специальная сущность, автоматизирующая создание экземпляров нужных классов и помогающая в управлении их жизненным циклом. В TypeScript Inversify предоставляет API для создания контейнеров.

**Контрол-фрик (Control Freak)** — запах кода, при котором зависимости явно создаются внутри конструктора через ключевое слово `new`. Класс становится жёстко зацепленным, нарушаются принципы OCP, LSP, ISP и DIP. При тестировании требуется создание глобальных ресурсозатратных моков, засоряющих область видимости.

**Локатор служб (Service Locator)** — шаблон, который подключает зависимости в момент, когда они нужны (скрыто извлекает из глобального регистра). Нарушает инкапсуляцию. Код становится непрозрачным, IDE не может подсказать необходимые зависимости для инициализации класса. Допускается только при работе с legacy-кодом.

## Как устроено

### Внедрение зависимостей через конструктор (Constructor Injection)

1. Объявить типы зависимостей (интерфейсы) в параметрах конструктора класса (например, `constructor(private chair: Chair)`)
2. Зарегистрировать в DI-контейнере соответствие абстрактного типа конкретной реализации (`container.registerSingleton<Chair, WoodenChair>()`)
3. При инициализации приложения DI-контейнер автоматически анализирует типы, находит зарегистрированные классы и подставляет их экземпляры в конструктор

### Внедрение через сеттер (Property/Setter Injection)

1. Объявить в классе публичные опциональные поля для зависимостей (`public couch?: Couch`)
2. Инициализировать объект без аргументов (`new Room()`)
3. Присвоить зависимости напрямую в свойства объекта (`livingRoom.couch = new LeatherCouch()`)

### Внедрение через интерфейс (Method Injection)

1. Создать интерфейс с методами-инжекторами (`interface RoomBuilder { injectChair(dep: Chair): void }`)
2. Реализовать интерфейс в целевом классе, сохраняя переданные аргументы во внутренние свойства
3. Внутри функции-фабрики или строителя вызвать методы-инжекторы и передать экземпляры зависимостей

## Детали и нюансы

### Раздувшийся конструктор

Слишком большое количество зависимостей при инъекции через конструктор указывает на нарушение SRP — класс выполняет слишком много задач и требует дробления.

### Проблемы DI-контейнеров

Неграмотное использование DI-контейнеров приводит к тому, что логика взаимодействия компонентов оказывается "размазанной" по разным частям приложения. Появляются циклические зависимости. Поток управления становится неявным. Снижается тестируемость и увеличивается стоимость изменений.

### Ограничения Property Injection

Невозможно гарантировать наличие всех зависимостей в момент инициализации объекта. Зависимости должны быть строго опциональными, поля становятся `public`, что может нарушать инкапсуляцию.

### Схема терморегулятора (DIP)

До рефакторинга: стрелки идут напрямую от одного конкретного модуля к другому (высокое зацепление). После рефакторинга: между сущностями вводится прослойка в виде интерфейсов. Модули зависят не друг от друга, а направляют зависимости в абстракцию (низкое зацепление).

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Constructor Injection | Все зависимости видны явно, объект полностью готов после создания | Конструктор может раздуться (сигнал нарушения SRP) | Основной способ — для обязательных зависимостей |
| Property/Setter Injection | Гибкость, можно менять зависимости на лету (паттерн Стратегия) | Нет гарантии наличия зависимостей при инициализации, нарушение инкапсуляции (public поля) | Только для строго опциональных зависимостей |
| Method Injection | Явные методы-инжекторы, совместим с паттерном Фабрика | Больше кода, нужен отдельный интерфейс для инжекторов | Когда DI происходит через строитель или фабрику |
| DI-контейнер | Автоматизация создания экземпляров, управление жизненным циклом | Неявный поток управления, риск циклических зависимостей, "размазанная" логика | Крупные приложения с множеством зависимостей |
| Контрол-фрик (антипаттерн) | Простота написания | Жёсткое зацепление, нетестируемость, нарушение OCP/LSP/ISP/DIP | Никогда |
| Локатор служб (антипаттерн) | Подключение зависимостей по мере необходимости | Скрытые зависимости, нарушение инкапсуляции, непрозрачный код | Только для legacy-кода |

## Примеры

### Нарушение DIP — прямая зависимость от низкоуровневого модуля

```typescript
class MySqlConnection {/*...*/}

class Auth {
  connection: MySqlConnection

  constructor(connection: MySqlConnection) {
    this.connection = connection
  }

  async authenticate(login: string, password: string): Promise<AuthResult> {/*...*/}
}
```

Класс `Auth` (высокоуровневый) жёстко зависит от конкретной реализации `MySqlConnection` (низкоуровневый). При смене БД придется менять логику `Auth`.

### Исправление DIP — зависимость от абстракции

```typescript
interface DataBaseConnection {
  connect(host: string, user: string, password: string): void
}

class MySqlConnection implements DataBaseConnection {
  constructor() {/*...*/}
  connect(host: string, user: string, password: string): void {/*...*/}
}

class Auth {
  connection: DataBaseConnection

  constructor(connection: DataBaseConnection) {
    this.connection = connection
  }

  authenticate(login: string, password: string) {/*...*/}
}
```

Введена абстракция `DataBaseConnection`. `Auth` зависит только от интерфейса, что позволяет подставить любую БД (например, `DbMock` для тестов) без изменения `Auth`.

### Внедрение зависимостей через интерфейс (Method Injection)

```typescript
interface RoomBuilder {
  injectChair(dep: Chair): void
  injectCouch(dep: Couch): void
  injectTable(dep: Table): void
}

class Room implements RoomBuilder {
  private chair?: Chair
  private couch?: Couch
  private table?: Table

  injectChair(chair: Chair) {
    this.chair = chair
  }
// ...
}

function roomFactory(): Room {
  const room = new Room()
  room.injectChair(new WoodenChair())
  return room
}
```

Используются отдельные методы-инжекторы вместо конструктора или открытых свойств. Применяется совместно с паттерном Фабрика.

### Антипаттерн "Контрол-фрик"

```typescript
class Auth {
  // Обратите внимание на отсутствие аргументов у конструктора
  // и ключевое слово `new` внутри:
  constructor() {
    this.connection = new MySqlConnection()
  }
}
```

Зависимость от конкретного низкоуровневого класса (`MySqlConnection`) жёстко зашита внутри конструктора с помощью `new`. Класс невозможно протестировать без создания глобального мока соединения.

## Связи с другими темами

- Соблюдение DIP автоматически удовлетворяет [[03. SRP и OCP|OCP]] (замена БД не требует изменения модуля) и [[04. LSP и ISP|LSP]] (подстановка другой реализации не ломает систему)
- Если при внедрении зависимостей конструктор разрастается или появляются циклические зависимости — это прямой сигнал о нарушении [[03. SRP и OCP|SRP]]
- Зацепление (coupling) и связность (cohesion) — ключевые метрики, на которые влияет DIP: минимизирует зацепление и максимизирует связность — подробнее в [[02. Coupling, Cohesion и Композиция vs Наследование]]
- Паттерн "Наблюдатель" инвертирует контроль (IoC) так же, как обработчики событий (GUI, клик мыши) — фреймворк или наблюдаемый объект берёт на себя ответственность за вызов нужных методов у зависимостей — подробнее в [[06. Паттерны проектирования, антипаттерны и запахи кода]]
- Property Injection реализует паттерн [[06. Паттерны проектирования, антипаттерны и запахи кода|Стратегия]] — позволяет заменять зависимости во время работы программы

## Важно / подводные камни / best practices

- **Контрол-фрик** — зависимости создаются через `new` внутри конструктора. Нарушает OCP, LSP, ISP и DIP одновременно. Класс становится нетестируемым
- **Локатор служб** — скрывает зависимости, нарушает инкапсуляцию. Допустим только в legacy-коде
- **Раздувшийся конструктор** — сигнал нарушения SRP. Класс выполняет слишком много задач
- **Циклические зависимости** — признак неграмотного использования DI-контейнеров
- **Property Injection** — поля становятся public, нет гарантии наличия зависимостей. Использовать только для строго опциональных зависимостей
