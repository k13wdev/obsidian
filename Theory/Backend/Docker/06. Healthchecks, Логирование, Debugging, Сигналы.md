# 06. Healthchecks, Логирование, Debugging, Сигналы

## Суть и контекст

Мониторинг контейнеров включает healthchecks, сбор логов через logging drivers и инструменты отладки. Корректное завершение контейнеров требует понимания механизма SIGTERM/SIGKILL и проблемы PID 1.

---

## Ключевые идеи

- HEALTHCHECK в Dockerfile: в источниках не раскрыто детально (упоминаются только Liveness/Readiness probes на уровне Kubernetes)
- Logging driver перехватывает STDOUT/STDERR контейнера; `docker logs` работает только с драйверами `json-file` и `journald`
- `json-file` без лимитов бесконечно растёт → переполнение диска хоста
- `docker stop` = SIGTERM + grace period (10 с) + SIGKILL; `docker kill` = немедленный SIGKILL
- Если под PID 1 запущен shell-скрипт или sudo — SIGTERM не доходит до приложения
- `dumb-init` / `tini` — решение проблемы PID 1: проксируют сигналы и предотвращают зомби-процессы
- Exit code 137 = SIGKILL (128+9); Exit code 143 = SIGTERM (128+15)

---

## Определения и термины

**Logging driver (драйвер журналирования)** — механизм, перехватывающий всё, что приложение отправляет в STDOUT и STDERR, и перенаправляющий эти данные в заданное место (файл, системный демон, удалённый сервер).

**json-file** — драйвер журналирования Docker по умолчанию; сохраняет логи в структурированном JSON в локальной ФС хоста (`/var/lib/docker/containers/<id>/<id>-json.log`).

**SIGTERM** — POSIX-сигнал (Exit Code 143 = 128+15), предоставляющий процессу льготный период (grace period) для финализации, очистки ресурсов и штатного завершения.

**SIGKILL** — POSIX-сигнал (Exit Code 137 = 128+9), немедленно и принудительно прерывающий процесс на уровне ядра ОС. Процесс лишается возможности выполнить процедуры очистки.

**PID 1** — идентификатор корневого процесса в пространстве имён контейнера. Жизненный цикл контейнера жёстко привязан к PID 1; именно этот процесс получает сигналы от демона Docker.

**dumb-init / tini** — лёгкие системы инициализации процессов для запуска под PID 1; перехватывают системные сигналы, корректно проксируют их дочерним процессам приложения, предотвращают зомби-процессы.

---

## Как устроено

### Healthchecks

> В предоставленных источниках не раскрыто детально для Docker уровня. Упоминаются только Liveness/Readiness/Startup probes на уровне Kubernetes.

### Логирование контейнеров

**Доступные драйверы:** json-file (по умолчанию), syslog, journald, gelf, fluentd, splunk, none

**Алгоритм работы json-file:**
1. Процесс в контейнере пишет в STDOUT/STDERR
2. Демон Docker структурирует строку: добавляет поле лога, тип потока (`stdout`/`stderr`) и точную временную метку
3. Данные записываются в: `/var/lib/docker/containers/<container_id>/<container_id>-json.log`

**Конфигурация глобального драйвера** — в файле `/etc/docker/daemon.json`, требует перезапуска демона.

### Debugging контейнеров

**`docker inspect`** — клиент опрашивает REST API; возвращает JSON-документ с метаданными; физически данные хранятся в `config.json` и `hostconfig.json` в директории контейнера

**`docker stats`** — Docker API обращается к эндпоинту `/v1/containers/<id>/stats`, непрерывно опрашивает метрики cgroups (`blkio`, `cpu_usage`, `memory`) и транслирует JSON-поток

**`docker exec`** — создаёт новый (вторичный) процесс внутри уже работающих namespaces и cgroups целевого контейнера

**`docker top`** — выполняет `ps` в контексте пространств имён контейнера

**`nsenter`** — в источниках упоминается только в оглавлении, детальная механика не раскрыта.

### Сигналы и graceful shutdown

**`docker stop` (алгоритм):**
1. Демон Docker отправляет SIGTERM (в одном источнике упоминается SIG_HUP; в большинстве — SIGTERM) процессу с PID 1
2. Стартует grace period (по умолчанию 10 секунд) — время на закрытие сетевых сессий, сброс данных из RAM на диск
3. Если процесс с PID 1 всё ещё активен — Docker отправляет SIGKILL; контейнер → `Exited`

**`docker kill` (алгоритм):**
1. Демон немедленно отправляет SIGKILL (минуя фазу ожидания)
2. Процесс жёстко завершается ОС

## Детали и нюансы

- **`docker logs` работает только с `json-file` и `journald`** — при других драйверах (syslog, splunk) команда выдаёт ошибку; логи нужно читать нативными средствами
- **json-file без лимитов растёт бесконечно** — может переполнить диск хоста
- **Управление ротацией json-file через `logrotate`:** использовать директиву `copytruncate` (Docker держит дескриптор открытым; обычный `logrotate` ломает логирование)
- **Запись логов в файл внутри ФС контейнера** (напр. `/var/log/app.log`) — демон Docker не видит эти данные; `docker logs` возвращает пустоту
- **SIGHUP Docker-демону** = перезапуск всех контейнеров (НЕ переоткрытие логов)
- **Проблема PID 1:** если под PID 1 запущен `sudo` или shell-скрипт (shell-форма CMD) — SIGTERM отправляется shell/sudo, который игнорирует или не пробрасывает сигнал дочернему процессу; приложение убивается SIGKILL по истечении grace period

---

## Сравнения и trade-offs

| Logging Driver | Persistence | `docker logs` | Когда использовать |
|---------------|------------|---------------|-------------------|
| json-file | Локально на хосте | Да | Dev, малые объёмы (с лимитами ротации) |
| syslog | Системный журнал хоста | Нет | Интеграция с системным логированием |
| journald | systemd journal | Да | Хосты с systemd |
| fluentd | Внешний агрегатор | Нет | Централизованное логирование (ELK) |
| none | Нет | Нет | Отладка, контейнеры без логов |

| Команда | Сигнал | Grace Period | Когда использовать |
|---------|--------|-------------|-------------------|
| `docker stop` | SIGTERM → SIGKILL | 10 с (настраивается) | Production, нормальное завершение |
| `docker kill` | SIGKILL немедленно | Нет | Экстренное завершение |
| `docker kill --signal=USR1` | Произвольный сигнал | — | Reload конфига, rotate логов |

## Примеры

```bash
# Чтение логов с метками времени в реальном времени
docker logs -f -t streamtest
# -f: follow (непрерывный вывод); -t: временная метка

# Инспекция метаданных контейнера
docker inspect <container_id>

# Мониторинг ресурсов в реальном времени
docker stats

# Выполнение команды внутри работающего контейнера
docker exec -it <container_id> /bin/sh

# Просмотр процессов контейнера
docker top 106ead0d55af

# Отправка произвольного сигнала (напр. для reload конфига)
docker kill --signal=USR1 6b785f78b75e
```

```json
// Глобальная конфигурация logging driver в daemon.json
{
  "log-driver": "syslog",
  "log-opts": {}
}
// /etc/docker/daemon.json; требует перезапуска демона
```

```dockerfile
# Решение проблемы PID 1 через dumb-init
CMD ["/sbin/dumb-init", "/bin/bash", "/run.sh"]
# dumb-init: PID 1, принимает SIGTERM от Docker, пробрасывает дочернему процессу
```

## Важно / подводные камни / best practices

- **Всегда направлять логи в STDOUT/STDERR** — позволяет использовать Docker Logging Drivers; снимает с приложения управление файлами
- **Никогда не запускать stateful-контейнеры (БД) через `docker kill` или `docker rm -f`** — SIGKILL не даёт сбросить кэш → повреждение файлов
- **`dumb-init` или `tini` в production-образах** — корректная обработка сигналов PID 1, предотвращение зомби-процессов
- **Настраивать лимиты ротации для json-file** (`--log-opt max-size=10m --log-opt max-file=3`) — предотвращение переполнения диска
- **Exec-форма `CMD ["binary"]`** вместо Shell-форм — гарантирует, что приложение занимает PID 1
