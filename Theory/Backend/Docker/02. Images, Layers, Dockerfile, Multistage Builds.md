# 02. Images, Layers, Dockerfile, Multistage Builds

## Суть и контекст

Docker-образы строятся как стопка неизменяемых слоёв (layers) на базе Union Filesystem. Dockerfile — декларативный файл сборки, каждая инструкция которого создаёт новый слой. Multistage builds позволяют радикально уменьшить итоговый образ, отбрасывая инструменты сборки.

---

## Ключевые идеи

- Каждая инструкция `RUN`, `COPY`, `ADD` в Dockerfile создаёт новый read-only слой
- Контейнер добавляет поверх слоёв образа один тонкий read-write слой (Thin R/W Layer / upperdir)
- Copy-on-Write: файл из read-only слоя копируется в upperdir только при попытке изменения
- После Cache Miss кэш инвалидируется для ВСЕХ последующих инструкций до конца Dockerfile
- `COPY` предпочтительнее `ADD` в большинстве случаев
- `ENTRYPOINT` — неизменяемый бинарный файл; `CMD` — аргументы по умолчанию к нему
- Multistage: финальный образ содержит только артефакт сборки, без исходников и инструментов
- Shell-форма (`CMD node app.js`) запускает /bin/sh -c, что ломает обработку сигналов; нужна Exec-форма (`["node", "app.js"]`)

---

## Определения и термины

**Слой образа (Layer)** — файл, защищённый от записи, описывающий изменение состояния образа в сравнении с предыдущим слоем.

**Union File System (UnionFS)** — файловая система с каскадно-объединённым монтированием, позволяющая монтировать несколько файловых систем с перекрытием в общую иерархию, выглядящую как единая файловая система.

**OverlayFS** — реализация UnionFS, интегрированная в ядро Linux (начиная с 3.18). Компоненты: `lowerdir` (нижние read-only слои), `upperdir` (верхний writable слой, diff), `workdir` (внутренняя директория для атомарных операций), `merged` (объединённое представление).

**Copy-on-Write (CoW) / Copy-on-change** — механизм, при котором файл из read-only слоя (lowerdir) при необходимости изменения сначала копируется целиком в upperdir, а затем модифицируется там. Оригинал в lowerdir остаётся неизменным.

**Multistage build (многоступенчатая сборка)** — процесс сборки с использованием нескольких инструкций `FROM` в одном Dockerfile, позволяющий копировать артефакты из одной ступени в другую и отбрасывать ненужные инструменты сборки.

**Build Context (контекст сборки)** — директория и все её поддиректории и файлы, отправляемые демону Docker клиентом при запуске `docker build`.

## Как устроено

### Структура OverlayFS

```
merged (то, что видит контейнер — единая FS)
   ↑ объединение
upperdir (diff, R/W — writable слой контейнера)
workdir  (внутренняя директория для атомарных операций)
lowerdir (read-only слои образа, снизу вверх):
   └── Layer N (последняя инструкция RUN/COPY/ADD)
   └── Layer N-1
   └── ...
   └── Layer 1
   └── Base layer (FROM образ)
```

### Операции с файлами

**Чтение:** поиск начинается с `upperdir`. Если файла нет — запрос «проваливается» ниже сквозь `lowerdir`, пока файл не найден.

**Изменение (Copy-on-Write):** если процессу нужно изменить файл из `lowerdir`, ядро копирует его целиком в `upperdir`, затем вносит изменения. Оригинал в `lowerdir` остаётся неизменным.

**Удаление:** в `upperdir` создаётся специальный символьный файл-маркер (character special file с маркером `c`). В `merged` файл исчезает, но физически из `lowerdir` не удаляется и продолжает занимать место.

### Механизм Layer Caching

1. Docker читает инструкции из Dockerfile последовательно сверху вниз
2. Для каждой инструкции Docker проверяет локальный кэш: есть ли промежуточный образ с точно такой же инструкцией и таким же родительским слоем
3. Для `ADD` и `COPY` дополнительно вычисляется и сверяется контрольная сумма файлов из контекста сборки
4. **Cache Hit** → слой переиспользуется
5. **Cache Miss** → инструкция выполняется заново
6. **После Cache Miss кэш инвалидируется для ВСЕХ последующих инструкций** до конца Dockerfile

## Детали и нюансы

### Инструкции Dockerfile

| Инструкция | Назначение | Ключевые особенности |
|-----------|-----------|----------------------|
| `FROM` | Базовый образ | Первая инструкция; `AS name` для именования ступени в multistage |
| `RUN` | Выполнить команду | Создаёт новый слой; изменения фиксируются |
| `CMD` | Команда по умолчанию | Переопределяется аргументами `docker run`; допускается только одна (учитывается последняя) |
| `ENTRYPOINT` | Неизменяемый исполняемый файл | Не переопределяется аргументами `docker run`; только через `--entrypoint` |
| `COPY` | Копировать файлы из контекста | Все файлы получают владельца root; предпочтительнее ADD |
| `ADD` | Копировать + распаковывать | Автоматически распаковывает локальные `.tar`; может скачивать по URL |
| `ARG` | Переменная времени сборки | Доступна при `docker build --build-arg`; в рантайме контейнера недоступна |
| `ENV` | Переменная окружения | Доступна при сборке (как `$VAR`) и в рантайме контейнера |
| `EXPOSE` | Документировать порт | Только документация; фактически порт не открывает и не пробрасывает |
| `WORKDIR` | Рабочая директория | Для всех последующих `RUN`, `CMD`, `ENTRYPOINT`, `COPY`, `ADD`; создаётся если нет |
| `USER` | UID/GID процесса | Для последующих инструкций и процесса контейнера |
| `VOLUME` | Точка монтирования | Создаёт анонимный managed volume; нельзя указывать директорию хоста |

### CMD vs ENTRYPOINT

- **`ENTRYPOINT`** — жёстко зафиксированное ядро команды запуска
- **`CMD`** — гибкие параметры, передаваемые ENTRYPOINT (если пользователь не указал иные при `docker run`)
- Exec-форма: `["binary", "arg"]` — процесс получает PID 1, сигналы ОС работают корректно
- Shell-форма: `binary arg` — Docker оборачивает в `/bin/sh -c`, приложение получает PID не 1, сигналы не обрабатываются корректно

### COPY vs ADD

- **`COPY`** — просто копирует файлы; предсказуемо и прозрачно
- **`ADD`** — дополнительно: распаковывает локальные `.tar`, может скачивать по URL
- `ADD` по URL создаёт слой, который нельзя очистить → раздувает образ; риск MITM-атак
- Правильный способ загрузки по URL: `RUN curl -SL <url> | tar -xC /dest && rm -rf ...` (всё в одном слое)

### Ограничение на количество слоёв

Исторически AUFS имел лимит в 42 слоя, позже расширенный до 127. При превышении образ собрать невозможно.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Single-stage build | Простота | Большой образ (включает инструменты сборки, исходники) | Dev/прототипирование |
| Multistage build | Минимальный финальный образ, без исходников и инструментов | Сложнее Dockerfile | Production, CI/CD |
| `COPY` | Предсказуемо, безопасно | Не распаковывает архивы | Почти всегда |
| `ADD` | Распаковывает `.tar`, загружает URL | Непредсказуемое поведение, риск раздутия образа | Только для распаковки локальных `.tar` |
| `CMD` | Легко переопределить аргументы | Полностью перезаписывается командой `docker run` | Аргументы по умолчанию к ENTRYPOINT |
| `ENTRYPOINT` | Исполняемый файл фиксирован | Сложнее переопределить | Основной процесс контейнера |

## Примеры

```dockerfile
# FROM с именованием ступени
FROM node:7 AS base

# RUN — выполнить команду
RUN apt-get update && apt-get install -y git

# CMD — exec-форма
CMD ["supervisord", "-n"]

# ENTRYPOINT — exec-форма
ENTRYPOINT ["node", "app.js"]

# COPY — копировать файлы
COPY . /src

# ADD — URL (антипаттерн!)
ADD https://example.com/file.tar.gz /app/

# ARG — переменная времени сборки
ARG VERSION=1.0

# ENV — переменная окружения
ENV AP="/data/app" VERSION="0.6"

# EXPOSE — документировать порт
EXPOSE 8080/tcp

# WORKDIR — рабочая директория
WORKDIR /app

# USER — UID/GID процесса
USER 1000:1000

# VOLUME — точка монтирования
VOLUME ["/var/log"]
```

```dockerfile
# Правильный порядок для оптимизации кэша
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
# Антипаттерн: COPY . /app перед RUN pip install — инвалидирует кэш при любом изменении кода
```

```dockerfile
# Multistage build: сборка Maven + минимальный production образ
FROM maven:latest AS appserver
WORKDIR /usr/src/atsea
COPY pom.xml .
RUN mvn -B -f pom.xml dependency:resolve
COPY . .
RUN mvn -B package -DskipTests

FROM java:8-jdk-alpine AS production
WORKDIR /app
COPY --from=appserver /usr/src/atsea/target/AtSea.jar .
ENTRYPOINT ["java", "-jar", "/app/AtSea.jar"]
# Первая стадия >500МБ (исходники + Maven)
# Финальный образ содержит только AtSea.jar на базе alpine
```

## Важно / подводные камни / best practices

- **Файл удалённый в отдельной `RUN rm` НЕ уменьшает образ** — он физически остаётся в предыдущем слое; объединять операции создания и удаления в одну `RUN` команду
- **Shell-форма ломает сигналы** — всегда использовать Exec-форму `["binary", "arg"]` для CMD и ENTRYPOINT
- **Порядок инструкций критичен**: копировать файлы зависимостей (`requirements.txt`, `package.json`) до `COPY . .` — это сохраняет кэш слоя зависимостей при изменении кода
- **`--no-install-recommends`** при `apt-get` в Ubuntu/Debian — исключает необязательные пакеты, уменьшает размер
- **Никогда не запускать от root** — явно создавать пользователя через `RUN groupadd -r app && useradd -r -g app app` и указывать `USER app`
- **`COPY` вместо `ADD`** для всего, кроме распаковки локальных `.tar`; `ADD` по URL — антипаттерн

Компоненты OverlayFS
?
- lowerdir — read-only слои образа
- upperdir (diff) — writable слой контейнера
- workdir — внутренняя директория для атомарных операций
- merged — объединённое представление (то, что видит контейнер)

Алгоритм Layer Caching
?
1. Docker читает Dockerfile сверху вниз
2. Для каждой инструкции ищет совпадение в кэше (та же инструкция + тот же родительский слой)
3. COPY/ADD: дополнительно сверяет контрольную сумму файлов
4. Cache Hit → слой переиспользуется
5. Cache Miss → выполняется заново + все последующие инструкции тоже инвалидируются

Что происходит при удалении файла из lowerdir в контейнере?::В upperdir создаётся символьный файл-маркер (character special file с маркером c); в merged файл исчезает, но физически из lowerdir не удаляется и продолжает занимать место

В чём разница CMD vs ENTRYPOINT?
?
ENTRYPOINT — жёстко зафиксированный исполняемый файл; переопределяется только через --entrypoint
CMD — аргументы по умолчанию; полностью перезаписывается аргументами docker run
Вместе: ENTRYPOINT — ядро, CMD — параметры по умолчанию

В чём разница COPY vs ADD?
?
COPY — просто копирует файлы из контекста сборки
ADD — дополнительно: распаковывает локальные .tar, может загружать по URL
Best practice: всегда COPY, ADD только для локальных .tar

Почему Shell-форма CMD/ENTRYPOINT проблематична?::Docker оборачивает в /bin/sh -c → /bin/sh получает PID 1, приложение не получает сигналы ОС (SIGTERM и др.) корректно; нужна Exec-форма: ["binary", "arg"]

Edge case: `RUN dd if=/dev/zero of=/bigfile` + `RUN rm /bigfile` — уменьшит ли образ?::Нет. Файл физически остаётся в слое первой RUN; вторая RUN создаёт лишь маркер удаления в upperdir; образ только увеличивается

Как правильно организовать COPY для оптимизации кэша?::Сначала копировать только файл зависимостей (requirements.txt / package.json), установить зависимости (RUN pip install), затем копировать исходный код (COPY . .) — изменение кода не инвалидирует кэш зависимостей
