# 04. Volumes, Bind Mounts, tmpfs

## Суть и контекст

Docker предоставляет три механизма хранения данных вне writable-слоя контейнера: Docker Volumes (управляемые тома), Bind mounts (монтирование директорий хоста) и tmpfs mounts (in-memory). Все три решают проблему persistence — данные не исчезают при удалении контейнера.

---

## Ключевые идеи

- Данные в локальной (non-persistent) ФС контейнера уничтожаются при его удалении
- Docker Volumes — рекомендованный способ для production; их управляет Docker, путь генерируется автоматически
- Bind mounts — прямое монтирование директории хоста; зависят от структуры ФС хоста
- tmpfs — in-memory хранилище; данные исчезают при остановке контейнера; используется для безопасности и производительности
- Флаг `--mount` предпочтительнее `-v` / `--volume` — более читаемый, один механизм для всех типов
- `docker rm` без `-v` создаёт осиротевшие тома (orphan volumes)

---

## Определения и термины

**Том (Volume)** — точка монтирования в дереве каталогов контейнера, в которую смонтирована часть дерева каталогов хоста. Инструмент для сегментирования и совместного использования данных, жизненный цикл которых не зависит от отдельного контейнера.

**Bind mount volume (связанное монтирование)** — том, который указывает на определённое пользователем местоположение в файловой системе хост-ОС.

**Docker-managed volume (управляемый том)** — том, создаваемый демоном Docker в части файловой системы хоста, которая принадлежит и контролируется исключительно демоном Docker (Docker-managed space).

**tmpfs mount** — временное хранилище информации, использующее оперативную память хоста (RAM), обеспечивающее высокую скорость операций чтения/записи.

**Осиротевший том (Orphan volume)** — управляемый том Docker, который больше не используется ни одним контейнером, но продолжает занимать дисковое пространство.

---

## Как устроено

### Хранение данных по умолчанию (Thin R/W Layer)

- Непостоянные данные пишутся в Thin writable layer, контролируемый Storage Driver (AUFS, OverlayFS, BTRFS, Device Mapper)
- Слой создаётся при старте контейнера и жёстко связан с его жизненным циклом
- При удалении контейнера — все данные уничтожаются физически
- Пути на хосте: `/var/lib/docker/<storage-driver>/...` (Linux), `C:\ProgramData\Docker\windowsfilter\...` (Windows)

### Механика Bind mounts

- Абсолютный путь ФС хоста монтируется напрямую в ФС контейнера
- Содержимое монтируемой директории хоста **перекрывает (shadows)** любые файлы, заложенные в образ по тому же целевому пути
- Изменения видны немедленно как в контейнере, так и на хосте

### Механика Docker Volumes (управляемые тома)

1. Пользователь вызывает `docker volume create` или запускает контейнер с указанием нового тома
2. Docker создаёт физическую директорию на хосте: `/var/lib/docker/volumes/<имя_тома>/_data`
3. Если целевая директория внутри образа уже содержит файлы — Docker автоматически копирует их в свежесозданный том
4. Том монтируется в контейнер

### Механика tmpfs

- Файловая система размещается исключительно в оперативной памяти хоста
- Данные никогда не сбрасываются на диск
- При остановке контейнера данные стираются из памяти

---

## Детали и нюансы

- **`--volumes-from` race condition:** если контейнер A и контейнер B монтируют разные тома в один и тот же путь (напр. `/library/ss`), и создаётся контейнер C с `--volumes-from A --volumes-from B` — только один том будет успешно подключён, второй игнорируется
- **SELinux (RedHat/CentOS):** при использовании bind mounts процесс в контейнере получает `Permission denied`. Решение: суффикс `:z` (доступ для нескольких контейнеров) или `:Z` (эксклюзивный доступ); Docker применит правильный контекст безопасности SELinux через `chcon`
- **VOLUME в Dockerfile:** инструкция создаёт только анонимный managed volume; привязка к путям хост-ОС в Dockerfile запрещена из соображений переносимости и безопасности
- **Безопасность + tmpfs:** комбинация `--read-only + --tmpfs /tmp` реализует принцип минимальных привилегий — при компрометации процесса злоумышленник не может записать вредоносные скрипты на диск; данные в `/tmp` исчезают при остановке

---

## Сравнения и trade-offs

| Тип | Управляет | Persistence | Производительность | Когда использовать |
|-----|-----------|-------------|-------------------|-------------------|
| Thin R/W Layer | Docker (Storage Driver) | Нет | Ниже (через storage driver) | Только временные данные сборки |
| Docker Volume | Docker daemon | Да | Высокая | Production БД, persistent data |
| Bind mount | Пользователь / ОС хоста | Да | Высокая (прямой доступ) | Dev-окружение, монтирование конфигов |
| tmpfs | Ядро Linux (RAM) | Нет | Максимальная | Временные файлы, кэши, секреты в памяти |

| Флаг | Синтаксис | Читаемость | Поддержка Swarm |
|------|-----------|-----------|-----------------|
| `-v` / `--volume` | `host:container:options` | Низкая | Ограниченная |
| `--mount` | `key=value,...` | Высокая | Полная |

## Примеры

```bash
# Создание управляемого тома
docker volume create myvol

# Запуск с управляемым томом (формат --mount — предпочтительный)
docker container run -dit --name voltainer \
  --mount source=bizvol,target=/vol alpine
# Если bizvol не существует — Docker создаст его автоматически

# Запуск с Bind mount (формат -v)
docker run -d --name bmweb \
  -v ~/example-docs:/usr/local/apache2/htdocs/:ro \
  -p 80:80 httpd:latest
# :ro — read-only; процессы в контейнере не могут изменять файлы на хосте

# Контейнер с read-only ФС + tmpfs для временных файлов (безопасный шаблон)
docker run --read-only --tmpfs /tmp alpine sh -c 'echo "whatever" > /tmp/file'
# --read-only: запрет любой записи в корневую ФС контейнера
# --tmpfs /tmp: in-memory хранилище по пути /tmp

# Удаление осиротевших томов
docker volume prune

# Удаление контейнеров вместе с managed volumes (без orphan volumes)
docker rm -v $(docker ps -aq)
# -v удаляет managed volumes, если на них нет других ссылок; bind mounts не удаляет
```

## Важно / подводные камни / best practices

- **Никогда не хранить persistent data (БД, ценные логи) в writable-слое контейнера** — при пересоздании контейнера данные стираются
- **`--mount` вместо `-v`**: читаемее, предотвращает синтаксические ошибки, поддерживает специфические параметры драйверов и Swarm
- **Docker Volumes вместо Bind mounts в production**: управляемые тома абстрагируют контейнер от ФС хоста, поддерживают сторонние плагины (NFS, AWS S3, Azure, Pure Storage)
- **Паттерн Volume Container**: остановленный контейнер-обёртка, хранящий ссылки на тома; другие контейнеры получают доступ через `--volumes-from` — упрощает бэкапы и миграцию
- **`docker rm` всегда с `-v`**: предотвращает накопление orphan volumes
