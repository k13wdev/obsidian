# 03. Жизненный цикл контейнера, Управление ресурсами

## Суть и контекст

Контейнер проходит через строго определённые состояния (Created → Running → Paused/Exited → Removed). Управление ресурсами осуществляется через cgroups ядра Linux: CPU (shares, quota, pinning) и память (hard limit + OOM Killer).

---

## Ключевые идеи

- Состояние контейнера напрямую привязано к состоянию процесса с PID 1 внутри него
- `docker stop` = SIGTERM + тайм-аут + SIGKILL; `docker kill` = немедленный SIGKILL
- `docker pause` замораживает процесс через cgroup freezer без отправки сигналов
- CPU Shares — относительный приоритет (работает только при конкуренции); CPU Quota — жёсткий лимит в мкс
- Лимит памяти — жёсткий: превышение вызывает OOM Killer → SIGKILL → `Exited (137)`
- `--memory-swap` = RAM + Swap в сумме; `-1` отключает лимит Swap

---

## Определения и термины

**Control Groups (cgroups)** — механизм ядра Linux для агрегирования и разделения наборов задач и всех их будущих потомков в иерархические группы со специализированным поведением. Docker использует cgroups для установки лимитов на память, swap и процессорное время.

**CPU Shares (относительный вес процессора)** — значение, определяющее долю времени процессора, которую контейнер должен получить относительно других работающих контейнеров. По умолчанию пул share равен 1024. Применяется ядром только при возникновении конкуренции (contention) за процессорное время — не жёсткий лимит.

**CPU Quota** — жёсткий лимит времени процессора в микросекундах (us), которое процессы в cgroup могут использовать в течение заданного CFS-периода. Если `cpu.cfs_quota_us = 50000`, а `cpu.cfs_period_us = 100000`, контейнер ограничен ровно 50% времени одного ядра CPU.

**OOM Killer (Out-Of-Memory Killer)** — функция ядра Linux, отправляющая сигнал SIGKILL процессу при исчерпании разрешённой или доступной оперативной памяти. Защищает систему от падения.

**SIGTERM** — POSIX-сигнал, дающий процессу шанс очистить ресурсы и штатно завершить работу.

**SIGKILL** — POSIX-сигнал, немедленно и принудительно убивающий процесс.

## Как устроено

### Состояния контейнера и переходы

```
                   docker create
                        ↓
                    [Created]
                        ↓ docker start / docker run
                    [Running]  ←──────────────────────┐
                   ↙        ↘                         │
        docker pause      docker stop / docker kill    │
           ↓                  ↓                       │
        [Paused]          [Exited]   ←── PID 1 сам   │
           ↓ docker unpause    │        завершился    │
        [Running]              │                      │
                               │ docker start         │
                               └──────────────────────┘
                               │ docker rm
                               ↓
                           [Removed]
```

**Все состояния:** Created, Running/Up, Paused, Restarting, Exited/Stopped, Deleted

### Переходы состояний: команды и механизмы

**`docker create`** — инициализация файловой системы и параметров; состояние → Created

**`docker start` / `docker run`** — запуск основного процесса (PID 1); состояние → Running

**`docker pause`** — использует cgroup freezer: планировщик ОС запрещает выделять время процессу; сигналы процессу не отправляются; данные остаются в RAM; состояние → Paused

**`docker unpause`** — процесс размораживается в cgroups и продолжает работу с того же места; состояние → Running

**`docker stop`** — отправляет SIGTERM процессу PID 1 → тайм-аут (по умолчанию 10 или 30 секунд) → если не завершился, SIGKILL; состояние → Exited

**`docker kill`** — немедленный SIGKILL; процесс убивается без шанса на очистку; состояние → Exited

**PID 1 сам завершился** → контейнер автоматически → Exited

**`docker rm`** — удаление всех метаданных и слоёв файловой системы остановленного контейнера

### Управление ресурсами CPU

**CPU Shares:** контролируется через `cpu.shares` в cgroups. При конкуренции контейнеры получают CPU пропорционально:
- Контейнер A: 1024 shares, контейнер B: 512 shares → при 100% нагрузке A: 66%, B: 33%
- При простое контейнер с любым share может использовать 100% CPU

**CPU Quota (CFS):** жёсткий лимит — контейнер получает не более N мкс за каждые M мкс CFS-периода:
- `cpu.cfs_quota_us = 50000`, `cpu.cfs_period_us = 100000` → не более 50% одного ядра

**CPU Pinning:** жёсткая привязка контейнера к конкретным ядрам — устраняет задержки на контекстное переключение между ядрами

### Управление памятью

**Лимит памяти — жёсткий (hard limit):** в отличие от CPU, память нельзя «сжать» или «заторможенить». При превышении лимита → OOM Killer → SIGKILL → `Exited (137)`

**`--memory-swap`:** управляет суммарным объёмом RAM + Swap:
- `--memory 512m --memory-swap 768m` → 512 МБ RAM + 256 МБ Swap
- `--memory-swap=-1` → лимит на Swap отключён

---

## Детали и нюансы

- Ручные изменения в `/sys/fs/cgroup/...` применяются мгновенно, но при `docker stop` директория cgroup уничтожается; при `docker start` демон воссоздаёт её с исходными значениями — ручные изменения теряются
- Если лимит памяти превышает физическую RAM хоста — контейнер может использовать swap, как обычный Linux-процесс; при активном swap производительность критически падает
- В Kubernetes настоятельно рекомендуется **полностью отключать swap на узлах**
- При `restart=on-failure:5` и OOM → контейнер перезапускается 5 раз, каждый раз падая; в K8s это приводит к статусу `CrashLoopBackOff`
- `docker pause` через cgroup freezer — не SIGSTOP (который процесс может игнорировать); заморозка происходит прямо в планировщике ядра

## Сравнения и trade-offs

| Параметр | CPU Shares | CPU Quota (CFS) |
|----------|-----------|-----------------|
| Тип ограничения | Относительный приоритет | Жёсткий лимит |
| Когда работает | Только при конкуренции за CPU | Всегда |
| Единица | Доля от пула 1024 | Микросекунды (us) за период |
| Предсказуемость | Непредсказуемо при простое соседей | Точный кап |
| Когда использовать | Мягкий приоритет между сервисами | Строгое ограничение в production |

| Команда | Сигнал | Graceful? | Когда использовать |
|---------|--------|-----------|-------------------|
| `docker stop` | SIGTERM → (тайм-аут) → SIGKILL | Да | Production, нормальное завершение |
| `docker kill` | SIGKILL сразу | Нет | Экстренное завершение |
| `docker rm -f` | SIGKILL сразу + удаление | Нет | Принудительная очистка |

## Примеры

```bash
# Запуск с лимитами памяти и политикой перезапуска
docker run -ti --restart=on-failure:3 -m 200m --memory-swap=300m \
  progrium/stress --cpu 2 --io 1 --vm 2 --vm-bytes 128M --timeout 120s
# -m 200m: лимит физической памяти 200 МБ
# --memory-swap=300m: суммарно RAM+Swap = 300 МБ (то есть 100 МБ Swap)
# --restart=on-failure:3: до 3 перезапусков при падении
```

```bash
# Ограничение CPU через относительные веса (Shares)
docker run --rm -ti -c 512 progrium/stress \
  --cpu 2 --io 1 --vm 2 --vm-bytes 128M --timeout 120s
# -c 512: половина стандартного приоритета 1024; работает только при конкуренции
```

```bash
# Привязка контейнера к ядру 0 (CPU Pinning)
docker run -d --cpuset-cpus 0 --name ch6_stresser dockerinaction/ch6_stresser
# Форматы: 0,1,2 (список) или 0-2 (диапазон)
```

```bash
# Корректное удаление контейнера с томами
docker stop cass1
docker rm -vf cass1
# -v: удалить связанные managed volumes (предотвращает orphan volumes)
# -f: принудительно, для страховки
```

```bash
# Ручное изменение CPU shares через cgroups (применяется немедленно, но не сохраняется)
echo 512 > /sys/fs/cgroup/cpu/docker/dcbbaa763daf/cpu.shares
cat /sys/fs/cgroup/cpu/docker/dcbbaa763daf/cpu.shares
```

## Важно / подводные камни / best practices

- **`docker stop` перед `docker rm`**: SIGTERM гарантирует приложению время на безопасное сохранение состояния; `docker rm -f` = выстрел в затылок — может привести к порче файлов и незакрытым сессиям
- **`--rm` для краткосрочных контейнеров**: автоматически уничтожает контейнер при выходе, предотвращая накопление мёртвых контейнеров на диске
- **CPU Shares недостаточно для жёсткого капа**: при простое соседей контейнер с низким share всё равно утилизирует 100% CPU; для точного ограничения — CPU Quota
- **Orphan volumes**: при удалении контейнера без флага `-v` тома остаются на диске; использовать `docker rm -v` или периодически `docker volume prune`
- **Ручные изменения cgroups** исчезают при перезапуске контейнера — всегда задавать лимиты через `docker run` флаги
