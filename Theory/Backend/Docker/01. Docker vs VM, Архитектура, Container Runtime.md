# 01. Docker vs VM, Архитектура, Container Runtime

## Суть и контекст

Docker — платформа для упаковки, распространения и запуска приложений в изолированных процессах (контейнерах), использующих ядро хост-ОС напрямую. Применяется в DevOps, CI/CD, микросервисной архитектуре и cloud-native разработке.

---

## Ключевые идеи

- Контейнер — изолированный процесс хост-ОС, а не виртуальная машина; разделяет ядро хоста через namespaces и cgroups
- Виртуальная машина несёт «VM tax» — каждая Guest OS потребляет значительный объём RAM и CPU на системные процессы
- Docker Engine состоит из четырёх слоёв: dockerd → containerd → shim → runc
- Жизненный цикл контейнеров полностью отвязан от демона (daemonless containers): остановка dockerd не убивает запущенные контейнеры
- OCI (Open Container Initiative) определяет image-spec и runtime-spec — промышленный стандарт совместимости рантаймов
- Образ = класс (class) в ООП; контейнер = экземпляр (instance) этого класса во время выполнения

---

## Определения и термины

**Виртуальная машина (VM)** — эмуляция аппаратного обеспечения, использующая гипервизор для разделения физических ресурсов и запускающая собственную полноценную гостевую операционную систему (Guest OS) для каждого экземпляра.

**Контейнер (Container)** — изолированный процесс, выполняющийся в операционной системе хоста, потребляющий только ресурсы самого приложения без накладных расходов на дополнительную ОС. Вызванный к жизни образ Docker.

**Docker Image (Образ)** — неизменяемый пакет, содержащий файловую систему ОС, код приложения, библиотеки и все зависимости, необходимые для его запуска.

**Docker Daemon (dockerd)** — серверный процесс Docker, ожидающий запросов к API Docker и управляющий образами, контейнерами, сетями и томами.

**Docker Client (docker)** — инструмент командной строки (CLI) для взаимодействия и отправки инструкций демону Docker по протоколу HTTP.

**Registry (Реестр)** — удалённая платформа или служба для хранения и распространения образов Docker (самым крупным публичным реестром является Docker Hub).

**containerd** — высокоуровневая среда выполнения контейнеров (container supervisor), управляющая всем жизненным циклом контейнеров, включая загрузку образов (pull), создание сетевых интерфейсов, запуск и остановку.

**runc** — низкоуровневая среда выполнения, эталонная реализация OCI container-runtime-spec. CLI-обёртка над libcontainer, предназначенная исключительно для интерфейса с ядром ОС и непосредственного создания контейнеров.

**CRI (Container Runtime Interface)** — gRPC API, определяющий спецификации и библиотеки для интеграции различных сред выполнения контейнеров с компонентом kubelet в Kubernetes.

**namespaces** — функция ядра Linux для сокрытия глобальных системных ресурсов от контейнера. Типы: MNT, UTS, IPC, PID, NET.

**cgroups (Control Groups)** — функция ядра Linux для ограничения доступных ресурсов: CPU, RAM, диска и сети.

**OCI (Open Container Initiative)** — организация, определяющая промышленные стандарты image-spec и runtime-spec для совместимости контейнерных экосистем.

**shim** — процесс docker-containerd-shim, подхватывающий контейнер после завершения runc и удерживающий открытыми потоки STDIN/STDOUT.

## Как устроено

### VM vs Docker: механика изоляции

**VM:**
- Внизу: Physical Server → Host OS → Hypervisor
- Каждая VM содержит: виртуальные ресурсы → Guest OS kernel → приложения
- Гипервизор абстрагирует физическое железо, каждая ВМ требует собственной полноценной Guest OS

**Docker:**
- Внизу: Physical Server → Host OS → Docker layer
- Container 1, Container 2: только приложения и библиотеки
- Контейнеры напрямую разделяют единое ядро ОС без прослоек виртуализации

Контейнеры физически являются дочерними процессами в дереве процессов хостовой ОС (видно через `ps axlfww`).

### Механизм изоляции контейнеров

Docker использует две функции ядра Linux:
- **namespaces** — сокрытие глобальных системных ресурсов (MNT, UTS, IPC, PID, NET)
- **cgroups** — ограничение доступных ресурсов CPU, RAM, диска и сети

### Архитектура Docker Engine: 4 слоя

```
$ Docker client
      ↕  (REST API, HTTP)
Docker daemon (dockerd)
      ↓  (gRPC)
containerd (Container supervisor)
      ↓
   shim    shim    shim    ...
      ↓
   runc    runc    runc    ...
      ↓
Container  Container  ...
```

### Алгоритм запуска контейнера

1. **Docker Client** преобразует команду пользователя в API payload (REST) и выполняет POST-запрос к API endpoint демона (сокет `/var/run/docker.sock` на Linux или `\pipe\docker_engine` на Windows)
2. **Docker Daemon** принимает запрос и делает gRPC-вызов к containerd с инструкцией создать контейнер
3. **containerd** берёт Docker-образ, преобразует его в совместимый OCI-bundle и инструктирует runc
4. **containerd** делает fork нового процесса runc для этого контейнера
5. **runc** обращается к ядру Linux, создаёт нужные namespaces и cgroups, запускает процесс контейнера как свой дочерний процесс
6. Как только процесс стартует, **runc немедленно завершает свою работу** (exits)
7. Процесс **docker-containerd-shim (shim)** подхватывает контейнер, становится его родительским процессом, удерживает открытыми потоки STDIN/STDOUT и передаёт код завершения обратно демону

### CRI в Kubernetes

Чтобы kubelet мог работать с любыми рантаймами (containerd, CRI-O, Docker), рантайм должен реализовать два служебных gRPC-интерфейса:
- **ImageService** — для работы с образами
- **RuntimeService** — для управления жизненным циклом подов и контейнеров

---

## Детали и нюансы

- **Контейнер живёт ровно столько, сколько живёт его процесс с PID 1** — если процесс завершился, контейнер останавливается
- Минималистичные образы (например, Alpine Linux) занимают ~5 MB, стартуют за миллисекунды
- При перезаписи тега (например, `ubuntu:latest`) старый образ теряет тег и переходит в состояние **Dangling Image** (помечается как `<none>:<none>`), но остаётся на диске
- Windows-контейнеры требуют Windows-ядра, Linux-контейнеры требуют Linux-ядра — запуск контейнера другой платформы требует дополнительных легковесных VM (Docker Desktop через Hyper-V/WSL2)
- При изменении драйвера исполнения (с native/libcontainer на LXC) контейнеры, созданные одним драйвером, не запустятся в другом — конфигурации несовместимы

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Виртуальная машина | Полная изоляция, разные ОС, сильные security границы | VM tax (высокое потребление RAM/CPU), медленный старт (минуты) | Нужна полная изоляция ОС, разные ядра на одном хосте |
| Docker-контейнер | Лёгкий (МБ), быстрый старт (мс), разделяет ядро хоста, эффективное использование ресурсов | Разделяет ядро хоста (меньше изоляции), Linux-контейнеры только на Linux-ядре | Микросервисы, CI/CD, cloud-native приложения, одинаковые ОС |

## Примеры

```bash
# Запуск контейнера через CLI клиента
docker container run -it alpine:latest sh
# Флаги -it запускают контейнер в интерактивном режиме и привязывают терминал

# Просмотр иерархии запущенных процессов Docker Engine на хосте Linux
ps axlfww
# Демонстрирует, что контейнеры физически являются дочерними процессами хостовой ОС
```

```json
// Настройка Docker демона для безопасного TLS-подключения (daemon.json)
{
  "tlsverify": true,
  "tlscacert": "/etc/docker/ca.pem",
  "tlscert": "/etc/docker/server-cert.pem",
  "tlskey": "/etc/docker/server-key.pem",
  "hosts": ["tcp://node3:2376"]
}
```

```bash
# Ручной запуск демона Docker с переопределением среды выполнения на LXC
docker -d -H tcp://0.0.0.0:2375 -H unix:///var/run/docker.sock -e lxc
```

```proto
// Определение gRPC-интерфейса RuntimeService спецификации CRI
service RuntimeService {
    rpc Version(VersionRequest) returns (VersionResponse) {}
    rpc RunPodSandbox(RunPodSandboxRequest) returns (RunPodSandboxResponse) {}
    rpc CreateContainer(CreateContainerRequest) returns (CreateContainerResponse) {}
    rpc StartContainer(StartContainerRequest) returns (StartContainerResponse) {}
}
// Часть Protobuf-спецификации, которую должна реализовать CRI-совместимая среда
```

```bash
# Выполнение команды через CRI (crictl) для создания пода
POD_ID=$(crictl runp pod.yaml)
CONTAINER_ID=$(crictl create $POD_ID container.yaml pod.yaml)
crictl start $CONTAINER_ID
# Утилита crictl общается напрямую с CRI-совместимым рантаймом через сокет
```

## Важно / подводные камни / best practices

- **root в контейнере = root на хосте** (при отсутствии user namespaces) — всегда задавать непривилегированного пользователя через `USER <UID>` (например, `USER 1001`) в Dockerfile
- **Никогда не хранить данные** (БД, логи) в writable-слое контейнера — при удалении контейнера данные уничтожаются; использовать Docker Volumes
- **Избегать тега `:latest` в production** — тег может быть перезаписан другой версией в реестре; использовать точные теги (`:6.2.3-alpine`) или криптографические хеши (image digests / SHA256)
- Использовать флаг `--no-install-recommends` при `apt-get` в Dockerfile — уменьшает размер образа и количество слоёв
- **Daemonless containers**: обновление dockerd в production не убивает контейнеры — shim удерживает их живыми

Шаги запуска контейнера в Docker Engine
?
1. Docker Client → POST REST API → /var/run/docker.sock
2. Docker Daemon → gRPC → containerd (создать контейнер)
3. containerd: образ → OCI-bundle → инструктирует runc
4. containerd делает fork runc
5. runc: создаёт namespaces + cgroups → запускает процесс
6. runc немедленно завершается (exits)
7. shim подхватывает контейнер, удерживает STDIN/STDOUT, передаёт exit code демону

В чём ключевое отличие контейнера от VM по использованию ресурсов?::Контейнер разделяет ядро хост-ОС, не загружает собственную Guest OS — стартует за миллисекунды; VM несёт «VM tax» (полная Guest OS на каждый экземпляр) — старт занимает минуты

Почему после обновления dockerd контейнеры не умирают?::Архитектура «daemonless containers»: жизненный цикл контейнеров отвязан от демона через процесс shim, который удерживает контейнер живым независимо от состояния dockerd

Что происходит с образом при перезаписи его тега?::Старый образ лишается тега и переходит в состояние Dangling Image (помечается как `<none>:<none>`), но остаётся на диске

Edge case: Windows-контейнер на Linux-хосте?::Невозможно напрямую — контейнеры разделяют ядро хоста, Windows-контейнеры требуют Windows-ядра; нужны дополнительные лёгкие VM (Docker Desktop через Hyper-V/WSL2)

Edge case: контейнер завершается сразу после старта?::Контейнер живёт ровно столько, сколько живёт его процесс с PID 1 — если процесс (напр. фоновый демон) завершился, контейнер останавливается

Почему нельзя запускать процессы в контейнере под root?::Контейнер разделяет ядро хоста — при компрометации root в контейнере может получить доступ к хостовой системе; нужно задавать USER в Dockerfile
