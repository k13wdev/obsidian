# 05. Конфигурация, Секреты, Stateless vs Stateful

## Суть и контекст

Конфигурация контейнеров передаётся через переменные окружения. Секреты требуют отдельного механизма (Docker Secrets) — ENV недостаточно безопасны. Проектирование контейнеров как stateless-приложений — ключевой архитектурный принцип.

---

## Ключевые идеи

- `ENV` подходит для нечувствительных данных конфигурации; для секретов — Docker Secrets
- ENV-переменные автоматически наследуются дочерними процессами и часто попадают в логи ошибок
- Любой пользователь с доступом к Docker API может прочитать ENV через `docker inspect`
- Docker Secrets шифруются в покое и при передаче; монтируются как in-memory файлы
- Stateless контейнеры: весь persistent state — во внешнем хранилище (Redis, PostgreSQL, S3)
- Конфигурация в переменных окружения соответствует методологии The Twelve-Factor App

---

## Определения и термины

**Environment variables (переменные окружения)** — пары «ключ-значение», доступные программам через их контекст выполнения. Позволяют изменять конфигурацию программы без модификации файлов или команды запуска. Значения внедряются в контейнер на этапе его создания (creation time).

**Stateless приложение** — приложение, спроектированное для немедленного ответа на один изолированный запрос без необходимости отслеживать информацию между запросами от одного или нескольких клиентов.

**Stateful приложение** — приложение, отслеживающее важные данные в оперативной памяти, файлах или базе данных. При перезапуске stateful-сервиса теряется информация, не записанная из оперативной памяти.

**Docker Secrets (секреты Docker)** — объекты первого класса в Docker API (с версии 1.13) для безопасной доставки конфиденциальных данных. Шифруются в состоянии покоя и при передаче, монтируются в контейнеры как in-memory файловые системы, работают по модели наименьших привилегий.

---

## Как устроено

### Конфигурация контейнеров через ENV

- Переменные окружения передаются через `-e KEY=VALUE` при `docker run` или через `env_file` в `docker-compose.yml`
- Значения внедряются при создании контейнера
- Конфигурация в Docker Compose: ключ `environment` (список/словарь) или `env_file` (путь к файлу)

### Docker Secrets: механизм работы

1. Секрет создаётся через CLI: `docker secret create`
2. Секрет прикрепляется к сервису флагом `--secret` при `docker service create`
3. Зашифрованный секрет передаётся по сети (in-flight) на worker node, где запускается задача сервиса
4. На worker node секрет расшифровывается и монтируется в контейнер в in-memory файловую систему (`/run/secrets/` на Linux, `C:\ProgramData\Docker\secrets\` на Windows)
5. Приложение считывает секрет как обычный файл — без методов дешифровки
6. После завершения работы контейнера in-memory ФС демонтируется, секрет немедленно удаляется с узла

**Интеграция с Swarm:** кластерное хранилище (на базе etcd) реплицирует конфигурацию и секреты на все узлы-менеджеры в зашифрованном виде.

## Детали и нюансы

### Риски ENV для секретов

- ENV-переменные **автоматически наследуются** всеми дочерними процессами
- Приложения **часто дампят переменные окружения в логи ошибок** при сбоях или старте
- Любой пользователь с доступом к Docker API может прочитать их через **`docker inspect`**
- Данные в ENV **невозможно стереть из памяти (unset) с гарантией безопасности** после использования
- Значения ENV **запекаются в метаданные слоёв образа** — хардкод секретов в Dockerfile = хардкод в исходном коде

### Хранение state в контейнерах

- По умолчанию данные stateful-приложения пишутся в Thin R/W Layer контейнера
- Скорость I/O через UnionFS значительно ниже, чем прямой доступ к диску
- Данные уничтожаются при удалении/пересоздании контейнера

### Риски потери ключей в Docker Swarm

Корневой ключ подписи генерируется один раз. При сбое узла-менеджера, если ключи не были скопированы — восстановить зашифрованные конфигурации, секреты и механизм Docker Content Trust невозможно; все пользователи теряют возможность загружать обновления образов.

## Сравнения и trade-offs

| Метод | Данные | Безопасность | Когда использовать |
|-------|--------|-------------|-------------------|
| `ENV` в Dockerfile | Нечувствительная конфигурация | Низкая (видна через `docker inspect`, попадает в логи) | PORT, ENVIRONMENT, флаги |
| `-e` / `--env-file` | Конфигурация runtime | Средняя | Адреса сервисов, флаги окружения |
| Docker Secrets | Пароли, токены, TLS-сертификаты | Высокая (in-memory, зашифровано) | Credentials, ключи шифрования |

| Тип приложения | Сложность | Масштабируемость | Рекомендация |
|----------------|-----------|-----------------|--------------|
| Stateless | Ниже | Горизонтальная, легко | Проектировать так по умолчанию |
| Stateful | Выше | Сложнее | Выносить state во внешние хранилища |

## Примеры

```dockerfile
# Задание переменных окружения в Dockerfile
ENV AP /data/app
ENV SCPATH /etc/supervisor/conf.d
```

```bash
# Внедрение конфигурации через -e при запуске
docker run -e ENVIRONMENT=testing -e API_KEY=12345 -i -t awesome_app:version1 /opt/awesome_app/test.sh

# Создание Docker Secret из STDIN
echo staging | docker secret create staging_token -
```

```yaml
# Docker Compose: загрузка переменных из файла
services:
  identidock:
    env_file:
      - .env
# .env файл: KEY=VALUE пары
```

## Важно / подводные камни / best practices

- **Никогда не хранить секреты в ENV в Dockerfile** — запекаются в слои образа; доступны через `docker inspect`; попадают в логи
- **Конфигурация в ENV, секреты в Docker Secrets** (или монтировать как файлы в read-only tmpfs)
- **Проектировать контейнеры как stateless** — весь persistent state во внешнем хранилище (Redis, PostgreSQL, S3); перезапуск/пересоздание контейнера не влияет на данные
- **The Twelve-Factor App**: конфигурация в среде выполнения, отдельно от исходного кода — позволяет деплоить один образ в dev/staging/production без пересборки
- **Хранить данные stateful-приложения в Docker Volumes**, не в R/W слое контейнера
