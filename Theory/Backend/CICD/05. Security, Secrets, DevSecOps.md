# 05. Security, Secrets, DevSecOps

## Суть и контекст

DevSecOps — встраивание целей информационной безопасности во все стадии цикла создания ПО. Безопасность становится частью повседневной работы, а не отдельным этапом аудита в конце. Ключевой принцип: **Shift-left security** — проверки безопасности запускаются при каждом коммите, выявляют уязвимости, пока объём нового кода мал и контекст свеж. Секреты никогда не хранятся в репозитории — только в специализированных хранилищах.

---

## Ключевые идеи

- DevSecOps: безопасность = часть CI/CD pipeline, не отдельный ручной аудит
- SAST — white-box, без выполнения кода; DAST — black-box, против работающего приложения
- Secret Detection — форма SAST; regex; сканирует любые текстовые файлы (language-agnostic)
- Dependency Scanning: версии зависимостей vs CVE; рекурсивно (прямые + транзитивные)
- Container Scanning: пакеты базового образа ОС + пакеты из Dockerfile vs CVE
- IaC Scanning (KICS): Terraform/Ansible конфиги сканируются до деплоя; отображается как SAST
- Простое удаление секрета из кода — НЕ решение: секрет остаётся в истории Git навсегда
- Protected + Masked: секрет только в защищённых ветках, [MASKED] в логах
- Привилегированный runner (root/privileged mode) = угроза: малварь получает доступ к хост-системе

## Определения и термины

**DevSecOps (Rugged DevOps / DevOpsSec)** — процессы встраивания целей информационной безопасности во все стадии цикла создания ПО, делающие безопасность частью повседневной работы всех сотрудников компании. Подразумевает интеграцию превентивного контроля в репозитории исходного кода и конвейеры развёртывания, снижая зависимость от ручных проверок.

**SAST (Static Application Security Testing)** — сканер, проверяющий исходный код проекта на предмет плохих практик кодирования, антипаттернов и уязвимостей без выполнения самого кода. Представляет собой «white-box» тестирование. Использует специфичные для языков анализаторы: Semgrep, Bandit, Brakeman.

**DAST (Dynamic Application Security Testing)** — форма «black-box» тестирования, сканирующая работающие веб-приложения (URLs) или Web API эндпоинты: отправляет входные данные и анализирует ответы. Выполняет обход (spidering) страниц. Режимы:
- **Пассивный** — базовые проверки
- **Активный** (full scan / active attacks) — симулирует действия хакера

**Secret Detection (Обнаружение секретов)** — специализированная форма SAST: анализирует репозиторий с использованием регулярных выражений для поиска строк с конфиденциальными данными (пароли, SSH ключи, AWS deploy keys). Независима от языков (language-agnostic): сканирует любые текстовые файлы.

**Dependency Scanning (Проверка зависимостей)** — сравнение имён и версий каждой зависимости с базой данных известных уязвимостей (CVE). Не анализирует сам исходный код зависимости. Рекурсивно сканирует прямые и транзитивные зависимости.

**Container Scanning (Сканирование контейнеров)** — проверка базового образа Linux, на котором работает Docker-приложение, по базе CVE. Проверяет пакеты по умолчанию в ОС и дополнительные пакеты, указанные в Dockerfile.

**IaC Scanning (Инфраструктура как код — сканирование)** — специализированная форма SAST, которая анализирует репозиторий на наличие конфигурационных файлов поддерживаемых IaC-инструментов (Terraform, Ansible) и идентифицирует уязвимости или плохие практики до развёртывания инфраструктуры. В GitLab базируется на open-source инструменте KICS (Keeping Infrastructure as Code Secure). В интерфейсе Vulnerability Report результаты классифицируются как SAST.

**OWASP Dependency-Check** — инструмент статического анализа, проверяющий зависимости проекта на наличие бинарных и исполняемых файлов с публично известными уязвимостями. Упоминается как стандартный плагин Jenkins для отчётов по критичности находок.

**GitLab CI Variables: Protected** — флаг, гарантирующий экспорт переменной только в pipeline, запущенные на защищённых ветках (protected branches) и тегах. Предотвращает утечку секретов в pipeline с обычных feature-веток.

**GitLab CI Variables: Masked** — флаг, скрывающий значение переменной в логах выполнения job. Попытки вывода заменяются на `[MASKED]`. Требует строгого соответствия форматам регулярных выражений.

**Secret Rotation (Ротация секретов)** — процесс отзыва (revoke) и замены паролей, токенов доступа, приватных ключей и других учётных данных новыми значениями. Должна выполняться немедленно после обнаружения секрета в системе контроля версий. Простое удаление из файла в следующем коммите не устраняет угрозу — секрет сохраняется в истории репозитория.

**Shift-left security** — интеграция принципов безопасности и проверок на самых ранних стадиях SDLC (сдвиг влево по временной шкале). Требует автоматизированных проверок при каждом коммите до слияния. Ошибки архитектуры дешевле исправлять при малом объёме кода. Способствует сотрудничеству разработчиков и инженеров безопасности через Merge Requests.

**HashiCorp Vault / AWS Secrets Manager / AWS SSM Parameter Store** — управляемые системы для хранения, управления, контроля доступа и шифрования секретов (пароли БД, ключи шифрования, токены). Устраняют необходимость хранения учётных данных в plaintext. Инструменты CI/CD извлекают секреты динамически только на время выполнения сборки или деплоя.

**SOPS** — В предоставленных источниках не раскрыто.

## Как устроено

### Управление уязвимостями в GitLab

1. Сканер (SAST, Secret Detection и др.) обнаруживает уязвимость → публикует со статусом **Needs Triage**
2. Команда анализирует: исправление не нужно → **Dismissed**; нужно → **Confirmed**
3. Опционально создаётся issue с информацией и инструкциями по устранению
4. Разработчик создаёт ветку и Merge Request
5. Вкладка «Security» в отчёте MR показывает **дельту**: уязвимость есть в целевой ветке, но больше не детектируется в ветке разработчика

### Историческое сканирование секретов (Historic Mode)

1. `SECRET_DETECTION_HISTORIC_SCAN: "true"` в `.gitlab-ci.yml`
2. Сканер анализирует абсолютно все коммиты в истории репозитория Git (включая «откаченные»)
3. Любой найденный секрет — даже удалённый в последующих коммитах — помечается как скомпрометированный и рекомендуется к отзыву (revoke)

### Интеграция стороннего сканера в pipeline

1. Создаётся job в секции `test`
2. Указывается Docker-образ стороннего сканера (`image`)
3. В блоке `script` — CLI-команда сканера
4. `allow_failure: true` — не роняет весь pipeline при находках
5. В `artifacts:reports:sast:` — путь к JSON с результатами → отображается в Security Dashboard GitLab

## Детали и нюансы

- **Dependency Scanning vs Container Scanning**: Dependency Scanning проверяет пакеты кода приложения (например, `requirements.txt`); Container Scanning — системные пакеты базового Linux-дистрибутива внутри Docker-образа
- **Vulnerability Report vs MR Report**: Vulnerability Report — глобальное состояние проекта (все уязвимости в main); MR Report — только «дельта» (новые уязвимости добавленные конкретной веткой)
- **IaC Scanning в GitLab = SAST**: результаты Terraform/Ansible сканирования отображаются в Vulnerability Report с меткой инструмента SAST
- **Masked variable ограничения**: значение должно соответствовать строгим regex; многострочные значения не маскируются
- **Привилегированный runner**: пайплайны разных проектов могут читать кэши друг друга; вредоносный код из собираемого образа получает доступ к хост-системе runner-а

## Сравнения и trade-offs

| Тип сканирования | Метод | Уровень | Инструменты |
|-----------------|-------|---------|------------|
| SAST | White-box (без запуска) | Исходный код | Semgrep, Bandit, Brakeman |
| DAST | Black-box (работающее приложение) | HTTP/API | OWASP ZAP и др. |
| Secret Detection | SAST (regex) | Любые текстовые файлы | Встроен в GitLab |
| Dependency Scanning | Версии vs CVE | Файлы зависимостей (requirements.txt и др.) | OWASP Dependency-Check |
| Container Scanning | Пакеты образа vs CVE | Базовый образ Docker | Trivy, Clair |
| IaC Scanning | SAST (конфиги) | Terraform, Ansible файлы | KICS (GitLab) |

| Хранение секретов | Безопасность | Сложность |
|------------------|-------------|---------|
| Hardcoded в коде / Dockerfile | Критическая уязвимость | Нет |
| CI Variables (Protected + Masked) | Средняя | Низкая |
| Vault / AWS Secrets Manager | Высокая (dynamic secrets, rotation) | Высокая |

## Примеры

```yaml
# Добавление IaC Scanning (Terraform/Ansible) в GitLab pipeline
stages:
  - test
include:
  - template: SAST-IaC.latest.gitlab-ci.yml
```

```yaml
# Secret Detection: историческое сканирование + исключение директорий
secret_detection:
  variables:
    SECRET_DETECTION_HISTORIC_SCAN: "true"
    SECRET_DETECTION_EXCLUDED_PATHS: "tests,docs"
```

```yaml
# Безопасное использование секрета в pipeline (GitHub Actions синтаксис)
- name: Use my secret token
  env:
    MY_SECRET_TOKEN: ${{ secrets.MY_SECRET_TOKEN }}
  run: |
    curl -d $MY_SECRET_TOKEN https://some-endpoint
```

> Секрет загружается из менеджера в переменную среды только на время выполнения шага — не зашивается в конфигурацию.

```hcl
# Уязвимый Terraform код — обнаружится IaC Scanning (KICS)
resource "aws_s3_bucket" "testBucket" {
  bucket = "myBucket"
  acl = "authenticatedRead"
}
```

> `acl = "authenticatedRead"` — уязвимость: предоставляет доступ всем аутентифицированным пользователям AWS, не только владельцу.

**Vault agent в Jenkins / OWASP Dependency-Check в Jenkinsfile** — В предоставленных источниках не раскрыто. Инструменты упоминаются только контекстно (OWASP Plugin), конкретный синтаксис в Jenkinsfile отсутствует.

## Важно / подводные камни / best practices

- **Shift-left + delta в MR**: разработчики видят только уязвимости от своего коммита, не тысячи старых — устраняют в нужный момент, пока контекст понятен
- **Vault / AWS Secrets Manager вместо CI Variables**: секреты динамически извлекаются только на время сборки; не хранятся нигде в plaintext
- **IaC Scanning обязателен**: Terraform/Ansible создают инфраструктуру автоматически; без сканирования S3-бакеты и БД разворачиваются с дырами «из коробки»
- **Антипаттерн: удалить токен из кода = безопасность**: секрет навсегда в истории Git; злоумышленник уже мог извлечь его; обязательна ротация на стороне провайдера (перевыпустить AWS Access Key)
- **Protected + Masked**: без Protected — секрет утечёт в feature-ветки; без Masked — появится в логах plaintext
- **Runner non-root**: привилегированный режим = container escape → доступ к хост-системе + данным других pipeline
- **Устаревший базовый образ**: CentOS 6 с непатченными CVE → Container Scanning как обязательный gate
