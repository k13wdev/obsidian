# 09. Observability, Платформы, Антипаттерны

## Суть и контекст

Зрелый CI/CD включает три ключевых измерения: **Observability** (телеметрия pipeline и приложения), **платформенный подход** (Self-service инфраструктура, где разработчики не зависят от других отделов) и **устойчивость** (Chaos Engineering, Disaster Recovery, Audit trails). Важнейший антипаттерн — долгоживущие ветки и «усталость от алертов» (alert fatigue). Цель: система, самостоятельно обнаруживающая и устраняющая проблемы.

---

## Ключевые идеи

- Self-service: разработчики нажимают кнопку → развёртывают в production; не ждут Ops/Security/QA
- Observability 4 уровня: бизнес (транзакции) → приложение (5XX) → инфраструктура (CPU) → деплой (статус CI)
- Audit trails: pipeline-логи — юридическое доказательство соответствия (PCI-DSS, HIPAA); Filebeat → Logstash → ELK
- Chaos Monkey («Обезьянья армия Netflix»): постоянное случайное уничтожение инстансов → выявление слабых мест
- Security Monkey: находит уязвимые Security Groups → отключает
- Long-running branches — антипаттерн: большие пакеты изменений → конфликты CI → скрытые причины сбоев
- Alert fatigue: вместо превентивных слабых сигналов — поток алертов после падения production
- Service Handback: хрупкие сервисы возвращаются к разработчикам до автоматизации деплоя

---

## Определения и термины

**Self-service infrastructure (Самообслуживающаяся платформа)** — платформа для надёжного, быстрого и безопасного тестирования, развёртывания и запуска IT-сервисов без ожидания результатов от смежных групп. Разработчики нажимают кнопку для развёртывания и видят production среду. Создаётся командами эксплуатации с встроенным опытом Ops + Security.

**Chaos Monkey / Chaos Engineering** — сервис, симулирующий сбои инфраструктуры (AWS) постоянно и в случайном порядке для оценки устойчивости систем. Превратился в **«Обезьянью армию Netflix»**:
- **Chaos Monkey** — уничтожает случайные инстансы AWS
- **Janitor Monkey** — сканирует и уничтожает «мусорные» ресурсы
- **Security Monkey** — обнаруживает экземпляры с уязвимостями (неверно настроенные Security Groups) и отключает их

**Телеметрия (Deployment telemetry)** — сбор данных о статусе сборки (красный/зелёный для тестов), времени на развёртывание и частоте развёртываний. Предупреждает о проблемах, если выполнение сборки или тестов занимает вдвое больше обычного.

**Audit trails (Аудиторские доказательства)** — подтверждения соответствия требованиям безопасности, автоматически создаваемые и сохраняемые в pipeline в ходе повседневной работы. Должны отправляться в общую систему логирования (ELK и др.), где связываются с нормативными актами (PCI-DSS, HIPAA).

**Long-running branch (Долгоживущая ветка)** — стратегия разработки, при которой код долгое время не объединяется с основной веткой. Ведёт к значительным сбоям при слиянии. Противоположность мелкосерийной разработки и непрерывной интеграции.

**Internal Developer Platform (IDP) / Platform Engineering** — в источниках нет прямого определения; описывается как «создание общедоступных платформ», в которые встраивается коллективный опыт Ops и Security, что делает деплой для разработчиков простым, надёжным и безопасным.

**Golden paths** — В предоставленных источниках не раскрыто.

**Multi-tenant CI/CD** — В предоставленных источниках не раскрыто.

## Как устроено

### Observability: архитектура мониторинга

```
Бизнес-логика ──┐
Приложение      ├──→ Маршрутизатор событий ──→ Хранилище
Операционная ОС ┘         │                 ──→ Построение графиков (Grafana, ELK)
                          └──────────────────→ Оповещение (при отклонениях)
```

4 уровня телеметрии:
1. **Бизнес**: транзакции, регистрации пользователей, конверсия
2. **Приложение**: ошибки 5XX, латентность
3. **Инфраструктура**: CPU, memory
4. **Деплой**: статус CI (красный/зелёный), время деплоя, частота деплоев

Инструменты агрегации: Prometheus, Grafana, ELK, Splunk, Datadog

### Audit trails: DevOps Audit Defence Toolkit

1. Изменения фиксируются в VCS с параметрами контроля конфигураций
2. Автоматические проверки зависимостей: OWASP Dependency Check, Nexus Lifecycle
3. Централизованный сбор: события деплоя + изменения AWS Security Groups + логи серверов
4. **Filebeat → Logstash → ELK** — неизменяемая система логов с привязкой к требованиям (PCI-DSS, HIPAA)

### Chaos Engineering: «Обезьянья армия Netflix»

1. **Chaos Monkey** внедряется в production: постоянно случайно уничтожает инстансы или компоненты AWS
2. **Janitor Monkey** сканирует среду → уничтожает «мусорные» ресурсы
3. **Security Monkey** обнаруживает уязвимые экземпляры (неверные Security Groups) → отключает
4. Система адаптируется (переключение уровней, жёсткие лимиты) без длительного простоя

### Service Handback: процесс передачи управления

```
[Разработчики] ──→ Hand-Off Readiness Review ──→ [SRE управляет]
                                                        │
                                               Метрика "Стабильность" падает
                                                        │
                         [Разработчики снова] ←─── Возврат управления
                                │
                     Исправление + Readiness Review ──→ [SRE управляет]
```

## Детали и нюансы

- **Телеметрия видна всей организации**: если деплой ломает метрики (регистрации падают на 20%) — все сотрудники видят в реальном времени → быстрое обнаружение и исправление
- **Pipeline-логи = юридический Audit trail**: статические анализаторы (OWASP) в pipeline + централизованное логирование = автоматическое доказательство соответствия; нет ручных аудитов
- **Chaos Engineering тестирует Disaster Recovery**: постоянные случайные уничтожения узлов проверяют auto-scaling и self-healing кластеров Jenkins
- **CircleCI test split**: `circleci tests split --split-by=timings` — шардинг тестов на основе хронометража предыдущих запусков

---

## Сравнения и trade-offs

| Антипаттерн | Проблема | Решение |
|------------|---------|---------|
| Long-running branches | Конфликты слияния, скрытые причины сбоев CI | Trunk-Based Development, мелкие коммиты |
| Alert fatigue | Поток алертов после сбоя; нет превентивных сигналов | Shift-left метрики, слабые сигналы в начале pipeline |
| Передача «через стену» (Hand-Off) | Хрупкие приложения в эксплуатации, Ops чинит незнакомый код | Self-service платформы + Service Handback |
| Ручные аудиты в облаке | Auto Scaling: серверы появляются/исчезают; физическая выборка невозможна | IaC + централизованный стриминг логов + метрики к требованиям |

## Примеры

```bash
# CircleCI: шардинг тестов по времени выполнения
circleci tests split --split-by=timings
```

> Утилита анализирует хронометраж предыдущих тестов и разделяет набор на подмножества для параллельного выполнения.

```
# Пример Audit trail лога деплоя (Deployinator)
[web] 2010-05-18 22:20:50 | PRODUCTION | sandrews | Production deploy: old 25134,
[web] 2010-05-18 22:18:00 | PRINCESS   | sandrews | Princess Deploy: old: 25144, new:
```

> Запись фиксирует: метку времени, окружение (PRODUCTION/PRINCESS/QA), ID пользователя, версии коммитов до и после деплоя.

## Важно / подводные камни / best practices

- **Self-service infrastructure**: Ops создают централизованные сервисы (аутентификация, логирование, API деплоя); разработчики не ждут Security/QA/Ops → устраняет узкие места и межкомандный ручной труд
- **Телеметрия в общих дашбордах**: метрики бизнес-уровня (конверсия, транзакции) видны всей компании → мгновенная обратная связь после деплоя
- **Антипаттерн Long-running branches**: интеграция больших порций кода → серьёзные конфликты CI/CD + скрытые причины сбоев; решение — частые малые коммиты в main (Trunk-Based)
- **Антипаттерн Alert fatigue**: поток алертов после падения production; решение — слабые превентивные сигналы на ранних стадиях CI (shift-left мониторинг)
- **Audit trail автоматически из pipeline**: вместо ручных аудитов — стриминг логов (Filebeat → ELK) с привязкой к PCI-DSS/HIPAA
- **Chaos Engineering в production**: не проверка концепций в staging, а постоянное уничтожение в production → выявление реальных слабых мест; Janitor/Security Monkey = непрерывный автоматический аудит

?

3. Security Monkey: обнаруживает уязвимые Security Groups → отключает

Edge case: Compliance audit в Auto Scaling среде?::Серверы постоянно появляются/исчезают; физическая выборка тысяч серверов невозможна. Решение: IaC + централизованный стриминг логов (AWS CloudWatch) с привязкой к требованиям контроля

Edge case: Service Handback — хрупкое приложение передано в SRE?::Метрики стабильности падают → управление возвращается разработчикам (SRE Handback) → исправление + автоматизация → Readiness Review → повторная передача в SRE

Edge case: Disaster Recovery — полное падение AWS?::Ожидание до 6 часов. Механизмы: жёсткое отключение тормозящих компонентов + деградация сервиса + auto-scaling groups для самовосстановления

Почему Chaos Engineering в production, а не staging?::Staging не отражает реальную нагрузку и конфигурацию; реальные слабые места обнаруживаются только под настоящим трафиком; постоянное уничтожение = непрерывное тестирование устойчивости
