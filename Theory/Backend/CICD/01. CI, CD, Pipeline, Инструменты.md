# 01. CI, CD, Pipeline, Инструменты

## Суть и контекст

CI/CD — набор практик непрерывной интеграции, доставки и развёртывания кода. Позволяет автоматизировать сборку, тестирование и деплой при каждом изменении кода. Pipeline-as-code — ключевой принцип: конфигурация пайплайна хранится в репозитории как код.

---

## Ключевые идеи

- CI — каждое изменение кода немедленно интегрируется и проверяется; отсутствие ночных сборок
- Continuous Delivery: продукт всегда готов к релизу, но деплой в production активируется **вручную** (manual failsafe)
- Continuous Deployment: деплой в production **автоматически** при успешном прохождении всех тестов
- Pipeline-as-code: конфигурация пайплайна в системе контроля версий — аудит, версионирование, code review
- Shift-left: проверки выполняются как можно раньше — снижает стоимость обнаружения ошибок
- Артефакты vs Кэш: артефакты — результаты работы job для передачи между стадиями; кэш — зависимости для ускорения последующих запусков
- DAG (`needs:` в GitLab): job не ждёт завершения всей стадии, а запускается сразу после нужной job
- Hosted runners: GitLab даёт 400 мин/месяц бесплатно; self-hosted runners — без ограничений на своём сервере

---

## Определения и термины

**Continuous Integration (CI)** — постоянный процесс внесения изменений в код, когда каждое изменение проверяется при добавлении его к уже имеющимся и проверенным изменениям. В контексте DevOps включает работу в среде, близкой к производственной.

**Continuous Delivery (CD)** — методика разработки, при которой релиз рабочего программного продукта производится так быстро, как того требует проект, а сборка — таким образом, чтобы релизы можно было гарантированно безопасно выпускать в любое время. Продукт постоянно готов к релизу, но деплой в production активируется вручную (manual failsafe).

**Continuous Deployment (CD)** — выпуск релиза рабочей версии ПО для пользователей осуществляется **автоматически** при каждом коммите, если пройдены все проверки.

**Pipeline** — серия шагов, выполняемых при каждом коммите в репозиторий. Состоит из стадий (Stage) и заданий (Job).

**Pipeline as Code (PaC)** — набор возможностей, позволяющих определять пайплайны в виде кода, хранить и версионировать их в репозитории. Логическое продолжение Infrastructure as Code.

**Stage (Стадия)** — тематически связанный набор задач пайплайна. Определяет порядок: Build → Test → Deploy.

**Job (Задача)** — единица работы внутри стадии; содержит команды, выполняемые в одной среде (Docker-контейнер или shell-среда runner-а).

**Артефакт (Artifact)** — файлы, сгенерированные job и сохранённые для скачивания или использования в последующих job. По умолчанию удаляются после завершения job.

**GitLab Runner** — программа, принимающая задания CI/CD от GitLab, выполняющая их в изолированной среде (Docker-контейнере) и отправляющая результаты обратно в GitLab.

**Кэш (Cache)** — сохранённые зависимости (npm, Maven, pip) между запусками пайплайна. Ускоряет сборку. Не предназначен для передачи данных между стадиями (для этого — артефакты).

**DAG (Directed Acyclic Graph) Pipeline** — режим, при котором job не ждёт завершения всей стадии, а запускается сразу по готовности конкретной зависимой job (через `needs:` в GitLab).

## Как устроено

### Continuous Delivery vs Continuous Deployment

```
CI:    Commit → Build → Test → Artifact
                                    ↓
CD-Delivery:                  → Pre-prod → [Manual Approval] → Production
CD-Deployment:                → Pre-prod → Production (автоматически)
```

### Архитектура GitLab Pipeline

```
Разработчик → коммит в ветку с .gitlab-ci.yml
                    ↓
GitLab instance → считывает конфигурацию → планирует jobs
                    ↓
Job назначается свободному GitLab Runner
                    ↓
Runner → забирает коммит → выполняет команды shell/script
                    ↓
Runner → отправляет результаты (pass/fail) + артефакты → GitLab
```

### Типовые стадии CI/CD пайплайна

```
Start → Checkout → Quality Tests → Unit Tests → Security Tests
     → Build → Push → Deploy → Acceptance Tests → End
```

### Артефакты vs Кэш

| | Артефакты | Кэш |
|--|----------|-----|
| Назначение | Передача результатов между стадиями | Ускорение сборки при повторных запусках |
| Хранение | На сервере GitLab/Jenkins | На runner-е |
| Пример | Скомпилированный JAR, отчёт тестов | node_modules, .m2, pip packages |

### DAG с `needs:` (GitLab)

```yaml
deploy_app:
  needs:
    - build_app    # не ждёт завершения всей стадии build
```
Как только `build_app` завершается → `deploy_app` немедленно стартует, не дожидаясь других job той же стадии.

### Jenkins: Declarative vs Scripted Pipeline

| | Declarative | Scripted |
|-|------------|---------|
| Синтаксис | `pipeline { agent... stages { steps {` | `node('workers') { stage(...) {` |
| Версия | Новый (с v2.5) | Старый |
| Groovy | Ограничен блоком `script {}` | Полный Groovy |
| Сложная логика | Только в `script {}` | Нативно (for, if/else, переменные) |

---

## Детали и нюансы

- **YAML-ошибка в `.gitlab-ci.yml`**: GitLab показывает `yaml invalid` и не запускает pipeline
- **Declarative Jenkinsfile: ограничения**: нельзя использовать `for`, `if/else`, объявления переменных вне `script {}` — ошибка парсера
- **Недетерминированная сборка**: разные зависимости или флаги компилятора в CI и production → отличные артефакты → баги на деплое
- **Hosted GitLab**: 400 бесплатных минут в месяц. Self-hosted runners — без ограничений на своём сервере
- **GitHub Actions hosted/self-hosted**: публичный GitHub — GitHub размещает (hosted); GitHub Enterprise — пользователь сам отвечает за запуск платформы
- **ArgoCD**: в источниках не найдено прямого упоминания; упоминается Argo Workflows — система CI/CD, где каждый шаг выполняется как контейнер на Kubernetes
- **Mono-repo**: один репозиторий для всех микросервисов — логичная связность, но пайплайны разрастаются в «spaghetti mix». Multi-repo (один репозиторий на сервис) — изоляция, но дублирование настроек пайплайнов

---

## Сравнения и trade-offs

| Подход | CI | Continuous Delivery | Continuous Deployment |
|--------|-----|--------------------|-----------------------|
| Деплой в production | Вручную | Вручную (по готовности) | Автоматически |
| Риск | Высокий (редкие релизы) | Средний | Низкий (мелкие изменения) |
| Скорость | Низкая | Средняя | Высокая |

| Инструмент | Тип | Особенности |
|------------|-----|-------------|
| Jenkins | Open Source, self-hosted | Declarative/Scripted Pipeline; GitHub webhook |
| GitLab CI | Встроен в GitLab | `.gitlab-ci.yml`; GitLab Runners; DAG |
| GitHub Actions | SaaS / self-hosted | `.github/workflows/`; marketplace actions |
| CircleCI | SaaS | Hosted/self-hosted runners |
| Travis CI | SaaS | Hosted |
| AWS CodePipeline | Cloud-managed | Интеграция с AWS |

## Примеры

```yaml
# .gitlab-ci.yml: простая сборка Java с артефактами
stages:
  - build

compile-java-with-javac:
  stage: build
  image: openjdk:latest
  script:
    - javac src/com/hatsforcats/*.java -d target/
    - ls target/com/hatsforcats
    - cd target
    - java com.hatsforcats.Login
```

```yaml
# GitLab: сохранение артефакта
build_app:
  stage: build
  script:
    - make my_app
    - tar -czvf my_app.tar.gz .
  artifacts:
    paths:
      - my_app.tar.gz
```

```yaml
# GitLab: rules + needs (DAG) — деплой только на main
deploy_app:
  stage: deploy
  script:
    - echo "Run deploy only on main branch"
  only:
    - main
  needs:
    - build_app
```

```groovy
// Jenkins Declarative Pipeline
pipeline{
    agent any
    stages {
        stage('Test'){
            steps {
                sh 'npm run test'
                sh 'npm run coverage'
            }
        }
    }
}
```

```groovy
// Jenkins Scripted Pipeline
node('workers'){
    stage('Checkout'){
        checkout scm
    }
}
```

```groovy
// Jenkins: параллельные задачи
stage('Push'){
    def fileName = commitID()
    def parallelStagesMap = functions.collectEntries {
        ["${it}" : {
            stage("Lambda: ${it}") {
                sh "aws s3 cp ${fileName}.zip s3://${bucket}/${it}/"
            }
        }]
    }
    parallel parallelStagesMap
}
```

## Важно / подводные камни / best practices

- **Pipeline-as-code**: вся конфигурация в репозитории — аудит, откат, code review
- **Shift-left**: проверки как можно раньше (качество, безопасность, тесты) — ошибки дешевле исправить в начале
- **DAG вместо последовательных стадий**: сокращает общее время пайплайна
- **Ночные сборки — антипаттерн**: утром команда видит сломанную сборку и тратит время на поиск виновника среди сотен коммитов
- **Sequential workflows — антипаттерн**: команды передают код «через стену» → большие пакеты изменений → дорогое исправление ошибок интеграции
- **DRY в CI-конфигурации**: anchors, `extends:`, `include:` в GitLab для избежания дублирования

## Карточки для повторения

#flashcards/cicd/basics

**Continuous Integration (CI)**:::Практика немедленной интеграции каждого изменения кода с автоматической проверкой; исключает ночные сборки и «интеграционный ад»

**Continuous Delivery**:::Код всегда готов к релизу; деплой в production — вручную (manual failsafe); автоматизация до pre-prod

**Continuous Deployment**:::Полная автоматизация; каждый коммит, прошедший все тесты, автоматически попадает в production

**Pipeline**:::Серия шагов (stages/jobs) при каждом коммите: Checkout → Build → Test → Deploy

**Pipeline as Code (PaC)**:::Конфигурация пайплайна в репозитории; версионирование, code review, аудит; продолжение IaC

**Stage**:::Тематически связанная группа jobs; определяет порядок выполнения (Build → Test → Deploy)

**Job (CI/CD)**:::Единица работы внутри стадии; выполняется в изолированной среде (контейнер/shell)

**Артефакт (CI/CD)**:::Файлы, сгенерированные job и сохранённые для последующих стадий или скачивания; удаляются без декларации

**GitLab Runner**:::Программа, принимающая jobs от GitLab и выполняющая их в изолированной среде (Docker/shell)

Артефакты vs Кэш в CI/CD?::Артефакты — передача результатов между стадиями (JAR, отчёты); Кэш — ускорение повторных сборок (node_modules, .m2); хранятся на runner-е

DAG pipeline в GitLab (`needs:`)?::Job не ждёт завершения всей стадии; стартует сразу после завершения указанной зависимой job → сокращение общего времени пайплайна

Jenkins Declarative vs Scripted Pipeline?::Declarative: структурированный синтаксис (pipeline/agent/stages/steps), ограничения Groovy вне script {}. Scripted: полный Groovy (node/stage), старый стиль, максимальная гибкость

CI/CD: Continuous Delivery vs Continuous Deployment?::Delivery — деплой в production требует ручного подтверждения (manual approval). Deployment — полностью автоматический деплой при успешном прохождении тестов

Shift-left: что означает?::Сдвиг проверок влево (раньше) в жизненный цикле разработки; раннее обнаружение ошибок снижает стоимость их исправления

Edge case: YAML-ошибка в .gitlab-ci.yml?::GitLab → статус `yaml invalid`; pipeline не запускается; jobs не назначаются runner-ам

Edge case: Declarative Jenkinsfile + for/if вне script {}?::Ошибка парсера: "Method calls on objects not allowed outside 'script' blocks"; сложная логика только внутри `script {}`

Hosted vs self-hosted runners?::Hosted: GitLab/GitHub хостит (GitLab — 400 мин/мес бесплатно, ограничения). Self-hosted: свой сервер, нет ограничений по времени, управление runner самостоятельно
