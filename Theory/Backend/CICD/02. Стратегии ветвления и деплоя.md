# 02. Стратегии ветвления и деплоя

## Суть и контекст

Стратегия ветвления определяет, как команда организует параллельную разработку в системе контроля версий. Стратегия деплоя определяет, как новая версия кода доставляется пользователям. Выбор стратегии напрямую влияет на частоту релизов, риск простоя, сложность отката и скорость обратной связи от pipeline-а.

Ключевой принцип: **развёртывание (deployment) и релиз (release) — разные события**. Deployment — доставка кода в среду. Release — открытие доступа клиентам. Их разделение позволяет деплоить в рабочее время без стресса и активировать функцию в нужный маркетинговый момент.

---

## Ключевые идеи

- Trunk-Based Development: фиксации в главную ветку ежедневно (или чаще), даже если функция не завершена — исключает «ад слияний»
- Feature Branching: долгоживущие ветки → редкие слияния → большой пакет изменений → высокий риск дестабилизации master
- GitFlow: структурированная иерархия из 5 типов веток (master / preprod / develop / feature / hotfix); логична для больших команд, избыточна для маленьких
- Feature Flags: разделяют deployment и release; позволяют включать функцию для выбранных пользователей без нового деплоя
- Rolling Deployment: последовательная замена экземпляров — простое решение, но откат медленный (6–20 мин суммарного простоя)
- Blue-Green Deployment: два полных окружения, мгновенный откат, но двукратные ресурсы и проблема схемы БД
- Canary Release: инкрементальный деплой через один «канареечный» экземпляр; экономия ресурсов, но сложная маршрутизация
- Cluster Immune System: расширение Canary с автоматическим откатом по метрикам мониторинга
- A/B Testing: одновременные версии для разных сегментов пользователей через Feature Flags; проверяет бизнес-гипотезы

## Определения и термины

**Trunk-Based Development (Магистральная разработка)** — подход, при котором разработчики часто сливают изменения в магистральную (trunk) ветку репозитория (главную ветку), даже если функция целиком ещё не завершена. Подразумевает отказ от долгоживущих веток функций и слияния по возможности несколько раз в день.

**Feature Branching (Ветвление функции)** — стратегия ветвления, согласно которой в начале разработки новой функции создаётся новая ветка (ответвление от главной), разработка продолжается в ней до завершения, после чего она целиком сливается с главной веткой. Ветки становятся «долгоживущими», что усложняет последующее слияние.

**GitFlow** — стратегия ветвления, состоящая из веток: `master` (текущий производственный код), `preprod` (ветка релизов), `develop` (интеграционная ветка), `feature/X` (разрабатываемая функция) и `hotfix/X` (исправления для производственной среды). Не решает всех проблем ветвления, но предлагает логичную структуру для больших команд. Может быть избыточной для небольших проектов.

**Feature Flag (Переключатель функциональности)** — механизм для выборочного включения и отключения функции без развёртывания кода в производственную среду. В GitLab базируется на библиотеке third-party Unleash. Позволяет разделять развёртывание кода и релиз функциональности для клиентов.

**Rolling Deployment (Скользящее обновление)** — обновление одного набора экземпляров по месту его размещения путём последовательной замены старой версии на новую. Вносит от 6 до 20 минут в общее время простоя при ошибках.

**Blue-Green Deployment (Сине-зелёное развёртывание, Red-Black)** — создание абсолютно нового набора экземпляров (зелёных), который обновляется. Только после того, как эти экземпляры готовы, трафик переключается с оригинальных (синих) на новые (зелёные). Быстрый откат осуществляется простым перенаправлением трафика обратно на «синюю» среду.

**Canary Release (Канареечное развёртывание)** — обновляется один экземпляр («канарейка»), на него направляется небольшой процент трафика. Если он функционирует нормально, развёртывание продолжается. Автоматизирует процесс релиза, последовательно распространяя его на всё более крупные среды.

**Cluster Immune System** — расширение шаблона канареечного релиза, в котором системы мониторинга связываются с процессами релиза и автоматизируется откат на предыдущую версию кода при отклонении производительности от заданных показателей. Применяется для минимизации последствий без участия человека.

**A/B Testing (A/B-тестирование)** — отправка в эксплуатацию нескольких версий кода одновременно для разных сегментов заказчиков с использованием переключателей функциональности для проверки бизнес-гипотез. Контролирует процент пользователей, участвующих в эксперименте.

**Progressive Delivery** — В предоставленных источниках не раскрыто. Частично описано в рамках «Канареечных релизов».

## Как устроено

### GitFlow: алгоритм работы

1. Ветка `develop` создаётся от `master`
2. Ветка `preprod` создаётся от `develop`
3. Разработчик создаёт `feature/X` от `develop`; по завершении работы создаёт Pull Request (PR)
4. CI запускает тесты на PR; после успеха: `feature/X` сливается в `develop`, ветка удаляется
5. Сборка деплоится в sandbox/development окружение
6. Создаётся PR из `develop` в `preprod`; после слияния код деплоится в staging
7. После валидации: `preprod` сливается в `master`, изменения развёртываются в production (ручное одобрение)
8. При сбое в production: создаётся ветка `hotfix/X` от `master`; после фикса она сливается и в `master`, и в `develop`

```
hotfix/X ----·---------·--
            ↑           ↓ merge
master  ----·-----------·----  v1.0.0 → v1.0.1
            ↑           ↑ merge
preprod ----·-----------·----
            ↑           ↑ merge
develop ----·-·---------·----
                ↑     ↑ merge
feature/A ------·-----·
```

### Feature Flags через Unleash (GitLab)

1. Флаг создаётся в UI GitLab (`Deployments → Feature Flags`) с указанием стратегии (`Percent rollout 100%`, `Based on User ID`)
2. В коде приложения импортируется клиент Unleash, настраивается подключение к GitLab API с передачей ID проекта (`instance_id`)
3. Формируется контекст (`unleash_context`) с данными текущего пользователя (`user_id`)
4. Приложение вызывает метод проверки флага (`is_enabled?`); если флаг активирован для контекста — выполняется новый код; иначе — старый код

### Канареечное развёртывание: алгоритм

1. Запускается новый экземпляр (канареечный) версии 1.4.0 наряду с тремя базовыми экземплярами версии 1.3.2
2. Одинаковое (небольшое) количество трафика направляется на один выделенный базовый экземпляр и на канареечный
3. Если метрики канареечной версии идентичны или лучше метрик базовой — развёртывание продолжается на остальные узлы

### Blue-Green: схема трафика

```
            Маршрутизатор
           /              \
 [Зелёная секция]   [Синяя секция — неактивна]
  Веб-сервер          Веб-сервер
  App-сервер          App-сервер
  Зелёная БД          Синяя БД
```

## Детали и нюансы

- **Аналогия ветвления**: ветвление в Git сравнивается с движением по улицам. `Main Street` — основная ветка. `First Street` — ветка функции. Разработчик (машина) делает коммиты (бросает шарики из окна). `git switch` (вертолёт с магнитом) переносит машину между улицами в любой момент.
- **GitFlow: масштабируемость**: не подходит для маленьких проектов — требует поддержки 4–5 параллельных уровней веток и строгой координации pull request-ов.
- **Shadow launch**: код развёртывается в production, но скрыт под флагом. Приложение обрабатывает реальный трафик в фоне — нагрузочное тестирование без риска. При сбоях функциональность отключается мгновенно изменением параметра, без отката релиза.
- **Разделение deployment и release**: деплой выполняется в рабочее время без стресса; маркетинг активирует функцию в нужный момент независимо от команды разработки.

## Сравнения и trade-offs

| Стратегия деплоя | Ресурсы | Откат | Риск простоя | Сложность |
|-----------------|---------|-------|-------------|-----------|
| Rolling Deployment | ×1 (инкрементально) | Новое rolling (5–15 мин) | 6–20 мин суммарно | Низкая |
| Blue-Green | ×2 (два полных окружения) | Мгновенный (переключить трафик) | Минимальный | Средняя + проблема БД |
| Canary Release | ×1 (инкрементально) | Сложная маршрутизация + изоляция метрик | Ограничен одним экземпляром | Высокая |
| Cluster Immune System | ×1 (инкрементально) | Автоматический по метрикам | Минимальный | Высокая |

| Стратегия ветвления | Merge hell | Скорость обратной связи | Размер пакета | Подходит для |
|--------------------|-----------|------------------------|--------------|-------------|
| Trunk-Based Development | Нет | Максимальная | Минимальный | Любые команды |
| Feature Branching | Да (долгоживущие ветки) | Задержана до слияния | Большой | Небольшие изолированные задачи |
| GitFlow | Управляемо (структура) | Средняя (уровни PR) | Контролируемый | Большие команды, несколько сред |

| Подход | Монолит | Микросервисы |
|--------|---------|-------------|
| Деплой | «Всё или ничего» | Независимый деплой компонентов |
| Масштабирование | Вертикальное, сложное | Горизонтальное, независимое |
| Canary/Blue-Green | Для всего приложения сразу | Для конкретного сервиса |
| Управление зависимостями | Простое | Сложное |

## Примеры

```ruby
# Интеграция и проверка Feature Flag (Ruby) через Unleash
require 'unleash'
require 'unleash/context'
unleash = Unleash::Client.new({
    url: 'http://gitlab.com/api/v4/feature_flags/unleash/42',
    app_name: 'production',
    instance_id: '29QmjsW6KngPR5JNPMWx'
})
unleash_context = Unleash::Context.new
unleash_context.user_id = "123"
if unleash.is_enabled?("my_feature_name", unleash_context)
    puts "Feature enabled"
else
    puts "Feature disabled!"
end
```

```groovy
// Jenkins Declarative Pipeline: Manual Approval для релиза в Production
stage('Deploy'){
  if(env.BRANCH_NAME == 'develop' || env.BRANCH_NAME == 'preprod'){
     build job: "watchlist-deployment/${env.BRANCH_NAME}"
  }
  if(env.BRANCH_NAME == 'master'){
     timeout(time: 2, unit: "HOURS") {
        input message: "Approve Deploy?", ok: "Yes"
     }
     build job: "watchlist-deployment/master"
  }
}
```

> `timeout(time: 2, unit: "HOURS")` — ждёт подтверждения до 2 часов. `input message` выводит диалог «Approve Deploy?». При нажатии «Yes» запускается job развёртывания в master. Если 2 часа истекают — pipeline прерывается.

## Важно / подводные камни / best practices

- **Trunk-Based Development + Gated Commits**: разработчики фиксируют в основную ветку ежедневно (или чаще); pipeline автоматически проверяет изменения до фактического слияния; уменьшает размер пакета, исключает «ад слияний», код всегда в состоянии «готов к релизу»
- **Shadow launch с Feature Flags**: деплой без риска — реальный трафик в фоне; мгновенное отключение без отката релиза
- **Blue-Green + изменение схемы БД**: нельзя изменять или удалять таблицы/колонки, пока обе версии работают на общей БД; решение — шаблон «расширение-сжатие» (expand-contract): сначала добавить новые поля, потом удалить старые после полного перехода
- **Никогда не сливать deployment и release**: деплой в рабочее время без стресса, релиз — по решению бизнеса через Feature Flag
- **Антипаттерн: пакетные релизы за месяцы**: долгая настройка тестовых сред → «героические усилия» на выходных; решение — частые мелкие деплои
- **Антипаттерн: Feature Branching с редкими слияниями**: чем реже фиксация в основную ветку, тем больше пакет изменений и тем выше риск дестабилизации master; тестирование интегрированной системы откладывается до самого конца

## Карточки для повторения

#flashcards/cicd/branching

**Trunk-Based Development**:::Стратегия разработки, при которой разработчики часто сливают изменения в главную ветку (trunk), даже если функция не завершена; отказ от долгоживущих веток; слияния несколько раз в день

**Feature Branching**:::Стратегия ветвления: новая ветка создаётся для каждой функции, разрабатывается до завершения, затем сливается с главной; ветки становятся «долгоживущими» → merge hell

**GitFlow**:::Стратегия из 5 типов веток: master (production) / preprod (staging) / develop (интеграция) / feature/X (разработка) / hotfix/X (исправления production); логична для больших команд, избыточна для маленьких

**Feature Flag (Unleash)**:::Механизм выборочного включения/отключения функции без деплоя кода; GitLab использует библиотеку Unleash; разделяет deployment и release

**Rolling Deployment**:::Последовательная замена старых экземпляров новыми по месту размещения; при сбое — 6–20 мин суммарного простоя (обнаружение + откат = новое rolling)

**Blue-Green Deployment**:::Создаётся новый набор экземпляров; трафик переключается на них только после готовности; откат — мгновенный переброс трафика обратно; требует двукратных ресурсов

**Canary Release**:::Обновляется один «канареечный» экземпляр + небольшой % трафика; если метрики в норме — деплой продолжается; экономия ресурсов, но сложная маршрутизация

**Cluster Immune System**:::Расширение Canary: мониторинг связан с pipeline; автоматический откат при отклонении метрик от заданных показателей; без участия человека

**A/B Testing**:::Несколько версий кода одновременно для разных сегментов пользователей через Feature Flags; цель — проверка бизнес-гипотез

Алгоритм GitFlow
?
1. develop создаётся от master
2. preprod создаётся от develop
3. feature/X создаётся от develop; PR после завершения
4. CI проверяет PR → merge в develop, ветка удаляется
5. Деплой в sandbox/development
6. PR: develop → preprod → деплой в staging
7. preprod → master → production (ручное одобрение)
8. Hotfix: hotfix/X от master → merge в master и develop

Алгоритм Feature Flags (Unleash) в коде
?
1. Создать флаг в GitLab UI (Deployments → Feature Flags) со стратегией
2. Импортировать Unleash Client, настроить URL GitLab API + instance_id
3. Создать контекст (unleash_context) с user_id
4. Вызвать is_enabled?("flag_name", context) → new code / old code

Алгоритм Canary Release
?
1. Запустить канареечный экземпляр v_new рядом с базовыми v_old
2. Направить небольшой % трафика на канарейку
3. Сравнить метрики канарейки с базовыми
4. Если метрики ≥ базовых → деплой на остальные узлы

Edge case: Rolling Deployment + программный сбой?::Обнаружение займёт 3–10 мин (трафик идёт на сломанный экземпляр); откат = новое rolling обновление назад (5–15 мин); суммарный простой у клиентов: 6–20+ мин

Edge case: Blue-Green + изменение схемы БД (удаление колонок)?::Старая (синяя) и новая (зелёная) версии не могут работать с одной БД → нужны две БД → риск потери транзакций при откате; решение: шаблон «расширение-сжатие» (никогда не изменять существующие объекты)

Edge case: Feature Branching + редкие слияния?::«Ад слияний» (merge hell); тестирование интегрированной системы откладывается; большой пакет изменений → высокий риск дестабилизации master при слиянии; задержка обратной связи от CI

Edge case: GitFlow в маленьком проекте?::Избыточно: 4–5 параллельных уровней веток, строгая координация PR; накладные расходы превышают пользу

Развёртывание (deployment) vs Релиз (release)?::Deployment — доставка кода в среду; Release — открытие доступа клиентам. Разделение: деплой в рабочее время без стресса, бизнес активирует Feature Flag в нужный момент

Shadow launch с Feature Flags — что это?::Код развёртывается в production, но скрыт под флагом; реальный трафик обрабатывается в фоне (нагрузочное тестирование); при сбое — мгновенное отключение без отката релиза

Blue-Green vs Canary: ресурсы и откат?::Blue-Green: ×2 ресурсы, мгновенный откат (переключить трафик). Canary: ×1 (инкрементально), откат сложнее (маршрутизация + изоляция метрик)

Jenkins: ручное подтверждение деплоя в production?::Блок `timeout(time: 2, unit: "HOURS") { input message: "Approve Deploy?", ok: "Yes" }` — ждёт 2 часа; при истечении — pipeline прерывается
