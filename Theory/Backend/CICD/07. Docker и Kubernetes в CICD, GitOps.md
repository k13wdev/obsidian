# 07. Docker и Kubernetes в CI/CD, GitOps

## Суть и контекст

CI/CD pipeline собирает Docker-образы, публикует их в реестр и разворачивает в Kubernetes через kubectl или Helm. GitOps — более зрелый подход: конфигурация инфраструктуры и приложения хранится в Git, изменения применяются автоматически через агент внутри кластера (Pull-based), без push-деплоев из CI runner-а. Преимущество: кластер всегда синхронизирован с состоянием Git.

---

## Ключевые идеи

- Multi-stage build: промежуточные слои не попадают в финальный образ; Go: 300 МБ → 8 МБ
- Docker-in-Docker (dind): сборка контейнеров внутри CI job; --docker-privileged = root к хост-машине
- GitOps Pull-model: агент (agentk) в кластере сам синхронизирует состояние с Git; нет push из CI
- GitOps Push-model (kubectl apply в CI): проще, но дрейф конфигурации при изменениях вне CI
- Rollback немедленно: не ждать fix forward; в Helm: `helm rollback`
- Docker layer cache: порядок инструкций критичен — зависимости до исходного кода
- Zero-downtime: K8s RollingUpdate + корректные probe в Deployment
- ArgoCD / Flux — В предоставленных источниках не раскрыто

---

## Определения и термины

**Multistage build (Многоэтапная сборка)** — функция Docker для создания контейнеров, при которой промежуточные слои сборки (с инструментами компиляции) не сохраняются в финальном образе — копируется только скомпилированный бинарный файл. Позволяет уменьшить размер образа (например, Go-приложение: 300 МБ → 8 МБ).

**Docker-in-Docker (dind)** — тип сервисного контейнера в pipeline, позволяющий выполнять сборку новых контейнеров внутри уже контейнеризованной CI/CD среды. По умолчанию использует флаг `--docker-privileged`, предоставляющий root-доступ к хост-машине.

**GitOps** — практика DevOps, при которой конфигурация инфраструктуры и приложения хранится в системе контроля версий (Git). Изменения применяются автоматически через фиксацию кода. Pull-based модель GitOps использует агента внутри кластера Kubernetes, устраняя необходимость push-деплоев через CI pipeline.

**Helm Chart** — пакет предварительно сконфигурированных ресурсов Kubernetes. Включает `Chart.yaml` (метаданные), `values.yaml` (настройки по умолчанию) и `templates/` (манифесты). Установленный экземпляр чарта — **Release**. Откат: `helm rollback`.

**Rollback (Откат)** — возврат к предыдущему релизу с помощью переключателей, удаления сломанных сервисов или системных команд при сбое развёртывания. Выполняется немедленно, без ожидания исправления бага (fix forward) — для быстрого восстановления работоспособности. В Helm — `helm rollback`.

**Blue-Green Deployment** — создание полностью нового набора экземпляров (зелёная среда) и перевод трафика с оригинальных (синяя среда) только после их полной готовности.

**Canary Deployment** — метод, при котором развёртывание ПО выполняется на небольшой группе серверов для небольшого числа клиентов для проверки на реальном трафике.

**Zero-downtime deployment (RollingUpdate)** — стратегия Kubernetes `RollingUpdate`, позволяющая развёртывать изменения с 0% времени простоя.

**ArgoCD / Flux** — В предоставленных источниках не раскрыто. Инструменты упоминаются только в запросе, данные отсутствуют.

## Как устроено

### GitLab Agent для Kubernetes (Pull-based GitOps)

1. Агент (`agentk`) устанавливается в кластер Kubernetes
2. Агент устанавливает двунаправленное соединение (gRPC bidirectional streaming) с сервером GitLab (`kas`)
3. Разработчик обновляет конфигурацию в Git-репозитории
4. Агент автоматически выявляет изменения конфигурации
5. Агент обновляет состояние кластера, извлекая данные из GitLab — в обход сетевых экранов, препятствующих входящим соединениям

### Multi-stage Docker build (Go-пример)

1. Стадия 1: `FROM golang:1.16.5 AS builder` → `go build` (компиляция)
2. Стадия 2: `FROM alpine:latest` (минимальный базовый образ)
3. `COPY --from=builder /.../app .` — переносится только готовый бинарный файл
4. `docker build` в CI компилирует слои, промежуточные файлы не попадают в итоговый контейнер

### Деплой Helm Chart в Kubernetes через CI

1. `helm package chart` → создаёт архив `.tgz`
2. `helm repo index` → обновляет метаданные `index.yaml`
3. Архив + индекс загружаются в Helm-репозиторий (например, GitHub Pages)
4. Этап Deploy: `helm upgrade --install watchlist ./watchlist -f values.override.yaml --set metadata.git.commitId=...`
5. Kubernetes применяет RollingUpdate → 0% downtime

## Детали и нюансы

- **Docker layer caching**: инструкции в Dockerfile должны идти строго: `COPY requirements.txt .` → `RUN pip install` → `COPY main.py .`; если исходный код копируется до установки зависимостей — cache инвалидируется при любом изменении кода → долгая переустановка зависимостей при каждом коммите
- **dind + privileged mode**: требует root-доступа к хост-машине runner-а; потенциальный container escape (см. Security); альтернативы: Kaniko, Buildah — сборка без privileged
- **Push-model дрейф**: kubectl apply в CI runner — состояние кластера может разойтись с Git при ручных изменениях; Pull-based GitOps (agentk) — кластер сам синхронизирует состояние
- **Docker не на worker**: pipeline c `docker build` падает с "docker: command not found"; решение: label `workers` для маршрутизации заданий только на узлы с Docker Engine

## Сравнения и trade-offs

| Подход деплоя | Модель | Дрейф конфигурации | Сложность |
|--------------|--------|-------------------|---------|
| Push (kubectl apply в CI runner) | CI runner → K8s | Возможен | Низкая |
| Pull (GitOps agentk) | K8s агент ← Git | Минимальный (self-healing) | Выше |

| Сборка образа | Размер | Сложность Dockerfile |
|-------------|-------|---------------------|
| Single-stage (один FROM) | Большой (300 МБ для Go) | Простой |
| Multi-stage (builder + runtime) | Маленький (8 МБ для Go) | Сложнее |

| Docker Registry | Хостинг | Интеграция |
|----------------|---------|-----------|
| GitLab Container Registry | GitLab (SaaS/Self-hosted) | Нативная с GitLab CI |
| Amazon ECR | AWS | aws ecr get-login-password |
| Google Container Registry (GCR) | GCP | gcloud auth configure-docker |
| Nexus / JFrog | Self-hosted | docker login + --insecure-registry |

## Примеры

```yaml
# GitLab CI: сборка и push Docker-образа с Docker-in-Docker
publish_to_container_registry:
  stage: publish
  image: docker:stable
  services:
    - docker:dind
  script:
    - docker login -u "hfc-generic-build" -p "<deploy token password>" $CI_REGISTRY
    - docker build -t $IMAGE .
    - docker push $IMAGE
```

```bash
# Аутентификация Jenkins в Amazon ECR
aws ecr get-login-password --region REGION | docker login \
  --username AWS --password-stdin \
  ACCOUNT_ID.dkr.ecr.REGION.amazonaws.com/mlabouardy/movies-loader
```

```bash
# Аутентификация и push в Google Container Registry (GCR)
gcloud auth configure-docker
docker tag mlabouardy/movies-loader eu.gcr.io/PROJECT_ID/mlabouardy/movies-loader
docker push eu.gcr.io/PROJECT_ID/mlabouardy/movies-loader
```

```groovy
// Jenkins: деплой манифестов в Kubernetes
stage('Deploy'){
  sh 'kubectl apply -f deployments/'
}
```

```groovy
// Jenkins: обновление Helm-релиза с инъекцией метаданных
helm upgrade --install watchlist ./watchlist -f values.override.yaml \
  --set metadata.jenkins.buildTag=${env.BUILD_TAG} \
  --set metadata.git.commitId=${commitID()}
```

> `--set` переопределяет аннотации на лету: Build ID и Commit ID инъектируются в K8s манифесты → точная диагностика работающего кластера.

## Важно / подводные камни / best practices

- **Multi-stage build обязателен для production**: Go/Java/C++ образы с build-инструментами — сотни МБ лишнего; runtime-слой должен быть минимальным (alpine или distroless)
- **Порядок инструкций в Dockerfile**: зависимости (`COPY requirements.txt` + `RUN pip install`) строго перед исходным кодом (`COPY . .`) — иначе cache инвалидируется при каждом коммите
- **Pull-based GitOps предпочтительнее Push**: кластер самостоятельно синхронизируется с Git; нет дрейфа при ручных изменениях; agentk не требует входящих соединений (firewall-friendly)
- **Rollback немедленно**: не ждать исправления бага; `helm rollback` или смена алиаса Lambda; MTTR важнее MTBF
- **Инъекция метаданных в Helm**: `--set metadata.git.commitId` → `kubectl describe pod` показывает точный коммит → упрощает диагностику
