# RabbitMQ — основы, архитектура и routing

## Суть и контекст

RabbitMQ — брокер сообщений (message broker), реализующий протокол AMQP 0-9-1. Выступает посредником между приложениями, обеспечивая маршрутизацию, доставку и временное хранение сообщений. Снижает связность (coupling) микросервисов, позволяя реализовать асинхронное взаимодействие (Event-Driven Architecture). RabbitMQ работает по модели «умный брокер, тупой потребитель» — брокер активно следит за статусом доставки, проталкивает (push) сообщения потребителям и удаляет их после подтверждения обработки.

---

## Ключевые идеи

- Message broker — middleware, реализующий паттерны Pub-Sub и Point-to-Point
- RabbitMQ реализует спецификацию AMQP 0-9-1, использующую бинарное кодирование и двунаправленные RPC-запросы
- Сообщение состоит из тела (Body, бинарные данные), свойств (Properties) и заголовка (Header)
- RabbitMQ не парсит тело сообщения — передаёт как «чёрный ящик» (opaque byte array)
- Маршрутизация построена на цепочке: Producer → Exchange → Binding → Queue → Consumer
- Тип Exchange определяет алгоритм маршрутизации (Direct, Fanout, Topic, Headers)
- Channel — логическое соединение внутри TCP Connection, обеспечивает мультиплексирование
- Virtual Host изолирует Exchange, Queue и пользователей в рамках одного сервера
- При ошибке AMQP (попытка записать в несуществующий exchange) RabbitMQ закрывает Channel, а не Connection
- Тело сообщения хранится в памяти в единственном экземпляре; в очереди хранятся только ссылки

## Определения и термины

**Message Broker (Брокер сообщений)** — программная система (middleware), полностью или частично реализующая паттерны обмена данными Pub-Sub и Point-to-Point, выступающая посредником между приложениями для маршрутизации, доставки и временного хранения сообщений. Берёт на себя инфраструктурные задачи коммуникации, снижая связность (coupling) микросервисов.

**AMQP (Advanced Message Queuing Protocol)** — открытый протокол промежуточного ПО для передачи сообщений, определяющий сетевой wire-уровень и логическую модель (обменники, очереди, привязки). RabbitMQ реализует спецификацию AMQP 0-9-1. Протокол использует бинарное кодирование и поддерживает двунаправленные RPC-запросы.

**Producer (Publisher)** — приложение, формирующее и отправляющее сообщения в брокер. Продюсер указывает сетевой адрес брокера, формирует заголовок и тело сообщения. Не взаимодействует напрямую с получателями.

**Consumer (Потребитель)** — приложение-подписчик, устанавливающее TCP-соединение с брокером для получения и обработки сообщений из очереди. Может работать в синхронном режиме (pull/get) или асинхронном (push/consume).

**Message (Сообщение)** — основная единица (атомарный блок) данных в системе AMQP, состоящая из тела (Body) с бинарными данными, свойств (Properties) и заголовка (Header). Тело сообщения RabbitMQ не парсит и передаёт как «чёрный ящик» (opaque byte array).

**Exchange (Точка обмена / Обменник)** — компонент модели AMQ, получающий сообщения от продюсера и перенаправляющий их в ноль или более очередей. Алгоритм маршрутизации определяется типом обменника и настроенными правилами привязки.

**Queue (Очередь)** — структура данных в памяти или на диске для хранения сообщений до их доставки потребителю по принципу FIFO (с учётом приоритетов). Работает по модели «weak-FIFO», так как сообщения могут потребляться приоритетными консьюмерами быстрее остальных. Сообщения удаляются из очереди после подтверждения (ack).

**Binding (Привязка)** — правило, связывающее Exchange и Queue, указывающее обменнику, в какую очередь направлять определённые сообщения. Привязка содержит Binding Key, который сверяется с Routing Key сообщения.

**Routing Key (Ключ маршрутизации)** — строковый атрибут сообщения, используемый Exchange для принятия решения о маршрутизации в очередь при сравнении с ключом Binding. В Direct exchange требуется точное совпадение; в Topic exchange может содержать логические блоки, разделённые точками.

**Virtual Host (vhost)** — механизм RabbitMQ для создания множества изолированных логических сред в рамках одного физического сервера. Каждый vhost имеет собственные группы пользователей, свои Exchange, очереди и права доступа. Клиент подключается строго к одному vhost.

**Connection (Соединение)** — физическое TCP-соединение между приложением-клиентом и сервером RabbitMQ. Инициализируется через RPC handshake (Connection.Start → Connection.StartOk).

**Channel (Канал)** — логическое (виртуальное) соединение внутри установленного TCP-соединения (Connection), обеспечивающее мультиплексирование. Каждому каналу присваивается целочисленный ID. На каждый канал внутри RabbitMQ выделяется память. Если происходит ошибка на уровне AMQP (например, попытка записать в несуществующий exchange), RabbitMQ закрывает Channel, а не Connection.

## Как устроено

### Жизненный цикл сообщения

1. Producer открывает TCP Connection и создаёт в нём Channel
2. Producer формирует сообщение (бинарное тело + свойства) и отправляет команду `Basic.Publish`. Сообщение уходит в виде трёх фреймов: Method frame, Content Header frame, Body frame
3. Команда направляется в определённый Exchange с указанием Routing Key
4. Exchange ищет совпадение Routing Key в своих Binding с очередями
5. Если совпадение найдено, RabbitMQ помещает ссылку на сообщение в подходящую очередь (или несколько). Само тело сообщения хранится в памяти в единственном экземпляре
6. Если Consumer подключён с командой `Basic.Consume`, RabbitMQ проталкивает (Push) ему сообщение
7. Consumer обрабатывает сообщение и отправляет `Basic.Ack` обратно в брокер
8. RabbitMQ удаляет сообщение из очереди и из памяти, как только удалены все ссылки на него

### Типы Exchange

**Direct Exchange (amq.direct)** — отправляет сообщение в очередь, если Routing Key сообщения строго равен Binding Key. Идеален для адресной (point-to-point) доставки.

**Fanout Exchange (amq.fanout)** — игнорирует Routing Key. Безусловно дублирует и отправляет сообщение во все привязанные к нему очереди. Используется для pub-sub или broadcast.

**Topic Exchange (amq.topic)** — выполняет маршрутизацию по паттернам. Routing Key разбивается точками (например, `image.new.profile`). В Binding используются wildcards: `*` заменяет ровно одно слово, `#` заменяет ноль или больше слов.

**Headers Exchange (amq.headers / amq.match)** — игнорирует Routing Key, проверяет таблицу заголовков (headers property). Binding содержит аргумент `x-match`: при значении `any` достаточно одного совпадения заголовка, при `all` должны совпасть все.

### Анатомия фрейма AMQP

Блок «Frame Header» (3 байта), состоящий из трёх полей: 1 байт — тип фрейма (Frame type), 2 байта — номер канала (Channel number), 4 байта — размер фрейма (Frame size). Далее блок «Frame payload» — размер меняется в зависимости от типа фрейма (Method, Content Header, Body). В конце — маркер завершения «End-byte marker» (значение ASCII 206 / 0xCE).

### Атрибуты очередей

| Атрибут | Описание |
|---------|----------|
| Durable | При `durable=True` очередь сохраняется на диск и переживает рестарт брокера |
| Exclusive | Доступна только для того Connection, в котором создана. Удаляется при закрытии соединения |
| Auto-delete | Удаляется автоматически, когда отписывается последний consumer |
| TTL (x-message-ttl) | Время жизни (в мс) всех сообщений внутри очереди |
| x-expires | Время в мс, после которого очередь удалит сама себя, если к ней нет обращений |
| Priorities (x-max-priority) | Задаётся при объявлении (до 255). Обрабатывает сообщения с учётом свойства `priority` |
| delivery-mode | Для сохранения сообщения на диск при рестарте: очередь должна быть `durable`, и у сообщения `delivery-mode=2` |

### Паттерн Request-Response (RPC)

Publisher отправляет сообщение, указывая временную `reply-to` очередь (и `correlation-id` для сопоставления). Consumer (Worker) обрабатывает данные и шлёт результат обратно в Exchange, используя значение `reply-to` в качестве Routing Key.

### Exchange-to-Exchange bindings (E2E)

В RabbitMQ спецификация AMQP расширена: Exchange можно привязать к другому Exchange (`Exchange.Bind`). Роутинг работает по тем же принципам. Можно комбинировать топологии (например, Topic Exchange направляет сообщение в Headers Exchange).

### Ordering и Consistent Hashing Exchange

RabbitMQ гарантирует **global ordering** в рамках одной очереди: сообщения доставляются строго в хронологическом порядке. Однако при использовании **competing consumers** (несколько консьюмеров на одну очередь) порядок обработки нарушается — сообщения обрабатываются параллельно с разной скоростью.

**Решение: Consistent Hashing Exchange** (`x-consistent-hash`):

1. Создаётся обменник типа `x-consistent-hash`
2. К нему привязываются несколько очередей с указанием веса (weight)
3. Обменник хэширует Routing Key сообщения и закрепляет хэш за конкретной очередью
4. Сообщения с одинаковым ключом (например, ID заказа) всегда попадают строго в одну очередь и обрабатываются строго одним консьюмером — порядок сохраняется

## Детали и нюансы

### Unroutable messages

Если Producer отправляет сообщение в Exchange, но для его Routing Key не находится ни одной привязанной очереди:
- Если **не** установлен флаг `mandatory=True`, RabbitMQ молча отбрасывает (silently drop) сообщение — **риск потери данных**
- Если установлен `mandatory=True`, брокер возвращает сообщение обратно клиенту командой `Basic.Return`
- Альтернатива: механизм «Alternate Exchange», который определит «запасной» обменник для маршрутизации таких сообщений

### Фрагментация больших сообщений

При превышении максимального размера фрейма (более 131 КБ по умолчанию), payload (body) разбивается на несколько Body Frames. Ограничение на размер каждого фрейма данных: макс. размер − 7 байт (затраты на заголовки фрейма).

### Иммутабельность настроек очередей

Если приложение пытается объявить (`Basic.QueueDeclare`) уже существующую очередь, но с другими параметрами (например, меняет `durable` или добавляет `x-message-ttl`), RabbitMQ немедленно закрывает Channel (403 ошибка/прекондиция). Решение: удалить очередь и пересоздать заново.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|-------------------|
| Direct Exchange | Простой, точное совпадение ключей, предсказуемый роутинг | Только point-to-point, нет паттернов | Адресная доставка конкретному сервису |
| Fanout Exchange | Broadcast во все очереди, не нужен routing key | Нет фильтрации, все получают всё | Pub-sub, рассылка уведомлений |
| Topic Exchange | Гибкая маршрутизация по паттернам с wildcards | Сложнее настройка и отладка | Сложная маршрутизация по категориям |
| Headers Exchange | Маршрутизация по заголовкам, не зависит от routing key | Наименее производительный, сложная настройка | Когда routing key недостаточен |

| Свойство | durable=True + delivery-mode=2 | Transient (в памяти) |
|----------|-------------------------------|---------------------|
| Сохранение при рестарте | Да | Нет |
| Производительность | Ниже (запись на диск) | Выше (только RAM) |
| Когда использовать | Критичные данные | Метрики, логи, некритичные данные |

## Примеры

### Базовая публикация сообщения (Java Client)

```java
ConnectionFactory factory = new ConnectionFactory();
factory.setHost("localhost");
Connection connection = factory.newConnection();
Channel channel = connection.createChannel();

channel.queueDeclare("hello", false, false, false, null);
String message = "Hello World!";
channel.basicPublish("", "hello", null, message.getBytes());
```

Создаётся фабрика, открывается TCP connection, создаётся channel. Метод `queueDeclare` объявляет не-durable, не-exclusive, не-autodelete очередь без дополнительных аргументов. Отправляется массив байт в default exchange (пустая строка `""`) с ключом маршрутизации по имени очереди.

### Временная exclusive очередь для RPC (Python/rabbitpy)

```python
queue_name = 'response-queue-%i' % os.getpid()
response_queue = rabbitpy.Queue(channel, queue_name,
                                exclusive=True, auto_delete=True)
response_queue.declare()
response_queue.bind('direct-rpc-requests', 'detect-faces')

for message in response_queue:
    message.ack()
    # process response
```

Имя генерируется динамически с учётом PID процесса. Очередь эксклюзивна и автоудаляема. Привязывается к exchange `direct-rpc-requests` по ключу `detect-faces`. Подтверждение сообщений `message.ack()`.

### Включение Management UI через CLI

```bash
rabbitmq-plugins enable rabbitmq_management rabbitmq_federation_management
```

Активирует веб-интерфейс (порт 15672 по умолчанию) и возможности федерации без перезагрузки брокера (в версиях ≥ 3.4.0).

## Важно / подводные камни / best practices

- **Задавайте content-type** — использовать самоописываемые сообщения, задавая стандартизированный `content-type` (например, `application/json`) в `Basic.Properties`. Избавляет consumer'ы от необходимости «угадывать» структуру данных для десериализации, предотвращает краши при неизвестном формате
- **Используйте Dead-Letter Exchanges (DLX)** — аргумент `x-dead-letter-exchange` в параметрах очереди. Позволяет настроить «корзину» для отброшенных или истёкших по TTL сообщений. Помогает при отладке «битых» сообщений без их потери
- **Не используйте Basic.Get в цикле** — синхронный метод `Basic.Get` в бесконечном цикле (Polling) на каждое сообщение вызывает полный цикл RPC, снижая пропускную способность минимум в 2 раза по сравнению с `Basic.Consume` (Push модель)
- **Настраивайте prefetch (QoS)** — `Basic.QoS prefetch_count` ограничивает количество неподтверждённых сообщений, защищая RAM потребителя от переполнения и позволяя балансировать нагрузку. Без явного ограничения при `no_ack=True` брокер будет безудержно слать сообщения
- **Настройки очередей иммутабельны** — попытка переобъявить очередь с другими параметрами приведёт к закрытию Channel

Жизненный цикл сообщения в RabbitMQ
?
1. Producer открывает TCP Connection и создаёт Channel
2. Producer формирует сообщение и отправляет Basic.Publish (3 фрейма: Method, Content Header, Body)
3. Команда направляется в Exchange с указанием Routing Key
4. Exchange ищет совпадение Routing Key в Binding с очередями
5. Ссылка на сообщение помещается в подходящую очередь (тело хранится в единственном экземпляре)
6. RabbitMQ проталкивает (Push) сообщение Consumer'у
7. Consumer отправляет Basic.Ack
8. RabbitMQ удаляет сообщение из памяти

Как работает Direct Exchange?::Отправляет сообщение в очередь при строгом равенстве Routing Key и Binding Key. Идеален для point-to-point доставки

Как работает Fanout Exchange?::Игнорирует Routing Key, безусловно дублирует сообщение во все привязанные очереди. Используется для pub-sub или broadcast

Как работает Topic Exchange?::Маршрутизация по паттернам. Routing Key разбивается точками (image.new.profile). Wildcards: * — ровно одно слово, # — ноль или больше слов

Как работает Headers Exchange?::Игнорирует Routing Key, проверяет таблицу заголовков. x-match=any — достаточно одного совпадения, x-match=all — все должны совпасть

Что произойдёт, если для Routing Key нет привязанной очереди (unroutable message)?::Без mandatory=True — RabbitMQ молча отбросит сообщение. С mandatory=True — вернёт через Basic.Return. Альтернатива — Alternate Exchange

Что произойдёт при попытке переобъявить очередь с другими параметрами?::RabbitMQ немедленно закрывает Channel (403 ошибка). Настройки очередей иммутабельны — нужно удалить и пересоздать

Из каких фреймов состоит AMQP-сообщение при отправке?::Method frame, Content Header frame, Body frame. Frame Header: 1 байт тип + 2 байта канал + 4 байта размер. End-byte marker: ASCII 206 (0xCE)

Чем опасен Basic.Get в цикле (Polling)?::Синхронный RPC на каждое сообщение, пропускная способность минимум в 2 раза ниже, чем у Basic.Consume (Push модель)

Зачем настраивать prefetch (QoS)?::Ограничивает неподтверждённые сообщения, защищает RAM от переполнения. Без ограничения при no_ack=True брокер шлёт сообщения непрерывно до заполнения TCP-буфера

Что такое Exchange-to-Exchange bindings?::Расширение AMQP в RabbitMQ: Exchange можно привязать к другому Exchange через Exchange.Bind. Позволяет комбинировать топологии роутинга

RabbitMQ работает по модели ==«умный брокер, тупой потребитель»== — брокер ==активно следит== за статусом доставки

В Direct exchange требуется ==точное совпадение== Routing Key и Binding Key

В Topic exchange wildcard `*` заменяет ==ровно одно слово==, `#` заменяет ==ноль или больше слов==

Default exchange имеет имя =="" (пустая строка)==, маршрутизирует по ==имени очереди==

Максимальный размер фрейма по умолчанию — ==131 КБ==, при превышении payload разбивается на несколько Body Frames

Для сохранения сообщения на диск: очередь должна быть ==durable==, а у сообщения ==delivery-mode=2==

Consistent Hashing Exchange:::Тип обменника (x-consistent-hash), хэширующий Routing Key и закрепляющий сообщения с одинаковым ключом за одной очередью. Решает проблему нарушения порядка при competing consumers

Что происходит с ordering при competing consumers?::Несколько консьюмеров обрабатывают сообщения параллельно с разной скоростью — хронологический порядок обработки нарушается. Решение: Consistent Hashing Exchange
