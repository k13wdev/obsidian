## Суть и контекст

Конспект охватывает внутренние механизмы PostgreSQL, критически важные для оптимизации производительности и обслуживания:

1. **Алгоритмы соединений** (Nested Loop, Hash Join, Merge Join) и Bitmap Index Scan.
2. **WAL (Write-Ahead Logging)** — механизм обеспечения durability и crash recovery.
3. **VACUUM** — очистка мёртвых версий строк, autovacuum, защита от Transaction ID Wraparound.
4. **Write amplification, long-running transactions** — практические последствия MVCC и страничного хранения.

Общие сведения об оптимизаторе, стоимостной модели и EXPLAIN см. в [[07. Оптимизация запросов]].

---

## Ключевые идеи

- Три основных алгоритма соединения: Nested Loop (для малых наборов с индексом), Hash Join (для больших без сортировки), Merge Join (для предварительно отсортированных данных).
- Bitmap Index Scan — "мост" между случайным I/O и последовательным сканированием.
- При большом числе JOIN планировщик использует эвристики и генетический алгоритм (GEQO) для сужения пространства поиска.
- WAL гарантирует: изменения файлов данных записываются на диск только после сброса журнальных записей. При crash recovery — REDO от последнего checkpoint.
- VACUUM удаляет мёртвые версии строк (dead tuples), обновляет Free Space Map и Visibility Map, замораживает старые XID.
- VACUUM FULL переписывает таблицу целиком, но требует ACCESS EXCLUSIVE lock — парализует всю работу с таблицей.

## Определения и термины

Базовые термины (Query planner, Cost, Sequential/Index Scan, Statistics) — в [[07. Оптимизация запросов]]. Ниже — термины, специфичные для внутреннего устройства PostgreSQL.

**Bitmap Index Scan / Bitmap Heap Scan** — двухэтапный метод: сканирование индекса → построение битовой карты (bitmap) расположения строк → чтение heap в физическом порядке, минимизируя случайный доступ к диску.

**OID (Object Identifier)** — 32-битное целое число, первичный ключ во всех таблицах системного каталога PostgreSQL.

**Кластер баз данных** — набор баз данных, управляемых одним экземпляром PostgreSQL. Хранится в каталоге PGDATA.

**LSN (Log Sequence Number)** — 64-битное смещение в байтах до журнальной записи относительно начала WAL. Записывается в заголовке каждой страницы данных.

**Грязная страница (Dirty page)** — страница в буферном кеше, изменённая в памяти, но ещё не записанная на диск.

**Nested Loop** — алгоритм соединения, при котором для каждой строки внешнего (outer) узла вызывается внутренний (inner) узел-сканер.

**Hash Join** — алгоритм соединения: данные внутреннего узла полностью загружаются в хеш-таблицу в оперативной памяти, затем внешний узел ищет совпадения по хешу.

**Merge Join** — алгоритм соединения, требующий предварительной сортировки обоих входов по ключам соединения, затем параллельное слияние.

**WAL (Write-Ahead Logging)** — стандартный метод обеспечения целостности данных, гарантирующий, что изменения файлов данных записываются на диск только после сброса журнальных записей в постоянную память.

**Checkpoint (Контрольная точка)** — точка в последовательности транзакций, в которой все грязные (изменённые) страницы данных принудительно сбрасываются на диск, а в WAL записывается специальная метка.

**VACUUM** — команда для освобождения дискового пространства, занятого устаревшими версиями строк (dead tuples), оставшимися после UPDATE и DELETE.

**VACUUM FULL** — версия очистки, требующая эксклюзивной блокировки (ACCESS EXCLUSIVE lock) и полностью переписывающая таблицу в новый файл для возврата пустого места операционной системе.

## Как устроено

### Физическая организация (PGDATA и Fork-файлы)

**Структура PGDATA**:
- `base/` — данные баз данных (каталог для каждой БД)
- `global/` — общие объекты кластера
- `pg_wal/` — файлы журнала WAL (сегменты по 16 МБ)
- `pg_control` — управляющий файл кластера (версия, LSN checkpoint)

**Размер файлов**: растут сегментами по **1 Гбайт**; порядковый номер сегмента в имени файла.

**Слои (Forks)**:
- `main` (основной слой) — табличные / индексные данные; имя = relfilenode
- `_fsm` (Free Space Map) — дерево-структура, хранит степень заполнения страниц; используется при вставках для поиска свободного места
- `_vm` (Visibility Map) — 2 бита на страницу: all-visible + all-frozen; ускоряет Index-Only Scan и агрессивный VACUUM
- `_init` (Initialization) — для `UNLOGGED` таблиц; файл нулей для восстановления после сбоя

### Буферный кеш — Clock Sweep

**Компоненты**: массив буферов в shared_buffers + хеш-таблица `(fileID, blockNum) → номер буфера`.

**Алгоритм Clock Sweep (вытеснение)**:
1. «Стрелка» обходит буферы по кругу.
2. При встрече буфера с `usage_count > 0` — уменьшить на 1.
3. Вытесняется первый незакреплённый (`pin_count = 0`) буфер с `usage_count = 0`.
4. Если буфер грязный — сначала сбросить на диск.

**Pinning**: при работе с буфером процесс увеличивает `pin_count`. Закреплённый буфер не может быть вытеснен.

### Источники статистики планировщика

`pg_class.reltuples`, `pg_class.relpages`, `pg_statistic`. Cost-формула и EXPLAIN ANALYZE разобраны в [[07. Оптимизация запросов]].

При большом числе JOIN планировщик переходит на эвристики и генетический алгоритм (GEQO) — отсекает декартовы произведения и сужает пространство поиска.

### WAL — механика

**Компоненты**: файлы сегментов WAL (по 16 МБ, разделены на страницы по 8 КБ). Каждая запись содержит CRC-32C чексумму и ссылку на предыдущую запись. LSN = байтовое смещение от начала WAL.

**Процесс записи**:
1. Транзакция вносит изменения в страницы данных в оперативной памяти.
2. Изменения регистрируются в WAL-буферах.
3. При COMMIT: XLogWrite переносит WAL-буферы в кеш ядра, issue_xlog_fsync сбрасывает на диск (fsync).

**Crash Recovery**:
1. Система читает WAL начиная с REDO-записи последнего checkpoint.
2. Повторяет (REDO / roll-forward) все модификации, не попавшие в файлы данных.

### VACUUM — механика очистки

1. Демон autovacuum проверяет статистику (изменённые/удалённые строки превышают `autovacuum_vacuum_threshold + scale_factor × reltuples`).
2. Сканирование таблицы с использованием Visibility Map (страницы all-visible пропускаются).
3. TID мёртвых строк накапливаются в **префиксном дереве** в памяти (ограничено `maintenance_work_mem`).
4. Очистка **всех индексов** таблицы от ссылок на найденные TID.
5. Очистка самой таблицы — освобождение указателей на dead tuples.
6. Обновление **Free Space Map** для повторного использования при вставках.
7. Обновление **Visibility Map** — помечает страницы all-visible (ускоряет Index-Only Scan).
8. **Заморозка (Freeze)**: когда возраст транзакции достигает `autovacuum_freeze_max_age`, запускается агрессивная очистка — все строки замораживаются (FrozenXID). Защита от Transaction ID Wraparound (переполнения 32-битного XID счётчика).

## Детали и нюансы

- **Bloat при UPDATE/DELETE**: из-за MVCC версии не удаляются сразу. Обычный VACUUM помечает место как свободное, но не уменьшает размер файла на диске. Для возврата места ОС нужен VACUUM FULL (или CLUSTER).
- **VACUUM FULL** захватывает ACCESS EXCLUSIVE lock — полная блокировка таблицы для чтения и записи.
- **Параметры autovacuum**: запуск происходит, когда количество изменённых/удалённых строк превышает порог: `autovacuum_vacuum_threshold` (абсолютное число) + `autovacuum_vacuum_scale_factor` × reltuples (доля таблицы). Для снижения нагрузки на I/O используется **cost-based vacuum delay** — искусственные паузы autovacuum на основе оценки стоимости прочитанных/записанных страниц.
- **Write amplification (усиление записи)** — отношение физического объёма данных, записанных на диск, к логическому объёму изменённых данных. Причины:
    1. **Страничный I/O**: СУБД со страничной структурой (PostgreSQL, Oracle, MySQL InnoDB) не может изменить 1 байт на диске — считывает страницу в буферный кеш, меняет данные, сбрасывает всю страницу целиком. Для записи 50 байт на странице 8 КБ WA может достигать нескольких сотен. Метаданные страницы и неполное заполнение создают WA на уровне 1.05–1.10 даже в идеальных условиях.
    2. **WAL Journaling**: принцип опережающей записи требует писать данные дважды — сначала в WAL (последовательно), затем в файлы данных (произвольно). Объём записи в файлы данных обычно в 5–7 раз превосходит объём записи в журнал из-за страничной структуры файлов данных против компактной записи в WAL. Параметр `full_page_writes`: при первом изменении страницы после контрольной точки в WAL записывается образ всей страницы целиком для защиты от частичной записи (torn pages) — резко увеличивает WA.
    3. **MVCC Heap Bloat**: UPDATE не перезаписывает старую строку, а создаёт новую версию; старая помечается как `xmax`, но физически остаётся в файле. VACUUM очищает место, но не всегда возвращает его ОС, требуя перезаписи страниц для дефрагментации.
    4. **B-Tree Page Splits**: при вставке в заполненную страницу индекса происходит её разделение на две; примерно половина ключей переносится в новую страницу; обновляется родительская страница (информация о границах ключей) — лавинообразная запись нескольких страниц из-за одного ключа.
- **HOT-updates (Heap-Only Tuples)**: если новая версия строки помещается на ту же страницу (благодаря `FILLFACTOR`), WA для индексов становится равным нулю — индексные записи продолжают указывать на ту же страницу, и физический адрес не меняется.
- **`FILLFACTOR` trade-off**: низкий fillfactor (например, 50%) радикально снижает WA при обновлениях, но удваивает объём занимаемого места на диске и замедляет Sequential Scan (нужно читать в 2 раза больше страниц для выборки тех же данных).
- **BRIN для time-series (логов)**: BRIN хранит только min/max значения для диапазонов страниц, занимает в тысячи раз меньше места, чем B-Tree, и почти не генерирует WA при вставках в конец таблицы (обновляется крайне редко).
- **Write amplification в LSM-деревьях**: в LSM-деревьях (RocksDB, Cassandra, Tarantool Vinyl) данные пишутся только добавлением в новые файлы. Фоновые процессы слияния файлов (Level-based или Size-tiered) многократно перечитывают и переписывают одни и те же данные — суммарный WA может достигать нескольких десятков. LSM-движки (например, Aerospike) стараются писать блоками фиксированного размера (wblocks) в обход файловой системы для минимизации износа SSD.
- **Аналитические СУБД (ClickHouse, Vertica)**: WA при записи ниже (данные пишутся огромными блоками, сжимаются и редко обновляются), но WA может резко возрастать при слиянии кусков (merge).
- **SSD и WA**: высокий WA критичен для SSD — флеш-память имеет ограниченный ресурс циклов перезаписи; чрезмерный WA от LSM-сжатия или частых `full_page_writes` физически изнашивает накопитель.
- **Антипаттерн `VACUUM FULL` под нагрузкой**: полностью переписывает таблицу в новый файл (гигантский всплеск записи и WA) и эксклюзивно блокирует доступ даже на чтение.
- **Long-running transactions**: напрямую мешают VACUUM (старый snapshot удерживает dead tuples) и порождают каскады блокировок. Полный разбор, включая взаимодействие таймаутов, `hot_standby_feedback` и риск переполнения `pg_wal/`, см. в [[11. Транзакции, ACID, изоляция и MVCC]].
- **Миграции (ALTER TABLE) и даунтайм**: любой DDL (ALTER TABLE) требует ACCESS EXCLUSIVE lock — ждёт завершения всех текущих транзакций и блокирует старт новых. Для построения индексов под нагрузкой используется `CREATE INDEX CONCURRENTLY` / `REINDEX INDEX CONCURRENTLY` (блокировка понижается до `SHARE UPDATE EXCLUSIVE`), см. [[17. Безопасные миграции и обслуживание]].
- **Nested Loop на больших данных**: если внутренний источник — Seq Scan (не индекс), стоимость возрастает экспоненциально (перемножение строк × стоимость полного сканирования). В таких случаях Hash Join или Merge Join эффективнее.
- **LIMIT + ORDER BY**: actual time в EXPLAIN ANALYZE может быть значительно меньше estimated cost, так как LIMIT прерывает запрос досрочно.
- **Bitmap Index Scan** работает как "мост" между Random I/O и Sequential Scan: индекс выдаёт адреса вразнобой, но bitmap сортирует физические адреса для почти последовательного чтения.
- Оптимизатор не ищет абсолютно идеальный план — используются эвристики и GEQO для отсечения декартовых произведений на ранних этапах.

## Сравнения и trade-offs

| Алгоритм соединения | Плюсы | Минусы | Когда использовать |
|---------------------|-------|--------|--------------------|
| Nested Loop | Эффективен для малых наборов; хорошо работает с Index Scan на inner | Экспоненциальная стоимость при больших inner без индекса | Малый inner set, есть индекс на ключ соединения |
| Hash Join | Не требует сортировки; эффективен для больших наборов | Требует memory для хеш-таблицы inner | Большие наборы, нет индекса, нет предварительной сортировки |
| Merge Join | Эффективен для предварительно отсортированных данных | Требует сортировки обоих входов | Данные уже отсортированы или нужен ORDER BY |

| Метод доступа | Плюсы | Минусы | Когда использовать |
|--------------|-------|--------|--------------------|
| Seq Scan | Последовательное чтение (быстрый I/O) | Читает всю таблицу | Низкая селективность запроса, нет индексов |
| Index Scan | Точечный доступ по индексу | Random I/O к heap | Высокая селективность, точечные запросы |
| Bitmap Index Scan | Сортирует адреса для последовательного heap-чтения | Две фазы (индекс + heap) | Средняя селективность, OR-условия |

| Тип VACUUM | Действие | Блокировка | Возвращает место ОС |
|-----------|---------|-----------|---------------------|
| VACUUM (обычный) | Помечает свободное место в FSM | Не блокирует чтение/запись | Нет |
| VACUUM FULL | Переписывает таблицу в новый файл | ACCESS EXCLUSIVE (полная блокировка) | Да |

## Примеры

```sql
-- Мониторинг Write Amplification (PostgreSQL 16+)
SELECT
    backend_type,
    object,       -- relation (таблицы/индексы) или wal
    context,      -- normal (через shared buffers), vacuum (вне буферов), bulkwrite (COPY)
    writes,
    write_bytes,  -- общий объём физически записанных данных в байтах
    fsyncs
FROM pg_stat_io
WHERE writes > 0;

-- Снижение WA через HOT-updates: резерв свободного места на странице
ALTER TABLE my_table SET (fillfactor = 70);
CREATE INDEX idx_name ON table_name (col) WITH (fillfactor = 90);

-- Обнаружение long-running транзакций
SELECT pid, age(backend_xid), age(backend_xmin), state, query
FROM pg_stat_activity WHERE state = 'idle in transaction';

-- Автоматическое завершение зависших транзакций
SET idle_in_transaction_session_timeout = '5min';

-- Ручное завершение
SELECT pg_terminate_backend(pid);
```

## Важно / подводные камни / best practices

- Антипаттерн: VACUUM FULL для регулярного обслуживания — медленно, эксклюзивная блокировка. Настраивать autovacuum так, чтобы обычный VACUUM справлялся в фоне.
- После массовой загрузки (COPY) запускать `VACUUM ANALYZE` — VACUUM обновит Visibility Map (Index-Only Scan), ANALYZE освежит статистику для планировщика.
- EXPLAIN / EXPLAIN ANALYZE, оценка планов и рекомендации — в [[07. Оптимизация запросов]].

- main (основные данные)
- _fsm (Free Space Map — дерево заполнения страниц)
- _vm (Visibility Map — 2 бита на страницу: all-visible, all-frozen)
- _init (для UNLOGGED таблиц — восстановление после сбоя)

Алгоритм Clock Sweep (вытеснение из буферного кеша)
?
1. Стрелка обходит буферы по кругу
2. usage_count > 0 → уменьшить на 1, пропустить
3. usage_count = 0 и pin_count = 0 → вытеснить (если грязный — сначала сбросить на диск)

LSN (Log Sequence Number):::64-битное смещение в байтах до журнальной записи от начала WAL. Записывается в заголовке каждой страницы данных.

Файлы WAL имеют размер сегмента ==16 МБ==, разделены на страницы по ==8 КБ==

Почему VACUUM FULL не подходит для регулярного обслуживания?::Требует ACCESS EXCLUSIVE lock (полная блокировка таблицы), медленно работает. Лучше настроить autovacuum.

В чём отличие VACUUM от VACUUM FULL?
?
VACUUM: помечает свободное место в FSM, не блокирует чтение/запись, не уменьшает файл
VACUUM FULL: переписывает таблицу, ACCESS EXCLUSIVE lock, возвращает место ОС

Для чего Visibility Map обновляется при VACUUM?::Помечает страницы, где все строки видны всем транзакциям — ускоряет Index-Only Scan.

Что такое GEQO в планировщике PostgreSQL?::Генетический алгоритм для сужения пространства поиска планов при большом числе JOIN-ов.

Write amplification (усиление записи):::Отношение физического объёма данных, записанных на диск, к логическому объёму изменённых данных

Причины write amplification
?
1. Страничный I/O — изменение 50 байт сбрасывает всю 8 КБ страницу
2. WAL — данные пишутся дважды (журнал + файлы данных; объём файлов в 5–7 раз больше WAL)
3. MVCC bloat — новые версии рядом со старыми
4. B-Tree page splits — каскадное обновление родительских узлов

Как снизить write amplification?::Использование FILLFACTOR/PCTFREE для резервирования места в страницах, предотвращая расщепление узлов и миграцию строк

Когда autovacuum запускает очистку таблицы?::Когда число изменённых/удалённых строк превышает autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × reltuples.

Что такое cost-based vacuum delay?::Механизм снижения I/O нагрузки autovacuum — искусственные паузы на основе оценки стоимости прочитанных/записанных страниц.

Write amplification в LSM-деревьях::Данные пишутся компактно (memtable → SSTable), но фоновый compaction перезаписывает данные многократно — мультипликатор может достигать нескольких десятков.

HOT-updates (Heap-Only Tuples):::Если новая версия строки помещается на ту же страницу (благодаря `FILLFACTOR`), WA для индексов становится равным нулю — индексные записи продолжают указывать на ту же страницу.

full_page_writes в PostgreSQL:::Параметр, при котором при первом изменении страницы после контрольной точки в WAL записывается образ всей страницы для защиты от torn pages; резко увеличивает WA.

Trade-off низкого FILLFACTOR (50%)::Радикально снижает WA при обновлениях, но удваивает объём диска и замедляет Sequential Scan (вдвое больше страниц для тех же данных).

Почему BRIN-индексы хороши для time-series/логов?::Хранят только min/max значения для диапазонов страниц — в тысячи раз меньше места, чем B-Tree, и почти не генерируют WA при вставках в конец таблицы.

Почему VACUUM FULL — антипаттерн под нагрузкой?::Полностью переписывает таблицу в новый файл (всплеск записи и WA) и эксклюзивно блокирует доступ даже на чтение.

Причины высокого WA на странице 8 КБ при записи 50 байт::СУБД со страничной структурой не может изменить 1 байт — читает страницу в буферный кеш, меняет данные, сбрасывает всю страницу целиком. WA может достигать нескольких сотен.

Во сколько раз объём записи в файлы данных превосходит объём записи в WAL?::Обычно в ==5–7 раз== из-за страничной структуры файлов данных против компактной записи в WAL.

Алгоритм B-Tree page split
?
1. При вставке в заполненную страницу индекса происходит её разделение на две.
2. Примерно половина ключей переносится в новую страницу.
3. Обновляется родительская страница (информация о границах ключей).
4. Лавинообразная запись нескольких страниц из-за изменения одного ключа.

Ключевые контексты `pg_stat_io` (context)
?
- `normal` — запись через shared buffers
- `vacuum` — запись вне буферов (autovacuum)
- `bulkwrite` — операции типа COPY

Объекты `pg_stat_io` (object)::`relation` (таблицы/индексы) или `wal`.

Как LSM-движок Aerospike снижает износ SSD?::Пишет блоками фиксированного размера (wblocks) в обход файловой системы.

Поведение WA в аналитических СУБД (ClickHouse, Vertica)::Ниже при записи (данные пишутся огромными блоками, сжимаются, редко обновляются), но резко возрастает при слиянии кусков (merge).

Почему ALTER TABLE вызывает даунтайм?::Требует ACCESS EXCLUSIVE lock → ждёт завершения всех транзакций → все новые SELECT встают в очередь за DDL → лавинообразный простой
