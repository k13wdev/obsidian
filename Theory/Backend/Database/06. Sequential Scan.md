## Суть и контекст

Sequential Scan (Seq Scan / полный просмотр) — метод физического доступа в PostgreSQL, при котором сервер последовательно считывает все страницы таблицы с диска. Планировщик выбирает Seq Scan, когда стоимость последовательного чтения ниже стоимости random I/O через индекс.

---

## Ключевые идеи

- Seq Scan считывает страницы данных **многостраничными блоками** (multi-block read) напрямую с диска, минуя индексы.
- Стоимость Seq Scan **фиксирована** — не зависит от количества выбираемых строк (N).
- Стоимость Index Scan **линейно растёт** с N. Точка пересечения — обычно 5–15% строк таблицы.
- Для маленьких таблиц (1–2 страницы) Seq Scan всегда дешевле индекса.
- PostgreSQL выделяет **ring buffer** для Seq Scan, чтобы не вымывать полезные данные из shared_buffers.

---

## Определения и термины

**Sequential Scan (Seq Scan / полный просмотр)** — метод физического доступа, при котором сервер последовательно считывает содержимое таблицы страницу за страницей с диска, перебирая все строки для проверки соответствия условиям фильтрации.

**Ring Buffer (Буферное кольцо)** — ограниченная область буферного кеша (256 Кбайт = 32 страницы) для больших таблиц. Предотвращает вытеснение «горячих» данных из shared_buffers при массовом сканировании. Стратегия `bulkread` активируется для таблиц >1/4 от shared_buffers.

**Synchronized Scans** — механизм объединения нескольких параллельных Seq Scan одной таблицы в одно буферное кольцо: последующий процесс начинает чтение с текущей позиции первого, затем дочитывает начало таблицы (wrap-around).

**TID Scan** — сканирование конкретных версий строк по физическому адресу (ctid). Используется при `WHERE ctid = '(0,1)'`.

**Sample Scan** — сканирование случайной выборки строк (команда `TABLESAMPLE`). Применяется для сбора статистики или аналитических запросов.

**Foreign Scan** — метод доступа к данным во внешних системах через Foreign Data Wrappers (FDW). Логика сканирования определяется обработчиком конкретного FDW.

---

## Как устроено

### Алгоритм

1. Открыть файл основного слоя (main fork) таблицы.
2. Страницы последовательно считываются в буферный кеш.
3. Для каждой страницы — итерация по указателям версий строк (line pointers).
4. **Visibility Check (MVCC)**: для каждой версии строки проверяются поля `xmin` / `xmax` в заголовке. Определяется, видима ли строка в текущем снимке транзакции.
5. Применяется **предикат Filter** — условие WHERE. Несовпадающие строки отбрасываются.
6. Совпадающие строки передаются вышестоящему узлу плана.

**Ring Buffer**: для таблиц, превышающих 1/4 от `shared_buffers`, используется стратегия **bulkread** — выделяется кольцо 256 Кбайт (32 страницы). Новые страницы читаются в кольцо, вытесняя старые страницы того же сканирования. «Горячие» данные в кеше не вымываются.

**Edge case**: грязные страницы в буферном кольце не записываются на диск принудительно — грязный буфер просто исключается из кольца и заменяется новым.

### Формула стоимости

```
cost = (relpages × seq_page_cost) + (reltuples × cpu_tuple_cost)
```

При наличии Filter добавляется `reltuples × cpu_operator_cost` за проверку условия.

### Когда планировщик выбирает Seq Scan

- Выборка **более 5–15%** строк таблицы — Index Scan дороже из-за random I/O.
- Таблица очень маленькая (**1–2 страницы**) — полное чтение дешевле индексного доступа.
- На таблице **нет подходящего индекса**.

---

## Детали и нюансы

- **synchronize_seqscans**: при параллельном Seq Scan новое сканирование присоединяется к уже идущему (wrap-around), экономя I/O. Порядок строк непредсказуем без ORDER BY.
- **enable_seqscan = off**: увеличивает штраф Seq Scan до бесконечности, заставляя планировщик выбирать индексные пути. Если индексов нет — Seq Scan всё равно используется, но с пометкой `Disabled: true`.
- **Parallel Sequential Scan**: ведущий процесс (leader) запрашивает рабочие процессы (workers). Таблица делится на блоки-чанки; процессы разбирают их из общей очереди в разделяемой памяти. Каждый worker получает диапазон страниц (а не по одной) для нормальной работы prefetching ОС.
- **Bloat и Seq Scan**: при отсутствии VACUUM таблица раздувается (dead tuples). Seq Scan обязан прочитать все страницы, включая страницы только с мёртвыми строками — время выполнения растёт линейно с физическим размером.

### Другие методы сканирования таблиц

| Метод | Описание | Когда |
|-------|---------|-------|
| TID Scan | Чтение по физическому адресу (ctid) | `WHERE ctid = '(0,1)'` |
| Sample Scan | Случайная выборка строк | `TABLESAMPLE SYSTEM(10)` |
| Foreign Scan | Доступ к внешним данным через FDW | Foreign Tables (postgres_fdw, file_fdw) |

## Примеры

```sql
-- План с Seq Scan
EXPLAIN SELECT * FROM tenk1 WHERE unique1 < 7000;
-- Seq Scan on tenk1  (cost=0.00..470.00 rows=7000 width=244)
--   Filter: (unique1 < 7000)

-- Отключение Seq Scan для принудительного использования индекса
SET enable_seqscan = off;
```

## Важно / подводные камни / best practices

- **Seq Scan на маленькой таблице — это нормально**. Не пытайтесь форсировать индекс для таблиц в 1–2 страницы.
- **enable_seqscan = off — диагностический инструмент**, не production-настройка.
- **Без ORDER BY порядок строк при Seq Scan не гарантирован**, особенно при synchronize_seqscans.
