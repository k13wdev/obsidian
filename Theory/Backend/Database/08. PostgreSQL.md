## Суть и контекст

PostgreSQL — объектно-реляционная СУБД с открытым исходным кодом. Конспект охватывает архитектуру экземпляра PostgreSQL: многопроцессную модель, разделяемую память, буферный менеджер, фоновые процессы и организацию хранения данных (heap-таблицы).

---

## Ключевые идеи

- PostgreSQL использует **многопроцессную архитектуру**: каждый пользовательский сеанс обслуживается отдельным процессом ОС (backend process).
- Общая память реализована как **сегмент разделяемой памяти ОС (shared memory)**, к которому обращаются все процессы.
- Данные таблиц лежат в **heap-файлах** (неупорядоченных). Индексы содержат ctid — физический идентификатор строки в heap.
- Буферный кеш (shared_buffers) — область в разделяемой памяти, разбитая на буферы размером со страницу БД, для кэширования дисковых страниц.
- Транзакции записывают изменения в **WAL buffer** (кольцевой буфер в памяти), который асинхронно сбрасывается на диск, переводя random I/O в sequential I/O.

---

## Определения и термины

**Экземпляр СУБД (Instance)** — набор ресурсов операционной системы, выделенных для работы с общими данными в памяти и на диске. Состоит из процессов и области общей памяти.

**Буферный кеш (Buffer cache / shared_buffers)** — область в разделяемой памяти экземпляра, разбитая на буферы размером со страницу базы данных, используемая для хранения копий страниц, прочитанных с диска, и буферизации записи изменённых ("грязных") страниц.

**Неупорядоченная таблица (Heap-organized table / Heap)** — изменяемая страничная структура данных, в которой новые строки записываются либо в конец файла, либо в страницы со свободным пространством без поддержания какого-либо порядка сортировки.

## Как устроено

### История

PostgreSQL основан на проекте POSTGRES версии 4.2, разработанном в Калифорнийском университете в Беркли. В 1995 году появился **Postgres95** с акцентом на стандартный SQL. Впоследствии переименован в PostgreSQL для отражения поддержки стандарта SQL. Развивается сообществом добровольцев и Postgres Global Development Group.

### Компоненты экземпляра

- **Процесс-почтмейстер (postmaster / postgres)** — главный процесс, слушает соединения; порождает backend-процессы через `fork()`.
- **Обслуживающие процессы (backend processes)** — по одному на каждый пользовательский сеанс.
- **Фоновые служебные процессы**:
  - `startup` — восстановление после сбоев
  - `checkpointer` — контрольные точки
  - `writer` (bgwriter) — асинхронный сброс грязных страниц
  - `wal writer` — запись WAL-буферов на диск
  - `autovacuum launcher / worker` — очистка мёртвых версий строк
  - `wal sender / receiver` — передача журналов для репликации
- **Разделяемая память (shared memory)** — общий сегмент памяти ОС для всех процессов. Включает: буферный кеш, WAL-буферы, таблицы блокировок, статистику.

### Буферный менеджер

1. Процесс ищет страницу в shared_buffers.
2. Если не найдена — находит свободный буфер, считывает данные с диска.
3. Накладывает кратковременную блокировку (spin-lock / latch) для защиты от конкурентных изменений.
4. Модифицирует данные, помечает страницу как "грязную" (dirty).
5. Снимает блокировку.
6. Фоновые процессы bgwriter и checkpointer асинхронно сбрасывают грязные страницы на диск.

### Хранение данных

- Таблицы — heap-файлы (неупорядоченные страницы).
- B-Tree индексы содержат ключи и **ctid** (номер страницы + номер кортежа) — прямой физический указатель на строку в heap.

### Роли и привилегии

**Ролевая модель**: PostgreSQL не разделяет «пользователь» и «группа» — только роль. Атрибуты: `LOGIN`, `SUPERUSER`, `CREATEDB`, `BYPASSRLS`.

**Предопределённые роли**:
- `pg_read_all_data` — чтение всех данных
- `pg_maintain` — право на VACUUM/ANALYZE
- `pg_monitor` — мониторинг

**Управление доступом**: `GRANT <привилегия> ON <объект> TO <роль>`, `REVOKE`, `WITH GRANT OPTION`. RLS (Row-Level Security) — политики `CREATE POLICY`.

### Конфигурация

- `postgresql.conf` — основные параметры (память, WAL, autovacuum).
- `pg_hba.conf` — host-based authentication: сеть + метод (trust, md5, gssapi).
- `pg_ident.conf` — сопоставление пользователей ОС с ролями БД.

### Утилиты

| Утилита | Назначение |
|---------|-----------|
| `psql` | Интерактивный терминал (`\d`, `\dt`, `\h`, `\q`) |
| `pg_dump` | Логический бэкап одной базы |
| `pg_restore` | Восстановление из архивов pg_dump |
| `pg_dumpall` | Бэкап всего кластера (включая роли, tablespaces) |

| `reindexdb` | CLI-обёртка для REINDEX |
| `initdb` | Инициализация нового кластера в PGDATA |

### Расширения и FDW

**Extensions**: `pageinspect` (анализ страниц), `pg_buffercache` (анализ кеша), `pg_stat_statements` (статистика запросов).

**FDW**: `file_fdw` (CSV), `postgres_fdw` (другой PG-сервер).

### Материализованные представления

```sql
-- Обновление данных
REFRESH MATERIALIZED VIEW my_view;

-- Обновление без блокировки чтения (требует UNIQUE индекс)
REFRESH MATERIALIZED VIEW CONCURRENTLY my_view;
```

## Детали и нюансы

- **Процесс-на-соединение**: каждое подключение порождает отдельный backend-процесс ОС; это определяет необходимость pooling'а на стороне приложения при большом числе клиентов.
- Обращение к страницам через shared_buffers защищается короткими spin-lock / latch — в отличие от долговременных блокировок уровня строк/таблиц.
- Детали Sequential Scan, включая ring buffer и `synchronize_seqscans`, см. в [[06. Sequential Scan]].

## Сравнения и trade-offs

| Характеристика | PostgreSQL | InnoDB (MySQL) |
|---------------|-----------|----------------|
| Модель | Многопроцессная (process per connection) | Многопоточная (threads) |
| Хранение таблиц | Heap (неупорядоченные страницы) | Clustered Index (B-Tree с данными в листьях) |
| Указатель в индексе | ctid (физический адрес в heap) | Значение первичного ключа |
| Обновление PK | Дёшево (обновляется ctid) | Дорого (физическое перемещение строки + каскадное обновление ВСЕХ вторичных индексов) |

## Важно / подводные камни / best practices

- **Регулярно выполняйте ANALYZE** — без актуальной статистики оптимизатор считает распределение равномерным и выбирает катастрофически медленные планы.
- **Не создавайте индексы на все поля "на всякий случай"** — ускоряет чтение, но экспоненциально замедляет INSERT/UPDATE/DELETE из-за перестроения B-Tree.
- **Используйте параметризованные запросы** (prepared statements), а не динамическую склейку строк — забивает plan cache и заставляет СУБД заново разбирать каждый запрос.
