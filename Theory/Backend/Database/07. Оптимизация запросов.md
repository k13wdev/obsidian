## Суть и контекст

Оптимизация запросов — процесс преобразования декларативного SQL-запроса в эффективный план физического выполнения. Оптимизатор (Query Optimizer) использует стоимостную модель (cost-based optimization), статистику распределения данных и алгоритмы перебора планов для выбора оптимального пути доступа.

---

## Ключевые идеи

- Оптимизатор преобразует **декларативное дерево логических операций** в эквивалентное **дерево физических операций** (план выполнения) на основе оценки стоимости.
- Стоимость — количественная оценка затрат в произвольных единицах; базовая единица (1.0) — стоимость последовательного чтения одной дисковой страницы.
- **Стоимость узла** = оценочное количество операций × константа стоимости одной операции.
- Без актуальной статистики оптимизатор считает распределение равномерным и выбирает катастрофически медленные планы.
- Переписывание запросов (Query Rewrite) работает как система макросов: views разворачиваются до начала cost-based оптимизации.

---

## Определения и термины

**Оптимизатор запросов (Query Optimizer / Planner)** — компонент СУБД, преобразующий декларативное дерево логических операций запроса в эквивалентное дерево физических операций (план выполнения) на основе оценки стоимости.

**Стоимость (Cost)** — количественная оценка затрат в произвольных единицах; базовая единица (1.0) — стоимость последовательного чтения одной дисковой страницы. Стоимость `seq_page_cost = 1.0`, `random_page_cost = 4.0` (по умолчанию; для SSD рекомендуется снижать до 1.1–2.0).

**Статистика (Statistics)** — информация о профиле данных: `reltuples`, `relpages` в `pg_class`; `null_frac`, `n_distinct`, `most_common_vals`, `most_common_freqs`, `histogram_bounds` в `pg_statistic` (доступ через `pg_stats`). Обновляется командой ANALYZE.

**Гистограмма (Histogram)** — массив `histogram_bounds` в `pg_statistic`, разбивающий данные на «корзины» с примерно равным количеством строк. Используется для оценки селективности range-запросов.

**Selectivity Estimation** — для равенства `col = const` берётся из `most_common_freqs`; для диапазонов `col > const` — интерполяция по `histogram_bounds`.

**GEQO (Genetic Query Optimizer)** — генетический алгоритм планирования, включается при числе таблиц в FROM > `geqo_threshold` (по умолчанию 12). Вместо точного перебора используется вероятностный алгоритм.

**Generic Plan / Custom Plan** — подготовленные запросы (PREPARE) первые 5 раз планируются с учётом фактических параметров (custom plan). На 6-й раз планировщик сравнивает среднюю стоимость и может переключиться на обобщённый план (generic plan).

## Как устроено

### Query Processing Pipeline

1. **Parser (Разбор)** — лексический и синтаксический анализ текста SQL, построение абстрактного синтаксического дерева (AST).
2. **Rewrite (Переписывание)** — замена имён представлений (views) их определениями, применение правил, внедрение фильтров RLS (Row-Level Security).
3. **Planner / Optimizer** — перебор вариантов соединений и методов доступа; расчёт стоимости на основе статистики. Использует динамическое программирование. При >8 таблиц ограничивается `join_collapse_limit`, при >12 — переключается на GEQO.
4. **Executor (Исполнение)** — работает по принципу конвейера (pull-model): узлы запрашивают строки у дочерних узлов.

### Алгоритмы JOIN

| Алгоритм | Механика | Когда используется |
|----------|---------|-------------------|
| Nested Loop | Для каждой строки внешнего набора — поиск во внутреннем | Малые наборы; есть индекс на внутреннем наборе |
| Hash Join | Построить хеш-таблицу внутреннего набора в `work_mem`; сканировать внешний | Большие наборы без сортировки; нет индексов |
| Merge Join | Отсортировать оба набора по ключу; синхронное слияние | Данные уже отсортированы (индекс); нужен отсортированный результат |

Hash Join при нехватке `work_mem` использует несколько **batches** (пакетов на диске).

### Алгоритмы агрегации

- **HashAggregate**: строит хеш-таблицу уникальных ключей в `work_mem`. Применяется, если все группы помещаются в память.
- **GroupAggregate**: агрегация предварительно отсортированного набора за один проход. Используется если данные уже отсортированы или набор слишком велик для хеширования.

### Формула стоимости Seq Scan

```
cost = (relpages × seq_page_cost) + (reltuples × cpu_tuple_cost)
```

Пример: таблица tenk1 (relpages=345, reltuples=10000) → cost = 345 × 1.0 + 10000 × 0.01 = 445.00.

### EXPLAIN vs EXPLAIN ANALYZE

**EXPLAIN** показывает **план выполнения** запроса без его выполнения. Для каждого узла выводятся:
- **Оценочная стоимость** (в условных единицах — обычно стоимость чтения одной дисковой страницы).
- **Ожидаемое количество строк** (rows).
- **Средняя ширина строки** в байтах (width).
- Стоимость узла верхнего уровня включает стоимость всех дочерних. Не учитывает время передачи результатов клиенту по сети.

**EXPLAIN ANALYZE** **реально выполняет** запрос и выводит фактические метрики:
- **Actual time** — фактическое время выполнения каждого узла (в мс).
- **Actual rows** — реальное количество строк.
- **Loops** — сколько раз узел был выполнен (для Nested Loop).
- Методы сортировки (в памяти или на диске), использование памяти хеш-таблицами.
- Количество поисков по индексу, число строк, отброшенных фильтром (Rows Removed by Filter).

**Ключевая метрика**: сравнение ожидаемого и фактического количества строк — сильное расхождение указывает на устаревшую статистику.

**Важно**: EXPLAIN ANALYZE добавляет накладные расходы на профилирование. Для анализа деструктивных запросов (UPDATE/DELETE) без изменения данных:

```sql
BEGIN;
EXPLAIN ANALYZE UPDATE ...;
ROLLBACK;
```

## Детали и нюансы

- **Граница переключения Seq Scan → Index Scan**: стоимость Seq Scan фиксирована (не зависит от N), стоимость Index Scan растёт линейно. Точка пересечения — обычно **5–15% строк таблицы**. При выборке большей доли оптимизатор переключается на Seq Scan.
- **Маленькие таблицы (1–2 страницы)**: планировщик игнорирует индекс и выбирает Seq Scan.
- **join_collapse_limit**: ограничивает количество таблиц на уровне дерева соединений при переборе порядка JOIN. По умолчанию 8. При превышении порядок явных JOIN фиксируется.
- **from_collapse_limit**: аналогично для подзапросов; контролирует «схлопывание» подзапросов во внешний запрос.
- **GEQO**: при числе таблиц > `geqo_threshold` (12 по умолчанию) — генетический алгоритм вместо точного перебора.
- **Ошибки оценки (Correlation)**: если две колонки зависимы (город и аэропорт), оценка селективности занижается (перемножение вероятностей). Лечение: `CREATE STATISTICS ... (dependencies) ON col1, col2 FROM table`.
- **Generic vs Custom Plans**: для prepared statements первые 5 выполнений используют custom plan (с параметрами). На 6-й — сравнивается средняя стоимость, может переключиться на generic plan.
- **Материализация CTE**: до PostgreSQL 12 `WITH`-выражения всегда материализовались (вычислялись один раз полностью). С версии 12 — только если CTE используется более одного раза или указано `MATERIALIZED`.

## Примеры

```sql
-- План без выполнения
EXPLAIN SELECT * FROM tenk1 WHERE unique1 < 7000;

-- Выполнить + фактические метрики
EXPLAIN ANALYZE SELECT * FROM flights;

-- + статистика буферного кеша (hits, reads, dirtied, written)
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM flights;

-- Сбор статистики
ANALYZE flights;

-- Увеличить детализацию гистограмм для столбца
ALTER TABLE flights ALTER COLUMN status SET STATISTICS 500;

-- Управление планировщиком
SET enable_seqscan = off;
SET enable_hashjoin = off;
SET work_mem = '64MB';

-- Расширенная статистика для зависимых колонок
CREATE STATISTICS flights_deps (dependencies) ON departure_airport, city FROM flights;

-- Просмотр статистики столбца
SELECT * FROM pg_stats WHERE tablename = 'flights' AND attname = 'status';

-- Запрос базовых характеристик таблицы
SELECT relpages, reltuples FROM pg_class WHERE relname = 'tenk1';
```

## Важно / подводные камни / best practices

- **Регулярно выполняйте ANALYZE** — без гистограмм и MCV оптимизатор занижает/завышает кардинальность и выбирает катастрофически медленные планы (Nested Loop вместо Hash Join).
- **Используйте prepared statements** — динамическая склейка SQL забивает plan cache и заставляет СУБД заново разбирать каждый запрос.
- **Не создавайте индексы на все поля** — ускоряет SELECT, но экспоненциально замедляет INSERT/UPDATE/DELETE.
- **Переписывание (Query Rewrite) бесплатно** — views разворачиваются до начала оптимизации, не создавая накладных расходов на этапе выполнения.
