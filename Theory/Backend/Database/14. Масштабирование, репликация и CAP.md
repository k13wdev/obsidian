## Суть и контекст

Конспект охватывает стратегии масштабирования баз данных за пределы одного сервера:

1. **Денормализация** — осознанное снижение нормальной формы ради производительности выборки.
2. **Шардирование** — горизонтальное и вертикальное разделение данных между серверами.
3. **Репликация** — master-slave, multi-leader, синхронная и асинхронная.
4. **CAP theorem** — фундаментальное ограничение распределённых систем и классификация PACELC.
5. **Leader election** — алгоритмы Raft и ZAB для автоматического выбора лидера.

---

## Ключевые идеи

- Денормализация — замена нескольких нормализованных таблиц их соединением для сокращения JOIN при выборке; применяется **исключительно** на физическом уровне как "последнее средство".
- Секционирование (partitioning) — разделение данных внутри одного экземпляра СУБД. Шардирование (sharding) — между несколькими независимыми серверами.
- Стратегии шардирования: hash-based (равномерное распределение, но невозможны range-запросы) vs range-based (локальность диапазонов, но hot spots).
- Виртуальные бакеты (virtual buckets) решают проблему решардинга: при добавлении сервера переносятся целые бакеты, а не пересчитываются хеши строк.
- Синхронная репликация: мастер ждёт подтверждения от реплик до возврата клиенту. Асинхронная: допускает потерю транзакций при отказе.
- CAP theorem: распределённая система не может одновременно обеспечить Consistency + Availability + Partition tolerance — максимум два из трёх.
- PACELC расширяет CAP: в штатном режиме (Else) компромисс между Latency и Consistency, а не только при разделении сети.
- Алгоритм Raft: leader election через голосование с эпохами (terms). ZAB (Zookeeper): обязательная фаза синхронизации после выбора лидера.

## Определения и термины

**Вертикальное масштабирование (Scale Up)** — увеличение производительности базы данных за счёт наращивания мощности одного сервера (CPU, RAM, Disk). Ограничено физическим пределом железа.

**Горизонтальное масштабирование (Scale Out)** — увеличение производительности путём добавления новых узлов (серверов) в кластер. Требует шардирования или репликации.

**RPO (Recovery Point Objective)** — максимальный допустимый объём потерянных данных при сбое (например, «не более 5 минут изменений»). При асинхронной репликации RPO > 0.

**RTO (Recovery Time Objective)** — целевое время восстановления системы до работоспособного состояния (например, «переключение на реплику ≤ 30 секунд»).

**Денормализация** — замена множества переменных отношения их соединением, цель — увеличение избыточности данных за счёт приведения к более низкому уровню нормализации для сокращения количества JOIN при выборке.

**Секционирование (Partitioning)** — физическое разделение данных на части (секции) с разными параметрами хранения внутри одного экземпляра СУБД. Логически таблица остаётся единой.

**Виды секционирования:**
- **Range** — данные распределяются по непересекающимся диапазонам значений ключа (например, по датам).
- **List** — каждая секция явно определяет список значений ключа (например, филиалы 'Москва', 'Омск').
- **Hash** — секция выбирается по остатку от деления хеша ключа на модуль.
- **Composite** — секции делятся на подсекции (Range по датам, внутри List по филиалам).
- **Reference** — дочерняя таблица секционируется зеркально родительской (по внешнему ключу).

**Шардирование (Sharding)** — разделение одной базы данных на несколько независимых баз (шардов), хранящих разные строки/фрагменты таблиц, для распределения нагрузки между узлами.

**Горизонтальное шардирование** — таблица делится **по строкам**: например, заказы с нечётными ID → Шард 1, с чётными → Шард 2.

**Вертикальное шардирование** — таблица делится **по столбцам**: тяжёлые поля (BLOB, медиа) выносятся в отдельную таблицу/шард, метаданные остаются в основной.

**Multi-Master репликация** — несколько узлов могут принимать записи одновременно. Требует сложных алгоритмов разрешения конфликтов; в open-source Redis — антипаттерн (бесконечный цикл).

**Ключ шардирования** — колонка (обычно PK) или набор колонок, на основании значения которой строка направляется в конкретный шард.

**Виртуальные бакеты (Virtual buckets)** — заранее созданное избыточное количество логических шардов на меньшем числе физических серверов для прозрачного переноса данных при добавлении узлов.

**Синхронная репликация (2-safe)** — транзакция не считается зафиксированной, пока все (или заданное число) серверы не подтвердят получение или применение изменений.

**Асинхронная репликация** — допускает задержку между COMMIT на мастере и распространением на реплики. Возможна потеря транзакций при отказе.

**Физическая репликация** — поблочное копирование «сырой» памяти (байтов) с диска на диск посредством передачи WAL. Реплика — побайтовая копия мастера, доступна строго только для чтения (hot standby).

**Логическая репликация** — репликация объектов на основе их идентичности (первичного ключа), при которой передаются логические изменения данных (INSERT/UPDATE/DELETE). Реплика может оставаться доступной для записи (Read-Write), строить свои индексы, реплицировать между разными мажорными версиями СУБД.

**Replication lag** — разница во времени или объёме WAL-данных между primary и standby.

**CAP-теорема (Brewer's theorem)** — распределённая система может обладать одновременно не более чем двумя свойствами из трёх: Consistency, Availability, Partition tolerance.

**Consistency (CAP)** — наличие глобального порядка операций и иллюзия одновременного отражения на всех узлах (линеаризуемость). Любое чтение возвращает результат последнего изменения.

**Availability (CAP)** — на каждый запрос работающему узлу должен быть получен ответ с конечным временем ожидания (содержимое не регламентируется).

**Partition tolerance** — система продолжает работать при разрыве сети между частями кластера.

**Eventual consistency** — данные постоянно реплицируются, и если остановить приложение, рано или поздно все узлы станут одинаковыми.

**Leader election** — процедура в алгоритмах распределённого консенсуса (Raft, ZAB), в ходе которой узлы автоматически выбирают новый источник изменений без вмешательства извне.

## Как устроено

### Шардирование — стратегии

**Hash-based**:
1. Извлекается значение ключа шардирования (ID).
2. Вычисляется хеш-функция от ключа.
3. Результат делится по модулю на количество шардов.
4. Строка помещается в шард с номером = остаток от деления.

**Range-based**:
1. Задаются границы значений ключа для каждого фрагмента.
2. Строки распределяются по попаданию ключа в диапазон.
3. При достижении максимального размера фрагмент делится на два.

**Решардинг без виртуальных бакетов**: система останавливается, пересчитываются хеши всех записей, миллионы строк копируются на новые шарды.

**Решардинг с виртуальными бакетами**: на сервере 4 логических бакета. При добавлении сервера 2 бакета целиком переносятся на новый узел, хеши строк не пересчитываются.

### Репликация — Master-Slave (Streaming Replication в PostgreSQL)

1. Standby подключается к Primary через replication connection.
2. Primary стримит WAL-записи на standby по мере генерации.
3. Standby применяет WAL к своей базе данных.

**Синхронная репликация (PostgreSQL)**:
- `remote_write`: мастер ждёт записи WAL в кеш ОС реплики.
- `on`: мастер ждёт flush WAL на диск реплики.
- `remote_apply`: мастер ждёт применения WAL (данные видимы читателям).

**Кворумная vs приоритетная** синхронизация:
- `ANY 2 (s1, s2, s3)` — кворумная, ждёт любые 2 из 3.
- `FIRST 3 (s1, s2, s3, s4)` — приоритетная, ждёт первые 3 по списку.

### Логическая репликация (PostgreSQL)

1. На мастере создаётся **публикация (Publication)** для определённых таблиц.
2. Логический процесс декодирует WAL в поток логических изменений (INSERT/UPDATE/DELETE строк).
3. **Подписчик (Subscription)** на реплике принимает поток и применяет логические команды.
4. Допускается фильтрация строк (Row Filters) и колонок (Column Lists).

**Ограничения**: логическая репликация **не реплицирует DDL** (CREATE/ALTER/DROP TABLE) и **последовательности** (sequences). При изменении схемы на мастере без синхронизации с подписчиком — ошибка.

### Секционирование — архивация данных

Удалить физический файл секции (`DROP TABLE`) быстрее, чем массивный DELETE с затратами на WAL и VACUUM. Секции за старые периоды можно мгновенно отключать или архивировать.

### Leader Election — алгоритм Raft

1. Реплика не получает heartbeat от лидера → увеличивает номер эпохи, переходит в Кандидат.
2. Рассылает RequestVote.
3. Узел голосует "за", если: его эпоха ≤ эпохи кандидата, он не голосовал в этой эпохе, lastLogIndex кандидата ≥ его собственного.
4. Кандидат с большинством голосов → Лидер → heartbeat.
5. Если голоса поровну → случайная пауза → перезапуск.

### Leader Election — алгоритм ZAB (Zookeeper)

1. Выбирается потенциальный лидер с максимально длинной историей.
2. Лидер собирает номера эпох, объявляет новую эпоху.
3. **Фаза синхронизации**: лидер выравнивает реплики (DIFF, SNAP или TRUNC).
4. После подтверждения от большинства → трансляция новых изменений.

### CAP theorem — доказательство (Гилберт и Линч, 2002)

1. Разрыв сети (P) — база распадается на изолированные части.
2. На Master поступает запись, не доходит до Slave.
3. Клиент запрашивает данные у Slave.
4. Выбор: отдать устаревшие данные (A, но не C) **или** отказать (C, но не A).

## Детали и нюансы

- **Hash-sharding и range-запросы**: СУБД вынуждена сканировать все шарды кластера — hash-стратегия не позволяет локализовать range.
- **Range-sharding и hot spots**: неравномерное распределение ключей перегружает отдельные фрагменты.
- **JOIN между шардами**: невыполнимы прямым SQL, требуют мультишардовых алгоритмов.
- **Транзакции между шардами**: не работают стандартными средствами.
- **Split-brain**: при потере связи резервный ЦОД может попытаться стать мастером. Решение: STONITH или нечётное число узлов (кворумный арбитр).
- **Инфляция эпохи в Raft**: узел с нестабильной сетью вызывает "мерцание" лидера. Решение: pre-vote без увеличения эпохи.
- **Отказ единственной синхронной реплики**: все write-транзакции на мастере зависают до бесконечности. Нужно уменьшить synchronous_standby_names и reload.
- **Денормализация и AVG**: дублирующиеся строки искажают агрегаты. Требуется явная фильтрация перед агрегацией.
- **Вставка без подходящей секции**: если значение ключа не попадает ни в один диапазон/список — ошибка. Требуется секция DEFAULT или автоматическое создание.
- **Потеря данных при failover на асинхронную реплику**: транзакции, успевшие закоммититься на мастере, но не переданные по сети — безвозвратно теряются (data loss пропорционален replication lag).
- **Логическая репликация и DDL**: DDL не реплицируется. Изменение схемы на мастере без обновления подписчика → ошибка. Последовательности (sequences) на реплике показывают старые значения при failover.
- **Логическая репликация требует Primary Key**: без PK UPDATE/DELETE не могут идентифицировать строку — логическая репликация таблицы без PK невозможна.

## Сравнения и trade-offs

| Стратегия шардирования | Плюсы | Минусы | Когда использовать |
|----------------------|-------|--------|--------------------|
| Hash-based | Равномерное распределение | Range-запросы → full scan всех шардов | Точечные запросы по ключу |
| Range-based | Локальность диапазонов | Hot spots при перекосе ключей | Временные ряды, диапазонные запросы |

| Репликация | Плюсы | Минусы | Когда использовать |
|-----------|-------|--------|--------------------|
| Синхронная | Нет потери данных | Latency = round-trip до реплики; зависание при падении реплики | Финансовые системы, критичные данные |
| Асинхронная | Минимальная latency | Потеря транзакций при отказе мастера | Высокий throughput, допустима потеря |

| Тип репликации | Плюсы | Минусы | Когда использовать |
|---------------|-------|--------|--------------------|
| Физическая (WAL) | Побайтовая копия, простая настройка | Реплика read-only, одна мажорная версия | Hot standby, HA failover |
| Логическая (Pub/Sub) | Реплика read-write, разные версии, фильтрация | DDL не реплицируется, sequences не синхронизируются | Выборочная репликация, агрегаты на реплике, миграция |

| Класс системы (CAP) | Поведение при partition | Примеры |
|---------------------|----------------------|---------|
| CP | Консистентность, часть узлов не отвечает | HBase, VoltDB |
| AP | Доступность, возможны устаревшие данные | DynamoDB, Cassandra, Riak KV |
| CA | Нет partition (один узел / shared disk) | PostgreSQL single-node, Oracle RAC |

| PACELC | При partition | В штатном режиме | Примеры |
|--------|--------------|-----------------|---------|
| PA/EL | Availability | Low Latency | DynamoDB, Cassandra |
| PA/EC | Availability | Consistency | MongoDB |
| PC/EC | Consistency | Consistency | HBase, VoltDB |
| PC/EL | Consistency | Low Latency | PNUTS (Yahoo) |

```python
# Расчёт номера шарда (hash-based)
shard_number = id % 1000
```

```sql
-- Синхронная репликация (приоритетная)
synchronous_standby_names = 'FIRST 3 (s1, s2, s3, s4)'

-- Синхронная репликация (кворумная)
synchronous_standby_names = 'ANY 2 (s1, s2, s3)'

-- Подключение standby к primary
primary_conninfo = 'host=192.168.1.50 port=5432 user=foo password=foopass'

-- Создание секции по диапазону дат (Range Partitioning)
CREATE TABLE measurement_y2007m12 PARTITION OF measurement
    FOR VALUES FROM ('2007-12-01') TO ('2008-01-01')
    TABLESPACE fasttablespace;

-- Настройка логической репликации
-- На мастере (pub):
CREATE PUBLICATION mypub FOR TABLE users, departments;
-- На реплике (sub):
CREATE SUBSCRIPTION mysub
    CONNECTION 'dbname=foo host=bar user=repuser'
    PUBLICATION mypub;

-- Ослабление консистентности в Cassandra
CONSISTENCY ONE
```

## Важно / подводные камни / best practices

- Денормализацию применять **исключительно** на физическом уровне, оставляя логическую структуру в максимальной НФ (5НФ). Только как "последнее средство" после исчерпания всех иных методов оптимизации.
- Антипаттерн: семантический атрибут (категория товара) как ключ шардирования → перекос данных, перегрузка одного шарда.
- Для High Availability указывать несколько кандидатов в синхронные реплики (`ANY n` или `FIRST n`) — падение одной не парализует мастер.
- Антипаттерн: `synchronous_commit = local` при синхронной репликации — мастер не ждёт реплику, гарантии 2-safe не выполняются.
- Использовать PACELC вместо CAP при проектировании: в штатном режиме важнее компромисс Latency vs Consistency.
- Consistency в CAP ≠ Consistency в ACID — это разные понятия (линеаризуемость реплик vs согласованность ограничений на одном узле).

Consistency в CAP:::Линеаризуемость — любое чтение возвращает результат последнего изменения данных. Глобальный порядок операций.

Availability в CAP:::Каждый запрос к работающему узлу получает ответ с конечным временем ожидания.

Eventual consistency:::Если остановить приложение, рано или поздно данные на всех узлах станут одинаковыми. Допускает временную несогласованность.

Доказательство CAP (Гилберт и Линч, 2002)
?
1. Разрыв сети — база на две части
2. Master получает запись, Slave не получает
3. Клиент запрашивает Slave
4. Выбор: устаревшие данные (AP) или отказ (CP)

Что такое PACELC?::Расширение CAP: если Partition → A или C; иначе (Else) → Latency или Consistency. Описывает компромисс в штатном режиме.

DynamoDB и Cassandra по классификации PACELC: ==PA/EL==

MongoDB по классификации PACELC: ==PA/EC==

HBase и VoltDB по классификации PACELC: ==PC/EC==

Алгоритм Raft — выбор лидера
?
1. Реплика не получает heartbeat → увеличивает эпоху, становится Кандидатом
2. Рассылает RequestVote
3. Узел голосует "за" если эпоха ≤ кандидата, не голосовал, lastLogIndex кандидата ≥ своего
4. Большинство → Лидер → heartbeat
5. Ничья → случайная пауза → перезапуск

В чём отличие Raft от ZAB?
?
Raft: лидер выбирается сразу, изменения накатываются после кворума
ZAB: после выбора лидера обязательная фаза синхронизации (DIFF/SNAP/TRUNC), затем новая эпоха

Что такое STONITH?::Shoot The Other Node In The Head — принудительное отключение старого мастера при split-brain.

Виды секционирования (Partitioning)
?
1. Range — по непересекающимся диапазонам значений ключа
2. List — по явному списку значений
3. Hash — по остатку от деления хеша ключа
4. Composite — секции делятся на подсекции
5. Reference — дочерняя таблица зеркально родительской

Физическая репликация:::Поблочное копирование WAL-байтов. Реплика — побайтовая копия мастера, строго read-only.

Логическая репликация:::Передача логических изменений (INSERT/UPDATE/DELETE) через Publication/Subscription. Реплика read-write, поддерживает фильтрацию и разные версии СУБД.

В чём ключевое отличие физической от логической репликации?
?
Физическая: побайтовая копия, реплика read-only, одна мажорная версия.
Логическая: передаёт SQL-события, реплика read-write, допускает фильтрацию строк/колонок, разные мажорные версии.

Какие ограничения у логической репликации PostgreSQL?::Не реплицирует DDL (CREATE/ALTER/DROP TABLE) и последовательности (sequences). При failover sequences на реплике покажут старые значения.

Почему DROP TABLE секции быстрее DELETE?::Удаляется физический файл целиком, без затрат на WAL и VACUUM. Мгновенная архивация старых данных.

Что произойдёт при вставке в секционированную таблицу без подходящей секции?::СУБД вернёт ошибку — строка не может быть маршрутизирована. Нужна секция DEFAULT.

Что происходит при падении единственной синхронной реплики?::Все write-транзакции на мастере зависают до бесконечности. Нужно уменьшить synchronous_standby_names и reload.

Вертикальное масштабирование (Scale Up):::Увеличение мощности одного сервера (CPU/RAM/Disk). Ограничено физическим пределом.

Горизонтальное масштабирование (Scale Out):::Добавление новых узлов в кластер. Требует шардирования или репликации.

RPO (Recovery Point Objective):::Максимально допустимый объём потерянных данных при сбое. При асинхронной репликации RPO > 0.

RTO (Recovery Time Objective):::Целевое время восстановления системы до работоспособного состояния.

Горизонтальное vs вертикальное шардирование
?
Горизонтальное: таблица делится по строкам (нечётные ID → Шард 1, чётные → Шард 2)
Вертикальное: таблица делится по столбцам (медиа-данные → отдельный шард)

Edge case: логическая репликация без Primary Key?::UPDATE/DELETE не могут идентифицировать строку — логическая репликация таблицы без PK невозможна.
