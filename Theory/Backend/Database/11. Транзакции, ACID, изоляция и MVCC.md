## Суть и контекст

Конспект охватывает механизмы обеспечения корректности данных в реляционных СУБД при конкурентном доступе:

1. **ACID** — четыре фундаментальных свойства транзакций (Atomicity, Consistency, Isolation, Durability) и механизмы их обеспечения.
2. **Уровни изоляции** — Read Uncommitted, Read Committed, Repeatable Read, Serializable — и соответствующие аномалии (dirty reads, phantom reads, lost updates).
3. **MVCC (Multi-Version Concurrency Control)** — ключевой механизм PostgreSQL, обеспечивающий неблокирующее чтение через версионирование строк и снимки транзакций.

---

## Ключевые идеи

- Транзакция — логическая единица работы, преобразующая БД из одного согласованного состояния в другое. Является единицей восстановления и параллельности.
- ACID — Atomicity (всё или ничего), Consistency (согласованное состояние), Isolation (скрытость изменений до фиксации), Durability (устойчивость при сбоях).
- Журнал опережающей записи (WAL) обеспечивает атомарность и долговечность: запись журнала на диск обязательна до изменений в БД и до маркера COMMIT.
- Строгий двухфазный протокол блокировки (Strict 2PL) математически гарантирует сериализуемость: все блокировки освобождаются только по завершении транзакции.
- MVCC в PostgreSQL: каждая SQL-команда видит снимок данных (snapshot) на момент в прошлом. Читатели не блокируют писателей, писатели не блокируют читателей.
- PostgreSQL не перезаписывает строки in-place — UPDATE создаёт новую физическую версию. Версии одной строки образуют односвязный список.
- READ UNCOMMITTED в PostgreSQL эквивалентен READ COMMITTED — грязное чтение невозможно из-за архитектуры MVCC.
- REPEATABLE READ в PostgreSQL строже стандарта ANSI SQL — фантомное чтение тоже не допускается.
- SERIALIZABLE в PostgreSQL реализован через SSI (Serializable Snapshot Isolation), отслеживающий циклы rw-конфликтов.
- XID ограничен 32 битами (4 млрд). Требуется VACUUM для замораживания старых xmin и предотвращения катастрофической потери видимости данных (Transaction ID Wraparound).

## Определения и термины

### Транзакции и ACID

**Транзакция** — логическая единица работы базы данных, преобразующая базу из одного правильного (согласованного) состояния в другое. Является единицей восстановления и параллельности.

**Атомарность (Atomicity)** — транзакция действует по принципу "все или ничего". Либо выполняются все операции транзакции, либо не выполняется ни одна.

**Согласованность (Consistency)** — транзакция переводит базу данных из одного согласованного (правильного) состояния в другое согласованное состояние. Включает проверку ограничений целостности (первичные, внешние ключи, check).

**Изолированность (Isolation)** — изменения, внесённые конкретной транзакцией, остаются скрытыми от всех других транзакций до тех пор, пока не произойдёт фиксация данной транзакции.

**Долговечность (Durability)** — если клиент получил подтверждение успешного завершения транзакции (зафиксирована), внесённые ею обновления сохраняются в базе данных на постоянной основе, даже если в дальнейшем произойдёт аварийный останов системы.

### Аномалии конкурентного доступа

**Грязное чтение (Dirty read)** — транзакция читает данные, изменённые, но не подтверждённые параллельной транзакцией.

**Неповторяющееся чтение (Nonrepeatable read)** — транзакция повторно считывает данные и обнаруживает, что они были модифицированы другой транзакцией (зафиксированной с момента первого чтения).

**Фантомное чтение (Phantom read)** — транзакция повторно выполняет запрос, возвращающий набор строк, и обнаруживает изменение набора из-за добавления новых строк другой недавно зафиксированной транзакцией.

**Потерянное обновление (Lost update)** — транзакция перезаписывает данные, которые уже были обновлены или обрабатываются другой транзакцией.

**Несогласованная запись (Write skew)** — транзакция делает расчёты на основании информации, которую изменяет другая транзакция, и сохраняет результат. Пример: две транзакции проверяют остаток баланса и обе делают списания — итоговый баланс уходит в минус.

**Грязная запись (Dirty write)** — две транзакции конкурентно изменяют одну и ту же строку (перезаписывают незафиксированные изменения).

**Несогласованное чтение (Read Skew)** — транзакция читает часть данных в одном состоянии, а часть — в другом, изменённом конкурентной транзакцией. По сути — разновидность Nonrepeatable Read, но затрагивает разные объекты в одной транзакции.

### MVCC

**MVCC (Multiversion Concurrency Control)** — модель многоверсионного управления конкурентным доступом, при которой каждая SQL-команда видит снимок данных (snapshot) на момент времени в прошлом, а блокировки для чтения не конфликтуют с блокировками для записи.

**Версия строки (Row version)** — индивидуальное состояние строки данных; каждое обновление строки создаёт новую физическую версию для той же логической строки.

**xmin** — идентификатор транзакции (XID), создавшей (вставившей) данную версию строки.

**xmax** — идентификатор транзакции, удалившей или обновившей данную версию строки. Значение 0 означает, что версия не удалена.

**cmin / cmax** — идентификаторы команды (command identifier) внутри транзакции, начиная с нуля. cmin — идентификатор команды, вставившей строку, cmax — удалившей строку.

**ctid** — физическое расположение версии строки внутри таблицы.

**Снимок транзакции (pg_snapshot)** — структура данных, хранящая информацию о видимости XID. Формат: `xmin:xmax:xip_list` (например, `10:20:10,14,15`).

Компоненты снимка:
- **xmin** в снимке — ID самой старой из всё ещё активных транзакций. Все транзакции с ID < xmin либо уже зафиксированы, либо откачены.
- **xmax** в снимке — ID первой ещё не начатой транзакции. Все транзакции с ID ≥ xmax невидимы.
- **xip_list** — список транзакций, находившихся в процессе выполнения (in-progress) на момент создания снимка.

**FrozenTransactionId** — специальный XID (исторически = 2), который всегда считается старше любой нормальной транзакции. Используется для защиты от Transaction ID Wraparound.

## Как устроено

### Механизмы атомарности и долговечности (журналирование)

1. Операции транзакции выполняются в буферах оперативной памяти.
2. Генерируются записи журнала с образами данных до и после обновления.
3. **Правило опережающей записи**: запись журнала должна быть физически внесена на диск до того, как изменения будут перенесены в саму базу данных, и до того, как будет записан маркер COMMIT.
4. При ROLLBACK система читает журнал и восстанавливает предыдущие значения.
5. При аварийном останове: сканирование контрольных точек (checkpoints), REDO зафиксированных транзакций, UNDO незавершённых.

### Механизмы изоляции — пессимистическая блокировка (Strict 2PL)

1. Транзакция, читающая кортеж, запрашивает S-блокировку (Shared).
2. Транзакция, обновляющая кортеж, запрашивает X-блокировку (Exclusive) или повышает S до X.
3. При конфликте (запрос X на ресурс с S-блокировкой) запрашивающая транзакция переходит в состояние ожидания.
4. Все блокировки освобождаются только по завершении транзакции (COMMIT/ROLLBACK), гарантируя сериализуемость.

### Уровни изоляции транзакций

| Уровень          | Dirty read            | Nonrepeatable read | Phantom read          | Serialization anomaly |
| ---------------- | --------------------- | ------------------ | --------------------- | --------------------- |
| READ UNCOMMITTED | Possible (но не в PG) | Possible           | Possible              | Possible              |
| READ COMMITTED   | Not possible          | Possible           | Possible              | Possible              |
| REPEATABLE READ  | Not possible          | Not possible       | Possible (но не в PG) | Possible              |
| SERIALIZABLE     | Not possible          | Not possible       | Not possible          | Not possible          |

- **READ UNCOMMITTED** в PostgreSQL = READ COMMITTED (dirty read невозможен из-за MVCC).
- **READ COMMITTED**: запрос видит данные, зафиксированные до начала текущего оператора (снимок оператора).
- **REPEATABLE READ**: запрос видит данные, зафиксированные до начала транзакции (снимок транзакции). В PostgreSQL фантомы тоже не допускаются.
- **SERIALIZABLE**: реализован через SSI (Serializable Snapshot Isolation), отслеживающий циклы rw-конфликтов.

### MVCC — версионирование строк

PostgreSQL не перезаписывает данные in-place. Версии одной логической строки объединены в односвязный список: старая версия → новая, самая свежая указывает сама на себя.

- **INSERT**: создаётся версия с xmin = текущий XID, xmax = 0.
- **DELETE**: в xmax актуальной версии записывается ID удаляющей транзакции.
- **UPDATE**: логический DELETE старой версии + INSERT новой.

### MVCC — правила видимости

1. Транзакция извлекает свой snapshot (snapshot.xmin, snapshot.xmax, snapshot.xip_list).
2. Если xmin строки в xip_list (активна) или xmin ≥ snapshot.xmax (будущая) → **невидима**.
3. Если xmin не в xip_list и < snapshot.xmax → проверяем статус в CLOG: если aborted → **невидима**.
4. Если xmin committed → проверяем xmax:
   - xmax = 0 (не удалена) или xmax в xip_list (удаляющая активна) → **видима**.
   - xmax committed и не в xip_list → **невидима**.

## Детали и нюансы

- **Типы конфликтов и аномалии**: RW → Nonrepeatable read, WR → Dirty read, WW → Lost Update.
- **Конкурентное обновление на RR/Serializable**: первая транзакция фиксируется успешно, вторая отменяется с ошибкой `could not serialize access due to concurrent update` (SQLSTATE 40001). Приложение **обязано** перехватывать код 40001 и выполнять retry всей бизнес-логики с начала.
- **Четыре условия возникновения deadlock**: (1) Взаимоисключение — ресурс используется только одним процессом. (2) Удержание и ожидание — процесс держит ресурс и запрашивает новый. (3) Отсутствие вытеснения — ресурс нельзя принудительно забрать. (4) Циклическая зависимость — замкнутое кольцо в графе ожиданий.
- **Deadlock при 2PL**: A ждёт B, B ждёт A. СУБД оптимистично ожидает `deadlock_timeout` (по умолчанию 1 сек), затем строит граф ожиданий (wait-for graph), при обнаружении цикла одна транзакция обрывается (victim selection, код 40P01 deadlock_detected).
- **Глобальный Deadlock в распределённой системе**: T1 ждёт T2 на узле X, T2 ждёт T1 на узле Y. Ни один узел локально не видит цикла — для обнаружения требуется обмен графами (сетевые расходы) или kill by timeout.
- **Стратегии предотвращения deadlock**: (1) Lock timeout — СУБД прерывает транзакцию, не получившую ресурс за пороговый период. (2) Retry logic — приложение перехватывает ошибку deadlock victim и повторяет транзакцию. (3) Wait-Die — «если выполнение транзакции А началось раньше, чем В, А переходит в состояние ожидания; в противном случае происходит её отмена» (откат и перезапуск А). (4) Wound-Wait — «если выполнение транзакции А началось позже, чем В, она переходит в состояние ожидания; в противном случае, она отменяет В» (откат и перезапуск В). (5) Блокировать объекты всегда в одинаковом строгом порядке во всех частях приложения. (6) Первой запрашивать блокировку самого строгого режима из всех, которые потребуются.
- **Выбор жертвы в PostgreSQL**: «как правило, отменяется последняя транзакция, замкнувшая цикл… но рассчитывать на это нельзя. Например, Db2 отменит произвольную транзакцию»; «точно, какая транзакция будет прервана, трудно предсказать, и на это не стоит полагаться».
- **Таймауты вместо обнаружения**: «на практике не все системы обнаруживают взаимоблокировки как таковые; в некоторых системах используется лишь механизм тайм-аутов и просто предполагается, что если некоторая транзакция не выполняет никакой работы в течение некоторого заранее установленного периода времени, то она находится в состоянии взаимоблокировки».
- **Deadlock vs Livelock vs Lock contention**:
    - *Deadlock* — циклическое ожидание, при котором ни одна транзакция не может продолжаться.
    - *Livelock (активный тупик / starvation)* — «ситуация называется активным тупиком — livelock, или истощением ресурсов — starvation… когда транзакция ожидает до бесконечности»; решается FIFO-очередью обслуживания блокировок.
    - *Lock contention (конкуренция за блокировки)* — неудовлетворённые блокировки, являющиеся «источником раздора (contention) среди клиентов базы данных» и влияющие на «общую производительность»; транзакции не попали в цикл, но ждут друг друга дольше ожидаемого.
- **Deadlock и индексы**: «B-tree, GiST и SP-GiST индексы… обеспечивают высочайшую параллельность без условий взаимоблокировки», однако «в Hash индексах… взаимоблокировка возможна, так как блокировки удерживаются дольше, чем одна операция с индексом».
- **Deadlock через внешние ключи**: «если мы в разных транзакциях со связанными таблицами работаем в разном порядке, то у нас ещё возможны дедлоки».
- **Предикатные блокировки SSI**: SERIALIZABLE в PostgreSQL использует предикатные блокировки (Predicate locks), которые «отображаются в системном представлении `pg_locks` с режимом `SIReadLock`». При обнаружении циклической зависимости, способной нарушить упорядочиваемость, происходит откат одной из транзакций (serialization failure).
- **Вложенные транзакции**: если внешняя A вызвала ROLLBACK, изменения внутренней B будут отменены. Durability применимо только к транзакциям самого внешнего уровня.
- **Практическая разница READ COMMITTED vs REPEATABLE READ при конкурентном UPDATE**: если две транзакции параллельно меняют одну строку — в READ COMMITTED вторая транзакция дождётся COMMIT первой, переоценит условие WHERE для новой версии строки и обновит её. В REPEATABLE READ вторая транзакция дождётся COMMIT первой и **немедленно упадёт** с ошибкой `ERROR: could not serialize access due to concurrent update`, так как она не имеет права изменять строку, изменённую после создания её снимка.
- **Read Committed + параллельный UPDATE/DELETE**: SQL-оператор может "не увидеть" часть строк, которые перемещаются/модифицируются (строка не попадает под WHERE из-за сдвига версий).
- **Последствия долгих транзакций (long-running transactions)**:
  1. **Bloat (разбухание таблиц)** — VACUUM не может удалить dead tuples, если они видимы для старого снимка бездействующей транзакции: «an open transaction prevents vacuuming away recently-dead tuples that may be visible only to this transaction; so remaining idle for a long time can contribute to table bloat». Файлы данных растут неограниченно.
  2. **Исчерпание лимитов** — общий менеджер блокировок может исчерпать `max_locks_per_transaction`, что заблокирует работу всего сервера.
  3. **Конфликты на репликах (early cleanup)** — «The most common reason for conflict between standby queries and WAL replay is "early cleanup"… cleanup on the primary will remove row versions that are still visible to a transaction on the standby». После превышения `max_standby_archive_delay` или `max_standby_streaming_delay` конфликтующие запросы отменяются (`canceled due to old snapshots`); при replay команды DROP DATABASE завершается вся конфликтующая сессия.
  4. **Блокировка DDL** — долгое удержание даже AccessShareLock блокирует DDL-операции, вызывая очереди ожидающих DML-запросов.
  5. **Накопление WAL** — «longer delays will result in a greater accumulation of WAL files, increasing disk space requirements for the standby's pg_wal directory». Если `pg_wal/` переполняется — PostgreSQL делает **PANIC shutdown** и БД остаётся offline до освобождения места. Replication slots могут удерживать WAL-сегменты и усугублять проблему.
  6. **Обнаружение**: `pg_stat_activity` — строки с большим `age(backend_xid)` или `age(backend_xmin)` (state `idle in transaction` означает «waiting for client inside a BEGIN block»). Для зависших prepared транзакций — `pg_prepared_xacts` с большим `age(transactionid)`.
  7. **Решение**: `idle_in_transaction_session_timeout` — «terminate any session that has been idle (that is, waiting for a client query) within an open transaction for longer than the specified amount of time». Прерывание сессии — `pg_terminate_backend(pid)` (также `pg_cancel_backend()` — оба работают на user backends, но **не** на startup process, выполняющий recovery).
  8. **Trade-off `hot_standby_feedback`** — «prevents VACUUM from removing recently-dead rows and so cleanup conflicts do not occur… you should note that this will delay cleanup of dead rows on the primary, which may result in undesirable table bloat». Устраняет отмены запросов на Standby ценой bloat на primary.
  9. **Взаимодействие таймаутов**: `lock_timeout` срабатывает только при ожидании блокировок — «if statement_timeout is nonzero, it is rather pointless to set lock_timeout to the same or larger value»; «If transaction_timeout is shorter or equal to idle_in_transaction_session_timeout or statement_timeout then the longer timeout is ignored».
  10. Для задач, требующих выполнения часы/дни — паттерн **Саги (Sagas)** с компенсирующими транзакциями вместо обычных ACID-транзакций.
- **Transaction ID Wraparound**: XID = 32 бита (4 млрд), пространство закольцовано. Старая транзакция, превысившая порог в 2 млрд, воспринимается как "будущая" → данные невидимы. Требуется VACUUM для замены xmin на FrozenTransactionId.
- **xmax может быть ненулевым для видимой строки** — если удаляющая транзакция активна или была откачена (rolled back). СУБД перепроверяет статус xmax в CLOG.
- **UPDATE увеличивает размер файлов** (bloat) — старые версии остаются для видимости прошлым транзакциям. Требуется VACUUM.
- **Replication lag на pg_stat_replication**: при простое мастера значения write_lag, flush_lag, replay_lag показывают время обработки последней WAL-записи, а не 0.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Пессимистические блокировки (2PL) | Математически гарантирует сериализуемость | Deadlocks, ожидание блокировок, снижение throughput | Когда нужна строгая изоляция без retry-логики |
| MVCC (PostgreSQL) | Читатели не блокируют писателей, COMMIT/ROLLBACK мгновенны | Bloat, требуется VACUUM, Transaction ID Wraparound | PostgreSQL — основной механизм |
| MVCC (Oracle/InnoDB) | UPDATE in-place + undo segment, нет bloat | ROLLBACK долгий (читается undo) | Oracle, MySQL InnoDB |
| READ COMMITTED | Нет dirty read, минимальные ограничения | Nonrepeatable read, phantoms | Дефолт PostgreSQL, OLTP |
| REPEATABLE READ | Нет dirty + nonrepeatable read | Retry при конфликте (40001) | Финансовые отчёты, агрегации |
| SERIALIZABLE (SSI) | Полная сериализуемость без блокировок чтения | Retry при конфликте, overhead отслеживания зависимостей | Критичная бизнес-логика |
| Оптимистическая блокировка | Нет lock wait, высокий throughput при редких конфликтах | Retry при конфликте на стороне приложения | Редкие конфликты, human interaction |
| Пессимистическая блокировка | Без retry, гарантированный доступ | Lock wait, deadlocks, снижение throughput | Высокая вероятность конфликтов |

### Виды блокировок строк (Row-level locks)

| Режим | Описание | Блокирует |
|-------|----------|-----------|
| **FOR UPDATE** | Эксклюзивная блокировка строк | UPDATE, DELETE, FOR UPDATE, FOR SHARE от других транзакций |
| **FOR NO KEY UPDATE** | Ослабленная эксклюзивная | FOR UPDATE, но разрешает FOR KEY SHARE |
| **FOR SHARE** | Разделяемая блокировка | UPDATE, DELETE, FOR UPDATE, но разрешает FOR SHARE |
| **FOR KEY SHARE** | Ослабленная разделяемая | DELETE и UPDATE ключевых значений, но не прочие UPDATE |

**Row-level locks в PostgreSQL** записываются прямо в физические страницы данных на диске (не в lock manager в RAM), что генерирует I/O трафик. В PostgreSQL нет эскалации блокировок (lock escalation) — в отличие от MS SQL/Db2, где миллионы row locks могут эскалироваться до table lock.

### Виды блокировок таблиц (Table-level locks в PostgreSQL)

| Режим | Устанавливается | Конфликтует с |
|-------|----------------|---------------|
| **ACCESS SHARE** | SELECT | ACCESS EXCLUSIVE |
| **ROW SHARE** | SELECT FOR UPDATE/SHARE | EXCLUSIVE, ACCESS EXCLUSIVE |
| **ROW EXCLUSIVE** | INSERT, UPDATE, DELETE | SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE |
| **SHARE UPDATE EXCLUSIVE** | VACUUM (не FULL), CREATE INDEX CONCURRENTLY | SHARE UPDATE EXCLUSIVE и выше |
| **SHARE** | CREATE INDEX | ROW EXCLUSIVE и выше |
| **SHARE ROW EXCLUSIVE** | CREATE TRIGGER, некоторые ALTER TABLE | ROW EXCLUSIVE и выше |
| **EXCLUSIVE** | Блокирует всё, кроме ACCESS SHARE | ROW SHARE и выше |
| **ACCESS EXCLUSIVE** | DROP TABLE, TRUNCATE, ALTER TABLE, VACUUM FULL | Все режимы, включая обычные SELECT |

### Намеченные блокировки (Intent locks)

Протокол, требующий установки блокировки на родительский объект (таблицу) перед блокировкой дочернего (строки) — для быстрого обнаружения конфликтов на уровне иерархии.

| Режим | Описание |
|-------|----------|
| **IS (Intent Shared)** | Намерение получить S-блокировку на строки |
| **IX (Intent Exclusive)** | Намерение получить X-блокировку на строки |
| **SIX (Shared Intent Exclusive)** | Таблица заблокирована для чтения (S), ниже по иерархии будут X-блокировки |

### Эскалация блокировок (Lock escalation)

**Эскалация блокировок** — механизм, при котором СУБД заменяет множество блокировок с тонкой детализацией (строки) на единственную с грубой детализацией (таблицы) при исчерпании лимита памяти диспетчера блокировок. Экономит RAM ценой катастрофического снижения параллелизма. В PostgreSQL эскалации нет.

**Advisory locks** — рекомендательные блокировки, семантика определяется приложением (аналог мьютексов ОС, но на уровне БД; хранятся в RAM, не вызывают bloat).

### Пессимистическая vs оптимистическая блокировка

**Пессимистическая**: SELECT FOR UPDATE → долгие расчёты (другие транзакции висят в очереди) → UPDATE → COMMIT.

**Оптимистическая**: SELECT с версией (_VERSION) → расчёты без блокировок → UPDATE WHERE _VERSION = Z → если 0 строк обновлено, retry. Аналогична паттерну **Compare-And-Swap (CAS)** из lock-free структур данных.

Пессимистические используют при **высокой** вероятности конфликтов (избегаем retry). Оптимистические — при **редких** конфликтах или когда human interaction может задержать транзакцию на часы.

### Предотвращение deadlock

| Стратегия | Механика |
|-----------|----------|
| Lock timeout | СУБД прерывает транзакцию, не получившую ресурс за пороговый период |
| Retry logic | Приложение перехватывает ошибку deadlock victim и повторяет транзакцию |
| Wait-Die | Старая ждёт ресурс молодой; молодая откатывается и перезапускается |
| Wound-Wait | Старая принудительно откатывает молодую; молодая ждёт |
| Advisory locks | Координация на уровне приложения (pg_advisory_lock / pg_try_advisory_lock) |

```sql
-- Явное задание уровня изоляции
START TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Перевод денег (иллюстрация атомарности)
BEGIN TRANSACTION ;
UPDATE ACC 123 { BALANCE := BALANCE - $100 } ;
IF <возникла ошибка> THEN GO TO UNDO ; END IF ;
UPDATE ACC 456 { BALANCE := BALANCE + $100 } ;
IF <возникла ошибка> THEN GO TO UNDO ; END IF ;
COMMIT ;
GO TO FINISH ;
UNDO :
ROLLBACK ;

-- Пессимистические блокировки строк
SELECT * FROM mytab WHERE id = 1 FOR UPDATE;      -- эксклюзивная блокировка
SELECT * FROM mytab WHERE id = 1 FOR SHARE;       -- разделяемая блокировка
SELECT * FROM mytab WHERE id = 1 FOR NO KEY UPDATE; -- ослабленная эксклюзивная
SELECT * FROM mytab WHERE id = 1 FOR KEY SHARE;   -- ослабленная разделяемая

-- Оптимистическая блокировка (версионирование)
-- 1. Чтение с версией: SELECT id, val, _VERSION FROM t WHERE id = 42;
-- 2. Расчёты в приложении (без блокировок в БД)
-- 3. Запись с проверкой версии:
UPDATE t SET val = X, _VERSION = _VERSION + 1 WHERE id = 42 AND _VERSION = 88;
-- Если 0 строк обновлено → конфликт, retry с шага 1

-- Рекомендательная блокировка (Advisory lock)
SELECT pg_advisory_lock(key);       -- ждёт, пока не освободится
SELECT pg_try_advisory_lock(key);   -- возвращает false немедленно, если занят

-- Классическая взаимоблокировка (deadlock) двух транзакций
-- Транзакция 1:
UPDATE accounts SET balance = balance + 100.00 WHERE acctnum = 11111;
-- Транзакция 2:
UPDATE accounts SET balance = balance + 100.00 WHERE acctnum = 22222;
UPDATE accounts SET balance = balance - 100.00 WHERE acctnum = 11111; -- ждёт Т1
-- Транзакция 1:
UPDATE accounts SET balance = balance - 100.00 WHERE acctnum = 22222; -- ждёт Т2 → deadlock!

-- Принудительное уничтожение "зависших" сессий
-- idle_in_transaction_session_timeout = 60000  -- в postgresql.conf (60 секунд)

-- Мониторинг ожиданий блокировок
SELECT pid, wait_event_type, wait_event FROM pg_stat_activity WHERE wait_event is NOT NULL;

-- Отношение с наибольшим числом непредоставленных (ungranted) блокировок
-- SELECT relation::regclass, count(*) FROM pg_locks WHERE NOT granted GROUP BY relation ORDER BY 2 DESC;

-- Предикатные блокировки SSI (SERIALIZABLE)
SELECT locktype, mode, relation::regclass FROM pg_locks WHERE mode = 'SIReadLock';

-- Просмотр системных полей MVCC
SELECT xmin, xmax, cmin, cmax, ctid, tableoid, * FROM my_table;

-- Получение текущего снимка транзакции
SELECT pg_current_snapshot();

-- Извлечение активных транзакций из снимка
SELECT pg_snapshot_xip(pg_current_snapshot());

-- Проверка видимости XID в рамках снимка
SELECT pg_visible_in_snapshot(xid8, pg_snapshot);
```

## Важно / подводные камни / best practices

- Использовать REPEATABLE READ или SERIALIZABLE для финансовых агрегатных проверок — на READ COMMITTED последовательные SELECT SUM(...) вернут разные результаты.
- При конфликте на RR/Serializable (ошибка 40001) приложение **обязано** повторить всю транзакцию с самого начала (retry loop).
- Антипаттерн: Cursor Stability — освобождает S-блокировку сразу после сдвига курсора (до COMMIT), допуская nonrepeatable read.
- Антипаттерн: антизапросы на отмену операций в коде приложения — ROLLBACK перекладывает ответственность на СУБД.
- Антипаттерн: использование ctid как логического идентификатора — ctid физически меняется при UPDATE и VACUUM FULL. Использовать Primary Key.
- Периодический VACUUM (настройка autovacuum) критически важен для предотвращения bloat и Transaction ID Wraparound.
- COMMIT и ROLLBACK в PostgreSQL практически мгновенны (меняется только статус в CLOG), но требуется фоновая очистка от мёртвых кортежей.

## Карточки для повторения

#flashcards/db/transactions

Транзакция:::Логическая единица работы БД, преобразующая базу из одного согласованного состояния в другое. Является единицей восстановления и параллельности.

Атомарность (Atomicity):::Транзакция действует по принципу "все или ничего" — либо выполняются все операции, либо не выполняется ни одна.

Согласованность (Consistency) в ACID:::Транзакция переводит БД из одного согласованного состояния в другое.

Изолированность (Isolation):::Изменения транзакции скрыты от других транзакций до момента фиксации (COMMIT).

Долговечность (Durability):::Если клиент получил подтверждение COMMIT — обновления сохраняются даже при аварийном останове системы.

Грязное чтение (Dirty read):::Транзакция читает данные, изменённые, но не подтверждённые параллельной транзакцией.

Неповторяющееся чтение (Nonrepeatable read):::Транзакция повторно считывает данные и обнаруживает их модификацию другой зафиксированной транзакцией.

Фантомное чтение (Phantom read):::Транзакция повторно выполняет запрос и обнаруживает новые строки, добавленные другой зафиксированной транзакцией.

Потерянное обновление (Lost update):::Транзакция перезаписывает данные, уже обновлённые другой транзакцией.

Write skew (Несогласованная запись):::Транзакция делает расчёты на основании информации, которую изменяет другая транзакция, и сохраняет результат.

Read Skew (Несогласованное чтение):::Транзакция читает часть данных в одном состоянии, а часть — в другом, изменённом конкурентной транзакцией. Разновидность Nonrepeatable Read для разных объектов.

MVCC (Multiversion Concurrency Control):::Модель, при которой каждая SQL-команда видит snapshot на момент в прошлом, блокировки чтения не конфликтуют с блокировками записи.

Что такое pg_snapshot и его формат?::Структура видимости XID. Формат: xmin:xmax:xip_list (например, 10:20:10,14,15).

Что означает xmin в снимке транзакции?::ID самой старой из активных транзакций. Все транзакции с ID < xmin уже зафиксированы или откачены.

Что означает xmax в снимке транзакции?::ID первой ещё не начатой транзакции. Все транзакции с ID ≥ xmax невидимы.

В чём разница MVCC PostgreSQL vs Oracle/InnoDB?::PostgreSQL хранит версии строк в heap (bloat + VACUUM). Oracle/InnoDB хранит in-place + undo segment (нет bloat, но медленный ROLLBACK).

Что хранится в поле xmin версии строки?::ID транзакции (XID), создавшей (вставившей) данную версию строки.

Что хранится в поле xmax версии строки?::ID транзакции, удалившей или обновившей данную версию. 0 = версия не удалена.

FrozenTransactionId:::Специальный XID (=2), который всегда считается старше любой нормальной транзакции. Защищает от Transaction ID Wraparound.

Правило опережающей записи (WAL)
?
Запись журнала должна быть физически внесена на диск:
1. До переноса изменений в файлы БД
2. До записи маркера COMMIT

Шаги восстановления при аварийном останове (WAL)
?
1. Сканирование контрольных точек (checkpoints)
2. REDO зафиксированных транзакций, не попавших на диск
3. UNDO незавершённых транзакций

Strict 2PL — порядок работы блокировок
?
1. Чтение кортежа → S-блокировка (Shared)
2. Обновление → X-блокировка (Exclusive) или повышение S→X
3. При конфликте — ожидание
4. Все блокировки освобождаются только по завершении (COMMIT/ROLLBACK)

Что видит запрос на уровне READ COMMITTED?::Данные, зафиксированные до начала текущего оператора (снимок оператора).

Что видит запрос на уровне REPEATABLE READ?::Данные, зафиксированные до начала транзакции (снимок транзакции). В PostgreSQL фантомы тоже не допускаются.

Чем READ UNCOMMITTED отличается от READ COMMITTED в PostgreSQL?::Ничем — PostgreSQL реализует READ UNCOMMITTED идентично READ COMMITTED из-за архитектуры MVCC.

Через какой механизм реализован SERIALIZABLE в PostgreSQL?::SSI (Serializable Snapshot Isolation), отслеживающий циклы зависимостей rw-конфликтов.

Что происходит при UPDATE в MVCC PostgreSQL?
?
1. Логический DELETE старой версии (в xmax пишется ID транзакции)
2. INSERT новой физической версии (в xmin пишется ID транзакции)
3. Старая и новая версии связаны в односвязный список

Правила видимости версии строки в MVCC
?
1. Извлечь snapshot (xmin, xmax, xip_list)
2. Если xmin в xip_list или ≥ snapshot.xmax → невидима
3. Если xmin не в xip_list и < snapshot.xmax → проверить CLOG: aborted → невидима
4. Если xmin committed → проверить xmax: 0 или в xip_list → видима; committed → невидима

Конфликт RW приводит к аномалии ==Nonrepeatable read==

Конфликт WR приводит к аномалии ==Dirty read==

Конфликт WW приводит к аномалии ==Lost Update==

XID в PostgreSQL ограничен ==32 битами== (==4 млрд== значений)

Что такое Transaction ID Wraparound?::XID закольцован: старая транзакция, превысившая порог в 2 млрд, воспринимается СУБД как "будущая" → данные становятся невидимыми. Требуется VACUUM для замораживания xmin.

Что делать при ошибке serialization_failure (SQLSTATE 40001)?::Приложение обязано перехватить код ошибки и повторить всю транзакцию с самого начала (retry).

Как СУБД обнаруживает Deadlock?::Строит граф ожиданий (wait-for graph), при обнаружении цикла одна транзакция обрывается (victim selection, код 40P01).

Стратегии предотвращения deadlock
?
1. Lock timeout — прерывание при превышении порога ожидания
2. Retry logic — приложение перехватывает ошибку и повторяет
3. Wait-Die — старая ждёт молодую; молодая откатывается
4. Wound-Wait — старая откатывает молодую; молодая ждёт

Пессимистическая блокировка:::Транзакция захватывает эксклюзивную блокировку при чтении для изменения и удерживает до COMMIT. Используют при высокой вероятности конфликтов

Оптимистическая блокировка:::Строка не блокируется при чтении; при записи проверяется, не изменилась ли версия (_VERSION). Аналог Compare-And-Swap (CAS). Используют при редких конфликтах

Что делает FOR UPDATE?::Эксклюзивная блокировка строк, предотвращающая UPDATE, DELETE и FOR UPDATE/SHARE от других транзакций до COMMIT

Чем FOR NO KEY UPDATE отличается от FOR UPDATE?::Ослабленная эксклюзивная блокировка — блокирует FOR UPDATE, но разрешает FOR KEY SHARE

Advisory lock (рекомендательная блокировка):::Блокировка, семантика которой определяется приложением, а не СУБД. Аналог мьютексов ОС на уровне БД; хранится в RAM, не вызывает bloat

Где хранятся row-level locks в PostgreSQL?::Прямо в физических страницах данных на диске (не в lock manager в RAM), что генерирует I/O трафик. В PostgreSQL нет эскалации блокировок.

Четыре условия возникновения deadlock
?
1. Взаимоисключение — ресурс используется только одним процессом
2. Удержание и ожидание — процесс держит ресурс и запрашивает новый
3. Отсутствие вытеснения — ресурс нельзя принудительно забрать
4. Циклическая зависимость — замкнутое кольцо в графе ожиданий

Эскалация блокировок (Lock escalation):::Замена множества блокировок с тонкой детализацией (строки) на единственную с грубой детализацией (таблицы) при исчерпании лимита памяти. Экономит RAM ценой снижения параллелизма. В PostgreSQL отсутствует.

Намеченная блокировка (Intent lock):::Блокировка родительского объекта (таблицы) перед блокировкой дочернего (строки) — для быстрого обнаружения конфликтов на уровне иерархии. Виды: IS, IX, SIX.

Какую блокировку устанавливает ACCESS EXCLUSIVE?::Монопольный доступ (DROP TABLE, TRUNCATE, ALTER TABLE, VACUUM FULL). Блокирует всё, включая обычные SELECT.

Какую блокировку устанавливает SHARE UPDATE EXCLUSIVE?::VACUUM (не FULL), CREATE INDEX CONCURRENTLY. Не блокирует чтение и запись.

Практическая разница READ COMMITTED vs REPEATABLE READ при конкурентном UPDATE
?
READ COMMITTED: вторая транзакция дождётся COMMIT первой, переоценит WHERE для новой версии строки и обновит.
REPEATABLE READ: вторая транзакция дождётся COMMIT первой и упадёт с ошибкой serialization_failure — нельзя менять строку, изменённую после создания снимка.

Последствия долгих транзакций (long-running)
?
1. Bloat — VACUUM не может удалить dead tuples, видимые для старого снимка
2. Исчерпание max_locks_per_transaction — блокировка сервера
3. Конфликты на Hot Standby — ошибки conflict with recovery
4. Блокировка DDL — очереди ожидающих DML-запросов

Что делает idle_in_transaction_session_timeout?::Автоматически убивает сессии, находящиеся в состоянии idle-in-transaction дольше заданного времени (в мс). Предотвращает bloat и блокировки.

Что такое паттерн Саги (Sagas)?::Альтернатива длинным ACID-транзакциям — цепочка коротких транзакций с компенсирующими операциями для отката. Используется для задач, длящихся часы/дни.

Глобальный Deadlock в распределённой системе::T1 ждёт T2 на узле X, T2 ждёт T1 на узле Y. Ни один узел локально не видит цикла. Обнаружение: обмен графами ожиданий или kill by timeout.

Livelock (активный тупик / starvation):::Ситуация, когда транзакция «ожидает до бесконечности», не попадая в deadlock. Решается FIFO-очередью обслуживания блокировок.

Lock contention (конкуренция за блокировки):::Неудовлетворённые блокировки — «источник раздора (contention) среди клиентов базы данных», влияющий на общую производительность. В отличие от deadlock, цикла нет.

В чём разница deadlock vs livelock vs lock contention?
?
Deadlock: циклическое ожидание, прогресс невозможен.
Livelock: транзакция ждёт бесконечно без цикла (starvation), решается FIFO.
Lock contention: транзакции ждут дольше ожидаемого, но без цикла — просто конкуренция.

Какие индексы PostgreSQL не подвержены взаимоблокировкам?::B-tree, GiST и SP-GiST — обеспечивают высочайшую параллельность без условий взаимоблокировки.

В каких индексах PostgreSQL возможен deadlock?::В Hash-индексах — блокировки удерживаются дольше, чем одна операция с индексом.

Edge case: deadlock через внешние ключи::Если в разных транзакциях со связанными таблицами работают в разном порядке, возможны дедлоки.

Что такое SIReadLock в pg_locks?::Режим предикатной блокировки SSI (SERIALIZABLE). При обнаружении цикла rw-конфликтов одна из транзакций откатывается с serialization failure.

Как PostgreSQL выбирает жертву при deadlock?::Обычно отменяется последняя транзакция, замкнувшая цикл, но «точно, какая транзакция будет прервана, трудно предсказать, и на это не стоит полагаться».

Альтернатива обнаружению deadlock::Механизм тайм-аутов — если транзакция не выполняет работы в течение заранее установленного периода, предполагается что она в состоянии взаимоблокировки.

Мониторинг ожиданий блокировок::`SELECT pid, wait_event_type, wait_event FROM pg_stat_activity WHERE wait_event is NOT NULL;`

Как найти отношение с наибольшим количеством непредоставленных блокировок?::Через `pg_locks` с фильтром `WHERE NOT granted`, группировкой по `relation`.

Почему idle_in_transaction вызывает bloat?::Открытая транзакция удерживает снимок; VACUUM не может удалить dead tuples, которые всё ещё видимы этой транзакции («recently-dead tuples that may be visible only to this transaction»).

Причина конфликтов на Standby::«Early cleanup» — cleanup на primary удаляет row versions, которые ещё видимы транзакции на standby.

Что происходит после превышения max_standby_streaming_delay?::Конфликтующие запросы отменяются (`canceled due to old snapshots`). При replay DROP DATABASE — завершается вся сессия.

Trade-off hot_standby_feedback
?
Плюс: предотвращает cleanup conflicts — запросы на Standby не отменяются.
Минус: задерживает cleanup dead rows на primary → может вызвать bloat.

Риск переполнения pg_wal/::PostgreSQL делает PANIC shutdown — БД остаётся offline до освобождения места. Replication slots усугубляют, удерживая WAL-сегменты.

pg_stat_activity: признаки long-running транзакций::Большие значения `age(backend_xid)` или `age(backend_xmin)`; state `idle in transaction`.

Как найти зависшие prepared транзакции?::`pg_prepared_xacts` — строки с большим `age(transactionid)`.

pg_cancel_backend vs pg_terminate_backend::Оба работают на user backends, но НЕ на startup process (recovery).

Взаимодействие lock_timeout и statement_timeout::«If statement_timeout is nonzero, it is rather pointless to set lock_timeout to the same or larger value» — lock_timeout работает только при ожидании блокировок.

Взаимодействие transaction_timeout и других таймаутов::«If transaction_timeout is shorter or equal to idle_in_transaction_session_timeout or statement_timeout then the longer timeout is ignored».

В чём отличие MVCC PostgreSQL от Oracle/InnoDB?
?
PostgreSQL: версии лежат вперемешку в heap, COMMIT/ROLLBACK мгновенны, но нужен VACUUM (bloat)
Oracle: UPDATE in-place + undo segment, ROLLBACK долгий, но нет bloat

Почему ctid нельзя использовать как идентификатор строки?::ctid физически меняется при UPDATE (новая версия в новом месте) и при VACUUM FULL. Использовать Primary Key.

Edge case: xmax ≠ 0, но строка видима — когда?::Если удаляющая транзакция всё ещё активна (не зафиксирована) или была откачена (rolled back).
