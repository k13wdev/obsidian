## Суть и контекст

InnoDB — движок хранения MySQL, использующий принципиально отличную от PostgreSQL архитектуру: многопоточную модель вместо многопроцессной, и кластеризованный индекс (Clustered Index) вместо heap-таблиц. Каждая таблица физически организована как B-Tree.

---

## Ключевые идеи

- InnoDB использует **потоки в рамках одного процесса ОС** (многопоточная модель), в отличие от многопроцессной модели PostgreSQL.
- Каждая таблица в InnoDB физически организована как **кластеризованный индекс** (B-Tree, где в листовых узлах хранятся полные строки данных).
- **Вторичные индексы** содержат не физические указатели (ctid), а **значение первичного ключа**. Поиск по вторичному индексу требует двух проходов по B-Tree.
- Обновление первичного ключа в InnoDB — **катастрофически дорогая** операция.
- MVCC в InnoDB: старые версии строк хранятся в **Undo Log** (отдельные сегменты отката), а не в heap-таблице, как в PostgreSQL.
- **Doublewrite Buffer** защищает от torn pages — повреждения страниц при частичной записи.
- **Change Buffer** асинхронно откладывает обновления вторичных индексов, если страница индекса отсутствует в Buffer Pool.

---

## Определения и термины

**Кластеризованный индекс (Clustered Index / Index-Organized Table)** — B-дерево, в листовых узлах которого хранятся не ссылки на строки, а сами данные всех полей таблицы. Значение первичного ключа выступает в роли ключа B-дерева. Если PK не определён явно — InnoDB выбирает первое поле UNIQUE NOT NULL; если такого нет — создаёт скрытый суррогатный ключ.

**Buffer Pool** — область памяти InnoDB, где кешируются страницы данных и индексов. Управляется алгоритмом LRU: вытесняются страницы, к которым давно не было обращений.

**Redo Log** — журнал WAL в InnoDB. Физические изменения страниц записываются в redo log перед применением к файлам данных. Гарантирует восстановление после сбоя.

**Undo Log (сегменты отката)** — область хранения старых версий строк при MVCC. Новая версия записывается поверх старой; старая версия сохраняется в undo space и удаляется фоновым процессом Purge после завершения транзакции.

**Doublewrite Buffer** — механизм защиты от torn pages. Изменённые страницы сначала записываются в непрерывный буфер на диске (doublewrite buffer), и только после успешного подтверждения — в основные файлы данных.

**Change Buffer** — структура в памяти, откладывающая изменения вторичных индексов, если страница индекса отсутствует в Buffer Pool. Изменения применяются асинхронно при следующей загрузке страницы в память.

**Adaptive Hash Index (AHI)** — динамически строимые в памяти хэш-индексы для часто запрашиваемых страниц B-дерева. Ускоряет поиск до O(1) для горячих данных.

**Страница InnoDB** — базовая единица хранения. Размер по умолчанию 16 КБ; допустимый диапазон 2–64 КБ.

**Row formats:**
- **COMPACT / DYNAMIC** — стандартные форматы. Оставляют свободное место в страницах для новых версий при обновлении (аналог `fillfactor`).
- **COMPRESSED** — сжатие страниц на уровне файловой системы (sparse files). В памяти хранятся одновременно сжатая и несжатая копии. Если буфер изменений сжатой страницы переполняется — страница делится на две.

## Как устроено

### Архитектура хранения

1. Таблица = кластеризованный индекс (B-Tree по первичному ключу).
2. В листовых узлах — полные строки данных (все поля).
3. Данные физически упорядочены по первичному ключу.

### Вторичные индексы

1. Листовые узлы вторичного индекса содержат **значение первичного ключа** (не физический адрес).
2. Поиск по вторичному индексу требует **двух проходов**:
   - Спуск по вторичному B-дереву → получение значения первичного ключа.
   - Спуск по первичному кластеризованному B-дереву → получение данных строки.

### Buffer Pool и LRU

1. Процесс запрашивает страницу.
2. Если страница найдена в Buffer Pool — данные возвращаются из памяти.
3. Если не найдена — страница считывается с диска, загружается в Buffer Pool.
4. При необходимости вытеснить страницу — выбирается страница с наименьшим временем последнего обращения (LRU).
5. Грязные (dirty) страницы перед вытеснением сбрасываются на диск.

### Запись через Doublewrite Buffer

1. Изменённые страницы записываются в Doublewrite Buffer (непрерывный фрагмент на диске).
2. После подтверждения записи в Doublewrite Buffer — страницы записываются в основные файлы данных.
3. При сбое между шагами 1 и 2 — страница восстанавливается из Doublewrite Buffer.

### MVCC в InnoDB (vs PostgreSQL)

- **PostgreSQL**: версии строк хранятся в heap-таблице (xmin/xmax). Очистка — VACUUM.
- **InnoDB**: новая версия записывается поверх старой; старая версия сохраняется в Undo Log. Очистка — процесс Purge.

### Обновление первичного ключа

1. СУБД физически **удаляет строку** из одного листового узла B-дерева.
2. **Вставляет** в другой узел, соответствующий новому ключу.
3. Поскольку вторичные индексы хранят значения PK, изменение PK требует **каскадного массового обновления ВСЕХ вторичных индексов**.

## Сравнения и trade-offs

| Характеристика | InnoDB (MySQL) | PostgreSQL (Heap) |
|---------------|---------------|-------------------|
| Базовая структура | Индекс-организованная таблица (B-Tree) | Неупорядоченная таблица (heap) |
| Хранение строк | В листьях B-Tree по PK | В heap-файле (неупорядоченно) |
| Указатель в вторичном индексе | Значение PK | ctid (физический адрес) |
| Поиск по вторичному индексу | Два прохода по B-Tree | Один проход по индексу + один random I/O к heap |
| Стоимость UPDATE PK | Катастрофическая (перемещение строки + обновление ВСЕХ вторичных индексов) | Дешёвая (обновление ctid) |
| Range-запросы по PK | Быстрые (данные физически упорядочены) | Медленнее (heap неупорядочен) |
| Хранение версий MVCC | Undo Log (сегменты отката) | В heap-таблице (xmin/xmax) |
| Очистка старых версий | Процесс Purge | Процесс VACUUM |
| Защита от torn pages | Doublewrite Buffer | Full Page Writes в WAL |

## Важно / подводные камни / best practices

- **Никогда не обновляйте первичный ключ в InnoDB** — каскадное обновление всех вторичных индексов вызывает катастрофическое падение производительности.
- **Выбирайте PK осознанно** — в InnoDB PK определяет физический порядок данных; автоинкрементный integer — оптимальный выбор.
- **Помните о двух проходах** при поиске по вторичному индексу — это удваивает количество дисковых операций по сравнению с точечным поиском по PK.
