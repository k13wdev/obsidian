## Суть и контекст

Триггер — заранее откомпилированная процедура, хранящаяся вместе с базой данных и вызываемая сервером автоматически при возникновении указанных событий. Триггеры позволяют реализовать сложную логику на уровне СУБД, но их использование для бизнес-логики и ограничений целостности считается антипаттерном.

---

## Ключевые идеи

- Триггер состоит из трёх частей: **событие** (INSERT/UPDATE/DELETE), **условие** (WHEN), **действие** (хранимая процедура).
- Параметры момента срабатывания: **BEFORE** (до), **AFTER** (после), **INSTEAD OF** (вместо действия).
- Уровни срабатывания: **FOR EACH ROW** (для каждой строки) и **FOR EACH STATEMENT** (один раз на оператор).
- Эксперты строго рекомендуют **не использовать триггеры** для бизнес-логики и ограничений целостности.
- Каскадные триггеры (T1 → T2 → T1) могут привести к переполнению стека/памяти СУБД.

---

## Определения и термины

**Триггер (Trigger)** — заранее откомпилированная процедура, которая хранится вместе с базой данных и вызывается сервером автоматически при возникновении указанных событий модификации данных или системных событий.

**Активная база данных** — база данных, имеющая связанные с ней триггеры.

**Правило ECA (Event-Condition-Action)** — структура триггера: событие (операция INSERT/UPDATE/DELETE), условие (логическое выражение WHEN), действие (процедура).

**ROW-level триггер (FOR EACH ROW)** — триггер, активизирующийся для каждой отдельной строки, затронутой оператором.

**STATEMENT-level триггер (FOR EACH STATEMENT)** — триггер, активизирующийся один раз для данного оператора, рассматриваемого как единое целое. Применяется по умолчанию.

**BEFORE** — триггер выполняется до применения изменений к данным и до проверки ограничений целостности. Может изменять данные NEW ROW перед их записью. Возврат NULL отменяет операцию для текущей строки.

**AFTER** — триггер выполняется после применения изменений и после проверки ограничений (но до COMMIT). Изменения уже видны другим операторам в той же транзакции.

**INSTEAD OF** — система отменяет стандартное действие и выполняет только логику триггера. Применяется исключительно для VIEW, чтобы СУБД знала, как распределить DML по базовым таблицам.

**Событийный триггер (Event Trigger)** — объект, реагирующий на команды DDL (CREATE, DROP, ALTER) на уровне всей базы данных.

**Переменные триггерной функции (PL/pgSQL):**
- `NEW` — RECORD с данными новой строки (INSERT, UPDATE).
- `OLD` — RECORD со старыми данными строки (UPDATE, DELETE).
- `TG_OP` — строка с типом операции: `'INSERT'`, `'UPDATE'`, `'DELETE'`.
- `TG_TABLE_NAME` — имя таблицы, на которой сработал триггер.
- `TG_TAG` — тег команды для событийных триггеров (например, `'DROP TABLE'`).
- `NEW TABLE` / `OLD TABLE` — ссылки на переходные таблицы (transition tables) с наборами всех изменённых строк одного оператора.

## Как устроено

### Алгоритм срабатывания

1. Возникает событие (INSERT/UPDATE/DELETE), попадающее под условие триггера.
2. **BEFORE**: выполняется до применения изменений. Может модифицировать NEW ROW перед записью.
3. **INSTEAD OF**: стандартное действие отменяется, выполняется только логика триггера.
4. **AFTER**: выполняется после применения изменений (но до COMMIT).
5. Если уровень FOR EACH ROW и оператор затрагивает 100 строк — триггер выполнится 100 раз.

### Порядок срабатывания

- Если несколько триггеров назначены на одно событие — они активируются в **алфавитном порядке** имён.
- BEFORE-триггеры срабатывают до проверки ограничений целостности; AFTER-триггеры — после.

### suppress_redundant_updates_trigger (PostgreSQL)

Встроенный триггер, который отменяет UPDATE, если OLD == NEW (значения не изменились). Экономит дисковое пространство, так как иначе UPDATE генерирует новую версию строки (dead tuple) и запись в WAL, даже если ничего не изменилось. Вносит процессорный overhead на каждую строку.

---

## Детали и нюансы

- **Каскадные триггеры**: запуск T1 может активировать T2, который снова активирует T1 (рекурсия) → переполнение стека/памяти СУБД.
- **Порядок срабатывания**: при нескольких триггерах на одно событие — алфавитный порядок. Трудновоспроизводимые "плавающие" баги при сложных цепочках.
- **Скрытый процедурный код**: разработчику сложнее понять логику системы; "простые" INSERT начинают вести себя непредсказуемо.
- **Оптимизатору сложно** анализировать запросы с триггерами.
- **Отложенные ограничения vs триггеры**: декларативные ограничения могут проверяться в конце транзакции (DEFERRED), триггеры обычно срабатывают немедленно после оператора.
- **Триггер может быть отключён** администратором — нет гарантий, какие изменения были пропущены за время его неактивности.
- **Триггеры vs Правила (Rules)**: правила трансформируют дерево запроса (переписывают SQL), эффективны для массовых операций, но сложны в отладке. Триггеры выполняются для каждой строки/оператора.

## Примеры

```sql
-- Полный синтаксис CREATE TRIGGER
CREATE TRIGGER <имя_триггера>
{ BEFORE | AFTER | INSTEAD OF } { INSERT | UPDATE | DELETE [ OR ... ] }
ON <имя_таблицы>
[ REFERENCING { OLD TABLE AS <имя> | NEW TABLE AS <имя> } ]
[ FOR EACH { ROW | STATEMENT } ]
[ WHEN ( <условие> ) ]
EXECUTE FUNCTION <имя_функции> ( <аргументы> );
```

```sql
-- Простой BEFORE ROW триггер
CREATE TRIGGER LONDON_SUPPLIER_INSERT
BEFORE INSERT ON LONDON_SUPPLIER
FOR EACH ROW
EXECUTE FUNCTION insert_to_s();
```

Если INSERT вставляет 100 строк, код триггера выполнится 100 раз (FOR EACH ROW).

```sql
-- Триггерная функция на PL/pgSQL (партиционирование)
CREATE OR REPLACE FUNCTION measurement_insert_trigger() RETURNS TRIGGER AS $$
BEGIN
    IF ( NEW.logdate >= DATE '2006-02-01' AND NEW.logdate < DATE '2006-03-01' ) THEN
        INSERT INTO measurement_y2006m02 VALUES (NEW.*);
    END IF;
    RETURN NULL; -- предотвращает вставку в корневую таблицу
END;
$$ LANGUAGE plpgsql;
```

```sql
-- Event trigger на DDL-событие sql_drop
CREATE EVENT TRIGGER test_event_trigger_for_drops
ON sql_drop
EXECUTE FUNCTION test_event_trigger_for_drops();
```

## Важно / подводные камни / best practices

- **Не используйте триггеры для бизнес-логики и ограничений целостности** — отдавайте предпочтение API, декларативным ограничениям (CHECK, RLS) или слою приложения.
- **Остерегайтесь каскадных триггеров** — рекурсия T1 → T2 → T1 приводит к переполнению памяти.
- **suppress_redundant_updates_trigger** — полезен при массовых обновлениях, экономит WAL и dead tuples ценой CPU overhead.
