## Суть и контекст

Тема охватывает механизмы хранения данных в реляционных СУБД и структуры индексов, ускоряющих поиск. Рассматриваются два основных способа организации хранения — неупорядоченная таблица (heap) и кластеризованная таблица (Index-Organized Table), а также типы индексов: B-Tree, Hash, составной (multicolumn) и покрывающий (covering). Материал актуален для проектирования схем баз данных, оптимизации запросов и понимания внутренней работы СУБД (PostgreSQL, InnoDB, Oracle).

---

## Ключевые идеи

- Индекс ускоряет чтение, но замедляет запись и расходует дополнительную память.
- B-Tree — основной тип индекса; хранит отсортированные значения в листовых страницах, обеспечивая поиск за логарифмическое время (3–4 дисковых скачка).
- Hash индекс хранит 32-битный хеш-код и поддерживает **только** операции точного равенства (`=`).
- Составной индекс определяется на нескольких столбцах; его эффективность зависит от порядка столбцов и типа операторов сравнения.
- Покрывающий индекс содержит все колонки запроса, что позволяет не обращаться к самой таблице (устраняет random I/O из heap).
- Heap-organized table хранит строки без упорядоченности; Index-Organized Table (IOT) хранит все данные таблицы непосредственно в листовых узлах кластеризованного индекса.
- Вторичные индексы на IOT используют значение первичного ключа как указатель, а не физический адрес строки.

## Определения и термины

**Индекс** — структура данных, хранящая информацию для поиска (ключ и значение-указатель), ускоряющая чтение, но замедляющая запись и расходующая память.

**B-Tree индекс** — сбалансированное дерево, в котором столбцы сортируются, а отсортированные значения блоками хранятся в листовых страницах.

**Hash индекс** — индекс, хранящий 32-битный хеш-код, вычисленный из значения индексируемой колонки.

**Составной индекс (Multicolumn / Composite index)** — индекс, определённый более чем на одном столбце таблицы.

**Покрывающий индекс (Covering index)** — индекс, в котором присутствуют все колонки, извлекаемые запросом, что позволяет не обращаться к самой таблице для выборки данных.

**Неупорядоченная таблица (Heap-organized table)** — изменяемая структура данных, где каждая страница хранит несколько строк таблицы без требования их упорядоченности.

**Кластеризованная таблица / Index-Organized Table (IOT)** — структура, в которой кластеризованный индекс хранит все данные таблицы непосредственно в своих листовых узлах.

**GiST (Generalized Search Tree)** — каркас сильно ветвящихся деревьев для создания индексов по геометрическим данным (поиск перекрытий, содержимости и т.д.).

**GIN (Generalized Inverted Index)** — инвертированный индекс для значений с множественными компонентами (массивы, JSON-документы). Содержит отдельную запись для каждого компонента.

**BRIN (Block Range INdex)** — индекс, хранящий минимальные и максимальные значения для последовательных физических блоков (страниц) таблицы. Занимает в тысячи раз меньше места, чем B-Tree.

**Битовый индекс (Bitmap Index)** — структура (Oracle), где в листовых узлах содержится битовая карта. Каждый установленный бит соответствует номеру строки с данным значением ключа. Эффективен на колонках с низкой кардинальностью (пол, статус).

**Частичный индекс (Partial Index)** — индекс, содержащий записи только для строк, удовлетворяющих указанному условию WHERE. СУБД использует его только если условие в запросе логически соответствует предикату индекса.

**Индекс по выражению (Expression Index / Functional Index)** — индекс, построенный на результате вычисления выражения или функции, а не на значении колонки напрямую.

**Селективность (Selectivity)** — доля строк (от 0 до 1), которая остаётся на выходе узла плана после фильтрации. Чем ближе к 0, тем эффективнее индекс. При выборке >25% строк индекс обычно игнорируется.

**Кардинальность (Cardinality)** — мера уникальности данных: отношение количества уникальных значений к общему числу строк. Высокая кардинальность (ИНН, UUID) — идеальна для B-Tree; низкая (пол, статус) — для Bitmap.

**Фактор кластеризации (Clustering factor)** — метрика, показывающая насколько близко друг к другу в физической таблице находятся строки с одинаковым значением индексированного поля.

**TID (Tuple Identifier)** — физический адрес версии строки в heap, состоящий из номера страницы и номера строки внутри неё.

**Карта видимости (Visibility Map, vm)** — файл с суффиксом `_vm`, хранящий по 2 бита на страницу: признак видимости всеми транзакциями (all-visible) и признак заморозки (all-frozen). Используется при Index-Only Scan.

**HOT-update (Heap-Only Tuple)** — оптимизация, при которой UPDATE не создаёт новую запись в индексе, если изменённые столбцы не входят в индекс и на странице есть место для новой версии строки.

**SP-GiST (Space-Partitioned Generalized Search Tree)** — метод доступа для несбалансированных структур (префиксные деревья, квадродеревья). Эффективен для неперекрывающихся областей данных.

## Как устроено

### Структура B-Tree

B-Tree состоит из трёх уровней компонентов:

1. **Корневой узел (Root page)** — верхняя точка дерева.
2. **Промежуточные страницы** — содержат только ссылки на диапазоны ключей.
3. **Листовые страницы (Leaf)** — хранят ключи и физические адреса строк.

Корневая и промежуточные страницы состоят только из ссылок на диапазоны ключей. Листовые узлы образуют **двунаправленный список** для перебора диапазонов. Высота дерева обычно составляет 3 или 4 уровня.

B-Tree по сути представляет собой ассоциативный массив ключ-значение (ID -> физический адрес строки, 3–4 дисковых скачка).

### Алгоритм поиска в B-Tree

1. Считывание корневой страницы, определение нужной ссылки на основе сравнения с интервалами.
2. Переход по ссылке к следующей странице.
3. Достижение листовой страницы, нахождение ключа и извлечение физического адреса строки.

Поиск занимает логарифмическое время, обходя H страниц (H = высота дерева).

### Алгоритм вставки в B-Tree

1. Спуск до листового узла.
2. Если есть свободное пространство — ключ вставляется с сохранением сортировки.
3. Если узел заполнен — происходит **split**: половина ключей остаётся в старом узле, половина переносится в новый. Ключ разделения поднимается в родительский узел. Деление может каскадно дойти до корня.

### Структура Hash индекса

Компоненты:

- **Мета-страница (meta)** — хранит метаданные индекса.
- **Страницы корзин (buckets)** — основные контейнеры для хеш-значений.
- **Страницы переполнения (overflow pages)** — используются при переполнении корзин.

На уровне корзины используются Share/Exclusive блокировки (hash-bucket-level locks). Hash индекс поддерживает **только** операции точного равенства (`=`).

### Физическая организация страницы

Размер страницы в PostgreSQL по умолчанию — **8 Кбайт**. Структура (сверху вниз):

1. **Заголовок (Header / Metadata)** — метаданные страницы.
2. **Массив указателей (Line Pointers)** — массив ссылок на версии строк.
3. **Свободное пространство (Free Space)** — находится между указателями и данными.
4. **Версии строк (Tuples)** — сами данные, хранятся снизу вверх.
5. **Специальная область (Special)** — для вспомогательной информации индексов; в табличных страницах обычно нулевого размера.

**TID** строки = (номер страницы, номер строки в массиве указателей). Физический адрес — 6 байт.

### Heap-хранение

- Строки записываются в страницы со свободным пространством через freelist или bitmap карты заполнения.
- Порядок строк — произвольный (heap = куча).
- СУБД ищет свободное место через **Free Space Map (fsm)** — дерево-структура для быстрого поиска страниц с нужным объёмом свободного места.

### Index-Only Scan и Visibility Map

**Index-Only Scan** позволяет вернуть данные прямо из индекса без обращения к heap:
1. Все запрашиваемые столбцы должны быть в индексе (покрывающий индекс).
2. Индекс должен поддерживать свойство RETURNABLE.
3. СУБД проверяет **Visibility Map**: если бит all-visible для страницы установлен — обращение к heap не требуется. Если нет — нужна проверка по heap для MVCC-снимка.

### Bitmap Index Scan

Двухэтапный метод для сочетания случайного I/O индекса с последовательным чтением heap:

1. **Этап 1 (Index Scan)**: из индекса извлекаются все TID, строится битовая карта страниц в памяти.
2. **Этап 2 (Heap Scan)**: страницы таблицы читаются строго по порядку номеров, каждая — только один раз.

**Особенности**:
- Позволяет комбинировать несколько индексов через `AND`/`OR`.
- При нехватке `work_mem` карта становится **lossy** (грубой) — тогда каждую строку на странице нужно перепроверять (recheck).

### Heap vs IOT (Index-Organized Table)

- В IOT (InnoDB, Oracle IOT) первичный ключ = ключ B-дерева, в листах хранятся все поля таблицы.
- В Heap лист B-Tree содержит ссылку в кучу (физический адрес).
- В IOT данные — payload в листах B-Tree.
- Вторичные индексы: в Heap — физический адрес строки, в IOT — значение первичного ключа.

## Детали и нюансы

### Составной индекс и неравенства

Составной индекс эффективен **до первого оператора неравенства включительно**. Пример: `WHERE A=1 AND B>2 AND C=3` — часть по `C` не используется индексом.

### Hash индекс и range-запросы

Hash индекс игнорируется для range-запросов. СУБД выполняет Sequential Scan (Seq Scan) вместо использования Hash индекса.

### Низкая селективность

При низкой селективности (малая кардинальность столбца) оптимизатор игнорирует B-Tree индекс и сканирует таблицу целиком, так как Sequential Scan выгоднее random I/O.

### Функция от колонки

Обычный индекс по колонке **не используется** для выражений с функциями, например `trunc(date)`. Необходим функциональный индекс.

### HOT-updates (Heap-Only Tuples)

Если UPDATE меняет столбцы, **не входящие** в индексы, и на странице есть свободное место (обеспечивается `FILLFACTOR`), PostgreSQL применяет HOT:
- Новая версия строки записывается на ту же страницу.
- Записи в индексе **не обновляются** — они продолжают указывать на старую версию, от которой идёт цепочка к новой.
- WA для индексов = 0.

При нехватке места на странице HOT невозможен — в индексе создаётся новая полноценная запись.

### Монотонно возрастающие PK в highload

Монотонно возрастающие первичные ключи в highload-системах создают contention за правый листовой узел B-Tree.

### Почему индекс может НЕ использоваться

1. **Низкая селективность** — запрос возвращает >25% строк → Sequential Scan выгоднее random I/O.
2. **Функция от индексированной колонки** — `WHERE trunc(date) = ...` → обычный индекс по `date` не применим. Нужен функциональный индекс.
3. **Неполный префикс составного индекса** — при индексе (A, B, C) запрос `WHERE B = 5` не может эффективно использовать индекс.
4. **Маленькая таблица (1–2 страницы)** — Random I/O по индексу дороже полного Seq Scan.
5. **Устаревшая статистика** — планировщик неверно оценивает кардинальность и пропускает индекс.

### Что хуже: много индексов или их отсутствие

- **Отсутствие индексов** → замедление выборки (O(N)), но максимальная скорость записи.
- **Множество индексов** → экспоненциальное замедление INSERT, UPDATE, DELETE (перестроение деревьев, page splits). Расход дискового пространства.
- Каждое обновление (UPDATE) колонки, входящей в индекс, вызывает синхронное обновление всех связанных B-деревьев.
- Лишние индексы «на всякий случай» — антипаттерн.

### Периодическое перестроение индексов (REINDEX)

Свежесозданный индекс работает быстрее — логически смежные страницы физически рядом на диске. Со временем UPDATE/DELETE порождают фрагментацию и разбухание (bloat) узлов. Периодический REINDEX восстанавливает плотность.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| B-Tree индекс | Поддерживает равенство, диапазоны, сортировку; логарифмическое время поиска (3–4 уровня) | Замедляет запись; split при заполнении узла; contention на правом листе при монотонном PK | Основной тип индекса для большинства запросов |
| Hash индекс | Быстрый точный поиск (`=`); блокировки на уровне корзины | Только операции равенства; не поддерживает range-запросы | Поиск по точному совпадению |
| Heap-organized table | Простая вставка (в любую страницу со свободным местом); физический адрес в индексах | Random I/O при чтении через индекс | Стандартное хранение (PostgreSQL, Oracle heap tables) |
| Index-Organized Table (IOT) | Данные уже в листах B-Tree — нет дополнительного I/O по PK | Вторичные индексы используют PK как указатель (менее эффективно); вставки дороже из-за поддержания порядка | InnoDB (MySQL), Oracle IOT; запросы по первичному ключу |
| Покрывающий индекс | Устраняет random I/O из heap; все данные из индекса | Расходует память; замедляет запись при большом числе колонок в индексе | Аналитические системы; частые SELECT-запросы по конкретным колонкам |
| SP-GiST | Эффективен для несбалансированных структур (префиксные деревья, квадродеревья) | Только неперекрывающиеся данные | IP-диапазоны, точки в квадрантах |
| Bitmap Index Scan | Сочетает random I/O индекса с последовательным чтением heap; можно комбинировать индексы | При нехватке work_mem — lossy-режим и recheck | Запросы с несколькими условиями по разным индексам |

## Примеры

### Создание Hash индекса

```sql
-- Hash индекс
CREATE INDEX name ON table USING HASH (column);
```

### Составной индекс

```sql
-- Составной индекс
CREATE TABLE test2 ( major int, minor int, name varchar );
SELECT name FROM test2 WHERE major = constant AND minor = constant;
```

### Покрывающий индекс (INCLUDE)

```sql
-- Покрывающий индекс: x — ключ поиска, y — полезная нагрузка в листе
CREATE INDEX tab_x_y ON tab(x) INCLUDE (y);
-- Запрос SELECT y FROM tab WHERE x = 'key' берёт y прямо из индекса (Index-Only Scan)
```

### Частичный индекс (Partial Index)

```sql
-- Индексирует только неоплаченные заказы (~2% данных)
CREATE INDEX orders_unbilled_index ON orders (order_nr) WHERE billed is not true;
```

### Индекс по выражению + GIN для JSON

```sql
-- GIN индекс по JSON-полю
CREATE INDEX idxgintags ON api USING GIN ((jdoc -> 'tags'));
-- Позволяет оптимизатору использовать индекс для WHERE jdoc -> 'tags' ? 'qui'
```

### Edge case: составной индекс при неравенствах

Запрос `WHERE A=1 AND B>2 AND C=3` — индекс используется только для колонок `A` и `B`; условие по `C` не покрывается индексом.

## Важно / подводные камни / best practices

- **Составные индексы:** условия с `=` помещать в начало, единственное неравенство — в конец.
- **Покрывающий индекс:** добавлять SELECT-колонки в индекс для аналитических систем (устраняет random I/O из heap).
- **Антипаттерн:** B-Tree на колонках с низкой кардинальностью (пол, статус) — использовать bitmap indexes.
- **Антипаттерн:** Монотонно возрастающие PK в highload — contention за правый листовой узел. Использовать REVERSE индекс.
- **Функциональные индексы:** обычный индекс по колонке не используется для выражений с функциями (например, `trunc(date)`). Нужен функциональный индекс.
- **Hash индекс:** бесполезен для range-запросов — СУБД откатывается к Seq Scan.

Как связаны листовые узлы B-Tree между собой?::Образуют двунаправленный список для перебора диапазонов

Какова типичная высота B-Tree?::3 или 4 уровня

Алгоритм поиска в B-Tree
?
1. Считывание корневой страницы, определение нужной ссылки на основе сравнения с интервалами
2. Переход по ссылке к следующей странице
3. Достижение листовой страницы, нахождение ключа и извлечение физического адреса строки

Сложность поиска в B-Tree — ==логарифмическое время (H страниц, H = высота дерева)==

Алгоритм вставки в B-Tree
?
1. Спуск до листового узла
2. Если есть свободное пространство — ключ вставляется с сортировкой
3. Если узел заполнен — split: половина ключей в старом узле, половина в новом; ключ разделения поднимается в родительский узел; деление может каскадно дойти до корня

Что происходит при вставке в заполненный узел B-Tree?::Split: половина ключей остаётся в старом узле, половина переносится в новый. Ключ разделения поднимается в родительский узел. Деление может каскадно дойти до корня.

Из каких компонентов состоит Hash индекс?::Мета-страница (meta), страницы корзин (buckets), страницы переполнения (overflow pages)

Какие блокировки используются в Hash индексе?::Share/Exclusive блокировки на уровне корзины (hash-bucket-level locks)

Какие операции поддерживает Hash индекс?::Только операции точного равенства (=)

Как создать Hash индекс в PostgreSQL?::CREATE INDEX name ON table USING HASH (column);

Как организована страница в Heap-хранении?
?
1. В начале страницы — метаданные
2. Затем заголовки строк
3. Данные строк — в конце страницы
4. Свободное пространство — между заголовками и данными

Куда записываются новые строки в Heap?::В страницы со свободным пространством (freelist или bitmaps)

В чём отличие Heap от IOT?
?
Heap: лист B-Tree содержит ссылку (физический адрес) в кучу; данные хранятся отдельно
IOT: данные — payload в листах B-Tree; первичный ключ = ключ B-дерева

Что используют вторичные индексы как указатель в Heap?::Физический адрес строки

Что используют вторичные индексы как указатель в IOT?::Значение первичного ключа

Edge case: что происходит при составном индексе с неравенствами (WHERE A=1 AND B>2 AND C=3)?::Индекс эффективен до первого оператора неравенства включительно — часть по C не используется

Edge case: что происходит при range-запросе на Hash индексе?::Hash индекс игнорируется, СУБД делает Seq Scan

Edge case: что происходит при низкой селективности индекса?::Оптимизатор игнорирует B-Tree, сканирует целиком (Sequential Scan выгоднее random I/O)

Edge case: используется ли обычный индекс по колонке для trunc(date)?::Нет, обычный индекс не используется для выражений с функциями. Нужен функциональный индекс.

Как правильно упорядочить колонки в составном индексе?::Условия с = в начало, единственное неравенство — в конец

Зачем использовать покрывающий индекс в аналитических системах?::Добавление SELECT-колонок в индекс устраняет random I/O из heap

Почему B-Tree на колонках с низкой кардинальностью — антипаттерн?::При низкой кардинальности (пол, статус) B-Tree неэффективен; следует использовать bitmap indexes

Почему монотонно возрастающий PK — антипаттерн в highload?::Создаёт contention за правый листовой узел B-Tree. Решение — использовать REVERSE индекс.

GiST (Generalized Search Tree):::Каркас сильно ветвящихся деревьев для индексов по геометрическим данным (перекрытия, содержимость).

GIN (Generalized Inverted Index):::Инвертированный индекс для значений с множественными компонентами (массивы, JSON). Отдельная запись для каждого компонента.

BRIN (Block Range INdex):::Индекс, хранящий min/max значения для последовательных физических блоков таблицы. Занимает в тысячи раз меньше места, чем B-Tree.

Битовый индекс (Bitmap Index):::Структура (Oracle), где листовые узлы содержат битовую карту — каждый бит соответствует строке с данным значением. Эффективен на колонках с низкой кардинальностью.

Селективность (Selectivity):::Количество строк, возвращаемое запросом. Чем меньше строк, тем эффективнее индекс. При >25% строк индекс обычно игнорируется.

Фактор кластеризации (Clustering factor):::Метрика, показывающая насколько близко в физической таблице находятся строки с одинаковым значением индексированного поля.

Частичный индекс (Partial Index):::Индекс, содержащий записи только для строк, удовлетворяющих WHERE-условию. Радикально уменьшает размер индекса.

Индекс по выражению (Expression Index):::Индекс, построенный на результате вычисления выражения или функции, а не на значении колонки напрямую.

Когда BRIN эффективнее B-Tree?::Для колонок time series (логи, датчики), где время монотонно возрастает с физическим добавлением строк.

B-Tree эффективен на колонках с ==высокой== кардинальностью, Bitmap — с ==низкой==

TID (Tuple Identifier):::Физический адрес версии строки: номер страницы + номер строки внутри неё. 6 байт.

Карта видимости (Visibility Map):::Файл с суффиксом _vm; 2 бита на страницу: all-visible (видима всем транзакциям) и all-frozen. Используется для Index-Only Scan без обращения к heap.

HOT-update (Heap-Only Tuple):::Оптимизация: UPDATE не создаёт запись в индексе, если изменённые столбцы не входят в индекс и на странице есть свободное место. WA для индексов = 0.

Этапы Bitmap Index Scan
?
1. Этап 1 (Index Scan): из индекса извлекаются все TID, строится битовая карта страниц в памяти
2. Этап 2 (Heap Scan): страницы читаются строго по порядку номеров, каждая — один раз

Что происходит при нехватке work_mem в Bitmap Index Scan?::Карта становится lossy (грубой) — каждую строку на странице нужно перепроверять (recheck)

SP-GiST (Space-Partitioned GiST):::Метод доступа для несбалансированных структур (префиксные деревья, квадродеревья). Эффективен для неперекрывающихся областей.

Структура страницы heap (сверху вниз)
?
1. Заголовок (Header)
2. Массив указателей (Line Pointers)
3. Свободное пространство
4. Версии строк (Tuples) — снизу вверх
5. Специальная область (Special)

Почему индекс может НЕ использоваться?
?
1. Низкая селективность (>25% строк) → Seq Scan выгоднее
2. Функция от колонки → нужен функциональный индекс
3. Неполный префикс составного индекса
4. Маленькая таблица (1–2 страницы)
5. Устаревшая статистика

Что хуже: много индексов или их отсутствие?::Отсутствие → медленная выборка. Много → экспоненциальное замедление записи (перестроение деревьев, page splits), расход диска. Лишние индексы «на всякий случай» — антипаттерн.

Hash индекс хранит ==32-битный== хеш-код

B-Tree можно рассматривать как ==ассоциативный массив ключ-значение (ID -> физический адрес строки)==
