## Суть и контекст

Конспект описывает механизмы СУБД (на примере PostgreSQL), позволяющие изменять структуру таблиц и обслуживать индексы без остановки сервиса. Темы: поведение DDL при высокой нагрузке (lock contention cascade), опция `CONCURRENTLY` для построения и перестроения индексов, миграция через логическую репликацию, ограничения встроенной логической репликации и хранимого кода.

Источники фокусируются на механизмах СУБД, а не на паттернах оркестрации приложений: конкретные пошаговые стратегии Expand and Contract, Blue-Green deployment для схем БД и Shadow writes в предоставленных выдержках не описаны.

---

## Ключевые идеи

- Любой DDL требует сильной блокировки (вплоть до `ACCESS EXCLUSIVE`); пока он ждёт в очереди, за ним встают и все новые запросы, включая простые `SELECT` → каскадная остановка обслуживания таблицы.
- Опция `CONCURRENTLY` понижает требуемый уровень блокировки до `SHARE UPDATE EXCLUSIVE`, позволяя `INSERT`/`UPDATE`/`DELETE` продолжаться во время построения/перестроения индекса.
- Построение индекса в режиме `CONCURRENTLY` выполняется в два прохода по таблице и ожидает завершения всех транзакций, которые могут «видеть» таблицу, ради согласованности.
- Миграция через логическую репликацию: новая база (возможно, на новой версии СУБД) принимает поток изменений от старой; реплика доступна на запись, что позволяет заранее подготовить индексы или изменить структуру; приложение переключается после синхронизации.
- Встроенная логическая репликация PostgreSQL **не** реплицирует DDL: схемы на источнике и приёмнике приходится поддерживать синхронно вручную.
- Канареечные (rolling) релизы несовместимы с хранимым кодом в БД: подсистемы хранения и обработки жёстко связаны внутри одного экземпляра СУБД.
- Антипаттерн: выполнять DDL в периоды высокой нагрузки — даже «быстрая» операция может надолго повиснуть в очереди за блокировкой.

## Определения и термины

**DDL (Data Definition Language)** — команды изменения структуры БД (`ALTER TABLE`, `CREATE INDEX`, `DROP TABLE` и т.п.), которые в общем случае требуют сильной блокировки таблицы.

**Lock contention cascade («лавинообразная остановка обслуживания таблицы»)** — ситуация, когда DDL-команда ждёт свою блокировку, и все новые входящие запросы (даже простые `SELECT`) встают в очередь **после** неё, парализуя работу с таблицей.

**CONCURRENTLY** — опция PostgreSQL для `CREATE INDEX` и `REINDEX`, при которой СУБД запрашивает более слабый уровень блокировки — `SHARE UPDATE EXCLUSIVE` — вместо стандартного `ACCESS EXCLUSIVE`.

**Логическая репликация (миграционный сценарий)** — способ миграции, при котором создаётся новая база (возможно на новой версии СУБД), настраивается логическая репликация от старой к новой; реплика доступна на запись, что позволяет заранее подготовить индексы и изменения структуры; после синхронизации приложение переключается на новую базу.

**Канареечный (rolling) релиз** — стратегия, при которой часть серверов приложений работает со старой версией кода, а часть — с новой. Несовместима с наличием хранимого кода в БД, так как подсистемы хранения и обработки жёстко связаны в одном экземпляре СУБД.

**lock_timeout** — параметр PostgreSQL, ограничивающий время ожидания блокировки. Если DDL-команда не может получить блокировку за указанное время — прерывается с ошибкой. Предотвращает lock contention cascade.

**Bloat (раздувание)** — накопление мёртвых версий строк в страницах таблицы из-за модели MVCC, если VACUUM не успевает их удалять. Страницы остаются полупустыми, файл данных растёт, Seq Scan замедляется.

**pg_repack** — расширение для онлайн-перестройки таблицы и индексов практически без прерывания обслуживания. Создаёт временную таблицу, переносит данные, изменения фиксируются триггерами. Эксклюзивная блокировка нужна только в самом начале и конце операции.

**pgcompacttable** — утилита, перемещающая актуальные версии строк в начало файла через фиктивные UPDATE, позволяя VACUUM постепенно усекать файл с конца без монопольных блокировок.

**VACUUM FULL** — полная перестройка таблицы в новый файл с максимально плотной упаковкой данных. Требует `ACCESS EXCLUSIVE` и свободное место на диске, равное размеру таблицы.

**pg_stat_user_tables** — системное представление: `n_dead_tup` (мёртвые строки), `n_ins_since_vacuum`, время последних очисток.

**pgstattuple** — расширение для точной оценки плотности данных. `tuple_percent` — доля полезных данных; `free_percent` — доля свободного места.

**pg_stat_io** (PostgreSQL 16+) — детальная статистика ввода-вывода с контекстами vacuum и bulkread.

## Как устроено

### Поведение DDL при высокой нагрузке

1. DDL-команда запрашивает сильную блокировку (обычно `ACCESS EXCLUSIVE`) и встаёт в очередь, ожидая завершения уже открытых транзакций.
2. Пока DDL-команда ожидает в очереди, все новые поступающие запросы (даже простые `SELECT`) встают в очередь **после** неё.
3. Это приводит к лавинообразной остановке обслуживания таблицы (lock contention).
4. Даже если сама DDL-операция (например, добавление колонки) выполняется быстро, длительное ожидание в очереди блокирует все остальные запросы и вызывает простой всей системы.

### Создание индекса в режиме CONCURRENTLY

Алгоритм:

1. СУБД запрашивает более слабый уровень блокировки — `SHARE UPDATE EXCLUSIVE` — вместо `ACCESS EXCLUSIVE`.
2. Система выполняет **два прохода** по таблице для построения индекса.
3. Процесс ожидает завершения всех транзакций, которые могут «видеть» таблицу, чтобы гарантировать согласованность.
4. На всём протяжении операции `INSERT`, `UPDATE` и `DELETE` продолжают работать.

### Перестроение индекса в фоновом режиме (REINDEX CONCURRENTLY)

- Требует только блокировки `SHARE UPDATE EXCLUSIVE`.
- Обычный `REINDEX` (без `CONCURRENTLY`) требует `ACCESS EXCLUSIVE`.

### Миграция через логическую репликацию

Алгоритм:

1. Создаётся новая база данных (возможно, на новой версии СУБД).
2. Настраивается логическая репликация данных от старой базы к новой.
3. Поскольку логическая репликация позволяет реплике быть доступной на запись, в ней можно заранее подготовить индексы или изменить структуру.
4. После синхронизации данных приложение переключается на новую базу.

## Детали и нюансы

- **Встроенная логическая репликация PostgreSQL не поддерживает автоматическую репликацию DDL** (таких как `ALTER TABLE`). Схемы на источнике и приёмнике должны поддерживаться в синхронном состоянии вручную.
- **Канареечные релизы и хранимый код в БД**: если бизнес-логика находится внутри БД (процедуры), «канареечный» выкат серверов приложений невозможен — подсистемы хранения и обработки жёстко связаны в рамках одного экземпляра СУБД.
- **`ALTER TABLE ... ADD COLUMN DEFAULT`**: все существующие строки расширяются, значение новой колонки устанавливается в указанное значение по умолчанию.
- **Информация, не найденная в источниках**: конкретные пошаговые стратегии Expand and Contract, Blue-Green deployment для схем БД, Shadow writes как стратегии миграции прикладного уровня, а также использование параметра `lock_timeout` для безопасных миграций в предоставленных выдержках не описаны.

## Сравнения и trade-offs

| Подход | Уровень блокировки | Что продолжает работать | Минусы | Когда использовать |
|--------|--------------------|--------------------------|--------|--------------------|
| `CREATE INDEX` (обычный) | `ACCESS EXCLUSIVE` | Только `SELECT` (в ряде случаев) | Длительный простой пишущих операций | Обслуживание в окно простоя |
| `CREATE INDEX CONCURRENTLY` | `SHARE UPDATE EXCLUSIVE` | `INSERT`, `UPDATE`, `DELETE` | Два прохода по таблице; ожидание транзакций, видящих таблицу | Построение индекса под нагрузкой |
| `REINDEX` (обычный) | `ACCESS EXCLUSIVE` | — | Полная блокировка таблицы | Только в окна обслуживания |
| `REINDEX INDEX CONCURRENTLY` | `SHARE UPDATE EXCLUSIVE` | `INSERT`, `UPDATE`, `DELETE` | Тот же порядок ограничений, что и у `CREATE INDEX CONCURRENTLY` | Фоновое перестроение индекса |
| Миграция через логическую репликацию | — (на стороне источника без блокировок DDL) | Рабочая нагрузка на старой базе до переключения | DDL не реплицируется автоматически; схемы нужно синхронизировать вручную | Переход на новую версию СУБД / существенное изменение структуры |

## Примеры

```sql
-- Создание индекса без блокировки пишущих транзакций
CREATE INDEX CONCURRENTLY idx_name ON table_name (column_name);
```

```sql
-- Перестроение индекса в фоновом режиме
REINDEX INDEX CONCURRENTLY index_name;
```

```sql
-- Добавление колонки со значением по умолчанию
ALTER TABLE S ADD COLUMN DISCOUNT INTEGER DEFAULT -1 ;
```

## Важно / подводные камни / best practices

- **Антипаттерн**: выполнять DDL-операции в периоды высокой нагрузки. Даже быстрая операция (например, добавление колонки) может надолго зависнуть в очереди на блокировку, блокируя все остальные запросы и вызывая простой всей системы.
- Для построения/перестроения индексов под нагрузкой использовать `CREATE INDEX CONCURRENTLY` / `REINDEX INDEX CONCURRENTLY` — они понижают уровень блокировки до `SHARE UPDATE EXCLUSIVE`.
- При миграции через логическую репликацию помнить, что встроенная логическая репликация PostgreSQL не реплицирует DDL: схемы на источнике и приёмнике должны поддерживаться синхронно вручную.
- Если бизнес-логика находится в хранимых процедурах БД, канареечные релизы прикладного уровня не применимы.

В чём отличие `CREATE INDEX` от `CREATE INDEX CONCURRENTLY`?
?
Обычный: требует `ACCESS EXCLUSIVE` — блокирует запись на время построения.
CONCURRENTLY: требует `SHARE UPDATE EXCLUSIVE` — `INSERT`/`UPDATE`/`DELETE` продолжают работать; но выполняется в два прохода и ждёт завершения транзакций, видящих таблицу.

В чём отличие `REINDEX` от `REINDEX INDEX CONCURRENTLY`?
?
Обычный: `ACCESS EXCLUSIVE` — полная блокировка таблицы.
CONCURRENTLY: `SHARE UPDATE EXCLUSIVE` — параллельные DML продолжают работать.
