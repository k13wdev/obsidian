# Event-driven архитектура, backpressure и сравнение RabbitMQ vs Kafka

## Суть и контекст

Конспект охватывает Event-Driven Architecture (EDA) и связанные паттерны (CQRS, SEDA, Transactional Outbox), механизмы backpressure и fault tolerance в системах обмена сообщениями, observability/мониторинг, а также детальное сравнение архитектурных подходов RabbitMQ и Kafka. Эти темы определяют выбор брокера и архитектурные решения при проектировании распределённых систем.

---

## Ключевые идеи

- Event-Driven Architecture (EDA) — парадигма, обеспечивающая слабую связность (loose coupling), изоляцию сбоев и независимое масштабирование
- CQRS разделяет команды (запись) и запросы (чтение); Kafka выступает event-sourced журналом для питания read-проекций
- SEDA — конвейеризация обработки через топики, каждая стадия масштабируется независимо
- Transactional Outbox решает проблему двойной записи (БД + Kafka) через промежуточную таблицу
- Backpressure: RabbitMQ — push-модель с QoS prefetch + TCP backpressure; Kafka — pull-модель (backpressure «из коробки»)
- Fault tolerance: Kafka — репликация (RF, ISR, acks=all); RabbitMQ — HA Queues, Federation, Shovel
- RabbitMQ = «Smart Broker, Dumb Consumer» (Push, удаление после ACK, сложная маршрутизация)
- Kafka = «Dumb Broker, Smart Consumer» (Pull, append-only лог, replay, высокая пропускная способность)
- Observability: consumer lag — ключевая метрика Kafka; UnderReplicatedPartitions — барометр здоровья кластера

## Определения и термины

**Event-Driven Architecture (EDA)** — парадигма, способствующая созданию, обнаружению, потреблению и реакции на события (существенные изменения состояния). Эмиттер события не знает о существовании потребителей ниже по потоку. Обеспечивает push-based коммуникацию и крайне слабосвязанную (extremely loosely-coupled) архитектуру.

**CQRS (Command-Query Responsibility Segregation)** — архитектурный паттерн, разделяющий действия, изменяющие состояние (Command), от действий, запрашивающих состояние (Query). Базируется на event-sourced журналах (Kafka) для раздельного питания оптимизированных для чтения хранилищ (view projections).

**SEDA (Staged Event-Driven Architecture)** — применение принципа конвейеризации (pipelining) к событийно-ориентированным данным. События передаются однонаправленно через серию стадий обработки, связанных топиками. Каждая стадия масштабируется автономно и независимо.

**Transactional Outbox (Outbox Pattern)** — паттерн, при котором микросервис публикует сообщение в специальную таблицу БД («outbox»), а отдельный сервис-ретранслятор (relay service) считывает событие и обновляет целевую систему (Kafka). Решает проблему двойной записи.

**«Smart Broker, Dumb Consumer»** — модель работы RabbitMQ: брокер следит за статусом сообщений, удаляет их из очереди после прочтения и сам организует процесс распределения данных (Push) между подписчиками.

**«Dumb Broker, Smart Consumer»** — модель работы Kafka: брокер выступает только хранилищем неизменяемого лога коммитов, а клиенты-потребители (Pull) сами запрашивают данные и отслеживают свои смещения (offsets).

**Backpressure (обратное давление)** — механизм защиты получателя от переполнения данными.

**QoS / prefetch_count** — ограничение на количество предварительно выбранных и неподтвержденных (unacknowledged) сообщений, отправленных RabbitMQ консьюмеру. Действует как предохранитель (circuit breaker).

**TCP Backpressure** — механизм RabbitMQ для остановки приёма данных на уровне TCP-сокета от продюсеров при перегрузке брокера. Заменил устаревший `Channel.Flow`.

**Consumer Lag** — разница между последним сообщением в партиции брокера и последним смещением, зафиксированным консьюмер-группой.

**Under-Replicated Partitions (URP)** — метрика Kafka: количество партиций, для которых ведомые реплики отстают и не синхронизированы. Значение > 0 указывает на отказ брокеров или исчерпание ресурсов.

**Federation** — плагин RabbitMQ для прозрачной передачи сообщений между узлами и кластерами, устойчивый к высоким задержкам (WAN) и разделениям сети.

**Burrow** — инструмент от LinkedIn для мониторинга consumer lag в Kafka, определяющий статус консьюмер-группы без статических порогов.

## Как устроено

### Event-Driven Architecture — изоляция сбоев

1. Сервис А пишет события в топик
2. Сервис Б (потребитель) выходит из строя
3. Сервис А продолжает работу, накапливая сообщения в брокере
4. После восстановления Сервис Б вычитывает накопившиеся сообщения
5. Сбой локализован — каскадного отказа не происходит

### CQRS в связке с Kafka

1. Клиент отправляет команду в блок Command Handling
2. Command Handling валидирует команду и записывает изменения в Master Data (БД)
3. Генерируется событие и публикуется в Kafka Topic
4. Query Handling (потребитель) вычитывает события из Topic
5. Query Handling обновляет View Projection (хранилище, оптимизированное под чтение)
6. Клиенты отправляют запросы на чтение только к View Projection

### SEDA — конвейер через топики

1. Данные из исходного топика поступают на стадию обработки (фильтрация, трансформация)
2. Промежуточная стадия выступает одновременно потребителем и продюсером
3. Преобразованное событие публикуется в следующий топик конвейера
4. Поддерживаются топологии **fan-in** (чтение из нескольких топиков, аналог SQL JOIN) и **fan-out** (публикация в несколько топиков)

### Transactional Outbox

**Проблема двойной записи (Dual-write):** микросервису нужно обновить БД и отправить сообщение в Kafka. Kafka Transactions не поддерживают атомарность с внешними СУБД. При падении после записи в БД, но до отправки в Kafka — неконсистентность.

**Решение:**
1. Приложение в единой SQL-транзакции записывает бизнес-данные в основную таблицу и сообщение в таблицу Outbox
2. Транзакция СУБД фиксируется (commit)
3. Отдельный Worker-процесс (или Kafka Connect с Debezium CDC) опрашивает или отслеживает изменения таблицы Outbox
4. Worker публикует события в Kafka
5. После подтверждения от Kafka Worker помечает запись как отправленную (или удаляет)

### Backpressure — RabbitMQ

**QoS (Consumer Backpressure):**
1. Консьюмер отправляет `Basic.QoS` с указанием `prefetch_count`
2. RabbitMQ выталкивает (push) сообщения до достижения лимита неподтверждённых
3. Окно смещается только при получении `Basic.Ack`

**TCP Backpressure (Producer):**
1. Брокер отслеживает потребление памяти и диска
2. При достижении `vm_memory_high_watermark` (0.4 от RAM) или `disk_free_limit` (50 MB) — генерируется alarm
3. RabbitMQ перестаёт читать из TCP-сокета продюсера
4. Брокер отправляет асинхронный AMQP-фрейм `Connection.Blocked`

### Backpressure — Kafka

1. Pull-модель: брокер не навязывает данные, консьюмер запрашивает через `poll()`
2. `max.poll.records` — лимит записей на один вызов poll
3. `fetch.min.bytes` — брокер ждёт накопления байт перед ответом
4. `fetch.max.wait.ms` — брокер отвечает по таймауту, если данных мало
5. На стороне продюсера: брокер задерживает ACK при нехватке ресурсов

### Fault Tolerance — Kafka

1. Топик настраивается с `replication.factor` (обычно 3)
2. Один брокер — Leader для партиции, остальные — Followers, пассивно вытягивают данные
3. Фолловеры, успевающие за лидером — In-Sync Replicas (ISR)
4. Продюсер использует `acks=all`: лидер не подтверждает, пока все ISR не сохранят
5. При сбое лидера Controller назначает нового из ISR
6. Если `unclean.leader.election.enable=true` — лидером может стать отставшая реплика не из ISR (потеря данных)

### Fault Tolerance — RabbitMQ

1. Узлы кластеризуются через Erlang IPC по TCP/IP
2. Очередь декларируется с `x-ha-policy: all` (или `nodes`) — Highly Available Queue
3. Сообщения зеркалируются на узлы кластера
4. При падении узла с основной очередью — прозрачный выбор нового зеркала
5. Federation — асинхронная передача данных в удалённый кластер для DR (устойчива к WAN-задержкам)

### Observability — ключевые метрики

**Kafka:**
- `kafka.server:type=ReplicaManager,name=UnderReplicatedPartitions` — должно быть 0
- `kafka.controller:type=KafkaController,name=ActiveControllerCount` — строго 1
- `kafka.server:type=BrokerTopicMetrics,name=BytesInPerSec / BytesOutPerSec` — интенсивность трафика
- `kafka.network:type=RequestMetrics,name=TotalTimeMs,request=Fetch/Produce` — задержка запросов
- Consumer Lag — через `kafka-consumer-groups.sh` или Burrow

**RabbitMQ:**
- `messages_unacknowledged` — неподтверждённые сообщения
- `messages_uncommitted`, `acks_uncommitted` — метрики транзакций
- `vm_memory_high_watermark`, `file_descriptors` — ресурсы

**Инструменты:**
- Kafka: JMX → kafka_exporter → Prometheus → Grafana
- RabbitMQ: `rabbitmqctl status`, HTTP Management API, Nagios, Munin, Zabbix

## Детали и нюансы

- **Unclean Leader Election + все ISR упали**: лидером становится out-of-sync реплика, происходит безвозвратная потеря данных, но кластер возобновляет приём запросов
- **Мониторинг Kafka самой Kafka**: если кластер падает, система мониторинга лишается возможности узнать об этом. Алертинг должен использовать независимые внешние системы
- **records-lag-max для алертинга**: если консьюмер упал, метрика перестаёт обновляться. Необходим Burrow, отслеживающий оффсеты в брокере независимо
- **records-consumed-rate без учёта продюсера**: низкая скорость консьюмера может быть вызвана отсутствием сообщений от продюсера — ложные алерты
- **Channel.Flow устарел**: продюсеры могли игнорировать его. Заменён жёстким TCP Backpressure — брокер блокирует I/O на уровне ОС
- **CQRS — усложнение**: требует нескольких хранилищ (для записи и чтения), синхронизации через event-sourced журналы, данные дублируются
- **Kafka Transactions не откатывают внешнюю БД**: Outbox Pattern обязателен для атомарности БД + Kafka
- **Параллелизм Kafka ограничен партициями**: одно приложение не может читать одну партицию в несколько потоков. Для 10 консьюмеров нужно минимум 10 партиций
- **RabbitMQ спроектирован под Push**: использование pull (Basic.Get в цикле) создаёт неравномерное распределение и потерю порядка

## Сравнения и trade-offs

| Аспект | RabbitMQ | Kafka |
|--------|----------|-------|
| Философия | Smart Broker, Dumb Consumer | Dumb Broker, Smart Consumer |
| Доставка | Push (проталкивание) | Pull (вытягивание) |
| Хранение | Удаление после ACK | Append-only лог, retention policy |
| Replay | Невозможен | Перемотка offset в прошлое |
| Backpressure | QoS prefetch + TCP backpressure | Pull-модель (из коробки) |
| Маршрутизация | Сложная (Direct, Fanout, Topic, Headers) | По ключу → партиция |
| Параллелизм | Добавить консьюмеров к очереди | Ограничен числом партиций |
| Latency | Низкая (push по одному) | Batching (выше throughput) |
| Fault tolerance | HA Queues, Federation, Shovel | Replication Factor, ISR |
| Мониторинг | rabbitmqctl, Management API, Nagios | JMX, Burrow, Prometheus |

| Когда выбрать RabbitMQ | Когда выбрать Kafka |
|------------------------|---------------------|
| Микросервисная архитектура с низкой задержкой | Обработка сотен тысяч сообщений в секунду |
| Сложная маршрутизация по ключам и заголовкам | Потоковая обработка (Stream Processing) |
| Классические очереди задач (point-to-point) | Долговременное хранение на диске |
| Нет необходимости в хранении данных на брокере | Многократное чтение (replay) тысячами подписчиков |
| | Строгая упорядоченность на уровне партиций |

| Паттерн | Суть | Когда применять |
|---------|------|-----------------|
| CQRS | Разделение Command (запись) и Query (чтение) | Разные требования к моделям чтения/записи |
| SEDA | Конвейер стадий через топики | Поэтапная обработка с независимым масштабированием |
| Transactional Outbox | Запись в БД + Outbox в одной транзакции | Атомарность БД + Kafka (проблема двойной записи) |

## Примеры

### Конфигурация RF++ для Kafka Fault Tolerance

```
replication.factor = 3
min.insync.replicas = 2
producer: acks = all
```

### Проверка consumer lag через CLI

```bash
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --group my-group
```

Выводит: `CURRENT-OFFSET`, `LOG-END-OFFSET` и `LAG` для каждой партиции.

### Создание отказоустойчивого топика

```bash
kafka-topics.sh --create --topic my-topic \
  --partitions 3 --replication-factor 3 \
  --bootstrap-server localhost:9092
```

### Аналогии

- **TCP Backpressure в RabbitMQ** — аналогия с физическим перекрытием трубы (клапаном) при нарастании давления
- **Pull-модель Kafka** — аналогия с водопроводным краном: консьюмер открывается только когда готов вместить порцию (batch)
- **SEDA** использует топики как bounded buffers между стадиями, объединяя fan-in и fan-out для эмуляции операций SQL JOIN над потоками событий
- **Transactional Outbox** часто реализуется через Kafka Connect Source-коннекторы (Debezium CDC) вместо кастомных polling-воркеров

## Важно / подводные камни / best practices

- **RF++**: для Fault Tolerance — `replication.factor=3`, `min.insync.replicas=2`, `acks=all`
- **Consumer lag — главная метрика**: объективно оценивает разрыв между генерацией и потреблением, независимо от абсолютных показателей
- **Burrow вместо records-lag-max**: если консьюмер упал, JMX-метрика перестаёт обновляться. Burrow отслеживает оффсеты в брокере независимо
- **Мониторинг Kafka — внешней системой**: если Kafka падает, внутренний мониторинг тоже. Алертинг через независимые системы или кластер другого ДЦ
- **Debezium CDC для Outbox**: специализированные коннекторы надёжнее кастомных polling-воркеров
- **SEDA Conditioning Stages**: промежуточные стадии трансформируют данные в специализированные топики, адаптированные под нужды конкретных Query-сервисов — вместо чтения всеми из «сырого» топика
- **Антипаттерн — синхронное ожидание в EDA**: ожидание подтверждения от другого микросервиса создаёт tight coupling. Транзакции должны быть локальными, коммуникация — асинхронной
- **acks=1 — скрытая потеря**: лидер отвечает ACK до репликации. При его падении — data loss без ведома приложения

Как EDA обеспечивает изоляцию сбоев?
?
1. Сервис А пишет в топик
2. Сервис Б падает
3. Сервис А продолжает, сообщения копятся в брокере
4. Сервис Б восстанавливается и вычитывает накопленное
5. Каскадного отказа нет

Алгоритм CQRS с Kafka
?
1. Клиент → Command Handling → запись в Master Data
2. Генерация события → Kafka Topic
3. Query Handling читает из Topic → обновляет View Projection
4. Клиенты читают только из View Projection

Алгоритм Transactional Outbox
?
1. SQL-транзакция: бизнес-данные + сообщение в таблицу Outbox
2. Commit СУБД
3. Worker/Debezium CDC читает Outbox
4. Публикация в Kafka
5. Пометка записи как отправленной

Как работает backpressure в RabbitMQ (QoS)?
?
1. Консьюмер шлёт Basic.QoS с prefetch_count
2. Брокер push-ит до лимита unacked
3. Окно смещается при получении Basic.Ack

Как работает TCP Backpressure в RabbitMQ?
?
1. Достигнут vm_memory_high_watermark или disk_free_limit → alarm
2. Брокер перестаёт читать из TCP-сокета продюсера
3. Отправляет Connection.Blocked клиенту

Как работает backpressure в Kafka?::Pull-модель: консьюмер сам запрашивает через poll(). max.poll.records ограничивает батч. fetch.min.bytes и fetch.max.wait.ms управляют сетевым трафиком

Алгоритм Fault Tolerance в Kafka
?
1. replication.factor=3 (Leader + 2 Followers)
2. Followers вытягивают данные, синхронные входят в ISR
3. acks=all — лидер ждёт подтверждения всех ISR
4. При сбое Controller выбирает нового лидера из ISR

Чем отличается Push (RabbitMQ) от Pull (Kafka)?
?
- Push: брокер проталкивает данные, минимальная latency, равномерное распределение
- Pull: консьюмер сам запрашивает batch, выше throughput, backpressure из коробки

Когда выбрать RabbitMQ?::Микросервисы с низкой задержкой, сложная маршрутизация, классические очереди задач, нет нужды в хранении

Когда выбрать Kafka?::Сотни тысяч msg/sec, Stream Processing, долговременное хранение, replay, тысячи подписчиков, строгий порядок в партиции

В чём проблема двойной записи?::Kafka Transactions не откатывают внешнюю БД. При падении после записи в БД, но до отправки в Kafka — неконсистентность. Решение: Transactional Outbox

Чем опасен unclean.leader.election.enable=true?::При падении всех ISR лидером становится out-of-sync реплика → безвозвратная потеря данных

Почему records-lag-max ненадёжна для алертинга?::Если консьюмер полностью упал, метрика исчезает. Нужен Burrow, отслеживающий оффсеты в брокере независимо

Антипаттерн в EDA: синхронное ожидание подтверждения?::Создаёт tight coupling и блокировки. Отказ потребителя → отказ продюсера. Транзакции локальные, коммуникация асинхронная

vm_memory_high_watermark по умолчанию — ==0.4== от RAM

disk_free_limit по умолчанию — ==50 MB==

ActiveControllerCount в Kafka должен быть строго ==1==

UnderReplicatedPartitions должно быть ==0==

Рекомендуемая конфигурация Kafka Fault Tolerance: RF=3, min.insync.replicas=  ==2==, acks=  ==all==

Параллелизм в Kafka ограничен числом ==партиций== в топике

SEDA поддерживает топологии ==fan-in== (аналог JOIN) и ==fan-out== (публикация в несколько топиков)

Kafka Connect с ==Debezium CDC== — рекомендуемый способ реализации Outbox Pattern
