## Суть и контекст

Тема охватывает стратегии кэширования данных (cache-aside, read-through, write-through, write-behind), управление временем жизни ключей через TTL и механизмы вытеснения (eviction) при переполнении памяти в Redis. Кэширование используется для ускорения доступа к данным путём хранения копий в оперативной памяти, при этом ключевой проблемой является поддержание консистентности между кэшем и базовым хранилищем.

---

## Ключевые идеи

- Существуют два принципиально разных подхода к организации кэша: **cache-aside** (кэш отделён от хранилища, приложение управляет обоими) и **inline cache** (кэш располагается перед хранилищем и проксирует все обращения к нему)
- Inline cache делится на три стратегии: read-through, write-through и write-behind, различающиеся моментом и способом синхронизации с базовым хранилищем
- TTL позволяет автоматически удалять ключи по истечении заданного времени, независимо от заполненности памяти
- При переполнении памяти (maxmemory) срабатывает политика вытеснения (eviction policy), которая определяет, какие ключи будут удалены
- Redis использует аппроксимированные алгоритмы LRU и LFU на основе случайной выборки, а не точный поиск наименее используемого ключа
- Политики вытеснения делятся на allkeys-* (применяются ко всем ключам) и volatile-* (применяются только к ключам с установленным TTL)
- Волатильный кэш (volatile cache) размещается в RAM, и его содержимое стирается при перезагрузке
- Неправильная конфигурация вытеснения или недостаточный размер кэша приводит к cache thrashing

## Определения и термины

**Cache-aside** - стратегия кэширования, при которой ответственность за консистентность лежит на приложении. Приложение сначала проверяет кэш, при промахе запрашивает базовое хранилище, а затем сохраняет полученное значение в кэш.

**Inline cache (встроенный кэш)** - кэш, располагающийся перед хранилищем данных, через который осуществляются все доступы к этому хранилищу.

**Read-through** - разновидность встроенного кэша (inline cache), который при отсутствии данных (промахе) берёт на себя ответственность за чтение значения из базового хранилища, сохранение его в кэше и возврат вызывающему приложению.

**Write-through** - разновидность встроенного кэша, при которой приложение обновляет значение в кэше, а кэш берёт на себя ответственность за синхронное обновление базового хранилища данных.

**Write-behind (write-back)** - разновидность встроенного кэша, при которой приложение обновляет значение в кэше, вызов записи завершается немедленно, а обновление базового хранилища происходит асинхронно позже в фоновом режиме.

**Volatile cache (волатильный кэш)** - кэш, размещённый в оперативной памяти (RAM), содержимое которого стирается при отключении питания или перезагрузке системы.

**Cache thrashing** - состояние снижения эффективности кэша, при котором данные постоянно удаляются из переполненного кэша, но вскоре запрашиваются снова, что приводит к циклическому процессу вытеснения и повторного считывания.

**Cache consistency (консистентность кэша)** - мера соответствия данных в кэше данным в базовом хранилище или сервисе.

> Термины cache stampede, thundering herd, cache penetration, cache avalanche, lazy expiration, active expiration - в предоставленных источниках не раскрыты.

## Как устроено

### Cache-aside (чтение)

1. Потребитель запрашивает данные в кэше
2. Если запись есть (**cache hit**), результат возвращается потребителю
3. Если записи нет (**cache miss**), потребитель запрашивает данные у целевого сервиса или БД
4. Результат сохраняется в кэш

### Cache-aside (инвалидация при записи)

Для поддержания консистентности при обновлении данных приложение записывает новое значение в БД и удаляет старое значение из кэша.

### Read-through (чтение через inline cache)

При промахе кэш самостоятельно запрашивает данные из базового хранилища, сохраняет их и возвращает вызывающему приложению. Приложение не знает о промахе - оно всегда обращается только к кэшу.

### Write-through (запись через inline cache)

1. Приложение передаёт команду на запись новых данных напрямую в кэш
2. Кэш сохраняет значение
3. Кэш **синхронно** обновляет значение в базовом хранилище данных

### Write-behind / write-back (асинхронная запись через inline cache)

1. Приложение передаёт команду на запись новых данных напрямую в кэш
2. Кэш сохраняет значение и **немедленно** возвращает ответ приложению
3. Асинхронный процесс кэша позднее записывает обновлённые данные в базовую БД

### TTL (управление временем жизни)

Ключу назначается период времени (секунды или миллисекунды). По истечении этого периода значение автоматически удаляется из кэша системой Redis, независимо от заполненности памяти.

### Механика вытеснения (cache eviction)

1. Когда память кэша заполняется до предела (`maxmemory`), срабатывает алгоритм вытеснения, заданный в `maxmemory-policy`
2. В режимах LRU и LFU Redis **не ищет** абсолютно старейший/наименее используемый ключ в памяти
3. Redis выполняет **аппроксимацию**: формирует случайную выборку (sample) из нескольких ключей
4. Из сформированной выборки вычисляется и удаляется наименее подходящий ключ, что экономит процессорное время и память на поддержание структур метаданных доступа

## Детали и нюансы

- **Временная неконсистентность в режиме write-behind.** Кэш хранит обновлённое значение, но ещё не сбросил его в базовую БД. Если приложение или другой процесс считает данные напрямую из БД в обход кэша, он получит старое (некорректное) значение.

- **TTL на элементы вложенных структур (LIST, SET, HASH, ZSET) не поддерживается.** Время жизни (TTL) применимо исключительно к ключу целиком, а не к его внутренним элементам.

- **Неполная точность вытеснения LRU/LFU.** Redis использует аппроксимированные алгоритмы выборки. Нет строгой гарантии, что будет удалён именно самый редко используемый ключ во всей базе.

- **noeviction**: если политика вытеснения не настроена, при переполнении записи будут **отклоняться с ошибкой**.

- Политики **allkeys-\*** применяются ко всем ключам в базе, политики **volatile-\*** - исключительно к ключам, у которых явно установлен TTL, позволяя сохранять постоянные ключи нетронутыми.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| **Cache-aside** | Кэш физически не связан с БД; приложение полностью контролирует логику | Ответственность за консистентность лежит на приложении; необходимо вручную оркестрировать чтение/запись | Когда нужен гибкий контроль над кэшированием |
| **Read-through** (inline) | Кэш выступает прозрачным прокси; приложение обращается только к кэшу | Кэш должен поддерживать логику чтения из хранилища | Когда нужна прозрачность доступа для приложения |
| **Write-through** (inline) | Синхронная запись гарантирует консистентность кэша и БД | Увеличенная задержка записи из-за синхронного обновления хранилища | Когда консистентность важнее скорости записи |
| **Write-behind** (inline) | Немедленный ответ приложению; высокая скорость записи | Временная неконсистентность: БД может содержать устаревшие данные до момента асинхронной синхронизации | Когда скорость записи важнее мгновенной консистентности |

| Критерий | LRU | LFU |
|----------|-----|-----|
| Принцип | Опирается на **возраст последнего доступа** | Опирается на **счётчик обращений** |
| Удаляет | Ключ, к которому дольше всего не обращались | Ключ, к которому обращались реже всего |

| Система | Поддержка политик вытеснения |
|---------|------------------------------|
| **Memcached** | Исключительно LRU |
| **Redis** | Множественные настраиваемые политики (LRU, LFU, random, noeviction; allkeys/volatile варианты) |

| Политика вытеснения | Применение | Описание |
|---------------------|-----------|----------|
| `allkeys-lru` | Все ключи | Удаляет наименее недавно использованный ключ |
| `allkeys-lfu` | Все ключи | Удаляет наименее часто используемый ключ |
| `allkeys-random` | Все ключи | Удаляет случайный ключ |
| `volatile-lru` | Только ключи с TTL | Удаляет наименее недавно использованный ключ среди имеющих TTL |
| `volatile-lfu` | Только ключи с TTL | Удаляет наименее часто используемый ключ среди имеющих TTL |
| `volatile-random` | Только ключи с TTL | Удаляет случайный ключ среди имеющих TTL |
| `noeviction` | - | Отклоняет записи с ошибкой при переполнении |

## Примеры

### Команды управления TTL

```
EXPIRE key-name seconds
EXPIREAT key-name timestamp
PTTL key-name
PEXPIRE key-name milliseconds
PEXPIREAT key-name timestamp-milliseconds
TTL key-name
PERSIST key-name
```

- `EXPIRE` и `EXPIREAT` задают таймаут в секундах или Unix timestamp
- `PTTL`, `PEXPIRE` и `PEXPIREAT` работают с точностью до миллисекунд (появились в Redis 2.6)
- `TTL` возвращает оставшееся время жизни
- `PERSIST` удаляет счётчик времени жизни, делая ключ постоянным

### Конфигурация политик вытеснения

```
maxmemory-policy allkeys-lru
maxmemory-policy allkeys-lfu
maxmemory-policy allkeys-random
maxmemory-policy volatile-lru
maxmemory-policy volatile-lfu
maxmemory-policy volatile-random
```

## Важно / подводные камни / best practices

- **LRU или LFU - лучший выбор для большинства сценариев кэширования.** Использование этих политик является наилучшим выбором для подавляющего большинства сценариев кэширования.
- **Случайное вытеснение опасно.** Использование `allkeys-random` / `volatile-random` ведёт к частой потере активно используемых данных и резкому росту cache misses.
- **Cache thrashing - следствие неправильной конфигурации.** Неправильная конфигурация алгоритма вытеснения или недостаточный размер кэша приводит к cache thrashing.
- **TTL нельзя установить на элементы вложенных структур** (LIST, SET, HASH, ZSET) - только на ключ целиком.
- **Write-behind создаёт окно неконсистентности.** Если данные читаются из БД в обход кэша до момента асинхронной записи, будет получено устаревшее значение.
