## Суть и контекст

Тема охватывает три взаимосвязанных аспекта работы с Redis: **снижение сетевых задержек** через механизм pipeline, **оптимизацию потребления памяти** через компактные внутренние представления данных (ziplist, intset) и шардирование, а также **оценку производительности** с помощью бенчмарков и формул расчёта latency при кэшировании. Всё это критично при масштабировании Redis-решений, где каждый миллисекунда RTT и каждый байт памяти на элемент имеют значение.

---

## Ключевые идеи

- Pipeline позволяет отправлять несколько команд за один round-trip, сокращая суммарную задержку в разы.
- Существуют два вида pipeline: транзакционный (с MULTI/EXEC) и не-транзакционный (без MULTI/EXEC). Они различаются по влиянию на других клиентов.
- Ziplist и intset — компактные внутренние представления, которые радикально снижают накладные расходы на память по сравнению со стандартными структурами (linkedlist, hashtable).
- Шардирование данных внутри хешей позволяет удерживать размеры в пределах ziplist и экономить до 70% памяти.
- Сжатие данных на стороне клиента (lz4, gzip, bzip2) — дополнительный способ уменьшить объём хранимых данных.
- Реальная производительность приложения на Python составляет 50–60% от результатов redis-benchmark из-за накладных расходов на парсинг протокола.
- Формула расчёта среднего времени запроса при кэшировании учитывает cache hit rate и cache miss rate.

---

## Определения и термины

**Транзакционный Pipeline** — механизм отправки последовательности команд, обёрнутых в MULTI и EXEC, в рамках одного сетевого пакета и получения всех ответов одновременно.

**Не-транзакционный Pipeline** — механизм локальной буферизации команд на стороне клиента без использования MULTI/EXEC, отправляющий команды единым пакетом.

**Round-trip time (RTT)** — время, затрачиваемое на отправку запроса по сети и получение ответа. Pipeline сокращает количество RTT.

**Ziplist** — неструктурированное компактное сериализованное представление структур данных (LIST, HASH, ZSET), устраняющее накладные расходы на указатели.

**Intset** — компактное представление множеств (SET) в виде отсортированного массива целых чисел (base-10 signed long).

**Cache Overhead (накладные расходы кэша)** — дополнительное время на проверку наличия данных в кэше и запись результата в кэш при cache miss.

> Термины jemalloc, фрагментация, used_memory, used_memory_rss, mem_fragmentation_ratio, SLOWLOG, OBJECT ENCODING, CONFIG SET, hot keys — в источниках не найдены. Упоминается только базовый вывод потребления памяти в команде INFO.

## Как устроено

### Механика не-транзакционного pipeline

1. Вызов `pipe = conn.pipeline(False)` — инициализация pipeline без MULTI.
2. Команды добавляются в локальную очередь на стороне клиента. В сеть ничего не отправляется.
3. Вызов `pipe.execute()` отправляет весь пакет команд за один RTT и возвращает все ответы.

Результат: время выполнения 3–5 команд снижается с 3–10 мс до 1–2 мс в локальной сети.

### Memory layout ziplist

Каждый элемент в ziplist хранится последовательно в виде:

1. **Первая длина** — размер предыдущего элемента (используется для реверсивного сканирования).
2. **Вторая длина** — размер текущего элемента.
3. **Полезная нагрузка** — строка данных.

Для сравнения: классический двусвязный список требует 21 байт накладных расходов на 32-битной платформе для хранения 3-байтовой строки. Ziplist тратит ~2 байта на элемент.

### Intset

Intset работает аналогично ziplist для экономии памяти, но применяется только для SET, содержащих исключительно целые числа.

### Сжатие данных на стороне клиента

Сжатие выполняется на стороне клиента перед отправкой данных в Redis. Доступные библиотеки: lz4, gzip, bzip2.

- Уровень gzip 1 обрабатывает 24–52 МБ/с.
- Уровень gzip 9 требует в 5–10 раз больше CPU ради улучшения всего на 1–5% по сравнению с уровнем 5.

### Вычисление latency при кэшировании

Формула расчёта среднего времени запроса:

```
Request_Time_With_Cache = (Cache_Miss_Rate × Request_Time_Cache_Miss) + (Cache_Hit_Rate × Request_Time_Cache_Hit)
```

Пример: при базовом запросе 500 мс, кэше 1 мс и 90% hit rate:

```
= (0.1 × 500) + (0.9 × 1) = 50 + 0.9 ≈ 50.9 мс
```

> В источнике указано значение 51.1 мс (с учётом cache overhead). Улучшение — 89.8%.

### Шардирование хешей

Пример: 320,000 ключей в одном HASH занимают 44 МБ (encoding: hashtable). При разбиении на шарды по 1024 элемента — 12 МБ (encoding: ziplist). Экономия — 70%.

### Пороги переключения encoding

Redis автоматически переключает внутреннее представление структур данных с компактного (ziplist/intset) на стандартное (linkedlist/hashtable) при превышении порогов:

```
list-max-ziplist-entries 512
list-max-ziplist-value 64
hash-max-ziplist-entries 512
hash-max-ziplist-value 64
zset-max-ziplist-entries 128
zset-max-ziplist-value 64
set-max-intset-entries 512
```

- `*-entries` — максимальное количество элементов.
- `*-value` — максимальный размер значения в байтах.

## Детали и нюансы

- **Big keys и ziplist**: ziplist может содержать до 50,000–100,000 элементов, но при таких объёмах производительность деградирует. Операция RPOPLPUSH с ~50,000 элементами показывает всего ~553 оп/сек.
- **Проблема создания нового соединения на каждую команду**: производительность падает на 25–30%. При интенсивной нагрузке возникает ошибка "Cannot assign requested address" — исчерпание доступных портов.
- **Предел 1 ядро при локальном pipelining в Python**: при использовании Unix domain socket с RTT < 0.015 мс, узким местом становится скорость парсинга протокола — ~6394 пакетов/с. Упирается в одно ядро CPU.
- **Redis-benchmark**: на 2.4 GHz показывает ~34,000 req/s для PING, ~33,000 для GET/SET (при `-c 1` — одно соединение).
- **Реальный клиент vs. benchmark**: не следует сравнивать приложение с redis-benchmark. Реальный клиент на Python выдаёт 50–60% от результатов benchmark из-за парсинга.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Транзакционный pipeline (MULTI/EXEC) | Атомарность группы команд, один RTT | Блокирует других клиентов на время выполнения | Когда нужна атомарность группы операций |
| Не-транзакционный pipeline (`pipeline(False)`) | Один RTT, не блокирует других клиентов | Нет атомарности — команды могут чередоваться с командами других клиентов | Когда нужна только оптимизация сетевых задержек |
| Ziplist | ~2 байта накладных на элемент, компактно | Деградация при >50,000 элементах (RPOPLPUSH ~553 оп/с) | До 500–2048 элементов, значения до 64–128 байт |
| Hashtable / Linkedlist | Быстрый доступ при больших объёмах | 21+ байт накладных на элемент (32-бит) | Автоматически при превышении порогов ziplist |
| gzip уровень 1 | 24–52 МБ/с обработки | Менее эффективное сжатие | Когда важна скорость |
| gzip уровень 9 | Максимальное сжатие | В 5–10 раз больше CPU, выигрыш всего 1–5% vs. уровень 5 | Практически не оправдан |
| gzip уровень 5 | Баланс CPU и сжатия | — | Оптимальный компромисс |
| Один большой HASH (320K ключей) | Простая структура | 44 МБ (hashtable encoding) | — |
| Шардированные HASH (по 1024) | 12 МБ (ziplist encoding), экономия 70% | Сложнее логика доступа | Когда важна экономия памяти |

## Примеры

### redis-benchmark

```
redis-benchmark -c 1 -q
```

На 2.4 GHz: ~34,000 req/s для PING, ~33,000 для GET/SET.

### debug_object — просмотр encoding

```python
conn.debug_object('test')
# {'encoding': 'ziplist', 'serializedlength': 24, ...}
```

Возвращает метаданные: encoding (ziplist, intset, linkedlist, hashtable) и serializedlength.

### Не-транзакционный pipeline

```python
pipe = conn.pipeline(False)
pipe.hset('login:', token, user)
pipe.zadd('recent:', token, timestamp)
pipe.execute()
```


## Важно / подводные камни / best practices

- **Короткие ключи**: использовать `user:joe` вместо `username:joe` для экономии памяти.
- **Лимиты ziplist**: рекомендуемый диапазон — 500–2048 элементов, значения до 64–128 байт. Превышение приводит к деградации производительности.
- **Шардирование хешей**: при большом количестве ключей разбивать на шарды, чтобы удерживать encoding в ziplist. Экономия до 70%.
- **Не сравнивать приложение с redis-benchmark**: реальный клиент на Python выдаёт 50–60% от benchmark из-за накладных расходов на парсинг протокола.
- **Не создавать новое соединение на каждую команду**: падение производительности на 25–30%, риск исчерпания портов ("Cannot assign requested address").
- **gzip уровень 9 не оправдан**: выигрыш 1–5% по сравнению с уровнем 5 при росте CPU в 5–10 раз.
