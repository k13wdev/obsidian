# Kafka — обработка ошибок, производительность и ecosystem

## Суть и контекст

Обработка ошибок в Kafka строится на механизмах retry продюсера, идемпотентности для предотвращения дубликатов и контроле порядка сообщений через `max.in.flight.requests.per.connection`. Производительность оптимизируется через batching (batch.size, linger.ms), компрессию и использование sequential I/O с page cache. Экосистема Kafka включает Kafka Connect (интеграция с внешними системами), Schema Registry (валидация схем), Kafka Streams (потоковая обработка) и архитектурный паттерн Transactional Outbox для консистентности двойной записи.

---

## Ключевые идеи

- retries по умолчанию = Integer.MAX_VALUE, управляется через delivery.timeout.ms
- max.in.flight.requests.per.connection > 1 + retries без идемпотентности = потеря порядка
- batch.size + linger.ms определяют стратегию батчинга продюсера
- compression.type — сжатие батча на стороне клиента, распаковка у консьюмера
- buffer.memory — буфер продюсера (32 MiB), при переполнении send() блокируется
- Kafka Connect — фреймворк source/sink коннекторов для интеграции
- Schema Registry — централизованное хранилище схем (Avro, Protobuf, JSON Schema)
- Kafka Streams — клиентская библиотека потоковой обработки с RocksDB и changelog
- Transactional Outbox — паттерн для консистентности БД + Kafka
- Kafka не поддерживает вторичные индексы — только offset и timestamp

## Определения и термины

**retries** — параметр продюсера, задающий количество повторных попыток отправки при транзитной ошибке от брокера. По умолчанию `Integer.MAX_VALUE`. Работает совместно с `delivery.timeout.ms`.

**retry.backoff.ms** — интервал ожидания в миллисекундах перед каждой последующей попыткой повторной отправки. По умолчанию 100 мс.

**max.in.flight.requests.per.connection** — верхняя граница количества неподтверждённых запросов (batch), которые продюсер может отправить по одному TCP-соединению без блокировки. По умолчанию 5. При значении > 1 и включённых retries без идемпотентности — возможна потеря порядка (out-of-order).

**batch.size** — максимальный размер памяти в байтах для группировки сообщений в одну партицию. По умолчанию 16384 байт (16 KiB). Продюсер может отправлять полупустые батчи при срабатывании linger.ms.

**linger.ms** — искусственная задержка в миллисекундах, заставляющая продюсер ожидать накопления сообщений в батч. По умолчанию 0 мс. Если batch.size достигнут — отправка немедленно, игнорируя linger.ms.

**buffer.memory** — объём памяти продюсера для буферизации сообщений. По умолчанию 33554432 (32 MiB). При переполнении `send()` блокируется на `max.block.ms`.

**fetch.min.bytes** — минимальный объём данных, который брокер должен собрать перед ответом на Fetch. По умолчанию 1 байт. Работает с тайм-аутом `fetch.max.wait.ms`.

**Sequential I/O** — паттерн append-only записи в конец файла, обеспечивающий скорость, сопоставимую с пропускной способностью сети.

**Kafka Connect** — фреймворк для масштабируемого копирования данных между Kafka и другими хранилищами. Работает как кластер worker-процессов. Source connectors (чтение из систем) и sink connectors (запись в системы).

**Schema Registry** — общий репозиторий для хранения и валидации схем данных (Avro, Protobuf, JSON Schema). Декаплирует продюсеров и консьюмеров, позволяя менять формат без синхронизации обновлений.

**Kafka Streams** — клиентская библиотека для потоковой обработки (map, reduce, join) в реальном времени. Сохраняет локальное состояние в RocksDB, дублирует изменения в changelog-топиках для отказоустойчивости.

**Transactional Outbox** — паттерн, решающий проблему неконсистентности при двойной записи. Бизнес-объект и сообщение сохраняются в БД в рамках одной СУБД-транзакции. Отдельный worker вычитывает и публикует в Kafka.

## Как устроено

### Retry и образование дубликатов

1. Продюсер отправляет батч лидер-реплике
2. Лидер записывает в лог и реплицирует
3. Лидер падает ДО отправки ACK продюсеру
4. Продюсер по истечении `request.timeout.ms` инициирует retry
5. Запрос попадает к новому лидеру, который уже получил копию через репликацию → **дубликат**
6. Решение: `enable.idempotence=true` — брокер кэширует последние 5 sequence ID и отклоняет повторы (`DuplicateSequenceException`)

### Производительность (Batching + Compression + Zero-copy)

1. `send()` → сообщения сериализуются и помещаются в `buffer.memory`
2. Накапливаются в батч до `batch.size` или `linger.ms`
3. При `compression.type` — весь батч сжимается на стороне клиента
4. Брокер записывает сжатый батч в Page Cache (sequential I/O)
5. Чтение: zero-copy (sendfile) из Page Cache в сетевой сокет без распаковки на брокере

### Kafka Connect — архитектура

| Компонент | Роль |
|-----------|------|
| Worker | REST API, балансировка нагрузки, отказоустойчивость, хранение оффсетов |
| Connector | Определяет количество Tasks, распределяет работу (какие таблицы читать) |
| Task | Физически копирует данные. Source Task опрашивает внешнюю БД |
| Converter | Преобразует объекты Connect Data API в формат для Kafka (JSON, Avro, Protobuf) |

### Kafka Streams — механика стейта

1. **Локальное состояние**: агрегации и джоины кэшируются в RocksDB (память + диск)
2. **Отказоустойчивость**: все изменения (changelog) непрерывно отправляются в системный топик Kafka с политикой log compaction
3. **Восстановление**: при падении узла новая инстанция восстанавливает RocksDB из changelog-топика

### Transactional Outbox

1. Приложение в рамках одной СУБД-транзакции записывает бизнес-данные (Business Table) и сообщение (Outbox Table)
2. Отдельный Worker/Kafka Connect регулярно вычитывает (polling) новые записи из Outbox Table
3. Worker публикует их в Kafka
4. Гарантия: сообщение отправлено ↔ бизнес-транзакция зафиксирована

### Dead Letter Queue (DLQ) — программная реализация на стороне консьюмера

Kafka не имеет встроенного механизма DLQ (в отличие от RabbitMQ DLX). Реализуется программно:

1. Консьюмер получает пакет сообщений и пытается обработать
2. Возникает неисправимая ошибка (сбой десериализации, инвалидный формат)
3. Консьюмер перехватывает исключение в коде, не прерывая цикл `poll()`
4. Консьюмер (используя встроенный `KafkaProducer`) отправляет «ядовитое» сообщение в специально созданный DLQ-топик
5. Консьюмер фиксирует смещение оригинального сообщения (commit) и продолжает чтение партиции

**Антипаттерн:** бесконечный цикл попыток десериализовать битое сообщение. Исключение на уровне `poll()` прервёт поток консьюмера.

## Детали и нюансы

### Потеря порядка сообщений

Если `max.in.flight.requests.per.connection > 1` и `retries > 0` без идемпотентности: первый батч падает, второй (уже в сети) принимается успешно, retry первого записывает его ПОСЛЕ второго → out-of-order.

### message.max.bytes

При превышении (по умолчанию 1 МБ) брокер отклонит запрос. Увеличение снижает пропускную способность, блокирует сетевые потоки и замедляет репликацию. Консьюмер с `fetch.message.max.bytes` меньше, чем `message.max.bytes` на брокере, застрянет.

### Отсутствие вторичных индексов

В Kafka невозможно сделать выборку по значению поля (payload). Данные индексируются строго по offset и timestamp.

### Невозможность уменьшения партиций

Увеличение числа партиций меняет `hash(key) % numPartitions`. Сообщение, попадавшее в партицию 1, начнёт маршрутизироваться в другую → нарушение ordering для ключа.

### transactional.id утечка памяти

ID хранятся `transactional.id.expiration.ms` (по умолчанию 7 дней). 3 новых ID/сек за неделю = 5 ГБ ОЗУ брокера. Переиспользуйте ID.

### Kafka Transactions не связывают внешние ресурсы

Невозможно откатить Kafka-транзакцию, если упал коммит в СУБД-приёмнике. Для этого — Transactional Outbox pattern.

## Сравнения и trade-offs

| Параметр | По умолчанию | Описание | Trade-off |
|----------|-------------|----------|-----------|
| batch.size | 16 KiB | Размер батча | Больше → выше throughput, выше latency |
| linger.ms | 0 мс | Задержка для накопления | Больше → лучше батчинг, выше latency |
| buffer.memory | 32 MiB | Буфер продюсера | Мало → send() блокируется |
| fetch.min.bytes | 1 байт | Минимум данных для ответа | Больше → меньше запросов, выше latency |
| retries | MAX_VALUE | Повторные попытки | Дубликаты без идемпотентности |
| max.in.flight | 5 | Неподтверждённые запросы | >1 + retries → out-of-order без idempotence |

| Компонент экосистемы | Назначение | Плюсы | Минусы |
|---------------------|-----------|-------|--------|
| Kafka Connect | Интеграция с внешними системами | Готовые коннекторы, масштабируемость | Сложность настройки |
| Schema Registry | Валидация схем данных | Декаплинг, эволюция схем | Дополнительный сервис |
| Kafka Streams | Потоковая обработка | Без отдельного кластера, RocksDB | Только Kafka ↔ Kafka |

## Примеры

### Настройка продюсера для производительности

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("batch.size", 65536);        // 64 KiB
props.put("linger.ms", 50);            // 50ms задержка
props.put("compression.type", "lz4");  // сжатие батча
props.put("buffer.memory", 67108864);  // 64 MiB
```

### Идемпотентный продюсер

```java
Properties props = new Properties();
props.put("enable.idempotence", "true");
props.put("acks", "all");
props.put("retries", Integer.MAX_VALUE);
props.put("max.in.flight.requests.per.connection", "5");
```

Включение `enable.idempotence=true` требует строго `acks=all`, `retries > 0` и `max.in.flight ≤ 5`.

### Запуск Kafka Connect и создание коннектора

```bash
# Запуск в распределённом режиме
bin/connect-distributed.sh config/connect-distributed.properties

# Создание source-коннектора через REST API
curl -X POST http://localhost:8083/connectors -H "Content-Type: application/json" -d '{
  "name": "jdbc-source",
  "config": {
    "connector.class": "io.confluent.connect.jdbc.JdbcSourceConnector",
    "connection.url": "jdbc:postgresql://localhost/mydb",
    "topic.prefix": "db-",
    "mode": "incrementing"
  }
}'
```

## Важно / подводные камни / best practices

- **Настраивайте delivery.timeout.ms, не retries** — реалистичное время (120 сек) лучше, чем угадывать число попыток
- **Не используйте enable.auto.commit=true при записи во внешние БД** — offset коммитится по таймеру, при crash данные в БД не записаны (потеря)
- **Включайте идемпотентность** — предотвращает дубликаты при ретраях и гарантирует порядок при in-flight ≤ 5
- **Не превышайте message.max.bytes** — брокер отклонит, большие сообщения снижают throughput
- **Переиспользуйте transactional.id** — новые ID утекают в памяти брокера
- **Kafka Transactions не связывают внешние ресурсы** — для консистентности БД + Kafka используйте Transactional Outbox
- **Не создавайте десятки тысяч партиций** — файловые дескрипторы, Page Cache, перегрузка Controller
