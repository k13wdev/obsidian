# Fuzzing

## Суть и контекст

Fuzzing — метод тестирования, при котором случайные (мутированные) входные данные используются для обнаружения ошибок и граничных случаев (edge cases), которые сложно предусмотреть вручную. Встроен в стандартный пакет `testing` начиная с Go 1.18. Запускается через `go test -fuzz`.

---

## Ключевые идеи

- Функция фазз-теста начинается с префикса `Fuzz`, принимает `f *testing.F`
- `f.Add()` — задаёт начальный корпус (seed corpus), от которого отталкивается фаззер
- `f.Fuzz(func(t *testing.T, ...))` — анонимная функция-цель, которую фаззер запускает с мутированными данными
- Без флага `-fuzz` тест выполняется ровно один раз только с seed corpus
- Без флага `-fuzztime` фаззер работает бесконечно
- Найденный баг сохраняется в `testdata/fuzz/<ИмяТеста>/` как регрессионный тест — автоматически воспроизводится при обычном `go test`
- Фаззинг дополняет tabletests — автоматически покрывает ветки, которые сложно предусмотреть вручную
- Поддерживаются только базовые типы данных

---

## Определения и термины

**Fuzz test (фазз-тест)** — метод тестирования, при котором случайные входные данные используются для обнаружения ошибок или граничных случаев. Размещается в файле `_test.go`, имя функции начинается с `Fuzz`, параметр — `f *testing.F`.

**Seed corpus (начальный корпус)** — начальные значения, определяемые в фазз-тесте через `f.Add()`. Фаззер использует эти данные как базу для генерации новых комбинаций. Если запустить тест без флага `-fuzz`, он выполнится только один раз с использованием seed corpus.

**Fuzz target (цель фаззинга)** — анонимная функция, передаваемая в `f.Fuzz()`, которая выполняется фаззером с использованием сгенерированных входных данных. Принимает параметры: `t *testing.T` и набор входных переменных, которые фаззер будет мутировать.

**Corpus directory** — директория `testdata/fuzz/<ИмяТеста>/`, где хранятся сохранённые входные данные. Файлы из этой директории автоматически прогоняются как регрессионные тесты при обычном `go test`.

## Как устроено

### Алгоритм работы фаззера

1. `go test -fuzz` сканирует файлы на наличие функций с префиксом `Fuzz` и параметром `*testing.F`
2. Разработчик задаёт начальные значения через `f.Add()` (seed corpus)
3. Фаззер запускает цель `f.Fuzz()` с начальными данными
4. Автоматически генерирует новые комбинации входных данных (побитовые инверсии, усечения, вставка случайных байтов) для покрытия новых ветвлений кода
5. Процесс выполняется бесконечно до выявления ошибки или до истечения `-fuzztime`
6. При сбое: минимальный набор входных данных, приводящий к ошибке, сохраняется в `testdata/fuzz/<ИмяТеста>/`

### Поддерживаемые типы для f.Fuzz()

В функцию-цель `f.Fuzz(func(t *testing.T, ...))` можно передавать только базовые примитивы:
- `string`, `[]byte`, `rune`
- `bool`
- Целые числа: `int`, `int8`, `int16`, `int32`, `int64`, `uint` и их варианты
- Числа с плавающей точкой: `float32`, `float64`

Собственные структуры (`struct`), каналы и интерфейсы — не поддерживаются.

## Примеры

### Базовый фазз-тест

```go
func FuzzFooer(f *testing.F) {
    // Seed corpus — начальные значения
    f.Add(3)
    f.Add(0)
    f.Add(-1)

    // Цель фаззинга
    f.Fuzz(func(t *testing.T, a int) {
        result := Fooer(a)
        if result == "" {
            t.Errorf("Fooer(%d) вернул пустую строку", a)
        }
    })
}
```

### Команды запуска

```bash
# Запуск фаззинга (бесконечно)
go test -fuzz=FuzzFooer

# Ограничить время работы
go test -fuzz=FuzzFooer -fuzztime=10m

# Ограничить время минимизации найденного бага
go test -fuzz=FuzzFooer -fuzzminimizetime=30s

# Прогон regression tests без фаззинга (обычный go test)
go test ./...
```

## Сравнения и trade-offs

| Подход | Что покрывает | Ограничения |
|--------|--------------|-------------|
| Table-driven tests | Вручную перечисленные кейсы | Ограничен фантазией разработчика |
| Fuzzing | Автоматически расширяет покрытие ветвлений | Только базовые типы; не подходит для проверки бизнес-логики |

## Важно / подводные камни / best practices

**Бесконечное выполнение:** Если в тестируемом коде нет ошибок, фаззер работает вечно. В CI/CD обязателен флаг `-fuzztime`.

**Регрессионные тесты автоматически:** Найденные баги сохраняются в `testdata/fuzz/` и воспроизводятся при каждом `go test` без флага `-fuzz`. Этот каталог нужно коммитить в репозиторий.

**Фаззинг ≠ проверка логики:** Цель фаззинга — вызвать крэш, панику, зависание, неожиданную ошибку. Не проверяй через фаззинг, что `2+2=4`.

**Запуск без `-fuzz`:** Без флага `-fuzz` функция выполняется ровно один раз с seed corpus — полезно как smoke test.

**Ограничение типов:** Нельзя передавать `struct` напрямую. Обходной путь — принять `[]byte` и десериализовать внутри функции-цели.
