# Benchmarks

## Суть и контекст

Benchmarks в Go — инструмент измерения производительности кода: времени выполнения (`ns/op`) и потребления памяти (`B/op`, `allocs/op`). Входят в стандартный пакет `testing`, размещаются в файлах `_test.go` и запускаются через `go test -bench`.

---

## Ключевые идеи

- Функция бенчмарка начинается с префикса `Benchmark`, принимает `b *testing.B`
- `b.N` — счётчик итераций, автоматически увеличивается средой выполнения до статистической стабильности
- По умолчанию бенчмарки **не запускаются**; требуется флаг `-bench`
- `-benchmem` добавляет метрики памяти (`B/op`, `allocs/op`)
- `b.ResetTimer()` / `b.StopTimer()` / `b.StartTimer()` — управление таймером для исключения накладных расходов на setup
- Компилятор может удалить вызов функции, если результат игнорируется (dead code elimination); результат нужно сохранять в глобальную переменную

---

## Определения и термины

**Benchmark test (бенчмарк)** — метод тестирования производительности кода, при котором измеряется время выполнения и использование памяти алгоритмом путём многократного запуска одной и той же функции. Размещаются в файлах `_test.go`; имя функции начинается с `Benchmark`, за которым следует заглавная буква или цифра.

**b *testing.B** — параметр, передаваемый в функции бенчмаркинга, предоставляющий доступ к управлению циклом измерений и таймерами.

**b.N** — верхняя граница цикла `for` внутри функции бенчмарка. Значение корректируется автоматически: начинается с 1, затем увеличивается (1, 2, 5, 10, 20, 50...) до тех пор, пока бенчмарк не будет выполняться не менее одной секунды.

**ns/op** — наносекунд на операцию. Среднее время, затраченное на выполнение одной итерации целевой функции.

**B/op** — байт на операцию. Средний объём памяти, выделенный за одно выполнение функции. Выводится при использовании `-benchmem`.

**allocs/op** — количество операций выделения памяти за одно выполнение функции. Выводится при использовании `-benchmem`.

---

## Как устроено

### Жизненный цикл бенчмарка

1. `go test -bench` сканирует тестовые файлы на наличие функций `BenchmarkXxx`
2. Инструмент инициализирует `b.N = 1` и запускает функцию
3. Если выполнение завершается быстрее 1 секунды — `b.N` увеличивается по схеме 1 → 2 → 5 → 10 → 20 → 50...
4. Функция запускается заново с новым `b.N`
5. Процесс повторяется до статистически стабильной метрики `ns/op`
6. Формируется отчёт: имя бенчмарка, суффикс GOMAXPROCS (`-8`), `b.N`, `ns/op`

### Управление таймером

| Метод | Действие |
|-------|----------|
| `b.ResetTimer()` | Обнуляет счётчики времени и аллокаций; удаляет влияние кода до цикла |
| `b.StopTimer()` | Ставит учёт времени на паузу (для тяжёлой подготовки внутри цикла) |
| `b.StartTimer()` | Возобновляет учёт времени |

## Детали и нюансы

### Dead code elimination

Компилятор Go может полностью удалить вызов тестируемой функции, если возвращаемое значение игнорируется (`_ = fibo1(i)`). Бенчмарк покажет нереалистично высокую скорость (доли наносекунд). Обход: сохранять результат в локальную переменную, а после цикла — в глобальную.

### Бесконечное выполнение

Если в тело бенчмарка передать `b.N` как аргумент тестируемой функции (`fibo2(b.N)` внутри `for i := 0; i < b.N; i++`), время одной итерации будет расти вместе с `b.N`. Бенчмарк зависнет — никогда не достигнет стабильной метрики.

### Влияние окружения

Запуск бенчмарка на загруженной ОС (выполняющей сторонние задачи) приводит к искажению результатов. Среда бенчмаркинга требует изоляции ресурсов.

### Пустой цикл и GC

Пустой цикл `for i := 0; i <= 255; i++ {}` внутри бенчмарка заблокирует GC: горутина с пустым циклом никогда не «засыпает», и сборщик мусора не перехватит управление.

## Примеры

### Базовый бенчмарк с защитой от dead code

```go
var result int

func BenchmarkFibo1(b *testing.B) {
    var r int
    for i := 0; i < b.N; i++ {
        r = fibo1(i)
    }
    // Сохраняем в глобальную переменную — компилятор не удалит вызов
    result = r
}
```

### Исключение времени инициализации (ResetTimer)

```go
func BenchmarkChannels(b *testing.B) {
    ch := make(chan int, 100)
    // Инициализация завершена — сбрасываем таймер
    b.ResetTimer()

    for i := 0; i < b.N; i++ {
        ch <- i
        <-ch
    }
}
```

### Исключение тяжёлой подготовки итераций (StopTimer/StartTimer)

```go
func BenchmarkFindInLargeSlice(b *testing.B) {
    for i := 0; i < b.N; i++ {
        b.StopTimer()
        // Тяжёлая подготовка данных — не входит в замер
        data := make([]Record, 1_000_000)
        b.StartTimer()

        // Тестируемый метод — входит в замер
        Find(data, "target")
    }
}
```

### Запуск бенчмарков

```bash
# Запуск всех бенчмарков
go test -bench=.

# С метриками памяти
go test -bench=. -benchmem

# Только конкретный бенчмарк
go test -bench=BenchmarkFibo1

# Увеличить время замера до 5 секунд
go test -bench=. -benchtime=5s

# Обойти кеш результатов
go test -bench=. -count=1
```

### Пример вывода

```
BenchmarkFibo1-8    5000000    245 ns/op
BenchmarkFibo1-8    5000000    245 ns/op    48 B/op    2 allocs/op  # с -benchmem
```

- `-8` — значение GOMAXPROCS
- `5000000` — итоговое значение `b.N`
- `245 ns/op` — среднее время одной операции
- `48 B/op` — байт выделено за операцию
- `2 allocs/op` — аллокаций за операцию

## Важно / подводные камни / best practices

- Имя функции строго `BenchmarkXxx` — начинается с заглавной буквы или цифры. Строчная `benchmarkFoo` — тест будет проигнорирован
- `b.N` — только для ограничения числа итераций цикла, **не** как динамический аргумент бизнес-функции
- Всегда сохранять результат функции в глобальную переменную, чтобы избежать dead code elimination
- `b.ResetTimer()` вызывать строго **перед циклом**, а не внутри него
- Тестировать с разными размерами входных данных (1, 4, 10, 1000 байт) — эмпирически найти оптимальный trade-off
- Бенчмарки тесно связаны с `runtime/pprof` — `go test -bench -cpuprofile` позволяет строить call-графы
