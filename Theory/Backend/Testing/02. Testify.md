# Testify

## Суть и контекст

Testify — де-факто стандартная библиотека для расширенного тестирования в Go. Создана Мэтом Райером. Предоставляет богатый набор утверждений (`assert`, `require`), инструменты для работы с моками (`mock`) и ООП-подход к организации тестов (`suite`). Устанавливается командой `go get github.com/stretchr/testify`.

---

## Ключевые идеи

- `assert` и `require` предоставляют одинаковый API, но отличаются поведением при провале
- `assert` — продолжает выполнение теста; `require` — реализует «fail fast», останавливает тест немедленно
- `testify/mock` позволяет создавать моки зависимостей через `On` / `Return` / `AssertExpectations`
- `mockery` — утилита для автогенерации моков из интерфейсов
- `testify/suite` организует тесты как набор методов структуры с shared setup/teardown
- Сгенерированный `mockery` конструктор автоматически регистрирует `AssertExpectations` через `t.Cleanup`

---

## Определения и термины

**assert** — пакет Testify, при провале проверки фиксирует ошибку и выводит diff, но позволяет тесту продолжить выполнение.

**require** — пакет Testify, реализует паттерн «fail fast»: при провале проверки немедленно останавливает выполнение текущего теста (аналог `t.FailNow()` / `t.Fatal()`). Критически важен для проверок, после провала которых дальнейшее выполнение бессмысленно.

**Мок** — объект, имитирующий поведение реальных зависимостей (базы данных, стороннего API) для тестирования кода в изоляции.

**mockery** — утилита с открытым исходным кодом для автогенерации моков из интерфейсов Go через аннотацию `//go:generate`.

## Как устроено

### assert vs require

```go
func TestMapWithTestify(t *testing.T) {
    // require: если провалится — выполнение остановится здесь
    require.Equal(t, map[int]string{1: "1"}, myMap)

    // assert: накапливает ошибки, идёт дальше
    assert.Equal(t, "Foo", Fooer(0))
    assert.NotEqual(t, "Foo", Fooer(1))
}
```

### Жизненный цикл мока

1. Создаётся объект мока: `mockProvider := new(mocks.UserProvider)`
2. Настраиваются ожидания: `mockProvider.On("MethodName", args...).Return(vals...)`
3. Вызывается тестируемый код, который использует мок
4. `AssertExpectations(t)` проверяет, что все ожидания выполнились

### testify/suite — жизненный цикл

1. `SetupSuite()` — вызывается один раз до старта всех тестов в suite
2. `SetupTest()` — вызывается перед каждым тестом
3. Выполняется тестовый метод (`TestXxx`)
4. `TearDownTest()` — вызывается после каждого теста
5. `TearDownSuite()` — вызывается один раз после завершения всех тестов

## Детали и нюансы

### Методы assert/require

| Метод | Назначение |
|-------|-----------|
| `Equal(t, expected, actual)` | Рекурсивная проверка равенства (структуры, мапы, слайсы) |
| `NotEqual(t, expected, actual)` | Проверка неравенства |
| `Nil(t, object)` / `NotNil` | Проверка на nil |
| `True(t, value)` / `False` | Проверка bool |
| `Error(t, err)` / `NoError` | Проверка наличия/отсутствия ошибки |
| `Contains(t, s, contains)` | Содержит ли строка/слайс/мапа элемент |
| `Len(t, object, length)` | Проверка длины слайса/массива/мапы/строки |
| `ElementsMatch(t, listA, listB)` | Одинаковые элементы независимо от порядка |
| `EqualValues(t, expected, actual)` | Равенство с приведением типов |
| `WithinDuration(t, expected, actual, delta)` | Сравнение `time.Time` с допустимой погрешностью |

### mock.On — настройка ожиданий

```go
// Конкретный аргумент
mockProvider.On("User", "test@test.com").Return(nil).Once()

// Любой аргумент
mockProvider.On("User", mock.Anything).Return(nil).Times(2)

// Любой аргумент заданного типа
mockProvider.On("User", mock.AnythingOfType("string")).Return(nil)
```

| Метод | Назначение |
|-------|-----------|
| `On("Method", args...)` | Задаёт ожидаемый вызов и аргументы |
| `Return(vals...)` | Что вернёт мок при вызове |
| `.Once()` | Ровно один вызов |
| `.Times(N)` | Ровно N вызовов |
| `mock.Anything` | Принимает любой аргумент |
| `mock.AnythingOfType("string")` | Принимает аргумент заданного типа |
| `AssertExpectations(t)` | Проверяет, что все ожидания выполнились |

## Примеры

### Базовые assert/require

```go
import (
    "testing"
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func TestService(t *testing.T) {
    user, err := service.GetUser("test@test.com")

    require.NoError(t, err)        // стоп если ошибка
    require.NotNil(t, user)        // стоп если nil

    assert.Equal(t, "John", user.Name)
    assert.True(t, user.Active)
    assert.Len(t, user.Roles, 2)
}
```

### Работа с моками

```go
// Создаём мок
mockProvider := new(mocks.UserProvider)

// Настраиваем ожидание
mockProvider.On("User", "test@test.com").Return(nil).Once()
mockProvider.On("User", mock.Anything).Return(nil).Times(2)

// Передаём мок в тестируемый код
handler := NewHandler(mockProvider)
handler.Process("test@test.com")

// Проверяем, что все ожидания выполнились
mockProvider.AssertExpectations(t)
```

### testify/suite

```go
import (
    "testing"
    "github.com/stretchr/testify/suite"
)

type MyLogicSuite struct {
    suite.Suite
    myMock    *MyMockObject
    myService *Service
}

func (s *MyLogicSuite) SetupTest() {
    s.myMock = new(MyMockObject)
    s.myService = NewService(s.myMock)
}

func (s *MyLogicSuite) TestCalculation() {
    s.myMock.On("DoCalc").Return(42).Once()

    res := s.myService.PerformCalc()

    s.Equal(42, res)
}

// Точка входа для go test
func TestMyLogicSuite(t *testing.T) {
    suite.Run(t, new(MyLogicSuite))
}
```

## Важно / подводные камни / best practices

**Type Panic в моках:** `Return()` возвращает интерфейсы. Если указать значение не того типа (например, строку вместо числа), произойдёт `panic` при запуске (жёсткое приведение типов), а не просто fail теста.

**Unexpected Call:** Если тестируемый код вызовет мок с аргументами, не прописанными в `On()`, тест завершится с ошибкой «unexpected call».

**Сравнение типов:** `assert.Equal` строго проверяет типы. `int32(5)` и `int64(5)` — разные типы, тест упадёт. Для числовых сравнений разных типов используй `assert.EqualValues`.

**Сравнение time.Time:** Два объекта `time.Time` могут совпадать по значению, но отличаться значением monotonic clock. Используй `assert.WithinDuration` с дельтой.

**Указатели в mock.On:** `mock.On("Save", MyStruct{})` не сработает, если код вызывает `Save(&MyStruct{})`. Используй `mock.Anything` или `mock.MatchedBy(func(v *MyStruct) bool { ... })`.

**AssertExpectations в современном mockery:** Сгенерированный конструктор мока автоматически регистрирует `AssertExpectations` через `t.Cleanup()` — вызывать вручную не обязательно, но и не вредно.

## Карточки для повторения

#flashcards/go/testify

**assert vs require** — в чём разница?
?
`assert`: при провале фиксирует ошибку и продолжает выполнение теста
`require`: при провале немедленно останавливает тест (fail fast, аналог `t.Fatal`)

Когда использовать `require` вместо `assert`?::Когда после провала проверки дальнейшее выполнение бессмысленно или опасно (например, продолжение с nil-указателем вызовет панику)

Что делает `mockProvider.On("Method", args...).Return(vals...)`?::Настраивает ожидание: задаёт, какой метод и с какими аргументами будет вызван, и что он вернёт

Что делает `mockProvider.AssertExpectations(t)`?::Проверяет, что все настроенные через `On` ожидания выполнились; если вызов не состоялся или был лишним — тест падает

`mock.Anything` в Testify?::Константа, которая принимает любой аргумент в `mock.On()`

Что такое **Type Panic** при работе с testify/mock?::Если в `Return()` передать значение не того типа, мок выполнит жёсткое приведение типов и вызовет `panic` во время выполнения

Метод для сравнения двух слайсов с одинаковыми элементами в любом порядке?::`assert.ElementsMatch(t, listA, listB)`

Метод для сравнения `time.Time` с допустимой погрешностью?::`assert.WithinDuration(t, expected, actual, delta)`

Почему `assert.Equal(t, int32(5), int64(5))` падает?::Testify строго проверяет типы; `int32` и `int64` — разные типы; для числовых сравнений без учёта типов используй `assert.EqualValues`

Структура testify/suite — шаги жизненного цикла
?
1. `SetupSuite()` — один раз до всех тестов
2. `SetupTest()` — перед каждым тестом
3. Выполнение `TestXxx`
4. `TearDownTest()` — после каждого теста
5. `TearDownSuite()` — один раз после всех тестов

Как запустить testify/suite?
?
```go
func TestMySuite(t *testing.T) {
    suite.Run(t, new(MySuite))
}
```
