# 11. RBAC, Network Policies, Pod Security

## Суть и контекст

Безопасность Kubernetes строится на трёх уровнях: RBAC контролирует доступ к API-серверу (кто что может делать с ресурсами кластера), Network Policies ограничивают сетевой трафик между подами (whitelist-брандмауэр уровня подов), Pod Security Context задаёт ограничения ОС для процессов внутри контейнеров.

---

## Ключевые идеи

- RBAC: запрос авторизуется если хотя бы одно правило разрешает действие; по умолчанию — запрещено
- Role + RoleBinding — Namespace-скоп; ClusterRole + ClusterRoleBinding — кластерный скоп
- ClusterRole через RoleBinding — доступ только внутри конкретного Namespace (не кластера)
- Network Policy: без политик — Default Allow (весь трафик); при применении политики к поду — Default Deny + whitelist
- NetworkPolicy работает только с поддерживающим CNI (Calico, Cilium, WeaveNet); Flannel без интеграции — не применяется
- Несколько NetworkPolicy на один под объединяются через логическое ИЛИ
- `capabilities: drop: [all]` + `runAsNonRoot: true` — основа Pod Security
- `automountServiceAccountToken: false` для подов без доступа к API-серверу — защита от RCE-атак
- `hostNetwork: true` / `hostPID: true` / `hostPath` mount — container breakout векторы

---

## Определения и термины

**Role (Роль)** — ресурс, содержащий правила, представляющие набор разрешений в рамках одного конкретного пространства имён (Namespace).

**RoleBinding (Привязка роли)** — ресурс, предоставляющий разрешения, определённые в Role, одному или нескольким пользователям, группам или ServiceAccount в рамках определённого пространства имён.

**ClusterRole (Кластерная роль)** — ресурс, содержащий правила и разрешения, не привязанные к конкретному пространству имён, применяемые на уровне всего кластера или для предоставления доступа к не-ресурсным URL-адресам (например, `/healthz`).

**ClusterRoleBinding (Кластерная привязка роли)** — ресурс, предоставляющий разрешения ClusterRole набору пользователей на уровне всего кластера (во всех пространствах имён).

**ServiceAccount (Служебная учётная запись)** — объект, представляющий идентификатор приложения в Kubernetes. По умолчанию монтируется в каждый создаваемый под и используется контейнерами для отправки запросов к API-серверу Kubernetes.

**Principle of least privilege (Принцип минимальных привилегий)** — принцип, согласно которому каждый процесс и контейнер должен запускаться только с минимальным набором прав доступа и ресурсов, необходимых для выполнения поставленной задачи.

**Network Policy (Сетевая политика)** — набор правил брандмауэра (работающий по принципу «белых списков»), применяемый к пространствам имён и подам с использованием селекторов меток, специфицирующий, каким группам подов разрешено общаться друг с другом и с внешними конечными точками сети.

**Pod Security Policy (PSP)** — ресурс уровня кластера, определяющий условия безопасности (контекст), которым должны удовлетворять поды для допуска в кластер (например, запрет привилегированных контейнеров, использование hostNetwork). Применяется через Admission Controller.

**Security Context (Контекст безопасности)** — набор параметров уровня ОС (UID, GID, capabilities, SELinux, Seccomp, AppArmor), определяющий ограничения доступа и привилегий для подов и контейнеров во время выполнения.

**AppArmor** — модуль безопасности ядра Linux, применяющий к процессам профили, ограничивающие привилегии на уровне системных параметров ядра и прав доступа к файлам.

**seccomp (secure computing mode)** — функция ядра Linux для ограничения процессов в возможностях выполнения системных вызовов (syscalls).

## Как устроено

### RBAC: алгоритм авторизации

1. Запрос поступает на API-сервер → этап аутентификации (TLS-сертификаты, токены)
2. Аутентифицированный запрос → авторизация. Включает: username, verb (get/list/create/delete), ресурс (pods/secrets)
3. API-сервер последовательно опрашивает модули авторизации (ABAC, RBAC, Webhook)
4. RBAC: проверяет все RoleBindings и ClusterRoleBindings субъекта
5. Если хотя бы одно правило **разрешает** действие → запрос авторизован
6. Если ни один модуль не разрешает → **HTTP 403 Forbidden**

### Матрица RBAC ресурсов

| Ресурс | Скоп | Субъекты | Назначение |
|--------|------|---------|-----------|
| Role | Namespace | Пользователи/Группы/SA | Правила в пределах namespace |
| RoleBinding | Namespace | Пользователи/Группы/SA | Привязка Role или ClusterRole к namespace |
| ClusterRole | Кластер | — | Правила на уровне кластера / non-namespaced ресурсы |
| ClusterRoleBinding | Кластер | Пользователи/Группы/SA | ClusterRole для всего кластера |

> **ClusterRole + RoleBinding** (не ClusterRoleBinding): ClusterRole используется как **шаблон** — права применяются только в Namespace, указанном в RoleBinding. Доступ к Node, PV и другим кластерным ресурсам — всё равно 403.

### Network Policies: механизм

- Без политик: **Default Allow** — весь трафик между всеми подами разрешён
- При применении NetworkPolicy к поду (через `podSelector`): под переходит в **изоляцию** (Default Deny)
- Трафик разрешается только при явном совпадении с правилами `ingress` / `egress`
- Несколько NetworkPolicy на один под: итог = **Union (логическое ИЛИ)** всех правил
- Реализацию правил обеспечивают CNI-плагины (Calico, Cilium, WeaveNet, Canal) — Flannel без интеграции политики **не применяет**

### Pod Security Context: что контролирует

| Параметр | Уровень | Эффект |
|---------|---------|--------|
| `runAsUser: 1000` | Pod/Container | Процесс запускается под UID 1000 |
| `runAsNonRoot: true` | Pod/Container | Запрет запуска как root |
| `readOnlyRootFilesystem: true` | Container | Корневая ФС только для чтения |
| `capabilities: drop: [all]` | Container | Сброс всех Linux capabilities |
| `capabilities: add: [MKNOD]` | Container | Добавление конкретных capabilities |
| `allowPrivilegeEscalation: false` | Container | Блокировка эскалации привилегий |
| `fsGroup: 2000` | Pod | GID для файловой системы томов |

## Детали и нюансы

- **ClusterRole через RoleBinding для кластерных ресурсов**: 403 Forbidden — RoleBinding не может предоставлять доступ к ресурсам уровня кластера. Для Node/PV нужен ClusterRoleBinding
- **NetworkPolicy + несовместимый CNI**: объект создаётся в API успешно, но правила не применяются — трафик не фильтруется
- **Эскалация привилегий через RBAC**: пользователь не может создать Role с привилегиями, которых у него самого нет — API-сервер отклонит запрос
- **PSP + существующие поды**: изменение PodSecurityPolicy не влияет на уже запущенные поды — применяется только при создании/обновлении манифеста
- **automountServiceAccountToken**: токен `serviceaccount/token` монтируется по умолчанию в `/var/run/secrets/kubernetes.io/` — при RCE злоумышленник может его прочитать и получить доступ к API
- **hostNetwork/hostPID/hostPath**: разрушают изоляцию контейнера → vector container breakout (доступ к сетевым интерфейсам узла, процессам хоста, файловой системе)

---

## Сравнения и trade-offs

| Подход | Role + RoleBinding | ClusterRole + ClusterRoleBinding | ClusterRole + RoleBinding |
|--------|-------------------|----------------------------------|--------------------------|
| Область | Один Namespace | Весь кластер | Один Namespace (шаблон) |
| Кластерные ресурсы | Нет | Да | Нет |
| Переиспользование | Нет | Да | Да |

## Примеры

```yaml
# Role: разрешение на получение подов и их логов
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-and-pod-logs-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list"]
```

```yaml
# RoleBinding: привязка ClusterRole к ServiceAccount в Namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: sample
subjects:
- kind: ServiceAccount
  name: read-pods
  namespace: sample
roleRef:
  kind: ClusterRole
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

```yaml
# NetworkPolicy: ingress + egress whitelist
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: test-network-policy
  namespace: default
spec:
  podSelector:
    matchLabels:
      role: db
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
      - ipBlock:
          cidr: 172.17.0.0/16
          except: [ 172.17.1.0/24 ]
      - namespaceSelector:
          matchLabels:
            project: myproject
      ports:
        - protocol: TCP
          port: 6379
  egress:
    - to:
      - ipBlock:
          cidr: 10.0.0.0/24
      - podSelector:
          matchLabels:
            role: frontend
      ports:
        - protocol: TCP
          port: 5978
```

```yaml
# Pod Security Context: drop all capabilities, readOnly FS, runAsUser
apiVersion: v1
kind: Pod
metadata:
  name: security-context-demo
spec:
  securityContext:
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
  containers:
  - name: sec-ctx-demo
    image: busybox
    command: ["sh", "-c", "sleep 1h"]
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
          - all
        add: ["MKNOD"]
```

```yaml
# AppArmor профиль через аннотацию
metadata:
  annotations:
    container.apparmor.security.beta.kubernetes.io/<container-name>: localhost/<profile-name>
```

## Важно / подводные камни / best practices

- **Drop all capabilities по умолчанию**: большинству приложений не нужны дополнительные Linux capabilities
- **runAsNonRoot: true**: если образ запускается как root — снижает поверхность атаки при компрометации
- **NetworkPolicy Default Deny**: создать политику с пустыми ingress/egress в каждом Namespace, затем точечно разрешать
- **automountServiceAccountToken: false**: для подов без доступа к K8s API — защита от использования токена при RCE
- **Запрет hostNetwork/hostPID/hostPath** без явной необходимости — риск container breakout
- **ClusterRole как шаблон через RoleBinding** — переиспользование политик без дублирования

## Карточки для повторения

#flashcards/k8s/security

**Role**:::Набор разрешений в рамках одного Namespace; содержит apiGroups + resources + verbs

**RoleBinding**:::Привязывает Role или ClusterRole к субъектам (User/Group/ServiceAccount) в рамках одного Namespace

**ClusterRole**:::Набор разрешений на уровне кластера; для non-namespaced ресурсов (Node, PV) и URL; может использоваться как шаблон через RoleBinding

**ClusterRoleBinding**:::Привязывает ClusterRole к субъектам на уровне всего кластера (все Namespaces)

**ServiceAccount**:::Идентификатор приложения в K8s; токен монтируется в /var/run/secrets/kubernetes.io/; используется для запросов к API-серверу

**Network Policy**:::Whitelist-брандмауэр для подов (ingress/egress); без политик — Default Allow; при наличии — Default Deny + явные разрешения

**Security Context**:::Параметры ОС для контейнера: runAsUser, capabilities, readOnlyRootFilesystem, seccomp, AppArmor

RBAC: алгоритм авторизации запроса
?
1. Аутентификация (TLS/токен) → получение username
2. Запрос: username + verb (get/list/...) + resource (pods/secrets/...)
3. RBAC: проверка всех RoleBindings/ClusterRoleBindings субъекта
4. Если хотя бы одно правило разрешает → авторизован
5. Иначе → HTTP 403 Forbidden

Edge case: ClusterRole + RoleBinding для Node/PV?::403 Forbidden — RoleBinding не предоставляет доступ к кластерным ресурсам, даже через ClusterRole. Нужен ClusterRoleBinding

Edge case: NetworkPolicy + Flannel без интеграции Calico?::Объект создаётся в API, но правила не применяются — трафик не фильтруется. NetworkPolicy работает только с поддерживающим CNI

Edge case: несколько NetworkPolicy на один под?::Объединяются через логическое ИЛИ (Union): если одна разрешает порт 80, вторая — 443, под принимает оба порта

Edge case: пользователь создаёт Role с привилегиями, которых у него нет?::API-сервер отклонит запрос — защита от эскалации привилегий через RBAC

Default Allow vs Default Deny в NetworkPolicy?::Без NetworkPolicy = Default Allow (весь трафик). При применении NetworkPolicy к поду = Default Deny для неразрешённого + whitelist из правил

Зачем `automountServiceAccountToken: false`?::Токен SA монтируется по умолчанию в каждый под; при RCE злоумышленник прочитает его и может обращаться к K8s API. Отключить если приложению не нужен доступ к API

Риски `hostNetwork: true` / `hostPID: true`?::Разрушают изоляцию контейнера; доступ к сетевым интерфейсам узла и процессам хоста → vector container breakout

PSP: применяется ли к существующим подам при изменении?::Нет — Admission Controller применяет PSP только при создании/обновлении манифеста; существующие поды не затрагиваются
