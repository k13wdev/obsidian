# 08. HPA, Autoscaling

## Суть и контекст

Kubernetes предоставляет три уровня автомасштабирования: HPA (Horizontal Pod Autoscaler) — изменяет количество подов; VPA (Vertical Pod Autoscaler) — изменяет ресурсы одного пода; Cluster Autoscaler — изменяет количество узлов кластера. HPA работает на основе метрик (CPU, Memory, custom) и периодически пересчитывает необходимое число реплик.

---

## Ключевые идеи

- HPA работает через Scale sub-resource — единообразно масштабирует Deployment, ReplicaSet, StatefulSet
- Метрики CPU вычисляются от `requests` контейнера, не от Capacity узла или `limits`
- При нескольких метриках HPA выбирает наибольшее требуемое количество реплик (`Max(...)`)
- Cooldown: scale-up не чаще 3 мин, scale-down не чаще 5 мин; максимальный шаг — удвоение реплик
- Масштабирование по памяти нецелесообразно: приложения не освобождают память принудительно → бесконечный scale-up
- `minReplicas: 0` (scale to zero) нативный HPA не поддерживает
- HPA всегда привязывать к Deployment, не к ReplicaSet напрямую
- Cluster Autoscaler: scale-up при Pending подах; scale-down при < 50% requests на узле

---

## Определения и термины

**HorizontalPodAutoscaler (HPA)** — ресурс и контроллер Kubernetes, который периодически проверяет метрики подов, вычисляет необходимое количество реплик для достижения целевого значения метрики и автоматически обновляет поле `replicas` в целевом ресурсе (Deployment, ReplicaSet, StatefulSet).

**Vertical Pod Autoscaling (VPA)** — механизм вертикального масштабирования, который автоматически настраивает или изменяет запросы ресурсов (CPU/Memory) для контейнеров пода (через плагин `InitialResources` или динамически во время работы).

**Cluster Autoscaler** — механизм, который автоматически запрашивает у облачного провайдера дополнительные узлы кластера при нехватке ресурсов для запуска подов и удаляет (de-provision) узлы, если они недоиспользуются.

**TCUP (targetCPUUtilizationPercentage)** — целевой процент использования процессора, который контроллер HPA пытается поддерживать путём регулирования количества реплик.

**PodDisruptionBudget (PDB)** — ресурс, указывающий минимальное количество подов, которое всегда должно быть доступно (или максимальное количество недоступных), для предотвращения прерывания сервиса при добровольном завершении работы узлов (например, при scale-down Cluster Autoscaler).

## Как устроено

### Поток метрик HPA

```
Контейнеры → cAdvisor (в Kubelet на каждом узле)
           → Heapster / Metrics API (агрегация)
           → HPA Controller (периодические REST-запросы)
           → Scale sub-resource (adjusts replicas в Deployment/RS)
```

### Алгоритм вычисления реплик (одна метрика)

1. Суммирует значения метрики по всем подам
2. Делит сумму на целевое значение (`targetAverageUtilization`)
3. Округляет вверх до ближайшего целого

**Пример:** суммарная загрузка CPU = 108%, цель = 30%
→ `108 / 30 = 3.6` → округляем вверх → **4 реплики**

### Алгоритм при нескольких метриках

HPA вычисляет требуемое количество реплик для каждой метрики отдельно и выбирает **наибольшее**:

```
Метрика CPU (цель 50%): (15+30+12)/20 = 57/20 → 3 реплики
Метрика QPS (цель 50):  (60+90+50)/50 = 200/50 → 4 реплики
→ Max(3, 4) = 4 реплики
```

### Cooldown периоды

| Направление | Cooldown | Максимальный шаг |
|-------------|----------|-----------------|
| Scale-up | Минимум 3 мин между изменениями | Удвоение (при > 2 реп.) или до 4 (при 1–2 реп.) |
| Scale-down | Минимум 5 мин между изменениями | — |

### Типы метрик HPA

| Тип | Описание | Пример |
|-----|---------|--------|
| Resource | CPU/Memory контейнера; % от `requests` | `targetAverageUtilization: 30` |
| Pods | Произвольная custom-метрика пода (среднее) | `qps: targetAverageValue: 100` |
| Object | Метрика другого объекта кластера (одно значение) | `latencyMillis` объекта Ingress |

### VPA (Vertical Pod Autoscaling)

Плагин Admission Control `InitialResources`: при создании пода без указанных `requests` — анализирует исторические данные потребления ресурсов контейнерами (по образу и тегу) и автоматически внедряет вычисленные `requests` в спецификацию пода.

### Cluster Autoscaler

**Scale-up алгоритм:**
1. Обнаруживает под в состоянии `Pending` (Scheduler не может разместить)
2. Проверяет node pools облачного провайдера
3. Если найден подходящий pool → запрашивает новый узел через облачное API
4. Новый узел регистрируется → kubelet присоединяется → под планируется

**Scale-down алгоритм:**
1. Мониторит сумму `requests` подов на каждом узле
2. Если сумма `< 50% Capacity` длительное время → узел помечается как ненужный
3. Узел помечается `unschedulable`
4. Поды вытесняются (evicted) → пересоздаются ReplicaSet-контроллером на других узлах
5. Узел физически удаляется из облака

---

## Детали и нюансы

- **CPU > 100% в `kubectl describe hpa`** — не ошибка: % вычисляется от `requests.cpu`, если `requests.cpu: 100m` а фактически потребляется 108m → метрика = 108%
- **Масштабирование по памяти**: K8s масштабирует поды вверх, но старые поды не освобождают RAM (приложение не может принудительно вызвать GC и вернуть страницы ОС) → метрика не снижается → бесконечный scale-up до `maxReplicas`
- **scale to zero (minReplicas: 0)**: нативный HPA не поддерживает
- **HPA + kubectl rolling-update + ReplicationController**: при rolling-update создаётся новый RC, HPA остаётся привязан к старому → автомасштабирование ломается. Всегда привязывать к Deployment
- **HPA без requests.cpu**: HPA не может вычислить % использования → не работает

---

## Сравнения и trade-offs

| Уровень | HPA | VPA | Cluster Autoscaler |
|---------|-----|-----|--------------------|
| Что масштабирует | Количество подов | Ресурсы одного пода | Количество узлов |
| Когда | Изменение нагрузки | Неправильные requests | Нехватка узлов |
| Downtime | Нет | Возможен (перезапуск пода) | Нет (при PDB) |

## Примеры

```bash
# Быстрое создание HPA через CLI
kubectl autoscale deployment kubia --cpu-percent=30 --min=1 --max=5
```

```yaml
# HPA декларативно (CPU-based)
apiVersion: autoscaling/v2beta1
kind: HorizontalPodAutoscaler
metadata:
  name: kubia
spec:
  scaleTargetRef:
    apiVersion: extensions/v1beta1
    kind: Deployment
    name: kubia
  minReplicas: 1
  maxReplicas: 5
  metrics:
  - type: Resource
    resource:
      name: cpu
      targetAverageUtilization: 30
```

```yaml
# HPA на основе custom-метрики подов (QPS)
spec:
  metrics:
  - type: Pods
    resource:
      metricName: qps
      targetAverageValue: 100
```

```yaml
# HPA на основе метрики объекта Ingress (latency)
spec:
  metrics:
  - type: Object
    resource:
      metricName: latencyMillis
    target:
      apiVersion: extensions/v1beta1
      kind: Ingress
      name: frontend
    targetValue: 20
```

```yaml
# PodDisruptionBudget: защита от eviction при Cluster Scale-down
apiVersion: policy/v1beta1
kind: PodDisruptionBudget
metadata:
  name: kubia-pdb
spec:
  minAvailable: 3
  selector:
    matchLabels:
      app: kubia
```

## Важно / подводные камни / best practices

- **HPA всегда к Deployment**, не к ReplicaSet/ReplicationController — корректная работа при Rolling Update
- **targetAverageUtilization ≤ 80%**: запас времени на запуск новых подов до пика нагрузки (HPA реагирует с задержкой до нескольких минут)
- **Масштабирование по памяти нецелесообразно** если приложение не освобождает память пропорционально снижению нагрузки
- **Custom метрики (QPS)**: использовать только если значение метрики линейно падает при добавлении реплик
- **Всегда задавать maxReplicas**: защита от runaway scaling при баге (100% CPU без реальной нагрузки)
- **PodDisruptionBudget** при Cluster Autoscaler: защита от одновременного eviction всех реплик при scale-down узла
