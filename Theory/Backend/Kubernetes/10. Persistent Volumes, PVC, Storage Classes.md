# 10. Persistent Volumes, PVC, Storage Classes

## Суть и контекст

Kubernetes разделяет управление хранилищем на два уровня: администраторы создают PersistentVolume (или StorageClass для динамического создания), разработчики запрашивают хранилище через PersistentVolumeClaim. CSI — стандартный интерфейс для внешних хранилищных плагинов. StatefulSet использует `volumeClaimTemplates` для создания уникального PV/PVC на каждый под.

---

## Ключевые идеи

- PV — ресурс уровня кластера (вне Namespace); PVC — ресурс Namespace
- Binding PV↔PVC: контроллер ищет наименьший подходящий PV (capacity ≥ request, access modes совместимы)
- Dynamic Provisioning: StorageClass + CSI-плагин → провизионер создаёт физический диск и PV автоматически
- `storageClassName: ""` (пустая строка) → запрет динамического провижионинга; только статические PV
- Access mode — ограничение на уровне узлов (node), не подов; RWO = только один узел
- StatefulSet volumeClaimTemplates: имя PVC = `<template-name>-<pod-name>`; при scale-down PVC не удаляется
- Reclaim Policy Retain: PV переходит в Released после удаления PVC; повторное использование требует ручного вмешательства
- Deployment + один PVC + replicas=2 на разных узлах → FailedAttachVolume (Multi-Attach) для RWO томов

---

## Определения и термины

**PersistentVolume (PV)** — ресурс кластера, используемый для отображения физических томов хранения (например, NFS-директории, GCE Persistent Disk, AWS EBS). PV не принадлежит какому-либо пространству имён (namespace) — это ресурс уровня кластера (аналогично Node).

**PersistentVolumeClaim (PVC)** — запрос приложения на использование PersistentVolume. Без действительного PVC приложение не может использовать PV. Строго привязан к конкретному пространству имён.

**StorageClass (Класс хранилища)** — ресурс, определяющий тип доступного хранилища и провизионера (provisioner), используемый для динамического создания PersistentVolume в ответ на PersistentVolumeClaim.

**Dynamic Provisioning (Динамическое выделение)** — механизм, при котором кластерный провизионер автоматически выделяет физический диск и создаёт объект PersistentVolume «на лету» при получении PersistentVolumeClaim от пользователя.

**Container Storage Interface (CSI)** — опубликованный промышленный стандарт взаимодействия между механизмами оркестрации контейнеров (Kubernetes, Mesos) и внешними провайдерами хранилищ. CSI повторяет архитектуру CNI: замена in-tree драйверов на out-of-tree плагины (в виде DaemonSet подов).

**volumeClaimTemplates** — поле в спецификации контроллера StatefulSet, которое используется для автоматической генерации уникальных объектов PersistentVolumeClaim для каждого создаваемого пода.

**Access Modes (Режимы доступа):**
- **ReadWriteOnce (RWO)** — том может быть смонтирован для чтения и записи только на одном рабочем узле одновременно
- **ReadOnlyMany (ROX)** — том может быть смонтирован только для чтения одновременно на множестве узлов
- **ReadWriteMany (RWX)** — том может быть смонтирован для чтения и записи одновременно на множестве узлов

**Reclaim Policy (Политика возврата):**
- **Retain** — том и его содержимое сохраняются, переходя в статус `Released`; для повторного использования требуется ручное вмешательство администратора
- **Delete** — удаляется как объект PV в кластере, так и базовое физическое хранилище (диск в GCE/AWS)
- **Recycle** — содержимое тома очищается (`rm -rf /volume/*`), становится доступным для новых PVC (только для NFS и HostPath)

## Как устроено

### Статическое выделение: алгоритм Binding

1. Администратор создаёт PersistentVolume с указанием `capacity` и `accessModes`
2. Пользователь создаёт PersistentVolumeClaim, указывая минимально требуемый размер и `accessModes`
3. PersistentVolume Controller ищет подходящий PV (если в PVC указан `storageClassName` — поиск ограничен им)
4. Контроллер выбирает **наименьший** из доступных PV, где `size ≥ PVC request` и `access modes` совместимы
5. Контроллер связывает (binds) PV и PVC — **эксклюзивно (1:1)**
6. Если подходящий PV не найден — PVC остаётся в статусе `Pending`

### Dynamic Provisioning: алгоритм

```
Администратор:
  Развёртывает CSI-плагин (в виде DaemonSet)
  Создаёт StorageClass → provisioner = CSI-плагин
                                    ↓
Пользователь создаёт PVC с именем StorageClass
                                    ↓
PV Controller → обращается к CSI-провизионеру
                                    ↓
Провизионер → облачное API → создаёт физический диск (AWS EBS / GCE PD / ...)
                                    ↓
Провизионер → создаёт объект PV в кластере → связывает с PVC
                                    ↓
Kubelet монтирует хранилище в контейнер
```

### volumeClaimTemplates в StatefulSet

- Контроллер StatefulSet создаёт поды **последовательно** (0 → N-1)
- Перед каждым подом: из `volumeClaimTemplates` генерируется уникальный PVC с именем `<template-name>-<pod-name>`
- PVC инициирует Dynamic Provisioning (или привязывается к статическому PV)
- **При scale-down**: под удаляется, PVC **строго сохраняется** (данные не теряются)
- **При scale-up**: новый под с тем же именем переподключает **существующий** PVC

---

## Детали и нюансы

- **Deployment + RWO PVC + replicas=2 на разных узлах**: первый под монтирует том → второй зависает в `ContainerCreating` с `FailedAttachVolume: Multi-Attach error`. Облачный провайдер аппаратно блокирует монтирование диска к нескольким VM. Решение: RWX или StatefulSet
- **Reclaim Policy Retain + пересоздание PVC**: при удалении PVC → PV переходит в `Released`; новый PVC с тем же именем **не привяжется** к нему (K8s блокирует повторное использование Released тома — он может содержать чужие данные). Администратор должен вручную очистить диск и пересоздать PV
- **Изменение accessModes в существующем PVC**: не допускается платформой. Нужно удалить и пересоздать ресурс
- **PVC 80Gi vs PV 100Gi и 500Gi**: K8s привяжет к 100Gi; 20Gi останутся неиспользованными (перерасход в облаке)
- **`storageClassName: ""`**: без этого поля K8s применит дефолтный StorageClass и динамически создаст новый PV, проигнорировав статический

---

## Сравнения и trade-offs

| Подход | Управление | Гибкость | Когда использовать |
|--------|-----------|----------|--------------------|
| Static Provisioning | Ручное создание PV | Жёсткая привязка к инфраструктуре | On-premise, legacy storage |
| Dynamic Provisioning (StorageClass + CSI) | Автоматическое | Переносимость между провайдерами | Cloud, современный production |

| Reclaim Policy | Данные при удалении PVC | Управление | Когда использовать |
|----------------|------------------------|------------|-------------------|
| Retain | Сохраняются (Released) | Ручное | Критичные данные, ценные диски |
| Delete | Удаляются физически | Автоматическое | Облачные эфемерные диски |
| Recycle | Очищаются (rm -rf) | Автоматическое | NFS/HostPath (legacy) |

| Access Mode | Монтирование | Поддержка | Когда использовать |
|-------------|-------------|----------|--------------------|
| RWO | 1 узел (R/W) | Большинство провайдеров (EBS, PD) | БД на 1 экземпляр |
| ROX | N узлов (R) | NFS, специальные | Shared read-only данные |
| RWX | N узлов (R/W) | NFS, CephFS, специальные | Shared R/W (legacy, media) |

## Примеры

```yaml
# PVC для статического PV (запрет динамического провижионинга)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: manual
spec:
  storageClassName: ""   # пустая строка = только статические PV
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Mi
```

```yaml
# StorageClass для динамического создания GCE SSD дисков
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
  zone: europe-west1-b
```

```yaml
# volumeClaimTemplates в StatefulSet (PostgreSQL)
  volumeClaimTemplates:
  - metadata:
      name: postgres-volume
    spec:
      storageClassName: longhorn
      accessModes:
        - ReadWriteOnce
      resources:
        requests:
          storage: 1Gi
# Создаст: postgres-volume-postgres-0, postgres-volume-postgres-1 ...
```

```yaml
# PV с volumeMode: Block (raw-устройство для процесса)
apiVersion: v1
kind: PersistentVolume
metadata:
  name: manual
spec:
  capacity:
    storage: 100Mi
  accessModes:
    - ReadWriteOnce
  volumeMode: Block
  csi:
    driver: driver.longhorn.io
    volumeHandle: manual
```

## Важно / подводные камни / best practices

- **StorageClass + Dynamic Provisioning** вместо статических PV: манифест переносим между провайдерами (AWS EBS / GCE PD / Minikube)
- **Никогда не использовать Deployment для stateful-приложений (БД)**: все реплики ссылаются на один PVC → одновременная запись в файлы БД → повреждение данных. StatefulSet = уникальный PVC на каждый под
- **`storageClassName: ""`**: явно запрещает Dynamic Provisioning; без этого дефолтный StorageClass может создать неожиданный PV
- **Reclaim Policy Retain для production данных**: защита от случайного удаления диска при удалении PVC
- **RWO ≠ multi-replica**: не использовать RWO PVC в Deployment с несколькими репликами на разных узлах

**PersistentVolume (PV)**:::Ресурс уровня кластера (вне Namespace); представляет физическое хранилище (NFS, EBS, PD); создаётся администратором или провизионером

**PersistentVolumeClaim (PVC)**:::Запрос приложения на хранилище; привязан к Namespace; без PVC приложение не может использовать PV; Binding с PV — 1:1

**StorageClass**:::Определяет тип хранилища и provisioner; основа Dynamic Provisioning; переносимость манифестов между провайдерами

**Dynamic Provisioning**:::При создании PVC провизионер (CSI-плагин) автоматически создаёт физический диск и объект PV; пользователь не видит инфраструктуру

**CSI (Container Storage Interface)**:::Стандарт плагинов хранилища для K8s (аналог CNI для сети); плагины работают как DaemonSet; заменяет in-tree драйверы

**volumeClaimTemplates**:::Поле StatefulSet; создаёт уникальный PVC для каждого пода по шаблону `<name>-<pod-name>`; PVC сохраняется при scale-down

Алгоритм Binding PVC к PV
?
1. PVC создаётся с capacity и accessModes
2. PV Controller ищет PV: capacity ≥ request, accessModes совместимы, storageClassName совпадает
3. Выбирает наименьший подходящий PV
4. Связывает 1:1 (эксклюзивно)
5. Если нет подходящего → PVC остаётся Pending

Reclaim Policy: Retain vs Delete vs Recycle?
?
- Retain: PV → Released; данные сохраняются; нужно ручное вмешательство
- Delete: PV и физический диск удаляются автоматически
- Recycle: rm -rf /volume/*; доступен для новых PVC (только NFS/HostPath)

Edge case: Deployment + RWO PVC + 2 реплики на разных узлах?::Первый под монтирует том, второй → ContainerCreating + FailedAttachVolume: Multi-Attach error. Облако блокирует монтирование RWO диска к нескольким VM. Решение: RWX или StatefulSet

Edge case: удалить PVC → пересоздать PVC → PV с политикой Retain?::PV переходит в Released (данные сохранены); новый PVC не привяжется автоматически (K8s блокирует повторное использование Released томов). Администратор должен вручную очистить и пересоздать PV

`storageClassName: ""` в PVC — что означает?::Запрет Dynamic Provisioning; поиск только среди статических pre-provisioned PV без StorageClass

StatefulSet scale-down: что происходит с PVC?::Под удаляется, PVC строго сохраняется (данные не теряются). При scale-up новый под с тем же именем переподключает тот же PVC

Почему Deployment нельзя использовать для БД с несколькими репликами?::Все реплики ссылаются на один PVC → один физический диск → одновременная запись → повреждение данных. StatefulSet создаёт уникальный PVC для каждой реплики

PV vs PVC: уровень Namespace?::PV — ресурс уровня кластера (вне Namespace, как Node). PVC — привязан к Namespace
