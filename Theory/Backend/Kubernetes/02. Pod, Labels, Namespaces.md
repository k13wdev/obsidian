# 02. Pod, Labels, Namespaces

## Суть и контекст

Pod — базовая развёртываемая единица в Kubernetes: группа тесно связанных контейнеров, разделяющих сетевое пространство, хранилище и жизненный цикл. Labels + Selectors — механизм организации и фильтрации ресурсов. Namespaces — виртуальные кластеры для логического разграничения ресурсов.

---

## Ключевые идеи

- Pod = логическая машина с собственным IP, hostname и процессами
- Контейнеры одного пода разделяют Network, IPC, UTS namespaces; общаются через localhost
- Один процесс — один контейнер; несколько связанных процессов — несколько контейнеров в одном поде
- Init-контейнеры запускаются последовательно до основных; каждый должен завершиться успешно
- Sidecar — вспомогательный контейнер рядом с основным (синхронизация данных, proxy, адаптер)
- Labels — произвольные пары key-value; Selectors — операторы фильтрации (In, NotIn, Exists, DoesNotExist)
- При удалении Namespace — все ресурсы внутри автоматически уничтожаются
- Node и PersistentVolume — глобальные ресурсы вне Namespaces

---

## Определения и термины

**Pod** — базовая развёртываемая единица в Kubernetes. Группа из одного или нескольких тесно связанных контейнеров, всегда выполняющихся вместе на одном рабочем узле в одних Linux namespaces. Выступает как логическая машина с собственным IP, hostname и процессами.

**Фаза Pending** — под принят системой K8s, но один или несколько образов контейнеров ещё не созданы или не загружены на узел.

**Фаза Running** — под привязан к узлу, все контейнеры созданы; хотя бы один контейнер работает, запускается или перезапускается.

**Фаза Succeeded / Completed** — все контейнеры успешно завершили работу (Exit Code 0) и не будут перезапущены (обычно в Job).

**Фаза Failed** — контейнеры завершились с ошибкой (Exit Code: 137, 1, и т.д.); система может переводить в CrashLoopBackOff.

**Фаза Unknown** — невозможно получить информацию о поде (как правило, потеря связи с узлом).

**Состояние контейнера Waiting / ContainerCreating** — контейнер в процессе создания (загрузка образа) или ожидания preconditions.

**Состояние контейнера Terminated** — контейнер завершил выполнение; содержит Exit Code, Reason (OOMKilled, Error), время запуска и завершения.

**Init containers** — контейнеры, запускаемые последовательно до основных; каждый должен успешно завершиться перед запуском следующего.

**Sidecar containers** — вспомогательные контейнеры в поде (синхронизация данных из Git, ротация логов, адаптеры).

**Label (метка)** — произвольная пара «ключ-значение», прикрепляемая к ресурсам K8s для идентификации и организации в группы.

**Label selector (селектор меток)** — критерий фильтрации для выбора подмножества ресурсов по меткам; операторы: `In`, `NotIn`, `Exists`, `DoesNotExist`.

**Namespace (пространство имён)** — абстрактный объект, логически разграничивающий ресурсы в едином кластере. Виртуальный кластер; позволяет использовать одни имена ресурсов несколько раз в разных namespace.

**Default namespaces:**
- `default` — для пользовательских объектов без указания namespace
- `kube-system` — компоненты Control Plane и K8s
- `kube-public` — доступно для чтения всем пользователям

## Как устроено

### Изоляция и совместное использование в Pod

- Контейнеры пода разделяют: **Network** (один IP, одно пространство портов), **IPC**, **UTS** (hostname) namespaces
- Изолированы: **PID** (по умолчанию), **MNT** (каждому контейнеру своя root FS)
- Общение между контейнерами пода — через **localhost**
- Между разными подами — через плоскую (flat) сеть кластера без NAT

### Жизненный цикл пода

```
kubectl apply/create → Pending (планировщик назначает узел)
                     ↓
             kubelet загружает образ → ContainerCreating
                     ↓
                  Running
                 ↙      ↘
          Succeeded    Failed / CrashLoopBackOff
```

При потере связи с узлом → `Unknown` → `Terminating` (NodeLost)

### Init-контейнеры: механизм

- Определяются в `spec.initContainers`
- Запускаются **последовательно**
- Каждый должен **успешно завершиться** (Exit Code 0) перед запуском следующего
- `spec.containers` запускаются только после успешного завершения **всех** init-контейнеров

### Labels и Selectors: механизм

Контроллеры (ReplicaSet, Service) опрашивают API-сервер, применяют операторы к метаданным подов:
- `In`, `NotIn`, `Exists`, `DoesNotExist`
- Если метка пода изменяется и перестаёт соответствовать селектору контроллера → под исключается из управления → контроллер создаёт замену

### Namespaces: изоляция и DNS

- ServiceAccount по умолчанию (`default`) назначается каждому поду; полномочия ограничены текущим namespace
- DNS-имя сервиса: `<service-name>.<namespace>.svc.cluster.local`
- Короткое имя (`<service-name>`) разрешается только в рамках текущего namespace
- Node и PersistentVolume — глобальные, вне namespace

---

## Детали и нюансы

- Несколько контейнеров в одном поде с одинаковым портом → **Port conflict** (один localhost, одно сетевое пространство)
- При удалении Namespace → **все ресурсы внутри уничтожаются** (поды, Services, Deployments, etc.)
- Namespaces обеспечивают **логическое** разделение, не сетевую изоляцию; без Network Policies поды из разных namespaces могут общаться

---

## Сравнения и trade-offs

| Паттерн | Описание | Когда использовать |
|---------|---------|-------------------|
| Один контейнер в поде | Простота, независимость | Большинство случаев |
| Несколько контейнеров (sidecar) | Вспомогательные задачи рядом с основным | Log shipping, git sync, proxy/mesh |
| Init-контейнер | Предварительная инициализация | Ожидание зависимостей, настройка конфигурации |

| Selector | Синтаксис | Описание |
|---------|---------|---------|
| Equality-based | `env=prod` | Точное совпадение |
| Set-based | `env in (prod,devel)` | Значение из набора |
| Negation | `!env` | Ключ отсутствует |

## Примеры

```yaml
# Pod с метками
apiVersion: v1
kind: Pod
metadata:
  name: kubia-manual-v2
  labels:
    creation_method: manual
    env: prod
spec:
  containers:
  - image: luksa/kubia
    name: kubia
    ports:
    - containerPort: 8080
      protocol: TCP
```

```yaml
# Init-контейнер: ожидание доступности сервиса
spec:
  initContainers:
  - name: init
    image: busybox
    command:
    - sh
    - -c
    - 'while true; do echo "Waiting for fortune service to come up..."; wget http://fortune -q -T 1 -O /dev/null >/dev/null 2>/dev/null && break; sleep 1; done; echo "Service is up! Starting main container."'
```

```bash
# Добавление метки и перезапись существующей
kubectl label pod kubia-dmdck type=special
kubectl label pod kubia-dmdck app=foo --overwrite

# Фильтрация подов по меткам
kubectl get po -l creation_method=manual
kubectl get po -l '!env'              # поды без ключа env
kubectl get po -l 'env in (prod,devel)'

# Namespace
kubectl create -f custom-namespace.yaml
kubectl delete ns custom-namespace
```

## Важно / подводные камни / best practices

- **Один основной процесс — один контейнер**; несколько процессов в одном контейнере усложняет мониторинг PID 1
- **Frontend и backend — разные поды**: независимое масштабирование, размещение на разных узлах
- **Namespace ≠ сетевая изоляция**: по умолчанию поды разных namespaces могут общаться; нужны Network Policies
- **Init-контейнеры для зависимостей**: ожидание готовности баз данных, конфигурационных сервисов — не делать это в основном контейнере
