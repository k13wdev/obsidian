# Dynamic Programming

## Суть и контекст

Dynamic Programming (динамическое программирование, DP) — метод решения сложных задач оптимизации путём разбиения на меньшие подзадачи, решения каждой ровно один раз и хранения результатов в таблице. Применяется когда задача имеет **перекрывающиеся подзадачи** и **оптимальную подструктуру**. Превращает экспоненциальные рекурсивные алгоритмы в полиномиальные.

## Ключевые идеи

- Две ключевые характеристики: optimal substructure + overlapping subproblems
- Два подхода: memoization (top-down) и tabulation (bottom-up)
- Bottom-up часто быстрее (нет накладных расходов рекурсии), top-down вычисляет только нужные подзадачи
- Время работы = количество уникальных подзадач × выборов на каждую
- Пространство часто оптимизируется: хранить только последний(е) слой(и) таблицы

## Определения и термины

**Dynamic Programming (DP)** — метод: разбить задачу на подзадачи, решить каждую один раз, сохранить результат, использовать при необходимости.

**Optimal Substructure (оптимальная подструктура)** — оптимальное решение исходной задачи содержит оптимальные решения подзадач. Можно строить глобальный оптимум из локальных оптимумов.

**Overlapping Subproblems (перекрывающиеся подзадачи)** — наивный рекурсивный алгоритм решает одни и те же подзадачи снова и снова. DP решает каждую ровно один раз.

**Memoization (мемоизация, Top-Down)** — рекурсивная реализация + кэш (массив или HashMap). При вызове сначала проверяется кэш, при cache miss — вычисляется и сохраняется.

**Tabulation (Bottom-Up)** — итеративная реализация. Подзадачи решаются от меньшего к большему в цикле. К моменту решения подзадачи все её зависимости уже вычислены и хранятся в таблице.

**DP Table** — таблица (массив, матрица) для хранения результатов подзадач.

**State** — описание подзадачи. Параметры, уникально идентифицирующие подзадачу (например, `dp[i][j]` = LCS первых i символов строки X и j символов строки Y).

**Recurrence Relation (рекуррентное соотношение)** — формула, выражающая решение задачи через решения меньших подзадач.

## Как устроено

### Два необходимых условия

**1. Optimal Substructure**: решение задачи = оптимальные решения подзадач, скомбинированные правильным образом. Позволяет уверенно строить глобальный ответ из локальных.

**2. Overlapping Subproblems**: наивная рекурсия многократно решает одно и то же. DP устраняет это хранением результатов.

**Важно: подзадачи должны быть автономны (independent)**. DP работает ТОЛЬКО когда подзадачи не зависят друг от друга — решение одной подзадачи не влияет на другую. Если выбор для одной подзадачи ограничивает выбор для другой — DP неприменим напрямую.

### Top-Down (Memoization)

```
function dp(state):
    if state in cache:
        return cache[state]
    result = compute(state)  // может вызывать dp() рекурсивно
    cache[state] = result
    return result
```

**Преимущество**: строго вычисляет только те подзадачи, которые реально нужны.
**Недостаток**: накладные расходы рекурсии; риск stack overflow при большой глубине.

### Bottom-Up (Tabulation)

```
for i from smallest to largest subproblem:
    dp[i] = f(dp[i-1], dp[i-2], ...)  // зависимости уже вычислены
return dp[target]
```

**Преимущество**: нет рекурсии → лучшие константы, нет stack overflow.
**Недостаток**: нужно правильно определить порядок вычисления.

Оба подхода дают одинаковую асимптотическую сложность.

### Как распознать DP-задачу

- Задача просит **оптимизировать** (maximize/minimize) характеристику при ограничениях
- Задача просит найти «n-й» элемент или **подсчитать количество** способов
- Каждый выбор оставляет **меньшую подзадачу того же вида**
- Комбинаторные задачи с последовательностью выборов

### Временная и пространственная сложность

**Время**: количество уникальных подзадач × число выборов на каждую подзадачу.

**Пространство**: размер DP-таблицы для хранения всех состояний.

**Оптимизация памяти**: если алгоритм использует только последний слой таблицы — хранить только его (O(n) вместо O(n²)) или даже два значения (O(1)).

## Классические примеры

### Fibonacci Numbers

**Наивная рекурсия**: O(2ⁿ) — множество перекрывающихся вызовов.

**DP top-down**: O(n) время, O(n) память — кэшируем предыдущие значения.

**DP bottom-up оптимизированный**: O(n) время, **O(1) память** — для следующего значения нужны только два предыдущих. Можно хранить лишь две переменные.

### 0-1 Knapsack (Рюкзак)

Максимизировать ценность предметов при ограниченном весе рюкзака.

Таблица `dp[i][w]` = максимальная ценность, используя первые i предметов при ёмкости w.

На каждом шаге: взять предмет или не взять → `max(dp[i-1][w], dp[i-1][w - weight[i]] + value[i])`.

**Fractional Knapsack**: если разрешено дробить предметы — задача решается жадным алгоритмом (сортировка по value/weight, берём максимально ценные по удельной стоимости). DP не нужен. 0-1 Knapsack (нельзя дробить) — только DP.

### Coin Change (Making Change)

Количество способов набрать n центов из монет заданных номиналов.

Рекурсивное дерево ветвится — использовать 0, 1, 2... монет каждого номинала. Появляются дублирующиеся пары (остаток, индекс_монеты) → мемоизация через 2D массив.

**Развёрнутая версия (подсчёт способов)**: `dp[amount][denomination]` = количество способов набрать `amount` используя монеты вплоть до `denomination`. Размер таблицы: (amount+1) × (len(coins)+1).

```
dp[0][j] = 1 для всех j  // 0 центов = 1 способ (ничего не брать)
dp[i][j] = dp[i][j-1]                    // не использовать j-ю монету
         + dp[i - coins[j]][j]  // использовать j-ю монету (если i >= coins[j])
```

### Robot in Grid (Робот в сетке)

Робот в верхнем левом углу сетки r×c, двигается только вправо или вниз. Некоторые клетки заблокированы. Найти путь до нижнего правого угла.

**Рекурсия + мемоизация неудач**: `failed` — Set заблокированных или тупиковых клеток. Если клетка приводит только в тупики — добавить в `failed`. O(rc) — каждая клетка посещается не более двух раз.

### Power Set (Множество всех подмножеств) через DP

**Техника Base-Case-and-Build**: начать с пустого множества `{{}}`, для каждого нового элемента `x` добавить ко всем существующим подмножествам копию с `x`.

```
P(∅) = {{}}
P({a}) = P(∅) ∪ {S ∪ {a} | S ∈ P(∅)} = {{}, {a}}
P({a,b}) = P({a}) ∪ {S ∪ {b} | S ∈ P({a})} = {{}, {a}, {b}, {a,b}}
```

**Сложность**: O(n·2ⁿ) — 2ⁿ подмножеств, каждое копируется за O(n).

### LIS — Восстановление последовательности

Для восстановления самой подпоследовательности (не только длины) ведётся дополнительная таблица предшественников `prev[i]` — индекс предыдущего элемента в LIS.

```go
// После заполнения dp[] и prev[]:
// Восстановить путь: найти i с max dp[i], затем следовать prev[i] назад
idx := 0
for i := 1; i < n; i++ {
    if dp[i] > dp[idx] {
        idx = i
    }
}
path := []int{}
for idx != -1 {
    path = append([]int{nums[idx]}, path...)
    idx = prev[idx]
}
```

### Longest Common Subsequence (LCS)

Найти максимальную общую подпоследовательность двух строк X и Y.

Таблица `c[i][j]` = длина LCS первых i символов X и j символов Y.

Рекуррентное соотношение:
- Если `x[i] == y[j]`: `c[i][j] = c[i-1][j-1] + 1`
- Иначе: `c[i][j] = max(c[i][j-1], c[i-1][j])`

Сложность: O(mn) время и пространство. Оптимизация: можно хранить только две строки таблицы → O(min(m,n)) пространство.

**LCS vs Longest Common Substring**: LCS — подпоследовательность (символы не обязаны быть смежными). Longest Common Substring — подстрока (символы идут подряд). Разница в рекуррентном соотношении: при несовпадении символов LCS берёт max(c[i][j-1], c[i-1][j]), а Longest Common Substring обнуляет c[i][j] = 0. Максимум LCS находится в c[m][n], а максимум Longest Common Substring — в любой ячейке таблицы (нужно искать глобальный max).

## Детали и нюансы

**Bottom-up лучше на практике**: нет накладных расходов рекурсивных вызовов → лучшие константы. Но top-down иногда удобнее при неочевидном порядке вычисления.

**Пространственная оптимизация Fibonacci**: т.к. `fib(n)` зависит только от `fib(n-1)` и `fib(n-2)` — достаточно двух переменных. O(1) пространство.

**Пространственная оптимизация LCS**: если нужна только длина, не сам путь — достаточно двух строк таблицы. O(min(m,n)) пространство.

**Backtracking → DP**: если backtracking-алгоритм имеет перекрывающиеся подзадачи → добавить мемоизацию → DP.

## Сравнения и trade-offs

| | Top-Down (Memoization) | Bottom-Up (Tabulation) |
|--|------------------------|------------------------|
| Реализация | Рекурсия + кэш | Итеративные циклы |
| Порядок вычислений | Автоматически (по требованию) | Нужно определить явно |
| Вычисляет лишние подзадачи? | Нет — только нужные | Иногда да |
| Накладные расходы | Рекурсия, стек | Нет |
| Риск stack overflow | Есть | Нет |
| Константы | Хуже | Лучше |

## Примеры

```go
// Fibonacci — top-down с мемоизацией
func fibMemo(n int, memo map[int]int) int {
    if n <= 1 {
        return n
    }
    if v, ok := memo[n]; ok {
        return v
    }
    memo[n] = fibMemo(n-1, memo) + fibMemo(n-2, memo)
    return memo[n]
}

// Fibonacci — bottom-up, O(n) время, O(1) память
func fibBottomUp(n int) int {
    if n <= 1 {
        return n
    }
    prev, curr := 0, 1
    for i := 2; i <= n; i++ {
        prev, curr = curr, prev+curr
    }
    return curr
}

// 0-1 Knapsack
func knapsack(weights, values []int, capacity int) int {
    n := len(weights)
    dp := make([][]int, n+1)
    for i := range dp {
        dp[i] = make([]int, capacity+1)
    }
    for i := 1; i <= n; i++ {
        for w := 0; w <= capacity; w++ {
            dp[i][w] = dp[i-1][w] // не берём предмет i
            if weights[i-1] <= w {
                take := dp[i-1][w-weights[i-1]] + values[i-1]
                if take > dp[i][w] {
                    dp[i][w] = take // берём предмет i
                }
            }
        }
    }
    return dp[n][capacity]
}

// Coin Change — минимальное количество монет
func coinChange(coins []int, amount int) int {
    dp := make([]int, amount+1)
    for i := range dp {
        dp[i] = amount + 1 // infinity
    }
    dp[0] = 0
    for i := 1; i <= amount; i++ {
        for _, coin := range coins {
            if coin <= i && dp[i-coin]+1 < dp[i] {
                dp[i] = dp[i-coin] + 1
            }
        }
    }
    if dp[amount] > amount {
        return -1
    }
    return dp[amount]
}

// Longest Common Subsequence
func lcs(x, y string) int {
    m, n := len(x), len(y)
    dp := make([][]int, m+1)
    for i := range dp {
        dp[i] = make([]int, n+1)
    }
    for i := 1; i <= m; i++ {
        for j := 1; j <= n; j++ {
            if x[i-1] == y[j-1] {
                dp[i][j] = dp[i-1][j-1] + 1
            } else {
                if dp[i][j-1] > dp[i-1][j] {
                    dp[i][j] = dp[i][j-1]
                } else {
                    dp[i][j] = dp[i-1][j]
                }
            }
        }
    }
    return dp[m][n]
}

// Longest Increasing Subsequence (LIS) — длина + восстановление последовательности
func lengthOfLIS(nums []int) int {
    n := len(nums)
    dp := make([]int, n)
    for i := range dp {
        dp[i] = 1
    }
    result := 1
    for i := 1; i < n; i++ {
        for j := 0; j < i; j++ {
            if nums[j] < nums[i] && dp[j]+1 > dp[i] {
                dp[i] = dp[j] + 1
            }
        }
        if dp[i] > result {
            result = dp[i]
        }
    }
    return result
}

// LIS — восстановление самой последовательности через prev[]
func lisSequence(nums []int) []int {
    n := len(nums)
    dp := make([]int, n)
    prev := make([]int, n)
    for i := range dp {
        dp[i] = 1
        prev[i] = -1
    }
    bestIdx := 0
    for i := 1; i < n; i++ {
        for j := 0; j < i; j++ {
            if nums[j] < nums[i] && dp[j]+1 > dp[i] {
                dp[i] = dp[j] + 1
                prev[i] = j
            }
        }
        if dp[i] > dp[bestIdx] {
            bestIdx = i
        }
    }
    var path []int
    for idx := bestIdx; idx != -1; idx = prev[idx] {
        path = append([]int{nums[idx]}, path...)
    }
    return path
}

// Robot in Grid — мемоизация неудач
func findPath(grid [][]bool) [][2]int {
    r, c := len(grid), len(grid[0])
    failed := make(map[[2]int]bool)
    var dfs func(row, col int) [][2]int
    dfs = func(row, col int) [][2]int {
        if row >= r || col >= c || !grid[row][col] {
            return nil
        }
        if failed[[2]int{row, col}] {
            return nil
        }
        if row == r-1 && col == c-1 {
            return [][2]int{{row, col}}
        }
        // Попробовать вправо, затем вниз
        path := dfs(row, col+1)
        if path == nil {
            path = dfs(row+1, col)
        }
        if path == nil {
            failed[[2]int{row, col}] = true
            return nil
        }
        return append([][2]int{{row, col}}, path...)
    }
    return dfs(0, 0)
}

// Power Set через Base-Case-and-Build
func powerSet(nums []int) [][]int {
    result := [][]int{{}} // начать с пустого подмножества
    for _, num := range nums {
        newSubsets := make([][]int, len(result))
        for i, subset := range result {
            newSubset := make([]int, len(subset)+1)
            copy(newSubset, subset)
            newSubset[len(subset)] = num
            newSubsets[i] = newSubset
        }
        result = append(result, newSubsets...)
    }
    return result
}

// Climbing Stairs (Fibonacci variant)
func climbStairs(n int) int {
    if n <= 2 {
        return n
    }
    prev, curr := 1, 2
    for i := 3; i <= n; i++ {
        prev, curr = curr, prev+curr
    }
    return curr
}

// Maximum Subarray (Kadane's DP)
func maxSubArray(nums []int) int {
    dp := nums[0]  // dp[i] = max subarray ending at i
    maxSum := dp
    for i := 1; i < len(nums); i++ {
        if dp < 0 {
            dp = nums[i]
        } else {
            dp += nums[i]
        }
        if dp > maxSum {
            maxSum = dp
        }
    }
    return maxSum
}
```

## Важно / подводные камни / best practices

- Сначала убедись в наличии **двух признаков**: optimal substructure + overlapping subproblems
- Определи **state** (что описывает подзадачу) и **recurrence** (как из подзадач строится решение)
- При bottom-up: правильный порядок итерации — от меньших подзадач к большим
- Оптимизируй пространство: если нужен только последний слой — храни только его
- Инициализация DP-таблицы критична: 0, -∞, +∞ — в зависимости от смысла задачи
- DP не всегда оптимально по памяти — иногда greedy-подход экономнее

#flashcards/algorithms/dp

Два необходимых условия для применения DP?::1) Optimal Substructure — оптимальное решение содержит оптимальные решения подзадач; 2) Overlapping Subproblems — наивная рекурсия решает одни и те же подзадачи снова и снова.

Что такое Memoization (Top-Down)?::Рекурсивная реализация + кэш. При вызове сначала проверяется кэш. Cache miss → вычислить и сохранить. Строго вычисляет только нужные подзадачи.

Что такое Tabulation (Bottom-Up)?::Итеративная реализация. Решаем подзадачи от меньшего к большему. К моменту решения подзадачи все её зависимости уже в таблице.

Чем Bottom-Up лучше Top-Down?::Нет накладных расходов рекурсии → лучшие константы. Нет риска stack overflow. Часто быстрее на практике.

Чем Top-Down лучше Bottom-Up?::Вычисляет только нужные подзадачи (не все). Проще реализовать при неочевидном порядке вычисления.

Как определить время работы DP?::Количество уникальных подзадач × количество выборов на каждую подзадачу.

Как восстановить саму LIS-последовательность?::Дополнительный массив `prev[i]` — индекс предыдущего элемента в LIS. После заполнения: найти i с max dp[i], затем следовать по prev[] назад до -1.

Как работает Robot in Grid с мемоизацией?::Рекурсия DFS (вправо или вниз). `failed` — Set клеток, которые ведут только в тупики. Если путь из клетки не найден → добавить в failed. Избегает повторной работы за O(rc).

Как построить Power Set через DP?::Base-Case-and-Build: начать с {{}}. Для каждого нового элемента x добавить ко всем существующим подмножествам копию с x. O(n·2ⁿ).

Как решить Coin Change (подсчёт способов) через DP?::2D таблица dp[amount][denomination]. dp[0][j]=1. dp[i][j] = dp[i][j-1] (без j-й монеты) + dp[i-coins[j]][j] (с j-й монетой). Размер: (amount+1)×(len(coins)+1).

Рекуррентное соотношение LCS?::Если x[i]==y[j]: c[i][j] = c[i-1][j-1] + 1. Иначе: c[i][j] = max(c[i][j-1], c[i-1][j]). Сложность O(mn).

Как оптимизировать память Fibonacci DP?::fib(n) зависит только от двух предыдущих → хранить только два значения → O(1) память вместо O(n).

Как распознать DP-задачу?::Задача просит оптимизировать/максимизировать/минимизировать; подсчитать количество способов; найти n-й элемент; каждый выбор порождает подзадачу того же вида.

0-1 Knapsack — рекуррентное соотношение?::dp[i][w] = max(dp[i-1][w], dp[i-1][w-weight[i]] + value[i]) — не берём или берём i-й предмет.

Почему подзадачи в DP должны быть автономны (independent)?::Решение одной подзадачи не должно влиять на другую. Если выбор для одной подзадачи ограничивает выбор для другой — DP неприменим напрямую.

Чем отличается Fractional Knapsack от 0-1 Knapsack?::Fractional — можно дробить предметы → жадный алгоритм (sort by value/weight). 0-1 — нельзя дробить → только DP.

Чем отличается LCS от Longest Common Substring?::LCS — подпоследовательность (символы не смежные), max в c[m][n]. Substring — подстрока (символы подряд), при несовпадении c[i][j]=0, max ищется глобально по всей таблице.
