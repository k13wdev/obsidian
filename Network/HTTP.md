# HTTP (HyperText Transfer Protocol)

## Суть и контекст

HTTP — протокол прикладного уровня для передачи гипертекстовых документов и данных в веб. Работает по схеме «запрос-ответ»: клиент (браузер) инициирует, сервер возвращает объекты. HTTP — протокол без сохранения состояния (stateless).

---

## Ключевые идеи

- HTTP/1.0 — одно соединение на один запрос/ответ
- HTTP/1.1 — persistent connections + pipelining
- HTTP/2 — бинарный формат, мультиплексирование, сжатие заголовков, Server Push
- HTTP/3 — поверх QUIC (UDP), устраняет head-of-line blocking
- HTTP stateless — состояние поддерживается через cookies
- Кэширование: Cache-Control, ETag, If-Modified-Since / 304 Not Modified

---

## Определения и термины

**HTTP** — HyperText Transfer Protocol; протокол прикладного уровня для передачи данных в веб по схеме запрос-ответ поверх TCP.

**Persistent connection** — одно TCP-соединение для множества запросов/ответов (HTTP/1.1+).

**Pipelining** — отправка следующего запроса до получения ответа на предыдущий.

**Мультиплексирование (HTTP/2)** — несколько параллельных запросов в одном TCP-соединении.

**Server Push (HTTP/2)** — сервер отправляет ресурсы клиенту до того, как тот их запросил.

**QUIC** — транспортный протокол поверх UDP; основа HTTP/3.

**Cookie** — идентификатор, отправляемый сервером через `Set-Cookie` и возвращаемый браузером в заголовке `Cookie` для поддержания сессии.

**ETag** — уникальный тег контента для проверки его актуальности.

## Как устроено

### Версии HTTP

| Версия | Ключевые особенности |
|---|---|
| **HTTP/1.0** | Одно соединение — один запрос/ответ; TCP разрывается после каждого ответа |
| **HTTP/1.1** | Persistent connections; pipelining; поддержка виртуальных хостов (Host) |
| **HTTP/2** | Бинарный формат; мультиплексирование; сжатие заголовков (HPACK); Server Push |
| **HTTP/3** | Поверх QUIC (UDP); быстрое установление соединений; нет head-of-line blocking |

### Структура HTTP-запроса

```
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
(пустая строка)
[тело — только для POST/PUT]
```

### Структура HTTP-ответа

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 1234
Cache-Control: max-age=3600
ETag: "abc123"
(пустая строка)
<html>...</html>
```

### Методы HTTP

| Метод | Описание |
|---|---|
| **GET** | Запрос объекта с сервера |
| **POST** | Отправка данных на сервер (формы, API) |
| **PUT** | Загрузка или замена объекта по URL |
| **DELETE** | Удаление объекта на сервере |
| **HEAD** | Аналог GET, но возвращает только заголовки (без тела) |
| **OPTIONS** | Запрос информации о поддерживаемых методах и параметрах соединения |

### Коды ответов

| Диапазон | Категория | Примеры |
|---|---|---|
| **1xx** | Информационные | 100 Continue |
| **2xx** | Успех | 200 OK, 201 Created, 204 No Content |
| **3xx** | Перенаправление | 301 Moved Permanently, 302 Found, 304 Not Modified |
| **4xx** | Ошибка клиента | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found |
| **5xx** | Ошибка сервера | 500 Internal Server Error, 502 Bad Gateway, 505 HTTP Version Not Supported |

## Детали и нюансы

### Cookies и сессии

HTTP — stateless протокол. Для идентификации пользователей:
1. Сервер отправляет `Set-Cookie: id=12345` в ответе
2. Браузер сохраняет cookie
3. Браузер возвращает `Cookie: id=12345` во всех последующих запросах к этому домену
4. Сервер идентифицирует пользователя и поддерживает сессию (корзина покупок, авторизация)

### Кэширование

| Механизм | Описание |
|---|---|
| **Cache-Control** | Директивы управления кэшем (`max-age`, `no-cache`, `private`) |
| **ETag** | Уникальный тег версии контента; клиент отправляет в `If-None-Match` |
| **Last-Modified** | Дата изменения; клиент отправляет в `If-Modified-Since` |
| **304 Not Modified** | Сервер говорит: «Используй кэш, контент не изменился» |

## Сравнения и trade-offs

### HTTP/1.1 vs HTTP/2 vs HTTP/3

| Аспект | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| Транспорт | TCP | TCP | QUIC (UDP) |
| Формат | Текстовый | Бинарный | Бинарный |
| Параллельность | Pipelining (с проблемами) | Мультиплексирование | Мультиплексирование |
| Заголовки | Без сжатия | HPACK-сжатие | QPACK-сжатие |
| Server Push | Нет | Да | Да |
| Head-of-line blocking | Есть (TCP) | Есть (TCP) | Нет (QUIC потоки независимы) |

## Важно / подводные камни / best practices

1. **HTTP stateless** — каждый запрос независим; состояние — только через cookies/сессии
2. **Persistent connections по умолчанию в HTTP/1.1** — `Connection: keep-alive`
3. **304 Not Modified** — не тратит трафик, но требует запрос к серверу; кэш не всегда избавляет от RTT
4. **HEAD** — полезен для проверки доступности ресурса без скачивания тела
5. **GET не должен изменять состояние** — идемпотентный метод

#flashcards/network/http

Что такое HTTP?::HyperText Transfer Protocol — протокол прикладного уровня для передачи данных в веб по схеме запрос-ответ поверх TCP. Stateless протокол.

В чём ключевые отличия HTTP/1.0, 1.1, 2, 3?
?
- **HTTP/1.0** — одно соединение на запрос
- **HTTP/1.1** — persistent connections, pipelining
- **HTTP/2** — бинарный, мультиплексирование, Server Push, сжатие заголовков
- **HTTP/3** — поверх QUIC (UDP), без head-of-line blocking

Какие методы HTTP существуют?
?
- **GET** — запрос объекта
- **POST** — отправка данных
- **PUT** — загрузка/замена объекта
- **DELETE** — удаление
- **HEAD** — только заголовки
- **OPTIONS** — поддерживаемые методы

Что означают коды 2xx, 3xx, 4xx, 5xx?
?
- **2xx** — успех (200 OK, 201 Created)
- **3xx** — перенаправление (301, 304 Not Modified)
- **4xx** — ошибка клиента (400, 401, 403, 404)
- **5xx** — ошибка сервера (500, 502)

Как работают cookies в HTTP?::Сервер отправляет Set-Cookie в ответе. Браузер сохраняет и возвращает Cookie в последующих запросах. Это позволяет поддерживать состояние (сессию) поверх stateless протокола.

Что такое ETag и зачем нужен?::Уникальный тег версии контента. Клиент отправляет If-None-Match с ETag — если контент не изменился, сервер возвращает 304 Not Modified без тела.

Что такое Server Push в HTTP/2?::Сервер может отправить ресурсы клиенту до того, как тот их явно запросил — например, CSS и JS вместе с HTML.
