# TCP (Transmission Control Protocol)

## Суть и контекст

TCP — надёжный транспортный протокол интернета с установлением логического соединения. Обеспечивает передачу данных между прикладными процессами в виде неструктурированного, но упорядоченного потока байтов. Преобразует ненадёжную службу IP в надёжную.

---

## Ключевые идеи

- TCP гарантирует безошибочность, упорядоченность и отсутствие пропусков/дубликатов
- Перед передачей данных — трёхстороннее рукопожатие (SYN → SYN-ACK → ACK)
- Закрытие соединения — четырёхстороннее (FIN → ACK → FIN → ACK + TIME_WAIT)
- Управление потоком (flow control) — sliding window с rwnd от получателя
- Управление перегрузкой — slow start + AIMD (cwnd)
- Заголовок TCP — 20 байт (без опций)
- Потеря сегмента обнаруживается по тайм-ауту или 3 дублирующим ACK

---

## Определения и термины

**TCP** — надёжный, ориентированный на соединение транспортный протокол; передаёт данные как непрерывный поток байтов.

**SYN** — флаг сегмента TCP для инициации соединения (synchronize sequence numbers).

**ACK** — флаг подтверждения; номер подтверждения (ack) = следующий ожидаемый байт.

**FIN** — флаг завершения соединения.

**RTT (Round-Trip Time)** — время кругового обхода: время от отправки сегмента до получения ACK.

**cwnd (congestion window)** — окно перегрузки; ограничивает количество неподтверждённых данных в сети.

**rwnd (receive window)** — окно приёма; сообщается получателем; ограничивает объём данных, не дожидаясь ACK.

**ssthresh (slow start threshold)** — порог медленного старта; при cwnd ≥ ssthresh переходим к линейному росту.

**MSS (Maximum Segment Size)** — максимальный размер сегмента данных TCP.

**TIME_WAIT** — состояние после отправки последнего ACK; ожидание 2MSL для гарантированного завершения.

**SYN flood** — атака: злоумышленник заваливает сервер SYN-запросами без завершения рукопожатия. Защита — SYN cookies.

## Как устроено

### Трёхстороннее рукопожатие (Three-way Handshake)

```
Клиент                          Сервер
   │  SYN (SYN=1, seq=client_isn) │
   │ ─────────────────────────────►│
   │                               │
   │ SYN-ACK (SYN=1, ACK=1,       │
   │  seq=server_isn,              │
   │  ack=client_isn+1)            │
   │ ◄─────────────────────────────│
   │                               │
   │  ACK (ACK=1,                  │
   │   ack=server_isn+1)           │
   │ ─────────────────────────────►│
   │                               │
   │       ESTABLISHED             │
```

1. Клиент отправляет SYN (SYN=1), выбирает начальный порядковый номер `client_isn`
2. Сервер отвечает SYN-ACK (SYN=1, ACK=1), подтверждает `ack=client_isn+1`, сообщает `server_isn`
3. Клиент отправляет ACK (`ack=server_isn+1`) — соединение установлено (ESTABLISHED)

### Четырёхстороннее закрытие соединения

```
Инициатор                       Получатель
   │  FIN                          │
   │ ─────────────────────────────►│
   │  ACK                          │
   │ ◄─────────────────────────────│  (может продолжать слать данные)
   │  FIN                          │
   │ ◄─────────────────────────────│
   │  ACK                          │
   │ ─────────────────────────────►│
   │  TIME_WAIT (2MSL)             │
```

### Структура TCP-сегмента (заголовок — 20 байт без опций)

| Поле | Размер | Описание |
|---|---|---|
| Порт отправителя | 16 бит | Идентифицирует сокет приложения-отправителя |
| Порт получателя | 16 бит | Идентифицирует сокет приложения-получателя |
| Порядковый номер | 32 бита | Номер первого байта данных сегмента в потоке |
| Номер подтверждения | 32 бита | Следующий ожидаемый байт (накопительный ACK) |
| Длина заголовка | 4 бита | Размер заголовка в 32-разрядных словах |
| Флаги | — | URG, ACK, PSH, RST, SYN, FIN |
| Размер окна (rwnd) | 16 бит | Свободное место в буфере получателя |
| Контрольная сумма | 16 бит | Проверка целостности заголовка + данных |
| Указатель срочности | 16 бит | Смещение к последнему байту URG-данных |
| Опции | переменная | MSS, масштаб окна и др. |

### Управление потоком (Flow Control)

Предотвращает переполнение буфера получателя:

- Получатель сообщает значение **rwnd** (receive window) = свободное место в буфере
- Отправитель гарантирует: `неподтверждённые данные ≤ rwnd`
- Если получатель объявил `rwnd = 0` — отправитель периодически шлёт пробные сегменты для проверки освобождения буфера

### Управление перегрузкой (Congestion Control)

#### Медленный старт (Slow Start)
```
cwnd = 1 MSS
Каждый RTT: cwnd × 2 (экспоненциальный рост)
До тех пор, пока cwnd < ssthresh
```

#### AIMD — предотвращение перегрузки
```
При cwnd ≥ ssthresh:
  cwnd += 1 MSS за каждый RTT (линейный рост)

При потере по тайм-ауту:
  ssthresh = cwnd / 2
  cwnd = 1 MSS → повтор slow start

При 3 дублирующих ACK (TCP Reno / Fast Retransmit):
  ssthresh = cwnd / 2
  cwnd = ssthresh
  → Fast Recovery → линейный рост
```

## Детали и нюансы

### Флаги TCP

| Флаг | Назначение |
|---|---|
| **SYN** | Инициация соединения |
| **ACK** | Подтверждение данных |
| **FIN** | Завершение соединения |
| **RST** | Сброс соединения (аварийное закрытие) |
| **PSH** | Немедленно передать данные приложению |
| **URG** | Срочные данные (используется редко) |

### Порядковые номера и ACK

- Порядковый номер — номер **первого байта** сегмента в общем потоке
- ACK (номер подтверждения) — номер **следующего ожидаемого байта** (накопительный)
- Начальные порядковые номера (`client_isn`, `server_isn`) выбираются случайно при handshake

## Сравнения и trade-offs

### TCP vs UDP

| Характеристика | TCP | UDP |
|---|---|---|
| Соединение | С установлением (3-way handshake) | Без соединения |
| Надёжность | Гарантированная доставка | Ненадёжная |
| Поток данных | Байтовый поток | Сообщения (сохраняет границы) |
| Управление потоком | Есть (rwnd) | Нет |
| Управление перегрузкой | Есть (cwnd, AIMD) | Нет |
| Заголовок | 20+ байт | 8 байт |
| Задержка | Выше (handshake, подтверждения) | Ниже |
| Применение | HTTP, SMTP, FTP, SSH | DNS, VoIP, потоковое видео, игры |

## Важно / подводные камни / best practices

1. **SYN flood** — атака использует уязвимость трёхстороннего рукопожатия; защита — SYN cookies
2. **TIME_WAIT** — состояние после закрытия длится 2MSL; может приводить к нехватке портов при высокой нагрузке
3. **Медленный старт на каждое новое соединение** — HTTP/1.1 создаёт новое соединение на каждый запрос, что влечёт повторный slow start; HTTP/2 решает это через мультиплексирование
4. **ACK — накопительный** — подтверждает все байты до указанного номера, не отдельные сегменты
5. **3 дублирующих ACK → Fast Retransmit** — не ждём тайм-аута, сразу ретрансмит

Что такое TCP и какие гарантии он даёт?
?
TCP — надёжный транспортный протокол с установлением соединения. Гарантирует:
- Безошибочность данных
- Упорядоченность сегментов
- Отсутствие пропусков и дубликатов

Опиши трёхстороннее рукопожатие TCP
?
1. Клиент → SYN (SYN=1, seq=client_isn)
2. Сервер → SYN-ACK (SYN=1, ACK=1, seq=server_isn, ack=client_isn+1)
3. Клиент → ACK (ack=server_isn+1) → соединение ESTABLISHED

Опиши четырёхстороннее закрытие TCP
?
1. Инициатор → FIN
2. Получатель → ACK (может продолжать слать данные)
3. Получатель → FIN
4. Инициатор → ACK → TIME_WAIT (2MSL)

Что такое flow control в TCP?::Механизм согласования скорости отправителя со скоростью чтения получателем через поле rwnd (receive window). Отправитель не должен превышать rwnd неподтверждёнными данными.

Что такое slow start в TCP?::Начальная фаза управления перегрузкой: cwnd начинается с 1 MSS и удваивается каждый RTT до достижения порога ssthresh.

Что происходит при обнаружении потери по тайм-ауту в TCP?::ssthresh = cwnd/2, cwnd сбрасывается до 1 MSS, повтор slow start.

Что такое Fast Retransmit в TCP?::При получении 3 дублирующих ACK — немедленная повторная отправка потерянного сегмента без ожидания тайм-аута. ssthresh = cwnd/2, cwnd = ssthresh (TCP Reno).

Какие флаги есть в TCP-заголовке?::SYN (инициация), ACK (подтверждение), FIN (завершение), RST (сброс), PSH (протолкнуть данные), URG (срочность).

Чем отличается порядковый номер от номера подтверждения в TCP?::Порядковый номер — номер первого байта данных сегмента в потоке. Номер подтверждения (ACK) — номер следующего ожидаемого байта (накопительный).

Что такое TIME_WAIT в TCP?::Состояние после отправки последнего ACK при закрытии соединения. Длится 2MSL для гарантированного завершения обмена.
