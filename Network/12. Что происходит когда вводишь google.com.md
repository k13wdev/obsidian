## Суть и контекст

Ввод адреса в браузере запускает цепочку сетевых взаимодействий, которая задействует все уровни модели OSI — от прикладного (DNS, HTTP) до физического (передача битов по кабелю). Рассматривается только сетевая часть: от резолвинга имени до получения HTTP-ответа.

---

## Ключевые идеи

- DNS переводит доменное имя в IP-адрес через иерархию серверов
- ARP связывает IP-адрес с MAC-адресом для передачи кадра в локальной сети
- TCP устанавливает надёжное соединение через трёхстороннее рукопожатие
- TLS поверх TCP шифрует канал до отправки HTTP-запроса
- HTTP-запрос инкапсулируется вниз по уровням OSI и декапсулируется на сервере
- Все 7 уровней OSI участвуют в обработке одного запроса к google.com

---

## Как устроено: полная цепочка

### Шаг 0. Обработка ввода в браузере

- Нажатие клавиши вызывает аппаратное прерывание ОС
- Браузер запускает механизм автоподстановки из кэша/истории (зависит от режима «инкогнито»)
- Не-ASCII символы домена конвертируются в **Punycode** перед DNS-запросом
- DNS-сервер может вернуть десяток IP-адресов по принципу **Round Robin** — клиент выбирает первый
- Chrome пытается согласовать апгрейд до **SPDY/HTTP2** вместо HTTP/1.1

---

### Шаг 1. DNS-резолвинг

**Цель:** преобразовать `google.com` → IP-адрес.

**Иерархия поиска (от быстрого к медленному):**
1. Кэш браузера
2. Кэш ОС
3. Файл `/etc/hosts`
4. Рекурсивный резолвер (DNS-сервер провайдера или 8.8.8.8)
5. Корневые серверы (Root servers) — 13 групп, возвращают адреса TLD-серверов
6. TLD-серверы (`.com`) — возвращают адреса авторитетных серверов Google
7. Авторитетные серверы Google — возвращают финальный IP

**Протокол:** UDP, порт 53.

**Типы записей в ответе:**
- `A` — IPv4-адрес
- `AAAA` — IPv6-адрес
- `CNAME` — псевдоним (alias)

Каждая запись содержит **TTL** (Time to Live) — время хранения в кэше резолверов.

---

### Шаг 2. ARP (если MAC неизвестен)

**Задача:** перед отправкой IP-пакета (с DNS-запросом или TCP-сегментом) нужно знать MAC-адрес шлюза (default gateway) или локального DNS-сервера.

**Процесс:**
1. Хост отправляет **широковещательный ARP-запрос**: «У кого IP X.X.X.X?»
2. Целевое устройство отвечает своим **MAC-адресом**
3. MAC заносится в **ARP-таблицу** отправителя
4. Только после этого IP-пакет инкапсулируется в Ethernet-кадр и отправляется

---

### Шаг 3. TCP: трёхстороннее рукопожатие

**Параметры подключения:**
- Браузер выбирает **эфемерный порт** (1024–65535)
- Порт назначения: **80** (HTTP) или **443** (HTTPS)

**Три шага:**

| Шаг | Отправитель | Флаги | Содержимое |
|-----|-------------|-------|------------|
| 1 | Клиент → Сервер | `SYN` | ISN клиента (`client_isn`), выбранный случайно |
| 2 | Сервер → Клиент | `SYN + ACK` | ISN сервера (`server_isn`), ACK = `client_isn + 1` |
| 3 | Клиент → Сервер | `ACK` | ACK = `server_isn + 1` |

После шага 3 соединение переходит в состояние **ESTABLISHED**.

**Порядковые номера (Sequence numbers):** 32-битные числа, отслеживают каждый байт данных — обеспечивают правильный порядок доставки и исключают дубликаты.

---

### Шаг 4. TLS-рукопожатие (только для HTTPS, порт 443)

**Цель:** установить шифрованный канал до отправки HTTP-запроса.

#### Фазы TLS-рукопожатия:

**1. ClientHello** (Клиент → Сервер)
- Поддерживаемые наборы шифров (cipher suites)
- Версия TLS
- Случайное число клиента **RA**

**2. ServerHello** (Сервер → Клиент)
- Выбранный cipher suite
- Случайное число сервера **RB**

**3. Certificate**
- Сервер отправляет **X.509-сертификат**
- Клиент проверяет подлинность через **цепочку доверия CA** (Certificate Authority)

**4. Обмен ключами**
- Клиент генерирует **Pre-Master Secret (PMS)**
- Шифрует его **открытым ключом сервера** и отправляет

**5. Генерация сеансовых ключей**
- Обе стороны независимо вычисляют **Master Secret (MS)** из PMS + RA + RB
- MS нарезается на **4 симметричных ключа**:
  - 2 ключа шифрования (по одному для каждого направления)
  - 2 ключа MAC (для проверки целостности)

**6. Change Cipher Spec**
- Клиент сигнализирует, что дальнейшие данные будут зашифрованы

**7. Finished**
- Обе стороны обмениваются сообщениями **Finished**, содержащими MAC всех предыдущих handshake-сообщений — защита от подмены во время рукопожатия

---

### Шаг 5. HTTP-запрос

Браузер формирует запрос и отправляет через установленный (и зашифрованный) канал:

```
GET / HTTP/1.1
Host: google.com
User-Agent: ...
Cookie: ...
```

#### Инкапсуляция (движение вниз по уровням OSI):

| Уровень | Действие |
|---------|----------|
| Прикладной (L7) | Формируется HTTP-сообщение `GET /` |
| Представления (L6) | TLS шифрует сообщение |
| Сеансовый (L5) | TLS управляет состоянием сессии |
| Транспортный (L4) | HTTP-сообщение разбивается на TCP-сегменты |
| Сетевой (L3) | Сегменты упаковываются в IP-пакеты (src IP, dst IP) |
| Канальный (L2) | Пакеты инкапсулируются в Ethernet-кадры (MAC-адрес шлюза через ARP) |
| Физический (L1) | Кадры передаются как биты — электрические, оптические или радиосигналы |

---

### Шаг 6. Обработка на сервере

#### Обработка на HTTPD-сервере (детально)

1. Получение запроса, разбор: Метод, Домен (Host), Путь
2. Проверка существования виртуального хоста и разрешений метода
3. Проверка прав доступа клиента (IP, аутентификация)
4. Применение модулей перезаписи (mod_rewrite / URL Rewrite)
5. Поиск локального контента; если динамический — запуск интерпретатора (PHP)
6. Отправка результата клиенту

- Веб-сервер (Nginx, Apache, Google Web Server) принимает пакеты
- Проводит **декапсуляцию**: снимает заголовки L1 → L2 → L3 → L4 → L6 → L7
- Передаёт HTTP-запрос приложению

**HTTP Response:**
```
HTTP/1.1 200 OK
Content-Type: text/html
...

<html>...</html>
```

Браузер получает ответ, рендерит страницу, запрашивает дополнительные объекты (CSS, JS, изображения) — повторяя цикл DNS → TCP → HTTP для каждого.

---

### Шаг 7. Завершение соединения

**4-way handshake:**
1. Инициатор → `FIN`
2. Получатель → `ACK`
3. Получатель → `FIN`
4. Инициатор → `ACK` → состояние **TIME_WAIT** (2MSL)

**Persistent connections (HTTP/1.1):** соединение может оставаться открытым для последующих запросов — без повторного TCP-рукопожатия.

## Детали и нюансы

### Уровни OSI: полная таблица участия

| Уровень OSI | Протокол/Механизм | Роль |
|-------------|-------------------|------|
| L7 — Прикладной | DNS, HTTP/HTTPS | Резолвинг имени, формирование GET-запроса |
| L6 — Представления | TLS, X.509 | Шифрование/дешифрование, форматы сертификатов |
| L5 — Сеансовый | TLS session | Управление состоянием сессии |
| L4 — Транспортный | TCP | 3-way handshake, порты, sequence numbers, ретрансмиссия |
| L3 — Сетевой | IP, BGP | IP-адресация, глобальная маршрутизация |
| L2 — Канальный | Ethernet, ARP | Кадрирование, MAC-адреса |
| L1 — Физический | Кабель / оптика / Wi-Fi | Передача битов по физической среде |

### HTTP/1.1 vs HTTP/2

| Характеристика | HTTP/1.1 | HTTP/2 |
|----------------|----------|--------|
| Формат | Текстовый | Бинарный |
| Соединения | 1 запрос / соединение или pipelining | Мультиплексирование по одному TCP |
| Head-of-line blocking | Есть | Устранено |
| Сжатие заголовков | Нет | Есть (HPACK) |
| Server Push | Нет | Есть |

## Важно / подводные камни

- **ARP-спуфинг:** злоумышленник может подменить MAC-адрес в ARP-ответе — MitM-атака на канальном уровне
- **ISN рандомизация:** начальные порядковые номера TCP выбираются случайно — защита от атак предсказания последовательности
- **TLS Finished:** MAC всех handshake-сообщений защищает от подмены cipher suite на слабый (downgrade attack)
- **DNS кэширование:** устаревший кэш может отдавать неактуальный IP — TTL критичен
- **Persistent connections:** без `Connection: close` браузер не закрывает TCP-соединение сразу — экономия на повторных рукопожатиях

Что такое Pre-Master Secret (PMS) в TLS?::Случайное число, сгенерированное клиентом, зашифрованное открытым ключом сервера и отправленное серверу для выработки общего секрета

Что проверяет сообщение TLS Finished?::MAC всех предыдущих handshake-сообщений — защита от downgrade-атаки (подмены параметров во время рукопожатия)

Какой уровень OSI отвечает за TLS?::L6 (Представления) — шифрование/дешифрование и формат X.509 сертификатов; L5 (Сеансовый) — управление состоянием TLS-сессии

Какую роль играет L2 (Канальный) при запросе к google.com?::Инкапсуляция в Ethernet-кадры, MAC-адресация, ARP для определения MAC-адреса шлюза
