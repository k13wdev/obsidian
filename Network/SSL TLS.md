## Суть и контекст

SSL (Secure Sockets Layer) и его преемник TLS (Transport Layer Security) — криптографические протоколы, обеспечивающие защищённую передачу данных в сети. TLS располагается между транспортным (TCP) и прикладным уровнями. В модели OSI относится к уровню представления.

---

## Ключевые идеи

- SSL — устаревший предшественник TLS; современный стандарт — TLS 1.2 и TLS 1.3
- TLS обеспечивает три свойства: конфиденциальность, целостность, аутентификация
- Handshake: асимметричное шифрование для согласования → симметричное для данных
- Сертификаты X.509 — стандарт формата; выдаются CA в рамках PKI
- HSTS — механизм принудительного использования HTTPS

---

## Определения и термины

**SSL (Secure Sockets Layer)** — криптографический протокол, предшественник TLS. SSL 3.0 → TLS 1.0 → TLS 1.2 → TLS 1.3.

**TLS (Transport Layer Security)** — современный криптографический протокол; работает поверх TCP, под прикладными протоколами (HTTP, SMTP, FTP).

**Симметричное шифрование** — один и тот же ключ для шифрования и расшифровки. Быстрое; используется для шифрования данных.

**Асимметричное шифрование** — пара ключей: публичный (открытый) и приватный (закрытый). Медленное; используется для согласования ключей и подписей.

**Master Secret** — общий секрет, выработанный клиентом и сервером в ходе handshake; на его основе создаются сеансовые ключи.

**Сеансовые ключи** — симметричные ключи, созданные на основе Master Secret; используются для шифрования данных сессии.

**X.509** — стандарт формата цифровых сертификатов.

**CA (Certificate Authority)** — доверенная организация, выдающая и подписывающая X.509-сертификаты.

**PKI (Public Key Infrastructure)** — система CA и протоколов для управления цифровыми сертификатами и подписями.

**Cipher Suite** — набор алгоритмов: обмен ключами + шифрование + MAC (Message Authentication Code).

**Diffie-Hellman** — протокол обмена ключами, позволяющий установить симметричные сеансовые ключи по незащищённому каналу. Уязвим для атаки посредника без аутентификации.

**Pre-Master Secret (PMS)** — случайное число, сгенерированное клиентом, зашифрованное публичным ключом сервера и отправленное для выработки общего секрета.

**SNI (Server Name Indication)** — поле TLS для идентификации посещаемого сайта. В TLS 1.2 передаётся открытым текстом; в TLS 1.3 — шифруется.

**HSTS (HTTP Strict Transport Security)** — заголовок HTTP, принуждающий браузер всегда использовать HTTPS для данного домена.

## Как устроено

### Место TLS в стеке протоколов

```
┌─────────────────────┐
│  Прикладной уровень │  HTTP, SMTP, FTP...
├─────────────────────┤
│       TLS/SSL       │  ← Шифрование, аутентификация, целостность
├─────────────────────┤
│  Транспортный (TCP) │
├─────────────────────┤
│  Сетевой (IP)       │
└─────────────────────┘
```

В модели OSI — **уровень представления** (6).

### TLS Handshake

```
Клиент                                     Сервер
  │  1. Client Hello                          │
  │     — версия TLS                          │
  │     — cipher suites                       │
  │     — случайное число (client_random)     │
  │ ──────────────────────────────────────────►│
  │                                           │
  │  2. Server Hello                          │
  │     — выбранная версия TLS                │
  │     — выбранный cipher suite              │
  │     — случайное число (server_random)     │
  │     — сертификат X.509                    │
  │ ◄──────────────────────────────────────────│
  │                                           │
  │  3. Проверка сертификата                  │
  │     — цепочка: сертификат → CA → root CA  │
  │     — срок действия                       │
  │     — соответствие домена                 │
  │                                           │
  │  4. Обмен ключами (Key Exchange)          │
  │     — асимметричное шифрование            │
  │     — клиент шифрует pre-master secret    │
  │       публичным ключом сервера            │
  │ ──────────────────────────────────────────►│
  │                                           │
  │  5. Генерация Master Secret               │
  │     — обе стороны вычисляют независимо   │
  │     — Master Secret = f(pre-master,       │
  │       client_random, server_random)       │
  │                                           │
  │  6. Генерация сеансовых ключей            │
  │     — 4 ключа из Master Secret            │
  │     — EA (шифрование клиента)             │
  │     — MA (MAC клиента)                    │
  │     — EB (шифрование сервера)             │
  │     — MB (MAC сервера)                    │
  │                                           │
  │  7. Finished (зашифрованные)              │
  │ ◄────────────────────────────────────────►│
  │                                           │
  │  8. Данные шифруются симметрично          │
  │ ◄────────────────────────────────────────►│
```

### Генерация сеансовых ключей (детально)

Master Secret = f(Pre-Master Secret, Client Random, Server Random)

MS разделяется на **4 сеансовых ключа**:
- Ключ шифрования клиента (EA)
- MAC-ключ клиента (MA)
- Ключ шифрования сервера (EB)
- MAC-ключ сервера (MB)

Раздельные ключи для каждого направления обеспечивают защиту от replay-атак.

### Сертификаты и PKI

**Структура X.509-сертификата:**
- Публичный ключ владельца
- Доменное имя (Common Name / SAN)
- Срок действия (Not Before / Not After)
- Подпись CA (удостоверяет подлинность)
- Информация об издателе

**Цепочка доверия:**
```
Root CA (самоподписанный, встроен в браузер/ОС)
    └── Intermediate CA (подписан Root CA)
            └── Сертификат сайта (подписан Intermediate CA)
```

Браузер проверяет каждое звено цепочки до доверенного Root CA.

### HSTS

Сервер отправляет заголовок:
```
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

- Браузер запоминает: только HTTPS для этого домена в течение `max-age` секунд
- Защищает от SSL stripping (атака понижения до HTTP)
- Защищает от кражи cookie через HTTP

## Детали и нюансы

### Эволюция SSL/TLS

| Версия | Статус |
|---|---|
| SSL 2.0 | Устарел, уязвим |
| SSL 3.0 | Устарел (POODLE-атака) |
| TLS 1.0 | Устарел |
| TLS 1.1 | Устарел |
| TLS 1.2 | Актуален, широко поддерживается |
| TLS 1.3 | Актуален, рекомендуется; 1 RTT (TLS 1.2 — 2 RTT) |

### Diffie-Hellman и обмен ключами

Классический обмен (RSA): клиент шифрует PMS публичным ключом сервера → сервер расшифровывает приватным. Компрометация приватного ключа раскрывает все прошлые сессии.

Diffie-Hellman (ECDHE): обе стороны генерируют эфемерные ключи → Forward Secrecy. TLS 1.3 требует только DH-обмен (RSA key exchange удалён).

### SSL v3 → TLS

Между SSL v3 и TLS есть несовместимость в способе вычисления ключа сеанса. TLS стандартизован IETF (RFC 5246).

### Симметричное vs Асимметричное шифрование

| Аспект | Симметричное | Асимметричное |
|---|---|---|
| Ключи | Один общий | Публичный + приватный |
| Скорость | Быстрое | Медленное |
| Применение в TLS | Шифрование данных | Согласование ключей, подписи |
| Примеры | AES, ChaCha20 | RSA, ECDH, Ed25519 |

## Сравнения и trade-offs

| Аспект | SSL | TLS 1.2 | TLS 1.3 |
|---|---|---|---|
| Статус | Устарел | Актуален | Рекомендуется |
| RTT handshake | 2 | 2 | 1 (0-RTT для повторных) |
| Forward Secrecy | Нет | Опционально | Обязательно |
| Поддерживаемые алгоритмы | Слабые | Широкий выбор | Только современные |

## Важно / подводные камни / best practices

1. **SSL устарел** — не использовать SSL 2.0/3.0 и TLS 1.0/1.1; минимум TLS 1.2, лучше TLS 1.3
2. **Forward Secrecy** — TLS 1.3 обязательно поддерживает; даже при компрометации приватного ключа прошлые сессии не расшифровать
3. **HSTS без includeSubDomains** — субдомены остаются уязвимы к SSL stripping
4. **Самоподписанные сертификаты** — браузер не доверяет без добавления в хранилище доверенных; не для продакшена
5. **Срок действия** — сертификаты истекают; Let's Encrypt автоматизирует обновление

#flashcards/network/ssl

Чем TLS отличается от SSL?::SSL — устаревший предшественник. TLS — современный стандарт (TLS 1.2 и 1.3). SSL имеет критические уязвимости (POODLE и др.).

Какие три свойства обеспечивает TLS?::1. Конфиденциальность (шифрование). 2. Целостность (MAC/подписи). 3. Аутентификация (сертификаты X.509).

Почему в TLS используется сначала асимметричное, потом симметричное шифрование?
?
- Асимметричное (медленное) — для безопасного согласования Master Secret
- Симметричное (быстрое) — для шифрования самих данных
- Оптимальный баланс безопасности и производительности

Что такое Master Secret в TLS?::Общий секрет, выработанный клиентом и сервером в ходе handshake. На его основе генерируются симметричные сеансовые ключи.

Что такое цепочка доверия в PKI?::Root CA → Intermediate CA → сертификат сайта. Каждое звено подписано вышестоящим CA. Браузер доверяет Root CA (встроен в ОС/браузер).

Чем TLS 1.3 лучше TLS 1.2?::1 RTT вместо 2 RTT для handshake (0-RTT для повторных). Обязательный Forward Secrecy. Только современные алгоритмы. Удалены устаревшие cipher suites.

Что такое Forward Secrecy?::Свойство: компрометация приватного ключа сервера не позволяет расшифровать прошлые сессии. Достигается через эфемерные ключи (ECDHE). В TLS 1.3 — обязательно.

Что такое HSTS и от чего защищает?::HTTP Strict Transport Security — заголовок, принуждающий браузер использовать только HTTPS. Защищает от SSL stripping (понижения до HTTP) и кражи cookie.

Что такое Cipher Suite?::Набор алгоритмов: алгоритм обмена ключами + алгоритм шифрования + MAC. Клиент и сервер согласуют cipher suite в ходе TLS handshake.

Что такое Pre-Master Secret?::Случайное число, генерируемое клиентом в TLS. Шифруется публичным ключом сервера. Обе стороны вычисляют из него Master Secret = f(PMS, Client Random, Server Random).

Сколько сеансовых ключей генерируется из Master Secret?::4 ключа: ключ шифрования клиента (EA), MAC-ключ клиента (MA), ключ шифрования сервера (EB), MAC-ключ сервера (MB).

Что такое SNI?::Server Name Indication — поле TLS для идентификации посещаемого сайта. В TLS 1.2 — открытый текст. В TLS 1.3 — шифруется.

Чем RSA key exchange отличается от Diffie-Hellman в TLS?::RSA: клиент шифрует PMS публичным ключом сервера. Компрометация ключа раскрывает прошлые сессии. DH (ECDHE): эфемерные ключи → Forward Secrecy. TLS 1.3 использует только DH.
