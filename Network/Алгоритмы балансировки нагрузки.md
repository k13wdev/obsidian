# Алгоритмы балансировки нагрузки

## Суть и контекст

Балансировка нагрузки — распределение входящего трафика между несколькими серверами или сетевыми путями. Цель: равномерное использование ресурсов, предотвращение перегрузки отдельных узлов, обеспечение отказоустойчивости. Применяется в веб-инфраструктуре, CDN, дата-центрах и сетевой маршрутизации.

---

## Ключевые идеи

- Балансировщик должен отслеживать отказы узлов и немедленно прекращать отправку трафика на вышедшие из строя серверы (Health Checks)
- Sticky Sessions необходимы для корректной работы веб-запросов в рамках одной TCP-сессии
- IP Hash — единственный алгоритм из рассмотренных, гарантирующий Sticky Sessions по природе
- DNS и IP Anycast реализуют балансировку на уровне глобальной маршрутизации
- Traffic Engineering решает задачу глобального равномерного использования сетевых ресурсов

---

## Определения и термины

- **Балансировщик нагрузки (Load Balancer)** — устройство или ПО, распределяющее запросы между серверами
- **Sticky Sessions (сеансовая привязка)** — все пакеты одного TCP-соединения/сессии направляются на один и тот же сервер
- **Health Check** — периодическая проверка доступности и работоспособности серверов
- **IP Anycast** — один IP-адрес назначается нескольким кластерам; маршрутизаторы направляют трафик к топологически ближайшему
- **LAG (Link Aggregation Group)** — агрегирование нескольких физических каналов в один логический для увеличения пропускной способности
- **Traffic Engineering** — управление маршрутизацией с целью равномерной загрузки всей сети

---

## Как устроено: алгоритмы

### Round Robin (Карусель)

**Принцип:** серверы перебираются по очереди циклически — каждый следующий запрос уходит к следующему серверу.

- **Плюсы:** простота реализации, равномерное распределение при одинаковых запросах
- **Минусы:** не учитывает текущую загрузку серверов и разницу в мощности

---

### Weighted Round Robin (Взвешенная карусель)

**Принцип:** каждому серверу присваивается вес, пропорциональный его производительности. В каждом цикле сервер обрабатывает количество запросов (или данных), пропорциональное своему весу.

**Пример:** видеосерверу с большей мощностью назначается больший вес, чем файловому серверу.

- **Плюсы:** эффективное использование ресурсов в гетерогенных системах
- **Минусы:** требует ручной настройки весов

---

### Least Connections / Resource Based (Минимум соединений)

**Принцип:** запрос направляется серверу с наименьшим количеством активных задач или минимальным коэффициентом текущей загрузки.

- **Плюсы:** адаптивен к реальному состоянию системы; предотвращает перегрузку отдельных узлов
- **Минусы:** требует постоянного мониторинга состояния всех серверов

---

### Least Response Time (Минимальное время отклика)

**Принцип:** учитывает время ожидания в очереди или задержку отклика сервера. Путь с наименьшим «весом» (временем) считается приоритетным.

- **Когда использовать:** в сетях, чувствительных к задержкам — потоковое видео, VoIP

---

### IP Hash (Хеширование)

**Принцип:** вычисляется хеш-функция от признаков пакета (IP-адрес источника/назначения, порты TCP/UDP). Результат однозначно определяет конкретный сервер или порт в группе агрегирования (LAG).

**Пример:** сервер отвечает только если последние 4 бита IP-источника совпадают с его номером.

- **Плюсы:** обеспечивает **Sticky Sessions** — пакеты одного потока всегда попадают на один и тот же сервер; исключает нарушение порядка данных
- **Минусы:** при малом количестве потоков нагрузка может распределиться неравномерно

---

### Random (Случайный выбор)

**Принцип:** выбор сервера осуществляется случайным образом.

- **Применение:** в сильно связанных топологиях дата-центров для обхода узких мест

---

## Балансировка на уровне сети

### DNS-балансировка

При DNS-запросе сервер возвращает IP-адрес **ближайшей к клиенту реплики** (client mapping). Широко используется в CDN.

### IP Anycast

Одному IP-адресу назначаются несколько кластеров в разных частях мира. Адрес публикуется через **BGP**. Маршрутизаторы автоматически направляют запрос клиента к **топологически ближайшей** точке.

**Преимущества:**
- Распределённое «впитывание» трафика при DDoS-атаках
- Минимальная задержка для клиента

### Датаграммный баланс

В таблице маршрутизации для одного адреса назначения указывается **несколько следующих хопов** — пакеты распределяются между ними.

## Сравнения и trade-offs

| Алгоритм | Учитывает загрузку | Sticky Sessions | Сложность | Лучше для |
|----------|-------------------|-----------------|-----------|-----------|
| Round Robin | Нет | Нет | Низкая | Однородные запросы |
| Weighted Round Robin | Частично (вес статичный) | Нет | Средняя | Гетерогенные серверы |
| Least Connections | Да (динамически) | Нет | Высокая | Долгие соединения |
| Least Response Time | Да (задержка) | Нет | Высокая | Latency-sensitive (VoIP, видео) |
| IP Hash | Нет | **Да** | Средняя | Сессионные приложения |
| Random | Нет | Нет | Низкая | Mesh-топологии ДЦ |
| DNS Anycast | Нет (гео-близость) | Нет | Высокая (инфра) | CDN, глобальные сервисы |

## Детали и нюансы

### Sticky Sessions

Необходимы для корректной работы веб-запросов: балансировщик должен «помнить» соответствие клиент → сервер, чтобы все последующие пакеты в рамках одного TCP-соединения уходили на тот же сервер.

Без Sticky Sessions возможна ситуация: два пакета одной сессии попадают на разные серверы → некорректная обработка.

### Health Checks

Балансировщик обязан:
1. Фиксировать отказы узлов
2. Немедленно прекращать отправку запросов на вышедшие из строя серверы

**Health Checks в SDN:** контроллер использует «измерители» (meters) для резервирования ресурсов и проверки состояния путей в реальном времени.

### Traffic Engineering

Глобальная задача определения маршрутов так, чтобы:
- Ресурсы сети были загружены равномерно
- Коэффициенты использования не превышали заданных порогов (например, **0,5** для трафика реального времени)

## Важно / подводные камни / best practices

- **IP Hash и малое число потоков:** при немногих клиентах хеширование может дать неравномерное распределение — один сервер перегружен, остальные простаивают
- **Без Health Checks** балансировщик продолжает слать трафик на упавший сервер — часть запросов теряется
- **Round Robin без весов** на гетерогенных серверах — слабые серверы перегружаются быстрее мощных
- **Threshold 0,5** для real-time трафика в Traffic Engineering — жёсткое требование: при превышении QoS деградирует

Перечисли основные алгоритмы балансировки нагрузки::Round Robin, Weighted Round Robin, Least Connections (Resource Based), Least Response Time, IP Hash, Random, DNS Anycast

Как работает Round Robin?::Серверы перебираются по очереди циклически — каждый следующий запрос уходит к следующему серверу в списке

Как работает Weighted Round Robin?::Каждому серверу назначается вес. В каждом цикле сервер обрабатывает количество данных, пропорциональное своему весу. Мощный сервер получает больше запросов.

Как работает Least Connections?::Запрос направляется серверу с наименьшим числом активных соединений или минимальной загрузкой. Требует мониторинга состояния всех серверов.

Как работает Least Response Time?::Запросы направляются на сервер с наименьшим временем отклика (задержкой). Приоритет — путь с минимальным «весом». Используется для latency-sensitive трафика (VoIP, видео).

Как работает IP Hash и какое его ключевое свойство?::Хеш от IP/портов источника и назначения однозначно определяет сервер. Ключевое свойство: **Sticky Sessions** — все пакеты одного потока всегда попадают на один сервер.

Что такое Sticky Sessions и зачем нужны?::Привязка всех пакетов одной TCP-сессии к одному серверу. Необходима для корректной обработки: без неё пакеты одной сессии могут попасть на разные серверы.

Что такое IP Anycast?::Один IP-адрес назначается нескольким кластерам мира, публикуется через BGP. Маршрутизаторы направляют трафик к топологически ближайшему кластеру. Используется в CDN и для защиты от DDoS.

Чем DNS-балансировка отличается от IP Anycast?::DNS возвращает разные IP разным клиентам (ближайшую реплику). Anycast — один IP у всех кластеров, выбор делают маршрутизаторы через BGP.

Что такое Health Check и почему критичен?::Периодическая проверка работоспособности серверов. Без него балансировщик продолжает слать трафик на упавший сервер — запросы теряются.

Какой порог коэффициента использования задаётся для трафика реального времени в Traffic Engineering?::0,5 — коэффициент использования канала не должен превышать 50% для сетей с требованиями к QoS.

Когда IP Hash даёт неравномерное распределение?::При малом количестве потоков (клиентов): хеш может отобразить большинство потоков на один сервер, остальные простаивают.
