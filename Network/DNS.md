## Суть и контекст

DNS — распределённая база данных, реализованная с помощью иерархии серверов имён, и протокол прикладного уровня, который обеспечивает трансляцию символьных имён хостов в числовые IP-адреса. Маршрутизаторы используют IP-адреса, пользователям удобнее работать с мнемоническими именами — DNS является мостом между ними. Критически важный компонент интернет-инфраструктуры.

---

## Ключевые идеи

- DNS — это распределённая иерархическая база данных, не централизованная
- Пространство имён имеет древовидную структуру, начинающуюся с безымянного корня (точка)
- Существует 4 типа DNS-серверов: root, TLD, authoritative, recursive resolver
- Запросы выполняются итеративно: локальный сервер сам обходит иерархию
- Ответы кэшируются; TTL определяет время жизни записи в кэше
- Атака Каминского позволяет отравить кэш целой зоны
- DNSSEC защищает от спуфинга с помощью цифровых подписей

## Определения и термины

**DNS (Domain Name System)** — распределённая база данных и протокол прикладного уровня для преобразования доменных имён в IP-адреса и обратно.

**DNS-запись ресурса (Resource Record, RR)** — запись в базе DNS формата (Name, Value, Type, TTL).

**TTL (Time-to-Live)** — время в секундах, в течение которого запись может считаться действительной в кэше DNS-сервера.

**Рекурсивный распознаватель (Recursive Resolver)** — DNS-сервер, принимающий запрос от хоста и самостоятельно выполняющий итеративный обход иерархии DNS для получения ответа.

**Авторитетный сервер (Authoritative Server)** — сервер, хранящий официальные записи ресурсов для конкретной зоны и возвращающий окончательный ответ.

**DNS-спуфинг** — атака, при которой злоумышленник навязывает поддельный IP-адрес, перехватывая ответ DNS.

**DNSSEC** — набор стандартов для защиты DNS с помощью цифровых подписей и ключей проверки подлинности ответов.

**Атака Каминского** — вид DNS-спуфинга, позволяющий отравить кэш для целой зоны путём подмены связующих записей (glue records).

**SOA (Start Of Authority)** — запись, содержащая параметры зоны: имя первичного источника, e-mail администратора, serial, тайм-ауты (refresh, retry, expire).

**SRV** — запись для идентификации хоста как определённого сервиса в домене; обобщение MX для других служб.

**RRSIG** — запись DNSSEC с криптографическим хешем набора записей (RRSET), подписанным закрытым ключом зоны.

**DNSKEY** — запись DNSSEC с открытым ключом зоны и алгоритмом подписи. Заменяет X.509 в DNS из-за громоздкости.

**DoT (DNS-over-TLS) / DoH (DNS-over-HTTPS)** — протоколы, направляющие DNS-трафик по зашифрованному каналу. Решают подслушивание DNS-запросов, но создают риск централизации.

**DDNS (Dynamic DNS)** — опция для автоматического обновления DNS-записей через DNS-сообщения (RFC 2136, 3007); согласует адресную базу при DHCP.

## Как устроено

### Иерархия доменных имён

Древовидная структура, начинающаяся с безымянного корня (обозначается точкой `.`):

```
. (root)
├── .com (TLD)
│   ├── google.com (2nd level)
│   └── amazon.com
├── .edu (TLD)
│   └── uchicago.edu
│       └── cs.uchicago.edu (subdomain)
└── .ru (ccTLD)
    └── yandex.ru
```

- **TLD (Top-Level Domain):** располагаются под корнем
  - **gTLD** — родовые: .com, .edu, .org
  - **ccTLD** — национальные: .ru, .uk
- **Домены второго уровня:** регистрируются организациями (google.com, yandex.ru)
- **Поддомены:** создаются организацией самостоятельно (cs.vu.nl)

### Типы DNS-серверов

| Тип | Функция |
|---|---|
| **Корневые (Root)** | Знают адреса всех TLD-серверов. 13 логических имён (a–m.root-servers.net), реплицированных физически по всему миру |
| **TLD-серверы** | Отвечают за зону TLD (.com, .edu и др.), направляют к авторитетным серверам |
| **Авторитетные (Authoritative)** | Хранят официальные записи для конкретной зоны, возвращают окончательный ответ |
| **Рекурсивный резолвер (Recursive Resolver)** | Обычно предоставляется провайдером; принимает запрос от клиента и обходит иерархию итеративно |

### Процесс резолвинга DNS-запроса

Пример: поиск IP-адреса для `www.cs.uchicago.edu`

```
1. Приложение вызывает stub resolver на хосте пользователя
       ↓
2. Stub resolver → Локальный DNS-сервер (рекурсивный резолвер)
       ↓ (кэш пуст)
3. Локальный сервер → Корневой сервер
   ← Корневой: "Спроси TLD .edu: [IP-адреса серверов .edu]"
       ↓
4. Локальный сервер → TLD-сервер .edu
   ← TLD: "Спроси авторитетный сервер uchicago.edu: [IP]"
       ↓
5. Локальный сервер → Авторитетный сервер uchicago.edu
   ← Авторитетный: "Спроси cs.uchicago.edu: [IP]"
       ↓
6. Локальный сервер → Авторитетный сервер cs.uchicago.edu
   ← Возвращает IP-адрес www.cs.uchicago.edu
       ↓
7. Локальный сервер → Хост пользователя (возвращает IP)
   + Сохраняет результат в кэш
```

## Детали и нюансы

### Типы DNS-записей

| Тип | Назначение | Пример |
|---|---|---|
| **A** | Имя хоста → IPv4-адрес | `example.com → 93.184.216.34` |
| **AAAA** | Имя хоста → IPv6-адрес | `example.com → 2606:2800::1` |
| **NS** | Домен → имя авторитетного DNS-сервера | `example.com → ns1.example.com` |
| **CNAME** | Псевдоним → каноническое имя хоста | `www.example.com → example.com` |
| **MX** | Домен → хост для приёма почты | `example.com → mail.example.com` |
| **TXT** | Произвольный текст (SPF, DKIM, верификация) | `"v=spf1 include:..."` |
| **PTR** | IP-адрес → имя (обратный поиск, зона in-addr.arpa) | `34.216.184.93.in-addr.arpa → example.com` |

### Кэширование и TTL

- Серверы имён кэшируют результаты запросов для снижения нагрузки и задержек
- TTL определяет срок жизни записи в кэше (в секундах)
- Формат записи ресурса: `(Name, Value, Type, TTL)`

### DNS-спуфинг и DNSSEC

**DNS-спуфинг:**
- Злоумышленник отправляет поддельный UDP-ответ раньше легального сервера
- Навязывает распознавателю ложный IP-адрес для перехвата трафика
- **Атака Каминского** — отравляет кэш для целой зоны, подменяя glue records в ответе

**DNSSEC:**
- Цифровые подписи и ключи для проверки подлинности ответов
- Позволяет убедиться, что запись получена от авторитетного источника и не изменена
- **RRSIG** — цифровая подпись набора записей
- **DNSKEY** — открытый ключ зоны

### Формат DNS-сообщения (Запрос/Ответ)

| Поле | Размер |
|---|---|
| Идентификатор транзакции | 16 бит |
| Флаги (QR, AA, RD, RA) | 16 бит |
| Число вопросов | 16 бит |
| Число ответных RR | 16 бит |
| Число RR авторитетного источника | 16 бит |
| Число дополнительных RR | 16 бит |
| Секция Вопросов (QNAME, Тип) | переменная |
| Секция Ответов (RR) | переменная |
| Секция Авторитетных источников | переменная |
| Секция Дополнительной информации | переменная |

### Зонная передача (Zone Transfer)

Вторичные серверы запрашивают файл зоны через запрос типа **AXFR**. Из-за большого объёма передача осуществляется по **TCP** (порт 53).

### Работа DNSSEC (пошагово)

1. Сервер группирует записи одного типа и имени в RRSET
2. Хеширует RRSET и подписывает закрытым ключом зоны → RRSIG
3. Открытый ключ помещается в DNSKEY; DNSKEY подписывается родительским доменом
4. Клиент запрашивает RRSET + RRSIG + DNSKEY
5. Использует DNSKEY для расшифровки RRSIG и хеширует RRSET → при совпадении данные подлинные

### Защита от спуфинга: рандомизация

Идентификатор транзакции — всего 16 бит (1:65536). Дополнительная защита: рандомизация UDP-порта отправителя и **0x20-кодирование** (случайный регистр букв в QNAME, например `wWw.GooGle.Com`).

### QNAME-минимизация (RFC 7816)

Локальный резолвер отправляет серверам верхних уровней только ту часть имени, за которую они отвечают (серверу .edu — только `uchicago.edu`, а не `www.cs.uchicago.edu`). Защищает конфиденциальность FQDN.

### EDNS Client Subnet (EDNS0 CS)

Публичный резолвер (8.8.8.8) вставляет в DNS-запрос IP-подсеть реального клиента → авторитетный CDN-сервер определяет ближайший узел кэширования.

## Сравнения и trade-offs

| Аспект | Итеративный запрос | Рекурсивный запрос |
|---|---|---|
| Кто обходит иерархию | Локальный сервер | Каждый сервер в цепочке |
| Нагрузка | На локальный сервер | Распределена по иерархии |
| Применение | Стандарт для серверов | Используется клиентом (stub resolver) |

## Важно / подводные камни / best practices

1. **DNS — не централизованная система** — распределённая база с иерархической организацией
2. **UDP по умолчанию** — DNS использует UDP порт 53; TCP — для зонных передач и ответов > 512 байт
3. **Кэш отравляется** — DNS-спуфинг возможен без DNSSEC
4. **TTL баланс** — низкий TTL: быстрые обновления, высокая нагрузка; высокий TTL: меньше запросов, медленное распространение изменений
5. **13 корневых имён** — не 13 серверов физически; каждое обслуживается множеством реплик через Anycast
6. **16-бит ID транзакции** — малая энтропия; дополняется рандомизацией порта и 0x20-кодированием
7. **Открытые рекурсивные серверы** — используются для DDoS Amplification; коэффициент усиления DNS до 54.6×
8. **DoT/DoH** — решают подслушивание, но создают риск централизации DNS в браузерах
