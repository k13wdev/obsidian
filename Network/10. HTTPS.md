## Суть и контекст

HTTPS — защищённая версия HTTP, работающая поверх TLS (Transport Layer Security). Обеспечивает конфиденциальность, целостность данных и аутентификацию сервера. По умолчанию использует порт 443.

---

## Ключевые идеи

- HTTPS = HTTP поверх TLS (ранее SSL)
- Обеспечивает три свойства: конфиденциальность, целостность, аутентификация
- Данные шифруются симметричным шифрованием; ключ согласуется через асимметричное
- Сертификат X.509 подтверждает подлинность сервера; выдаётся CA
- HSTS — защита от атак понижения протокола

---

## Определения и термины

**HTTPS** — HTTP Secure; HTTP поверх TLS. Порт 443.

**TLS (Transport Layer Security)** — криптографический протокол для защиты передачи данных; преемник SSL.

**Сертификат X.509** — стандарт формата цифрового сертификата, содержащего публичный ключ и удостоверяющего подлинность владельца.

**CA (Certificate Authority)** — центр сертификации; доверенная организация, выдающая и подписывающая цифровые сертификаты.

**PKI (Public Key Infrastructure)** — система CA и протоколов для управления цифровыми сертификатами и подписями.

**HSTS (HTTP Strict Transport Security)** — механизм, принуждающий браузер использовать только HTTPS, защищая от атак понижения протокола и кражи cookie.

**SNI (Server Name Indication)** — поле TLS для идентификации посещаемого сайта. В TLS 1.2 — открытый текст; в TLS 1.3 — шифруется.

**Симметричное шифрование** — один ключ для шифрования и расшифровки (быстрое, используется для данных).

**Асимметричное шифрование** — пара ключей: публичный + приватный (медленное, используется для согласования ключей).

## Как устроено

### TLS Handshake (рукопожатие)

```
Клиент                                  Сервер
  │  Client Hello                         │
  │  (версия TLS, cipher suites,          │
  │   случайное число клиента)            │
  │ ─────────────────────────────────────►│
  │                                       │
  │  Server Hello                         │
  │  (выбранный алгоритм,                 │
  │   случайное число сервера,            │
  │   сертификат X.509)                   │
  │ ◄─────────────────────────────────────│
  │                                       │
  │  Проверка сертификата через CA        │
  │  (браузер проверяет цепочку доверия)  │
  │                                       │
  │  Обмен ключами                        │
  │  (асимметричное шифрование →          │
  │   согласование Master Secret)         │
  │ ─────────────────────────────────────►│
  │ ◄─────────────────────────────────────│
  │                                       │
  │  Генерация сеансовых ключей           │
  │  (на основе Master Secret)            │
  │                                       │
  │  Данные шифруются симметрично         │
  │  (сеансовые ключи — быстро)           │
  │ ◄────────────────────────────────────►│
```

**Шаги handshake:**
1. **Client Hello** — клиент сообщает версию протокола и поддерживаемые алгоритмы шифрования (cipher suites)
2. **Server Hello** — сервер выбирает алгоритм и отправляет цифровой сертификат X.509
3. **Аутентификация** — клиент проверяет сертификат через CA (цепочка доверия)
4. **Обмен ключами** — асимметричное шифрование для безопасной передачи Master Secret
5. **Сеансовые ключи** — MS разделяется на **4 ключа**: ключ шифрования клиента (EA), MAC-ключ клиента (MA), ключ шифрования сервера (EB), MAC-ключ сервера (MB)
6. **Передача данных** — данные шифруются симметрично (быстрее, чем асимметрично)

### Сертификаты и PKI

```
Root CA (самоподписанный)
    └── Intermediate CA
            └── Сертификат сервера (example.com)
                    ├── Публичный ключ
                    ├── Доменное имя
                    ├── Срок действия
                    └── Подпись CA
```

- Браузер содержит список доверенных корневых CA
- Проверяется вся цепочка: сертификат сервера → intermediate CA → root CA

## Сравнения и trade-offs

### HTTP vs HTTPS

| Аспект | HTTP | HTTPS |
|---|---|---|
| Порт | 80 | 443 |
| Шифрование | Нет | TLS (симметричное + асимметричное) |
| Аутентификация | Нет | Сертификат X.509 от CA |
| Целостность | Нет | Цифровые подписи |
| Производительность | Быстрее | Незначительно медленнее (TLS handshake) |
| SEO | Хуже | Лучше (Google предпочитает HTTPS) |

## Важно / подводные камни / best practices

1. **HSTS** — без HSTS браузер может быть обманут на HTTP-соединение атакой понижения протокола (SSL stripping)
2. **Смешанный контент (Mixed Content)** — если HTTPS-страница загружает ресурсы по HTTP — браузер блокирует или предупреждает
3. **Срок действия сертификата** — истёкший сертификат блокируется браузером с ошибкой
4. **Самоподписанные сертификаты** — не доверяются браузерами по умолчанию (нет CA в цепочке)
5. **TLS handshake добавляет RTT** — TLS 1.2: 2 RTT; TLS 1.3: 1 RTT (0-RTT для повторных соединений)
