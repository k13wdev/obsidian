## Суть и контекст

Замыкания — одна из важнейших концепций JavaScript. Понимание замыканий необходимо для написания надёжного, инкапсулированного и асинхронного кода. В JS замыкания не нужно создавать специально — они возникают естественно каждый раз при создании функции. Технически, любая функция в JS образует замыкание.

---

## Ключевые идеи

- Замыкание — комбинация функции и лексического окружения, в котором эта функция была объявлена
- Замыкания возникают автоматически при каждом создании функции — специально создавать не нужно
- Функция **не копирует** значения переменных в момент создания — она захватывает их **по ссылке** и берёт текущее значение при обращении
- `LexicalEnvironment` — скрытый объект с переменными текущего вызова
- [[02. Scope (область видимости)|Scope]] — неизменная ссылка на лексическое окружение, в котором функция была **создана**
- Поиск переменной идёт по цепочке [[02. Scope (область видимости)|Scope]] снизу вверх до глобального объекта
- `var` не имеет блочной области видимости — classic loop bug
- `let`/`const` имеют блочную область; `let` в `for` создаёт отдельный binding для каждой итерации
- Замыкания удерживают весь `LexicalEnvironment` в памяти — источник утечек

---

## Определения и термины

**Замыкание (closure)** — комбинация функции и лексического окружения, в котором эта функция была объявлена. Ситуация, когда функция способна запоминать свою лексическую область видимости и иметь к ней доступ даже тогда, когда она вызывается вне этой области видимости.

**LexicalEnvironment (лексическое окружение)** — скрытый внутренний объект, создаваемый при запуске функции. В него записываются все аргументы, вложенные функции и локальные переменные текущего вызова.

**[[02. Scope (область видимости)|Scope]]** — скрытое системное свойство функции. Устанавливается при создании функции (ещё до выполнения) и ссылается на лексическое окружение, в котором функция была создана. Эта ссылка неизменна и навсегда привязывает функцию к месту её «рождения».

**Scope Chain (цепочка областей видимости)** — последовательность `LexicalEnvironment`-объектов от текущего контекста до глобального. Выстраивается через свойство [[02. Scope (область видимости)|Scope]].

**IIFE (Immediately Invoked Function Expression)** — немедленно вызываемое функциональное выражение. Функция оборачивается в круглые скобки (превращается в выражение) и немедленно вызывается. Создаёт изолированную область видимости.

**TDZ (Temporal Dead Zone — Временная мёртвая зона)** — период между поднятием (`hoisting`) переменных `let`/`const` и строкой их явного объявления. В этот период к переменной нельзя обратиться — выбрасывается ошибка.

**Hoisting (поднятие)** — механизм, при котором объявления переменных и функций перемещаются (концептуально) в начало своей области видимости. `var` поднимается с инициализацией `undefined`; `let`/`const` поднимаются, но попадают в TDZ.

## Как устроено

### Создание LexicalEnvironment и [Scope (область видимости)|Scope]]

При запуске функции:
1. Создаётся новый объект `LexicalEnvironment`, в который записываются все аргументы, вложенные функции и локальные переменные
2. Этот объект получает ссылку на внешнее окружение — то, на которое указывает [[02. Scope (область видимости)|Scope]] функции

При **создании** функции (ещё до вызова):
- Функция получает скрытое свойство [[02. Scope (область видимости)|Scope]], указывающее на текущее `LexicalEnvironment`
- Эта ссылка неизменна — функция навсегда связана с местом создания

### Алгоритм поиска переменной (Scope Chain)

1. Проверить текущий `LexicalEnvironment` функции
2. Если переменная не найдена — перейти по [[02. Scope (область видимости)|Scope]] во внешнее окружение
3. Повторять до глобального объекта (`window` в браузере)

### Захват переменных по ссылке

Функция не копирует значение переменной в момент создания. Значение берётся **текущее** в момент обращения:

```javascript
var phrase = 'Привет';
function sayHi(name) {
  alert(phrase + ', ' + name);
}
sayHi('Вася'); // Привет, Вася
phrase = 'Пока';
sayHi('Вася'); // Пока, Вася (берется текущее значение)
```

## Детали и нюансы

### var vs let/const

**`var`:**
- Нет блочной области видимости — ограничен только функцией
- Поднимается (`hoisting`) в верх функции с инициализацией `undefined`
- В цикле `for (var i ...)` — одна переменная `i` для всех итераций

**`let`/`const`:**
- Строгая блочная область видимости
- Поднимаются, но попадают в TDZ — до строки объявления обратиться нельзя
- В цикле `for (let i ...)` — для каждой итерации создаётся **своя независимая** переменная `i`

### V8 оптимизация памяти

Современные движки (Chrome, Node.js) умеют оптимизировать замыкания: если переменная абсолютно точно не используется во внутренних функциях, движок удаляет её из памяти. Поэтому при отладке в Chrome такая переменная может быть недоступна (`undefined`).

### Несколько экземпляров замыкания

При каждом вызове внешней функции создаётся **абсолютно новый** `LexicalEnvironment`. Это означает, что два вызова одной фабричной функции создают два **полностью независимых** замыкания. Изменения в одном никак не влияют на другое.

---

## Сравнения и trade-offs

| | `var` | `let` / `const` |
|---|---|---|
| Область видимости | Функциональная | Блочная |
| Hoisting | Поднимается с `undefined` | Поднимается, но в TDZ |
| В цикле `for` | Одна переменная для всех итераций | Своя переменная на каждую итерацию |
| Classic loop bug | Да — все замыкания видят последнее `i` | Нет — каждое замыкание видит своё `i` |

## Примеры

### Захват по ссылке (текущее значение)

```javascript
var phrase = 'Привет';
function sayHi(name) {
  alert(phrase + ', ' + name);
}
sayHi('Вася'); // Привет, Вася
phrase = 'Пока';
sayHi('Вася'); // Пока, Вася
```

### Фабрика функций

```javascript
function makeAdder(x) {
  return function(y) {
    return x + y; // замыкает переменную x
  };
}
var add5 = makeAdder(5);
var add10 = makeAdder(10);
console.log(add5(2));  // 7
console.log(add10(2)); // 12
```

`add5` и `add10` имеют одинаковое тело, но хранят разные лексические окружения: в одном `x = 5`, в другом `x = 10`.

### Модульный паттерн (приватные переменные через IIFE)

```javascript
var counter = (function() {
  var privateCounter = 0; // Приватная переменная
  function changeBy(val) { privateCounter += val; } // Приватная функция

  return { // Публичный API
    increment: function() { changeBy(1); },
    decrement: function() { changeBy(-1); },
    value: function() { return privateCounter; }
  };
})();

counter.increment();
counter.increment();
console.log(counter.value()); // 2
```

Методы `increment`, `decrement` и `value` разделяют одно лексическое окружение. Внешний код не может изменить `privateCounter` напрямую.

### Частичное применение (каррирование)

```javascript
function sum(a) {
  return function(b) {
    return a + b; // a берется из внешнего LexicalEnvironment
  };
}
alert( sum(1)(2) ); // 3
```

### Classic loop bug (var)

```javascript
for (var i = 0; i < 5; i++) {
  setTimeout(function() {
    console.log(i);
  }, 100);
}
// Выведет: 5, 5, 5, 5, 5
```

Все 5 функций замыкаются на **одну и ту же** переменную `i`. К моменту срабатывания таймеров цикл завершён, `i = 5`.

### Решение 1: IIFE

```javascript
for (var i = 0; i < 5; i++) {
  (function(j) {
    setTimeout(function() { console.log(j); }, 100);
  })(i);
}
```

IIFE создаёт новую область видимости для каждой итерации, копируя текущее значение `i` в параметр `j`.

### Решение 2: let (современный подход)

```javascript
for (let i = 0; i < 5; i++) {
  setTimeout(function() { console.log(i); }, 100);
}
// Выведет: 0, 1, 2, 3, 4
```

`let` перепривязывается для каждой итерации, сохраняя правильное значение.

### Замыкание в setTimeout (колбэк)

```javascript
function wait(message) {
  setTimeout(function timer() {
    console.log(message); // Замыкание сохраняет доступ к message
  }, 1000);
}
wait("Привет!");
```

Движок браузера удерживает ссылку на `timer`. `timer` удерживает своё замыкание, не давая сборщику мусора удалить `message`.

### Замыкание в addEventListener

```javascript
function hide() {
  var event = new Event("hide", { cancelable: true });
  rabbit.addEventListener('hide', function(event) {
    if (confirm("Вызвать preventDefault?")) {
      event.preventDefault();
    }
  });
  rabbit.dispatchEvent(event);
}
```

Анонимная функция, переданная в `addEventListener`, образует замыкание и успешно обращается к внешним данным.

### Несколько независимых экземпляров (makeCounter)

```javascript
function makeCounter() {
  var currentCount = 1; // Приватная переменная
  return {
    getNext: function() { return currentCount++; },
    set: function(value) { currentCount = value; }
  };
}

var counter1 = makeCounter();
var counter2 = makeCounter(); // независимый экземпляр
```

Каждый вызов `makeCounter()` создаёт новый `LexicalEnvironment`. `counter1` и `counter2` абсолютно независимы.

### Мемоизация через замыкание

```javascript
function makeCaching(f) {
  var cache = {}; // Кэш скрыт в замыкании
  return function(x) {
    if (!(x in cache)) { // Если результата еще нет
      cache[x] = f.call(this, x); // Вычисляем и сохраняем
    }
    return cache[x]; // Возвращаем из кэша
  };
}
```

Обёртка имеет эксклюзивный доступ к приватному `cache` и сохраняет его в памяти между вызовами.

### Утечка памяти — проблема

```javascript
function f() {
  var data = "Большой объем данных..."; // Занимает много памяти
  function inner() {
    // Внутренняя функция удерживает ВСЁ замыкание f()
  }
  return inner;
}
```

### Утечка памяти — решение

```javascript
function f() {
  var data = "Большой объем данных...";
  function inner() { /* ... */ }
  data = null; // Явно очищаем память от тяжелых данных
  return inner;
}
```

## Важно / подводные камни / best practices

- **`var` в цикле + замыкание = classic loop bug.** Все функции читают одно финальное значение `i`. Решение: `let` или IIFE.
- **Функция захватывает переменную по ссылке**, а не её значение на момент создания. Значение будет текущим в момент обращения.
- **Каждый вызов внешней функции создаёт новый `LexicalEnvironment`.** Несколько вызовов — несколько независимых замыканий.
- **Утечки памяти:** `LexicalEnvironment` живёт, пока есть хоть одна ссылка через [[02. Scope (область видимости)|Scope]]. Большие данные явно обнулять (`data = null`).
- **Циклические ссылки (старые браузеры IE8-):** DOM-элемент ↔ JS-объект через замыкание — память не очищается. Решение: явный разрыв ссылки (`xhr = null`).
- **V8 оптимизация:** если переменная не используется во внутренних функциях, движок может удалить её. При отладке такая переменная может быть `undefined`.
- **Неоправданное создание функций внутри функций** влияет на производительность и потребление памяти.
- **`recover` работает только внутри `defer`** — не относится к JS, игнорировать.

**var vs let в замыканиях**

В чём разница var и let при использовании в цикле for с замыканием?
?
`var`: одна переменная для всех итераций — все замыкания читают финальное значение
`let`: каждая итерация получает свой независимый binding — каждое замыкание видит правильное значение

Classic loop bug: что выведет код?
```javascript
for (var i = 0; i < 5; i++) {
  setTimeout(function() { console.log(i); }, 100);
}
```
?
5, 5, 5, 5, 5 — все функции замкнуты на одну переменную i, цикл уже завершён

Два способа исправить classic loop bug
?
1. IIFE: обернуть тело цикла в `(function(j) { ... })(i)` — создаёт новую область для каждой итерации
2. Использовать `let` вместо `var` — каждая итерация получает свой binding

**Паттерны**

Паттерн: фабрика функций (function factory)
?
Внешняя функция принимает параметр, возвращает внутреннюю функцию, которая замыкает этот параметр:
`function makeAdder(x) { return function(y) { return x + y; }; }`
`makeAdder(5)` и `makeAdder(10)` — разные замыкания с разными x

Паттерн: модульный (Module pattern)
?
IIFE с приватными переменными внутри + возврат объекта с публичными методами
Методы замыкают приватные переменные; внешний код не может обратиться к ним напрямую

Паттерн: частичное применение / каррирование
?
Функция возвращает функцию, запоминая первый аргумент:
`function sum(a) { return function(b) { return a + b; }; }`
`sum(1)(2) // 3`

Паттерн: мемоизация через замыкание
?
Внешняя функция создаёт приватный объект `cache`; возвращаемая функция при вызове проверяет кэш, при промахе вычисляет и сохраняет результат

**Память и утечки**

Как долго живёт LexicalEnvironment в памяти?::До тех пор, пока на него ссылается хотя бы одна внутренняя функция через [[02. Scope (область видимости)|Scope]]

Как предотвратить утечку памяти при замыкании большой переменной?::Явно обнулить переменную после использования: `data = null` — сборщик мусора сможет освободить память

Edge case: почему при отладке в Chrome переменная из замыкания может быть `undefined`?::V8 оптимизирует: если переменная не используется во внутренних функциях, движок удаляет её из памяти

Edge case: циклические ссылки DOM ↔ JS (старые браузеры IE8-)
?
DOM-элемент ссылается на JS-объект, JS-объект через замыкание ссылается обратно на DOM-элемент — память не очищается.
Решение: разорвать ссылку вручную (`xhr = null`)
