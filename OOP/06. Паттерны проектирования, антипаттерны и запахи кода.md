# Паттерны проектирования, антипаттерны и запахи кода

## Суть и контекст

Паттерны проектирования — повторяющиеся архитектурные решения, помогающие соблюдать принципы SOLID. Антипаттерны и запахи кода — распространённые ошибки проектирования, указывающие на нарушение одного или нескольких принципов. Все паттерны и антипаттерны рассматриваются в контексте ООП и SOLID, с примерами на TypeScript.

---

## Ключевые идеи

- Декоратор расширяет функциональность без изменения базового класса и без наследования
- Прокси контролирует доступ к объекту, предоставляя тот же интерфейс
- Фасад скрывает сложную логику за простым API
- Стратегия динамически выбирает алгоритм из словаря вместо if/else
- Наблюдатель создаёт механизм подписки — инвертирует контроль (IoC)
- Адаптер транслирует несовместимые интерфейсы
- Абстрактная фабрика создаёт сущности без привязки к конкретным реализациям
- Состояние меняет поведение объекта в зависимости от внутреннего состояния
- Божественный объект, Контрол-фрик и Локатор служб — ключевые антипаттерны
- Пустая реализация и instanceof-проверки — основные запахи кода

## Определения и термины

**Декоратор (Decorator)** — шаблон, который заключается в создании обёрток с дополнительной функциональностью для объектов. Позволяет расширять функциональность динамически, без изменения базового класса и без необходимости описывать каждого наследника.

**Прокси (Proxy)** — шаблон проектирования, при котором общение с каким-то объектом контролирует другой объект-заместитель. Расширяет функциональность существующих классов, не меняя их, но всегда предоставляет тот же интерфейс, что и оригинальный объект.

**Фасад (Facade)** — шаблон проектирования, при котором сложная логика скрывается за вызовом более простого API. Берёт ответственность за настройку и вызов специфических методов конкретных объектов на себя.

**Стратегия (Strategy)** — шаблон, позволяющий менять настройки, конфигурацию или алгоритм в зависимости от ситуации и требований. Динамический выбор нужных сущностей (например, из мапы) вместо условий `if/else` или `switch`.

**Наблюдатель (Observer)** — шаблон, который создаёт механизм подписки, когда некоторые сущности могут реагировать на поведение других. Инвертирует контроль за выполнением программы, аналогично обработчикам событий в GUI.

**Адаптер (Adapter)** — шаблон проектирования, который позволяет объектам с несовместимыми интерфейсами работать вместе. Выступает дополнительным слоем для трансляции сообщений, помогая справиться со сломанным API или преобразованием структур данных.

**Абстрактная фабрика (Abstract Factory)** — фабрика, которая создаёт фабрики. Позволяет создавать сущности без привязки к конкретным реализациям классов.

**Состояние (State)** — шаблон, позволяющий менять поведение объектов в зависимости от их внутреннего состояния. Может нарушать LSP, если реализация позволяет выброс новых исключений.

**Божественный объект (God Object)** — сущность, которая отвечает за слишком много. Внутри скапливается неоправданно большое количество данных.

**Контрол-фрик (Control Freak)** — запах кода, при котором зависимости явно создаются в конструкторе с помощью слова `new`. Вызывает жёсткое зацепление, делает класс нетестируемым без глобальных моков.

**Локатор служб (Service Locator)** — шаблон, который позволяет подключать зависимости в момент, когда они становятся нужны. Скрывает зависимости и нарушает инкапсуляцию, IDE не подсказывает нужные аргументы при инициализации.

**Спагетти-код** — в предоставленных источниках не раскрыто.

**instanceof-проверки (switch over types)** — проверка на принадлежность к типу или классу перед выполнением операции. Главный индикатор нарушения OCP и проблем с иерархией наследования.

**Пустая реализация** — запах кода, когда метод оставлен пустым, потому что класс обязан реализовать интерфейс, часть которого ему не нужна. Прямое нарушение ISP и LSP.

## Как устроено

### Механизм Адаптера

1. Создаётся общий интерфейс `Storage` с методом `getItem(key)`
2. Адаптер `SophisticatedStorageAdapter` реализует `Storage`
3. Внутри реализованного метода `getItem` вызывается несовместимое API целевого хранилища `verySophisticatedStorage.getByKey(key)`
4. Класс `StorageService` зависит только от интерфейса `Storage` и вызывает его методы, не зная о деталях реализации

### Механизм Наблюдателя (Observer)

1. Наблюдатели передаются в метод `subscribe` наблюдаемого объекта
2. Наблюдаемый объект сохраняет их во внутренний массив `this.listeners.push(applicant)`
3. При наступлении события вызывается метод `notify` наблюдаемого объекта
4. Метод `notify` перебирает массив подписчиков через `forEach` и синхронно вызывает метод `update` у каждого слушателя

### Механизм Декоратора

1. Оригинальный объект (базовое поведение) передаётся в конструктор декоратора и сохраняется в свойстве `this.decorated`
2. При вызове метода у декоратора, он сначала вызывает метод у сохранённого объекта (`this.decorated.greet()`)
3. Декоратор применяет дополнительную логику к результату (`toUpperCase()`) и возвращает новое значение

## Детали и нюансы

### Фасад → Божественный объект

Слишком активное использование фасадов приводит к тому, что фасад перехватывает управление огромным количеством подсистем. Фасад может легко превратиться в антипаттерн "Божественный объект", аккумулируя избыточную ответственность.

### Адаптер — дополнительная абстракция

Адаптер добавляет ещё одну абстракцию. При создании нового адаптера разработчику необходимо вручную найти и обновить все места, где его требуется использовать. Тем не менее, адаптер корректно транслирует данные из одной структуры в другую (например, XML в JSON).

### Грязный интерфейс — раздувание при добавлении функций

Сущности обязывают реализовывать неиспользуемые методы. Разработчик вынужден писать методы-заглушки (например, `sin(x: number) { return 0 }` для обычного арифметического калькулятора). Это запах "Пустая реализация".

### Состояние и LSP

Неправильная реализация паттерна "Состояние", при которой выбрасываются непредусмотренные исключения, напрямую нарушает принцип подстановки Барбары Лисков.

## Сравнения и trade-offs

| Паттерн | Назначение | Связь с SOLID | Ограничения |
|---------|-----------|--------------|-------------|
| Декоратор | Расширение функциональности обёртками | OCP — расширение без изменения | Может предоставлять расширенный интерфейс |
| Прокси | Контроль доступа к объекту | OCP — расширение без изменения | Обязан предоставлять тот же интерфейс, что и оригинал |
| Фасад | Упрощение сложного API | SRP — одна точка входа | Рискует стать Божественным объектом |
| Стратегия | Динамический выбор алгоритма | OCP — добавление через словарь | — |
| Наблюдатель | Механизм подписки и реакции | IoC — инвертирует контроль | Синхронный вызов подписчиков |
| Адаптер | Трансляция несовместимых интерфейсов | DIP — зависимость от абстракции | Дополнительный слой, ручное обновление мест использования |
| Абстрактная фабрика | Создание объектов без привязки к реализации | OCP/DIP — расширяемость, абстракции | — |
| Состояние | Смена поведения по внутреннему состоянию | Может нарушить LSP | Риск непредусмотренных исключений |

### Декоратор vs Прокси

Декоратор способен предоставлять расширенный интерфейс (добавлять новое поведение), тогда как Прокси обязан предоставлять ровно тот же интерфейс, что и оригинальный объект.

### Наблюдатель vs обработчики событий

Шаблон инвертирует контроль таким же образом, как среда реагирует на пользовательский ввод в графическом интерфейсе (GUI).

## Примеры

### Шаблон "Стратегия" — динамический выбор из словаря (OCP)

```typescript
function formatterStrategy(reportType: ReportTypes) {
  const formatters = {
    [ReportTypes.Html]: HtmlFormatterFactory,
    [ReportTypes.Txt]: TxtFormatterFactory,
    default: TxtFormatterFactory
  }

  return formatters[reportType] || formatters.default
}
```

Функция динамически извлекает нужную фабрику из словаря по ключу `reportType`. При добавлении нового формата достаточно добавить строчку в словарь `formatters`, не изменяя логику маршрутизации.

### Нарушение LSP через паттерн "Состояние"

```typescript
class Published extends Article {
  constructor() {
    super({ status: ArticleStatus.Published })
  }

  publish(): void {
    // Упс!
    throw new Error('article is already published')
  }
}
```

Метод `publish` в дочернем классе выбрасывает необъявленное в родительском классе исключение. Это усиливает предусловия и ломает ожидаемое поведение системы (нарушение LSP).

### Антипаттерн "Контрол-фрик"

```typescript
class Auth {
  // Обратите внимание на отсутствие аргументов у конструктора
  // и ключевое слово `new` внутри:
  constructor() {
    this.connection = new MySqlConnection()
  }
}
```

Зависимость от конкретного низкоуровневого класса (`MySqlConnection`) жёстко зашита внутри конструктора с помощью `new`. Класс невозможно протестировать без создания глобального мока соединения.

## Важно / подводные камни / best practices

- **Абстрактная фабрика и Стратегия** — позволяют создавать зависимости без привязки к конкретным классам. Добавление новых типов изменяет минимальное количество модулей (соответствие OCP)
- **instanceof-проверки** — жёстко привязывают модуль к реализации. Изменение требований заставит менять код модуля. Прямое нарушение OCP
- **Локатор служб** — скрывает зависимости, нарушает прозрачность кода. Использовать только при работе с legacy-кодом
- **Пустая реализация** — указывает на нарушение ISP (зависимость от ненужного) и LSP (метод-заглушка не меняет состояние ожидаемым образом)
- **Фасад** — рискует стать Божественным объектом при чрезмерном использовании

?

3. При событии вызывается метод notify
