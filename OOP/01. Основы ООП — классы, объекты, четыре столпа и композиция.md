# Основы ООП — классы, объекты, четыре столпа и композиция

## Суть и контекст

ООП — инструмент проектирования системы на языке, более близком к языку бизнес-правил. Код в ООП выступает как "карта", а бизнес-требования — как "территория". Если карта точная, система легко переносит изменения территории. ООП описывает отношения между сущностями и их поведение, а не только иерархию.

Конспект охватывает базовые концепции ООП: классы, объекты, конструкторы, четыре столпа (абстракция, наследование, полиморфизм, инкапсуляция), а также композицию как альтернативу наследованию. Примеры приведены на TypeScript.

---

## Ключевые идеи

- ООП описывает отношения между сущностями и их поведение, а не только иерархию.
- Код — "карта", бизнес-требования — "территория". Точная карта позволяет системе легко переносить изменения территории.
- Наследование — проектирование от общего к частному. Длинные цепочки иерархий типов хрупкие.
- Композиция — проектирование от частного к общему. Снимает ограничения единственной иерархии.
- Интерфейсы предпочтительнее абстрактных классов для связывания модулей: для связи достаточно контракта на поведение.
- Проблема "банана и джунглей": при наследовании дочерний класс получает нужную функцию (банан), но вместе с ней наследует кучу неиспользуемого кода от родительских классов (гориллу и джунгли).
- Математическая иерархия объектов реального мира не совпадает с иерархией поведения в коде.
- Полиморфизм и композиция применяются как приём рефакторинга для ослабления зацепления модулей через введение интерфейсов и обёрток.

## Определения и термины

**ООП** — инструмент проектирования системы на языке, более близком к языку бизнес-правил. ООП описывает отношения между сущностями и их поведение, а не только иерархию. Код выступает как "карта", а бизнес-требования — как "территория".

**Абстракция** — базовый тип или интерфейс, играющий роль контракта на поведение. Гарантирует, что экземпляр конкретного класса будет содержать указанные поля и методы. Применяется в местах стыка модулей для ослабления зацепления.

**Наследование** — проектирование от общего к частному в виде иерархии (Животные -> Млекопитающие -> Человек). Длинные цепочки иерархий типов хрупкие. Инструмент приводит к тупику, если иерархий одновременно несколько.

**Композиция** — проектирование от частного к общему в виде совокупности обособленных наборов "фич" (Человек = Скелет + Нервная система + Иммунная система). Разные свойства объектов сочетаются в новом объекте, снимая ограничения единственной иерархии.

**Полиморфизм** — в источниках явного определения нет. Упоминается исключительно как приём рефакторинга ("Замена прямого наследования на полиморфизм и композицию") для ослабления зацепления модулей через введение интерфейсов и обёрток.

**Инкапсуляция** — в источниках явного определения нет. Упоминается "нарушение инкапсуляции состояния" при использовании глобального синглтона и "нарушение инкапсуляции (сокрытие зависимостей)" при использовании локатора служб.

**Класс / Экземпляр / Объект** — часть кода (модуль), обособленная от других и имеющая границы. Экземпляр создаётся через конструктор с ключевым словом `new`. Классы описываются свойствами (полями) и поведением (методами).

**Конструктор** — метод создания экземпляра класса, принимающий зависимости и начальные аргументы. Используется для внедрения зависимостей (Constructor Injection).

**Интерфейс** — абстракция, описывающая контракт на поведение (методы и поля), который обязаны реализовать классы. Делится на роли (тип, поведение, ожидание).

**Абстрактный класс** — базовый тип (суперкласс), описывающий общую реализацию и абстрактные методы для наследников. Имеет предел вложенности. Может содержать protected методы и поля, доступные только наследникам.

**Божественный объект (God Object)** — антипаттерн, возникающий при чрезмерном вынесении общей функциональности в абстрактный суперкласс.

## Как устроено

### Структура класса

Класс описывается свойствами (полями) и поведением (методами). Экземпляр создаётся через конструктор с ключевым словом `new`. Конструктор принимает зависимости и начальные аргументы.

### Наследование через абстрактный класс

Абстрактный класс-родитель задаёт общую реализацию (например, protected-метод `setSide`) и объявляет абстрактные методы, которые наследники обязаны реализовать. Наследник расширяет суперкласс через `extends` и вызывает `super()` в конструкторе. Абстрактный класс ограничивает модель одной иерархией.

### Композиция объектов через TypeScript (алгоритм)

1. Выделить изолированные свойства в отдельные интерфейсы (`interface Shape { areaOf(): number }`, `interface WidthfulShape { setWidth(size: number): void }`)
2. Создать тип, объединяющий нужные интерфейсы через пересечение: `type RectShape = Shape & WidthfulShape & HeightfulShape`
3. Объявить класс, реализующий собранный контракт: `class Rectangle implements RectShape`
4. Реализовать методы, требуемые всеми интерфейсами контракта

Такой подход позволяет собирать поведение объекта из независимых "фич" без привязки к единственной иерархии наследования.

## Детали и нюансы

### Edge case: Квадрат наследуется от Прямоугольника

Попытка скопировать иерархию объектов напрямую из реального мира (например, Квадрат наследуется от Прямоугольника): класс-наследник ослабляет постусловия базового класса. Квадрат меняет обе стороны при вызове `setWidth`, тогда как Прямоугольник ожидал изменения только одной. Математическая иерархия не совпадает с иерархией поведения.

### Edge case: режим "Гостя" в классе User

Сущность не вписывается в существующую модель глубокого наследования (например, режим "Гостя" в классе "Пользователь" User): Гость получает унаследованный метод `updateProfile`, но так как профиля у него нет, метод выбрасывает необъявленное исключение `throw new Error`. Иерархии типов хрупки.

### Проблема "банана и джунглей"

При наследовании дочерний класс получает нужную функцию (банан), но вместе с ней наследует кучу неиспользуемого кода от родительских классов (гориллу и джунгли).

### Антипаттерн: проверки instanceof (switch over types)

Указывает на то, что сущность не вписывается в иерархию наследования, модуль жёстко завязан на конкретной реализации. Ломает полиморфизм.

### Антипаттерн: God Object

Возникает при чрезмерном вынесении общей функциональности в абстрактный суперкласс.

### Инкапсуляция — частичные упоминания

> В источниках упомянуто частично: инкапсуляция упоминается в контексте нарушений — "нарушение инкапсуляции состояния" при использовании глобального синглтона и "нарушение инкапсуляции (сокрытие зависимостей)" при использовании локатора служб. Полного определения не дано.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| **Наследование** | Проектирование от общего к частному, переиспользование общей реализации через суперкласс | Длинные цепочки хрупкие; приводит к тупику при нескольких иерархиях; проблема "банана и джунглей" | Когда иерархия строгая и неглубокая |
| **Композиция** | Проектирование от частного к общему; снимает ограничения единственной иерархии; сочетает разные свойства объектов | В источниках явных минусов не указано | Когда объект сочетает несколько независимых наборов "фич" |
| **Интерфейс** | Чистый абстрактный контракт; позволяет использовать композицию через пересечение типов; для связи модулей достаточно контракта на поведение | В источниках явных минусов не указано | При связывании модулей и ослаблении зацепления |
| **Абстрактный класс** | Задаёт общую реализацию (protected-методы и поля); наследники могут переиспользовать логику | Ограничивает модель одной иерархией; имеет предел вложенности; риск God Object | Когда нужна общая реализация для группы наследников |

## Примеры

### Структура класса

```typescript
class Rectangle {
  width: number
  height: number

  constructor(width: number, height: number) {
    this.width = width
    this.height = height
  }

  setWidth(width: number) {
    this.width = width
  }

  areaOf(): number {
    return this.width * this.height
  }
}
```

### Абстрактный класс и наследование

```typescript
abstract class RightAngleShape {
  protected setSide(size: number, side?: 'width' | 'height'): void {}
  abstract areaOf(): number
}

class Square extends RightAngleShape {
  edge: number
  constructor(size: number) {
    super()
    this.edge = size
  }
  protected setSide(size: number): void {
    this.edge = size
  }
  setWidth(size: number): void {
    this.setSide(size)
  }
  areaOf(): number {
    return this.edge ** 2
  }
}
```

### Композиция через интерфейсы

```typescript
interface Shape { areaOf(): number }
interface WidthfulShape { setWidth(size: number): void }

type SquareShape = Shape & WidthfulShape

class Square implements SquareShape {
  edge: number
  constructor(size: number) { this.edge = size }

  protected setSide(size: number): void { this.edge = size }

  setWidth(size: number) { this.setSide(size) }
  areaOf(): number { return this.edge ** 2 }
}
```

## Важно / подводные камни / best practices

- **Best practice:** использование интерфейсов вместо абстрактных классов при связывании модулей. Для связи модулей достаточно контракта на поведение. Интерфейсы позволяют применять композицию.
- **Антипаттерн:** проверки `instanceof` (switch over types). Указывает на то, что сущность не вписывается в иерархию наследования, модуль жёстко завязан на конкретной реализации. Ломает полиморфизм.
- **Антипаттерн:** God Object. Возникает при чрезмерном вынесении общей функциональности в абстрактный суперкласс.
- **Подводный камень:** математическая иерархия объектов реального мира не совпадает с иерархией поведения в коде. Попытка скопировать иерархию напрямую приводит к ослаблению постусловий базового класса.
- **Подводный камень:** длинные цепочки наследования хрупки. Наследование приводит к тупику, если иерархий одновременно несколько.
- **Подводный камень:** проблема "банана и джунглей" — дочерний класс вместе с нужной функцией наследует кучу неиспользуемого кода от родительских классов.
