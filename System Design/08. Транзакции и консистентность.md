## Суть и контекст

Транзакции — механизм, обеспечивающий корректность операций с данными через гарантии ACID. В распределённых системах задача усложняется: возникают проблемы консистентности между узлами, необходимость в consensus-алгоритмах и компромиссы CAP theorem. Выбор уровня изоляции и механизма координации напрямую влияет на производительность и корректность системы.

---

## Ключевые идеи

- ACID: Atomicity (all-or-nothing), Consistency (инварианты), Isolation (транзакции не видят друг друга), Durability (данные не теряются)
- Уровни изоляции: Read Uncommitted → Read Committed → Repeatable Read/Snapshot → Serializable
- Каждый уровень предотвращает определённые аномалии: dirty reads, read skew, lost update, write skew, phantom
- Три механизма Serializable: 2PL (пессимистичный), SSI (оптимистичный), фактическое последовательное выполнение
- 2PC для распределённых транзакций, но падение координатора блокирует участников (in doubt)
- Linearizability ≠ Serializability: первое — актуальность данных, второе — изоляция
- CAP: при partition выбор между Consistency и Availability
- Consensus (Raft, ZooKeeper) — фундамент для выбора лидера и total order broadcast
- В микросервисах вместо 2PC рекомендуются **Саги** с компенсационными транзакциями

## Определения и термины

**Atomicity (Атомарность)** — способность прервать транзакцию при ошибке и отбросить (undo) все частично выполненные записи. Гарантия all-or-nothing. Не связана с многопоточностью.

**Consistency (Согласованность)** — транзакция переводит БД из одного валидного состояния (отвечающего инвариантам приложения) в другое валидное состояние.

**Isolation (Изоляция)** — параллельно выполняющиеся транзакции не влияют друг на друга. По умолчанию приравнивается к serializability.

**Durability (Долговечность)** — данные успешно зафиксированной (committed) транзакции не будут утеряны при сбоях.

**Read Uncommitted** — самый слабый уровень: предотвращает dirty writes, но допускает dirty reads.

**Read Committed** — две гарантии: при чтении видны только закомиченные данные (no dirty reads), при записи перезаписываются только закомиченные данные (no dirty writes).

**Repeatable Read / Snapshot Isolation** — транзакция читает из консистентного снимка БД, зафиксированного на момент начала, игнорируя более поздние изменения.

**Serializable** — наивысший уровень: параллельное выполнение даёт тот же результат, что и последовательное (serial).

**Dirty Read** — транзакция видит незакомиченные записи другой транзакции.

**Dirty Write** — транзакция перезаписывает значение, изменённое другой незакомиченной транзакцией.

**Read Skew (Nonrepeatable Read)** — клиент видит разные части БД в разные моменты (баланс одного счёта до перевода, второго — после).

**Lost Update** — при конкурентном read-modify-write вторая транзакция перезаписывает результат первой.

**Write Skew** — две транзакции читают одни данные, принимают решение и обновляют разные объекты. К моменту записи исходная посылка неверна.

**Phantom** — запись в одной транзакции изменяет результаты поискового запроса в другой.

**Linearizability (Strong Consistency)** — гарантия актуальности (recency guarantee): система ведёт себя так, будто существует ровно одна копия данных, и все операции атомарны.

**CAP theorem** — при разделении сети (Partition) распределённая система выбирает между Consistency (Linearizability) и Availability.

**Consensus** — процесс достижения алгоритмического согласия между узлами со свойствами: Agreement, Integrity, Validity, Termination.

## Как устроено

### Аномалии и уровни изоляции

| Уровень | Dirty Read | Dirty Write | Read Skew | Lost Update | Write Skew | Phantom |
|---------|-----------|-------------|-----------|-------------|------------|---------|
| Read Uncommitted | Да | Нет | Да | Да | Да | Да |
| Read Committed | Нет | Нет | Да | Да | Да | Да |
| Snapshot Isolation | Нет | Нет | Нет | Частично | Да | Да |
| Serializable | Нет | Нет | Нет | Нет | Нет | Нет |

### Two-Phase Locking (2PL) — пессимистичный Serializable

1. Для чтения — Shared lock, для записи — Exclusive lock
2. Писатель блокирует и других писателей, **и читателей** (отличие от MVCC)
3. Читатель блокирует писателей, но позволяет параллельное чтение
4. Все блокировки накапливаются (фаза 1) и удерживаются до commit/abort (фаза 2)
5. Для предотвращения phantom: Index-range locks или Predicate locks

### MVCC (Multi-Version Concurrency Control) — Snapshot Isolation

1. При старте транзакция получает уникальный возрастающий txid
2. При записи строка маркируется `created_by = txid`
3. При удалении — маркируется `deleted_by = txid` (не удаляется физически)
4. Чтение игнорирует изменения от незакомиченных транзакций и от транзакций с txid > текущего
5. **Readers never block writers, and writers never block readers**

### SSI (Serializable Snapshot Isolation) — оптимистичный Serializable

1. Транзакции работают на базе Snapshot Isolation **без блокировок**
2. БД отслеживает чтение устаревших MVCC-снимков
3. БД отслеживает записи, влияющие на предшествующие чтения других транзакций
4. При коммите, если обнаружен конфликт сериализации — транзакция abort

### Two-Phase Commit (2PC) — распределённый коммит

1. Координатор выдает глобальный ID транзакции
2. **Phase 1 (Prepare)**: координатор отправляет prepare участникам. Они пишут в WAL, проверяют ограничения, голосуют "yes"/"no"
3. Отдавая "yes", участник **отказывается от права** самостоятельно прервать транзакцию
4. **Phase 2 (Commit)**: если все "yes" — координатор пишет решение в лог (commit point), отправляет commit всем

### Consensus (Raft / ZooKeeper)

1. Используется для выбора лидера и Total Order Broadcast
2. Выбор лидера голосованием кворума с номером эпохи (epoch/term)
3. Предложение принимается при подтверждении кворумом, если нет лидера с большей эпохой
4. Требуется строгое большинство (например, 3 из 5)

## Детали и нюансы

- **Падение координатора 2PC после Phase 1**: участники зависают in doubt, удерживая exclusive locks. Блокируют доступ к строкам до восстановления координатора или ручного вмешательства
- **Heuristic decisions в 2PC** — антипаттерн: одностороннее разрешение in doubt транзакций ломает атомарность
- **Clock skew + LWW**: хронологически последняя запись "тихо" отбрасывается из-за отставания часов
- **Длинные транзакции в SSI**: высокий процент aborts, т.к. предпосылки успевают измениться. SSI требует коротких и быстрых транзакций
- **Linearizability при partition**: линеаризуемая БД отклоняет запросы на отрезанных узлах → недоступна
- **XA-транзакции в микросервисах** — антипаттерн: производительность падает до 10 раз (MySQL), отказ любого узла прерывает транзакцию (failure amplification). Рекомендуется паттерн **Саги** с компенсационными транзакциями
- **Materializing conflicts** — создание таблиц-блокировок для слотов времени/ресурсов при отсутствии Serializable

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| 2PL | Истинная сериализуемость, предотвращает все аномалии | Блокирует читателей, deadlocks, низкая пропускная способность | Строгие транзакционные требования |
| MVCC/Snapshot | Readers не блокируют writers, хорошая пропускная способность | Не предотвращает write skew и phantom | Аналитика, длинные чтения |
| SSI | Сериализуемость без блокировок | Высокий процент aborts при длинных транзакциях | Короткие транзакции, OLTP |
| 2PC | Атомарность в распределённой системе | Падение координатора блокирует всё, failure amplification | Распределённые БД (не микросервисы) |
| Saga | Нет координатора, отказоустойчивость | Eventual consistency, компенсации | Микросервисы |
| Linearizable | Одна копия данных, атомарные операции | Недоступна при partition, высокая задержка | Выбор лидера, CAS-операции |
| Eventual consistent | Высокая доступность | Stale reads | Когда допустима задержка обновления |

## Примеры

### SELECT FOR UPDATE — пессимистичная блокировка

```sql
SELECT * FROM doctors
WHERE on_call = true AND shift_id = 1234
FOR UPDATE;
```

`FOR UPDATE` захватывает Exclusive lock на прочитанные строки. Другие транзакции ждут.

### Compare-And-Set (CAS) — линеаризуемая операция

```
CAS(x, old=1, new=2) → success
CAS(x, old=0, new=3) → fail (текущее значение не 0)
```

Изменение только если текущее значение совпадает с ожидаемым.

### Write Skew — аномалия

Транзакция A читает: 2 врача на дежурстве → снимает Alice с дежурства.
Транзакция B читает: 2 врача на дежурстве → снимает Bob с дежурства.
Итог: 0 врачей на дежурстве. Обе транзакции обновляли **разные объекты**, но логика нарушена.

### Mutex для Race Condition (Go)

```go
s.mu.Lock()
if part.Stock < qty {
    s.mu.Unlock()
    return errors.New("insufficient stock")
}
part.Stock -= qty
s.mu.Unlock()
```

## Важно / подводные камни / best practices

- **MVCC** — readers never block writers, and writers never block readers. Использовать для аналитики
- **SSI** — требует коротких транзакций; при длинных — высокий процент aborts
- **2PC** — не использовать в микросервисах (failure amplification, 10x замедление). Использовать **Saga**
- **Linearizability ≠ Serializability**: первое — актуальность одного объекта, второе — изоляция многих
- **2PL ≠ 2PC**: 2PL — изоляция на одной БД, 2PC — распределённый коммит
- **Clock skew** делает LWW небезопасным — нужны version vectors
- **Materializing conflicts** — обходной путь для write skew без Serializable

В чём разница Linearizability и Serializability?
?
- **Linearizability** — актуальность данных (recency) для одного объекта
- **Serializability** — изоляция операций над многими объектами

В чём разница 2PL и 2PC?
?
- **2PL** (Two-Phase Locking) — сериализуемая изоляция на одной БД (фаза блокировок + фаза снятия)
- **2PC** (Two-Phase Commit) — распределённый коммит (Prepare + Commit). Совершенно разные вещи

Шаги Two-Phase Commit (2PC)
?
1. Координатор выдает глобальный ID
2. Phase 1 (Prepare): координатор → prepare, участники → WAL + vote yes/no
3. Голос "yes" = отказ от права самостоятельного abort
4. Phase 2 (Commit): все "yes" → координатор пишет решение → commit всем

Что происходит при падении координатора 2PC после Phase 1?::Участники зависают in doubt, удерживая exclusive locks. Блокируют доступ до восстановления координатора

Три механизма Serializable изоляции
?
1. **2PL** — пессимистичный, блокировки
2. **SSI** — оптимистичный, на базе snapshot + обнаружение конфликтов
3. **Фактическое последовательное выполнение** — single thread

Как работает MVCC (Snapshot Isolation)?
?
1. Транзакция получает txid
2. Запись → строка маркируется created_by = txid
3. Удаление → маркируется deleted_by = txid
4. Чтение игнорирует незакомиченные и более новые txid
5. Readers never block writers

SSI (Serializable Snapshot Isolation):::Оптимистичный алгоритм: работает на snapshot без блокировок, отслеживает конфликты, при коммите abort если конфликт. Требует коротких транзакций

Почему 2PC не подходит для микросервисов?::Производительность падает до 10x (MySQL), отказ любого узла прерывает транзакцию (failure amplification). Использовать Saga с компенсациями

Consensus требует строгого большинства — например, =={3 из 5}== узлов

Принцип MVCC: ==Readers never block writers==, and ==writers never block readers==

Что такое Materializing conflicts?::Создание дополнительных таблиц-блокировок для слотов времени/ресурсов — обходной путь для write skew без Serializable изоляции
