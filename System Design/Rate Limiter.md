## Суть и контекст

Rate Limiter — компонент, контролирующий частоту трафика от клиента или сервиса. Блокирует избыточные вызовы, если количество запросов превышает порог. Предотвращает DoS-атаки, снижает затраты на платные API и защищает серверы от перегрузки ботами.

---

## Ключевые идеи

- Размещение: на клиенте ненадёжно, на сервере — контроль, в middleware/API Gateway — оптимально для микросервисов
- Пять алгоритмов: Token Bucket, Leaking Bucket, Fixed Window Counter, Sliding Window Log, Sliding Window Counter
- Счётчики хранятся в Redis (INCR + EXPIRE) — in-memory, быстрый доступ
- В распределённой среде: race condition при конкурентном read-check-increment; нужны Lua-скрипты или locks
- HTTP 429 Too Many Requests при превышении лимита

---

## Определения и термины

**Rate Limiter** — компонент сетевой системы, контролирующий частоту трафика. Блокирует избыточные вызовы, если запросов больше порога за период.

**Token Bucket** — контейнер с заданной ёмкостью (bucket size), в который периодически добавляются маркеры (tokens) с заданной скоростью (refill rate). Широко используется Amazon и Stripe. Допускает кратковременные всплески.

**Leaking Bucket** — запросы обрабатываются со строго фиксированной скоростью через очередь FIFO. Используется Shopify. Сглаживает трафик, но при всплеске очередь заполняется старыми запросами.

**Fixed Window Counter** — время делится на фиксированные окна, счётчик увеличивается для каждого окна. При превышении — отклонение.

**Sliding Window Log** — хранит timestamp каждого запроса. Отклоняет, если количество за последнее окно > лимита. Решает проблему границ окон, но потребляет много памяти (хранит даже отклонённые).

**Sliding Window Counter** — гибрид Fixed Window + Sliding Window Log. Оценивает частоту на основе взвешенного среднего предыдущего окна.

## Как устроено

### Где размещать Rate Limiter

| Место | Плюсы | Минусы |
|-------|-------|--------|
| Клиент | — | Ненадёжно, запросы легко подделать |
| Сервер | Полный контроль над алгоритмом | Нагрузка на сервер |
| Middleware / API Gateway | Оптимально для микросервисов, уже есть аутентификация | Зависимость от gateway |

### Архитектура Rate Limiter (распределённая)

1. Правила лимитов хранятся на диске. Workers загружают их в кэш
2. Клиент отправляет запрос в Middleware
3. Middleware читает правила из кэша и счётчик из Redis
4. Счётчик > порога → HTTP 429 (или в очередь)
5. Счётчик ≤ порога → запрос на API-сервер, Redis INCR

### Алгоритмы

| Алгоритм | Как работает | Плюсы | Минусы |
|----------|-------------|-------|--------|
| Token Bucket | Токены добавляются с refill rate; запрос забирает токен | Допускает burst, прост, мало памяти | Два параметра (bucket size, refill rate) сложно настроить |
| Leaking Bucket | FIFO очередь, обработка с фиксированной скоростью | Сглаживает трафик, стабильный output | Всплеск заполняет очередь старыми запросами |
| Fixed Window Counter | Счётчик на фиксированное окно времени | Прост, мало памяти | На стыке окон пропускает 2x лимита |
| Sliding Window Log | Хранит timestamps всех запросов | Точный, нет проблем границ | Много памяти (хранит даже отклонённые) |
| Sliding Window Counter | Взвешенное среднее предыдущего + текущего окна | Точнее Fixed Window, мало памяти | Приблизительная оценка |

---

## Детали и нюансы

- **Fixed Window Counter на стыке окон**: может пропустить в 2 раза больше лимита, если всплеск на границе двух окон
- **Race condition в распределённой среде**: конкурентный read-check-increment для счётчика Redis. Sticky sessions не масштабируются. Нужны **блокировки (Locks) или Lua-скрипты Redis**
- **Серверы Rate Limiter** географически близко к edge-серверам (CDN) для снижения задержки

## Примеры

### Redis команды

```
INCR user:123:requests       # +1 к счётчику
EXPIRE user:123:requests 60  # TTL 60 секунд
```

### HTTP ответ при превышении

```
HTTP/1.1 429 Too Many Requests
X-Ratelimit-Remaining: 0
X-Ratelimit-Limit: 100
X-Ratelimit-Retry-After: 30
```

## Важно / подводные камни / best practices

- Клиент должен обрабатывать HTTP 429 и добавлять **exponential backoff** в retry
- Кэш на клиенте — не спамить API
- **Lua-скрипты Redis** для атомарного read-check-increment в распределённой среде
