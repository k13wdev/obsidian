## Суть и контекст

**Application Layer (Слой приложения)** — слой, определяющий задачи, которые должно выполнять программное обеспечение, и управляющий выразительными доменными объектами для решения проблем. Не содержит бизнес-правил или знаний о предметной области — только координирует задачи и делегирует работу коллаборациям доменных объектов.

Application Layer — это «тонкий координатор»: он управляет состоянием хода выполнения задачи, но сам решений не принимает.

---

## Ключевые идеи

- Application Service принимает вызов от клиентского кода (контроллера) с параметрами в виде примитивов, строк, DTO или объектов команд.
- Сервис загружает агрегат из Repository, вызывает на нём доменный метод и сохраняет результат.
- Бизнес-логика **не должна** находиться в Application Service — только в доменных объектах.
- Если Application Service начинает выполнять вычисления бизнес-правил, **доменное знание перетекает в код приложения** — это антипаттерн.
- Application Services выступают в роли основного API (Primary Ports) в [[Архитектура (Hexagonal, CQRS, Event Sourcing)|Hexagonal Architecture]], к которому подключаются внешние адаптеры (REST контроллеры, SOAP интерфейсы).

---

## Определения и термины

**Application Service (Сервис приложения)** — координирующий класс в слое приложения, который принимает примитивные типы или объекты команд, использует репозитории для извлечения агрегатов и делегирует им выполнение бизнес-операций.

**Command (Команда)** — операция, которая вносит изменения в состояние системы. В архитектуре CQS имеет тип возвращаемого значения void.

**Query (Запрос)** — операция, которая извлекает информацию из системы, не вызывая наблюдаемых побочных эффектов (Side-Effect-Free Function).

**DTO (Data Transfer Object)** — объект для передачи данных между доменной моделью и пользовательским интерфейсом (или другими системами).

**DTO Assembler** — компонент, который читает данные из экземпляров агрегата и собирает DTO.

**Domain Payload Object (DPO)** — альтернатива DTO для передачи данных агрегата без создания отдельного DTO.

**Double Dispatch** — паттерн, при котором Application Service получает зависимости из репозиториев и передаёт их как аргументы в методы агрегата, избегая Dependency Injection внутрь агрегатов.

## Как устроено

### Структура Application Service

Application Service следует предсказуемому шаблону:

1. Принимает вызов с параметрами (примитивы, строки, DTO или Command-объекты).
2. Создаёт экземпляры идентификаторов (например, `TenantId`, `ProductId`) из переданных строк.
3. Обращается к Repository для извлечения **одного** (желательно) агрегата по идентификатору.
4. (Опционально) Извлекает другие зависимости (другие агрегаты или Domain Services).
5. Вызывает метод-команду на извлечённом агрегате, передавая необходимые данные и зависимости.
6. Сохраняет изменённый агрегат обратно через Repository.

### Управление транзакциями

Транзакции управляются **слоем приложения** (Application Service), а не доменными объектами или репозиториями. Транзакция должна охватывать модификацию **строго одного** экземпляра агрегата. При необходимости обновления других агрегатов используется Eventual Consistency через публикацию Domain Events.

### DTO и маппинг

Агрегаты **не передаются** напрямую клиентам. Данные из агрегатов конвертируются в DTO через DTO Assembler. Альтернативные подходы: Domain Payload Object (DPO) или публикация внутреннего состояния агрегата через паттерн Mediator.

### Валидация

В доменном слое валидация реализуется через механизм **Guards (охранные проверки)** — проверки корректности атрибутов при конструировании или изменении. Фабрики проверяют инварианты при создании сложных объектов и выбрасывают исключения, если безопасное создание невозможно. Для сложных композиций существует **Deferred Validation (отложенная валидация)** — проверка согласованности всего дерева агрегата перед транзакционным коммитом.

## Отличие Application Service от Domain Service

| | Application Service | Domain Service |
|---|---|---|
| Слой | Application Layer | Domain Layer |
| Содержит бизнес-логику | Нет | Да |
| Состояние | Управляет ходом задачи | Stateless |
| Что делает | Координирует: парсит вход, вызывает домен, отправляет уведомления | Выполняет бизнес-правила между агрегатами |
| Пример | Парсит XML-запрос, вызывает Domain Service, отправляет e-mail | Дебетует и кредитует Account, проверяет бизнес-правила |

**Transaction Script vs Application Service:** если Transaction Script самостоятельно принимает решения на основе данных, то Application Service только вызывает методы доменных объектов, оставляя принятие решений за ними.

## Примеры

### Use Case: создание элемента беклога

```java
public class ProductBacklogItemService ... {
    ...
    @Transactional
    public void planProductBacklogItem(
        String aTenantId, String aProductId,
        String aSummary, String aCategory,
        String aBacklogItemType, String aStoryPoints) {

        Product product =
            productRepository.productOfId(
                    new TenantId(aTenantId),
                    new ProductId(aProductId));

        BacklogItem plannedBacklogItem =
            product.planBacklogItem(
                    aSummary,
                    aCategory,
                    BacklogItemType.valueOf(aBacklogItemType),
                    StoryPoints.valueOf(aStoryPoints));

        backlogItemRepository.add(plannedBacklogItem);
    }
    ...
}
```

Метод принимает скалярные типы (String). Ищет `Product` через репозиторий. Вызывает фабричный метод `planBacklogItem` на агрегате, который гарантирует выполнение доменных инвариантов. Сохраняет созданный `BacklogItem`.

### Double Dispatch: разрешение зависимостей

```java
public class ProductBacklogItemService ... {
    ...
    @Transactional
    public void assignTeamMemberToTask(
        String aTenantId, String aBacklogItemId,
        String aTaskId, String aTeamMemberId) {

        BacklogItem backlogItem =
            backlogItemRepository.backlogItemOfId(
                new TenantId(aTenantId),
                new BacklogItemId(aBacklogItemId));

        Team ofTeam =
            teamRepository.teamOfId(
                backlogItem.tenantId(),
                backlogItem.teamId());

        backlogItem.assignTeamMemberToTask(
                new TeamMemberId(aTeamMemberId),
                ofTeam,
                new TaskId(aTaskId));
    }
    ...
}
```

Application Service извлекает `Team` (другой агрегат) из репозитория и передаёт его как аргумент в доменный метод `assignTeamMemberToTask` — вместо внедрения `teamRepository` в `BacklogItem`.

## Детали и нюансы

### Обработка ошибок

- **Optimistic Concurrency:** если два пользователя одновременно пытаются изменить один агрегат, ORM сравнивает версию объекта. Версия второго пользователя признаётся устаревшей (stale), транзакция отклоняется, генерируется исключение оптимистичной блокировки.
- **Ошибки доменной логики** обрабатываются выбросом исключений из доменных моделей или фабрик до завершения транзакции в Application Service.

### Batch Processing (исключение из правила)

Разрешается модифицировать несколько агрегатов в одной транзакции при пакетной обработке (например, создание списка элементов из массива `BacklogItemDescription[]` за один вызов), так как семантически это равноценно созданию объектов по одному.

## Важно / подводные камни

- **Application Service должен быть тонким.** Если логика взаимодействия агрегатов переносится в Application Service, доменное знание утекает в слой приложения, что приводит к потере целостности доменной модели.
- **Не внедряй Repository в агрегаты.** Получай зависимости в Application Service и передавай их как аргументы в методы агрегата (Double Dispatch).
- **Одна транзакция = один агрегат.** Если Use Case требует обновления нескольких агрегатов одновременно, используй Domain Events и Eventual Consistency.
- **DTO — не копия доменной модели.** DTO, которые слепо копируют поля доменной модели, превращаются в Anemic Domain Model.

Чем Application Service отличается от Domain Service?
?

5. Вызвать метод-команду на агрегате.

Кто управляет транзакциями в DDD?::Application Service. Транзакция охватывает модификацию строго одного агрегата. Обновление других агрегатов — через Eventual Consistency (Domain Events).

Что такое DTO Assembler?::Компонент, который читает данные из экземпляров агрегата и собирает DTO для передачи клиентам. Агрегаты не передаются напрямую.
