## Суть и контекст

Стратегический дизайн — высокоуровневый уровень DDD. Он отвечает на вопросы: **как разбить большую систему на части**, как эти части взаимодействуют и **куда направить лучшие ресурсы команды**.

---

## Distillation (Дистилляция)

**Дистилляция** — процесс разделения компонентов смеси для извлечения её ценной сути.

В DDD: постоянный процесс рефакторинга и очищения модели от второстепенных деталей с целью сфокусировать внимание на самом главном.

Стратегическая дистилляция:
- Помогает команде осознать дизайн системы в целом.
- Облегчает коммуникацию между командами.
- Направляет усилия по разработке.
- Подсказывает, какие компоненты стоит отдать на аутсорс, а какие — поручить лучшим разработчикам.

---

## Три типа поддоменов

Большая система состоит из множества компонентов, но не все они одинаково важны.

### Core Domain (Коренной домен)

**Core Domain** — наиболее важная, отличительная часть модели, которая является ключом к достижению целей пользователя, приносит основную ценность бизнесу и выделяет приложение среди конкурентов.

Именно сюда следует направлять **лучшие кадры и основные усилия** для создания глубокой модели и гибкого дизайна.

**Пример:** в системе контейнерных перевозок Core Domain — уникальные алгоритмы маршрутизации и консолидации грузов. Это то, что нельзя купить «с полки».

### Generic Subdomain (Универсальный поддомен)

**Generic Subdomain** — часть модели, которая усложняет систему, но не отражает специфических знаний бизнеса.

Примеры: учёт часовых поясов, отправка email, стандартная бухгалтерия, аутентификация.

Для Generic Subdomains рекомендуется использовать **готовые решения (off-the-shelf)** или отдавать реализацию на аутсорс, чтобы не отвлекать основную команду от Core Domain.

### Supporting Subdomain (Вспомогательный поддомен)

**Supporting Subdomain** — компоненты, выполняющие важную поддерживающую роль для Core Domain, но не являющиеся главным конкурентным преимуществом и не полностью универсальным решением.

| | Core Domain | Generic Subdomain | Supporting Subdomain |
|---|---|---|---|
| Уникальность | Уникален для бизнеса | Одинаков везде | Частично специфичен |
| Ресурсы | Лучшие разработчики | Аутсорс / готовые решения | Средний приоритет |
| Модель | Глубокая, сложная | Стандартная | Достаточная |

## Context Map (Карта контекстов)

**Context Map** — документ или схема, предоставляющая **глобальный обзор всех Bounded Contexts** в проекте и описывающая **фактические, реальные** (не идеальные) отношения между ними.

В проектах сосуществует множество моделей, каждая из которых имеет смысл только внутри своего Bounded Context. Context Map делает эти отношения явными.

---

## Паттерны интеграции между контекстами

### 1. Shared Kernel (Общее ядро)

Две или более команды соглашаются использовать **общее подмножество** доменной модели вместе с ассоциированным кодом и структурой базы данных.

Эта «общая» часть получает особый статус: её нельзя изменять без предварительной консультации с другой командой. Для предотвращения поломок необходима частая непрерывная интеграция и объединённые наборы автоматизированных тестов.

**Когда:** команды тесно связаны и готовы координировать изменения.

### 2. Customer-Supplier (Заказчик-Поставщик)

Одна подсистема **upstream (поставщик)** кормит данными другую — **downstream (заказчик)**. Отношения формализуются:
- Команда «заказчика» участвует в планировании итераций «поставщика», формулируя свои нужды.
- Команды совместно создают набор автоматизированных приёмочных тестов, которые «поставщик» постоянно выполняет у себя.

**Когда:** у поставщика есть мотивация учитывать нужды заказчика.

### 3. Conformist (Конформист)

Применяется, когда команда «поставщика» **не имеет мотивации** учитывать потребности «заказчика». Команда «заказчика» отказывается от создания собственной независимой модели и **подчиняется модели поставщика**.

Это сильно упрощает интеграцию и устраняет затраты на перевод данных, хотя и навязывает архитектурные ограничения нижестоящей команде.

**Пример:** интеграция с крупной платёжной платформой или ERP — проще принять их модель, чем бороться.

### 4. Anti-Corruption Layer — ACL (Предохранительный уровень)

Если внешняя система (часто унаследованная/legacy) имеет слабую или неподходящую модель — прямая интеграция «загрязнит» новую чистую модель.

**ACL** — изолирующий слой, который позволяет новой системе работать **исключительно в терминах своей доменной модели**, выполняя двусторонний семантический перевод запросов и данных.

Обычно строится на базе паттернов **Facade, Adapter и Translator**.

```
Новая система ──► ACL (Facade/Adapter/Translator) ──► Legacy система
     │                                                      │
 Модель A                                             Модель B (чужая)
```

**Когда:** интеграция с legacy, внешними API, чужими системами с «плохой» моделью.

### 5. Open Host Service (Служба с открытым хостом)

Когда к подсистеме необходимо подключить **множество других систем** — создание индивидуальных трансляторов для каждой парализует разработку.

В этом случае определяется единый **протокол**, открывающий доступ к подсистеме как к набору сервисов. Специфические требования единичных интеграторов решаются с помощью одноразовых расширений.

**Примеры:** публичное API, REST API с версионированием.

### 6. Published Language (Опубликованный язык)

Когда системы должны обмениваться информацией (особенно в B2B или Open Host Service), используется **хорошо документированный общий язык** вместо прямого перевода между внутренними моделями.

**Примеры:** XML-схемы, JSON Schema, отраслевые стандарты (Chemical Markup Language для химических формул).

Часто используется вместе с **Open Host Service**: сервис предоставляет доступ через Published Language.

### 7. Separate Ways (Разные пути)

Интеграция всегда обходится дорого и влечёт компромиссы. Если два набора функций **не имеют абсолютно необходимой связи** — нужно безжалостно отрезать их друг от друга.

Это позволяет разработчикам создавать простые и эффективные решения в узких рамках без затрат на трансляционные слои.

**Когда:** дешевле реализовать что-то дважды, чем тратить силы на интеграцию.

### 8. Big Ball of Mud (Большой комок грязи)

**Антипаттерн** — результат провала стратегического проектирования: границы контекстов размываются, код и данные спутываются в хаос без чётких правил.

В Context Map этот антипаттерн **явно обозначается** — чтобы очертить его границы и защитить другие «чистые» Bounded Contexts от его влияния с помощью Anti-Corruption Layer.

## Пример Context Map

```
┌─────────────────┐  Customer-Supplier  ┌──────────────────┐
│  Контекст       │ ──────────────────► │  Контекст        │
│  «Заказы»       │                     │  «Доставка»      │
│  (upstream)     │                     │  (downstream)    │
└────────┬────────┘                     └──────────────────┘
         │
         │ ACL
         ▼
┌─────────────────┐
│  Legacy ERP     │  ← Big Ball of Mud
│  (внешняя       │    (защищаем через ACL)
│   система)      │
└─────────────────┘

┌─────────────────┐  Published Language ┌──────────────────┐
│  Контекст       │ ──────────────────► │  Внешние         │
│  «Каталог»      │  (Open Host Service)│  партнёры        │
│  (Open API)     │                     │                  │
└─────────────────┘                     └──────────────────┘
```

## Важно / подводные камни

- **Context Map — реальность, не идеал.** Карта описывает фактические отношения «как есть», а не желаемое состояние. Это инструмент понимания, а не проектирования.
- **ACL — дорого, но необходимо.** Не экономь на предохранительном уровне при интеграции с legacy — «загрязнение» модели обходится дороже.
- **Core Domain — приоритет всегда.** Если Core Domain и Generic Subdomain конкурируют за ресурсы — Core Domain побеждает всегда.
- **Conformist — осознанный выбор.** Подчинение чужой модели — законная стратегия, не поражение. Главное — это явное архитектурное решение, а не случайная зависимость.

## Карточки для повторения

#flashcards/sd/ddd

Что такое Core Domain?::Наиболее важная, отличительная часть модели, приносящая основную ценность бизнесу и выделяющая продукт среди конкурентов. Сюда направляют лучшие ресурсы.

Что такое Generic Subdomain?::Часть системы, которая не отражает специфических знаний бизнеса (email, аутентификация, бухгалтерия). Рекомендуется использовать готовые решения или аутсорс.

Что такое Context Map?::Документ или схема с глобальным обзором всех Bounded Contexts и описанием фактических (не идеальных) отношений между ними.

Перечисли паттерны интеграции между контекстами.
?
1. Shared Kernel — общее подмножество модели
2. Customer-Supplier — поставщик учитывает нужды заказчика
3. Conformist — заказчик принимает модель поставщика
4. Anti-Corruption Layer — изолирующий слой перевода
5. Open Host Service — единый протокол для многих интеграторов
6. Published Language — общий документированный язык
7. Separate Ways — намеренный отказ от интеграции
8. Big Ball of Mud — антипаттерн (хаос без границ)

Что такое Anti-Corruption Layer?::Изолирующий слой, позволяющий новой системе работать в терминах своей модели при интеграции с legacy/внешними системами. Выполняет двусторонний семантический перевод. Строится на Facade, Adapter, Translator.

Когда применяется паттерн Conformist?::Когда поставщик не имеет мотивации учитывать нужды заказчика. Заказчик принимает модель поставщика целиком — упрощает интеграцию, но навязывает ограничения.

Чем Open Host Service отличается от Published Language?::Open Host Service — протокол/интерфейс доступа к подсистеме для множества интеграторов. Published Language — конкретный формат данных (схема, стандарт) для обмена. Часто используются вместе.

Что такое Distillation в DDD?::Процесс очищения модели от второстепенных деталей для фокуса на главном. Разделяет домен на Core Domain, Generic и Supporting Subdomains; направляет ресурсы туда, где они принесут наибольшую ценность.
