## Суть и контекст

**Domain Event** — механизм, позволяющий агрегатам «сообщать о произошедшем» без прямой связи с другими частями системы. Это ключ к масштабируемости: вместо строгой транзакционной согласованности — **eventual consistency** через события.

---

## Определение

**Domain Event (Доменное событие)** — факт, что в предметной области произошло что-то значимое.

Доменные события:
- Публикуются моделью (чаще всего Агрегатом).
- Описывают произошедшее в прошедшем времени (например, `OrderPlaced`, `PaymentReceived`, `BacklogItemCommitted`).
- Позволяют оповестить другие части системы, удалённые системы или другие Bounded Contexts.

---

## Зачем нужны

### Проблема без событий

Без Domain Events: если `Order` создан, нужно немедленно и синхронно обновить `Inventory`, `Notification`, `Analytics`. Это прямые зависимости. Одна транзакция на всё — медленно, хрупко, плохо масштабируется.

### Решение с событиями

`Order.place()` → публикует `OrderPlaced` → подписчики (Inventory, Notification, Analytics) обрабатывают событие асинхронно в своих транзакциях.

```
Order.place()
    └─► публикует OrderPlaced
             ├─► InventoryService обновляет остатки
             ├─► NotificationService отправляет email
             └─► AnalyticsService записывает метрику
```

**Eventual consistency** за пределами агрегата — жизненно важная для масштабируемости.

## Как устроено

### Структура события

```python
from dataclasses import dataclass
from datetime import datetime
from uuid import UUID

@dataclass(frozen=True)  # Events — Value Objects (неизменяемы)
class OrderPlaced:
    order_id: UUID
    customer_id: UUID
    total: Money
    occurred_at: datetime = field(default_factory=datetime.utcnow)
```

Событие — **Value Object** (иммутабельно, нет идентичности — важен сам факт).

### Публикация из агрегата

```python
class Order:
    def __init__(self):
        self._events: list = []

    def place(self, customer_id: UUID, items: list) -> None:
        # бизнес-логика...
        self._events.append(OrderPlaced(
            order_id=self.order_id,
            customer_id=customer_id,
            total=self.total,
        ))

    def pull_events(self) -> list:
        events = self._events.copy()
        self._events.clear()
        return events
```

### Диспетчеризация

```python
# Application Service после сохранения агрегата
def place_order(command: PlaceOrderCommand) -> None:
    order = order_factory.create(...)
    order.place(command.customer_id, command.items)
    order_repository.add(order)

    # диспетчер публикует события подписчикам
    for event in order.pull_events():
        event_dispatcher.dispatch(event)
```

## Пример из источников

При добавлении элемента в спринт публикуется событие `BacklogItemCommitted`. Подписчики реагируют асинхронно, обновляя связанные агрегаты в своих транзакциях — без необходимости менять два агрегата в одной транзакции.

```
BacklogItem.commitToSprint()
    └─► BacklogItemCommitted(backlog_item_id, sprint_id)
             └─► Sprint обновляет свою статистику (отдельная транзакция)
```

## Domain Events и интеграция между Bounded Contexts

Domain Events — главный механизм **интеграции между Bounded Contexts**:

```
┌──────────────────────────┐         ┌──────────────────────────┐
│  Контекст «Заказы»       │         │  Контекст «Склад»        │
│                          │         │                          │
│  Order.place()           │──────►  │  InventoryService        │
│  └─► OrderPlaced (Event) │ Event   │  .handle(OrderPlaced)    │
│                          │ Bus /   │  └─► резервирует товар   │
│                          │ Queue   │                          │
└──────────────────────────┘         └──────────────────────────┘
```

Контексты не знают ничего друг о друге — только о событиях.

## Детали и нюансы

### Именование

События называются в **прошедшем времени**:
- `OrderPlaced` — не `PlaceOrder` (это команда)
- `PaymentReceived` — не `ReceivePayment`
- `UserRegistered` — не `RegisterUser`

### Event vs Command

| | Command (Команда) | Domain Event (Событие) |
|---|---|---|
| Смысл | «Сделай это» | «Это произошло» |
| Время | Будущее (намерение) | Прошедшее (факт) |
| Может отклониться? | Да | Нет — факт уже случился |
| Примеры | `PlaceOrder`, `CancelOrder` | `OrderPlaced`, `OrderCancelled` |

### Транзакционность

- Событие публикуется **после** успешного сохранения агрегата.
- Если транзакция откатилась — событие не публикуется.
- Для надёжности: «Transactional Outbox» паттерн — событие записывается в ту же транзакцию, что и агрегат, затем отдельный процесс публикует его в очередь.

## Важно / подводные камни

- **Событие — факт, не команда.** Не называй событие `DoSomething` — это команда. Событие описывает то, что уже случилось.
- **Не меняй события.** Если подписчиков много и они в продакшене — изменение события ломает их. Версионируй: `OrderPlacedV2`.
- **Порядок событий.** При асинхронной обработке события могут приходить не по порядку. Подписчики должны быть идемпотентными.
- **Не перегружай событие данными.** Событие несёт минимум данных — достаточно для принятия решения подписчиком. Сложные данные подписчик запрашивает сам.

## Карточки для повторения

#flashcards/sd/ddd

Что такое Domain Event?::Факт того, что в предметной области произошло что-то значимое. Публикуется агрегатом, описывается в прошедшем времени (OrderPlaced, PaymentReceived). Позволяет оповестить другие части системы асинхронно.

Почему Domain Events — это Value Objects?::Событие иммутабельно (описывает уже произошедший факт, который нельзя изменить) и не имеет концептуальной идентичности — важен сам факт, а не «который экземпляр события».

Как Domain Events обеспечивают eventual consistency?::Агрегат публикует событие после успешного сохранения. Подписчики (другие агрегаты, сервисы) обрабатывают событие асинхронно в своих транзакциях — согласованность достигается «в конечном счёте».

Чем Command отличается от Domain Event?
?
- **Command** — намерение сделать что-то в будущем (может отклониться).
- **Domain Event** — факт, что что-то уже произошло (не отклоняется).
- Именование: Command — `PlaceOrder`; Event — `OrderPlaced`.

Почему события называются в прошедшем времени?::Они описывают уже произошедший факт. `OrderPlaced` — заказ уже размещён. Это фундаментально отличает события от команд (намерений).

Когда публикуется Domain Event?::После успешного сохранения агрегата в хранилище. Если транзакция откатилась — событие не публикуется.

Как Domain Events используются для интеграции между Bounded Contexts?::Контекст-источник публикует событие в шину/очередь. Контекст-потребитель подписывается и обрабатывает событие в своей транзакции. Контексты не знают ничего друг о друге — только о форме события.
