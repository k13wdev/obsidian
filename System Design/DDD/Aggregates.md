## Суть и контекст

**Aggregate** — главный паттерн тактического DDD для управления сложностью и согласованностью данных. Без агрегатов бизнес-правила «размазываются» по всей системе; с агрегатами — инкапсулируются в чётко очерченной границе.

---

## Определение

**Aggregate (Агрегат)** — кластер (группа) связанных Сущностей и Объектов-значений, которые рассматриваются как **единое целое** с точки зрения изменения данных.

Агрегат имеет:
- **Корень (Aggregate Root)** — одна Сущность, через которую происходит весь доступ к агрегату снаружи.
- **Границу (Aggregate Boundary)** — чёткая черта вокруг кластера объектов, составляющих агрегат.

---

## Как устроено

```
┌──────────────────────────────────────────┐
│             Aggregate: Автомобиль        │
│                                          │
│  ┌─────────────────┐                     │
│  │  Корень (Root)  │ ← Car (Entity)      │
│  │   car_id: UUID  │                     │
│  └────────┬────────┘                     │
│           │                              │
│    ┌──────┴──────┐                       │
│    │             │                       │
│  Tire(VO)     Tire(VO)                   │
│  Tire(VO)     Tire(VO)                   │
│                                          │
└──────────────────────────────────────────┘
```

Шины автомобиля находятся **внутри** границы агрегата, так как вне контекста конкретной машины идентичность её колёс никого не интересует. Снаружи к агрегату обращаются только через `Car`.

## Четыре правила проектирования агрегатов

### Правило 1: Защищай истинные инварианты внутри границы

**Инвариант** — бизнес-правило, которое должно соблюдаться всегда.

Внутри границы агрегата должна поддерживаться **строгая транзакционная согласованность**. Всё, что должно быть согласовано «прямо сейчас» — должно быть в одном агрегате.

Пример: Заказ и его позиции (OrderItems). Правило «общая сумма заказа должна совпадать с суммой позиций» — инвариант. Они должны быть в одном агрегате.

### Правило 2: Создавай небольшие агрегаты

Крупные агрегаты приводят к проблемам:
- **Производительность**: загружать весь огромный объект ради изменения одного поля.
- **Масштабируемость**: блокировка всего агрегата при каждом изменении.
- **Конкурентные сбои**: при одновременном редактировании разными пользователями транзакции конкурируют за один объект.

Пример плохого дизайна: агрегат «Продукт», в который включены все отзывы, все варианты, весь инвентарь. Любое добавление отзыва блокирует весь продукт.

Решение: вынести «Отзыв» в отдельный агрегат. «Вариант продукта» — в свой. Ссылаться по ID.

### Правило 3: Ссылайся на другие агрегаты только по ID

Агрегаты должны ссылаться на другие агрегаты только через их **глобально уникальный идентификатор (ID)**, а не через прямые ссылки на объекты.

```python
# ❌ Плохо: прямая ссылка на другой агрегат
class Order:
    def __init__(self):
        self.customer: Customer = ...  # прямая ссылка

# ✅ Хорошо: ссылка по ID
class Order:
    def __init__(self):
        self.customer_id: UUID = ...  # только идентификатор
```

**Почему:** прямые ссылки размывают границы агрегатов, приводят к загрузке лишних данных и нарушают принцип «маленького агрегата».

### Правило 4: Используй Eventually Consistent за пределами границы

Согласованность внутри агрегата — строгая (в рамках одной транзакции). Согласованность между агрегатами — **в конечном счёте (eventually consistent)**, через Domain Events.

Нельзя изменить два агрегата в одной транзакции. Если кажется, что нужно — скорее всего неправильно определены границы.

## Примеры

### Агрегат «Заказ (Order)»

```
Order (Корень)
├── order_id: UUID
├── customer_id: UUID        ← ссылка по ID на другой агрегат
├── status: OrderStatus      ← Value Object
├── items: List[OrderItem]   ← Value Objects внутри границы
└── total: Money             ← Value Object

Инварианты:
- total = sum(item.price * item.quantity for item in items)
- нельзя добавить позицию в закрытый заказ
```

### Агрегат «Продукт (Product)» — небольшой

```
Product (Корень)
├── product_id: UUID
├── name: str
└── price: Money

Отзывы → отдельный агрегат Review(product_id, ...)
Варианты → отдельный агрегат ProductVariant(product_id, ...)
```

## Детали и нюансы

### Как определить границу агрегата

Задай вопрос: **«Что должно быть согласовано прямо сейчас, в рамках одной транзакции?»**
- Если X и Y должны изменяться вместе и никак иначе — они в одном агрегате.
- Если X и Y могут быть не согласованы несколько секунд — они в разных агрегатах.

### Корень агрегата — единственная точка входа

Внешний код никогда не обращается к внутренним объектам агрегата напрямую. Все операции проходят через корень. Корень отвечает за соблюдение инвариантов.

```python
# ❌ Плохо: прямое изменение внутреннего объекта
order.items[0].price = 100

# ✅ Хорошо: через метод корня
order.update_item_price(item_id, Money(100, "RUB"))
# корень проверяет инварианты и обновляет total
```

## Сравнения и trade-offs

| | Маленький агрегат | Большой агрегат |
|---|---|---|
| Производительность | Высокая | Низкая (загрузка лишнего) |
| Конкурентность | Мало конфликтов | Частые блокировки |
| Согласованность | Eventually consistent | Строгая, но медленная |
| Простота дизайна | Требует событий | Проще на первый взгляд |

---

## Важно / подводные камни

- **«Большой агрегат»** — самая частая ошибка начинающих в DDD. Начни с маленьких агрегатов и добавляй что-то внутрь только если без этого невозможно соблюдать инвариант.
- **Один агрегат = одна транзакция.** Если меняешь два агрегата в одной транзакции — скорее всего, это один агрегат или нужны Domain Events.
- **Корень — не просто «главный объект».** Это контракт: весь доступ к агрегату — через корень. Нарушение этого правила сломает инкапсуляцию инвариантов.
