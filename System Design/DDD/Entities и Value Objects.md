## Суть и контекст

Первый шаг тактического дизайна — научиться различать два фундаментально разных типа объектов в доменной модели: объекты с **идентичностью** (Entity) и объекты с **характеристиками** (Value Object). Ошибка в классификации приводит к сложному и неправильному коду.

---

## Entities (Сущности)

### Определение

**Entity** — объект, который фундаментально определяется не своими атрибутами, а **непрерывностью и уникальной идентичностью (identity)**.

Сущности проходят через различные состояния и жизненные циклы, но их идентичность сохраняется независимо от изменений формы.

### Ключевое свойство

Два объекта могут иметь одинаковые атрибуты, но быть **разными сущностями**. И наоборот — сущность остаётся той же, даже если все её атрибуты изменились.

### Примеры

- **Человек**: может поменять имя, адрес, профессию — но остаётся тем же человеком (та же идентичность).
- **Место на стадионе** с уникальным номером места: конкретное кресло 14Б в ряду 5 — это Сущность. Каждый раз вы бронируете именно это место.
- Но если на концерте **свободная рассадка** (без номеров) — места не имеют идентичности и Сущностями не являются. Любое свободное место эквивалентно другому.

### Идентичность

- Обычно выражается через уникальный **ID** (UUID, автоинкрементный идентификатор базы данных).
- Логика сравнения двух Сущностей строится только на сравнении их ID, а не атрибутов.

### В коде

```python
class Order:
    def __init__(self, order_id: UUID):
        self.order_id = order_id  # идентичность
        self.items = []
        self.status = "pending"

    def __eq__(self, other):
        return self.order_id == other.order_id  # сравнение по ID
```

## Value Objects (Объекты-значения)

### Определение

**Value Object** — объект, который описывает характеристику или атрибут, но **не имеет концептуальной идентичности**.

Если нам важны только сами характеристики элемента модели, его следует моделировать как Объект-значение.

### Ключевые свойства

1. **Неизменяемость (Immutable)** — Value Object нельзя изменить после создания. Вместо изменения — создаётся новый объект.
2. **Концептуальное целое (Conceptual Whole)** — несколько атрибутов объединяются в одно неделимое целое.
3. **Взаимозаменяемость** — два Value Object с одинаковыми атрибутами полностью идентичны. Нет разницы, какой из них использовать.

### Примеры

- **Адрес**: улица + город + почтовый индекс — это цельный Объект-значение. Адрес «ул. Ленина, 5, Москва, 101000» и другой адрес с теми же данными — это одно и то же.
- **Деньги (Money)**: сумма + валюта. `Money(100, "RUB")` и другой `Money(100, "RUB")` — взаимозаменяемы.
- **Цвет**: RGB(255, 0, 0) — это Объект-значение. Не важно, какой «экземпляр» красного использовать.
- **Диапазон дат**: `DateRange(start, end)` — неделимое целое.

### Неизменяемость в коде

```python
from dataclasses import dataclass

@dataclass(frozen=True)  # frozen=True → неизменяемый
class Money:
    amount: Decimal
    currency: str

    def add(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("Different currencies")
        return Money(self.amount + other.amount, self.currency)
        # возвращаем новый объект, не изменяем текущий
```

## Как различать Entity и Value Object

| Вопрос | Entity | Value Object |
|---|---|---|
| Важна ли уникальность этого объекта? | Да | Нет |
| Может ли объект изменяться, оставаясь «тем же»? | Да | Нет |
| Важно ли, какой конкретно «экземпляр» использовать? | Да | Нет |
| Нужно ли отслеживать историю изменений? | Да | Нет |

**Практический вопрос:** Если заменить один экземпляр на другой с теми же данными — изменится ли что-то для системы?
- Да → это **Entity** (идентичность важна).
- Нет → это **Value Object** (характеристики важны, не «кто»).

## Сравнения и trade-offs

| | Entity | Value Object |
|---|---|---|
| Идентичность | Уникальная (ID) | Нет (определяется атрибутами) |
| Изменяемость | Мутабельна | Иммутабельна |
| Сравнение | По ID | По значениям всех атрибутов |
| Жизненный цикл | Непрерывный | Создаётся заново при «изменении» |
| Сложность | Выше | Ниже — проще тестировать |

## Детали и нюансы

- **Одно и то же может быть и Entity, и Value Object** — зависит от контекста. Место на стадионе: Entity (при нумерованных местах), Value Object (при свободной рассадке).
- **Предпочитай Value Object** там, где можно — иммутабельные объекты проще, предсказуемее и безопаснее в многопоточной среде.
- Value Object может **содержать другие Value Object** — например, `Address` содержит `PostalCode`.
- Value Object может **содержать Entity** — но редко, это сигнал пересмотреть модель.

## Важно / подводные камни

- **Не добавляй ID ради ID.** Если объект не нуждается в идентичности — не превращай его в Entity только потому, что «так удобнее для базы данных».
- **Изменяемый Value Object — ошибка дизайна.** Если Value Object можно изменить, это разрушает концепцию взаимозаменяемости и приводит к «неожиданным» изменениям в других частях кода.
- **База данных — не модель.** ORM-строка с ID не означает, что объект — Entity. ID строки — технический артефакт, а не доменная идентичность.

Как сравниваются Entity и Value Object?
?
- **Entity** — сравнение по ID (идентичности). Два объекта с одинаковыми атрибутами могут быть разными сущностями.
- **Value Object** — сравнение по значениям всех атрибутов. Два объекта с одинаковыми значениями — полностью идентичны.

Почему Value Object должен быть иммутабельным?::Неизменяемость обеспечивает взаимозаменяемость. Изменяемый Value Object может «неожиданно» измениться в другом месте кода. Вместо изменения — создаётся новый объект.

Как понять, что перед тобой Entity, а не Value Object?::Задать вопрос: если заменить этот объект на другой с теми же данными, изменится ли что-то для системы? Если да — Entity. Если нет — Value Object.

Что такое Conceptual Whole (концептуальное целое)?::Несколько атрибутов объединяются в один неделимый Value Object. Например, `Address` объединяет улицу, город и индекс в одно целое — нельзя передать «часть» адреса.
