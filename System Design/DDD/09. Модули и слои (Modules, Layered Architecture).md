## Суть и контекст

**Модули (Modules)** и **Layered Architecture (Слоистая архитектура)** — два механизма организации кода в DDD. Модули группируют объекты **внутри** доменного слоя по бизнес-концепциям. Layered Architecture изолирует доменную модель от технических деталей, разделяя систему на специализированные слои.

---

## Ключевые идеи

- Модули организуют части внутри одной модели в логически связные группы (high cohesion, low coupling) для снижения когнитивной нагрузки.
- Layered Architecture разделяет ПО на четыре слоя: User Interface, Application, Domain, Infrastructure.
- Зависимости направлены **сверху вниз** — доменный слой не зависит от инфраструктуры.
- Пакеты внутри Domain Layer группируются **по бизнес-концепциям**, а не по типам паттернов.

---

## Определения и термины

**Module (Модуль / Пакет)** — элемент дизайна, организующий части внутри одной модели в логически связные группы. Выполняет роль физического пространства имён (namespace).

**Layered Architecture (Слоистая архитектура)** — паттерн архитектуры, разделяющий программное обеспечение на специализированные слои для изоляции доменной модели от технических деталей.

**User Interface Layer (Слой представления)** — отвечает за отображение информации пользователю и интерпретацию его команд. Внешним актором может быть как человек, так и другая компьютерная система.

**Application Layer (Слой приложения)** — определяет задачи ПО, не содержит бизнес-правил, координирует задачи и делегирует работу доменным объектам.

**Domain Layer (Доменный слой)** — содержит концепции бизнеса, информацию о бизнес-ситуациях и бизнес-правила. Полностью изолирован от деталей хранения и отображения. «Сердце» бизнес-ПО.

**Infrastructure Layer (Слой инфраструктуры)** — предоставляет общие технические возможности: отправка сообщений, персистентность, отрисовка UI. Поддерживает паттерны взаимодействия слоёв.

## Как устроено

### Layered Architecture: четыре слоя

```
┌─────────────────────────────────────────────┐
│          User Interface Layer               │
│  REST контроллеры, gRPC, CLI, MQ            │
│  ── принимает запросы, отображает данные ──  │
├─────────────────────────────────────────────┤
│          Application Layer                  │
│  Application Services, Use Cases            │
│  ── координирует, управляет транзакциями ── │
├─────────────────────────────────────────────┤
│          Domain Layer                       │
│  Entities, Value Objects, Aggregates,       │
│  Domain Services, Domain Events             │
│  ── бизнес-правила и бизнес-логика ──       │
├─────────────────────────────────────────────┤
│          Infrastructure Layer               │
│  ORM, Email, Message Broker, HTTP-клиенты   │
│  ── технические возможности ──              │
└─────────────────────────────────────────────┘
```

### Направление зависимостей

Любой элемент слоя зависит **только** от элементов того же слоя или слоёв, находящихся ниже. Доменный слой **не зависит** от UI, Application или Infrastructure.

Взаимодействие «снизу вверх» реализуется через **косвенные механизмы**: callbacks, паттерн Observer. Компоненты Application и Domain вызывают технические функции Infrastructure через интерфейсы (Services).

```
User Interface ──► Application ──► Domain ──► Infrastructure
                                      ↑
                          (интерфейсы, а не реализации)
```

### Dependency Inversion

Layered Architecture эволюционирует в [[10. Архитектура (Hexagonal, CQRS, Event Sourcing)|Hexagonal Architecture]], реализуя Dependency Inversion Principle. Зависимости от адаптеров направлены **внутрь** к доменной модели. Доменный слой определяет интерфейсы (Ports), инфраструктура их реализует (Adapters).

## Modules: организация кода внутри слоёв

### Отношение Module и Bounded Context

- **Bounded Context** задаёт строгую границу применимости всей модели и языка.
- **Modules** используются **внутри** Bounded Context для группировки объектов.

### Правильная структура пакетов

```
com.companyname.context.domain.model.concept

Пример:
com.myapp.ordering.domain.model.order      ← модуль Order
com.myapp.ordering.domain.model.customer   ← модуль Customer
com.myapp.ordering.domain.model.shipping   ← модуль Shipping
com.myapp.ordering.application             ← Application Services
com.myapp.ordering.infrastructure          ← Repositories, Adapters
```

### Именование

Имена модулей берутся из **Ubiquitous Language**: `Customer`, `Shipping`, `Billing` — а не из технических терминов.

Модули — это «главы» (chapters) в истории системы. Модель рассказывает историю, модули структурируют её для читателя.

## Сравнения и trade-offs

| | Разбивка по концепциям | Разбивка по паттернам |
|---|---|---|
| Пример | `order/`, `shipping/`, `billing/` | `entities/`, `values/`, `services/` |
| Cohesion | Высокий — связанные объекты вместе | Низкий — разрозненные объекты вместе |
| Coupling | Минимальный между модулями | Хаотичные ссылки между пакетами |
| Читаемость | Передаёт доменные знания | Не передаёт — только технические артефакты |

## Детали и нюансы

### Infrastructure-Driven Packaging

Тяжеловесные архитектурные фреймворки (например, ранний J2EE) требуют разделять Entity Bean (данные) и Session Bean (логику) по разным техническим пакетам. Концептуальный объект (Entity) разрушается на несколько технических уровней, инкапсуляция теряется, когнитивная нагрузка возрастает.

Все элементы, реализующие единый концептуальный объект, **должны находиться в одном модуле**, даже если это противоречит техническим соглашениям фреймворка.

### Эволюция модулей

Модули должны рефакториться и развиваться (coevolve) вместе с кодом и моделью. Игнорирование рефакторинга модулей приводит к тому, что структура пакетов отражает **устаревшую** версию модели.

## Важно / подводные камни

- **Не группируй по паттернам.** Пакеты `entities/`, `values/`, `services/` — антипаттерн. Объекты без концептуальной связи сваливаются вместе, а связанные — разбрасываются.
- **Доменный слой не зависит от инфраструктуры.** Интерфейс Repository принадлежит домену, реализация — инфраструктуре.
- **Модули = части Ubiquitous Language.** Если название модуля не имеет смысла для бизнес-эксперта — оно неправильное.
- **Рефакторь модули.** Структура пакетов — не фиксация, а живой документ, эволюционирующий вместе с моделью.
