## Суть и контекст

Три паттерна тактического DDD, управляющих **жизненным циклом объектов**: создание (Factory), хранение и извлечение (Repository), операции без владельца (Service).

---

## Repositories (Репозитории)

### Определение

**Repository** — объект, предоставляющий доступ к корням Агрегатов, позволяющий находить и извлекать их в середине или конце их жизненного цикла.

Репозиторий создаёт **иллюзию коллекции объектов в памяти**, скрывая за собой сложную инфраструктуру доступа к базе данных и формирование SQL-запросов. Клиентский код запрашивает нужные объекты, говоря исключительно **на языке модели**.

### Зачем нужен

Без репозитория бизнес-логика была бы пронизана SQL-запросами и деталями хранилища. Репозиторий изолирует доменную модель от инфраструктурных деталей.

### Ключевые правила

- Репозиторий создаётся только для **корней Агрегатов** — не для внутренних сущностей.
- Репозиторий — это **коллекция в памяти**: `add()`, `remove()`, `find_by_id()`.
- Интерфейс репозитория принадлежит **доменному слою**; реализация — инфраструктурному.

### Пример

```python
# Интерфейс — в доменном слое (говорит языком модели)
class OrderRepository(ABC):
    def find_by_id(self, order_id: UUID) -> Order: ...
    def find_by_customer(self, customer_id: UUID) -> list[Order]: ...
    def add(self, order: Order) -> None: ...
    def remove(self, order: Order) -> None: ...

# Реализация — в инфраструктурном слое
class SqlOrderRepository(OrderRepository):
    def find_by_id(self, order_id: UUID) -> Order:
        row = self.db.query("SELECT * FROM orders WHERE id = ?", order_id)
        return self._map_to_domain(row)
```

### Жизненный цикл объекта

```
Factory ──► [Агрегат создаётся] ──► Repository.add()
                                           │
Repository.find_by_id() ◄── [Агрегат живёт в БД]
         │
         ▼
    [Агрегат извлекается] ──► [изменяется] ──► [сохраняется]
```

## Factories (Фабрики)

### Определение

**Factory** — инкапсулирует сложную логику **создания** объектов или целых Агрегатов, защищая клиента от необходимости знать их внутреннюю структуру.

Создание сложного объекта не является его прямой бизнес-обязанностью. Подобно тому, как сборка двигателя автомобиля требует промышленных роботов и механиков, которые не нужны вам при самом вождении, — Фабрика берёт на себя «сборку» валидного объекта, после чего её работа заканчивается.

### Два применения

1. **Создание нового агрегата** — когда логика создания сложна и требует проверки инвариантов.
2. **Восстановление (reconstitution)** — воссоздание объекта из сохранённого состояния в базе данных (обычно используется внутри Repository).

### Пример

```python
class OrderFactory:
    def create(
        self,
        customer_id: UUID,
        items: list[dict],
        shipping_address: Address,
    ) -> Order:
        # Сложная логика создания: проверка items, расчёт total,
        # применение начальных скидок, генерация ID
        order = Order(order_id=uuid4(), customer_id=customer_id)
        for item in items:
            order.add_item(ProductId(item["id"]), item["qty"], Money(item["price"]))
        order.set_shipping_address(shipping_address)
        return order  # возвращает валидный агрегат
```

### Когда не нужна Фабрика

Если конструктор агрегата прост и не нарушает Ubiquitous Language — фабрика не обязательна.

## Services (Сервисы / Службы)

### Определение

**Domain Service** — stateless операция, предлагаемая в виде отдельного интерфейса, когда важное действие или бизнес-процесс **не является естественной частью** Сущности или Объекта-значения.

Имена Сервисов обычно выражаются **глаголами** — они обозначают активность, а не состояние.

### Когда использовать

- Операция логически принадлежит домену, но «неудобно» размещать её в одном конкретном агрегате.
- Операция включает несколько агрегатов.
- Операция выполняет значимый бизнес-процесс.

### Пример

Перевод средств (Funds Transfer) включает дебетование одного банковского счёта и кредитование другого с учётом глобальных правил. Эта операция не принадлежит ни `Account` отдельно — её координацию выносят в доменный сервис:

```python
class FundsTransferService:
    def transfer(
        self,
        from_account: Account,
        to_account: Account,
        amount: Money,
    ) -> None:
        if not from_account.can_debit(amount):
            raise InsufficientFundsError()
        from_account.debit(amount)
        to_account.credit(amount)
        # публикуем Domain Event
        from_account.record_event(FundsTransferred(from_account.id, to_account.id, amount))
```

### Три слоя сервисов

| Слой | Тип | Назначение |
|---|---|---|
| Доменный (Domain Service) | Бизнес-логика | Операции между агрегатами |
| Прикладной (Application Service) | Оркестрация | Координация UC: загрузить → изменить → сохранить |
| Инфраструктурный (Infrastructure Service) | Технология | Email, HTTP, очередь сообщений |

> Не путать: **Domain Service** содержит бизнес-логику. **Application Service** — оркестрирует, но сам логики не содержит.

## Жизненный цикл агрегата: всё вместе

```
1. Factory.create()          → создаёт валидный агрегат
2. Repository.add()          → сохраняет в хранилище
3. Repository.find_by_id()   → загружает обратно
4. DomainService.transfer()  → выполняет операцию над несколькими агрегатами
5. Repository.save()         → сохраняет изменения
6. Domain Event              → уведомляет другие части системы
```

---

## Важно / подводные камни

- **Репозиторий — только для корней агрегатов.** Если нужен «репозиторий внутренней сущности» — значит, неправильно определены границы агрегата.
- **Domain Service ≠ Application Service.** Сервис с бизнес-логикой — доменный. Сервис, который просто вызывает репозиторий и агрегат — прикладной.
- **Фабрика не хранит объекты.** Её работа заканчивается с возвратом созданного объекта — хранение делает Repository.
- **Stateless.** Domain Service не имеет состояния — только получает данные, применяет бизнес-логику и возвращает результат.

## Карточки для повторения

#flashcards/sd/ddd

Что такое Repository в DDD?::Объект, предоставляющий доступ к корням Агрегатов. Создаёт иллюзию коллекции объектов в памяти, скрывая детали инфраструктуры. Клиентский код говорит языком модели.

Для каких объектов создаётся Repository?::Только для корней Агрегатов (Aggregate Root). Не для внутренних Сущностей.

Где определяется интерфейс Repository?::В доменном слое (принадлежит домену). Реализация — в инфраструктурном слое.

Что такое Factory в DDD?::Инкапсулирует сложную логику создания объектов или Агрегатов. Защищает клиента от необходимости знать внутреннюю структуру. Используется при создании и при восстановлении (reconstitution) из БД.

Что такое Domain Service в DDD?::Stateless операция, выраженная глаголом, которая не принадлежит ни одной Сущности или Value Object естественным образом. Применяется для операций над несколькими агрегатами.

В чём разница Domain Service и Application Service?
?
- **Domain Service** — содержит **бизнес-логику** домена.
- **Application Service** — **оркестрирует**: загрузить агрегат из репозитория, вызвать метод, сохранить. Сам бизнес-логики не содержит.

Каков жизненный цикл агрегата?
?
1. Factory.create() — создание
2. Repository.add() — сохранение
3. Repository.find_by_id() — загрузка
4. [изменение через методы агрегата или Domain Service]
5. Repository.save() — сохранение изменений
6. Domain Event — уведомление других частей системы
