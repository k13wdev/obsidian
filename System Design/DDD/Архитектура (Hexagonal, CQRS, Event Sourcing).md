## Суть и контекст

Доменная модель концептуально нейтральна по отношению к архитектуре — она может размещаться внутри разных архитектурных стилей. Три паттерна наиболее часто используются совместно с DDD: **Hexagonal Architecture**, **CQRS** и **Event Sourcing**.

---

## Hexagonal Architecture (Ports and Adapters)

### Определение

**Hexagonal Architecture** (Гексагональная архитектура, «Порты и Адаптеры») — архитектурный стиль, в котором тактическая доменная модель располагается в самом «сердце» (центре) программного обеспечения, окружённая слоем приложения и множеством Адаптеров.

### Структура

```
                ┌─────────────────────────────────────┐
                │           АДАПТЕРЫ (внешний мир)    │
                │                                     │
 HTTP/REST  ──► │  ┌────────────────────────────────┐ │
 gRPC       ──► │  │    Application Service Layer   │ │
 CLI        ──► │  │  ┌──────────────────────────┐  │ │
 MQ Message ──► │  │  │    Domain Model (Core)   │  │ │
                │  │  │  Entities, Aggregates,   │  │ │
 DB     ◄────── │  │  │  Value Objects, Services │  │ │
 Email  ◄────── │  │  └──────────────────────────┘  │ │
 S3     ◄────── │  └────────────────────────────────┘ │
                │                                     │
                └─────────────────────────────────────┘
```

- **Ядро (Core)** — доменная модель. Не зависит ни от чего снаружи.
- **Порты (Ports)** — интерфейсы, через которые внешний мир взаимодействует с ядром.
- **Адаптеры (Adapters)** — конкретные реализации: HTTP-контроллер, Kafka-консьюмер, SQL-репозиторий. Преобразуют внешние запросы во внутренние вызовы доменной модели.

### Плюсы

Использование Hexagonal Architecture облегчает внедрение и поддержку других архитектурных стилей: SOA, REST и Event-Driven Architecture. Доменная модель остаётся независимой от инфраструктуры и легко тестируемой.

### Направление зависимостей

```
HTTP Controller ──► Application Service ──► Domain Model
SQL Repository ──► (реализует порт домена)
                   ↑
            Domain Model знает только об интерфейсе
```

**Зависимости всегда направлены внутрь** — от адаптеров к ядру, никогда наоборот.

## CQRS (Command Query Responsibility Segregation)

### Определение

**CQRS** — паттерн разделения ответственности за команды и запросы. Система разбивается на две разные модели:
- **Command Model (Write Model)** — модель записи, изменяет состояние.
- **Query Model (Read Model)** — модель чтения, только читает.

### Проблема, которую решает

В DDD Агрегаты ссылаются друг на друга только по ID (не по прямым ссылкам). Это усложняет формирование «богатых» представлений для UI — нужно загружать много агрегатов по отдельности.

**CQRS решает это**, разделяя оптимизированную для записи модель (Агрегаты с инвариантами) и денормализованную модель для чтения (готовые плоские проекции для UI).

### Как работает

```
Команда (Command)
    │
    ▼
Command Handler
    │
    ├─► Aggregate (Domain Model)
    │        └─► публикует Domain Event
    │
    ▼
Event Store / Message Bus
    │
    ▼
Event Subscriber (обновляет Read Model)
    │
    ▼
Read Model (денормализованная проекция)
    │
    ▼
Query Handler ──► UI / Client
```

1. Клиент отправляет **команду** (PlaceOrder, CancelOrder).
2. Command Handler загружает агрегат, применяет команду, сохраняет.
3. Агрегат публикует **Domain Event**.
4. **Event Subscriber** обновляет **Read Model** — денормализованную проекцию для чтения.
5. При запросе клиент читает из **Read Model** напрямую — быстро и просто.

### Плюсы и минусы

| | Плюсы | Минусы |
|---|---|---|
| Производительность | Read Model оптимизирована под конкретные запросы | Дополнительная сложность |
| Масштабирование | Read и Write масштабируются независимо | Eventual consistency для чтения |
| Ясность | Чёткое разделение намерений (Command / Query) | Требует сдвига в мышлении |

**Ключевой компромисс:** CQRS означает строгий переход к **eventual consistency** для модели запросов. Read Model «немного отстаёт» от Write Model — это нужно принять и учитывать в проектировании UI.

## Event Sourcing (Порождение событий)

### Определение

**Event Sourcing** — подход к персистентности Агрегатов, при котором **текущее состояние не сохраняется напрямую**. Вместо этого сохраняется **непрерывная последовательность событий** (Events), произошедших с момента создания агрегата.

### Как работает

```
# Традиционный подход:
БД: { order_id: 1, status: "shipped", total: 500 }

# Event Sourcing:
Event Store:
  1. OrderPlaced    { order_id: 1, total: 500 }
  2. PaymentReceived{ order_id: 1, amount: 500 }
  3. OrderShipped   { order_id: 1, tracking: "..." }

Текущее состояние = воспроизведение (replay) всех событий по порядку
```

Чтобы получить текущее состояние агрегата — система извлекает события из **Event Store** и **воспроизводит (replays)** их в том же порядке.

### Связь с CQRS

Event Sourcing часто работает в тесной связке с CQRS:
- **Event Store** → источник событий для обновления Read Model.
- **Read Model** строится путём обработки потока событий из Event Store.

```
Write: Command ──► Aggregate ──► Event Store
Read:  Event Store ──► Projections ──► Read Model ──► Query
```

### Плюсы

- Полная **история изменений** агрегата — аудит из коробки.
- Возможность **воспроизвести** состояние на любой момент времени.
- Упрощает механизм сохранения сложных объектов с богатым поведением.
- Естественная основа для **Event-Driven Architecture**.

### Минусы

- Сложность запросов текущего состояния (нужна CQRS для Read Model).
- **Schema evolution**: изменение формата старых событий — нетривиальная задача.
- Требует **Event Store** — специализированного хранилища (EventStoreDB, Kafka).

## Интеграция между Bounded Contexts через события

При событийно-ориентированном подходе Domain Events пересылаются по всей системе через Message Bus / Event Store:

```
Bounded Context «Заказы»           Bounded Context «Склад»
────────────────────────           ────────────────────────
BacklogItem.commitToSprint()
    └─► BacklogItemCommitted {
            tenantId,              ──► Message Bus (RabbitMQ / Kafka)
            backlogItemId,                  │
            sprintId                        ▼
        }                          Event Subscriber получает событие
                                   └─► загружает Sprint из репозитория
                                   └─► применяет бизнес-логику
                                   └─► сохраняет в своей транзакции
```

**Ключевые детали:**
- Транзакции между распределёнными системами **не являются атомарными**.
- Различные системы приводят свои Агрегаты в согласованное состояние **с течением времени** (eventual consistency).
- Передача событий через: **RESTful ресурсы**, **Message Broker (RabbitMQ, Kafka)** или **Event Store**.
- Подписчик получает только идентификаторы и использует их для операций в **своей** доменной модели.

## Как паттерны работают вместе

```
┌──────────────────────────────────────────────────────┐
│              Hexagonal Architecture                  │
│                                                      │
│  Adapters         Application Layer    Domain Core   │
│  ─────────        ────────────────    ───────────    │
│  HTTP API ──► Command Handler ──► Aggregate          │
│  MQ       ──► Event Handler   │       └─► Event      │
│                               │              │       │
│         CQRS: Write side ─────┘              │       │
│                                              ▼       │
│              Event Store ◄───────────────────┘       │
│                    │                                 │
│         CQRS: Read side                              │
│              Event Store ──► Projection ──► Read DB  │
│                                              │       │
│  HTTP API ◄── Query Handler ◄────────────────┘       │
└──────────────────────────────────────────────────────┘
         Event Sourcing: Event Store = источник истины
```

## Важно / подводные камни

- **CQRS — не обязателен везде.** Начни без CQRS; добавляй только если производительность чтения или масштабируемость являются реальной проблемой.
- **Event Sourcing — не серебряная пуля.** Для простых CRUD-операций Event Sourcing создаёт лишнюю сложность. Применяй только там, где история изменений имеет ценность.
- **Eventual consistency — это контракт с пользователем.** Если UI показывает «немного устаревшие» данные — пользователь должен понимать это. Проектируй UX соответственно.
- **Schema evolution событий.** Изменение старых событий в Event Sourcing — сложная задача. Используй версионирование с самого начала.

## Карточки для повторения

#flashcards/sd/ddd

Что такое Hexagonal Architecture?::Архитектурный стиль, где доменная модель находится в центре, окружённая адаптерами. Зависимости направлены внутрь. Порты — интерфейсы; Адаптеры — конкретные реализации (HTTP, SQL, Kafka).

Что такое CQRS?::Паттерн разделения Command Model (запись, изменяет состояние) и Query Model (чтение, денормализованные проекции). Решает проблему формирования «богатых» представлений при ссылках агрегатов только по ID.

Каков главный компромисс CQRS?::Строгий переход к eventual consistency для Read Model. Модель чтения «немного отстаёт» от модели записи.

Что такое Event Sourcing?::Подход к персистентности, при котором сохраняется не текущее состояние агрегата, а полная последовательность событий. Текущее состояние восстанавливается воспроизведением (replay) событий.

В чём преимущество Event Sourcing?::Полная история изменений (аудит из коробки), воспроизведение состояния на любой момент времени, естественная основа для Event-Driven Architecture.

Как Event Sourcing и CQRS работают вместе?::Event Store — источник событий; события воспроизводятся для построения Read Model (проекций). Write side пишет в Event Store; Read side читает из проекций.

Как Domain Events используются для интеграции между Bounded Contexts?::Событие публикуется в Message Bus (RabbitMQ/Kafka) или Event Store. Подписчик в другом контексте получает событие, загружает свой агрегат по ID из события и выполняет бизнес-логику в своей транзакции. Eventual consistency — без двухфазных коммитов.
