## Суть и контекст

**Domain-Driven Design (DDD)** — подход к разработке программного обеспечения, который позволяет справляться со сложностью за счёт тесной связи модели предметной области с её реализацией в коде.

Проекты, которые просто реализуют одну функцию за другой без единой концептуальной модели, быстро поглощаются сложностью предметной области и становятся нечитаемыми и трудно поддерживаемыми.

---

## Ключевые идеи

- DDD ставит **предметную область (domain)** — бизнес-знание — в центр проектирования.
- Цель — создать **модель**, которая отражает реальные бизнес-процессы, а не просто хранит данные.
- DDD состоит из двух уровней: **стратегический** (как разбить систему) и **тактический** (как построить отдельные части).
- Разработчики и бизнес-эксперты **совместно** строят модель — это не односторонний процесс.
- Когда меняется понимание предметной области — меняется модель, а вслед за ней меняется и код.

---

## Зачем нужен DDD

Использование DDD приносит бизнесу и команде конкретную пользу:

- Организация получает **полезную модель** своей деятельности.
- Улучшается и уточняется **понимание бизнес-процессов** у всех участников.
- **Эксперты предметной области** активно участвуют в проектировании — не только в написании требований.
- В архитектуре создаются **чёткие и чистые границы**, снижающие зацепление между частями системы.

---

## Определения и термины

**Domain (Предметная область)** — сфера знаний и деятельности, которую автоматизирует программная система. Например: банковское дело, логистика, медицина.

**Domain Model (Модель предметной области)** — набор концепций, терминов и связей, отражающих глубокое понимание бизнеса. Это не просто схема данных — объекты модели обладают поведением, обеспечивают выполнение бизнес-правил и являются ключом к решению сложных проблем приложения.

**Стратегический дизайн** — высокоуровневый уровень DDD: как разбить большую систему на части (Bounded Contexts), как эти части взаимодействуют (Context Map), что является ядром системы (Core Domain).

**Тактический дизайн** — низкоуровневый уровень DDD: строительные блоки (Entities, Value Objects, Aggregates, Repositories, Factories, Services, Domain Events).

**Знание в коде** — принцип DDD: код должен явно выражать бизнес-концепции, а не скрывать их за техническими деталями.

**Ограниченный контекст (Bounded Context)** — стиль разделения (decoupling), при котором всё относящееся к данной предметной области доступно внутри контекста, но непрозрачно для других контекстов. Каждый контекст включает собственные классы и схемы БД. Дублирование кода (например, класса Customer) предпочтительнее высокой связности (coupling).

**Value Object (VO)** — неизменяемый объект (immutable), валидация которого происходит строго при создании. Примеры: HullStrength, Money, Email.

**Entity (Сущность)** — объект, имеющий уникальный идентификатор (ID) и содержащий поведение (бизнес-логику).

**Aggregate / Aggregate Root** — корень и точка входа, аккумулятор, объединяющий Value Objects и Entities. Пример: Spaceship.InstallEngine().

**Anti-corruption Layer (ACL)** — слой адаптера-переводчика, выступающий «погранконтролем» между внешними системами и доменом. Определяет интерфейс на терминах внутреннего домена (НАШЕМ языке), транслируя их во внешние вызовы.

**Domain Event** — паттерн публикации факта без знания о подписчиках.

**Event Sourcing** — паттерн хранения событий, а не текущего состояния системы.

## Два уровня DDD

```
DDD
├── Стратегический дизайн
│   ├── Ubiquitous Language    — единый язык команды
│   ├── Bounded Context        — явные границы модели
│   ├── Context Map            — карта взаимодействий контекстов
│   └── Core Domain            — ядро, которое даёт конкурентное преимущество
│
└── Тактический дизайн
    ├── Entity                 — объект с идентичностью
    ├── Value Object           — объект без идентичности (неизменяемый)
    ├── Aggregate              — кластер объектов с корнем
    ├── Repository             — доступ к агрегатам
    ├── Factory                — создание сложных объектов
    ├── Service                — stateless операции
    └── Domain Event           — уведомление о значимом событии
```

## Как устроено

### Anti-corruption Layer (ACL) — процесс внедрения

1. Внешняя система отдаёт данные в своём формате (например, JSON с vendor_sku и lead_time_cycles)
2. Создаётся интерфейс-шлюз (Gateway) в терминах домена
3. ACL-адаптер делает вызов во внешнюю систему и мапит чужие термины на Value Objects домена (например, lead_time_cycles → LeadDays)

### CQRS-lite в DDD

- **Aggregate** используется исключительно для ИЗМЕНЕНИЙ (Write path) и контроля бизнес-инвариантов
- **Read Model** используется для ОТОБРАЖЕНИЯ (Read path) — данные собираются напрямую плоским SQL-запросом для UI без участия логики Агрегата

## Детали и нюансы

- **Анемичная модель (Anemic Domain Model)** — антипаттерн: структуры данных не имеют поведения, вся логика выносится в слой сервиса (200+ строк if-ов). Дублирование бизнес-правил, сложность тестирования. Решается переходом к Богатой модели (Rich Model), где логика живёт рядом с данными в Entities
- **Разделение без DDD**: команды переиспользуют общие классы и БД → тесная связность (coupling). Изменение в одной части ломает зависимые сервисы. Микросервисы требуют дублирования кода между контекстами для полной изоляции
- **ACL не только для внешних API**: использовать ACL также для собственных баз данных (PostgreSQL, Redis), чтобы смена хранилища не ломала бизнес-домен

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| Анемичная модель (Anemic) | Простая структура, привычный стиль | Логика в сервисах (200+ строк if-ов), дублирование правил |
| Богатая модель (Rich / DDD) | Логика рядом с данными, тестируемость | Сложнее начальная архитектура |
| Общие классы между сервисами | Нет дублирования кода | Тесная связность, каскадные поломки |
| Дублирование в Bounded Contexts | Полная изоляция, независимость | Дублирование кода (осознанный trade-off) |

## Важно

- DDD **не подходит** для простых CRUD-систем — накладные расходы не оправданы.
- DDD наиболее ценен там, где **сложность предметной области** является главным вызовом.
- Начинать всегда со **стратегического дизайна** — сначала понять границы, потом строить детали.
