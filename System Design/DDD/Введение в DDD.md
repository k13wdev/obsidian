## Суть и контекст

**Domain-Driven Design (DDD)** — подход к разработке программного обеспечения, который позволяет справляться со сложностью за счёт тесной связи модели предметной области с её реализацией в коде.

Проекты, которые просто реализуют одну функцию за другой без единой концептуальной модели, быстро поглощаются сложностью предметной области и становятся нечитаемыми и трудно поддерживаемыми.

---

## Ключевые идеи

- DDD ставит **предметную область (domain)** — бизнес-знание — в центр проектирования.
- Цель — создать **модель**, которая отражает реальные бизнес-процессы, а не просто хранит данные.
- DDD состоит из двух уровней: **стратегический** (как разбить систему) и **тактический** (как построить отдельные части).
- Разработчики и бизнес-эксперты **совместно** строят модель — это не односторонний процесс.
- Когда меняется понимание предметной области — меняется модель, а вслед за ней меняется и код.

---

## Зачем нужен DDD

Использование DDD приносит бизнесу и команде конкретную пользу:

- Организация получает **полезную модель** своей деятельности.
- Улучшается и уточняется **понимание бизнес-процессов** у всех участников.
- **Эксперты предметной области** активно участвуют в проектировании — не только в написании требований.
- В архитектуре создаются **чёткие и чистые границы**, снижающие зацепление между частями системы.

---

## Определения и термины

**Domain (Предметная область)** — сфера знаний и деятельности, которую автоматизирует программная система. Например: банковское дело, логистика, медицина.

**Domain Model (Модель предметной области)** — набор концепций, терминов и связей, отражающих глубокое понимание бизнеса. Это не просто схема данных — объекты модели обладают поведением, обеспечивают выполнение бизнес-правил и являются ключом к решению сложных проблем приложения.

**Стратегический дизайн** — высокоуровневый уровень DDD: как разбить большую систему на части (Bounded Contexts), как эти части взаимодействуют (Context Map), что является ядром системы (Core Domain).

**Тактический дизайн** — низкоуровневый уровень DDD: строительные блоки (Entities, Value Objects, Aggregates, Repositories, Factories, Services, Domain Events).

**Знание в коде** — принцип DDD: код должен явно выражать бизнес-концепции, а не скрывать их за техническими деталями.

---

## Два уровня DDD

```
DDD
├── Стратегический дизайн
│   ├── Ubiquitous Language    — единый язык команды
│   ├── Bounded Context        — явные границы модели
│   ├── Context Map            — карта взаимодействий контекстов
│   └── Core Domain            — ядро, которое даёт конкурентное преимущество
│
└── Тактический дизайн
    ├── Entity                 — объект с идентичностью
    ├── Value Object           — объект без идентичности (неизменяемый)
    ├── Aggregate              — кластер объектов с корнем
    ├── Repository             — доступ к агрегатам
    ├── Factory                — создание сложных объектов
    ├── Service                — stateless операции
    └── Domain Event           — уведомление о значимом событии
```

## Важно

- DDD **не подходит** для простых CRUD-систем — накладные расходы не оправданы.
- DDD наиболее ценен там, где **сложность предметной области** является главным вызовом.
- Начинать всегда со **стратегического дизайна** — сначала понять границы, потом строить детали.

В чём разница между стратегическим и тактическим дизайном в DDD?
?
- **Стратегический** — высокоуровневый: Bounded Context, Ubiquitous Language, Context Map, Core Domain. Отвечает на вопрос «как разбить систему».

Когда DDD особенно ценен?::Когда **сложность предметной области** является главным вызовом. Для простых CRUD-систем накладные расходы DDD не оправданы.

Какую пользу приносит DDD бизнесу?
?
- Полезная модель деятельности организации.
- Улучшение и уточнение понимания бизнес-процессов.
- Активное участие экспертов предметной области в проектировании.
- Чёткие и чистые архитектурные границы между частями системы.
