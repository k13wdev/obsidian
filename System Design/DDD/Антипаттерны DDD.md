## Суть и контекст

Антипаттерны в DDD — типичные ошибки проектирования, которые разрушают преимущества доменно-ориентированного подхода. Знание антипаттернов позволяет распознать проблему до того, как она станет архитектурным долгом.

---

## Anemic Domain Model (Анемичная доменная модель)

### Определение

**Anemic Domain Model** — модель, лишённая поведения, которая сводится к простым структурам данных и удовлетворяет только требования доступа к базе данных. Вся бизнес-логика реализуется отдельными службами (Services).

### Почему это плохо

- Объекты предметной области разделяются на несколько технических уровней: отдельный слой данных и отдельный слой поведения (Session Beans или Services).
- Концептуальная сущность **фрагментируется** — инкапсуляция теряется.
- Система лишается преимуществ объектно-ориентированной парадигмы, бизнес-логика смешивается со сценариями доступа к БД.

### Как выглядит

```
❌ Анемичная модель:

Order                          OrderService
├── order_id                   ├── calculateTotal(order)
├── items[]                    ├── applyDiscount(order, code)
├── status                     ├── cancel(order)
├── get/set методы             └── validate(order)
└── (никакого поведения)       (вся логика здесь)
```

```
✅ Богатая доменная модель:

Order
├── order_id
├── items[]
├── status
├── place(customer_id, items)      ← поведение внутри
├── applyDiscount(discountCode)    ← бизнес-правила
├── cancel()                       ← инварианты защищены
└── calculateTotal()               ← домен решает
```

### Связь с Transaction Script

**Transaction Script** — процедурный паттерн, не предоставляющий объектной модели предметной области. Anemic Domain Model и Transaction Script тесно связаны: обе парадигмы переносят ответственность из структуры объектов в процедурный код сервисов.

## God Aggregate (Слишком большой агрегат)

### Определение

**Large-Cluster Aggregate / God Aggregate** — агрегат, спроектированный с избыточным количеством вложенных сущностей и коллекций, связанных прямыми ссылками.

### Проблемы

1. **Optimistic Concurrency конфликты:** объекты корня агрегата инкрементируют версию при любом изменении. Если агрегат велик — несколько пользователей, меняя разные части, вызывают откат транзакций.
2. **Производительность:** при ленивой загрузке добавление одного элемента в огромную коллекцию требует загрузки в память тысяч связанных объектов.
3. **Масштабируемость:** блокировка всего агрегата при каждом изменении.

### Пример

```java
// ❌ God Aggregate — Product владеет всем
public class Product extends ConcurrencySafeEntity {
    private Set<BacklogItem> backlogItems;  // тысячи элементов
    private String description;
    private String name;
    private ProductId productId;
    private Set<Release> releases;          // ещё коллекция
    private Set<Sprint> sprints;            // и ещё
    private TenantId tenantId;
}
```

Любое изменение в `BacklogItem`, `Release` или `Sprint` блокирует весь `Product`.

### Исправление

```java
// ✅ Small Aggregate — BacklogItem как независимый корень
public class BacklogItem extends ConcurrencySafeEntity {
    ...
    private ProductId productId;  // ссылка по ID, не прямая
    ...
}
```

`BacklogItem`, `Release`, `Sprint` становятся независимыми корнями агрегатов. Связь — только через идентификаторы.

## Утечка доменной логики в Application Service

### Определение

Если слой приложения начинает выполнять вычисления бизнес-правил, доменное знание перетекает в код приложения. Слой приложения должен оставаться **тонким координатором**: не содержать бизнес-правил и бизнес-состояния, а лишь делегировать работу доменным объектам.

### Признаки

- `if`/`else` с бизнес-условиями в Application Service
- Вычисления (цены, скидки, лимиты) в Application Service
- Доменные правила валидации в Application Service

## Dependency Injection внутрь агрегатов

### Определение

Антипаттерн внедрения ссылок на Repository или Domain Service внутрь экземпляров агрегатов.

### Почему это плохо

- В высоконагруженных системах создание множества ссылок на сервисы внутри **тысяч экземпляров** доменных объектов вызывает оверхед памяти.
- Увеличивается нагрузка на Garbage Collector.
- Нарушает чистоту модели и усложняет сериализацию.

### Исправление: Double Dispatch

Application Service сам получает зависимости из репозиториев и передаёт их как аргументы в методы агрегата:

```java
// ❌ Плохо: DI внутрь агрегата
class BacklogItem {
    private TeamRepository teamRepository;  // зависимость в поле

    void assignToTask(String teamMemberId, String taskId) {
        Team team = teamRepository.teamOfId(this.teamId);
        // ...
    }
}

// ✅ Хорошо: Double Dispatch
class BacklogItem {
    void assignTeamMemberToTask(
        TeamMemberId memberId, Team ofTeam, TaskId taskId) {
        // Team передана как аргумент — нет зависимости в полях
    }
}
```

## Smart UI (Умный пользовательский интерфейс)

### Определение

**Smart UI** — антипаттерн, при котором бизнес-логика помещается непосредственно в пользовательский интерфейс. Приложение разбивается на мелкие функции (отдельные UI), а реляционная база данных используется как общее хранилище.

### Последствия

- Объекты не абстрагируют бизнес-проблему.
- Переиспользование поведения невозможно.
- Миграция на другой дизайн невозможна без полного переписывания.

Smart UI **явно противопоставляется** Layered Architecture — это взаимоисключающая развилка.

> Smart UI рекомендуется только для проектов без высоких требований к абстракции, с короткими сроками и отсутствием планов на масштабирование.

## Нарушение правила одной транзакции

### Определение

Транзакционное изменение нескольких агрегатов в рамках одного Use Case нарушает правило DDD. При использовании распределённых транзакций (two-phase commits) система теряет способность к линейному масштабированию.

### Исправление

Использовать **Eventual Consistency** через Domain Events. Агрегат публикует событие, другие агрегаты асинхронно обновляются в отдельных транзакциях.

**Исключение:** Batch Processing — создание списка элементов за один вызов, где семантически это равноценно созданию по одному.

## Сравнения и trade-offs

| Антипаттерн | Симптом | Исправление |
|---|---|---|
| Anemic Domain Model | Объекты без поведения, логика в Service | Перенести логику в доменные объекты |
| God Aggregate | Частые конфликты транзакций, медленная загрузка | Разбить на Small Aggregates, связь по ID |
| Утечка логики в Application Service | `if`/`else` с бизнес-условиями в координаторе | Перенести в Domain Service или агрегат |
| DI в агрегаты | Repository/Service в полях агрегата | Double Dispatch — передавать как аргументы |
| Smart UI | Логика в представлении, нет модели | Layered Architecture |
| Нарушение 1 транзакция = 1 агрегат | Two-phase commits, блокировки | Domain Events, Eventual Consistency |

## Важно / подводные камни

- **Anemic Domain Model — самый распространённый антипаттерн.** Объекты с только get/set — сигнал, что бизнес-логика вытекла в сервисы.
- **God Aggregate — самая частая ошибка начинающих.** Начинай с маленьких агрегатов.
- **Smart UI — осознанный выбор,** но он закрывает путь к DDD навсегда.
- **Отключение оптимистичной блокировки** решает проблему транзакций God Aggregate, но приводит к неограниченному росту коллекций в памяти и падению производительности.
