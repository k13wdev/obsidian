## Суть и контекст

Выбор модели данных и storage engine — фундаментальное решение при проектировании системы. Реляционная модель (SQL) организует данные в таблицы с явной схемой, а документная (NoSQL) — в самодостаточные JSON-документы. На уровне хранения два основных подхода: B-Trees (оптимизированы для чтения) и LSM-Trees (оптимизированы для записи). Для аналитики применяется column-oriented storage.

---

## Ключевые идеи

- Реляционная модель нормализует данные для устранения дублирования, требуя JOIN
- Документная модель хранит данные иерархически (JSON), повышая data locality
- Schema-on-write (SQL) аналогична статической типизации, schema-on-read (NoSQL) — динамической
- B-Trees — стандарт для OLTP с предсказуемыми задержками чтения
- LSM-Trees — выше пропускная способность записи благодаря последовательной записи на диск
- Нормализация (1NF → 2NF → 3NF) устраняет аномалии обновления, но увеличивает количество JOIN
- Денормализация ускоряет чтение, но увеличивает избыточность и риск ошибок
- Column-oriented storage оптимизирован для OLAP (аналитика), сканирование миллионов строк

## Определения и термины

**Relational Model** — данные организованы в отношения (relations/tables), где каждое отношение является неупорядоченной коллекцией кортежей (tuples/rows). Схема задаётся явно при записи (schema-on-write).

**Document Model** — данные хранятся в виде самодостаточных документов (JSON, XML), структура которых интерпретируется только при чтении (schema-on-read). Иерархическая структура внутри документа (дерево).

**Normalization (Нормализация)** — процесс преобразования сложных структур данных в простые и стабильные:
- **1NF** — таблица не содержит повторяющихся групп (repeating groups). Первичный ключ расширяется
- **2NF** — таблица в 1NF, и каждый неключевой атрибут функционально зависит от **всего** первичного ключа (full functional dependency)
- **3NF** — таблица в 2NF, и ни один неключевой атрибут не зависит от другого неключевого атрибута (нет транзитивных зависимостей)

**Denormalization (Денормализация)** — процесс разделения или объединения нормализованных отношений в физические таблицы на основе степени совместного использования строк и полей. Снижает количество JOIN для ускорения чтения, но увеличивает избыточность.

**Data locality (Локальность данных)** — размещение нескольких фрагментов данных в одном месте, если они часто требуются одновременно. Документная модель имеет хорошую локальность (весь документ — непрерывная строка). Реляционная при нормализации разносит данные по таблицам.

**B-Tree** — структура индекса, которая разбивает БД на блоки или страницы фиксированного размера (обычно 4 КБ) и читает/пишет по одной странице за раз.

**SSTable (Sorted String Table)** — формат сегментного файла, в котором последовательность пар ключ-значение отсортирована по ключу. Каждый ключ встречается ровно один раз в каждом объединённом файле.

**LSM-Tree (Log-Structured Merge-Tree)** — индексирующая структура, основанная на принципе объединения и сжатия отсортированных файлов.

**Memtable** — сбалансированное дерево в оперативной памяти (например, красно-чёрное дерево), в которое добавляются новые записи в отсортированном виде перед сбросом на диск в SSTable.

**Secondary Index** — дополнительная структура данных, поддерживаемая параллельно с основным хранилищем, позволяющая эффективно искать записи по значению определённого поля.

**In-memory database** — БД, которая хранит данные целиком в оперативной памяти, используя диск только как append-only журнал для обеспечения долговечности (durability). Чтение обслуживается исключительно из памяти.

**Column-oriented storage** — хранилище, в котором все значения каждого столбца хранятся вместе в отдельном файле, а не все значения одной строки вместе.

## Как устроено

### B-Tree — поиск

1. Начать с корневой страницы (root page)
2. Найти диапазон, в который попадает искомый ключ
3. Перейти по ссылке на дочернюю страницу
4. Повторять до достижения листовой страницы (leaf page) с ключами и значениями

### B-Tree — вставка / обновление

1. Найти листовую страницу для ключа
2. **Обновление**: изменить значение и перезаписать страницу на диск
3. **Вставка (есть место)**: добавить ключ на страницу
4. **Вставка (переполнение)**: разделить страницу на две полупустые, обновить родительскую страницу новыми границами ключей

### LSM-Tree — запись и compaction

1. Запись добавляется в **WAL** (для защиты от сбоев) и в **in-memory memtable**
2. При превышении порога памяти (несколько МБ) memtable сбрасывается на диск как новый файл SSTable
3. Фоновый процесс **Compaction** берёт файлы SSTable, использует алгоритм mergesort для слияния, оставляет только самые свежие значения для каждого ключа и удаляет записи с меткой удаления (tombstone)

### Column-oriented — Bitmap Index запрос

1. Запрос `WHERE product_sk IN (30, 68, 69)`
2. Движок загружает три bitmap для значений 30, 68 и 69
3. Выполняется побитовое ИЛИ (bitwise OR) — аппаратно крайне быстро

## Детали и нюансы

- **Crash при split страницы B-Tree**: возможна частичная запись — "осиротевшая" (orphan) страница и повреждение индекса. Решение: WAL (Write-Ahead Log / redo log) — изменения строго пишутся в WAL перед изменением B-Tree
- **Падение LSM-Tree**: записи в memtable (не сброшенные в SSTable) теряются. Решение: отдельный on-disk WAL для восстановления memtable
- **Поиск отсутствующего ключа в LSM-Tree**: приходится проверять memtable + все файлы SSTable вплоть до самых старых. Решение: **Bloom filters** — компактная структура в памяти, с высокой вероятностью говорящая, что ключа точно нет
- **Hash index**: все ключи должны помещаться в RAM. Неэффективен для range queries, так как ключи не отсортированы
- **Update в Column-oriented storage**: невозможен in-place для сжатых колонок. Решение: LSM-Tree архитектура — записи идут в in-memory store, затем батчами сливаются с колоночными файлами (так работает Vertica)
- **Write amplification в LSM-Trees**: одна логическая запись → множественные физические перезаписи (compaction), влияет на срок службы SSD и забирает пропускную способность
- **Fragmentation в B-Trees**: страницы фиксированного размера → потери места при split или когда строка не помещается в остаток страницы
- **Shredding** — антипаттерн: разбиение простого документа (например, резюме) на десятки реляционных таблиц → неуклюжие схемы и переусложнённый код
- Связи M:N в документных БД решаются через **document references** — аналог foreign keys в реляционной модели

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Relational Model (SQL) | Нормализация, JOIN, строгая схема, транзакции | Shredding для древовидных данных, множественный поиск по индексу | Сильно связанные данные (many-to-many) |
| Document Model (NoSQL) | Data locality, гибкая схема, простые one-to-many | Сложно many-to-many, нет JOIN (или слабый) | Древовидные данные, загрузка целиком |
| Schema-on-write | Гарантии целостности при записи | Жёсткая миграция при изменении схемы | Когда важна строгость данных |
| Schema-on-read | Гибкость, нет миграций | Ошибки обнаруживаются позже | Гетерогенные данные |
| Нормализация | Нет дублирования, нет аномалий обновления | Много JOIN, сложные запросы | OLTP, целостность данных |
| Денормализация | Быстрое чтение, меньше JOIN | Избыточность, риск аномалий | Аналитика, Star Schema |
| B-Trees | Предсказуемое чтение, транзакционные блокировки | Fragmentation, random I/O при записи | Read-heavy, OLTP |
| LSM-Trees | Высокая пропускная способность записи, лучше сжатие | Write amplification, медленнее чтение отсутствующих ключей | Write-heavy |
| Row-oriented | Быстрый доступ к полной строке | Неэффективно при сканировании колонок | OLTP |
| Column-oriented | Сжатие, bitmap index, быстрое сканирование | Медленный update, не для точечных запросов | OLAP, аналитика |

## Примеры

### Append-only лог на Bash (простейшая БД)

```bash
db_set() { echo "$1,$2" >> database; }
db_get() { grep "^$1," database | sed "s/^$1,//" | tail -n 1; }
```

`db_set` дописывает пару в конец файла. `db_get` ищет все записи по ключу, забирает последнее (самое актуальное) значение через `tail -n 1`.

### Документная модель (JSON) — профиль пользователя

Отношения один-ко-многим (один пользователь — много должностей) вложены внутрь документа, создавая древовидную структуру.

### OLTP vs OLAP

| Характеристика | OLTP | OLAP |
|---------------|------|------|
| Storage | B-Tree, Row-oriented | Column-oriented |
| Запросы | Точечные (by key) | Сканирование миллионов строк |
| Шаблон | SELECT * WHERE id = X | SELECT SUM(...) GROUP BY ... |
| Оптимизация | Индексы, кэш | Сжатие, bitmap, векторные вычисления |

## Важно / подводные камни / best practices

- **Документная модель** — если данные древовидные (one-to-many), загружаются целиком, редко требуют M:N связей
- **Реляционная модель** — если данные сильно связаны (highly interconnected)
- **LSM-Trees для write-heavy**, **B-Trees для read-heavy** с предсказуемыми задержками
- **WAL обязателен** и для B-Trees (crash при split), и для LSM-Trees (потеря memtable)
- **Bloom filters** — обязательны для LSM-Trees для ускорения поиска отсутствующих ключей
- **Денормализация** — допустима для аналитики (Star Schema), но увеличивает риск аномалий
- **Shredding** — антипаттерн: не разбивать простой документ на десятки таблиц
- **Column-oriented** — для OLAP, не для OLTP

In-memory database:::БД целиком в RAM, диск — только append-only журнал для durability. Чтение исключительно из памяти

Нормализация 2NF::Таблица в 1NF + каждый неключевой атрибут зависит от всего первичного ключа (full functional dependency)

Нормализация 3NF::Таблица в 2NF + нет транзитивных зависимостей (неключевой атрибут не зависит от другого неключевого)

Шаги поиска в B-Tree
?
1. Начать с корневой страницы (root)
2. Найти диапазон для искомого ключа
3. Перейти по ссылке на дочернюю страницу
4. Повторять до leaf page с ключами и значениями

Шаги записи в LSM-Tree
?
1. Запись в WAL + in-memory memtable
2. При превышении порога memtable сбрасывается на диск как SSTable
3. Фоновый Compaction: mergesort SSTable файлов, удаление старых значений и tombstone

Что такое Write Amplification в LSM-Trees?::Одна логическая запись приводит к множественным физическим перезаписям из-за compaction. Влияет на срок службы SSD и забирает пропускную способность

Зачем нужен Bloom filter в LSM-Tree?::Для ускорения поиска отсутствующих ключей. Без него приходится проверять memtable + все SSTable файлы. Bloom filter с высокой вероятностью говорит, что ключа точно нет

Зачем нужен WAL в B-Tree?::Crash при split страницы может создать "осиротевшую" страницу и повредить индекс. WAL записывает изменения перед модификацией B-Tree

Schema-on-write аналогична ==статической типизации==, schema-on-read аналогична ==динамической типизации==

B-Tree страница имеет фиксированный размер ==4 КБ==

Когда использовать документную модель?::Данные древовидные (one-to-many), загружаются целиком, редко требуют связей many-to-many

Когда использовать реляционную модель?::Данные сильно связаны (highly interconnected), нужны JOIN и строгие транзакции

LSM-Trees vs B-Trees — когда что?
?
- **LSM-Trees** — write-heavy, выше пропускная способность записи, лучше сжатие
- **B-Trees** — read-heavy, предсказуемые задержки, транзакционные блокировки диапазонов

OLTP vs OLAP
?
- **OLTP** — B-Tree, row-oriented, точечные запросы по ключу
- **OLAP** — column-oriented, сканирование миллионов строк, bitmap index, векторные вычисления

Что такое Shredding (антипаттерн)?::Разбиение простого самодостаточного документа (например, резюме) на десятки реляционных таблиц → неуклюжие схемы и переусложнённый код

Hash index — ограничения
?
- Все ключи должны помещаться в RAM
- Неэффективен для range queries (ключи не отсортированы)

Как работает Column-oriented Bitmap Index запрос?
?
1. Загружаются bitmap для искомых значений
2. Побитовое ИЛИ (bitwise OR) над bitmap
3. Аппаратно выполняется крайне быстро
