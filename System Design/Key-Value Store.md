## Суть и контекст

Key-Value Store — не-реляционная база данных, где каждый уникальный идентификатор (ключ) хранится вместе с ассоциированным значением. Проектирование распределённого Key-Value Store требует выбора между согласованностью и доступностью (CAP theorem), реализации партиционирования (consistent hashing), репликации, обнаружения сбоев (gossip protocol) и механизмов восстановления (sloppy quorum, hinted handoff, Merkle tree).

---

## Ключевые идеи

- Ключи — обычный текст или хэш; значения — непрозрачные (opaque) объекты (строки, списки, объекты). Размер пары < 10 KB
- CAP theorem: сетевые разделения (P) неизбежны → выбор между CP и AP. CA-систем в реальном мире не существует
- Consistent hashing для партиционирования: ключ назначается первому серверу по часовой стрелке на hash ring
- Репликация: координатор реплицирует данные на N следующих узлов по часовой стрелке
- Quorum consensus: W + R > N гарантирует строгую согласованность (всегда есть перекрывающийся узел со свежими данными)
- Gossip protocol для децентрализованного обнаружения сбоев
- Sloppy quorum + hinted handoff для временных сбоев; Merkle tree для постоянных сбоев
- Vector Clocks для разрешения конфликтов при параллельном редактировании

## Определения и термины

**Key-Value Store (Key-Value Database)** — не-реляционная база данных, где каждый уникальный идентификатор сохраняется как ключ (key) вместе с ассоциированным значением (value). Ключи могут быть обычным текстом или хэшем. Значения рассматриваются как непрозрачные (opaque) объекты.

**CAP Theorem** — теорема, утверждающая, что распределённая система не может одновременно обеспечивать более двух из трёх гарантий: Consistency (согласованность), Availability (доступность) и Partition Tolerance (устойчивость к разделению сети). Поскольку сетевые разделения (P) неизбежны, система должна быть либо CP, либо AP.

**Gossip Protocol** — децентрализованный (peer-to-peer) протокол, используемый узлами для распространения информации об изменениях состояния кластера (membership and failure detection) без использования внешнего сервиса координации.

**Sloppy Quorum** — техника обработки временных сбоев: если нужные узлы недоступны, система временно записывает данные на другие случайные узлы.

**Hinted Handoff** — при возвращении оригинального узла в строй временные узлы передают ему накопленные данные.

**Anti-entropy / Merkle Tree** — структура данных (дерево хэшей), используемая для синхронизации и обнаружения неконсистентности данных между репликами в фоновом режиме для обработки постоянных сбоев (permanent failures).

**Vector Clocks** — механизм разрешения конфликтов при параллельном редактировании: D([Sx, version]). Позволяет определить связи "предок-потомок" между версиями данных.

**Quorum Consensus** — механизм настраиваемой согласованности. N — число реплик, W — размер кворума записи, R — размер кворума чтения. Если W + R > N — гарантируется строгая согласованность.

## Как устроено

### API

```
put(key, value)  — сохраняет пару ключ-значение
get(key)         — возвращает значение по ключу
```

### Consistent Hashing (партиционирование)

1. Вычисляется хэш серверов — размещаются на кольце (hash ring)
2. Вычисляется хэш ключа
3. Ключ назначается первому серверу, найденному при движении по кольцу по часовой стрелке

### Репликация

1. Данные записываются на первый узел (coordinator) по хэш-кольцу
2. Координатор реплицирует данные на N следующих узлов по часовой стрелке

### Read Path

1. Запрос на чтение поступает на узел
2. Узел проверяет Bloom filter, чтобы выяснить, в каких SSTable может содержаться ключ
3. SSTables возвращают искомый набор данных
4. Данные возвращаются клиенту

### Quorum Consensus

```
W + R > N → strong consistency

N = число реплик
W = узлы, подтверждающие запись
R = узлы, подтверждающие чтение

Всегда есть минимум один перекрывающийся узел со свежими данными
```

### Обработка сбоев

| Тип сбоя | Механизм | Как работает |
|----------|----------|-------------|
| Временный | Sloppy Quorum + Hinted Handoff | Запись на случайные узлы → передача данных при восстановлении |
| Постоянный | Anti-entropy + Merkle Tree | Фоновая синхронизация через дерево хэшей |

---

## Детали и нюансы

- **CAP в реальности**: CA-систем не существует, т.к. сетевые разделения (P) неизбежны. Выбор всегда между CP (жертва доступностью при сбое сети) и AP (жертва согласованностью, принимает запись всегда)
- **Vector Clocks — ограничение**: если длина вектора превышает лимит, старые пары [server: version] удаляются → невозможно точно определить связи "предок-потомок", что приводит к неэффективному разрешению конфликтов
- **Data center outage**: полный отказ дата-центра (потеря питания/сети) → требуется межрегиональная репликация (cross-data center replication)

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|-------------------|
| CP-система | Гарантирует согласованность данных | Отключает запись при сбоях сети | Банковские системы, критичные данные |
| AP-система | Принимает запись всегда, высокая доступность | Жертвует согласованностью (eventual) | Социальные сети, кэши, DNS |

## Примеры

### Архитектура Key-Value Store

```
Клиент → read/write → Coordinator (n6)
                           ├── Реплика n1
                           └── Реплика n2
                       ← response ← Coordinator
```

Узлы расположены на hash ring (n0...n7). Coordinator принимает запрос, реплицирует на N следующих узлов и возвращает ответ клиенту.

## Важно / подводные камни / best practices

- **Vector Clocks** для разрешения конфликтов при параллельной записи
- **Sloppy Quorum + Hinted Handoff** — обеспечивают высокую доступность записи при временных сбоях
- **Merkle Tree** — фоновая синхронизация для обнаружения неконсистентности при постоянных сбоях
- **Gossip Protocol** — децентрализованное обнаружение сбоев без единой точки отказа
- **Cross-data center replication** — защита от полного отказа дата-центра
