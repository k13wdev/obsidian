## Суть и контекст

Кэширование — одна из самых частых тем на System Design собеседовании. Кэш — временная область хранения в оперативной памяти, которая сохраняет результаты дорогостоящих ответов или часто запрашиваемые данные, чтобы последующие запросы обслуживались быстрее. Данные хранятся в энергозависимой памяти (volatile memory) — при перезапуске сервера они теряются.

---

## Ключевые идеи

- Кэш ускоряет чтение за счёт хранения данных в оперативной памяти
- Существует несколько уровней кэширования: клиентский, CDN, L1 (локальный), L2 (распределённый), database-level, hardware-level
- Основные стратегии: cache-aside, read-through, write-through, write-back, refresh-ahead
- Инвалидация — ключевая проблема: TTL, явная инвалидация, event-based
- Кэш **нельзя** использовать как единственный источник истины (Source of Truth)
- Связка L1 + L2 даёт оптимальный баланс между задержкой и разделяемым состоянием
- Проблемы: cache stampede, hot key, cache penetration, cache avalanche, SPOF кэша

---

## Определения и термины

**Кэш (Cache)** — временная область хранения (temporary storage area), которая сохраняет результаты дорогостоящих ответов или часто запрашиваемые данные в оперативной памяти (in-memory), чтобы последующие запросы обслуживались быстрее.

**L1 — локальный / application-level кэш** — кэш в памяти самого приложения (инстанса). Очень быстрый, нет сетевого перехода, но данные пропадают при рестарте и сложна инвалидация.

**L2 — распределённый кэш** — внешний общий кэш для всех инстансов (например, Redis). Проще делать единый слой кэширования, но добавляется сеть и сериализация.

**CDN** — сеть географически распределённых серверов для кэширования статического контента на периферии ближе к клиенту.

**Клиентский кэш** — кэширование результатов в браузере (заголовок `cache-control: private, max-age=3600`).

**Cache-aside (Lazy loading)** — паттерн, при котором сервис сначала читает из кэша. Если промах — идёт в БД, затем кладёт результат в кэш и отвечает клиенту. Хорошо подходит для систем с преобладанием чтения, так как кэшируется только то, что реально запрашивается.

**Read-through** — приложение читает только из кэша, а кэш сам обращается к источнику (БД) в случае промаха. Скрывает логику загрузки за кэшем, но при падении кэша приложение теряет доступ к данным.

**Write-through** — сначала запись идёт в БД, потом синхронно обновляется кэш. Чтение сразу видит актуальные данные, но замедляет путь записи.

**Write-back (Write-behind)** — сначала запись идёт в кэш, а в БД сбрасывается асинхронно. Подходит для write-heavy, но несёт риск потери данных при падении кэша.

**Refresh-ahead** — данные обновляются в кэше заранее по расписанию или по предсказанию спроса. Работает для очень популярных и стабильных объектов.

**Redis** — in-memory хранилище (key-value), используемое как распределённый кэш (L2) и для счётчиков (rate limiting). Предлагает команды `INCR` и `EXPIRE` для контроля времени жизни ключей.

**Инвалидация кэша:**
- **TTL (Time-To-Live)** — вытеснение данных по истечении времени жизни
- **Явная инвалидация (Explicit)** — удаление данных из кэша программно
- **По событиям (Event-based)** — обновление кэша привязано к событию из пути записи

## Как устроено

### Алгоритм работы Cache-aside

1. Клиент отправляет запрос к API-сервису
2. Сервис проверяет наличие данных в Redis Cache
3. Если данные есть (hit) — немедленно возвращает клиенту
4. Если данных нет (miss) — запрашивает из PostgreSQL
5. Получив ответ от БД, записывает (fill) данные в Redis Cache
6. Возвращает ответ клиенту

### Алгоритмы вытеснения (Eviction Policies)

| Алгоритм | Описание | Когда использовать |
|----------|----------|-------------------|
| LRU (Least Recently Used) | Удаляет данные, к которым давно не обращались | Универсальный, наиболее популярный |
| LFU (Least Frequently Used) | Удаляет то, что редко используется | При наличии устойчивых "горячих" ключей |
| FIFO (First In First Out) | Удаляет в порядке добавления | Простые сценарии |
| Clock / Second Chance / 2Q / SLRU | Оптимизированные алгоритмы | Избегание случайного вытеснения полезных данных |

## Детали и нюансы

### Cache Stampede / Thundering herd

Множество клиентов одновременно получают промах в кэше и одновременно идут в БД, вызывая её перегрузку. Решения:
- Блокировка по ключу (singleflight / per-key lock)
- Раннее обновление до истечения TTL
- Soft TTL + фоновое обновление
- Jitter на TTL

### Горячий ключ (Hot key)

Сверхчастое обращение к одному ключу (например, профиль знаменитости), из-за чего перегружается один сервер кэша. Решения:
- Отдельный TTL для ключа
- Локальный кэш L1 поверх L2
- Шардирование горячего ключа
- Объединение запросов

### Cache Penetration

Клиенты/злоумышленники массово запрашивают несуществующие ключи. Приложение всегда получает промах и бьёт БД. Решение: кэширование ответа "not found" с коротким TTL. Увеличивает расход памяти на мусорные ключи, но защищает базу.

### Cache Avalanche (при рестарте)

После рестарта кэш пуст. Если направить 100% трафика, БД рухнет от лавины промахов. Решения:
- Постепенный раскат трафика
- Blue/green с прогревом кэша
- Предварительная загрузка (preload) "горячих" объектов

### SPOF кэша

Отказ единственного сервера кэша = падение всей системы, если БД не выдержит нагрузку напрямую. Решение: кластер серверов кэша, разнесённых по дата-центрам, избыточное выделение памяти (overprovisioning).

### Коллизии в реплицированном кэше (Active-Active)

При высокой частоте обновления один инстанс обновляет данные, но из-за задержки репликации (replication lag) они затираются устаревшими данными от другого инстанса. При >100 МБ данных и частых апдейтах реплицированный кэш порождает неконсистентность.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Cache-aside | Кэшируется только запрашиваемое, приложение контролирует логику | Приложение усложняется кодом кэширования | Read-heavy, по умолчанию |
| Read-through | Скрывает логику за абстракцией | При падении кэша — нет доступа к данным | Когда хочется инкапсуляции |
| Write-through | Чтение сразу видит актуальные данные | Замедляет путь записи | Когда важна консистентность кэша |
| Write-back | Быстрая запись | Риск потери данных при падении кэша | Write-heavy сценарии |
| Refresh-ahead | Нет промахов для популярных объектов | Сложность предсказания спроса | Стабильные "горячие" объекты |
| L1 (локальный) | Сверхнизкая задержка, нет сети | Данные пропадают при рестарте, сложна инвалидация | Очень популярные объекты |
| L2 (распределённый) | Единый слой, проще инвалидация | Сеть + сериализация | Разделяемое состояние |
| Реплицированный кэш | Нет SPOF, сверхбыстрый | Лимит RAM (<100 МБ), неконсистентность при частых записях | Маленький read-only датасет |
| Распределённый кэш | Большой объём, высокая консистентность | Медленнее (сеть), SPOF без кластера | Большие данные, write-heavy |
| TTL длинный | Меньше нагрузки на БД | Stale данные | Редко меняющиеся данные |
| TTL короткий | Актуальные данные | Перегрузка источника | Часто меняющиеся данные |

## Примеры

### Базовая работа с кэшем (Redis/Memcached)

```
SETEX myKey 3600 "hi there"   # set с TTL 1 час
GET myKey                      # => "hi there"
```

### Rate Limiting через Redis

```
INCR user:123:requests    # увеличить счётчик на 1
EXPIRE user:123:requests 60  # TTL 60 секунд
```

### Схема Cache-aside

```
Клиент → API Сервис → Redis Cache
                         ├── hit → Ответ клиенту
                         └── miss → PostgreSQL
                                      ↓
                               API Сервис
                                  ├── fill → Redis Cache
                                  └── Ответ клиенту
```

### Защита от Cache Stampede (singleflight)

```
Множество запросов → API Сервис → Per-Key Lock / Singleflight
                                        ↓ (один запрос)
                                       DB
                                        ↓
                                   Cache fill
                                        ↓
                                 Fan-out responses → все ожидающие клиенты
```

## Важно / подводные камни / best practices

- **Кэш не Source of Truth** — нельзя использовать как единственный источник данных; он в энергозависимой памяти и не поддерживает ACID
- **Не кэшировать данные с требованием строгой консистентности** (например, баланс банковского счёта)
- **Связка L1 + L2** — best practice для баланса задержки и разделяемого состояния
- **Система должна выжить без кэша** — если при промахах всё падает, архитектура неверна
- **Singleflight / per-key lock** — обязательная защита от cache stampede
- **Кэширование "not found"** — защита от cache penetration (атаки несуществующими ключами)
- **Прогрев кэша при рестарте** — постепенный раскат или preload горячих объектов
- **Near-cache без синхронизации** — антипаттерн: разные ноды отдают рассинхронизированные данные
- **TTL баланс** — слишком длинный → stale, слишком короткий → перегрузка источника

?

- **FIFO** — в порядке добавления

Три способа инвалидации кэша
?
1. **TTL (Time-To-Live)** — вытеснение по истечении времени
2. **Explicit** — программное удаление
3. **Event-based** — обновление по событию из пути записи

Реплицированный vs Распределённый кэш
?
- **Реплицированный** — каждая нода содержит 100% данных, нет SPOF, сверхбыстрый, но лимит RAM (<100 МБ)
- **Распределённый** — выделенный сервер (Redis), медленнее (сеть), но масштабируется и даёт консистентность

Redis INCR + EXPIRE используется для ==Rate Limiting== — хранение счётчиков запросов с автоматическим сбросом

LRU расшифровывается как ==Least Recently Used== — удаляет данные, к которым ==давно не обращались==
