## Суть и контекст

Масштабирование — способность системы справляться с возрастающей нагрузкой за счёт добавления ресурсов. Существует два подхода: вертикальное (увеличение мощности одного сервера) и горизонтальное (добавление серверов в пул). Горизонтальное масштабирование требует балансировки нагрузки, stateless-архитектуры, CDN для статики и шардирования БД.

---

## Ключевые идеи

- Вертикальное масштабирование (scale up) просто, но упирается в физические лимиты и не даёт failover
- Горизонтальное масштабирование (scale out) дешевле на масштабе, но требует балансировки и управления состоянием
- Балансировщик нагрузки решает задачу отказоустойчивости и распределения трафика
- CDN разгружает origin-сервер, отдавая статику из географически ближайшего узла
- Consistent hashing минимизирует перераспределение данных при изменении числа серверов
- Stateless web tier — обязательное условие для горизонтального масштабирования веб-серверов
- Шардирование БД масштабирует данные горизонтально, но разрушает транзакции и усложняет JOIN

## Определения и термины

**Vertical scaling (Scale up)** — процесс масштабирования за счёт добавления большей вычислительной мощности (CPU, RAM, DISK) к существующему серверу. Отличный вариант при низком трафике благодаря простоте, но упирается в аппаратные лимиты сервера.

**Horizontal scaling (Scale out)** — практика добавления дополнительных серверов в пул ресурсов для распределения нагрузки. Требует stateless-архитектуры или балансировщика нагрузки. В контексте баз данных называется Shared-nothing architecture или шардированием.

**Балансировщик нагрузки (Load balancer)** — устройство или программа, которая равномерно распределяет входящий трафик между веб-серверами, определёнными в наборе балансировки. Пользователи подключаются к публичному IP балансировщика, а он общается с серверами через приватные IP.

**Content Delivery Network (CDN)** — сеть географически распределённых серверов, используемая для доставки статического контента (изображения, видео, CSS, JavaScript). Динамическое кэширование CDN также возможно для HTML-страниц на основе путей, query-строк, cookie и заголовков.

**Consistent hashing (Согласованное хэширование)** — особый вид хэширования, при котором в случае изменения размера хэш-таблицы в среднем перераспределяется только k/n ключей, где k — количество ключей, а n — количество слотов. Устраняет проблему классического `hash % N`.

**Virtual nodes (Виртуальные узлы)** — множественные точки на хэш-кольце, представляющие один физический сервер. Количество виртуальных узлов пропорционально ёмкости сервера, решает проблему неравномерного распределения данных.

**Stateless web tier** — архитектура, при которой данные о состоянии (например, сессии) вынесены из веб-серверов в постоянное хранилище (RDBMS или NoSQL). Любой веб-сервер в кластере может получить доступ к данным состояния из внешней БД.

**Stateful architecture** — архитектура, при которой компоненты хранят контекст между запросами. Требует sticky-маршрутизации. Применяется для шлюзов WebSocket, статусов присутствия пользователей, движков подбора в памяти.

## Как устроено

### Балансировка нагрузки — уровни и алгоритмы

**L4 (Transport)** — работает на уровне TCP/UDP, быстрее, не видит содержимое HTTP.

**L7 (Application)** — видит URL, заголовки, cookie, host; выполняет маршрутизацию по контенту, A/B-тестирование, аутентификацию.

#### Алгоритмы распределения

| Алгоритм | Описание | Когда использовать |
|----------|----------|-------------------|
| Round Robin | Распределение по кругу | Однотипные машины |
| Weighted Round Robin | С учётом разной мощности серверов | Гетерогенный кластер |
| Least Connections | На сервер с наименьшим числом активных соединений | Долгие соединения |
| По хэшу / sticky | Маршрутизация по ключу | Stateful-сессии |

---

### Как работает CDN

1. Пользователь запрашивает файл (например, `image.png`) по URL с доменом CDN
2. Если CDN-сервер не содержит файл в кэше, он запрашивает его с origin-сервера (веб-сервер или Amazon S3)
3. Origin возвращает файл CDN-серверу вместе с HTTP-заголовком **Time-to-Live (TTL)**
4. CDN кэширует файл до истечения TTL и отдаёт пользователю
5. При повторном запросе файл отдаётся из кэша CDN, если TTL не истёк

Инвалидация кэша CDN через версионирование:
```
https://cdn.example.com/image.png?v=2
```
Добавление параметра с номером версии в query string позволяет принудительно обновить кэш CDN до истечения TTL.

---

### Consistent hashing — алгоритм

1. Серверы отображаются на кольцевом хэш-пространстве с помощью равномерно распределённой хэш-функции (по IP или имени)
2. Ключи данных отображаются на то же кольцо
3. Для поиска сервера система двигается от позиции ключа **по часовой стрелке** до первого найденного сервера
4. При **добавлении** нового сервера перераспределяются только ключи, находящиеся между новым сервером и предыдущим против часовой стрелки
5. При **удалении** сервера перераспределяются только ключи, принадлежавшие ему; они переходят к следующему серверу по часовой стрелке

**Проблема без виртуальных узлов**: размер партиций между соседними серверами становится неравномерным — один сервер может получить гигантскую долю хэш-пространства.

**Решение — виртуальные узлы**: каждый физический сервер представлен множеством точек на кольце (s0_0, s0_1, s1_0, s1_1...), что дробит хэш-пространство на более мелкие участки для равномерного распределения.

---

### Database scaling — Read Replicas (Master-Slave)

1. Создаётся архитектура Master-Slave
2. Все операции изменения данных (write, update, delete) → Master DB
3. Master DB **асинхронно** реплицирует данные на Slave DB
4. Все операции чтения → Slave DB

---

### Database scaling — Sharding

1. Крупная база данных разбивается на мелкие части (шарды)
2. Каждый шард использует одинаковую схему данных, но содержит **уникальный** набор записей
3. Выбирается **Sharding Key (Partition Key)**, определяющий распределение данных
4. Запросы маршрутизируются в нужный шард на основе Sharding Key

## Детали и нюансы

- Максимальный сервер в Amazon RDS ограничен **24 TB RAM** — это потолок вертикального масштабирования
- Сервер вдвое мощнее стоит **сильно дороже** двух серверов — стоимость растёт быстрее, чем линейно
- Переход на горизонтальное масштабирование веб-серверов **жёстко требует** выноса пользовательских сессий в общую базу (Stateless Web Tier). Иначе каждый запрос должен балансироваться на конкретный сервер (sticky routing), что снижает отказоустойчивость
- Если один сервер умирает (private IP недоступен), балансировщик прекращает отправлять на него трафик
- При неравномерном распределении данных один шард может переполниться из-за **"горячих ключей"** (например, данные звёзд с миллионами подписчиков: Katy Perry, Justin Bieber)
- Шардирование **разрушает транзакции**, усложняет выполнение JOIN между шардами и глобальные индексы. Требуется перехэширование (resharding) с изменением sharding function и перемещением данных
- Стоимость CDN: при тарифе Amazon CloudFront $0.02 за гигабайт, стриминг длинных/тяжёлых видео обходится крайне дорого

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Vertical scaling (Scale up) | Простота, один сервер | Лимит CPU/RAM, нет failover, стоимость растёт нелинейно | Низкий трафик, ранние стадии |
| Horizontal scaling (Scale out) | Дешевле на масштабе, failover | Сложность балансировки, управление состоянием, согласованность | Высокий трафик, production |
| Stateless web tier | Любой сервер обслужит запрос, простое масштабирование | Требует внешнего хранилища сессий | По умолчанию для горизонтального масштаба |
| Stateful architecture | Быстрый доступ к контексту в памяти | Sticky routing, сложнее failover | WebSocket, in-memory matching |
| L4 балансировка | Быстрее, низкий overhead | Не видит HTTP | Высокая пропускная способность |
| L7 балансировка | Маршрутизация по контенту, A/B тесты | Медленнее, больше ресурсов | Сложная логика маршрутизации |
| CDN TTL длинный | Меньше нагрузки на origin | Stale контент | Редко меняющаяся статика |
| CDN TTL короткий | Актуальный контент | Частые перезагрузки с origin | Динамичный контент |
| hash % N | Простота | Массовая инвалидация при изменении N | Не использовать в распределённых системах |
| Consistent hashing | Минимальное перераспределение | Сложнее реализация | Распределённые кэши, шардинг |

## Примеры

### Антипаттерн: hash % N

```
server_index = hash(key) % N
```

При изменении N из-за добавления/удаления сервера модульная арифметика возвращает совершенно другие индексы, что вызывает массовую инвалидацию кэша (rehashing problem).

### Схема масштабируемой системы

```
Клиенты (Web, Mobile)
    |
    ├── DNS → CDN (статика: JS, CSS, изображения)
    |
    └── Load Balancer (публичный IP)
            |
        ┌───┴───┐
    Server 1  Server 2  (Stateless Web Tier)
        |         |
        └────┬────┘
             |
          Cache
             |
    ┌────────┴────────┐
  Master DB ──replicate──> Slave DB
  (write)                  (read)
```

## Важно / подводные камни / best practices

- **Stateless web tier** — обязательное условие для горизонтального масштабирования
- **Дублирование на каждом уровне (redundancy)** — устранение Single Point of Failure
- **Максимальное кэширование данных**, хостинг статики в CDN
- **hash % N** — антипаттерн для распределённых систем; использовать consistent hashing
- **Fallback для CDN** — клиенты должны уметь определять сбой CDN и запрашивать ресурсы напрямую с origin-сервера
- **Горячие ключи при шардировании** — один шард может переполниться; нужно учитывать распределение данных при выборе sharding key
- Шардирование — последнее средство; оно разрушает транзакции, усложняет JOIN и требует resharding
