## Суть и контекст

Выбор архитектурного стиля определяет структуру системы, способы масштабирования, развёртывания и взаимодействия компонентов. Основные стили: монолит, модульный монолит, микроядро, микросервисы и архитектура на основе пространства. Выбор зависит от зрелости команды, масштаба нагрузки и требований к независимости развёртывания.

---

## Ключевые идеи

- Монолит — простейший старт: один модуль, одна БД, простые транзакции
- Модульный монолит — промежуточный шаг: единица развёртывания, но разделение по доменам (bounded contexts)
- Микросервисы — независимые сервисы с собственными БД; требуют eventual consistency, Saga вместо ACID
- Микроядро — ядро + плагины; расширяемость без изменения ядра
- Архитектура на основе пространства — устраняет БД как bottleneck через in-memory data grids
- Паттерн "Три слоя": Handler (транспорт) → Service (бизнес-логика) → Repository (БД)
- REST для внешних API, gRPC для внутренних межсервисных вызовов
- Закон Конвея: архитектура отражает структуру команд

---

## Определения и термины

**Монолитная архитектура** — единый развёртываемый модуль, подключённый к единой базе данных, включающий все функциональные возможности системы. Проще разработка, меньше сетевых переходов, легче локальная отладка, проще транзакции.

**Модульный монолит** — единица развёртывания, связанная с единой БД и разделённая по задачам предметной области (ограниченным контекстам), а не по техническим возможностям. Внешне прост как монолит, внутри — границы ответственности.

**Архитектура микросервисов** — распределённая архитектура из отдельно развёртываемых узкоспециализированных сервисов, в каждом из которых смоделирована предметная область (ограниченный контекст). Каждый сервис включает собственные БД и классы.

**Микроядерная архитектура (Microkernel / Plug-in)** — монолитная архитектура из основной системы (ядра) и независимых подключаемых компонентов (плагинов). Расширяемость и настраиваемая логика. Плагины связываются с ядром "точка-точка" или через REST.

**Архитектура на основе пространства (Space-based)** — распределённая архитектура, устраняющая центральную БД как узкое место за счёт реплицированных сеток данных в оперативной памяти (in-memory data grids). Данные в RAM, обновления асинхронно в БД.

**gRPC** — среда удалённого вызова процедур, использующая бинарный формат и строгую схему (контракт).

**REST** — архитектурный стиль API, использующий HTTP-методы и ресурсную модель (существительные в URL).

## Как устроено

### Паттерн "Три слоя" в микросервисах

1. **Handler** — изолирует транспорт (HTTP/gRPC). Декодирование запроса, кодирование ответа
2. **Service** — бизнес-логика. Не зависит от транспорта
3. **Repository** — изолирует БД (PostgreSQL/MongoDB). Инкапсулирует SQL-запросы

### Микроядерная архитектура — процесс

1. Ядро принимает бизнес-запрос
2. Определяет необходимый плагин по реестру или контракту
3. Передаёт данные в плагин через метод/REST
4. Плагин выполняет специфичные правила и возвращает результат

### Space-based архитектура — компоненты

- **Блок обработки (processing unit)** — содержит in-memory кэш
- **Сетка сообщений (messaging grid)** — распределяет запросы
- **Сетка данных (data grid)** — репликация данных между блоками
- **Средство переноса данных (data pump)** + **data writer** — асинхронная запись в БД

Процесс:
1. Messaging grid направляет запрос на доступный блок обработки
2. Блок читает/обновляет in-memory кэш
3. Репликация синхронизирует кэши других блоков
4. Data pump асинхронно обновляет физическую БД

### Роли СУБД в микросервисной архитектуре

| Технология | Роль | Детали |
|-----------|------|--------|
| PostgreSQL | Primary DB (транзакционное ядро) | Write path, ACID, связи, строгая схема |
| Redis | Распределённый кэш (L2) | Сессии, счётчики rate limiting, горячие lookup |
| Elasticsearch/Kibana | Полнотекстовый поиск + визуализация логов | Индекс обновляется асинхронно из PostgreSQL через Kafka |

## Детали и нюансы

- **ACID транзакции через границы микросервисов** — антипаттерн: теряется независимость развёртывания, падение зависимого сервиса прерывает транзакцию. Использовать BASE, eventual consistency или **Saga**
- **Двойная запись** (запись в БД + событие в Kafka): если связь рвётся, возникает рассинхронизация. Решение: паттерн **Outbox** — данные и событие в одной транзакции PostgreSQL, relay-процесс публикует в брокер
- **Replica lag PostgreSQL**: клиент обновляет данные, делает redirect и видит старые данные с реплики. Решение: **закрепление чтения после записи** — маршрутизация на Primary DB короткое время после записи
- **Over-fetching в REST**: возврат 30+ полей, когда нужно 4. Решение: CQRS-lite (отдельная Read Model) или GraphQL
- **Закон Конвея (Inverse Conway Maneuver)**: монолит отражает структуру отделов (DBA, backend, frontend). Микросервисы требуют объединения всех слоёв для одной бизнес-команды

## Сравнения и trade-offs

| Стиль | Плюсы | Минусы | Когда использовать |
|-------|-------|--------|--------------------|
| Монолит | Простота, быстрый старт, простые транзакции | Масштабируется целиком, тесная связанность | Старт, маленькая команда |
| Модульный монолит | Границы доменов, простота развёртывания | Единая БД, зависимость модулей при деплое | Промежуточный шаг перед микросервисами |
| Микросервисы | Независимое развёртывание, масштабирование по частям | Сетевая задержка, partial failures, distributed tracing | Разное давление роста, независимые команды |
| Микроядро | Расширяемость плагинами, стабильное ядро | Связь ядра с плагинами, ограниченная масштабируемость | Продукты с настраиваемой логикой |
| Space-based | Устраняет БД bottleneck, горизонтальное масштабирование | Сложность, eventual consistency, стоимость RAM | Экстремальные пиковые нагрузки |

| Протокол | Плюсы | Минусы | Когда использовать |
|----------|-------|--------|--------------------|
| REST | Гибкость JSON, простота маршрутизации | Over-fetching, текстовый формат | Внешние клиентские API |
| gRPC | Типизация, бинарный формат, низкий оверхед | Сложнее отладка, нужен proto-файл | Внутренние межсервисные вызовы |
| GraphQL | Гибкие ответы, нет over-fetching | Сложность, кэширование | UI с разными представлениями |
| WebSocket | Realtime, bidirectional | Stateful, сложнее масштабирование | Чат, push-уведомления |

## Примеры

### Handler (Go) — изоляция транспорта

```go
func (h *Handler) Create(w http.ResponseWriter, r *http.Request) {
    var req CreateRequest
    json.NewDecoder(r.Body).Decode(&req)
    result, err := h.service.Create(req)
    json.NewEncoder(w).Encode(result)
}
```

Handler декодирует HTTP JSON и делегирует бизнес-логику в Service. Можно заменить транспорт на gRPC.

### Паттерн Outbox

```
DB Transaction:
  ├── INSERT INTO orders (business row)
  └── INSERT INTO outbox (event row)
  → COMMIT

Outbox Relay → Kafka Topic
```

## Важно / подводные камни / best practices

- **Не делать всё микросервисами с самого начала** — стартовать с модульного монолита, выделять сервисы при доказанной необходимости
- **REST для внешних API, gRPC для внутренних** — best practice по умолчанию
- **Паттерн Outbox** — решение двойной записи (БД + Kafka)
- **Закрепление чтения после записи** — маршрутизация на Primary DB после обновления
- **ACID через границы микросервисов** — антипаттерн; если нужны транзакции — сервисы слишком мелкие, объединить
- **Идемпотентность на стороне consumer** — Kafka даёт at-least-once, нужен ключ идемпотентности

3. **Repository** — изоляция БД

Что такое паттерн Outbox?::Данные и событие пишутся в одной транзакции БД (business row + outbox row). Relay-процесс асинхронно публикует из outbox в Kafka. Решает проблему двойной записи

Почему не стоит стартовать с микросервисов?::Усложняет систему: сетевая задержка, partial failures, distributed tracing. Стартовать с модульного монолита, выделять микросервисы при разном давлении роста или независимых релизах

Роли PostgreSQL, Redis, Elasticsearch в микросервисах
?
- **PostgreSQL** — Primary DB, write path, ACID
- **Redis** — кэш L2, сессии, счётчики rate limiting
- **Elasticsearch** — полнотекстовый поиск, визуализация логов (обновляется через Kafka)

Что такое Inverse Conway Maneuver?::Монолит отражает структуру отделов. Микросервисы требуют объединения всех слоёв для одной бизнес-команды — архитектура диктует оргструктуру

REST vs gRPC — когда что?
?
- **REST** — внешние клиентские API (гибкость JSON, простота)
- **gRPC** — внутренние вызовы (типизация, бинарный формат, низкий оверхед)

Kafka в микросервисах даёт ==at-least-once== доставку, поэтому consumer должен обеспечивать ==идемпотентность==
