## Суть и контекст

Очереди сообщений (Message Queues) — ключевой компонент распределённых систем для асинхронного взаимодействия между сервисами. Они декаплят producer и consumer, сглаживают пиковые нагрузки и позволяют обрабатывать задачи в фоне. Два основных подхода: task queue (RabbitMQ) и event stream (Apache Kafka).

---

## Ключевые идеи

- Очередь сообщений — буфер между producer и consumer для асинхронного взаимодействия
- Декаплинг: producer и consumer масштабируются независимо
- Асинхронная обработка радикально снижает задержку для пользователя (с 3100 мс до 25 мс)
- Apache Kafka — распределённый журнал событий с партициями, consumer groups и offset
- Kafka vs RabbitMQ: event stream (replay, несколько подписчиков) vs task queue (dispatch, DLQ)
- Гарантия доставки at-least-once требует идемпотентности на consumer
- Порядок сообщений гарантируется только внутри партиции Kafka

---

## Определения и термины

**Очередь сообщений (Message Queue)** — хранящийся в памяти надёжный компонент (durable component), поддерживающий асинхронное взаимодействие, служащий буфером и распределяющий асинхронные запросы.

**Apache Kafka** — распределённый журнал событий (distributed event log).

**Topic (Kafka)** — логическая категория событий.

**Partition (Kafka)** — физическое деление топика. Порядок сообщений гарантируется только внутри партиции.

**Consumer Group (Kafka)** — набор потребителей, совместно читающих поток.

**Offset (Kafka)** — позиция сообщения внутри партиции.

---

## Как устроено

### Базовая модель: Producer → Queue → Consumer

1. Input-сервисы (Producers) создают сообщения и публикуют в очередь
2. Очередь сохраняет сообщения как буфер
3. Consumers подключаются и асинхронно выполняют действия из сообщений

### Конвейер событий в Kafka

```
Producer Сервис → Kafka Topic
                      ├── Consumer Group 1: Поиск Indexer
                      ├── Consumer Group 2: Аналитика
                      └── Consumer Group 3: Notifications
```

Один поток событий читается несколькими независимыми подсистемами.

### Синхронная vs асинхронная обработка

Синхронный вызов долгого процесса (фото, платёж) → тайм-ауты, блокировка ресурсов.
Решение: producer кладёт задачу в очередь, отвечает клиенту `202 Accepted`. Consumer обрабатывает в фоне.

---

## Детали и нюансы

- **At-least-once доставка**: Kafka/RabbitMQ гарантируют минимум одну доставку. Для защиты от дублирования — **ключ идемпотентности** на стороне consumer
- **Порядок в Kafka**: гарантируется **только внутри партиции**. Для глобального порядка — одна партиция (но теряется параллелизм)
- **Двойная запись** (БД + Kafka): решается паттерном **Outbox** — данные и событие в одной транзакции PostgreSQL, relay публикует в брокер
- **Backpressure**: очередь сглаживает всплески нагрузки; consumers обрабатывают в своём темпе
- **Dead Letter Queue (DLQ)**: сообщения, которые consumer не смог обработать после нескольких попыток (RabbitMQ)

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Синхронное (HTTP/gRPC) | Простота, немедленный ответ | Задержка = сумма вызовов, каскадные сбои | Короткие операции, нужен ответ |
| Асинхронное (Queue) | Декаплинг, backpressure, быстрый ответ клиенту | Eventual consistency, дедупликация | Долгие операции, пиковые нагрузки |
| Kafka (Event stream) | Replay, несколько consumer groups, высокая пропускная способность | Сложнее настройка, eventual consistency | Стриминг, несколько подписчиков, аналитика |
| RabbitMQ (Task queue) | Гибкая маршрутизация, DLQ, удаление после обработки | Нет replay, один consumer на сообщение | Dispatch задач воркерам |

## Примеры

### Снижение задержки через асинхронность

Синхронно: Клиент → Сервис A → Сервис B → Сервис C = 3100 мс
Асинхронно: Клиент → Сервис A → Queue (202 Accepted) = 25 мс. Сервисы B, C обрабатывают в фоне.

### Паттерн Outbox для Kafka

```
BEGIN TRANSACTION;
  INSERT INTO orders (...);        -- business row
  INSERT INTO outbox (event, ...); -- event row
COMMIT;

-- Outbox Relay (отдельный процесс):
-- SELECT * FROM outbox WHERE published = false;
-- → Publish to Kafka Topic
-- → UPDATE outbox SET published = true;
```

## Важно / подводные камни / best practices

- **Идемпотентность consumer** — at-least-once означает возможные дубли; нужен ключ идемпотентности
- **Паттерн Outbox** — единственный надёжный способ записи в БД и публикации события
- **Порядок Kafka** — только внутри партиции; для строгого порядка ключи маршрутизации
- **Не использовать синхронный вызов для долгих операций** — ответить 202 Accepted и обработать через очередь
- **Backpressure** — очередь сглаживает всплески, но consumer должен справляться с объёмом

Kafka vs RabbitMQ
?
- **Kafka** — event stream: replay, несколько подписчиков, высокая пропускная способность
- **RabbitMQ** — task queue: dispatch воркерам, DLQ, удаление после обработки

Синхронное vs Асинхронное взаимодействие
?
- **Синхронное** (HTTP/gRPC) — ожидание ответа, задержка = сумма вызовов, каскадные сбои
- **Асинхронное** (Queue) — "выстрелил и забыл", задержка 3100 мс → 25 мс, backpressure

Почему нужна идемпотентность consumer?::Kafka/RabbitMQ дают at-least-once доставку — сообщение может прийти дважды. Ключ идемпотентности на стороне consumer предотвращает дублирование

Порядок сообщений в Kafka гарантируется только ==внутри одной партиции==

Что такое Dead Letter Queue (DLQ)?::Очередь для сообщений, которые consumer не смог обработать после нескольких попыток (RabbitMQ)

Как работает паттерн Outbox?
?
1. В одной транзакции: INSERT business row + INSERT outbox row → COMMIT
2. Relay-процесс читает outbox
3. Публикует в Kafka
4. Помечает как published
