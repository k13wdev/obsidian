## Раздел 2. По каждому месту работы

### 2.1. ИТ-Финанс (Январь 2019 — Декабрь 2021)

**Вопрос: Расскажи про ИТ-Финанс. Чем занимался?**

ИТ-Финанс — интегратор, работал на стороне заказчика в периметре Сбербанка. За три года участвовал в двух больших проектах.

Первый — API-слой модуля юридических лиц в корпоративном банкинге. Там у нас был gRPC-сервис, который агрегировал данные о контрагентах из четырёх внутренних систем Сбера. Плюс репликация данных из Oracle в PostgreSQL через CDC на Debezium — это давало нам возможность работать с актуальными данными без нагрузки на оригинальный Oracle. Кэшировали горячие данные через Redis Cluster с инвалидацией по событиям.

Второй проект — единый профиль клиента, ЕПК. Это агрегатор данных из семи источников в единую карточку. И к нему — сервис генерации PDF и DOCX через gotenberg с очередью на Kafka и приоритизацией по типу документа.

**Стек:** Go 1.16, Gin, gRPC, PostgreSQL, Oracle, Redis, Kafka, Docker, Prometheus, Grafana, gotenberg, S3, text/template.

**Вопрос: Расскажи подробнее про CDC-репликацию из Oracle. Почему именно Debezium?**

Oracle был основным источником данных по юридическим лицам в Сбере. Прямые запросы туда из нашего сервиса были нежелательны по нескольким причинам: нагрузка на OLTP-систему, жёсткие SLA на стороне Oracle, и риск деградации в пиковые часы.

Решение — CDC через Debezium с захватом из redo-логов Oracle. Изменения попадали в Kafka-топик, оттуда потребитель писал их в PostgreSQL. Это давало нам реплику с отставанием стабильно меньше пяти секунд при нагрузке около 3k RPS.

Трудности были в основном операционные. Oracle XE в тестовой среде вёл себя иначе, чем Oracle EE в проде по поводу redo-логов. Плюс схема в Oracle менялась без предупреждения — пришлось строить schema evolution в consumer'е, чтобы не падать на новых полях.

**Вопрос: Расскажи про сервис генерации документов. Как решали проблему с памятью?**

Изначальная проблема была простая: при генерации тяжёлых PDF мы писали весь документ в in-memory буфер, отдавали клиенту и освобождали память. На двух podах это давало пики потребления до 1.2 GB, и при нескольких одновременных запросах мы получали OOM-kill.

Решение — переход на io.Pipe и стриминг. Генерация начинается, данные пишутся в pipe, клиент читает одновременно с генерацией. Никакого буфера целиком в памяти. Потребление памяти упало с 1.2 GB до 340 MB на пиках.

Параллельно закэшировали скомпилированные шаблоны text/template в памяти. Компиляция шаблона при каждом запросе давала заметный CPU overhead — после кэширования CPU на pod-ах снизился примерно на 22%.

**Вопрос: Как тестировали генерацию документов? Что такое e2e-тест для PDF?**

У нас были e2e-тесты с реальной генерацией. Тест подавал данные, запускал полный пайплайн через gotenberg, получал итоговый PDF или DOCX, и сравнивал с эталоном. Сравнение шло не через бинарное diff — у PDF-формата это непредсказуемо из-за метаданных и timestamps — а через извлечение текстового содержимого и сравнение структуры.

**Вопрос: Про dataloader-паттерн для N+1 — расскажи подробнее.**

В слое работы с контрагентами у нас была классическая проблема N+1: при выдаче списка контрагентов каждый элемент отдельным запросом тянул связанные данные — статусы, оборотную аналитику. При списке из ста контрагентов это 100+ запросов к базе.

Решение — dataloader-паттерн: на уровне middleware собираем все ID контрагентов из запроса, один батч-запрос тянет всё что нужно, результат раскладывается по ключам. Это стандартный подход из GraphQL-мира, но хорошо работает и в gRPC-контексте.

Эффект — построение отчётов по контрагентам ускорилось примерно на 28%.

**Вопрос: Почему выбрали Redis Cluster, а не просто Redis?**

Redis Cluster обеспечивает горизонтальное масштабирование и встроенную репликацию. Для данных по контрагентам, где у нас тысячи ключей и пиковые паттерны чтения, это разумный выбор.

Инвалидация шла по событиям из Kafka. Когда CDC ловил изменение записи контрагента в Oracle, соответствующий ключ в Redis инвалидировался. Это давало нам эвентуальную консистентность с гарантией, что устаревший кэш живёт не дольше, чем лаг CDC — то есть меньше пяти секунд.

### 2.2. Zuzex (Февраль 2022 — Апрель 2024)

**Вопрос: Что такое CVM-платформа для МТС? Чем ты там занимался?**

CVM — Customer Value Management. По сути это B2B-платформа для управления продажами корпоративным клиентам МТС: управление лидами, сегментация, воронки, аналитика поведения клиентов. Продукт внутренний, но с интеграциями во внешние CRM.

Я пришёл туда на уровне middle и в итоге вырос до сеньора за время в компании. Первый период — сервис управления лидами и воронками, стриминг событий в ClickHouse, оптимизация горячих путей, тестирование бизнес-логики.

**Стек:** Go 1.18, gRPC, PostgreSQL, ClickHouse, Redis, Kafka, Kafka Connect, Docker, Kubernetes, GitLab CI.

**Вопрос: Расскажи про state machine для лида. Почему go-fsm?**

Жизненный цикл лида — это чётко определённый набор состояний: новый, назначен, в работе, закрыт успешно, закрыт неуспешно, и так далее. Плюс переходы с условиями — не все переходы допустимы из любого состояния.

Самописная state machine — это постоянный соблазн написать switch на switch. С библиотекой go-fsm переходы описываются декларативно, что делает код читаемым и защищает от случайного некорректного перехода.

Эффект был не только в надёжности: время от создания лида до назначения менеджеру сократилось с 8 минут до 2 минут — это за счёт того, что убрали избыточные polling-петли и перешли на событийную модель триггеров.

**Вопрос: Keyset pagination вместо OFFSET/LIMIT — в чём разница?**

OFFSET/LIMIT в PostgreSQL при больших смещениях работает плохо: база всё равно перебирает первые N строк, чтобы пропустить их. При тысячах лидов и глубокой пагинации это квадратичный рост latency.

Keyset pagination работает иначе: в запросе передаём курсор — значение последнего полученного элемента по ключу сортировки. Запрос выглядит как `WHERE id > :last_id ORDER BY id LIMIT 50`. База использует индекс, никакого перебора.

У нас p95 latency выдачи списка лидов упал с 1.4s до 420ms. Ограничение keyset — нельзя перепрыгнуть на произвольную страницу, только последовательная навигация. Для нашего кейса это было приемлемо.

**Вопрос: Как работал стриминг в ClickHouse через Kafka Connect?**

Kafka Connect с JDBC Sink Connector или ClickHouse Sink — берёт события из Kafka-топика, батчами пишет в ClickHouse. Преимущество этого подхода — не нужен кастомный consumer, отказоустойчивость из коробки через Kafka offset management, retry логика встроена.

Отставание в нашем пайплайне стабильно держалось меньше 30 секунд, что давало near-real-time аналитику по лидам. Это важно для CVM-системы — менеджеры видят свежую воронку, а не вчерашние данные.

**Вопрос: Расскажи про документационный сервис. Что такое diff на уровне AST?**

Задача была такая: хранить версии документов и показывать diff между ними. Текстовый diff — простейший вариант, но для структурированных документов он неудобен: меняешь форматирование и получаешь огромный diff без смысловых изменений.

Мы пошли на уровень AST — абстрактного синтаксического дерева документа. Документ парсился в дерево структурных элементов: разделы, параграфы, таблицы, формулы. Diff считался по этому дереву. Это давало смысловые изменения: "изменился текст в параграфе 3.2", а не "добавлен пробел и перенос строки".

Публикация новой версии ускорилась с 3 минут до 25 секунд. Откат к любой версии — меньше секунды.

**Вопрос: Расскажи про внедрение Temporal. Почему именно Temporal?**

До Temporal у нас был самописный механизм retry для длинных процессов квалификации лидов. Это выглядело как: таблица в базе, scheduler который её поллит, флаги состояния, ручные recovery после сбоя. Это работало, но:

- код был сложный и плохо тестируемый;
- добавление нового шага в процесс требовало правок в нескольких местах;
- при сбое нужно было вручную разбираться в состоянии.

Temporal решает эти проблемы декларативно. Workflow — это обычный Go-код, но с гарантиями durability. Если сервис падает посреди процесса — Temporal восстанавливает его с того же места. Retry policy, timeout, compensation — всё настраивается через аннотации.

Результат: самописный механизм удалён, кодовая база сократилась на 1200 строк.

**Вопрос: TimescaleDB — почему не ClickHouse для временных рядов?**

ClickHouse мы использовали для аналитических пайплайнов лидов. TimescaleDB выбрали для агрегатора поведенческих метрик по другой причине: там был сложный SQL с JOIN, subquery и оконными функциями. ClickHouse хорошо на простых аналитических запросах, но при сложных JOIN он заметно проигрывает.

TimescaleDB — это PostgreSQL с расширением для временных рядов. Continuous aggregates позволяют предрасчитывать агрегаты по окнам времени — в фоне, без нагрузки на время запроса. Построение отчётов по когортам ускорилось на 45%.

**Вопрос: Менторинг — как ты к нему подходишь?**

В Zuzex я менторил двух джуниоров. Подход простой: не делать за них, а создать структуру для роста.

Первое — регулярные 1-on-1, раз в неделю. Не отчёт о задачах, а обсуждение сложностей и вопросов. Второе — ревью с объяснением: не просто "это неправильно", а почему и какой подход лучше. Третье — постепенное усложнение задач — давать задачи чуть выше текущего уровня, но с подстраховкой.

Гайдлайны по Go — это тоже часть менторинга. Когда правила зафиксированы в документе, ревью становится объективным, а не вкусовщиной. Мы запустили weekly tech-sync, где разбирали интересные случаи из ревью. Количество мерж-конфликтов в спринте снизилось на 24% — это косвенный показатель того, что команда стала лучше координироваться.

### 2.3. Прикладные решения (Апрель 2024 — Январь 2025)

**Вопрос: Расскажи про брокер МФО. Что это за система?**

Брокер МФО — агрегатор займов. Человек заполняет заявку на сайте, система скорит его и роутит к подходящим партнёрам-МФО. Там 23 партнёра, у каждого своё API — разные форматы, разные правила, разные таймауты.

Моя основная задача — переработка движка роутинга и адаптерный слой под партнёров. Плюс JSON-DSL для продуктовой команды, чтобы они могли менять правила без деплоя.

**Стек:** Go 1.20, Echo, gRPC-gateway, PostgreSQL, Redis, RabbitMQ, ClickHouse, Docker, Kubernetes.

**Вопрос: Расскажи про JSON-DSL для бизнес-правил.**

Продуктовая команда постоянно меняет правила: какому партнёру слать заявки с каким скорингом, какие условия по сумме и срокам, как приоритизировать. Без DSL каждое изменение — это задача в бэклог, разработка, деплой. Цикл — от 3 дней и больше.

Мы спроектировали DSL в JSON: условия описываются как дерево предикатов с логическими операторами. Например: "если скоринг больше 650 И сумма меньше 100k — роутить в партнёров A, B, C в порядке приоритета". JSON Schema для валидации правил — продуктовая команда загружает правила через UI, валидация срабатывает мгновенно.

Цикл правки условий упал с 3 дней до 2 часов — теперь это делает сам продакт, без разработчика.

**Вопрос: Параллельная отправка заявок в 23 МФО — как это устроено?**

Заявка приходит, система скорит, затем параллельно отправляет в список подходящих МФО. Каждый запрос к партнёру — это отдельная goroutine, результаты собираются через channel с таймаутом.

Важный момент — не все партнёры должны ответить. Нам нужно хотя бы одно одобрение. Поэтому первое одобрение возвращается клиенту сразу, остальные запросы отменяются через context cancellation. Это значительно снижает время ответа по сравнению с последовательной отправкой.

Конверсия в одобренный займ выросла с 12% до 16% — за счёт лучшего роутинга и параллельности.

**Вопрос: RabbitMQ с exponential backoff — почему RabbitMQ?**

RabbitMQ хорошо подходит для task queues с routing key, priority queue и DLQ из коробки. Для нашей задачи — ретраи сломанных партнёрских API с backoff — это был готовый инструмент.

Механизм: при неудачном запросе к МФО сообщение переходит в delayed queue с экспоненциально растущей задержкой — 30s, 2min, 10min и так далее. После максимального числа попыток — в DLQ для ручного разбора.

Доля потерянных заявок снизилась с 3.2% до 0.4%.

**Вопрос: Property-based тесты на gopter.**

Скоринговые правила — это сложная логика с множеством граничных условий. Обычные unit-тесты покрывают то, что ты придумал. Property-based тесты генерируют случайные входные данные и проверяют инварианты.

Например, инвариант: "если заявка одобрена, скоринг должен быть выше минимального порога". Gopter генерирует тысячи случайных комбинаций параметров заявки и пытается нарушить этот инвариант.

До выката в прод таким образом было поймано 7 edge-case багов в правилах одобрения — вещи, которые никто не придумал бы написать в обычном тест-кейсе.

### 2.4. Тактикум (Январь 2025 — настоящее время)

**Вопрос: Расскажи про кредитный конвейер.**

Кредитный конвейер — это полный цикл от заявки до выдачи денег. Клиент подаёт заявку через мобильное приложение или партнёрский канал. Система проверяет его через 4 бюро кредитных историй и ФНС, принимает решение, и если одобрено — выдаёт средства.

Сложность — в нескольких вещах. Первое: множество внешних интеграций с нестабильными SLA — БКИ могут отвечать долго или не отвечать. Второе: финансовая идемпотентность — нельзя допустить двойного списания. Третье: длинные бизнес-процессы — от заявки до выдачи может проходить несколько минут с несколькими асинхронными шагами.

Архитектурно это 12 Go-микросервисов с оркестрацией через Temporal.

**Результаты:**
- Time-to-decision упал с 7 минут до 90 секунд
- Ноль инцидентов двойного списания за 18 месяцев
- p99 latency финального решения: с 3.1s до 780ms

**Вопрос: Как устроен адаптерный слой под БКИ и ФНС?**

Четыре бюро кредитных историй — каждое со своим API, форматом ответа, SLA и семантикой ошибок. Единый адаптерный слой — это интерфейс, который унифицирует запрос и ответ. Под каждый источник — конкретная реализация.

Паттерн — интерфейс + фабрика. Скоринговый сервис работает только с интерфейсом. Добавление нового источника данных — это написание нового адаптера и регистрация в фабрике. Это ускорило добавление новых интеграций и изолировало нестабильность каждого источника.

Одобрение качественных заёмщиков выросло на 19% — за счёт более глубокой и разнообразной проверки.

**Вопрос: Outbox-паттерн для идемпотентности.**

Outbox — это классический паттерн для гарантированной доставки событий без двухфазного коммита.

Логика: финансовая транзакция записывается в основную таблицу и одновременно в outbox-таблицу в той же PostgreSQL-транзакции. Отдельный publisher-процесс читает из outbox, публикует событие в Kafka, помечает запись как опубликованную.

На стороне потребителя — идемпотентный ключ. Если событие получено дважды (что бывает при at-least-once delivery), второй раз оно игнорируется.

Результат: за 18 месяцев эксплуатации ноль инцидентов двойного списания. Это самая важная метрика для финансового контура.

**Вопрос: Feature-flag сервис для скоринговых моделей.**

Самописный сервис на Go + Consul позволял хранить конфигурацию флагов в Consul KV, изменения применялись без деплоя, откат — это изменение конфига. Канареечный выкат скоринговой модели: 5% трафика на новую модель, мониторинг метрик, при проблемах — откат за 90 секунд вместо 40 минут.

**Вопрос: buf и CI-проверки на breaking changes.**

buf — это инструмент для работы с Protocol Buffers: линтинг, генерация кода, и самое важное — проверка обратной совместимости. В CI при каждом Pull Request автоматически запускается `buf breaking` — проверка, что новые .proto-файлы не ломают существующих потребителей.

Breaking changes в gRPC-контрактах — это серьёзная проблема в микросервисной архитектуре: один сервис обновился, другой — нет, всё сломалось. За год эксплуатации этой системы ни одного breaking change между релизами.

**Вопрос: Как ты перевёл Java-сервис на Go?**

Подход — strangler fig pattern. Не переписываем всё сразу, а создаём новый Go-сервис параллельно и постепенно переводим трафик.

Обратная совместимость контрактов: новый gRPC-сервис реализует те же proto-контракты. Это ключевое требование — все существующие клиенты не должны ничего менять. На этапе перехода оба сервиса работают параллельно, трафик переключается постепенно.

Результат: потребление инфраструктуры снизилось на 34%, cold-start ускорился в 8 раз — это характерная разница между JVM и Go-бинарником.

## Раздел 3. Технические вопросы по стеку

### Вопрос: Расскажи про goroutines и concurrency в Go. Как избегаешь race conditions?

Goroutines — это легковесные потоки выполнения, мультиплексируемые на OS-потоки планировщиком Go. Их можно создавать миллионами, стоимость — около 2-8 KB начального стека.

Race conditions избегаю несколькими способами:

**Первый — channels.** Передача данных через канал вместо shared memory. Go-философия: "communicate, don't share memory".

**Второй — sync-примитивы:** Mutex, RWMutex для явной защиты shared state. RWMutex использую когда читателей много, писателей мало.

**Третий — sync/atomic** для простых счётчиков и флагов без полного мьютекса.

**Четвёртый — `go test -race`.** Всегда. В CI обязательно. Детектор гонок ловит то, что визуальный ревью пропускает.

На практике использую context для cancellation, errgroup для группы goroutine с общим контекстом ошибки.

### Вопрос: Что такое context в Go и как ты его используешь?

context.Context — это стандартный способ передачи deadline, cancellation signal и произвольных значений через цепочку вызовов.

В нашем кредитном конвейере context используется везде: каждый запрос к БКИ или ФНС запускается с timeout-контекстом. Если внешний сервис не отвечает — контекст отменяется, goroutine прекращает работу, ресурсы освобождаются.

При параллельных запросах к нескольким источникам: первый успешный ответ отменяет контексты всех остальных через cancel-функцию.

Важное правило: context всегда первый аргумент функции, никогда не в структуре (кроме отдельных случаев с http.Request).

### Вопрос: Как работает gRPC? Чем отличается от REST?

gRPC — это RPC-фреймворк от Google на базе HTTP/2 и Protocol Buffers.

**Отличия:**

**Протокол:** HTTP/2 вместо HTTP/1.1. Это даёт multiplexing — несколько запросов по одному TCP-соединению, без head-of-line blocking. Плюс server push и bidirectional streaming.

**Сериализация:** Protocol Buffers — бинарный формат, компактнее JSON в 3-10 раз, быстрее сериализации/десериализации.

**Типизация:** .proto-файл — это строгий контракт. Код генерируется автоматически. Клиент и сервер всегда в синхронизированных схемах.

**Streaming:** gRPC поддерживает четыре режима — unary, server streaming, client streaming, bidirectional streaming. В скоринговом сервисе использую server streaming для передачи обновлений статуса заявки в реальном времени.

Недостаток gRPC — плохо читается человеком, сложнее дебажить без специальных инструментов вроде grpcurl или grpcui.

### Вопрос: Расскажи про partitioning в PostgreSQL.

Партиционирование — это разбиение большой таблицы на физически отдельные части (партиции) при сохранении единого логического доступа.

В Zuzex партиционировал таблицу лидов по дате (`RANGE` партиционирование по `created_at`). Причины:

**Производительность:** Planner может исключить нерелевантные партиции (partition pruning). Если запрос за последний месяц — сканируется только одна партиция, а не вся таблица за 3 года.

**Обслуживание:** старые данные можно удалить через `DROP PARTITION` — это мгновенно и не создаёт WAL-нагрузку в отличие от `DELETE`.

**Вакуумирование:** меньше bloat, более предсказуемый vacuum на каждой партиции.

Результат: время тяжёлых аналитических запросов снизилось на 38%.

### Вопрос: EXPLAIN ANALYZE — как ты его используешь?

`EXPLAIN ANALYZE` выполняет запрос и показывает реальный план выполнения с фактическими строками и временем.

На что смотрю в первую очередь:

**Seq Scan вместо Index Scan** на большой таблице — сигнал отсутствующего индекса или неправильной статистики.

**Rows Removed by Filter** — если высокое значение, фильтрация идёт после чтения большого количества строк.

**Hash Join vs Nested Loop** — Nested Loop плохо масштабируется при больших наборах данных.

**actual rows vs estimated rows** — большое расхождение означает устаревшую статистику — нужен `ANALYZE`.

### Вопрос: Расскажи про Kafka. Как гарантируешь exactly-once semantics?

Kafka — это distributed log. Продюсеры пишут в топики, разделённые по партициям. Консьюмеры читают с офсетом.

Exactly-once semantics — это сложная тема. На практике в большинстве систем мы реализуем это через паттерн на прикладном уровне:

**Outbox-паттерн:** событие записывается в транзакции с бизнес-данными. Kafka публикуется отдельно. На стороне потребителя — идемпотентный ключ.

**At-least-once + idempotent consumer:** потребитель может получить сообщение дважды при ребалансировке. Если обработка идемпотентна (ключ уже в базе — пропускаем), дублирование безопасно.

### Вопрос: Temporal — как работает durability?

Temporal хранит историю событий workflow. Каждый шаг — отдельная запись в event sourcing log Temporal-сервера.

При падении воркера посреди workflow: новый воркер получает задание, Temporal воспроизводит историю событий, и код workflow детерминированно "переигрывает" с начала, но все уже завершённые активити не исполняются повторно — их результаты берутся из истории.

Ключевое требование: workflow-код должен быть детерминированным. Нельзя использовать time.Now(), rand, IO напрямую — только через Temporal API.

### Вопрос: OpenTelemetry — что трейсируешь и как?

OpenTelemetry — это vendor-neutral SDK для distributed tracing, metrics и logs.

В кредитном конвейере трейсируем весь путь заявки: HTTP-запрос → скоринг → запросы к БКИ → Temporal activity → принятие решения → выдача. Trace_id сквозной — одинаковый от первого запроса до последнего шага.

Что даёт: при инциденте открываешь Jaeger, находишь trace по trace_id из логов, видишь весь стек вызовов с временными метками каждого шага. Среднее время поиска root cause упало с 55 минут до 14 минут.

Инструментация: middleware для HTTP/gRPC, ручные span'ы для критичных участков бизнес-логики, attribute'ы с бизнес-контекстом.

### Вопрос: Redis — какие паттерны использовал?

**Distributed cache** (Zuzex, ИТ-Финанс): кэширование результатов дорогих запросов с инвалидацией по событиям.

**TTL-инвалидация** (Тактикум): предрасчёт скоринговых фич с TTL. Фичи пересчитываются в фоне до истечения TTL, горячий путь не блокируется.

**Deduplication** (Zuzex): Redis + xxhash для дедупликации входящих лидов.

**Rate limiting** (ИТ-Финанс): token bucket на Redis для gRPC-gateway.

Важный принцип: Redis — это кэш, а не база данных. Всегда есть plan на случай его недоступности.

### Вопрос: Как работает ClickHouse?

ClickHouse — это column-oriented OLAP-база данных, оптимизированная для аналитических запросов над большими объёмами данных.

Column-oriented: данные хранятся по столбцам, а не по строкам. Для аналитики, где нужно агрегировать один-два столбца из таблицы с сотнями миллионов строк — это принципиально быстрее.

Когда выбирать: быстрые аналитические запросы по большому объёму данных, отчёты по временным рядам, агрегации с GROUP BY. Не подходит для транзакционной нагрузки с UPDATE/DELETE.

### Вопрос: Как устроен Kubernetes?

Kubernetes — это оркестратор контейнеров. Мы работаем с ним через Helm-чарты — это параметризованные шаблоны для деплоя.

При проблемах с pod типичный процесс диагностики:

```
kubectl get pods -n <namespace>
kubectl describe pod <pod-name>
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
```

Частые причины: OOM-kill, CrashLoopBackOff, Pending.

В продакшне автоскейлинг через HPA по CPU/memory или кастомным метрикам через KEDA.

### Вопрос: CDC на Debezium — расскажи подробнее.

CDC — Change Data Capture. Debezium читает не сам сервис, а WAL (Write-Ahead Log) базы данных. Это значит, что Debezium видит каждое изменение данных без нагрузки на приложение.

Для Oracle: Debezium читает redo logs. Для PostgreSQL: читает WAL через логическую репликацию (wal2json или pgoutput).

Каждое изменение публикуется в Kafka как событие с before/after состоянием.

Сложности:

**Schema evolution:** если схема меняется, Debezium нужно сообщить об изменении.

**Initial snapshot:** при первом запуске Debezium делает полный snapshot таблицы, потом переходит на CDC. Для больших таблиц это медленно.

**Tombstone messages:** удаление записи публикуется как null-сообщение с ключом — нужно правильно обрабатывать.

## Раздел 4. Проекты и достижения

### Вопрос: Какое достижение ты считаешь самым значимым?

Если выбрать одно — outbox-паттерн с результатом ноль инцидентов двойного списания за 18 месяцев в продакшне.

Не потому что это технически самое сложное. А потому что это прямое влияние на бизнес-критичный риск. В финтехе двойное списание — это регуляторные последствия, жалобы клиентов и потеря репутации. Архитектурное решение, которое элиминирует целый класс ошибок — это ценнее, чем любая оптимизация latency.

### Вопрос: Расскажи о самом сложном техническом решении.

Это миграция legacy Java-сервиса принятия решений на Go в кредитном конвейере.

Сложность была не в переписывании кода — это несложно. Сложность была в том, чтобы сделать это без downtime и с гарантией совместимости контрактов.

Решение: strangler fig. Новый Go-сервис поднимается параллельно, реализует те же gRPC-контракты. На уровне service mesh или балансировщика постепенно переводим процент трафика на новый сервис — сначала 1%, потом 10%, 50%, 100%. На каждом этапе сравниваем ответы обоих сервисов (shadow mode).

Результат без единого инцидента. Потребление инфраструктуры снизилось на 34%, cold-start ускорился в 8 раз.

### Вопрос: Расскажи о самом большом фейле.

При первом внедрении Temporal в Zuzex мы недооценили требование детерминированности workflow-кода. В один из воркеров прокралось использование `time.Now()` напрямую вместо `workflow.Now()`. Это создало проблему при replay: workflow вёл себя по-разному при восстановлении и при первичном исполнении.

Симптом — периодически workflow застревал в неожиданном состоянии после сбоя воркера. Диагностика заняла несколько часов, потому что проблема не воспроизводилась стабильно.

Урок: не инструмент виноват, а недостаточная глубина изучения его ограничений перед внедрением. После этого мы написали гайдлайн по Temporal с явным списком запрещённых паттернов в workflow-коде.

### Вопрос: Как ты подходишь к оптимизации производительности?

По шагам:

**Первый — не оптимизировать без измерения.** Сначала profiling: `pprof` в Go даёт CPU profile, heap profile, goroutine dump. Смотрю где реально тратится время.

**Второй — определить hotpath.** 80% времени тратится в 20% кода. Именно туда направляю усилия.

**Третий — выдвигаю гипотезу, делаю изменение, измеряю результат.** Не несколько изменений сразу.

**Четвёртый — фиксирую метрику до и после.**

Примеры: keyset pagination в Zuzex (p95 1.4s → 420ms), предрасчёт фич в Redis в Тактикуме (p99 3.1s → 780ms), io.Pipe вместо буфера в ИТ-Финанс (память 1.2GB → 340MB).

### Вопрос: Как ты проектируешь архитектуру? (кредитный конвейер)

Начинаю с domain-driven подхода: что такое кредитный конвейер как бизнес-процесс? Это последовательность шагов с ветвлением, внешними интеграциями и строгими требованиями к надёжности.

Исходя из этого — Temporal для оркестрации. Он даёт durability, retry, timeout, visibility из коробки.

Декомпозиция на 12 микросервисов по bounded context: приём заявок, скоринг, интеграция с БКИ, интеграция с ФНС, принятие решения, выдача.

Контракты через gRPC + buf — строгая типизация, CI-проверки на breaking changes.

Финансовая идемпотентность — outbox-паттерн, потому что это надёжнее.

Observability с первого дня — OpenTelemetry с trace_id сквозным.

## Раздел 5. Behavioural / Situational вопросы

### Вопрос: Расскажи о ситуации, когда ты не согласился с решением команды.

**Situation:** В Zuzex при внедрении нового сервиса агрегации метрик тимлид предложил хранить временные ряды в обычном PostgreSQL с индексами по timestamp.

### Вопрос: Расскажи о ситуации при сжатых сроках.

**Situation:** В Прикладных решениях партнёр-МФО изменил свой API без предупреждения. Наш адаптер начал возвращать ошибки, заявки перестали обрабатываться. Это заметили в пятницу вечером.

**Task:** Восстановить интеграцию максимально быстро без полного понимания новой схемы API.

**Action:** Включил DLQ — потерянные заявки не теряются, они скапливаются в dead letter queue. Это дало время. Затем прошёлся по ответам нового API, вычислил изменения в схеме, обновил адаптер, написал тест, задеплоил hotfix. Параллельно запустил обработку DLQ.

**Result:** За 3 часа интеграция восстановлена, все потерянные заявки обработаны.

### Вопрос: Расскажи о trade-off между скоростью и качеством.

**Situation:** В Тактикуме при запуске кредитного конвейера был жёсткий дедлайн.

**Task:** Запустить MVP в срок, не жертвуя безопасностью финансовых операций.

**Action:** Явно разделил что можно откладывать и что нельзя. Нельзя было откладывать: outbox-паттерн для идемпотентности. Можно было: аналитические дашборды, расширенный мониторинг, автоскейлинг. Зафиксировал tech debt явно — как конкретные задачи со сроками.

**Result:** Запустились в срок. Финансовый контур работал правильно с первого дня. Tech debt закрыт в следующие два спринта.

### Вопрос: Как ты объясняешь технические решения нетехническим стейкхолдерам?

Избегаю технического жаргона. Перевожу в бизнес-последствия.

Например, вместо "мы внедрили outbox-паттерн" — "мы сделали так, что система физически не может списать деньги дважды".

Вместо "оптимизировали keyset pagination" — "списки лидов теперь открываются за полсекунды вместо полутора".

Для принципиальных архитектурных решений показываю risk/benefit: что нас спасает это решение от, и что будет стоить, если этого не сделать.

### Вопрос: Как ты подходишь к code review?

Code review — это не поиск ошибок, это инструмент передачи знаний и повышения качества кодовой базы.

Что проверяю в первую очередь:

**Корректность:** делает ли код то, что должен. Граничные случаи. Error handling. Конкурентная безопасность.

**Понятность:** смогу ли я через 6 месяцев понять, что здесь происходит. Имена переменных, комментарии к нетривиальной логике.

**Согласованность:** не отклоняется ли от принятых паттернов команды.

**Тесты:** покрыты ли граничные случаи, тесты читаемы.

Что НЕ делаю: не придираюсь к форматированию — это работа линтера. Не блокирую PR из-за стилистических предпочтений.

### Вопрос: Расскажи о работе с legacy-кодом.

**Situation:** В Прикладных решениях legacy-backend админки заявок был на старом стеке — время загрузки таблицы заявок составляло 22 минуты.

**Task:** Переписать backend так, чтобы не сломать логику и улучшить производительность.

**Action:** Сначала — понять, что делает текущий код. Написал интеграционные тесты на существующее поведение — это дал "скриншот" ожидаемых результатов. Затем переписал backend на Go + gRPC-gateway, покрывая те же сценарии. Переход без downtime.

**Result:** Время загрузки упало с 22 минут до 14 секунд.

### Вопрос: Как ты внедрял что-то новое в команде?

**Situation:** В Zuzex хотел внедрить Temporal. Часть команды скептически отнеслась — "зачем новый инструмент, у нас работает".

**Task:** Аргументировать внедрение и снизить риски.

**Action:** Подготовил сравнение: сколько строк кода и скрытых багов в текущем механизме. Показал incident из прошлого, который Temporal исключил бы. Сделал proof of concept на одном процессе за неделю. Не внедрял сразу во всё.

**Result:** Temporal внедрили. Самописный механизм удалён. Кодовая база уменьшилась на 1200 строк. Команда потом сама предлагала переводить другие процессы.

### Вопрос: Как ты расставляешь приоритеты?

Первый критерий — бизнес-риск. Что сломает клиентов или потеряет деньги — немедленно.

Второй — зависимости команды. Если другие люди заблокированы — это приоритет.

Третий — сроки с внешними обязательствами.

Четвёртый — стратегический tech debt с накапливающимися последствиями.

При сжатых сроках — говорю явно о том, что не успею и что нужно деприоритизировать.

**Риск 1: Метрики звучат слишком хорошо**

Смягчение: каждый переход имеет внятную логику роста. ИТ-Финанс → Zuzex → Прикладные → Тактикум — понятная траектория от junior до senior-архитектора.

Смягчение: честно говорить "точную цифру не помню, но порядок был такой" лучше, чем выдумывать.

**Риск 5: Вопрос про работу в команде**
Смягчение: менторинг в Zuzex (двое джунов) и Тактикуме (двое middleов). Weekly tech-sync и гайдлайны — командная работа.
