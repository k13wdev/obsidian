## Суть и контекст

**Event Loop (Цикл обработки событий)** — механизм координации выполнения кода, работы с событиями и обновления интерфейса в **основном потоке браузера** (main thread). Поскольку JavaScript, DOM, расчёт стилей и рендеринг выполняются в **одном потоке**, они не могут работать одновременно — Event Loop строго упорядочивает их выполнение.

Тесно связан с [[Paint, Repaint и Compositing]] и [[Layout (Reflow)]], поскольку рендеринг — это один из шагов итерации Event Loop.

## Ключевые идеи

- Main thread отвечает за JS, DOM, стили и рендеринг — всё в одном потоке
- Event Loop — **бесконечный цикл**, который ожидает события и выполняет их в строгом порядке
- Один тик Event Loop: **1 Task → все Microtasks → (если пора) Rendering (rAF → Style → Layout → Paint)**
- **Microtask queue** опустошается **полностью** перед рендерингом — не одна, а все
- **Bесконечный цикл микрозадач заблокирует рендеринг**
- `requestAnimationFrame` выполняется **перед** вычислением стилей и Paint
- `setTimeout(fn, 0)` ≠ «сразу» — минимальная задержка ~4.7 мс + ставится в конец Task Queue
- Рендеринг происходит **не после каждой задачи** — браузер синхронизирует его с частотой обновления монитора (~60 Гц, каждые ~16.7 мс)

## Определения и термины

**Main Thread (Основной поток)** — единственный поток браузера, в котором выполняются JS, работа с DOM, расчёт стилей и рендеринг.

**Event Loop** — бесконечный цикл, поддерживающий жизнь процесса: ожидает события и выполняет их в строгом порядке.

**Call Stack (Стек вызовов)** — структура данных, отслеживающая, какая функция сейчас выполняется. Функция попадает в стек при вызове (push) и удаляется при завершении (pop).

**Task Queue / Macrotask Queue (Очередь задач)** — очередь макрозадач, из которой Event Loop берёт по одной задаче за итерацию.

**Task (Макрозадача)** — единица работы, которую браузер ставит в очередь задач. Примеры: коллбэки `setTimeout`/`setInterval`, обработчики пользовательского ввода (клики, скролл), сетевые ответы, парсинг HTML.

**Microtask Queue (Очередь микрозадач)** — очередь задач, обрабатываемая **целиком** после каждой макрозадачи (и после каждого коллбэка при пустом стеке), до рендеринга.

**Microtask (Микрозадача)** — задача для немедленного выполнения после завершения текущего кода, без задержки на новую полноценную Task. Примеры: коллбэки `Promise` (`.then()`, `.catch()`), `MutationObserver`, `queueMicrotask()`.

**requestAnimationFrame (rAF)** — API для постановки коллбэка в очередь рендеринга. Выполняется строго **перед** вычислением стилей, Layout и Paint.

**Rendering Step (Шаг рендеринга)** — шаг итерации Event Loop, включающий: rAF-коллбэки → Style Calculation → Layout → Paint → Compositing.

**queueMicrotask()** — стандартизированный API для добавления функции в очередь микрозадач напрямую, без создания промиса.

## Как устроено

### Полный порядок одного тика (итерации) Event Loop

```
1. Task (одна макрозадача из Task Queue)
        │
        ▼
2. Microtasks (все микрозадачи до полного опустошения очереди)
        │
        ▼
3. Rendering (только если браузер решил, что пора обновить экран):
   ├── requestAnimationFrame callbacks
   ├── Style Calculation (вычисление стилей)
   ├── Layout (Reflow)
   └── Paint & Compositing
        │
        ▼
4. Idle (ожидание, фоновые задачи)
        │
        ▼
1. Следующая Task...
```

**Важно**: рендеринг происходит не в каждой итерации. Браузер синхронизирует его с частотой обновления экрана (~60 Гц = каждые 16.7 мс). Если ничего не изменилось или вкладка в фоне — рендеринга не будет.

### Call Stack

- Микрозадачи и макрозадачи **не начнут выполняться**, пока Call Stack не станет **полностью пустым**
- Пока функция выполняется — она в стеке; завершилась — удалена из стека

### Task Queue (Макрозадачи)

- На каждой итерации берётся **только одна задача**
- После выполнения задачи браузер может (но не обязан) обновить рендеринг
- Новые задачи, добавленные во время выполнения текущей — ставятся в конец очереди

### Microtask Queue

- Обрабатывается **после каждого коллбэка** (если стек пуст) и в конце каждой задачи
- В отличие от Task Queue: берётся **вся очередь целиком**, не одна
- Если микрозадача добавляет новую микрозадачу — она выполняется **в этом же цикле**
- Бесконечный цикл микрозадач **заблокирует рендеринг** (и main thread)

### requestAnimationFrame

- Коллбэк rAF выполняется **перед** вычислением стилей и Paint
- Синхронизирован с частотой обновления экрана — вызывается **ровно один раз перед каждым кадром**
- Если вкладка в фоне — rAF не вызывается

**Старые браузеры** (старые Safari, Edge): rAF ошибочно вызывался **после** Paint, что вызывало задержки. По веб-стандартам и в современных Chrome/Firefox — rAF вызывается **до** вычисления стилей и Paint.

## Детали и нюансы

### setTimeout(fn, 0) vs Promise vs rAF

| | Тип | Когда выполняется | Синхронизация с кадром |
|---|---|---|---|
| `Promise.then()` | Microtask | Сразу после текущей задачи | Нет |
| `queueMicrotask()` | Microtask | Сразу после текущей задачи | Нет |
| `requestAnimationFrame()` | Rendering step | Перед следующим Paint | Да (≈60 Гц) |
| `setTimeout(fn, 0)` | Task (Macrotask) | Следующая Task + ≥4.7 мс задержка | Нет |

### Почему setTimeout(fn, 0) медленнее Promise

- `setTimeout` ставит **новую Task** в конец очереди макрозадач
- Браузеры устанавливают минимальную задержку таймеров (~4.7 мс)
- Коллбэки не синхронизированы с ~60 Гц → анимация через `setTimeout` вызывает jank:
  - Коллбэки могут выполниться несколько раз за один кадр без видимого результата
  - Или опоздать к моменту рендеринга

### queueMicrotask()

Явный API для добавления в Microtask Queue:
```javascript
queueMicrotask(() => {
  console.log('это микрозадача');
});
// Эквивалентно Promise.resolve().then(...), но без накладных расходов
```

### scheduler.postTask()

> **Примечание**: информация о `scheduler.postTask` **отсутствует в источниках ноутбука** — отмечена как внешние знания.

API для постановки задач с явным приоритетом (Prioritized Task Scheduling API).

## Сравнения и trade-offs

| Очередь | Порядок | Кол-во за итерацию | До рендеринга? |
|---------|---------|-------------------|----------------|
| Task Queue | FIFO | 1 | Зависит |
| Microtask Queue | FIFO | Все до опустошения | Да (всегда до рендеринга) |
| rAF Queue | FIFO | Все в момент вызова | Да (перед Style/Layout/Paint) |

## Примеры

### Порядок вывода: setTimeout vs Promise vs rAF

```javascript
console.log('script start');

// Макрозадача
setTimeout(() => {
  console.log('setTimeout');
}, 0);

// Микрозадачи
Promise.resolve().then(() => {
  console.log('promise1');
}).then(() => {
  console.log('promise2');
});

// Перед следующим кадром
requestAnimationFrame(() => {
  console.log('requestAnimationFrame');
});

console.log('script end');
```

**Порядок вывода:**
```
script start          ← синхронный код текущей задачи
script end            ← синхронный код текущей задачи
promise1              ← microtask (сразу после завершения текущего скрипта)
promise2              ← microtask (добавлена в той же очереди)
requestAnimationFrame ← rendering step (перед следующим обновлением экрана)
setTimeout            ← новая macrotask (следующая итерация event loop)
```

### Опасность бесконечного цикла микрозадач

```javascript
// ПЛОХО: заблокирует рендеринг навсегда
function infiniteMicrotasks() {
  Promise.resolve().then(infiniteMicrotasks);
}
infiniteMicrotasks();
// Microtask Queue никогда не опустеет → рендеринг не произойдёт
```

### Правильное использование rAF для анимации

```javascript
// ПЛОХО: setTimeout не синхронизирован с кадром
function animateWithTimeout() {
  element.style.left = (parseFloat(element.style.left) + 1) + 'px';
  setTimeout(animateWithTimeout, 0); // может вызваться несколько раз за кадр
}

// ХОРОШО: rAF синхронизирован с частотой обновления экрана
function animateWithRAF() {
  element.style.left = (parseFloat(element.style.left) + 1) + 'px';
  requestAnimationFrame(animateWithRAF); // ровно один раз перед каждым Paint
}
requestAnimationFrame(animateWithRAF);
```

## Связи с другими темами

- [[Paint, Repaint и Compositing]] — Rendering Step — часть итерации Event Loop
- [[Layout (Reflow)]] — Layout происходит в Rendering Step, после rAF-коллбэков
- [[Critical Rendering Path]] — рендеринг встроен в Event Loop как периодический шаг
- [[Парсинг HTML]] — парсинг HTML — это Task; `<script>` блокирует парсер в main thread

## Важно / подводные камни / best practices

- **Долгие задачи в main thread** (>50 мс) блокируют рендеринг и вызывают jank — дробить на части с `setTimeout` или `scheduler.postTask`
- **Бесконечный цикл микрозадач** заблокирует рендеринг навсегда — никогда не создавай рекурсивные промисы без базового случая
- **rAF — правильный инструмент для анимаций**, `setTimeout` — неправильный
- **Не обновляй DOM в цикле** без rAF — изменения могут быть применены несколько раз за кадр без отрисовки
- **Microtasks выполняются до рендеринга** — используй их для задач, которые должны завершиться до следующего Paint, но не блокируй очередь
- **Compositor Thread** (для `transform`/`opacity`) работает вне Event Loop — анимации на GPU не блокируются JS

#flashcards/browser/eventloop

Что такое Event Loop?::Бесконечный цикл, поддерживающий жизнь процесса браузера: ожидает события и выполняет их в строгом порядке в main thread.

Почему JS, DOM и рендеринг работают в одном потоке?::Чтобы избежать состояний гонки (race conditions) при одновременном доступе к DOM из нескольких потоков.

Перечисли полный порядок одного тика Event Loop::1. Task (одна макрозадача). 2. Все Microtasks до опустошения очереди. 3. Rendering (rAF → Style → Layout → Paint) — если браузер решил обновить экран. 4. Idle.

Сколько макрозадач берётся за одну итерацию Event Loop?::Одна.

Сколько микрозадач берётся за одну итерацию Event Loop?::Все — до полного опустошения очереди.

Что происходит, если микрозадача добавляет новую микрозадачу?::Новая микрозадача выполнится в этом же цикле обработки микрозадач (до рендеринга).

Что блокирует бесконечный цикл микрозадач?::Рендеринг — браузер не перейдёт к Rendering Step, пока Microtask Queue не опустеет.

Что такое Call Stack?::Структура данных, отслеживающая, какая функция сейчас выполняется. Задачи и микрозадачи не начнут выполняться, пока Call Stack не станет полностью пустым.

Когда выполняется requestAnimationFrame коллбэк в порядке Event Loop?::В Rendering Step — перед вычислением стилей, Layout и Paint. После всех микрозадач текущей итерации.

Чем отличается setTimeout(fn, 0) от Promise.then()?::Promise.then → микрозадача, выполняется сразу после текущей задачи до рендеринга. setTimeout → новая макрозадача с минимальной задержкой ~4.7 мс, выполняется в следующей итерации event loop.

Почему анимация через setTimeout вызывает jank?::setTimeout не синхронизирован с частотой обновления экрана (~60 Гц). Коллбэки могут выполниться несколько раз за кадр без видимого результата или опоздать к моменту Paint.

Что такое queueMicrotask()?::Стандартизированный API для прямого добавления функции в Microtask Queue. Аналог Promise.resolve().then(), но без накладных расходов на создание промиса.

Что такое Microtask?::Задача для немедленного выполнения после завершения текущего кода, без задержки на новую полноценную Task. Примеры: Promise.then(), MutationObserver, queueMicrotask().

Что такое Task (Macrotask)?::Единица работы в Task Queue. Примеры: коллбэки setTimeout/setInterval, обработчики кликов, сетевые ответы, парсинг HTML.

Каков минимальный интервал срабатывания setTimeout?::Около 4.7 миллисекунд (браузер устанавливает минимальную задержку для таймеров).
