## Суть и контекст

**Critical Rendering Path (Критический путь рендеринга)** — последовательность промежуточных этапов, которые браузер выполняет для преобразования полученных по сети байтов HTML, CSS и JavaScript в пиксели, отображаемые на экране пользователя. Оптимизация CRP позволяет значительно сократить время до первой отрисовки страницы.

Это сквозная тема, объединяющая все этапы работы браузера: [[02. Парсинг HTML]], [[03. Парсинг CSS (CSSOM)]], [[05. Layout (Reflow)]], [[06. Paint, Repaint и Compositing]].

## Ключевые идеи

- CRP состоит из **5 этапов**: DOM → CSSOM → Render Tree → Layout → Paint/Composite
- Первые ~**14 КБ** HTML получаются в первом TCP-пакете — критически важны для быстрого старта
- CSS **блокирует рендеринг** — пока CSSOM не готов, первый пиксель не появится
- `<script>` без `async`/`defer` **блокирует парсинг** HTML
- **Preload Scanner** обходит HTML вперёд и досрочно загружает ресурсы, пока парсер заблокирован
- `async`/`defer` для скриптов убирают parser blocking
- `<link rel="preload">` для критических ассетов ускоряет их загрузку
- Анимируй `transform`/`opacity` — они не триггерят Layout и Paint

## Определения и термины

**Critical Rendering Path (CRP)** — последовательность этапов от получения первого байта HTML до первого пикселя на экране.

**FCP (First Contentful Paint)** — время, когда запрошенный контент (текст, изображения) впервые становится видимым на экране.

**TTI (Time to Interactive)** — метрика от первого запроса (DNS, TCP) до момента, когда страница становится по-настоящему интерактивной (реагирует на ввод менее чем за 50 мс).

**INP (Interaction to Next Paint)** — метрика, оценивающая, насколько стабильно и быстро страница реагирует на пользовательский ввод на протяжении всего времени её использования.

**LCP (Largest Contentful Paint)** — в предоставленных источниках **не раскрыто**.

**Render-blocking Resource** — ресурс (обычно CSS или синхронный JS), блокирующий первую отрисовку страницы.

**Parser-blocking Resource** — ресурс (`<script>` без `async`/`defer`), блокирующий построение DOM-дерева.

**Preload Scanner** — дополнительный лёгкий парсер, запускаемый браузером параллельно с main-thread парсером для досрочной загрузки ресурсов.

**`async`** — атрибут `<script>`: скрипт загружается параллельно с парсингом HTML, выполняется немедленно после загрузки (не блокирует парсинг, но может выполниться в любой момент).

**`defer`** — атрибут `<script>`: скрипт загружается параллельно с парсингом HTML, выполняется после завершения парсинга (не блокирует парсинг, порядок выполнения сохраняется).

**`<link rel="preload">`** — директива приоритизации ресурсов: браузер загружает указанный ресурс с высоким приоритетом, не дожидаясь того момента, когда парсер доберётся до него.

**TTFB (Time to First Byte)** — время между нажатием на ссылку и загрузкой первого байта контента на новой странице.

**TBT (Total Blocking Time)** — прокси-метрика для INP, рассчитывающая, как долго основной поток был заблокирован во время загрузки страницы.

**SSR (Server-Side Rendering)** — рендеринг приложения на сервере для отправки HTML вместо JavaScript клиенту.

**CSR (Client-Side Rendering)** — рендеринг приложения в браузере с использованием JavaScript для изменения DOM.

**Hydration (Гидратация)** — запуск клиентских скриптов для добавления состояния и интерактивности к HTML, отрендеренному на сервере. Предполагается, что DOM не изменяется.

**Prerendering** — запуск клиентского приложения во время сборки для захвата его начального состояния в виде статического HTML.

## Как устроено

### Пять этапов Critical Rendering Path

```
Байты HTML (сеть)
        │
        ▼
1. DOM Tree
   Парсинг HTML → токенизация → построение DOM-дерева
   Корень: <html>. Включает все видимые и невидимые узлы.
        │
        ▼  (параллельно)
2. CSSOM Tree
   Парсинг CSS → построение CSSOM.
   Включает User Agent stylesheet + авторские стили.
   Рекурсивно вычисляет финальные стили каждого узла.
        │
        ▼
3. Render Tree (Style)
   DOM + CSSOM → только видимые узлы с вычисленными стилями.
   display: none → исключается. visibility: hidden → включается.
        │
        ▼
4. Layout (Reflow)
   Обход Render Tree → вычисление точных размеров и координат
   каждого узла с учётом размеров viewport.
        │
        ▼
5. Paint & Compositing
   Геометрические блоки → пиксели.
   Разбивка на слои → растеризация → GPU → экран.
```

### Что блокирует CRP

**Parser-blocking (блокирует DOM):**
- `<script>` без `async`/`defer`
- JS может изменить структуру документа через `document.write()`

**Render-blocking (блокирует первую отрисовку):**
- Загрузка внешних CSS-файлов
- CSS также блокирует выполнение JS (JS может запрашивать стили)

### Preload Scanner в CRP

Когда main-thread заблокирован скриптом:
1. Браузер запускает параллельный **Preload Scanner**
2. Он пробегает HTML вперёд и досрочно инициирует загрузку CSS, шрифтов, изображений, других скриптов
3. К моменту разблокировки парсера — ресурсы уже загружены или загружаются

**Ограничение**: Preload Scanner видит только **статически объявленные** в HTML ресурсы — динамически созданные он пропустит.

### Первые 14 КБ

Браузер получает первую порцию данных (~14 КБ — размер первого TCP-пакета) и **немедленно начинает парсинг**. Критически важно разместить в первых 14 КБ:
- Необходимый HTML для первой отрисовки
- Критический CSS (inline или первый внешний stylesheet)

## Детали и нюансы

### Метрики производительности

**FCP (First Contentful Paint)** — когда пользователь впервые видит любой контент (текст, изображение). Зависит от скорости построения Render Tree и первого Layout/Paint.

**TTI (Time to Interactive)** — измеряет время от первого запроса (DNS lookup, TCP handshake) до момента, когда страница реагирует на ввод менее чем за 50 мс. Зависит от объёма и времени выполнения JavaScript.

**INP (Interaction to Next Paint)** — оценивает стабильность реакции на ввод на протяжении всей сессии. Длительный Layout или Paint ухудшает INP.

### async vs defer vs без атрибута

| Атрибут | Парсинг HTML | Загрузка скрипта | Выполнение скрипта |
|---------|-------------|-------------------|-------------------|
| нет | ⛔ Блокируется | Последовательно | Немедленно после загрузки |
| `async` | ✅ Продолжается | Параллельно | Немедленно после загрузки (порядок не гарантирован) |
| `defer` | ✅ Продолжается | Параллельно | После завершения парсинга HTML (порядок сохраняется) |

## SSR, CSR и стратегии рендеринга

### SSR (Server-Side Rendering)

```
1. Навигационный запрос отправляется на сервер
2. Сервер полностью генерирует HTML на основе данных и логики
3. Сервер отправляет HTML вместе с сериализованными зависимостями
4. Клиент загружает HTML (улучшенный FCP)
5. Клиент загружает JS, выполняет его и привязывает обработчики (Hydration)
6. UI становится интерактивным
```

**Плюсы**: быстрый FCP, контент виден без ожидания JS.
**Минусы**: увеличенный TTFB (сервер генерирует HTML на лету), тяжёлая гидратация на мобильных может блокировать интерактивность на минуты. Создаётся эффект «зловещей долины» (uncanny valley) — UI выглядит готовым, но не реагирует на ввод.

### CSR (Client-Side Rendering)

```
1. Сервер отправляет минимальный HTML с ссылкой на JS
2. Браузер загружает JS
3. Вся логика, получение данных и генерация DOM выполняются JS на клиенте
```

**Минусы**: большой объём JS увеличивает TBT и задерживает TTI. На мобильных — особенно медленно из-за парсинга/компиляции JS.

### Static Rendering (Статический рендеринг)

Генерирует HTML-файлы на этапе сборки (build time). Гарантирует сверхнизкий TTFB и кэшируемость на CDN.

### Streaming SSR

Использование потокового SSR (например, `renderToPipeableStream` в React вместо `renderToString`), чтобы отправлять HTML частями (chunks). Позволяет браузеру начать парсинг раньше, снижая TTFB.

### Trisomorphic Rendering

Связывает потоковый SSR для начальных загрузок и Service Worker для рендеринга HTML кэшированных компонентов на последующих навигациях (SPA-переходы).

### Partial Rehydration

Техника, при которой статичные части серверного HTML исключаются из клиентского JS-бандла, сводя JS-footprint к нулю.

## Сравнения и trade-offs

| Оптимизация | Что ускоряет | Trade-off |
|-------------|-------------|-----------|
| `async` на скриптах | Убирает parser blocking | Порядок выполнения не гарантирован |
| `defer` на скриптах | Убирает parser blocking, сохраняет порядок | Скрипты выполняются позже (после парсинга) |
| `<link rel="preload">` | Ускоряет загрузку критичных ресурсов | Лишние preload = конкуренция за пропускную способность |
| Inline Critical CSS | Убирает render-blocking CSS для первого кадра | Увеличивает размер HTML, нет кэширования |
| `will-change: transform` | Выносит элемент на GPU-слой, плавные анимации | Layer Explosion при злоупотреблении |
| `transform`/`opacity` для анимаций | Только Composite, без Layout/Paint | Ограниченный набор анимируемых свойств |
| SSR | Быстрый FCP, SEO-friendly | Медленный TTFB, uncanny valley при тяжёлой гидратации |
| CSR | Богатая интерактивность, простая архитектура | Медленный FCP и TTI, большой JS-бандл |
| Static Rendering | Минимальный TTFB, CDN-кэширование | Не подходит для динамического контента |
| Streaming SSR | Снижает TTFB, прогрессивный рендеринг | Сложность серверной инфраструктуры |

## Примеры

### Пример: render-blocking CSS

```html
<head>
  <!-- Блокирует рендеринг до полной загрузки -->
  <link rel="stylesheet" href="styles.css">

  <!-- НЕ блокирует рендеринг для не-screen медиа -->
  <link rel="stylesheet" href="print.css" media="print">

  <!-- Приоритизированная загрузка критического ресурса -->
  <link rel="preload" href="critical-font.woff2" as="font" crossorigin>
</head>
```

### Пример: async vs defer

```html
<!-- Блокирует парсинг: ПЛОХО для некритических скриптов -->
<script src="analytics.js"></script>

<!-- Загружается параллельно, выполняется немедленно: для независимых скриптов -->
<script src="analytics.js" async></script>

<!-- Загружается параллельно, выполняется после парсинга: для зависимых скриптов -->
<script src="app.js" defer></script>
```

### Пример: оптимизация анимаций

```css
/* ПЛОХО: триггерит Layout + Paint на каждом кадре */
.box-slow {
  transition: width 0.3s, left 0.3s;
}

/* ХОРОШО: только Compositing, GPU-анимация */
.box-fast {
  transition: transform 0.3s;
  will-change: transform;
}
```

### Схема полного пути от запроса до пикселя

```
1. DNS lookup
2. TCP handshake
3. TLS negotiation (если HTTPS)
4. HTTP request → первый байт (TTFB)
5. Получение первых ~14 КБ HTML
6. Начало парсинга HTML → DOM
   └── Параллельно: загрузка CSS → CSSOM
   └── Preload Scanner: досрочная загрузка других ресурсов
7. DOM + CSSOM готовы → Render Tree
8. Layout → координаты и размеры
9. Paint → пиксели
10. Composite → GPU → экран (FCP)
```

## Важно / подводные камни / best practices

- **Первые 14 КБ** — помести критический HTML и CSS в первые 14 КБ ответа
- **`<link rel="stylesheet">` в `<head>`** — CSSOM должен строиться параллельно с DOM
- **`<script>` в конце `<body>` или с `async`/`defer`** — не блокировать парсинг HTML
- **Inline Critical CSS** — для первого кадра; остальные стили грузи асинхронно
- **`<link rel="preload">`** — для шрифтов, hero-изображений, критических JS
- **Не используй `@import` в CSS** — создаёт последовательные (не параллельные) запросы
- **Минимизируй render-blocking ресурсы** — каждый внешний CSS-файл задерживает FCP
- **Избегай Forced Synchronous Layout** в JS-обработчиках (читай геометрию до изменения стилей)
- **Анимируй `transform`/`opacity`** — они не выходят за рамки Composite

Что такое Critical Rendering Path?::Последовательность этапов, которые браузер выполняет для преобразования байтов HTML/CSS/JS в пиксели на экране пользователя.

Перечисли 5 этапов Critical Rendering Path::1. Построение DOM. 2. Построение CSSOM. 3. Построение Render Tree (DOM + CSSOM). 4. Layout (вычисление геометрии). 5. Paint & Compositing (пиксели и GPU).

Что такое render-blocking resource?::Ресурс (обычно CSS или синхронный JS), блокирующий первую отрисовку страницы. Браузер не покажет ни одного пикселя, пока такой ресурс не загружен и не обработан.

Что такое parser-blocking resource?::`<script>` без `async`/`defer`, блокирующий построение DOM-дерева. Браузер останавливает парсинг HTML и ждёт загрузки и выполнения скрипта.

Почему `<script>` блокирует HTML-парсер?::JS может изменить структуру документа через `document.write()`. Парсер должен дождаться завершения скрипта.

Чем отличаются `async` и `defer`?::Оба загружают скрипт параллельно с парсингом HTML. `async` выполняет скрипт немедленно после загрузки (порядок не гарантирован). `defer` — после завершения парсинга HTML (порядок сохраняется).

Что такое FCP?::First Contentful Paint — время, когда любой контент (текст, изображение) впервые становится видимым на экране.

Что такое TTI?::Time to Interactive — время от первого запроса до момента, когда страница реагирует на ввод менее чем за 50 мс.

Что такое INP?::Interaction to Next Paint — метрика стабильности реакции страницы на пользовательский ввод на протяжении всей сессии.

Почему важны первые 14 КБ HTML?::Это размер первого TCP-пакета. Браузер начинает парсинг немедленно при получении первых данных. Критический HTML и CSS в первых 14 КБ ускоряют FCP.

Что делает `<link rel="preload">`?::Сигнализирует браузеру загрузить указанный ресурс с высоким приоритетом немедленно, не дожидаясь, пока парсер доберётся до него в HTML.

Как Preload Scanner помогает CRP?::Запускается параллельно с заблокированным основным парсером, пробегает HTML вперёд и инициирует досрочную загрузку CSS, шрифтов, изображений.

Какие CSS-свойства не выходят за рамки Composite (не триггерят Layout и Paint)?::transform и opacity.

Что такое TTFB?::Time to First Byte — время между навигационным запросом и получением первого байта ответа.

Что такое TBT?::Total Blocking Time — прокси-метрика для INP, рассчитывающая, как долго основной поток был заблокирован во время загрузки страницы.

Что такое SSR?::Server-Side Rendering — рендеринг приложения на сервере для отправки готового HTML клиенту. Улучшает FCP, но увеличивает TTFB и требует Hydration для интерактивности.

Что такое CSR?::Client-Side Rendering — рендеринг полностью в браузере через JavaScript. Весь DOM строится на клиенте. Большой JS-бандл задерживает FCP и TTI.

Что такое Hydration?::Процесс запуска клиентских скриптов для добавления интерактивности к HTML, отрендеренному на сервере. Предполагает, что DOM не изменяется.

В чём опасность тяжёлой гидратации (Rehydration)?::На мобильных устройствах блокировка интерактивности может занимать минуты. Создаётся «uncanny valley» — UI визуально готов, но не реагирует на ввод. Дублируется объём данных (HTML + сырые данные для JS).

Что такое Streaming SSR?::Потоковый SSR (renderToPipeableStream) отправляет HTML частями (chunks), позволяя браузеру начать парсинг раньше и снижая TTFB.
