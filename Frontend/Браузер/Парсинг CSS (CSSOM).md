## Суть и контекст

Парсинг CSS — процесс преобразования CSS-правил в CSSOM-дерево (CSS Object Model). Происходит **параллельно** с построением DOM-дерева, но является **блокирующим для рендеринга**: браузер не отрисует ни одного пикселя до тех пор, пока не построит оба дерева — DOM и CSSOM. Это второй ключевой этап [[Critical Rendering Path|критического пути рендеринга]].

## Ключевые идеи

- CSS имеет **строгую контекстно-свободную грамматику** — в отличие от HTML, его можно парсить стандартными алгоритмами
- Парсинг CSS не блокирует парсинг HTML, но **блокирует рендеринг**
- CSS **блокирует выполнение JavaScript** (т.к. JS может запрашивать стили)
- Из CSS-правил строится CSSOM — независимая от DOM древовидная структура
- CSSOM включает как стили браузера по умолчанию (User Agent), так и авторские стили
- DOM + CSSOM → **Render Tree** (дерево рендеринга)
- Браузеры хэшируют правила по ключам (ID, класс, тег) для быстрого поиска
- **Специфичность** и **каскад** определяют, какое правило побеждает при конфликте

## Определения и термины

**CSSOM (CSS Object Model)** — независимая от DOM древовидная структура данных, представляющая CSS-правила. Содержит узлы с отношениями «родитель-потомок-брат», основанными на CSS-селекторах.

**Контекстно-свободная грамматика (Context-Free Grammar)** — тип грамматики, для которой существуют стандартные алгоритмы парсинга (в отличие от HTML).

**Лексер (Tokenizer)** — часть парсера CSS, разбивающая CSS-код на токены (идентификаторы, фигурные скобки, значения).

**Парсер** — часть, применяющая синтаксические правила и преобразующая токены в объекты (объект StyleSheet с объектами правил).

**StyleSheet** — результат парсинга CSS; содержит объекты правил (селекторы и объявления).

**Render Tree** — структура, получаемая объединением DOM и CSSOM. Содержит только видимые узлы с вычисленными стилями (называется **Frame Tree** в Gecko/Firefox).

**RenderObject** — прямоугольный блок в дереве рендеринга с визуальными атрибутами (цвета, размеры), расположенный в правильном порядке отрисовки.

**Computed Styles (вычисленные стили)** — финальные стили, вычисленные для каждого DOM-узла с учётом каскада, специфичности, наследования и стилей по умолчанию.

**Каскад (Cascade)** — механизм разрешения конфликтов между несколькими CSS-правилами, претендующими на один элемент.

**Специфичность (Specificity)** — числовой вес CSS-селектора, определяющий его приоритет при конфликте с другими правилами одинакового уровня каскада.

**Render-blocking** — блокировка первой отрисовки страницы до завершения загрузки и парсинга CSS.

## Как устроено

### Парсинг CSS

CSS имеет строгую контекстно-свободную грамматику, которую браузеры парсят по-разному:

- **WebKit**: использует инструменты **Flex** (для создания лексера) и **Bison** (генератор парсера «снизу вверх»)
- **Firefox (Gecko)**: написанный вручную парсер «сверху вниз»

Процесс:
1. Лексер разбивает CSS-код на токены
2. Парсер применяет синтаксические правила → объекты
3. Результат: объект **StyleSheet**, содержащий объекты правил (селекторы + объявления)
4. Из правил строится **CSSOM** — дерево узлов

### Блокировка рендеринга

```
Парсинг HTML  ────────────────────────────────────► DOM готов
                                                         │
Парсинг CSS   ──────────────────► CSSOM готов            │
                                       │                 │
                                       └─── Render Tree ─┤
                                                         │
                                                   Layout/Paint
```

- Загрузка внешних CSS **не блокирует парсинг HTML**
- Загрузка внешних CSS **блокирует рендеринг всей страницы**
- Загрузка CSS **блокирует выполнение JS**:
  - **Firefox**: блокирует все скрипты до загрузки таблицы стилей
  - **WebKit (Blink/Chrome)**: блокирует выполнение скрипта только если он пытается получить доступ к стилевым свойствам, которые могут зависеть от незагруженного CSS

### Вычисление стилей (Rule Matching)

Перебор всех правил для каждого элемента был бы слишком медленным. Браузеры (WebKit и Firefox) **хэшируют правила по ключам**:
- По **ID** — в карту ID
- По **имени класса** — в карту классов
- По **тегу** — в карту тегов

Пример: правило `div.error` попадёт в карту классов под ключом `error`. При вычислении стилей для `<p class="error">` браузер быстро найдёт это правило, но отбросит его — тег `p` ≠ `div`. Эта оптимизация **отсеивает более 95% правил** без необходимости их проверки.

### Каскад (Cascade)

Порядок приоритетов (от низшего к высшему):

| # | Источник | Тип |
|---|----------|-----|
| 1 | Браузер (User Agent) | Обычные объявления |
| 2 | Пользователь (User) | Обычные объявления |
| 3 | Автор (разработчик) | Обычные объявления |
| 4 | Автор (разработчик) | `!important` |
| 5 | Пользователь | `!important` |

### Специфичность (Specificity)

Вычисляется как 4 числа `a, b, c, d` в системе счисления с большим основанием:

| Компонент | Значение |
|-----------|----------|
| `a` | 1 если inline style, иначе 0 |
| `b` | Количество ID-селекторов (`#id`) |
| `c` | Количество классов, атрибутов, псевдоклассов (`.class`, `[type]`, `:hover`) |
| `d` | Количество тегов и псевдоэлементов (`div`, `::before`) |

Примеры:

| Селектор | a | b | c | d |
|----------|---|---|---|---|
| `ul li` | 0 | 0 | 0 | 2 |
| `ul ol+li` | 0 | 0 | 0 | 3 |
| `#myId` | 0 | 1 | 0 | 0 |
| `style="color: red"` | 1 | 0 | 0 | 0 |

При равной специфичности **побеждает правило, объявленное последним** в коде.

### Построение Render Tree

После построения DOM и CSSOM браузер объединяет их в **Render Tree**:

**Что включается:**
- Только видимые узлы
- `visibility: hidden` — включается (занимает место)

**Что исключается:**
- Тег `<head>` и его содержимое
- Элементы с `display: none`

**Отношение не 1:1** (один DOM-узел → несколько render-узлов):
- `<select>` → 3 объекта рендеринга: область отображения, выпадающий список, кнопка
- Перенос текста на новую строку → каждая строка = отдельный визуальный блок
- Смешанный контент (inline + block) → браузер создаёт **анонимные блочные рендереры**

## Детали и нюансы

- CSSOM строится полностью, прежде чем браузер может начать вычисление стилей — нельзя частично использовать CSSOM
- Загрузка CSS критична: если CSS приходит медленно, рендеринг заблокирован, даже если DOM уже готов
- Правило `@import` в CSS ещё хуже `<link>` — оно последовательно, а не параллельно
- Inline-стили (`style=""`) имеют наивысшую специфичность (a=1) среди авторских стилей
- `!important` от пользователя имеет наивысший приоритет вообще (защита доступности)

## Сравнения и trade-offs

| | `display: none` | `visibility: hidden` |
|---|---|---|
| В Render Tree | Нет | Да |
| Занимает место | Нет | Да |
| Триггерит Layout | Да (при изменении) | Нет (только Repaint) |

| Движок | CSS-парсер | Метод |
|--------|-----------|-------|
| WebKit / Blink | Flex + Bison | Сгенерированный, «снизу вверх» |
| Firefox (Gecko) | Написан вручную | «Сверху вниз» |

## Примеры

### Пример: CSS-правила → CSSOM

```css
body { font-size: 16px }
p    { font-weight: bold }
span { color: red }
p span { display: none }
img  { float: right }
```

→ CSSOM: дерево, где `p span` является потомком `p`, который является потомком `body`, унаследовавшего `font-size: 16px`.

### Пример: вычисление специфичности

```css
/* Специфичность: 0,0,0,2 */
ul li { color: red }

/* Специфичность: 0,1,0,0 */
#main { color: blue }

/* Специфичность: 1,0,0,0 */
/* <p style="color: green"> */
```

Для элемента `<li id="main">` победит `#main` (0,1,0,0 > 0,0,0,2).

### Пример: хэширование правил

```css
div.error { color: red }   /* → карта классов, ключ: "error" */
#header   { margin: 0 }    /* → карта ID, ключ: "header" */
p         { font-size: 1em }/* → карта тегов, ключ: "p" */
```

## Важно / подводные камни / best practices

- **Помещай `<link rel="stylesheet">` в `<head>`** — чтобы CSSOM был готов до JS
- **Не используй `@import` в CSS** — создаёт последовательные запросы, блокируя рендеринг дольше
- **Избегай глубоко вложенных селекторов** — увеличивают время сопоставления правил
- **CSS блокирует JS**: если JS нужен до загрузки стилей — убедись, что он не зависит от стилей
- **`display: none` убирает элемент из Render Tree** — изменение с/на `none` всегда вызывает Reflow
- **Специфичность `!important`** трудно переопределить — избегай злоупотреблений

## Карточки для повторения

#flashcards/browser/cssom

Что такое CSSOM?::CSS Object Model — независимая от DOM древовидная структура данных, представляющая CSS-правила. Браузер строит её из результата парсинга CSS.

Блокирует ли CSS парсинг HTML?::Нет. Парсинг CSS происходит параллельно с парсингом HTML. Но CSS блокирует **рендеринг** (первую отрисовку).

Блокирует ли CSS выполнение JavaScript?::Да. Firefox блокирует все скрипты, пока загружается CSS. WebKit/Blink — только если скрипт обращается к стилевым свойствам.

Почему CSS блокирует выполнение JavaScript?::JavaScript часто запрашивает стили элементов (offsetHeight, getComputedStyle). Если стиль ещё не применён — скрипт получит неверные данные.

Что такое Render Tree?::Структура, получаемая объединением DOM и CSSOM. Содержит только видимые узлы с вычисленными стилями. В WebKit называется Render Tree, в Gecko — Frame Tree.

Попадает ли `display: none` в Render Tree?::Нет. Элементы с `display: none`, а также тег `<head>` — исключаются из Render Tree.

Попадает ли `visibility: hidden` в Render Tree?::Да. Элемент невидим, но занимает геометрическое пространство — включается в Render Tree.

Как браузер оптимизирует сопоставление CSS-правил?::Хэширует правила по ключам: карта ID, карта классов, карта тегов. Это отсеивает более 95% правил без проверки.

Перечисли порядок каскада CSS от низшего к высшему приоритету::1. User Agent (браузер). 2. User обычные. 3. Author обычные. 4. Author !important. 5. User !important.

Как вычисляется специфичность CSS-селектора?::Четыре числа a,b,c,d: a=1 для inline style; b=кол-во ID; c=кол-во классов/атрибутов/псевдоклассов; d=кол-во тегов/псевдоэлементов.

Какова специфичность `#myId`?::0, 1, 0, 0

Какова специфичность `ul li`?::0, 0, 0, 2

Всегда ли соотношение DOM-узла к Render-узлу равно 1:1?::Нет. Один DOM-узел может давать несколько render-узлов (select → 3 объекта; перенос текста → несколько строк). Браузер также создаёт анонимные блочные рендереры для смешанного контента.
