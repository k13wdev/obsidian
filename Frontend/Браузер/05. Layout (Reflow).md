## Суть и контекст

**Layout** (в Chrome/Blink и Safari/WebKit) или **Reflow** (в Firefox/Gecko) — процесс, во время которого браузер вычисляет геометрическую информацию для каждого элемента в дереве рендеринга: точные размеры и координаты расположения на странице.

Это четвёртый этап [[01. Critical Rendering Path|критического пути рендеринга]], следующий после построения [[03. Парсинг CSS (CSSOM)|Render Tree]]. Напрямую влияет на производительность страницы и метрику INP (Interaction to Next Paint).

## Ключевые идеи

- HTML использует **потоковую модель макета** — большинство элементов можно обработать за один проход слева направо и сверху вниз
- Браузер использует механизм **dirty bit** — не пересчитывает весь макет при мелких изменениях
- Существует два вида: **глобальный** (весь документ) и **постепенный** (только dirty-элементы) layout
- Чтение геометрических свойств через JS **принудительно запускает синхронный layout** (Forced Synchronous Layout)
- **Layout Thrashing** — многократное принудительное синхронное перестроение макета в цикле — главный убийца производительности

## Определения и термины

**Layout / Reflow** — процесс вычисления точных размеров и координат элементов дерева рендеринга.

**Потоковая модель макета (Flow-based Layout)** — модель, в которой элементы обрабатываются слева направо и сверху вниз. Элементы, стоящие дальше в потоке, обычно не влияют на геометрию предшествующих.

**Dirty Bit** — флаг на элементе, означающий «мне нужен перерасчёт макета». Позволяет не пересчитывать весь документ.

**Dirty** — флаг: самому рендереру нужен layout.

**Children are dirty** — флаг: самому рендереру layout не нужен, но как минимум одному из его потомков — нужен.

**Global Layout (Глобальный макет)** — пересчёт макета для всего дерева рендеринга. Происходит синхронно при глобальных изменениях.

**Incremental Layout (Постепенный макет)** — пересчёт только dirty-элементов. Запускается асинхронно (по таймеру, пакетом).

**Forced Synchronous Layout (Принудительная синхронная компоновка)** — ситуация, когда JS изменяет стиль элемента, а затем немедленно запрашивает его геометрию. Браузер вынужден синхронно применить стили и запустить layout прямо во время выполнения JS.

**Layout Thrashing** — многократное возникновение Forced Synchronous Layout в быстрой последовательности (чаще всего в цикле `for`).

**INP (Interaction to Next Paint)** — метрика, оценивающая время реакции страницы на пользовательский ввод. Длительный layout напрямую ухудшает INP.

## Как устроено

### Алгоритм вычисления геометрии

Процесс рекурсивный:

1. **Родительский элемент** определяет свою ширину (с учётом ширины контейнера, `width`, полей и границ)
2. Родитель **перебирает детей** и для каждого:
   - Устанавливает координаты `x` и `y`
   - Вызывает layout дочернего элемента (если необходимо) для вычисления высоты
3. Родитель использует сумму высот детей + отступы/поля → устанавливает **свою высоту**
4. Родитель **снимает флаг dirty** (устанавливает `false`)

Система координат: относительно корневого кадра (`top`, `left`).

### Алгоритм вычисления ширины элемента (WebKit, RenderBox::calcWidth)

1. Вычисляется **доступная ширина контейнера** (`contentWidth`) — без учёта borders и scrollbar
2. Вычисляется ширина элемента: абсолютное значение или процент от ширины контейнера
3. Добавляются горизонтальные `border` и `padding` → «предпочтительная ширина»
4. Если предпочтительная ширина > `max-width` → применяется `max-width`
5. Если предпочтительная ширина < `min-width` (наименьшая неразрывная единица текста) → применяется `min-width`
6. При разрыве строки рендерер уведомляет родителя о необходимости создать дополнительный рендерер

### Внутренние объекты

- **RenderView** (WebKit) / **ViewPortFrame** (Firefox) — корневой рендерер, соответствующий `<html>`, размеры равны viewport
- **nsHTMLReflowState** (Firefox) — входной объект для layout, включает ширину родителей
- **nsHTMLReflowMetrics** (Firefox) — выходной объект, содержит вычисленную высоту

### Локальный layout

Изменение позиции рендерера (без изменения размера) или вставка текста вызывает **локальный layout** — размеры берутся из кэша, изменяется только субдерево. Без этого каждое нажатие клавиши вызывало бы полную перекомпоновку.

### Механизм Dirty Bit

```
Элемент изменился или добавлен
         │
         ▼
Помечает себя как "dirty"
Помечает детей как "dirty"
         │
         ▼
Браузер видит dirty-флаги
         │
         ▼
Запускает incremental layout только для dirty-элементов
```

Два флага:
- `dirty` — самому рендереру нужен layout
- `children are dirty` — хотя бы один потомок нуждается в layout

### Global vs Incremental Layout

**Глобальный layout** запускается синхронно при:
- Изменении размера окна браузера (`resize`)
- Изменении глобальных стилей (например, размер шрифта, влияющий на все элементы)

**Постепенный (Incremental) layout** запускается асинхронно:
- Браузер ставит команды reflow в очередь
- Выполняет их **пакетом по таймеру** (batching)
- Пример: загрузка нового контента из сети, добавление новых узлов в DOM

## Детали и нюансы

### Что триггерит Layout (Reflow)

Любое изменение **геометрических CSS-свойств** требует нового layout: `width`, `height`, `left`, `top`, `margin`, `padding`, `border`, `font-size`, etc.

**JavaScript-API, запускающие принудительный синхронный layout:**

| Категория | API |
|-----------|-----|
| Метрики блока | `offsetWidth`, `offsetHeight`, `offsetLeft`, `offsetTop` |
| Размеры клиента | `clientWidth`, `clientHeight` |
| Методы позиционирования | `getBoundingClientRect()`, `getClientRects()` |
| Прокрутка | `scrollWidth`, `scrollHeight`, `scrollTop` |
| Методы | `focus()`, `scrollIntoView()` |
| SVG | `computeCTM()`, `getBBox()`, `getComputedTextLength()` |
| Mouse Events | `mouseEvt.layerX`, `mouseEvt.layerY`, `mouseEvt.offsetX`, `mouseEvt.offsetY` |
| Window | `window.scrollX`, `window.scrollY`, `window.innerHeight`, `window.innerWidth` |
| Фокус и выделение | `elem.focus()`, `inputElem.select()`, `textareaElem.select()` |
| Scroll операции | `elem.scrollBy()`, `elem.scrollTo()`, `elem.scrollIntoView()`, `elem.scrollIntoViewIfNeeded()` |
| Стили | `window.getComputedStyle()` в определённых условиях |

**Условия, при которых `window.getComputedStyle()` форсирует layout:**
- Элемент находится в shadow tree
- Имеются viewport-зависимые медиа-запросы (`min-width`, `aspect-ratio`, `resolution`)
- Запрошено геометрическое свойство: `height`, `width`, `top`, `bottom`, `margin`, `padding`, `transform`, `grid`

### Стандартный конвейер кадра

```
JavaScript → Style Calculation → Layout → Paint → Composite
```

При запуске JS значения layout из **предыдущего кадра** известны и закэшированы. Forced Synchronous Layout ломает этот конвейер.

### Forced Synchronous Layout

```javascript
// ПЛОХО: изменяем стиль, затем сразу читаем геометрию
box.classList.add('super-big');
let currentHeight = box.offsetHeight; // ← браузер вынужден синхронно запустить layout!
```

Что происходит: браузер приостанавливает выполнение JS, применяет стили, запускает layout, возвращает значение.

```javascript
// ХОРОШО: сначала читаем (из кэша предыдущего кадра), потом пишем
let currentHeight = box.offsetHeight; // ← читаем старое значение из кэша
box.classList.add('super-big');       // ← пишем изменения
```

### Layout Thrashing

Возникает при чтении и записи геометрических свойств **в одном цикле**:

```javascript
// ПЛОХО: Layout Thrashing
function resizeParagraphs(paragraphs, box) {
  for (let i = 0; i < paragraphs.length; i++) {
    // Каждая итерация: запись (стиль) → чтение (offsetWidth) → принудительный layout
    paragraphs[i].style.width = box.offsetWidth + 'px';
  }
}
```

На каждой итерации: читается `offsetWidth` → меняется ширина → на следующей итерации браузер снова запрашивает `offsetWidth` → обнаруживает изменение стилей → **запускает полный layout снова**. Это происходит на каждой итерации.

```javascript
// ХОРОШО: вынести чтение за пределы цикла
function resizeParagraphs(paragraphs, box) {
  const width = box.offsetWidth; // ← читаем один раз до цикла

  for (let i = 0; i < paragraphs.length; i++) {
    paragraphs[i].style.width = width + 'px'; // ← только запись в цикле
  }
  // layout будет рассчитан асинхронно один раз после скрипта
}
```

## Сравнения и trade-offs

| | Global Layout | Incremental Layout |
|---|---|---|
| Охват | Всё дерево рендеринга | Только dirty-элементы |
| Запуск | Синхронно | Асинхронно (по таймеру, пакетом) |
| Триггер | Resize окна, глобальные стили | Добавление DOM-узлов, локальные изменения |

| | Forced Synchronous Layout | Обычный (асинхронный) Layout |
|---|---|---|
| Запуск | В момент запроса геометрии в JS | Браузер группирует и запускает пакетом |
| Влияние | Блокирует JS-поток | Не блокирует JS |
| Опасность | Да (особенно в циклах) | Нет |

## Примеры

### Пример: алгоритм вычисления геометрии

```
<div style="width: 800px">       ← родитель: ширина 800px
  <p style="width: 50%">         ← дочерний: ширина 400px (50% от 800)
    Hello, world!                ← текстовый узел: высота определяется шрифтом
  </p>
</div>
                                 ← родитель устанавливает высоту = высота <p>
```

### Пример: определение Forced Synchronous Layout в DevTools

В Chrome DevTools → вкладка **Performance** → предупреждения **"Forced reflow"** (Принудительная перекомпоновка).

### Пример: использование FastDOM для предотвращения Layout Thrashing

```javascript
// FastDOM группирует чтения и записи
fastdom.measure(() => {
  const width = box.offsetWidth; // читаем
  fastdom.mutate(() => {
    paragraphs.forEach(p => p.style.width = width + 'px'); // пишем
  });
});
```

## Важно / подводные камни / best practices

- **Правило «сначала читай, потом пиши»**: всегда читай геометрические свойства до изменения стилей
- **Выноси чтение за пределы циклов**: `offsetWidth`, `getBoundingClientRect()` и подобные — один раз до цикла
- **Используй `requestAnimationFrame`** для батчинга DOM-изменений — они будут применены перед следующим кадром
- **Используй библиотеки типа FastDOM** для автоматического группирования чтений и записей
- **Ищи "Forced reflow"** в Chrome DevTools Performance — это красный флаг производительности
- **`will-change: transform`** выносит элемент на compositing layer, анимация которого не требует Layout
- **Изменение `display: none`** всегда вызывает глобальный или инкрементальный layout — используй с осторожностью
- **Анимируй `transform` и `opacity`** вместо `left/top/width/height` — они не триггерят Layout

Как называется Layout в Firefox?::Reflow. В Chrome/Blink и Safari/WebKit — Layout.

Что такое Dirty Bit в контексте Layout?::Флаг на элементе, означающий «мне нужен перерасчёт макета». Позволяет не пересчитывать весь документ при мелких изменениях.

Назови 4 JS-API, которые триггерят Forced Synchronous Layout::offsetWidth, offsetHeight, getBoundingClientRect(), scrollTop (и другие: offsetLeft, offsetTop, clientWidth, clientHeight, scrollWidth, scrollHeight, focus(), scrollIntoView()).

Как исправить Layout Thrashing в цикле?::Вынести все чтения геометрических свойств за пределы цикла (одно чтение до цикла), оставив в цикле только записи.

Какой принцип предотвращает Forced Synchronous Layout?::«Сначала читай, потом пиши» — читай геометрические свойства до изменения стилей, а не после.

Как HTML-процессор вычисляет высоту родительского элемента?::Использует сумму высот дочерних элементов плюс отступы и поля (margins/padding).
