## Суть и контекст

Парсинг HTML — это процесс преобразования HTML-кода, полученного по сети, во внутреннее представление браузера (DOM-дерево), с которым можно взаимодействовать. Это первый этап [[01. Critical Rendering Path|критического пути рендеринга]].

В отличие от XML, грамматика HTML не является контекстно-свободной из-за своей «прощающей» природы (возможности опускать закрывающие теги и т.д.), поэтому для него нельзя использовать стандартные парсеры. Вместо этого браузеры используют специальные алгоритмы, детально описанные в спецификации HTML5.

## Ключевые идеи

- Парсинг HTML состоит из двух этапов: **токенизация** → **построение дерева**
- Алгоритм реализован как **конечный автомат** (state machine)
- HTML-парсер обладает высокой **устойчивостью к ошибкам** — он никогда не останавливается с ошибкой синтаксиса
- `<script>` без атрибутов **блокирует парсер** (parser blocking)
- **Preload Scanner** запускается параллельно, чтобы досрочно загрузить ресурсы
- Парсинг HTML является **реентерабельным**: JS может динамически изменить входной поток через `document.write()`

## Определения и термины

**Токен** — смысловая единица HTML: начальный тег, конечный тег, имя атрибута, значение атрибута, текстовый символ.

**Токенизация (Tokenization)** — этап лексического анализа, на котором входящий поток символов разбивается на токены.

**Конечный автомат (State Machine)** — модель алгоритма, где каждое состояние поглощает символы из потока и обновляет следующее состояние в зависимости от прочитанного.

**Режим вставки (Insertion Mode)** — состояние конечного автомата построения дерева (например: «до HTML», «в голове», «в теле»).

**Стек открытых элементов (Stack of Open Elements)** — структура данных для контроля правильной вложенности тегов и обработки незакрытых элементов.

**DOM (Document Object Model)** — внутреннее иерархическое представление HTML-документа в браузере.

**Parser Blocking** — приостановка построения DOM при встрече тега `<script>`.

**Preload Scanner** — дополнительный лёгкий парсер, который пробегает HTML вперёд и инициирует загрузку ресурсов, пока основной парсер заблокирован.

**Speculative Parsing** — техника досрочной (спекулятивной) инициализации сетевых запросов на основе ресурсов, найденных сканером предварительной загрузки.

**Speculative Mock Elements** — фиктивные элементы, создаваемые спекулятивным парсером исключительно для извлечения URL — они не изменяют настоящее DOM-дерево.

## Как устроено

### Этап 1: Токенизация

Алгоритм токенизации работает как конечный автомат. Пример для строки `<html><body>Hello world</body></html>`:

1. Парсер в состоянии **Data state** (состояние данных).
2. Встречает `<` → переходит в **Tag open state** (состояние открытого тега).
3. Читает `h`, `t`, `m`, `l` → создаёт токен начального тега в **Tag name state**.
4. Встречает `>` → завершает токен `<html>`, выдаёт его, возвращается в **Data state**.
5. Аналогично создаётся токен `<body>`.
6. Строка `Hello world` → текстовые токены для каждого символа по отдельности до встречи `<` от `</body>`.

### Этап 2: Построение DOM-дерева

Браузер берёт токены от токенизатора и создаёт соответствующие элементы DOM, прикрепляя их к объекту `Document`. Используется стек открытых элементов для контроля вложенности.

Пример:
- Токен `html` → режим «до html» → создаётся `HTMLHtmlElement` (корень дерева)
- Если парсер получает токен `body` без предшествующего `head` → неявно создаётся `HTMLHeadElement`, добавляется в дерево, затем создаётся `HTMLBodyElement`
- Символьные токены `Hello world` → создаётся текстовый узел `Text`

### Parser Blocking и выполнение `<script>`

Когда HTML-парсер встречает тег `<script>`:

1. Парсер **приостанавливает** построение DOM (parser blocking)
2. Браузер ждёт загрузки скрипта из сети (если внешний)
3. Парсит, компилирует и **выполняет** скрипт
4. Только потом **возобновляет** построение DOM-дерева

**Причина**: JS может динамически изменять HTML через `document.write()`, добавляя новые токены во входной поток.

### Preload Scanner и Speculative Parsing

Пока основной поток заблокирован:
1. Запускается **лёгкий параллельный парсер** (Preload Scanner)
2. Он пробегает по оставшемуся HTML вперёд
3. Находит внешние ресурсы: `<img>`, `<link>`, `<script>`
4. Браузер спекулятивно инициирует сетевые запросы в фоне
5. К моменту, когда основной парсер доходит до этих тегов — ресурсы уже загружены или загружаются

**Важно**: Speculative Parser создаёт **фиктивные mock-элементы** только для извлечения URL — настоящее DOM-дерево он не изменяет.

## Детали и нюансы

### Обработка ошибок (Error Tolerance)

Браузеры **никогда не останавливаются** с ошибкой синтаксиса при чтении HTML. Примеры автоматического исправления ошибок:

| Ошибка | Поведение браузера |
|--------|-------------------|
| `</br>` вместо `<br>` | Воспринимается как валидный `<br>` (для совместимости) |
| Таблица внутри таблицы вне `<td>` | Внутренняя таблица извлекается наружу, становится сестринским элементом |
| `<form>` внутри `<form>` | Вторая форма игнорируется парсером |
| Тысячи вложенных одинаковых тегов `<b>` | Ограничение вложенности ~20 тегами, остальные игнорируются |

### Реентерабельность парсера

HTML-парсинг является реентерабельным процессом: выполнение JS может добавить новые токены во входной поток (через `document.write()`), поэтому парсер обязан дождаться завершения скрипта перед продолжением.

### CDATA вне иностранного контента

Если парсер встречает конструкцию `<![CDATA[...]]>` **вне** SVG или MathML, возникает ошибка `cdata-in-html-content`. Парсер интерпретирует секцию CDATA как обычный HTML-комментарий.

### Символ NULL (U+0000)

При обнаружении символа `U+0000 NULL` в потоке возникает ошибка `unexpected-null-character`. Символ заменяется на `U+FFFD REPLACEMENT CHARACTER` (�).

### Float и Absolute элементы

Элементы `float` и `position: absolute` вынимаются из обычного потока при построении Render Tree и привязываются в другом месте дерева. На их исходном месте создаётся «рамка-заполнитель» (placeholder frame).

## Сравнения и trade-offs

| | HTML | XML | CSS |
|---|---|---|---|
| Тип грамматики | Не контекстно-свободная | Контекстно-свободная | Контекстно-свободная |
| Стандартные парсеры | Нельзя | Можно | Можно |
| Обработка ошибок | Прощающая (error tolerance) | Строгая (fatal error) | Прощающая |
| Алгоритм | Конечный автомат по спецификации HTML5 | Стандартные алгоритмы | Flex/Bison или ручной |

## Примеры

### Пример токенизации

```
Вход: <html><body>Hello world</body></html>

Состояния конечного автомата:
Data state → увидел '<' → Tag open state
Tag open state → читает 'h','t','m','l' → Tag name state
Tag name state → увидел '>' → выдаёт токен <html>, возврат в Data state
Data state → увидел '<' → Tag open state
...
Data state → 'H','e','l','l','o',' ','w','o','r','l','d' → текстовые токены
```

### Пример построения дерева

```
Токены:        <html>  <body>  "Hello world"  </body>  </html>
DOM-дерево:
Document
  └── HTMLHtmlElement
        └── HTMLBodyElement
              └── Text("Hello world")
```

## Важно / подводные камни / best practices

- **`<script>` без `async`/`defer`** — блокирует парсер: браузер стоит и ждёт загрузки и выполнения скрипта
- **Размещай `<script>` в конце `<body>`** или используй `async`/`defer`, чтобы не блокировать парсинг
- **`document.write()` в скриптах** делает парсинг реентерабельным и разрушает оптимизации
- **Preload Scanner** работает только если ресурсы объявлены статически в HTML — динамически созданные ресурсы он не найдёт
- **Невалидный HTML** будет исправлен браузером по-разному в разных браузерах — всегда используй валидную разметку
