# 08. Docker Networking

## Суть и контекст

Docker предоставляет подключаемую сетевую подсистему на основе драйверов (libnetwork). Каждый контейнер получает изолированный сетевой стек (через Linux network namespaces), а соединение между контейнерами и внешним миром организуется через один из четырёх основных драйверов: bridge, host, overlay, macvlan. Встроенный DNS-сервер (127.0.0.11) обеспечивает service discovery по именам контейнеров внутри пользовательских сетей.

---

## Ключевые идеи

- Bridge network — сеть по умолчанию: виртуальный мост docker0, пара veth-интерфейсов, NAT через iptables
- Host network — контейнер разделяет сетевой стек хоста; нет NAT и проброса портов
- Overlay network — L2-сеть поверх L3 через VXLAN; для мультихостовых кластеров (Swarm)
- MACVLAN — контейнер получает реальный MAC и IP в физической сети; требует promiscuous mode
- DNS работает только в user-defined сетях (не в дефолтном docker0)
- Port mapping (-p) создаёт правила iptables DNAT и запускает docker-proxy для перехвата трафика

---

## Определения и термины

**Bridge network (Мост)** — сетевой режим по умолчанию (основанный на 802.1d bridge), представляющий собой внутреннюю виртуальную подсеть хоста, к которой подключаются интерфейсы контейнеров для обеспечения их изоляции и связи друг с другом и с хостом. На Linux реализован в виде виртуального устройства docker0 (IP: 172.17.42.1/16), на Windows — с использованием драйвера nat.

**Host network (Режим хоста)** — сетевой режим, при котором контейнер использует пространство сетевых имён хоста, разделяя с ним IP-адрес и не требуя механизмов NAT и перенаправления портов.

**Overlay network (Оверлейная сеть)** — плоская сеть второго уровня (L2), охватывающая несколько физических хостов (соединённых сетями L3) и позволяющая распределённым контейнерам напрямую обмениваться данными. Инкапсулирует пакеты с помощью технологии VXLAN.

**MACVLAN** — сетевой драйвер, позволяющий подключать контейнеры напрямую к существующим физическим сетям и VLAN, предоставляя каждому контейнеру собственный реальный MAC- и IP-адрес без использования мостов и NAT. На Windows аналогом является драйвер transparent.

**Closed container / None network** — режим (--net=none), при котором сетевая среда контейнера полностью изолирована и контейнеру предоставляется исключительно интерфейс обратной петли (loopback).

**Port mapping (Перенаправление портов)** — механизм трансляции сетевых адресов и портов, связывающий порт на сетевом стеке хост-машины с интерфейсом и портом контейнера для пропуска входящего трафика.

**Встроенный DNS-сервер (Embedded DNS)** — DNS-сервер Docker, работающий по адресу 127.0.0.11. Обеспечивает разрешение имён контейнеров в IP-адреса внутри пользовательских (user-defined) сетей.

## Как устроено

### Bridge Network: архитектура

Внутренние компоненты: демон Docker, ядро Linux, виртуальный коммутатор docker0, пара интерфейсов veth, docker-proxy, подсистема iptables.

Алгоритм запуска контейнера в bridge-сети:

1. Демон Docker создаёт виртуальный мост docker0 (IP: 172.17.42.1/16) в корневом пространстве имён хоста.
2. При запуске контейнера создаётся пара виртуальных Ethernet-интерфейсов (veth). Один конец (vethf3e8733) привязывается к мосту docker0, другой помещается внутрь изолированного пространства имён контейнера и переименовывается в eth0.
3. Контейнеру присваивается приватный IP-адрес из подсети моста.
4. Весь исходящий трафик маршрутизируется через шлюз docker0 во внешнюю сеть хоста с использованием NAT (IP masquerading).

```
Physical network interface
         │
  ОС хоста (сетевой стек)
         │
     docker0 (172.17.42.1/16)
    ┌─────┴─────┐
  veth1       veth2
    │             │
Container 1   Container 2
  eth0 + lo    eth0 + lo
```

### Port Mapping: механизм (-p 8080:80)

1. Docker создаёт правила в iptables для Destination NAT (DNAT).
2. Запускается процесс docker-proxy для перехвата трафика на порту хоста (8080) и его проксирования на IP-адрес и порт (80) контейнера.
3. В режиме host этот этап пропускается — контейнер напрямую занимает порт хоста.

### Встроенный DNS и Service Discovery

1. При запуске контейнера с флагом --name или в составе сервиса демон Docker регистрирует его имя и IP-адрес во встроенном DNS-сервере (127.0.0.11).
2. Контейнеры, подключённые к пользовательским (user-defined) сетям, используют этот DNS-сервер для разрешения имён других контейнеров.
3. DNS-разрешение изолировано в рамках одной сети (network-scoped). Контейнеры в разных сетях не могут разрешать имена друг друга.

### Overlay Network (Swarm)

- Инкапсулирует пакеты через VXLAN поверх физической L3-сети.
- Флаг `-o encrypted` шифрует пользовательский трафик (data plane). По умолчанию шифруется только управляющий трафик (control plane).
- Ingress load balancing: входящий трафик на любой узел кластера распределяется по репликам сервиса через ingress routing mesh.

### Swarm Ingress Load Balancing

```
Внешние клиенты → порт 5000 (Ingress)
         │
  ┌──────┼──────┬──────┐
Node1  Node2  Node3  Node4
  │      │      │      │
 Replica Replica Replica Replica
         (overnet overlay)
```

## Детали и нюансы

- **DNS в дефолтной bridge-сети не работает**: дефолтная сеть docker0 не поддерживает встроенный DNS. `ping <container_name>` выдаёт ошибку unknown host. Для работы DNS-разрешения нужны пользовательские (user-defined) сети или устаревший механизм `--link`.
- **MACVLAN недоступен в публичных облаках**: требует перевода NIC хоста в promiscuous mode, что блокируется коммутаторами AWS/Azure. Ограничен корпоративными (on-premise) сетями.
- **Host mode и конфликт портов в Swarm**: при публикации порта в режиме `host mode` на одном узле нельзя запустить более одной реплики — порт хоста уже занят.
- **ICC (Inter-Container Communication)**: по умолчанию все контейнеры в bridge-сети могут свободно обмениваться трафиком (ICC включён). Если один контейнер скомпрометирован — он получает доступ ко всем локальным контейнерам, даже без проброса портов.
- **Узлы Swarm и оверлейные сети**: узлы в режиме Swarm видят оверлейные сети только если на них запущены контейнеры в этой сети (для снижения overhead gossip-протокола).

## Сравнения и trade-offs

| Драйвер | Изоляция | NAT | DNS | Мультихост | Когда использовать |
|---------|----------|-----|-----|------------|--------------------|
| bridge (default) | Высокая | Да | Нет (docker0) | Нет | Разработка, одиночный хост |
| bridge (user-defined) | Высокая | Да | Да | Нет | Изолированные группы контейнеров |
| host | Нет | Нет | — | Нет | Максимальная производительность сети |
| overlay | Средняя | Да | Да | Да | Swarm/кластеры на нескольких хостах |
| macvlan | Нет | Нет | — | Да | On-premise: нужен реальный MAC/IP |
| none | Полная | Нет | Нет | Нет | Максимальная изоляция, batch jobs |

## Примеры

```bash
# Создание пользовательской bridge-сети
docker network create -d bridge localnet
```

```bash
# Список всех сетей
docker network ls
```

```bash
# Просмотр конфигурации сети (подсети, IPAM, подключённые контейнеры)
docker network inspect bridge
```

```bash
# Создание оверлейной сети с шифрованием data plane (IPsec)
docker network create -d overlay -o encrypted prod-net
```

## Важно / подводные камни / best practices

- **Отключать ICC для production**: флаг демона `--icc=false` + явные пользовательские сети для контейнеров, которым нужно общаться. Снижает blast radius при компрометации контейнера.
- **Не использовать host mode без крайней необходимости**: теряется изоляция сети; открытые порты напрямую занимают порты хоста; конфликты и уязвимости напрямую затрагивают хост.
- **Предпочитать user-defined bridge default bridge**: в user-defined сетях работает встроенный DNS, контейнеры видят друг друга по именам.
- **MACVLAN только on-premise**: облачные провайдеры блокируют promiscuous mode на сетевом уровне.
- **Encrypted overlay обязателен для production Swarm**: без `-o encrypted` данные между узлами передаются в открытом виде (шифруется только control plane по умолчанию).
