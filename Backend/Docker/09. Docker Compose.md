# 09. Docker Compose

## Суть и контекст

Docker Compose — инструмент для декларативного определения, запуска и управления многоконтейнерными приложениями. Конфигурация описывается в YAML-файле (docker-compose.yml), а сам инструмент транслирует её в последовательность вызовов Docker API. Применяется в разработке, тестировании и (через Docker Stacks) в production-деплое на Swarm-кластерах.

---

## Ключевые идеи

- Compose — Infrastructure as Code: один YAML-файл декларирует все сервисы, сети, тома и секреты
- Жизненный цикл: `up` создаёт сети/тома → собирает образы → запускает контейнеры; `down` удаляет контейнеры и сети, но сохраняет тома
- При повторном `up` Compose пересоздаёт только изменившиеся контейнеры, переподключая существующие тома
- Docker Stacks — тот же синтаксис файла (v3+), но деплой в Swarm-кластер через `docker stack deploy`
- Секреты монтируются в `/run/secrets/<name>` как файлы в памяти (in-memory tmpfs)
- `docker-compose restart` **не** перечитывает YAML — для применения изменений нужен `up -d`

---

## Определения и термины

**Docker Compose** — инструмент (приложение на Python) для декларативного определения, запуска и управления приложениями, состоящими из одного или нескольких контейнеров (сервисов), с использованием файлов конфигурации YAML.

**Services (Сервисы)** — отдельные компоненты распределённого приложения («контейнеры в продакшне»). Сервис определяет, какой образ выполняется, какие порты используются и в каком количестве реплик.

**Stack (Стек)** — комплексное многосервисное приложение, развёртываемое из единого файла Compose в режиме Docker Swarm с помощью команды `docker stack deploy`.

**Secrets (Секреты)** — конфиденциальные данные, монтируемые в реплики сервисов в виде обычного файла. В Linux монтируются в in-memory файловую систему по пути `/run/secrets/<name>`.

**extends** — ключевое слово для наследования параметров из другого YAML-файла и указанного в нём сервиса. Значения в дочернем файле переопределяют родительские. Секции `links` и `volumes_from` не наследуются.

---

## Как устроено

### Жизненный цикл приложения Compose

1. `docker-compose up` запускает парсинг файла конфигурации (`docker-compose.yml` по умолчанию).
2. Создаются низкоуровневые инфраструктурные объекты: сначала сети (networks) и тома (volumes) — они потребляются контейнерами.
3. Выполняется скачивание образов (`pull`) из реестра или локальная сборка (`build`), если образы отсутствуют.
4. Запускаются контейнеры. При наличии зависимостей (через `links` или `volumes_from`) сервисы запускаются в строгом порядке.
5. При повторном `up` Compose сравнивает состояние с YAML. Контейнеры с изменённой конфигурацией останавливаются, удаляются и пересоздаются с новыми IP. Существующие тома переподключаются.
6. `docker-compose down` удаляет контейнеры и сети, но **сохраняет** тома, образы и исходный код.

### Механизм extends (переиспользование конфигураций)

- `extends` позволяет наследовать параметры из другого YAML-файла и сервиса в нём.
- Значения в дочернем файле переопределяют значения из родительского.
- `links` и `volumes_from` **не наследуются** — во избежание непредвиденных аварий.

### Порты в Swarm: Ingress vs Host mode

- По умолчанию порты маппятся в режиме `ingress` — доступны на всех узлах кластера, даже без реплики.
- Режим `host` привязывает порт только на узлах, где физически выполняется реплика.
- Режим `host` требует версии Compose-файла ≥ 3.2.

---

## Детали и нюансы

- **Директивы Compose → аргументы docker run**: `ports` = `-p`, `environment` = `-e`, `volumes` = `-v`.
- **Compose vs Stacks**: Compose деплоит на одиночный демон (single-engine mode); Stacks использует тот же синтаксис (v3+), но деплоит в Swarm.
- **`build` не работает в Stacks**: при `docker stack deploy` сборка образов не выполняется — все образы должны быть заранее в реестре.
- **Контейнеры-сироты (orphans)**: при переименовании сервиса в YAML старый контейнер остаётся запущенным в фоне — Compose не удаляет его автоматически.
- **Секреты через переменные окружения — антипаттерн**: значения видны в `docker inspect`; использовать механизм `secrets`.

---

## Сравнения и trade-offs

| | Docker Compose | Docker Stacks |
|---|---|---|
| Режим | Single-engine | Swarm mode |
| Команда деплоя | `docker-compose up` | `docker stack deploy` |
| Сборка образов | Поддерживается | **Не поддерживается** |
| Синтаксис файла | v2/v3 | v3+ |
| Реплики и constraints | Нет | `deploy:` секция |
| Использование | Разработка, тесты | Production кластер |

## Примеры

### Web + Redis приложение

```yaml
version: "3.8"
services:
  web-fe:
    build: .
    command: python app.py
    ports:
      - target: 5000
        published: 5000
    networks:
      - counter-net
    volumes:
      - type: volume
        source: counter-vol
        target: /code
  redis:
    image: "redis:alpine"
    networks:
      - counter-net
networks:
  counter-net:
volumes:
  counter-vol:
```

### Deплой стека в Swarm с секретами

```yaml
version: "3.2"
services:
  database:
    image: postgres
    environment:
      POSTGRES_DB_PASSWORD_FILE: /run/secrets/postgres_password
    secrets:
      - postgres_password
    deploy:
      placement:
        constraints:
          - 'node.role == worker'
secrets:
  postgres_password:
    external: true
```

Секрет `postgres_password` создаётся заранее в кластере (`external: true`) и монтируется в `/run/secrets/postgres_password`.

### Команды

```bash
# Запуск в фоновом режиме
docker-compose up -d

# Нестандартное имя файла конфигурации
docker-compose -f prod-equus-bass.yml up -d

# Пересборка образов без кэша
docker-compose build --no-cache

# Запуск одного сервиса для тестов (без зависимостей, с удалением после)
docker-compose run --no-deps --rm -e ENV=UNIT identidock

# Остановка и удаление контейнеров + томов
docker-compose stop
docker-compose rm --force -v

# Просмотр состояния сервисов
docker-compose ps

# Загрузка актуальных образов из реестра
docker-compose pull

# Остановка и удаление контейнеров и сетей (тома сохраняются)
docker-compose down
```

## Важно / подводные камни / best practices

- **Хранить docker-compose.yml в VCS**: файл самодокументируемый; стирает грань dev/ops; гарантирует согласованность сред.
- **Разделять конфигурации через `extends`**: одна base-конфигурация, отдельные файлы для dev (volume-монтирование, ENV=DEV) и prod (ENV=PROD).
- **Использовать длинный синтаксис для `ports`**: `target`/`published`/`mode` — более читаемо и позволяет явно задать `host`/`ingress` в Swarm.
- **Не передавать секреты через `environment`**: любой с доступом к `docker inspect` увидит значение; использовать механизм `secrets`.
- **`docker-compose restart` не применяет изменения**: после правки YAML всегда использовать `docker-compose up -d`.
- **Тома для stateful-сервисов обязательны**: без тома для PostgreSQL выполнение `docker-compose down` безвозвратно уничтожит данные.
- **Контейнеры-сироты**: при переименовании сервиса вручную удалять старые контейнеры (`docker rm`) или временно вернуть имя в YAML и выполнить `docker-compose rm`.
