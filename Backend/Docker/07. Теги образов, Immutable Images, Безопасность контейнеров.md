# 07. Теги образов, Immutable Images, Безопасность контейнеров

## Суть и контекст

Теги образов — изменяемые указатели; `latest` не гарантирует последнюю версию. Image digests (SHA256) — единственный способ получить иммутабельный образ. Безопасность контейнеров строится на ограничении Linux capabilities, seccomp-фильтрации syscalls, AppArmor-профилях и проверке подлинности образов через Docker Content Trust.

---

## Ключевые идеи

- Тег — изменяемый указатель на IMAGE ID; перезапись тега → старый образ становится dangling
- `latest` — floating tag без гарантии актуальности; опасен в production
- Image digest (SHA256) — строго иммутабелен; любое изменение образа → новый хэш
- Linux capabilities: дробление root-прав на гранулярные категории
- seccomp: фильтрация syscalls на уровне ядра; по умолчанию Docker блокирует ~300 из ~400 syscalls
- AppArmor: профили MAC (мандатного контроля доступа) на основе путей
- Docker Content Trust: криптографическая подпись и верификация образов
- `--privileged` = отключение всех механизмов изоляции Docker

---

## Определения и термины

**Тег образа (Image Tag)** — алфавитно-цифровой идентификатор, присваиваемый образам внутри репозитория (например, `14.04` или `stable`), позволяющий идентифицировать версию ПО. Теги изменяемы (mutable) — их можно перезаписать в любое время.

**Тег `latest`** — floating tag, применяемый Docker по умолчанию если при сборке или загрузке образа явно не указан другой тег. Является неофициальным соглашением и не гарантирует самую новую версию в репозитории.

**Дайджест образа (Image Digest / Content Hash)** — криптографическое хэш-значение (SHA256) содержимого образа (манифеста и слоёв файловой системы). Строго иммутабелен: любое изменение неизбежно генерирует новый хэш. Также обозначается как CAIID (content addressable image identifier).

**Capabilities (привилегии ядра Linux)** — наборы привилегий в ядре Linux, позволяющие раздробить права суперпользователя (root) на отдельные категории для гранулярного ограничения доступа процессов к системным функциям.

**seccomp (secure computing mode)** — функция ядра Linux для фильтрации и блокировки определённых системных вызовов (syscalls), выполняемых процессом из пользовательского пространства в ядро.

**AppArmor** — модуль безопасности Linux (LSM), реализующий мандатный контроль доступа (MAC) путём применения к процессам профилей, ограничивающих доступ к файлам и системным ресурсам на основе путей.

**Docker Content Trust (DCT)** — механизм Docker для криптографической подписи образов авторами и проверки этой подписи потребителями при загрузке. Базируется на программном каркасе Notary и спецификации The Update Framework.

## Как устроено

### Механика тегов и версионирования

- Теги служат лишь указателями на конкретный IMAGE ID
- При публикации обновлённого образа с тем же тегом — старый образ лишается тега (становится dangling), тег переносится на новый IMAGE ID
- Семантическое версионирование с перекрывающимися тегами: `1.4.2`, `1.4`, `1`, `latest` → все ссылаются на один IMAGE ID

### Механика иммутабельных образов (Digests)

- С Docker 1.10 используется модель content addressable storage
- Образ = манифест конфигурации + слои; для каждого вычисляется хэш SHA256

**Pull by Digest:**
1. Docker-клиент запрашивает манифест по точному SHA256 дайджесту
2. Реестр возвращает манифест
3. Docker скачивает слои, локально вычисляет хэши
4. Если хэши совпадают — образ криптографически верифицирован; изменение кода → несовпадение хэша → Docker отклоняет образ

**Нюанс:** для экономии трафика слои сжимаются при Push. Сжатый контент имеет другой хэш; Docker применяет дополнительный distribution hash (хэш сжатой версии) для проверки целостности при передаче.

### Механика безопасности

**Capabilities:** Docker по умолчанию сбрасывает часть привилегий Linux (напр. `SYS_TIME`, `SYS_MODULE`). Добавление/удаление через `--cap-add` / `--cap-drop`.

**seccomp:** при запуске Docker применяет JSON-профиль, перехватывающий syscalls. По умолчанию блокируются 32-bit syscalls, устаревшие сетевые функции и syscalls, не нужные стандартному контейнеру.

**Docker Content Trust:**
1. Автор создаёт корневой офлайн-ключ и ключ тегирования
2. Образ подписывается и выгружается в реестр
3. Клиент с `DOCKER_CONTENT_TRUST=1` при Pull скачивает сертификат автора по HTTPS, проверяет открытый ключ и математически валидирует подпись; неподписанные образы блокируются

**`--privileged`:** контейнер получает полный доступ ко всем устройствам хоста и всем Capabilities ядра — фактически отключает AppArmor, seccomp и caps.

## Детали и нюансы

- Root внутри контейнера (UID 0) = root на хосте (User Namespace по умолчанию не изолирован); при пробитии изоляции — root на сервере
- Бинарные файлы с битами SUID/SGID (`passwd`, `su`, `ping`) позволяют процессам выполняться с правами владельца файла; уязвимость → эскалация привилегий
- Если seccomp профиль с действием `kill` — запрещённый syscall → ядро немедленно прерывает процесс

## Сравнения и trade-offs

| Идентификатор | Изменяемость | Безопасность | Рекомендация |
|--------------|-------------|--------------|--------------|
| Тег `:latest` | Изменяемый | Низкая (подменяем) | Только для dev/экспериментов |
| Семантический тег (`:1.4.2`) | Изменяемый | Средняя | Удобен, но тег можно перезаписать |
| SHA256 digest | Иммутабелен | Максимальная | Production, CI/CD, FROM в Dockerfile |

| Механизм безопасности | Тип | Что ограничивает |
|----------------------|-----|-----------------|
| Linux Capabilities | Гранулярные права root | Системные привилегии ядра |
| seccomp | Фильтр syscalls | Системные вызовы к ядру |
| AppArmor | MAC-профили | Доступ к файлам и ресурсам |
| User Namespace | Изоляция UID | root в контейнере ≠ root на хосте |
| Docker Content Trust | Криптоподпись | Подлинность образа |

## Примеры

```bash
# Pull иммутабельного образа по SHA256 digest
docker image pull alpine@sha256:9a839e63da...9ea4fb9a54

# Включение Docker Content Trust (блокирует неподписанные образы)
export DOCKER_CONTENT_TRUST=1

# Максимальное ограничение capabilities: дропнуть всё, добавить только CHOWN
docker run --cap-drop all --cap-add CHOWN debian chown 100 /tmp

# Разрешение конкретного syscall через seccomp
docker run -d --security-opt seccomp:allow:clock_adjtime ntpd
```

```dockerfile
# Использование digest в FROM — гарантия иммутабельного базового слоя
FROM debian@sha256:d5e87cfcb730...

# Удаление SUID/SGID битов из всех бинарных файлов
RUN for i in $(find / -type f \( -perm +6000 -o -perm +2000 \)); \
    do chmod ug-s $i; done
```

## Важно / подводные камни / best practices

- **Никогда не использовать `:latest` в production** — floating tag обновляется без ведома; откатиться по тегу невозможно
- **FROM в Dockerfile с SHA256 digest** — гарантия, что базовый образ не подменён; защита от supply chain атак
- **Семантическое тегирование:** при сборке `1.4.2` тегировать как `1.4.2`, `1.4`, `1`, `latest` — пользователи выбирают уровень плавности обновлений
- **Сканирование образов (Trivy, Twistlock, Clair) в CI/CD** — базовые образы накапливают CVE; обнаруживать до деплоя
- **Никогда `--privileged` в production** — отключает все механизмы изоляции
- **Никогда root в контейнере** — явно указывать `USER app` в Dockerfile
- **Удалять SUID/SGID биты** из образа — предотвращают privilege escalation
