# RabbitMQ — масштабирование, отказоустойчивость и эксплуатация

## Суть и контекст

Масштабирование RabbitMQ включает горизонтальное добавление потребителей (competing consumers), кластеризацию узлов для отказоустойчивости и настройку Federation/Shovel для географически распределённых систем. Производительность зависит от выбора модели подтверждения, настройки prefetch (QoS) и типа хранения (persistent vs transient). Backpressure реализуется через TCP-уровень (система кредитов) и пороги memory/disk alarms. Мониторинг осуществляется через Management Plugin, CLI (rabbitmqctl) и интеграции с Nagios/Munin/Zabbix.

---

## Ключевые идеи

- Competing Consumers — несколько консьюмеров на одной очереди, распределение round-robin
- Prefetch count (QoS) — ограничение окна неподтверждённых сообщений, защита от переполнения
- HA Queues (Mirrored Queues) — зеркалирование очередей на несколько узлов кластера
- Quorum Queues — в источниках не найдено (описаны только HA-очереди на базе x-ha-policy)
- Federation — трансляция сообщений между брокерами без единого кластера (WAN-friendly)
- Shovel — плагин-клиент для перекачки сообщений между источником и приёмником
- TCP Backpressure — замена устаревшего Channel.Flow, работает на системе кредитов
- Memory/Disk Alarms — пороги vm_memory_high_watermark и disk_free_limit
- Кластеры RabbitMQ предназначены строго для LAN, не для WAN
- Максимальный рекомендуемый размер кластера — 32–64 узла
- Stats Node — единственный узел в кластере, собирающий статистику для Management UI

## Определения и термины

**Competing Consumers (Конкурирующие потребители)** — несколько приложений-подписчиков, подключённых к одной и той же очереди и конкурирующих за извлечение сообщений. RabbitMQ гарантирует доставку одного сообщения только одному из потребителей, распределяя нагрузку по алгоритму round-robin (по кругу).

**Prefetch count / Quality of Service (QoS)** — ограничение на количество скопившихся сообщений (window size), отправленных брокером потребителю, но ещё не получивших подтверждение (unacknowledged). Действует как предохранитель от переполнения памяти получателя. Если установлен в 0 (или не установлен при `no_ack=True`), брокер шлёт сообщения непрерывно.

**High Availability (HA) Queues / Mirrored Queues** — очереди, имеющие избыточные копии (зеркала) на нескольких или всех серверах внутри кластера RabbitMQ. При падении первичного (primary) узла очереди, вторичный узел берёт на себя роль первичного.

**Quorum Queues** — в источниках не найдено. Описаны только классические HA-очереди на базе `x-ha-policy`.

**Federation** — механизм (плагин) для трансляции сообщений между брокерами без создания единого кластера, устойчивый к высоким сетевым задержкам (WAN-friendly). Поддерживает Federated Exchanges (обменники) и Federated Queues (очереди).

**Shovel** — плагин RabbitMQ, работающий как клиентское приложение, которое подключается к источнику и приёмнику, читает, записывает сообщения и самостоятельно обрабатывает сбои соединений.

**TCP Backpressure** — механизм защиты брокера от перегрузки путём прекращения чтения данных из TCP-сокета публикующего клиента (вместо устаревшего `Channel.Flow`). Работает на основе системы кредитов для каждого соединения.

**Memory / Disk Alarms** — пороги `vm_memory_high_watermark` (доля RAM, по умолчанию 0.4) и `disk_free_limit` (минимум байт на диске, по умолчанию 50000000). При нарушении порогов срабатывает flow control (блокировка паблишеров).

**Stats Node** — узел в кластере RabbitMQ (при включённом rabbitmq-management), собирающий статистику и данные о состоянии всех остальных узлов. В кластере в любой момент времени может быть только один Stats Node.

## Как устроено

### Масштабирование consumers (Competing Consumers & Prefetch)

1. Брокер применяет push-модель: выталкивает сообщения в открытые TCP-каналы потребителей
2. Сообщения распределяются по принципу round-robin (циклически) среди всех активных подписчиков очереди
3. Если установлен `prefetch_count`, брокер ведёт счётчик неподтверждённых (unacked) сообщений для канала
4. Когда счётчик достигает лимита, push-отправка в данный канал приостанавливается до получения `Basic.Ack`

### Кластеризация (HA Queues)

1. Паблишер отправляет сообщение на любой узел кластера
2. Брокер синхронизирует сообщение между всеми узлами, указанными в `x-ha-policy` (primary и secondary)
3. Сообщение сохраняется на дисках/в памяти всех узлов
4. При потреблении сообщения с любого узла и получении `Basic.Ack`, все копии немедленно удаляются со всех узлов

### Federation Exchange

1. На downstream (получающем) узле объявляется политика федерации
2. Плагин федерации подключается к upstream (отправляющему) узлу по AMQP
3. Создаётся специальная work queue на upstream-узле
4. Downstream-узел потребляет сообщения из этой work queue и локально перепубликует их в свой локальный обменник

### Backpressure и Alarms (Flow Control)

1. При открытии соединения выделяется количество «кредитов»
2. Каждый RPC-запрос (публикация) уменьшает счётчик кредитов
3. После обработки запроса брокером кредит возвращается
4. Если кредиты исчерпаны (или сработали пороги `vm_memory_high_watermark` / `disk_free_limit`), брокер перестаёт читать из TCP-сокета
5. ОС клиента блокирует процесс публикации по I/O
6. Брокер отправляет асинхронный фрейм `Connection.Blocked` клиенту

### Мониторинг

- `rabbitmqctl status` или `rabbitmqctl report` — опрос локального Erlang-узла
- HTTP REST API (порт 15672 или 55672 в старых версиях) — агрегированная JSON-статистика от Stats Node
- Интеграции: Nagios (check_rabbitmq_server Perl-скрипты), Munin (плагин на Python/Perl), Zabbix (через externalscripts) — сбор данных через API или CLI

## Детали и нюансы

### Кластеризация только для LAN

Кластеры RabbitMQ спроектированы строго для LAN (локальных сетей) с низкими задержками. Использование кластера через интернет или между Availability Zones приведёт к рассинхронизации и «network partitions». Для WAN обязательно использовать Federation или Shovel.

### Ограничение размера кластера

Максимальный рекомендуемый размер кластера — от 32 до 64 узлов. Сложность синхронизации графа состояний нелинейна. При превышении управление кластером и сбор метрик замедляются.

### Отставание Secondary ноды

Возвращённая (или новая) нода не получает старые сообщения, уже находящиеся в очереди. Она будет синхронизирована только после того, как все старые сообщения будут потреблены и в очереди останутся только новые сообщения.

### Split-brain кластера

Если узлы потеряли связь и их состояния разошлись, восстановление кластера автоматически невозможно. Требуется ручная остановка всех узлов и запуск начиная с того Disk-узла, данные которого считаются наиболее актуальными.

### Cross-node routing

Сообщения всегда маршрутизируются на тот узел, где была физически объявлена очередь (кроме HA-зеркал). Если потребитель подключён к Узлу А, а очередь живёт на Узле Б, трафик будет прогоняться через весь кластер, удваивая сетевую нагрузку.

### Write bias при persistent messages

RabbitMQ при `delivery-mode: 2` (запись на диск) страдает от «write bias» (перекос в сторону записи), подобно OLTP базам данных. Кэширование записи ограничено контроллером диска (в отличие от кэширования чтения), что может вызывать задержки (I/O block).

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|-------------------|
| Clustering (HA Queues) | Единое состояние, автоматический failover | Только LAN, split-brain, нелинейная сложность синхронизации | Один дата-центр, высокая доступность |
| Federation | WAN-friendly, независимые серверы, устойчив к задержкам | Нет единого состояния, настройка политик | Между дата-центрами, геораспределение |
| Shovel | Простота, самостоятельная обработка сбоев | Ручная настройка, нет автоматической топологии | Перекачка данных между изолированными брокерами |

| Настройка prefetch | Эффект на throughput | Эффект на RAM | Рекомендация |
|-------------------|---------------------|---------------|-------------|
| Не установлен (0) | Максимальный push | Риск переполнения | Не рекомендуется |
| Малый (1-10) | Низкий (частые ack) | Минимальный | Тяжёлая обработка |
| Оптимальный (~2500) | Пиковый | Средний | Легковесные сообщения |
| multiple=True (batch ack) | Выше (меньше пакетов) | Средний | При падении — обработка всего батча заново |

| Тип узла | Роль | Особенности |
|----------|------|-------------|
| Disk Node | Хранение метаданных и сообщений | Минимум один в кластере, failover |
| RAM Node | Только обработка | Быстрее, но при рестарте теряет данные |
| Stats Node | Сбор статистики для Management UI | Только один в кластере |

## Примеры

### Объявление HA-очереди на всех узлах

```java
Map<String, Object> args = new HashMap<>();
args.put("x-ha-policy", "all");
channel.queueDeclare("my-ha-queue", true, false, false, args);
```

Аргумент `x-ha-policy` со значением `all` указывает кластеру зеркалировать очередь на все доступные узлы.

### HA-очередь на конкретных узлах

```java
Map<String, Object> args = new HashMap<>();
args.put("x-ha-policy", "nodes");
args.put("x-ha-nodes", Arrays.asList("rabbit@node1", "rabbit@node2"));
channel.queueDeclare("my-ha-queue", true, false, false, args);
```

Аргумент `nodes` включает точечный выбор, список `x-ha-nodes` задаёт имена Erlang-узлов для зеркалирования.

### HAProxy перед кластером RabbitMQ

```
listen rabbitmq
  bind *:5672
  mode tcp
  balance roundrobin
  server rabbit01 192.168.1.10:5672 check
  server rabbit02 192.168.1.11:5672 check backup
```

Включает балансировку TCP-пакетов, rabbit01 — основной, rabbit02 принимает соединения только при падении основного (backup).

### QoS (Prefetch) и batch ack

```python
channel.prefetch_count(10)

for i, message in enumerate(queue):
    process(message)
    if (i + 1) % 10 == 0:
        message.ack(all_previous=True)
```

Ограничивает push-окно 10 сообщениями. Метод `ack(all_previous=True)` (в AMQP — `multiple=True`) разом подтверждает все сообщения с 1 по 10, минимизируя сетевые задержки.

### Добавление узла в кластер через CLI

```bash
rabbitmqctl stop_app
rabbitmqctl reset
rabbitmqctl join_cluster rabbit@primary
rabbitmqctl start_app
```

`stop_app` останавливает AMQP-процессы (не убивая Erlang VM). `reset` стирает локальную метаинформацию узла. `join_cluster` подключает к `rabbit@primary`. `start_app` возобновляет работу.

## Важно / подводные камни / best practices

- **Не используйте Basic.Get в цикле** — синхронный RPC на каждое сообщение, как минимум в 2 раза медленнее `Basic.Consume`. Брокер не может оптимизировать доставку
- **Подключайте консьюмеров к узлу очереди** — определяйте топологию (queue locality) и подключайте напрямую к узлам, владеющим очередями. Иначе cross-node routing удваивает сетевую нагрузку
- **Не кластеризуйте через WAN** — используйте Federation или Shovel для геораспределения
- **Разделяйте узлы в крупных кластерах** — один Disk-узел как Stats Node, один вторичный Disk для failover, остальные — RAM-узлы
- **Оптимальный prefetch_count ~2500** — эмпирические тесты показывают пиковую пропускную способность для легковесных сообщений
- **Групповое подтверждение (multiple=True)** — снижает количество пакетов и увеличивает throughput, но при падении весь батч обрабатывается заново
- **Не используйте транзакции (TX) для производительности** — используйте Publisher Confirms
- **Split-brain требует ручного вмешательства** — автоматическое восстановление кластера невозможно при рассинхронизации состояний
