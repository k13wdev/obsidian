## Суть и контекст

Конспект охватывает ключевые паттерны, применяемые в распределённых системах баз данных:

1. **Saga (Хроника)** — паттерн замены длинных распределённых транзакций последовательностью коротких локальных транзакций с компенсациями.
2. **Идемпотентность (Idempotency)** — свойство, обеспечивающее безопасность повторного выполнения операций.
3. **Eventual Consistency** — модель согласованности с механизмами восстановления (read repair, anti-entropy, Merkle trees).
4. **Векторные часы (Vector clocks)** — структура для упорядочивания версий объектов в распределённых системах.
5. **Sloppy quorum и Hinted handoff** — механизмы обеспечения доступности при отсутствии кворума.

---

## Ключевые идеи

- Длинные ACID-транзакции (часы, дни, недели) блокируют доступ к ресурсам для других пользователей на неопределённо долгое время; при откате теряется огромный объём полезной работы. Вместо них используются саги.
- Сага — последовательность коротких локальных транзакций с гарантией: либо все выполнены успешно, либо выполнены компенсирующие транзакции.
- Все транзакции саги (и основные, и компенсирующие) **обязаны быть строго идемпотентными** — это критическое ограничение.
- Сага **не обеспечивает изоляцию** (свойство "I" из ACID) — параллельные транзакции видят данные в промежуточных состояниях.
- Eventual consistency: если остановить приложение, вносящее изменения, то рано или поздно данные на всех узлах станут одинаковыми.
- Два механизма восстановления согласованности: реактивный (read repair) и проактивный (anti-entropy с деревьями Меркла).
- Выбор "last write wins" в распределённых системах фактически случаен и зависит от рассинхронизации часов на узлах — приводит к потере обновлений.
- Альтернатива — перекладывание логики разрешения конфликтов на приложение (siblings в Riak KV).

## Определения и термины

**Идемпотентность (Idempotency)** — свойство транзакции, при котором её многократное исполнение приводит к тому же эффекту, что и однократное.

**Сага (Saga / Хроника)** — последовательность коротких (локальных) транзакций, выполняемых последовательно или параллельно, но независимо друг от друга, для которых гарантируется, что либо все они будут выполнены успешно, либо будут выполнены компенсирующие транзакции. Используется для замены длинных распределённых транзакций, удерживающих блокировки ресурсов продолжительное время (часы, дни).

**Компенсирующая транзакция** — транзакция, выполняющая обратные изменения для отмены результатов успешно завершённой локальной транзакции в случае незавершённого выполнения всей саги в целом.

**Согласованность в конечном счёте (Eventual consistency)** — состояние, при котором данные постоянно реплицируются между узлами, и если остановить приложение, вносящее изменения, то рано или поздно данные на всех узлах станут одинаковыми.

**Дерево Меркла (Merkle tree)** — иерархическая структура из контрольных сумм для небольших наборов объектов, используемая для быстрого поиска отличающихся данных между узлами кластера с минимальным сетевым трафиком.

**Векторные часы (Vector clock)** — массив, количество элементов в котором равно количеству узлов, используемый для упорядочивания версий объектов. При записи значения в базу вместе с ним записывается значение векторных часов.

## Как устроено

### Saga Pattern — нормальное выполнение

1. Оркестратор присваивает саге идентификатор и фиксирует её начало в своей внутренней БД.
2. Оркестратор последовательно (или параллельно) вызывает локальные транзакции на узлах-участниках.
3. Участники фиксируют изменения в своих БД и возвращают подтверждение.
4. Оркестратор фиксирует окончание саги в своей БД.

**Компоненты:** оркестратор (координатор саги), БД оркестратора, узлы-участники (микросервисы с собственными БД).

### Saga Pattern — отказ участника

1. Участник отказывается выполнять транзакцию (например, из-за логической ошибки).
2. Оркестратор вызывает компенсирующие транзакции для всех транзакций, которые уже были завершены предыдущими участниками саги.
3. Компенсирующие транзакции возвращают подтверждение.
4. Сага завершается неудачей.

**Визуализация отказа:** сверху вниз расположены 4 горизонтальные линии: "Оркестратор", "БД оркестратора", "Участник 1", "Участник 2". Оркестратор посылает сигнал "начало саги" в БД оркестратора. Затем отправляет "T1" к Участнику 1 → ответ "ok". Затем отправляет "T2" к Участнику 2 → ответ "отказ". После этого Оркестратор отправляет "T1'" (компенсация) к Участнику 1 → ответ "ok". В конце Оркестратор посылает "конец саги" в свою БД.

### Saga Pattern — отказ оркестратора

Запускается новый экземпляр оркестратора, который считывает из собственной БД информацию о наличии незавершённой саги и выполняет её повторно. Именно поэтому все транзакции обязаны быть идемпотентными.

### Eventual Consistency — реактивное восстановление (Read repair)

1. Читающий сеанс запрашивает данные с нескольких узлов (кворум).
2. База считывает данные и сравнивает их версии/отметки.
3. Выбираются самые новые данные, а на узлах с устаревшими данными происходит перезапись актуальной версией.

### Eventual Consistency — проактивное восстановление (Anti-entropy)

1. Фоновый процесс или команда администратора инициирует проверку.
2. Узлы обмениваются деревьями Меркла (деревьями контрольных сумм).
3. Сравниваются корневые суммы, при несовпадении происходит спуск по дереву до выявления конкретных отличающихся объектов.
4. Устаревшая версия объекта заменяется актуальной.

**Визуализация дерева Меркла:** иерархия хеш-сумм в виде пирамиды. На нижнем уровне — блоки сырых хешей данных: `180A`, `2511`, `CC85`, `9990`. На уровне выше их значения объединяются в агрегированные хеши: `CC85 A874`, `BF21 719A`. На вершине — единая контрольная сумма для всего набора данных. Узлы сначала сравнивают вершину пирамиды, и только при несовпадении спускаются на уровень ниже.

### Sloppy Quorum и Hinted Handoff

При отсутствии кворума для записи данные записываются на узлы-заместители (нестрогий кворум / sloppy quorum). Узлы-заместители сохраняют данные и пытаются передать их целевым узлам позже (направленная передача / hinted handoff).

## Детали и нюансы

- **Компенсирующая транзакция не должна завершаться откатом** — в идеальном случае в ней не должно быть возможности прерывания. Она должна быть выполнена безусловно.
- **Саги — рекурсивное обобщение savepoints**: в отличие от точек сохранения, обеспечивающих линейный откат, саги позволяют структурировать иерархию действий, выполняемых параллельно.
- **Контрольные точки в оркестраторе**: фиксация промежуточных результатов в БД оркестратора по результатам выполнения некоторых шагов саги позволяет быстрее проводить восстановление при сбое, не повторяя уже выполненные тяжёлые шаги. Выбор определяется балансом стоимости повторного шага и стоимости обращения к БД.
- **Sloppy quorum — риск потери данных**: если за заданное время передача данных с заместителей на целевые узлы не удалась, заместители стирают данные, и они теряются. Читающие сеансы не видят эти данные, так как ищут их только на целевых узлах.
- **Last write wins — псевдослучайный выбор**: выбор актуального значения зависит от настроек (рассинхронизации) часов на узлах кластера и фактически является случайным.
- **Siblings (Riak KV)**: СУБД сохраняет несколько конфликтующих значений с одним ключом. Приложение, получая массив значений при чтении, применяет бизнес-логику для корректного слияния (merge) данных.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Длинная ACID-транзакция | Полные гарантии ACID, включая изоляцию | Блокирует ресурсы на часы/дни; при откате теряется весь объём работы | Короткие операции внутри одной БД |
| Saga (Хроника) | Не удерживает блокировки; компенсация вместо отката | Нет изоляции (I из ACID); сложность реализации компенсаций; все шаги должны быть идемпотентными | Распределённые операции большой длительности |
| Read repair (реактивное) | Восстановление происходит «бесплатно» при чтении | Устаревшие данные обнаруживаются только при чтении | Часто читаемые данные |
| Anti-entropy (проактивное) | Систематическая проверка всех данных | Дополнительный сетевой трафик и I/O | Редко читаемые данные, фоновая синхронизация |
| Last write wins | Простота реализации | Зависит от синхронизации часов; потеря обновлений | Данные, где потеря некритична |
| Разрешение конфликтов приложением (siblings) | Бизнес-логика слияния; нет потерь | Сложность реализации на стороне приложения | Критичные данные с частыми конкурентными записями |

## Связи с другими темами

- Saga напрямую связана с концепцией [[14. Транзакции, ACID, изоляция и MVCC|ACID-транзакций]] — сага обеспечивает атомарность (A) и долговечность (D), но **не обеспечивает изоляцию (I)**.
- Саги рассматриваются как рекурсивное обобщение **точек сохранения (savepoints)** — связь с вложенными транзакциями.
- Eventual consistency связана с применением **деревьев Меркла** для минимизации I/O и сетевого трафика при синхронизации узлов.
- Идемпотентность — критическое требование для работы саг и для обеспечения корректности при повторных вызовах в распределённых системах.
- Sloppy quorum и hinted handoff связаны с [[17. Масштабирование, репликация и CAP|CAP theorem и репликацией]] — механизмы обеспечения доступности (A) за счёт согласованности (C).

## Важно / подводные камни / best practices

- **Все транзакции саги обязаны быть идемпотентными** — и основные, и компенсирующие. Это не рекомендация, а обязательное требование для корректной работы при отказе оркестратора.
- **Компенсирующая транзакция не должна откатываться** — в ней не должно быть возможности прерывания.
- **Сага не обеспечивает изоляцию** — приложение должно использовать собственные механизмы изоляции для защиты от чтения промежуточных состояний.
- **Не используйте длинные ACID-транзакции** для операций большой длительности — используйте саги.
- **Фиксируйте контрольные точки** в БД оркестратора для ускорения восстановления при сбоях.
- **Last write wins ≠ безопасный выбор** — выбор зависит от рассинхронизации часов и приводит к потере обновлений.
- **Sloppy quorum может терять данные** — если hinted handoff не успел передать данные за заданное время, они стираются.
- **Перекладывайте разрешение конфликтов на приложение** (как в Riak KV), если потеря обновлений недопустима.
