## Суть и контекст

Конспект охватывает три фундаментальных блока работы с реляционными базами данных:

1. **Реляционная модель** - математическая модель данных, основанная на теории множеств и логике предикатов. Определяет структуру хранения данных через отношения (relations), кортежи (tuples) и атрибуты (attributes).
2. **Нормализация** - процесс последовательной декомпозиции переменных отношений для устранения аномалий вставки, обновления и удаления. Иерархия нормальных форм: 1НФ, 2НФ, 3НФ, НФБК.
3. **SQL** - язык запросов к реляционным СУБД. Рассматриваются операции выборки (SELECT), соединения (JOIN), группировки (GROUP BY), подзапросы, CTE, оконные функции и порядок обработки запроса (query processing pipeline).

---

## Ключевые идеи

- Отношение - строго математический абстрактный объект без дубликатов и без порядка строк и столбцов. Таблица SQL - его конкретное представление, допускающее дубликаты, позиционное упорядочивание и NULL.
- Кортежи представляют собой истинные высказывания, а заголовок отношения - предикат. Реляционная модель базируется на теории множеств и логике предикатов.
- Нормализация должна доводиться минимум до НФБК для устранения аномалий - принцип "по одному факту в одном месте".
- Иерархия нормальных форм: 1НФ ⊃ 2НФ ⊃ 3НФ ⊃ НФБК - каждая следующая является строгим подмножеством предыдущей.
- Ссылочная целостность поддерживается через внешние ключи с правилами CASCADE, RESTRICT, NO ACTION, SET NULL.
- Порядок обработки SQL-запроса: FROM -> WHERE -> GROUP BY -> Агрегация -> HAVING -> Window Functions -> SELECT -> ORDER BY.
- Комбинация SELECT-FROM-WHERE соответствует проекции сокращения декартова произведения: FROM = декартово произведение, WHERE = Restrict, SELECT = Project.
- Оконные функции выполняют вычисления для набора строк, связанных с текущей строкой, не объединяя их в одну выходную строку (в отличие от агрегатных функций).
- CTE (Common Table Expression) определяет временную именованную таблицу, существующую только во время выполнения одного объемлющего запроса.
- Антипаттерн: NOT IN (подзапрос) при наличии NULL - из-за трехзначной логики SQL запрос не вернет ни одной записи.

## Определения и термины

### Реляционная модель

**Отношение (Relation) / Таблица (Table)** - математическая структура, определяющая взаимосвязи объектов. В реляционной модели это абстрактный объект, а таблица является конкретным физическим или визуальным представлением.

**Кортеж (Tuple) / Строка (Row)** - множество упорядоченных троек вида <Имя атрибута, Имя типа, Значение>. Кортежи в отношении не упорядочены и не содержат дубликатов.

**Атрибут (Attribute) / Столбец (Column)** - элемент информации, описывающий сущность. Атрибуты в отношении не имеют позиционного упорядочивания слева направо.

**Первичный ключ (Primary Key)** - поле (или набор полей), уникально идентифицирующее строку, в котором запрещены неопределенные значения (NULL).

**Внешний ключ (Foreign Key)** - множество атрибутов в ссылающейся переменной отношения, значения которого должны совпадать со значениями потенциального ключа в переменной отношения, указанной в ссылке.

### Нормальные формы

**1НФ (Первая нормальная форма)** - переменная отношения находится в 1НФ тогда и только тогда, когда в любом допустимом значении этой переменной каждый ее кортеж содержит только одно атомарное значение для каждого из атрибутов.

**2НФ (Вторая нормальная форма)** - переменная отношения находится в 2НФ тогда и только тогда, когда она находится в 1НФ и каждый неключевой атрибут неприводимо зависит от ее первичного ключа.

**3НФ (Третья нормальная форма)** - переменная отношения находится в 3НФ тогда и только тогда, когда она находится в 2НФ и ни один неключевой атрибут не является транзитивно зависимым от ее первичного ключа. Альтернативное определение (Дзаниоло): для каждой функциональной зависимости X -> A подмножество X включает A, либо X является суперключом, либо A входит в состав потенциального ключа.

**НФБК (Нормальная форма Бойса-Кодда)** - переменная отношения находится в НФБК тогда и только тогда, когда каждая ее нетривиальная и неприводимая слева функциональная зависимость имеет в качестве детерминанта некоторый потенциальный ключ.

**4НФ (Четвёртая нормальная форма)** — переменная отношения находится в 4НФ тогда и только тогда, когда она находится в НФБК и все многозначные зависимости (МЗЗ) в ней фактически представляют собой функциональные зависимости от её ключей.

**5НФ (Пятая нормальная форма / ПСНФ)** — проекционно-соединительная нормальная форма. Переменная отношения находится в 5НФ тогда и только тогда, когда каждая нетривиальная зависимость соединения в ней определяется её потенциальным ключом (ключами).

**6НФ (Шестая нормальная форма)** — переменная отношения находится в 6НФ тогда и только тогда, когда она удовлетворяет всем нетривиальным (в том числе обобщённым) зависимостям соединения.

**Денормализация** — замена переменных отношений их соединением, при котором проекция обратно восстанавливает исходные данные. Цель — увеличение избыточности для сокращения JOIN при чтении.

### SQL

**SELECT** - оператор, инициирующий извлечение данных из промежуточной виртуальной таблицы, формируя результирующий набор столбцов или вычисляемых выражений.

**CROSS JOIN (Декартово произведение)** - операция соединения, генерирующая все возможные комбинации строк из двух таблиц. При таблицах размером N и M строк результат содержит N * M строк.

**INNER JOIN (Внутреннее соединение)** - операция, при которой для каждой строки левой таблицы в результирующую таблицу добавляется строка для каждой строки правой таблицы, удовлетворяющей условию соединения (ON).

**LEFT OUTER JOIN** - сначала выполняется внутреннее соединение, а затем для каждой строки левой таблицы, не нашедшей совпадения в правой, добавляется строка с null значениями для столбцов правой таблицы.

**RIGHT OUTER JOIN** - для строк правой таблицы, не нашедших совпадений в левой, добавляются строки с пустыми значениями в столбцах левой таблицы.

**FULL OUTER JOIN** - объединяет результаты левого и правого внешних соединений. Строки без совпадений с обеих сторон дополняются null-значениями.

**GROUP BY** - конструкция, объединяющая наборы строк с одинаковыми значениями в указанных столбцах в одну групповую строку, применяемая для устранения избыточности или вычисления агрегатов.

**HAVING** - конструкция для фильтрации групповых строк после того, как группы и агрегатные функции уже вычислены.

**Подзапрос (Subquery)** - выражение SELECT-FROM-WHERE-GROUP BY-HAVING, вложенное внутрь другого запроса, заключенное в круглые скобки.

**Коррелированный подзапрос (Correlated subquery)** - подзапрос, содержащий ссылки на переменные области значений (столбцы), определенные во внешнем (охватывающем) запросе.

**CTE (Common Table Expression)** - именованное вспомогательное выражение (запрос WITH), определяющее временную таблицу, существующую только во время выполнения одного объемлющего запроса.

**Оконная функция (Window function)** - функция, выполняющая вычисления для набора строк таблицы (окна), связанных с текущей строкой, при этом не объединяющая эти строки в одну выходную строку (в отличие от агрегатных функций).

**PARTITION BY** - конструкция внутри предложения OVER, разделяющая строки на группы (партиции), обрабатываемые оконной функцией независимо друг от друга.

**Window frame (Оконный фрейм)** - заданное подмножество строк внутри партиции, над которым выполняется вычисление функции для текущей строки.

**ROW_NUMBER** - оконная функция, возвращающая последовательный номер текущей строки в ее партиции, начиная с 1.

**RANK** - оконная функция, возвращающая ранг текущей строки с возможными пропусками (gaps) при наличии дубликатов (peers).

**DENSE_RANK** - оконная функция, возвращающая ранг текущей строки без пропусков (gaps).

**LAG** - оконная функция, возвращающая значение выражения, вычисленного для строки, находящейся на offset строк перед текущей строкой в партиции.

**LEAD** - оконная функция, возвращающая значение выражения, вычисленного для строки, находящейся на offset строк после текущей строки в партиции.

## Как устроено

### Процесс нормализации

Нормализация выполняется последовательной декомпозицией переменных отношений:

1. **1НФ -> 2НФ**: разбить на проекции для исключения функциональных зависимостей, не являющихся неприводимыми.
2. **2НФ -> 3НФ**: разбить на проекции для исключения транзитивных функциональных зависимостей.
3. **3НФ -> НФБК**: разбить на проекции для исключения функциональных зависимостей, в которых детерминанты не являются потенциальными ключами.
4. **НФБК -> 4НФ**: разбить на проекции для исключения многозначных зависимостей (МЗЗ), которые не являются функциональными.
5. **4НФ -> 5НФ**: разбить на проекции для исключения зависимостей соединения (ЗС), не определяемых потенциальными ключами.

Иерархия: 1НФ ⊃ 2НФ ⊃ 3НФ ⊃ НФБК ⊃ 4НФ ⊃ 5НФ. Переход от 1НФ до НФБК основан на функциональных зависимостях (ФЗ), к 4НФ — на многозначных зависимостях (МЗЗ), к 5НФ — на зависимостях соединения (ЗС).

### Ссылочная целостность (Referential Integrity)

Поддержка ссылочной целостности через внешние ключи реализуется правилами:

- **CASCADE** - каскадное удаление/обновление зависимых кортежей
- **RESTRICT** - запрет операции при наличии зависимых кортежей
- **NO ACTION** - аналогично RESTRICT (запрет)
- **SET NULL** - установка NULL в зависимых кортежах

### Query Processing Pipeline (порядок обработки SQL-запроса)

1. **FROM** - вычисление декартова произведения таблиц
2. **WHERE** - применение предикатов фильтрации к строкам виртуальной таблицы
3. **GROUP BY** - группировка оставшихся строк
4. **Агрегация** - вычисление агрегатных функций (SUM, COUNT и т.д.)
5. **HAVING** - отбрасывание сгруппированных строк, не удовлетворяющих условию
6. **Window Functions** - вычисление оконных функций
7. **SELECT** - проекция - формирование финального списка столбцов
8. **ORDER BY** - физическая сортировка итогового набора строк

### Window Frames (оконные фреймы)

- **ROWS** - определяет границы физическим количеством строк
- **RANGE** - определяет границы по логической разнице значений столбца сортировки (требует ровно одного столбца в ORDER BY)
- Дефолтное состояние фрейма при наличии ORDER BY: `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`

### Recursive CTE (рекурсивный CTE)

1. Вычисление нерекурсивного базового термина, результаты помещаются в итоговый набор и во временную рабочую таблицу (working table)
2. Вычисление рекурсивного термина с использованием рабочей таблицы как источника данных
3. Замена содержимого рабочей таблицы результатами шага 2
4. Повторение шагов 2-3 до тех пор, пока рабочая таблица не станет пустой

## Детали и нюансы

### Реляционная модель и нормализация

- **Перекрывающиеся составные потенциальные ключи**: если переменная отношения имеет несколько перекрывающихся составных потенциальных ключей, определение 3НФ игнорирует приводимую зависимость атрибута, если он сам является компонентом другого потенциального ключа. Избыточность данных сохраняется. Требуется применение НФБК.
- **Декомпозиция с потерями (lossy decomposition)**: при выполнении обратной операции соединения проекций возникает утрата информации или появление фиктивных (spurious) кортежей. Декомпозиция должна выполняться строго в соответствии с теоремой Хита.
- **Нарушение ссылочной целостности при удалении**: при удалении родительского кортежа система прерывает транзакцию и активизирует исключение, если не задано ссылочное действие CASCADE.
- **Отношение vs Таблица**: Отношение - строго математический, абстрактный объект (без дубликатов, без порядка строк и столбцов). Таблица SQL допускает дубликаты строк (мультимножество), позиционное упорядочивание и значения NULL, нарушая классическую реляционную теорию.

### SQL

- **SELECT * в приложениях**: символ `*` разворачивается в список столбцов на этапе компиляции. Добавление/удаление столбцов через ALTER TABLE приведет к несовпадению структур.
- **Агрегатные функции и NULL**: агрегатные функции (кроме COUNT(*)) полностью игнорируют значения NULL. Если на входе только NULL, функция возвращает NULL, а не 0.
- **Вложенность агрегатных функций**: невозможно напрямую выполнить агрегат от агрегата (`SELECT AVG(SUM(QTY))`). Требуется подзапрос в FROM.
- **Границы оконного фрейма**: frame_start не может быть UNBOUNDED FOLLOWING; frame_end не может быть UNBOUNDED PRECEDING; frame_end не может находиться логически раньше frame_start.
- **NOT IN при наличии NULL**: из-за трехзначной логики SQL выражение NOT IN (подзапрос), где подзапрос возвращает NULL, не вернет ни одной записи.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Нормализация до НФБК+ | Устранение аномалий вставки, обновления и удаления. Принцип "по одному факту в одном месте" | Больше JOIN при выборке, потенциально медленнее чтение | По умолчанию при проектировании логической схемы (OLTP) |
| Денормализация (OLTP) | Сокращение JOIN при чтении | Аномалии вставки/удаления, усложнение агрегатных запросов (нужна проекция для устранения дубликатов), рост физического размера | Антипаттерн в OLTP; допустимо как последнее средство |
| Денормализация (OLAP/DWH) | Радикальное сокращение дискового I/O при массовых выборках | Рост объёма хранения, дублирование данных | Аналитические БД (Star schema, Snowflake, Data vault), где read:write от 1000:1 до 10⁶:1, данные не редактируются построчно |
| Неатомарные значения (вложенные списки, JSON в столбце) | В источниках не указаны | Нарушение 1НФ. Невозможность использования стандартных реляционных операторов | Антипаттерн (по источникам) |

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Агрегатные функции | "Схлопывают" строки в одну | Теряется идентичность исходных строк | Когда нужен единственный итог по группе |
| Оконные функции | Сохраняют идентичность исходных строк | В источниках не указаны | Когда нужно вычисление по набору строк без потери строк |

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| EXISTS | Возвращает TRUE при обнаружении первой строки - эффективнее | В источниках не указаны | Проверка существования строк |
| IN | В источниках не указаны | Выполняет полное сканирование; NOT IN при NULL не возвращает записей | Фильтрация по списку значений без NULL |

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| CTE | Разбивает запрос на читаемые блоки | В источниках не указаны | Вместо глубоко вложенных подзапросов |
| Вложенные подзапросы | В источниках не указаны | Ухудшается читаемость при глубокой вложенности | Простые одноуровневые вложения |

| Ссылочное действие | Поведение |
|--------------------|-----------|
| CASCADE | Каскадное удаление/обновление зависимых кортежей |
| RESTRICT | Запрет операции при наличии зависимых кортежей |
| NO ACTION | Аналогично RESTRICT |
| SET NULL | Установка NULL в зависимых кортежах |

## Примеры

### Создание таблицы с внешними ключами и CASCADE

```sql
CREATE TABLE SP (
    S# S# NOT NULL,
    P# P# NOT NULL,
    QTY QTY NOT NULL,
    PRIMARY KEY ( S#, P# ),
    FOREIGN KEY ( S# ) REFERENCES S ON DELETE CASCADE ON UPDATE CASCADE,
    FOREIGN KEY ( P# ) REFERENCES P ON DELETE CASCADE ON UPDATE CASCADE
) ;
```

### LEFT OUTER JOIN

```sql
SELECT * FROM weather LEFT OUTER JOIN cities ON weather.city = cities.name;
-- Строки weather без совпадений в cities выведутся с null в столбцах cities
```

### GROUP BY + HAVING

```sql
SELECT city, count(*), max(temp_lo)
FROM weather
GROUP BY city
HAVING max(temp_lo) < 40;
-- WHERE не может фильтровать по агрегатам. HAVING применяется после вычисления max
```

### Коррелированный подзапрос

```sql
SELECT P.P#, ( SELECT SUM ( SP.QTY ) FROM SP WHERE SP.P# = P.P# ) AS TOTQTY
FROM P ;
-- Подзапрос обращается к P.P# из внешнего запроса
```

### CTE (Common Table Expression)

```sql
WITH regional_sales AS (
    SELECT region, SUM(amount) AS total_sales FROM orders GROUP BY region
)
SELECT region, product_sales FROM regional_sales;
-- Именованное временное выражение переиспользуется в основном запросе
```

### Оконная функция: нарастающий итог

```sql
SELECT salary, sum(salary) OVER (ORDER BY salary) FROM empsalary;
-- Окно по умолчанию: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

### Оконная функция с явным фреймом ROWS

```sql
SELECT sum(salary) OVER (ORDER BY salary ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING)
-- Рамка: одна строка до + текущая + одна строка после
```

- Реляционная модель данных напрямую базируется на [[01. Теория множеств|теории множеств]] и [[02. Логика предикатов|логике предикатов]]. Кортежи представляют собой истинные высказывания, а заголовок отношения - предикат.
- Комбинация SELECT-FROM-WHERE соответствует операциям [[03. Реляционная алгебра|реляционной алгебры]]: FROM = декартово произведение, WHERE = Restrict (ограничение), SELECT = Project (проекция).
- Агрегатные vs оконные функции: агрегатные "схлопывают" строки в одну, оконные сохраняют идентичность исходных строк.
- EXISTS vs IN: EXISTS возвращает TRUE при обнаружении первой строки, что эффективнее полного сканирования оператором IN.
- [[05. Декомпозиция отношений|Теорема Хита]] - основа корректной декомпозиции без потерь при нормализации.

## Важно / подводные камни / best practices

### Best practices

- Проектировать схему данных, доводя нормализацию минимум до НФБК для устранения аномалий вставки, обновления и удаления (принцип "по одному факту в одном месте").
- Использование CTE вместо глубоко вложенных подзапросов - разбивает запрос на читаемые блоки.
- Осторожность с неявным поведением оконных фреймов при ORDER BY: без явного ROWS/RANGE фрейм дефолтный = нарастающий итог (`RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`).

### Антипаттерны

- **Денормализация на уровне логических переменных отношения** ради предполагаемого повышения производительности выборки. Усложняет аналитические запросы, увеличивает физический размер данных, порождает аномалии обновления.
- **Неатомарные значения** - использование переменных отношения с вложением списков или JSON-структур в реляционный столбец (нарушение 1НФ). Приводит к невозможности использования стандартных реляционных операторов.
- **NOT IN (подзапрос) при наличии NULL** - из-за трехзначной логики SQL запрос не вернет ни одной записи.

### Подводные камни

- `SELECT *` в приложениях: символ `*` разворачивается в список столбцов на этапе компиляции. Добавление/удаление столбцов через ALTER TABLE приведет к несовпадению структур.
- Агрегатные функции (кроме COUNT(*)) полностью игнорируют NULL. Если на входе только NULL - возвращается NULL, а не 0.
- Невозможно напрямую вложить агрегат в агрегат (`AVG(SUM(QTY))`). Требуется подзапрос в FROM.
- Семантически неверная декомпозиция при нормализации приводит к появлению фиктивных (spurious) кортежей при обратном соединении проекций.

## Карточки для повторения

#flashcards/db/normalization

Отношение (Relation):::Математическая структура, определяющая взаимосвязи объектов. Абстрактный объект без дубликатов и без порядка строк и столбцов. Таблица - его конкретное представление.

Кортеж (Tuple):::Множество упорядоченных троек вида <Имя атрибута, Имя типа, Значение>. Кортежи в отношении не упорядочены и не содержат дубликатов.

Атрибут (Attribute):::Элемент информации, описывающий сущность. Атрибуты в отношении не имеют позиционного упорядочивания слева направо.

Primary Key (Первичный ключ):::Поле (или набор полей), уникально идентифицирующее строку, в котором запрещены значения NULL.

Foreign Key (Внешний ключ):::Множество атрибутов в ссылающейся переменной отношения, значения которого должны совпадать со значениями потенциального ключа в указанной переменной отношения.

1НФ (Первая нормальная форма):::Переменная отношения находится в 1НФ тогда и только тогда, когда каждый ее кортеж содержит только одно атомарное значение для каждого из атрибутов.

2НФ (Вторая нормальная форма):::Переменная отношения находится в 2НФ, когда она в 1НФ и каждый неключевой атрибут неприводимо зависит от ее первичного ключа.

3НФ (Третья нормальная форма):::Переменная отношения находится в 3НФ, когда она в 2НФ и ни один неключевой атрибут не является транзитивно зависимым от ее первичного ключа.

НФБК (Нормальная форма Бойса-Кодда):::Переменная отношения находится в НФБК, когда каждая ее нетривиальная и неприводимая слева функциональная зависимость имеет в качестве детерминанта некоторый потенциальный ключ.

CROSS JOIN:::Операция соединения, генерирующая все возможные комбинации строк из двух таблиц. При таблицах размером N и M строк результат содержит N * M строк.

INNER JOIN:::Операция, при которой для каждой строки левой таблицы добавляется строка для каждой строки правой таблицы, удовлетворяющей условию ON.

LEFT OUTER JOIN:::Сначала выполняется INNER JOIN, затем для строк левой таблицы без совпадений добавляются строки с null для столбцов правой таблицы.

RIGHT OUTER JOIN:::Для строк правой таблицы без совпадений в левой добавляются строки с null в столбцах левой таблицы.

FULL OUTER JOIN:::Объединяет результаты LEFT и RIGHT OUTER JOIN. Строки без совпадений с обеих сторон дополняются null-значениями.

GROUP BY:::Конструкция, объединяющая строки с одинаковыми значениями в указанных столбцах в одну групповую строку для вычисления агрегатов.

HAVING:::Конструкция для фильтрации групповых строк после вычисления групп и агрегатных функций.

Подзапрос (Subquery):::Выражение SELECT-FROM-WHERE-GROUP BY-HAVING, вложенное внутрь другого запроса, заключенное в круглые скобки.

Коррелированный подзапрос (Correlated subquery):::Подзапрос, содержащий ссылки на столбцы, определенные во внешнем (охватывающем) запросе.

CTE (Common Table Expression):::Именованное вспомогательное выражение (запрос WITH), определяющее временную таблицу, существующую только во время выполнения одного объемлющего запроса.

Оконная функция (Window function):::Функция, выполняющая вычисления для набора строк (окна), связанных с текущей строкой, не объединяющая их в одну выходную строку (в отличие от агрегатных).

PARTITION BY:::Конструкция внутри OVER, разделяющая строки на группы (партиции), обрабатываемые оконной функцией независимо друг от друга.

Window frame (Оконный фрейм):::Заданное подмножество строк внутри партиции, над которым выполняется вычисление функции для текущей строки.

ROW_NUMBER:::Оконная функция, возвращающая последовательный номер текущей строки в ее партиции, начиная с 1.

RANK:::Оконная функция, возвращающая ранг текущей строки с возможными пропусками (gaps) при наличии дубликатов (peers).

DENSE_RANK:::Оконная функция, возвращающая ранг текущей строки без пропусков (gaps).

LAG:::Оконная функция, возвращающая значение для строки на offset строк перед текущей строкой в партиции.

LEAD:::Оконная функция, возвращающая значение для строки на offset строк после текущей строки в партиции.

Порядок обработки SQL-запроса (Query Processing Pipeline)
?
1. FROM - вычисление декартова произведения таблиц
2. WHERE - применение предикатов фильтрации
3. GROUP BY - группировка строк
4. Агрегация - вычисление агрегатных функций
5. HAVING - фильтрация сгруппированных строк
6. Window Functions - вычисление оконных функций
7. SELECT - проекция (формирование финального списка столбцов)
8. ORDER BY - физическая сортировка итогового набора

Шаги нормализации от 1НФ до НФБК
?
1. 1НФ -> 2НФ: разбить на проекции для исключения функциональных зависимостей, не являющихся неприводимыми
2. 2НФ -> 3НФ: разбить на проекции для исключения транзитивных функциональных зависимостей
3. 3НФ -> НФБК: разбить на проекции для исключения ФЗ, в которых детерминанты не являются потенциальными ключами
4. НФБК -> 4НФ: исключить многозначные зависимости, не являющиеся функциональными
5. 4НФ -> 5НФ: исключить зависимости соединения, не определяемые потенциальными ключами

4НФ (Четвёртая нормальная форма):::НФБК + все многозначные зависимости (МЗЗ) являются функциональными зависимостями от ключей

5НФ (Пятая нормальная форма / ПСНФ):::Каждая нетривиальная зависимость соединения определяется потенциальным ключом (ключами)

6НФ (Шестая нормальная форма):::Удовлетворяет всем нетривиальным (включая обобщённые) зависимостям соединения

Денормализация:::Замена переменных отношений их соединением для увеличения избыточности и сокращения JOIN при чтении. Допустима в OLAP/DWH (Star schema, Snowflake, Data vault).

Когда денормализация лучше нормализации?::Исключительно в аналитических БД (OLAP/DWH), где read:write от 1000:1 до 10⁶:1, данные не редактируются построчно, и I/O соединений — узкое место.

Переход от 1НФ до НФБК основан на ==функциональных зависимостях (ФЗ)==, к 4НФ — на ==многозначных зависимостях (МЗЗ)==, к 5НФ — на ==зависимостях соединения (ЗС)==

Алгоритм работы Recursive CTE
?
1. Вычисление нерекурсивного базового термина, результаты помещаются в итоговый набор и во временную рабочую таблицу (working table)
2. Вычисление рекурсивного термина с использованием рабочей таблицы как источника данных
3. Замена содержимого рабочей таблицы результатами шага 2
4. Повторение шагов 2-3 до тех пор, пока рабочая таблица не станет пустой

Иерархия нормальных форм: ==1НФ ⊃ 2НФ ⊃ 3НФ ⊃ НФБК ⊃ 4НФ ⊃ 5НФ== (каждая следующая - строгое подмножество предыдущей)

CROSS JOIN двух таблиц размером N и M строк даёт ==N * M== строк

Дефолтный оконный фрейм при наличии ORDER BY: ==RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW==

Типы оконных фреймов: ==ROWS== определяет границы физическим количеством строк, ==RANGE== - по логической разнице значений столбца сортировки

В чём разница между Отношением (Relation) и Таблицей (Table) в SQL?
?
Отношение - строго математический абстрактный объект без дубликатов, без порядка строк и столбцов.
Таблица SQL допускает дубликаты строк (мультимножество), позиционное упорядочивание и значения NULL, нарушая классическую реляционную теорию.

В чём отличие агрегатных функций от оконных?
?
Агрегатные функции: "схлопывают" строки в одну выходную строку.
Оконные функции: сохраняют идентичность исходных строк, выполняя вычисления для набора строк (окна).

В чём разница между EXISTS и IN?
?
EXISTS: возвращает TRUE при обнаружении первой строки - эффективнее.
IN: выполняет полное сканирование. NOT IN при наличии NULL не возвращает записей из-за трехзначной логики.

В чём разница между RANK и DENSE_RANK?
?
RANK: возвращает ранг с возможными пропусками (gaps) при наличии дубликатов.
DENSE_RANK: возвращает ранг без пропусков (gaps).

Что делает CASCADE как ссылочное действие?::Каскадное удаление/обновление зависимых кортежей при удалении/обновлении родительского кортежа.

Что делает RESTRICT как ссылочное действие?::Запрет операции при наличии зависимых кортежей.

Что делает SET NULL как ссылочное действие?::Установка NULL в зависимых кортежах при удалении/обновлении родительского кортежа.

Edge case: что происходит с SELECT * при ALTER TABLE?::Символ * разворачивается в список столбцов на этапе компиляции. Добавление/удаление столбцов приведет к несовпадению структур.

Edge case: как агрегатные функции обрабатывают NULL?::Агрегатные функции (кроме COUNT(*)) полностью игнорируют NULL. Если на входе только NULL - возвращается NULL, а не 0.

Edge case: можно ли вложить агрегат в агрегат (AVG(SUM(QTY)))?::Нет, невозможно напрямую. Требуется подзапрос в FROM.

Edge case: что происходит при NOT IN (подзапрос) с NULL?::Из-за трехзначной логики SQL запрос не вернет ни одной записи.

Edge case: что происходит при удалении родительского кортежа без ссылочного действия CASCADE?::Система прерывает транзакцию и активизирует исключение.

Edge case: что такое декомпозиция с потерями?::При обратном соединении проекций возникает утрата информации или появление фиктивных (spurious) кортежей. Декомпозиция должна выполняться по теореме Хита.

Edge case: перекрывающиеся составные потенциальные ключи и 3НФ?::Определение 3НФ игнорирует приводимую зависимость атрибута, если он - компонент другого потенциального ключа. Избыточность сохраняется. Требуется НФБК.

Edge case: ограничения на границы оконного фрейма?::frame_start не может быть UNBOUNDED FOLLOWING; frame_end не может быть UNBOUNDED PRECEDING; frame_end не может быть логически раньше frame_start.

Антипаттерн: денормализация на уровне логических переменных отношения?::Усложняет аналитические запросы, увеличивает физический размер данных, порождает аномалии обновления.

Антипаттерн: неатомарные значения в реляционном столбце (списки, JSON)?::Нарушение 1НФ. Приводит к невозможности использования стандартных реляционных операторов.

Best practice: до какой нормальной формы доводить схему?::Минимум до НФБК для устранения аномалий вставки, обновления и удаления (принцип "по одному факту в одном месте").

Best practice: CTE vs вложенные подзапросы?::Использовать CTE вместо глубоко вложенных подзапросов - разбивает запрос на читаемые блоки.

Best practice: неявное поведение оконных фреймов?::Без явного ROWS/RANGE при наличии ORDER BY фрейм = нарастающий итог (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW).
