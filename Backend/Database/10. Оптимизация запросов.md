## Суть и контекст

Оптимизация запросов — процесс преобразования декларативного SQL-запроса в эффективный план физического выполнения. Оптимизатор (Query Optimizer) использует стоимостную модель (cost-based optimization), статистику распределения данных и алгоритмы перебора планов для выбора оптимального пути доступа.

---

## Ключевые идеи

- Оптимизатор преобразует **декларативное дерево логических операций** в эквивалентное **дерево физических операций** (план выполнения) на основе оценки стоимости.
- Стоимость — количественная оценка затрат в произвольных единицах; базовая единица (1.0) — стоимость последовательного чтения одной дисковой страницы.
- **Стоимость узла** = оценочное количество операций × константа стоимости одной операции.
- Без актуальной статистики оптимизатор считает распределение равномерным и выбирает катастрофически медленные планы.
- Переписывание запросов (Query Rewrite) работает как система макросов: views разворачиваются до начала cost-based оптимизации.

---

## Определения и термины

**Оптимизатор запросов (Query Optimizer)** — компонент СУБД, преобразующий декларативное дерево логических операций запроса в эквивалентное дерево физических операций (план выполнения) на основе оценки стоимости.

**Стоимость (Cost)** — количественное выражение оценки затрат сервера на выполнение узла плана запроса в произвольных единицах. Базовая единица (1.0) — оценка стоимости последовательного чтения одной дисковой страницы.

**Статистика (Statistics)** — интегральная информация о профиле данных (количество строк, страниц, количество различных значений, гистограммы, MCV — Most Common Values), собираемая СУБД для расчёта кардинальности и стоимости операций.

---

## Как устроено

### Query Processing Pipeline

1. **Синтаксический и семантический разбор** — проверка корректности SQL.
2. **Преобразование (Rewrite / Переписывание)** — раскрытие представлений (views) в подзапросы, применение правил макросов, внедрение фильтров RLS.
3. **Каноническая форма** — применение логических законов (коммутативность, ассоциативность, идемпотентность), распределение WHERE как можно ближе к узлам сканирования для ранней фильтрации.
4. **Cost-based optimization** — динамическое программирование для перебора порядка соединений таблиц; отбрасывание декартовых произведений; для каждого узла — расчёт стоимости на основе статистики.

### Формула стоимости Seq Scan

```
cost = (relpages × seq_page_cost) + (reltuples × cpu_tuple_cost)
```

Пример: таблица tenk1 (relpages=345, reltuples=10000) → cost = 345 × 1.0 + 10000 × 0.01 = 445.00.

---

## Детали и нюансы

- **Граница переключения Seq Scan → Index Scan**: стоимость Seq Scan фиксирована (не зависит от N), стоимость Index Scan растёт линейно. Точка пересечения — обычно **5–15% строк таблицы**. При выборке большей доли оптимизатор переключается на Seq Scan.
- **Маленькие таблицы (1–2 страницы)**: планировщик игнорирует индекс и выбирает Seq Scan, так как стоимость random I/O по индексу превышает стоимость полного последовательного чтения.

---

## Примеры

```sql
-- Вывод плана запроса с оценкой стоимости
EXPLAIN SELECT * FROM tenk1 WHERE unique1 < 7000;
-- Seq Scan on tenk1  (cost=0.00..470.00 rows=7000 width=244)
--   Filter: (unique1 < 7000)

-- Запрос статистики из системного каталога
SELECT relpages, reltuples FROM pg_class WHERE relname = 'tenk1';
-- relpages=345, reltuples=10000
```

## Важно / подводные камни / best practices

- **Регулярно выполняйте ANALYZE** — без гистограмм и MCV оптимизатор занижает/завышает кардинальность и выбирает катастрофически медленные планы (Nested Loop вместо Hash Join).
- **Используйте prepared statements** — динамическая склейка SQL забивает plan cache и заставляет СУБД заново разбирать каждый запрос.
- **Не создавайте индексы на все поля** — ускоряет SELECT, но экспоненциально замедляет INSERT/UPDATE/DELETE.
- **Переписывание (Query Rewrite) бесплатно** — views разворачиваются до начала оптимизации, не создавая накладных расходов на этапе выполнения.
