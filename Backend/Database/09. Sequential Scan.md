## Суть и контекст

Sequential Scan (Seq Scan / полный просмотр) — метод физического доступа в PostgreSQL, при котором сервер последовательно считывает все страницы таблицы с диска. Планировщик выбирает Seq Scan, когда стоимость последовательного чтения ниже стоимости random I/O через индекс.

---

## Ключевые идеи

- Seq Scan считывает страницы данных **многостраничными блоками** (multi-block read) напрямую с диска, минуя индексы.
- Стоимость Seq Scan **фиксирована** — не зависит от количества выбираемых строк (N).
- Стоимость Index Scan **линейно растёт** с N. Точка пересечения — обычно 5–15% строк таблицы.
- Для маленьких таблиц (1–2 страницы) Seq Scan всегда дешевле индекса.
- PostgreSQL выделяет **ring buffer** для Seq Scan, чтобы не вымывать полезные данные из shared_buffers.

---

## Определения и термины

**Sequential Scan (Seq Scan / полный просмотр)** — метод физического доступа, при котором сервер последовательно считывает содержимое таблицы страницу за страницей с диска, перебирая все строки для проверки соответствия условиям фильтрации.

---

## Как устроено

### Алгоритм

1. Сервер считывает многостраничными блоками (multi-block read) страницы данных с диска в буферный кеш.
2. Для каждой прочитанной строки применяется **предикат Filter** — условие WHERE.
3. Несовпадающие строки отбрасываются, совпадающие — возвращаются.
4. Для больших таблиц используется **ring buffer** — жёстко ограниченная кольцевая область в shared_buffers, предотвращающая вымывание кеша.

### Формула стоимости

```
cost = (relpages × seq_page_cost) + (reltuples × cpu_tuple_cost)
```

При наличии Filter добавляется `reltuples × cpu_operator_cost` за проверку условия.

### Когда планировщик выбирает Seq Scan

- Выборка **более 5–15%** строк таблицы — Index Scan дороже из-за random I/O.
- Таблица очень маленькая (**1–2 страницы**) — полное чтение дешевле индексного доступа.
- На таблице **нет подходящего индекса**.

---

## Детали и нюансы

- **synchronize_seqscans**: при параллельном Seq Scan новое сканирование присоединяется к уже идущему, экономя I/O. Но порядок строк непредсказуем без ORDER BY.
- **enable_seqscan = off**: увеличивает штраф (стоимость) Seq Scan до бесконечности, заставляя планировщик выбирать индексные пути. Если индексов нет — Seq Scan всё равно используется, но с пометкой `Disabled: true`.

---

## Примеры

```sql
-- План с Seq Scan
EXPLAIN SELECT * FROM tenk1 WHERE unique1 < 7000;
-- Seq Scan on tenk1  (cost=0.00..470.00 rows=7000 width=244)
--   Filter: (unique1 < 7000)

-- Отключение Seq Scan для принудительного использования индекса
SET enable_seqscan = off;
```

## Важно / подводные камни / best practices

- **Seq Scan на маленькой таблице — это нормально**. Не пытайтесь форсировать индекс для таблиц в 1–2 страницы.
- **enable_seqscan = off — диагностический инструмент**, не production-настройка.
- **Без ORDER BY порядок строк при Seq Scan не гарантирован**, особенно при synchronize_seqscans.
