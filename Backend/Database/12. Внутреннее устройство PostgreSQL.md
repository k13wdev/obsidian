## Суть и контекст

Конспект охватывает внутренние механизмы PostgreSQL, критически важные для оптимизации производительности и обслуживания:

1. **Планировщик запросов (Query planner)** — стоимостная модель, выбор плана выполнения, алгоритмы соединений.
2. **EXPLAIN / ANALYZE** — инструменты анализа планов запросов: синтаксис, интерпретация вывода.
3. **WAL (Write-Ahead Logging)** — механизм обеспечения durability и crash recovery.
4. **VACUUM** — очистка мёртвых версий строк, autovacuum, защита от Transaction ID Wraparound.

---

## Ключевые идеи

- Планировщик преобразует декларативный SQL в дерево физических операций с минимальной расчётной стоимостью.
- Стоимость измеряется в условных единицах, привязанных к чтению одной дисковой страницы (seq_page_cost = 1.0).
- Три основных алгоритма соединения: Nested Loop (для малых наборов с индексом), Hash Join (для больших без сортировки), Merge Join (для предварительно отсортированных данных).
- Bitmap Index Scan — "мост" между случайным I/O и последовательным сканированием.
- WAL гарантирует: изменения файлов данных записываются на диск только после сброса журнальных записей. При crash recovery — REDO от последнего checkpoint.
- VACUUM удаляет мёртвые версии строк (dead tuples), обновляет Free Space Map и Visibility Map, замораживает старые XID.
- VACUUM FULL переписывает таблицу целиком, но требует ACCESS EXCLUSIVE lock — парализует всю работу с таблицей.

## Определения и термины

**Планировщик / Оптимизатор (Query planner)** — компонент СУБД, преобразующий логическое декларативное описание запроса в алгоритм физических операций (план запроса) с наименьшей расчётной стоимостью выполнения.

**План запроса (Query plan)** — дерево узлов плана (plan nodes), где нижние узлы сканируют базовые таблицы, а верхние выполняют объединения, агрегацию и сортировку.

**Стоимость (Cost)** — оценка затрат ресурсов на выполнение узла плана, измеряемая в произвольных единицах, привязанных к стоимости чтения одной дисковой страницы (page fetch).

**Sequential Scan (Seq Scan)** — метод доступа с последовательным чтением всех страниц таблицы для извлечения строк.

**Index Scan** — метод доступа с поиском по индексу, затем чтение строк из таблицы (heap).

**Bitmap Index Scan / Bitmap Heap Scan** — двухэтапный метод: сканирование индекса → построение битовой карты (bitmap) расположения строк → чтение heap в физическом порядке, минимизируя случайный доступ к диску.

**Nested Loop** — алгоритм соединения, при котором для каждой строки внешнего (outer) узла вызывается внутренний (inner) узел-сканер.

**Hash Join** — алгоритм соединения: данные внутреннего узла полностью загружаются в хеш-таблицу в оперативной памяти, затем внешний узел ищет совпадения по хешу.

**Merge Join** — алгоритм соединения, требующий предварительной сортировки обоих входов по ключам соединения, затем параллельное слияние.

**WAL (Write-Ahead Logging)** — стандартный метод обеспечения целостности данных, гарантирующий, что изменения файлов данных записываются на диск только после сброса журнальных записей в постоянную память.

**Checkpoint (Контрольная точка)** — точка в последовательности транзакций, в которой все грязные (изменённые) страницы данных принудительно сбрасываются на диск, а в WAL записывается специальная метка.

**VACUUM** — команда для освобождения дискового пространства, занятого устаревшими версиями строк (dead tuples), оставшимися после UPDATE и DELETE.

**VACUUM FULL** — версия очистки, требующая эксклюзивной блокировки (ACCESS EXCLUSIVE lock) и полностью переписывающая таблицу в новый файл для возврата пустого места операционной системе.

## Как устроено

### Планировщик запросов — стоимостная модель

**Источники статистики**: pg_class.reltuples, pg_class.relpages, pg_statistic.

**Формула базовой стоимости Seq Scan**:
```
(disk pages read × seq_page_cost) + (rows scanned × cpu_tuple_cost)
```
По умолчанию: `seq_page_cost = 1.0`, `cpu_tuple_cost = 0.01`. При наличии фильтра (WHERE) добавляется `rows × cpu_operator_cost`.

**Процесс выбора плана**:
1. Планировщик анализирует статистику таблиц, наличие индексов и селективность.
2. Генерируются возможные пути доступа и порядки соединений (A JOIN B→C или B JOIN C→A).
3. Для каждого пути рассчитывается стоимость (startup cost и total cost) с учётом вероятности нахождения страницы в кеше.
4. План с наименьшей итоговой стоимостью передаётся на выполнение (Executor).
5. При большом числе JOIN — эвристики и генетический алгоритм (GEQO) для сужения пространства поиска.

### EXPLAIN / EXPLAIN ANALYZE — чтение плана

**EXPLAIN**: каждая строка = узел плана. Выводятся: `cost=старт..всего`, `rows=ожидаемое число строк`, `width=средняя ширина строки в байтах`.

**EXPLAIN ANALYZE**: физически выполняет запрос. Добавляется: `actual time=старт..всего` (мс), `rows=реально полученное`, `loops=число циклов`. Общее время = `actual time × loops`.

### WAL — механика

**Компоненты**: файлы сегментов WAL (по 16 МБ, разделены на страницы по 8 КБ), LSN (Log Sequence Number = байтовое смещение), WAL buffers.

**Процесс записи**:
1. Транзакция вносит изменения в страницы данных в оперативной памяти.
2. Изменения регистрируются в WAL-буферах.
3. При COMMIT: XLogWrite переносит WAL-буферы в кеш ядра, issue_xlog_fsync сбрасывает на диск (fsync).

**Crash Recovery**:
1. Система читает WAL начиная с REDO-записи последнего checkpoint.
2. Повторяет (REDO / roll-forward) все модификации, не попавшие в файлы данных.

### VACUUM — механика очистки

1. Демон autovacuum проверяет системную статистику (количество обновлённых/удалённых строк).
2. Сканируется таблица (heap), выявляются мёртвые версии строк (dead tuples).
3. ID мёртвых строк собираются в память (до maintenance_work_mem), затем очистка индексов.
4. Свободное пространство размечается в Free Space Map для повторного использования INSERT/UPDATE.
5. Visibility Map обновляется: помечает страницы, где все строки видны всем (ускоряет Index-Only Scan).
6. Старые XID замораживаются (freezing) — защита от Transaction ID Wraparound.

## Детали и нюансы

- **Bloat при UPDATE/DELETE**: из-за MVCC версии не удаляются сразу. Обычный VACUUM помечает место как свободное, но не уменьшает размер файла на диске. Для возврата места ОС нужен VACUUM FULL (или CLUSTER).
- **VACUUM FULL** захватывает ACCESS EXCLUSIVE lock — полная блокировка таблицы для чтения и записи.
- **Параметры autovacuum**: запуск происходит, когда количество изменённых/удалённых строк превышает порог: `autovacuum_vacuum_threshold` (абсолютное число) + `autovacuum_vacuum_scale_factor` × reltuples (доля таблицы). Для снижения нагрузки на I/O используется **cost-based vacuum delay** — искусственные паузы autovacuum на основе оценки стоимости прочитанных/записанных страниц.
- **Write amplification (усиление записи)** — отношение физического объёма данных, записанных на диск, к логическому объёму изменённых данных. Причины: (1) Страничный I/O — изменение 50 байт сбрасывает всю 8 КБ страницу (мультипликатор в сотни раз). (2) WAL — данные пишутся дважды: сначала в журнал, затем в файлы данных (объём записи в файлы превосходит WAL в 5–7 раз). (3) MVCC bloat — новые версии пишутся рядом со старыми. (4) B-Tree page splits — расщепление каскадно обновляет родительские узлы. Снижение: FILLFACTOR/PCTFREE для резервирования места в страницах. Мониторинг: `pg_stat_io` (writes, write_bytes), `pg_stat_bgwriter`.
- **Write amplification в LSM-деревьях**: данные изначально пишутся компактно в memtable → immutable SSTable. Однако фоновые процессы слияния и сжатия файлов (compaction) перезаписывают данные многократно — мультипликатор записи может достигать нескольких десятков.
- **Long-running transactions**: (1) **Bloat** — VACUUM не может удалить dead tuples, если хотя бы одна транзакция со старым snapshot висит в системе → файлы разбухают. (2) **Snapshot too old на Standby** — долгие читающие транзакции конфликтуют с WAL-записями; Standby убивает транзакцию (`canceled due to old snapshots`). (3) **Lock contention каскад** — транзакция держит AccessShare; ALTER TABLE (требующий AccessExclusiveLock) встаёт в очередь; все последующие SELECT встают за ним → лавинообразный простой. Обнаружение: `age(backend_xid)` / `age(backend_xmin)` в `pg_stat_activity`; `pg_prepared_xacts` для зависших prepared-транзакций. Предотвращение: `idle_in_transaction_session_timeout`; `pg_terminate_backend(pid)`.
- **Миграции (ALTER TABLE) и даунтайм**: любой DDL (ALTER TABLE) требует ACCESS EXCLUSIVE lock — ждёт завершения всех текущих транзакций и блокирует старт новых. В источниках не описаны стратегии zero-downtime migrations (expand-contract, blue-green, shadow writes).
- **Nested Loop на больших данных**: если внутренний источник — Seq Scan (не индекс), стоимость возрастает экспоненциально (перемножение строк × стоимость полного сканирования). В таких случаях Hash Join или Merge Join эффективнее.
- **LIMIT + ORDER BY**: actual time в EXPLAIN ANALYZE может быть значительно меньше estimated cost, так как LIMIT прерывает запрос досрочно.
- **Bitmap Index Scan** работает как "мост" между Random I/O и Sequential Scan: индекс выдаёт адреса вразнобой, но bitmap сортирует физические адреса для почти последовательного чтения.
- Оптимизатор не ищет абсолютно идеальный план — используются эвристики и GEQO для отсечения декартовых произведений на ранних этапах.

## Сравнения и trade-offs

| Алгоритм соединения | Плюсы | Минусы | Когда использовать |
|---------------------|-------|--------|--------------------|
| Nested Loop | Эффективен для малых наборов; хорошо работает с Index Scan на inner | Экспоненциальная стоимость при больших inner без индекса | Малый inner set, есть индекс на ключ соединения |
| Hash Join | Не требует сортировки; эффективен для больших наборов | Требует memory для хеш-таблицы inner | Большие наборы, нет индекса, нет предварительной сортировки |
| Merge Join | Эффективен для предварительно отсортированных данных | Требует сортировки обоих входов | Данные уже отсортированы или нужен ORDER BY |

| Метод доступа | Плюсы | Минусы | Когда использовать |
|--------------|-------|--------|--------------------|
| Seq Scan | Последовательное чтение (быстрый I/O) | Читает всю таблицу | Низкая селективность запроса, нет индексов |
| Index Scan | Точечный доступ по индексу | Random I/O к heap | Высокая селективность, точечные запросы |
| Bitmap Index Scan | Сортирует адреса для последовательного heap-чтения | Две фазы (индекс + heap) | Средняя селективность, OR-условия |

| Тип VACUUM | Действие | Блокировка | Возвращает место ОС |
|-----------|---------|-----------|---------------------|
| VACUUM (обычный) | Помечает свободное место в FSM | Не блокирует чтение/запись | Нет |
| VACUUM FULL | Переписывает таблицу в новый файл | ACCESS EXCLUSIVE (полная блокировка) | Да |

## Примеры

```sql
-- План с реальным временем и статистикой буферов
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM tenk1 WHERE unique1 < 100 AND unique2 > 9000;

-- Отключение Seq Scan для тестирования альтернативных планов
SET enable_seqscan = off;
EXPLAIN SELECT * FROM unit;

-- Тестирование DML без реальных изменений
BEGIN;
EXPLAIN ANALYZE UPDATE tenk1 SET hundred = hundred + 1 WHERE unique1 < 100;
ROLLBACK;

-- Мониторинг Write Amplification
SELECT * FROM pg_stat_io;  -- writes, write_bytes, write_time

-- Обнаружение long-running транзакций
SELECT pid, age(backend_xid), age(backend_xmin), state, query
FROM pg_stat_activity WHERE state = 'idle in transaction';

-- Автоматическое завершение зависших транзакций
SET idle_in_transaction_session_timeout = '5min';

-- Ручное завершение
SELECT pg_terminate_backend(pid);
```

## Важно / подводные камни / best practices

- После массовой загрузки данных (bulk load / COPY) **строго рекомендуется** запускать ANALYZE (или VACUUM ANALYZE). Без актуальной статистики планировщик может выбрать катастрофически плохой план.
- Антипаттерн: VACUUM FULL для регулярного обслуживания — медленно, эксклюзивная блокировка. Настраивать autovacuum так, чтобы обычный VACUUM справлялся в фоне.
- Общее время узла в EXPLAIN ANALYZE = `actual time × loops` — не забывать умножать.
- EXPLAIN ANALYZE с BUFFERS показывает hit (из кеша) и read (с диска) — ключевой инструмент для диагностики I/O.
- Для тестирования DML (UPDATE/DELETE) через EXPLAIN ANALYZE — оборачивать в BEGIN/ROLLBACK, чтобы не изменять данные.

3. Lock contention каскад — ALTER TABLE ждёт транзакцию, все SELECT встают за ALTER
4. Исчерпание пула соединений

Как обнаружить long-running транзакции?::age(backend_xid)/age(backend_xmin) в pg_stat_activity; pg_prepared_xacts для зависших prepared-транзакций

Как предотвратить зависшие транзакции?::idle_in_transaction_session_timeout на уровне сервера; pg_terminate_backend(pid) вручную

Когда autovacuum запускает очистку таблицы?::Когда число изменённых/удалённых строк превышает autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × reltuples.

Что такое cost-based vacuum delay?::Механизм снижения I/O нагрузки autovacuum — искусственные паузы на основе оценки стоимости прочитанных/записанных страниц.

Write amplification в LSM-деревьях::Данные пишутся компактно (memtable → SSTable), но фоновый compaction перезаписывает данные многократно — мультипликатор может достигать нескольких десятков.

Почему ALTER TABLE вызывает даунтайм?::Требует ACCESS EXCLUSIVE lock → ждёт завершения всех транзакций → все новые SELECT встают в очередь за DDL → лавинообразный простой
