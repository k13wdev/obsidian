## Суть и контекст

Триггер — заранее откомпилированная процедура, хранящаяся вместе с базой данных и вызываемая сервером автоматически при возникновении указанных событий. Триггеры позволяют реализовать сложную логику на уровне СУБД, но их использование для бизнес-логики и ограничений целостности считается антипаттерном.

---

## Ключевые идеи

- Триггер состоит из трёх частей: **событие** (INSERT/UPDATE/DELETE), **условие** (WHEN), **действие** (хранимая процедура).
- Параметры момента срабатывания: **BEFORE** (до), **AFTER** (после), **INSTEAD OF** (вместо действия).
- Уровни срабатывания: **FOR EACH ROW** (для каждой строки) и **FOR EACH STATEMENT** (один раз на оператор).
- Эксперты строго рекомендуют **не использовать триггеры** для бизнес-логики и ограничений целостности.
- Каскадные триггеры (T1 → T2 → T1) могут привести к переполнению стека/памяти СУБД.

---

## Определения и термины

**Триггер (Trigger)** — заранее откомпилированная процедура, которая хранится вместе с базой данных и вызывается сервером автоматически при возникновении указанных событий модификации данных или системных событий.

**ROW-level триггер (FOR EACH ROW)** — триггер, активизирующийся для каждой отдельной строки, затронутой оператором.

**STATEMENT-level триггер (FOR EACH STATEMENT)** — триггер, активизирующийся один раз для данного оператора, рассматриваемого как единое целое.

**BEFORE** — триггер выполняется до применения изменений к данным. Может изменять данные NEW ROW перед их записью.

**AFTER** — триггер выполняется после применения изменений (но до COMMIT), имея доступ к итоговым состояниям.

**INSTEAD OF** — система отменяет стандартное действие и выполняет только логику триггера. Часто используется для обновляемых VIEW.

---

## Как устроено

### Алгоритм срабатывания

1. Возникает событие (INSERT/UPDATE/DELETE), попадающее под условие триггера.
2. **BEFORE**: выполняется до применения изменений. Может модифицировать NEW ROW перед записью.
3. **INSTEAD OF**: стандартное действие отменяется, выполняется только логика триггера.
4. **AFTER**: выполняется после применения изменений (но до COMMIT).
5. Если уровень FOR EACH ROW и оператор затрагивает 100 строк — триггер выполнится 100 раз.

### suppress_redundant_updates_trigger (PostgreSQL)

Встроенный триггер, который отменяет UPDATE, если OLD == NEW (значения не изменились). Экономит дисковое пространство, так как иначе UPDATE генерирует новую версию строки (dead tuple) и запись в WAL, даже если ничего не изменилось. Вносит процессорный overhead на каждую строку.

---

## Детали и нюансы

- **Каскадные триггеры**: запуск T1 может активировать T2, который снова активирует T1 (рекурсия) → переполнение стека/памяти СУБД.
- **Порядок срабатывания**: при нескольких триггерах на одно событие порядок часто не определён → трудновоспроизводимые "плавающие" баги.
- **Скрытый процедурный код**: разработчику сложнее понять логику системы; "простые" INSERT начинают вести себя непредсказуемо.
- **Оптимизатору сложно** анализировать запросы с триггерами.

---

## Примеры

```sql
-- Создание BEFORE ROW триггера
CREATE TRIGGER LONDON_SUPPLIER_INSERT
BEFORE INSERT ON LONDON_SUPPLIER
FOR EACH ROW
INSERT INTO S ... ;
```

Если INSERT вставляет 100 строк, код триггера выполнится 100 раз (FOR EACH ROW).

## Важно / подводные камни / best practices

- **Не используйте триггеры для бизнес-логики и ограничений целостности** — отдавайте предпочтение API, декларативным ограничениям (CHECK, RLS) или слою приложения.
- **Остерегайтесь каскадных триггеров** — рекурсия T1 → T2 → T1 приводит к переполнению памяти.
- **suppress_redundant_updates_trigger** — полезен при массовых обновлениях, экономит WAL и dead tuples ценой CPU overhead.
