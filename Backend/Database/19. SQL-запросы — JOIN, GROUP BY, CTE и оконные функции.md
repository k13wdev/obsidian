## Суть и контекст

Конспект охватывает ключевые конструкции языка SQL для выборки и агрегации данных:

1. **JOIN** — операция соединения таблиц (INNER, LEFT, RIGHT, FULL, CROSS, SELF) и правила выбора типа.
2. **GROUP BY / HAVING / WHERE** — группировка, фильтрация строк и фильтрация групп.
3. **CTE (Common Table Expressions)** — именованные подзапросы, рекурсия и data-modifying CTE.
4. **Оконные функции (Window Functions)** — вычисления по набору строк без схлопывания результата.
5. **SELECT \*** — почему это антипаттерн в production-коде.

---

## Ключевые идеи

- JOIN — операция получения комбинации двух таблиц на основе общих значений; допускается сцепление и вложенность соединений.
- CROSS JOIN порождает декартово произведение (N × M строк). INNER JOIN возвращает только строки, удовлетворяющие условию. LEFT/RIGHT/FULL OUTER JOIN дополняют результат строками без совпадений (с NULL).
- SELF JOIN — соединение таблицы с самой собой; обязательны псевдонимы (aliases).
- WHERE фильтрует **строки до** группировки и агрегации. HAVING фильтрует **группы после** агрегации.
- CTE (WITH-запросы) определяют временные именованные таблицы, существующие только в рамках одного запроса. Поддерживают рекурсию (RECURSIVE) и модификацию данных (INSERT/UPDATE/DELETE внутри WITH).
- Оконные функции вычисляются **после** GROUP BY, агрегации и HAVING — строки не схлопываются.
- PARTITION BY делит строки на независимые группы для обработки. ORDER BY внутри окна определяет порядок. Фрейм окна (window frame) задаёт диапазон строк для вычисления.
- SELECT * в production — антипаттерн: добавление/удаление столбца ломает прикладной код, ожидающий строгую структуру результата.

## Определения и термины

### JOIN (Соединение)

**JOIN (Соединение)** — таблица, производная от двух других (реальных или производных) таблиц в соответствии с правилами конкретного типа соединения. Операция получения комбинации двух таблиц на основе общих значений в общих столбцах.

**CROSS JOIN (Декартово произведение)** — соединение, при котором для каждой возможной комбинации строк из T1 и T2 формируется строка, состоящая из всех столбцов T1, за которыми следуют все столбцы T2. Если T1 имеет N строк, а T2 — M строк, результат содержит N × M строк.

**INNER JOIN (Внутреннее соединение)** — для каждой строки R1 из T1 в объединённой таблице создаётся строка для каждой строки из T2, удовлетворяющей условию соединения с R1.

**LEFT OUTER JOIN (Левое внешнее соединение)** — сначала выполняется внутреннее соединение, затем для каждой строки из T1, не удовлетворяющей условию соединения ни с одной строкой из T2, добавляется строка с NULL-значениями в столбцах T2.

**RIGHT OUTER JOIN (Правое внешнее соединение)** — обратная операция левому внешнему соединению. Для каждой строки из T2, не имеющей совпадений в T1, добавляется строка с NULL-значениями в столбцах T1.

**FULL OUTER JOIN (Полное внешнее соединение)** — включает результат внутреннего соединения, плюс несопоставленные строки из T1 (с NULL для T2), плюс несопоставленные строки из T2 (с NULL для T1).

**SELF JOIN (Самосоединение)** — соединение таблицы с самой собой. Требует обязательного использования псевдонимов (aliases) таблиц для устранения неоднозначности имён столбцов.

### GROUP BY / HAVING / WHERE

**GROUP BY** — конструкция, объединяющая строки с одинаковыми значениями в указанных столбцах в одну групповую строку, представляющую все строки в группе.

**WHERE** — фильтрует входные строки **до** вычисления групп и агрегатов. Определяет, какие строки попадут в агрегатное вычисление.

**HAVING** — фильтрует сгруппированные строки **после** вычисления групп и агрегатов. Логическое выражение в HAVING должно быть однозначным для каждой группы. WHERE отбрасывает отдельные строки, HAVING отбрасывает целые группы.

### CTE и оконные функции

**CTE (Common Table Expressions / WITH-запросы)** — вспомогательные запросы, определяющие временные таблицы, которые существуют только во время выполнения одного запроса.

**Оконная функция (Window function)** — функция, выполняющая вычисления для набора строк таблицы, которые каким-либо образом связаны с текущей строкой (фрейм окна), не вызывая при этом объединения строк в одну выходную строку.

## Как устроено

### Порядок выполнения SQL-запроса (логический конвейер)

1. **FROM / JOIN** — определение источников данных и соединения.
2. **WHERE** — фильтрация строк до группировки.
3. **GROUP BY** — формирование групп.
4. **Агрегатные функции** — вычисление SUM, AVG, COUNT и т.д.
5. **HAVING** — фильтрация групп после агрегации.
6. **Оконные функции** — вычисление OVER(...) по уже сгруппированным строкам.
7. **SELECT** — выбор столбцов результата.
8. **ORDER BY** — финальная сортировка.

### Механика JOIN

INNER JOIN возвращает только пересечение. LEFT JOIN сохраняет все строки левой таблицы. RIGHT JOIN — правой. FULL JOIN — обеих. CROSS JOIN — декартово произведение без условия.

Для SELF JOIN таблица соединяется сама с собой через псевдонимы:

```sql
SELECT w1.city, w1.temp_lo AS low, w1.temp_hi AS high,
       w2.city, w2.temp_lo AS low, w2.temp_hi AS high
    FROM weather w1 JOIN weather w2
        ON w1.temp_lo < w2.temp_lo AND w1.temp_hi > w2.temp_hi;
```

Псевдонимы `w1` и `w2` обязательны для различения левой и правой стороны соединения.

### Механика оконных функций

1. Конструкция `OVER` синтаксически отличает оконную функцию от обычной агрегатной.
2. `PARTITION BY` делит строки на группы (партиции) для независимой обработки.
3. `ORDER BY` внутри окна определяет порядок обработки строк внутри партиции.
4. **Фрейм окна (Window frame)** — для каждой строки существует набор строк внутри её партиции:
   - Если есть `ORDER BY` — фрейм по умолчанию: от начала партиции до текущей строки плюс все строки, равные текущей.
   - Если `ORDER BY` опущен — фрейм включает все строки партиции.
5. Оконные функции вычисляются **строго после** GROUP BY, агрегации и HAVING.

### Механика CTE (WITH)

1. Запрос `WITH` определяет именованное подвыражение, подставляемое или материализуемое в основном запросе.
2. Позволяет разбивать сложные запросы без глубокой вложенности подзапросов.
3. **Рекурсия**: при модификаторе `RECURSIVE` запрос WITH ссылается на свой собственный вывод (применяется для обхода графов, суммирования последовательностей).
4. **Data-Modifying CTE**: внутри WITH можно использовать INSERT, UPDATE, DELETE, MERGE — несколько операций модификации в рамках одного запроса (например, удалить строки и вернуть их через RETURNING для вставки в лог-таблицу).

## Детали и нюансы

### Условия фильтрации в OUTER JOIN

Ограничение, помещённое в `WHERE`, обрабатывается **после** соединения. Если сделать LEFT JOIN и указать `WHERE t2.value = 'xxx'`, то строки из левой таблицы, не имеющие совпадений (содержащие NULL в `t2.value`), будут отброшены условием WHERE. Левое внешнее соединение фактически **превратится во внутреннее (INNER JOIN)**.

Правильное решение — помещать условие в `ON`:

```sql
-- Неправильно: LEFT JOIN превращается в INNER JOIN
SELECT * FROM t1 LEFT JOIN t2 ON t1.id = t2.id WHERE t2.value = 'xxx';

-- Правильно: фильтрация в ON сохраняет семантику LEFT JOIN
SELECT * FROM t1 LEFT JOIN t2 ON t1.id = t2.id AND t2.value = 'xxx';
```

### Ограничения оконных функций

- Оконные функции **запрещены** в WHERE, GROUP BY и HAVING — парсер вернёт синтаксическую ошибку.
- Оконные функции разрешены **только** в списке SELECT и в ORDER BY.
- Включение агрегатной функции в аргументы оконной функции допустимо: `SUM(AVG(x)) OVER()`. Обратное — запрещено.

### Проблема SELECT * в production

Добавление или удаление столбца в таблице немедленно изменит структуру возвращаемого результата (количество и состав колонок). Это приводит к непредсказуемому поведению и поломке прикладного кода, ожидающего строгую структуру. Во встроенных операторах (embedded SQL) и динамическом SQL символ `*` меняет значение при `ALTER TABLE`, создавая скрытую опасность несовпадения типов и переменных привязки.

## Сравнения и trade-offs

| Конструкция | Что фильтрует | Когда срабатывает | Когда использовать |
|-------------|---------------|-------------------|--------------------|
| WHERE | Отдельные строки | До GROUP BY и агрегатов | Фильтрация исходных данных |
| HAVING | Целые группы | После GROUP BY и агрегатов | Фильтрация по результатам агрегации |

| Тип JOIN | Результат | Когда использовать |
|----------|-----------|-------------------|
| INNER JOIN | Только совпавшие строки | Нужны только строки с данными в обеих таблицах |
| LEFT JOIN | Все из левой + совпадения из правой | Сохранить все записи основной таблицы |
| RIGHT JOIN | Все из правой + совпадения из левой | Сохранить все записи присоединяемой таблицы |
| FULL JOIN | Все из обеих + совпадения | Объединение с сохранением всех данных |
| CROSS JOIN | Декартово произведение N × M | Генерация всех комбинаций |
| SELF JOIN | Таблица сама с собой | Сравнение строк одной таблицы |

## Примеры

### WHERE vs HAVING в одном запросе

```sql
SELECT product_id, p.name, (sum(s.units) * (p.price - p.cost)) AS profit
    FROM products p LEFT JOIN sales s USING (product_id)
    WHERE s.date > CURRENT_DATE - INTERVAL '4 weeks'
    GROUP BY product_id, p.name, p.price, p.cost
    HAVING sum(p.price * s.units) > 5000;
```

`WHERE` фильтрует только продажи за последние 4 недели **до** группировки. `HAVING` отбрасывает группы, где суммарная валовая выручка не превышает 5000.

### CTE для разбиения сложного запроса

```sql
WITH regional_sales AS (
    SELECT region, SUM(amount) AS total_sales
    FROM orders
    GROUP BY region
), top_regions AS (
    SELECT region
    FROM regional_sales
    WHERE total_sales > (SELECT SUM(total_sales)/10 FROM regional_sales)
)
SELECT region, product,
       SUM(quantity) AS product_units,
       SUM(amount) AS product_sales
FROM orders
WHERE region IN (SELECT region FROM top_regions)
GROUP BY region, product;
```

Определение `regional_sales` используется внутри `top_regions`, а `top_regions` — в основном запросе. Исключает необходимость двойной вложенности подзапросов.

### Оконная функция: сравнение зарплаты со средней по отделу

```sql
SELECT depname, empno, salary,
       avg(salary) OVER (PARTITION BY depname)
FROM empsalary;
```

`avg(salary)` вычисляется независимо для каждого окна, ограниченного `PARTITION BY depname`. Выходные строки **не схлопываются**, сохраняя `empno`.

## Важно / подводные камни / best practices

- Условия фильтрации для OUTER JOIN помещайте в `ON`, а не в `WHERE` — иначе LEFT JOIN превращается в INNER JOIN.
- Оконные функции нельзя использовать в WHERE, GROUP BY, HAVING — только в SELECT и ORDER BY.
- SELECT * в production — антипаттерн: ALTER TABLE ломает прикладной код.
- CTE с RECURSIVE позволяет обходить графы и иерархии, но требует условия остановки рекурсии.
- Data-Modifying CTE (INSERT/UPDATE/DELETE внутри WITH) выполняются в рамках одного запроса — удобно для операций "удалить и залогировать".
