## Суть и контекст

B-Tree (B-дерево) — основная структура данных для индексов в реляционных СУБД. Это сбалансированное дерево, оптимизированное для дисковых операций: каждый узел соответствует дисковой странице, а глубина дерева обычно составляет 3–4 уровня даже для сотен миллионов записей.

---

## Ключевые идеи

- B-Tree — направленный ациклический граф (сбалансированное дерево), узел которого представляет собой список пар «ключ—значение», перемежающихся ссылками на поддеревья, где все ключи уникальны и упорядочены.
- Каждый узел размещается в выделенной дисковой странице (обычно 8 КБ).
- Глубина дерева = логарифм по основанию N от количества ключей. При размере указателя 8 байт, ключа 30 байт и страницы 8 КБ, на 4-м уровне размещается до 8 000 000 страниц (~60 ГБ, >300 млн объектов).
- **B+Tree** — модификация, в которой значения содержатся только в листовых узлах, а листья образуют **двунаправленный список** для эффективных range-запросов.
- **B*Tree** — модификация, при переполнении узла пытающаяся перенести часть ключей в соседний узел вместо безусловного разделения, что увеличивает плотность хранения.

---

## Определения и термины

**B-Tree (B-дерево)** — направленный ациклический граф (сбалансированное дерево), узел которого представляет собой список пар «ключ—значение», перемежающихся ссылками на поддеревья, где все ключи уникальны и упорядочены.

**B+Tree (B+-дерево)** — модификация B-дерева, в которой значения содержатся только в листовых узлах (в промежуточных — только ключи), а листовые узлы образуют двунаправленный список.

**B*Tree** — модификация B-дерева, алгоритм которой при переполнении узла пытается перенести часть ключей в соседний узел вместо безусловного разделения, что увеличивает плотность хранения данных.

---

## Как устроено

### Структура узлов

- **Корневой узел (Root)** — верхний уровень дерева.
- **Промежуточные узлы** — содержат ключи и ссылки на поддеревья. Ключи в поддереве, на которое ссылается указатель между K_n и K_{n+1}, лежат в промежутке [K_n; K_{n+1}).
- **Листовые узлы (Leaf)** — содержат ключи и физические адреса строк (TID). В B+Tree ключи могут дублироваться в дочерних узлах для маршрутизации.
- Количество потомков у каждого узла на единицу больше, чем N (количество ключей, помещающихся в дисковую страницу).

### Алгоритм поиска

1. Чтение корневой страницы.
2. Сравнение искомого значения с ключами для выбора интервала и соответствующего указателя на дочерний узел.
3. Спуск по дереву до достижения листового узла (Leaf).
4. Поиск ключа в листовом узле и извлечение физического адреса (номера строки).
5. **В B+Tree** для range-запросов — после нахождения первого листа горизонтальное перемещение по двунаправленному списку листовых узлов.

### Алгоритм вставки и расщепления (Splitting)

1. Спуск до листового узла, в который должен попасть ключ.
2. Если узел заполнен менее максимума (с учётом FILLFACTOR), ключ вставляется с сохранением сортировки.
3. Если узел заполнен — **расщепление**: он делится пополам; половина ключей остаётся, половина переносится в новый узел.
4. Разделяющий ключ поднимается в узел-предок, обновляются границы указателей.
5. Если предок также переполнен, деление **каскадно** доходит до корня. При делении корня создаётся новый корневой узел с единственным ключом, и глубина дерева увеличивается на 1.

### Алгоритм удаления и балансировки

1. Удаление ключа из листового узла.
2. Если заполнение узла падает ниже допустимого порога (менее 50% для B-Tree, менее 2/3 для B*Tree), инициируется слияние.
3. Алгоритм пытается объединить узел с соседним.
4. Разделяющий ключ удаляется из узла-предка. Процесс может каскадно дойти до корня.

### Блокировки

Для B-Tree применяются кратковременные (short-term) блокировки на уровне страниц (page-level locks), которые снимаются сразу после чтения/записи строки индекса, предотвращая deadlocks.

## Детали и нюансы

- **Монотонно возрастающие ключи (автоинкремент)**: массированная вставка направляется исключительно в крайний правый листовой узел → высокая конкуренция за одну страницу → падение производительности. Решение: индекс по инвертированному представлению ключа (REVERSE в Oracle), равномерно размазывающий вставки, ценой потери эффективности range-запросов.
- **PCTFREE / FILLFACTOR**: оставляет резерв пустого места в страницах при начальном построении; при последующих вставках узлы не будут немедленно расщепляться.
- **Глубина 3–4 уровня** покрывает сотни миллионов объектов — на практике верхние уровни кэшируются в памяти.

---

## Сравнения и trade-offs

| Вариант | Плюсы | Минусы | Когда использовать |
|---------|-------|--------|--------------------|
| B-Tree | Логарифмический поиск O(log_N K), поддержка порядка, range-запросы | Медленнее O(1) для точного поиска | По умолчанию для большинства запросов |
| B+Tree | Эффективные range-запросы (двунаправленный список листьев) | Дублирование ключей в промежуточных узлах | PostgreSQL, MySQL (InnoDB) |
| B*Tree | Выше плотность хранения (перенос ключей вместо расщепления) | Сложнее алгоритм | Когда критична плотность |

## Примеры

**Визуализация поиска числа 61**: корневой блок (ROOT) с диапазонами → переход по ссылке 50–100 → промежуточный блок → переход по ссылке 50–65 → листовой узел → ключ 61, физический номер строки 10.

## Важно / подводные камни / best practices

- **Используйте FILLFACTOR** для таблиц с частым обновлением — предотвращает преждевременное расщепление узлов.
- **Осторожно с автоинкрементными ключами** в высоконагруженных системах — hot spot на правом листе.
- **B-Tree page-level locks кратковременны** — безопаснее для конкурентного доступа, чем Hash.
