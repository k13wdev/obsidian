## Суть и контекст

Hash таблица в контексте баз данных — структура данных, лежащая в основе Hash индексов. Обеспечивает константное время поиска O(1) для операций точного равенства, но не поддерживает порядок и range-запросы. В PostgreSQL реализована как дисковая структура с корзинами (buckets) и страницами переполнения (overflow pages).

---

## Ключевые идеи

- Hash индекс сохраняет **32-битный хеш-код**, вычисленный из значения индексируемой колонки, и поддерживает **исключительно операции проверки на равенство**.
- Дисковая структура: мета-страница, страницы корзин (buckets), страницы переполнения (overflow).
- При переполнении корзины используется механизм overflow pages (аналог цепочек / chaining).
- Hash индексы блокируют **всю корзину** (bucket-level lock) на время операции — в отличие от B-Tree с кратковременными page-level locks.

---

## Определения и термины

**Hash индекс** — индекс, сохраняющий 32-битный хеш-код, вычисленный из значения индексируемой колонки, и поддерживающий исключительно операции проверки на равенство.

**Корзины (Buckets)** — дисковые страницы, содержащие ключи и физические идентификаторы строк (TID).

**Страницы переполнения (Overflow pages)** — дополнительные страницы, выделяемые при нехватке места в основной корзине; связываются с корзиной подобно связному списку.

---

## Как устроено

### Дисковая структура Hash индекса

1. **Мета-страница (meta)** — содержит метаданные индекса, ссылается на корзины.
2. **Страницы корзин (buckets)** — хранят ключи и TID (физические адреса строк).
3. **Страницы переполнения (overflow)** — при переполнении корзины ключ и TID записываются в дополнительную страницу, связанную с основной корзиной.

### Алгоритм поиска

1. Вычисляется 32-битный хеш-код от значения колонки.
2. В мета-странице определяется нужная корзина.
3. Считывается дисковая страница корзины, выполняется поиск ключа и извлечение TID.
4. Если в корзине есть overflow pages — последовательный просмотр связанных страниц.

### Блокировки

Hash индексы используют блокировки на уровне **всей корзины** (hash-bucket-level locks). Они удерживаются дольше, чем page-level locks в B-Tree (пока не будет обработана вся корзина), что увеличивает вероятность **deadlocks**.

---

## Детали и нюансы

- **Только операция равенства (=)**: Hash индекс **не поддерживает** операторы <, >, BETWEEN, LIKE. Оптимизатор игнорирует Hash индекс и выполняет Seq Scan.
- **Bucket-level locks** увеличивают вероятность дедлоков по сравнению с B-Tree.
- В источниках не найдена информация по открытой адресации (open addressing), load factor и rehashing в контексте СУБД.

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Hash индекс | O(1) для точного равенства | Не поддерживает range-запросы; bucket-level locks → deadlocks | Только lookups по точному значению в низконагруженных сценариях |
| B-Tree индекс | O(log_N K); поддержка порядка и range-запросов; кратковременные page-level locks | Медленнее O(1) для точного поиска | Универсальный выбор по умолчанию |

---

## Примеры

```sql
-- Создание Hash индекса в PostgreSQL
CREATE INDEX name ON table USING HASH (column);
```

Явное указание `USING HASH` переопределяет метод доступа по умолчанию (B-Tree) на хеш-таблицу.

## Важно / подводные камни / best practices

- **Hash индексы не работают для range-запросов** — оптимизатор проигнорирует индекс и выполнит Seq Scan.
- **Bucket-level locks менее безопасны** — увеличенная вероятность deadlocks по сравнению с B-Tree.
- **B-Tree предпочтительнее в большинстве случаев** — Hash индекс оправдан только для чистых lookups по точному значению.
