## Суть и контекст

Конспект охватывает нереляционные базы данных — их модели данных, внутреннее устройство и паттерны применения:

1. **NoSQL** — класс СУБД с горизонтальным масштабированием за счёт ослабления ACID.
2. **Document DB (MongoDB)** — хранение документов в BSON, гибкая схема, моделирование связей.
3. **Key-Value (Redis)** — in-memory хранилище структур данных, кэширование, persistence, транзакции.
4. **Wide-column (Apache Cassandra)** — семейства колонок, partitioning key, eventual consistency.
5. **LSM-Tree** — структура данных, оптимизированная для записи (append-only).

---

## Ключевые идеи

- NoSQL жертвует строгими ACID-гарантиями в пользу BASE для горизонтального масштабирования.
- Redis — однопоточный in-memory сервер; любая команда или Lua-скрипт выполняется атомарно.
- MongoDB хранит документы в BSON (двоичный JSON): имена полей дублируются в каждом документе, что увеличивает объём по сравнению с реляционной моделью.
- Cassandra: первичный ключ = partitioning key (привязка к узлу) + clustering key (порядок сортировки внутри узла).
- LSM-Tree оптимизирован для записи: данные записываются последовательно и не изменяются in-place. Удаление — через маркеры (tombstones).
- Redis persistence: RDB (снепшоты через fork + Copy-on-Write) и AOF (append-only journal с fsync).
- Паттерны кэширования: cache-aside (приложение управляет), write-through (синхронная запись), write-behind (асинхронная).
- Redis Cluster: 16384 хеш-слота, алгоритмический шардинг, протокол Gossip.
- WATCH/MULTI/EXEC — оптимистичная блокировка в Redis; при высокой конкуренции → Lua-скрипты эффективнее в 4+ раза.

## Определения и термины

### Общие

**NoSQL** — класс СУБД, основным требованием к которым является горизонтальное масштабирование за счёт отказа от реляционной модели и строгих ACID в пользу BASE.

**LSM-дерево (Log-Structured Merge-tree)** — структура данных, в которой данные записываются на диск последовательно и не изменяются in-place; накапливаются в памяти и вытесняются в новые файлы.

**Маркер удаления (Tombstone)** — специальная запись в буфере LSM-дерева, скрывающая старые значения ключа на диске вместо физического удаления.

**Фильтр Блума (Bloom filter)** — вероятностная структура данных, позволяющая быстро определить, что ключа в файле гарантированно нет.

### Redis

**Redis (Remote Dictionary Service)** — опенсорсный in-memory сервер структур данных типа ключ-значение.

**Однопоточная архитектура** — обработка команд и скриптов в один поток; любая команда выполняется атомарно, блокируя другие до завершения.

**Ziplist** — компактное сериализованное представление данных (LIST, HASH, ZSET) в памяти; ~2 байта на элемент вместо 21 байт в двусвязном списке.

**Cache-aside** — паттерн: приложение проверяет кеш, при промахе обращается к БД, затем сохраняет в кеш. Ответственность за консистентность на приложении.

**Write-through** — паттерн: приложение пишет в кеш, кеш синхронно обновляет хранилище.

**Write-behind (Write-back)** — паттерн: значение обновляется в кеше, управление возвращается немедленно, обновление хранилища — асинхронно.

**TTL (Time-to-live)** — время до автоматического удаления ключа. Применяется только к ключам целиком, не к элементам внутри структур.

### MongoDB

**Документоориентированная БД (Document DB)** — хранилище ключ-значение, где значение — документ с описанием собственной структуры (JSON, BSON, XML).

### Cassandra

**Хранилище семейств колонок (Wide-column store)** — система, где значение ключа — множество пар «колонка — значение», структура хранится отдельно от значений.

## Как устроено

### Redis — архитектура и структуры данных

- **Strings**: байтовые строки, целые числа (32/64-бит), float IEEE 754. Макс. 512 МБ.
- **Lists**: двусвязные списки или ziplist. FIFO-очереди через LPUSH + RPOP.
- **Sets**: неупорядоченные уникальные строки. SISMEMBER, операции объединения/пересечения/разности.
- **Sorted Sets (ZSET)**: уникальные строки, упорядоченные по score (float IEEE 754). При равных scores — по бинарному значению строк.
- **Hashes**: неупорядоченные хеш-таблицы ключ→значение с атомарным обновлением отдельных полей.

### Redis — Persistence

**RDB (снепшоты)**:
1. BGSAVE → fork → дочерний процесс.
2. Родитель и дочерний используют общую память (Copy-on-Write).
3. При изменении данных — ядро ОС копирует изменённые страницы.
4. Дочерний записывает снимок на диск.

**AOF (Append-Only File)**:
1. Команды записи буферизуются.
2. Сбрасываются на диск через fsync (appendfsync: always / everysec / no).
3. При превышении порога → BGREWRITEAOF: переписывает лог, удаляя устаревшие команды.

**При одновременном RDB и AOF**: при перезапуске Redis загружает данные из AOF (более полная версия).

### Redis — Eviction Policies

При заполнении maxmemory:
- `allkeys-lru` — удаляет наименее недавно использованные (все ключи)
- `allkeys-lfu` — удаляет наименее часто используемые
- `allkeys-random` — случайное удаление
- `volatile-lru/lfu/random` — только ключи с TTL

LRU и LFU в Redis — **аппроксимация**: случайная выборка → удаление лучшего кандидата.

### Redis — Репликация и Cluster

**Master-Slave**:
1. Slave отправляет SYNC → Master делает BGSAVE + буферизует команды.
2. Master отправляет снепшот → Slave загружает.
3. Master передаёт буферизованные команды → постоянная трансляция.

**Redis Cluster**: 16384 хеш-слотов. Ключ → hash → modulo 16384 → узел. При решардинге переносятся целые слоты. Gossip-протокол для коммуникации узлов.

### Redis — Транзакции и Lua

**WATCH/MULTI/EXEC** (оптимистичная блокировка):
1. WATCH ключи → MULTI → команды в очередь → EXEC.
2. Если ключи изменены сторонним клиентом → транзакция сбрасывается (WatchError).

**Lua-скрипты**: SCRIPT LOAD → SHA1 → EVALSHA. Выполняются атомарно как единая команда.

### MongoDB — моделирование связей

- **Embedding**: дочерние объекты вложены в родительский (для кардинальности до ~10).
- **Массив ObjectID**: родитель хранит массив ID дочерних (десятки-сотни).
- **Ссылка на родителя**: дочерний документ содержит ObjectID родителя + вторичный индекс (тысячи+).

### LSM-Tree — механика

**Запись**: данные → буфер L0 в памяти (отсортированы) → при переполнении вытесняются на диск как файл L1.

**Поиск**: буфер → кеш → фильтры Блума → индексы файлов (от свежих к старым).

**Compaction**: фоновый процесс сливает файлы, оставляет последнюю версию ключа, удаляет tombstones на нижнем уровне.

### Cassandra — первичный ключ

- **Partitioning key** → привязка к физическому узлу кластера.
- **Clustering key** → порядок сортировки строк внутри узла.
- **Статические колонки** — одно значение для всего partitioning key.

Запись: любой узел может принять запрос → отправляет всем хранящим данные → возвращает успех при `CONSISTENCY` подтверждений (от ANY до ALL).

## Детали и нюансы

- **Ziplist при 50K-100K элементов**: производительность падает с ~50 000 ops/sec до 553 ops/sec из-за перекодирования и копирования памяти.
- **AOF + appendfsync always + SSD**: экстремальное write amplification → смерть SSD за несколько месяцев.
- **BGSAVE на 20 ГБ в Xen (AWS EC2)**: заморозка Redis на 4-6 секунд (fork: 200-300 мс/ГБ).
- **WATCH при высокой конкуренции**: 5 читателей + 5 писателей → 161 000 неудачных повторов, задержка 498 мс. Lua снижает до <3 мс.
- **Lua-скрипт с записью нельзя остановить** SCRIPT KILL → нужен SHUTDOWN NOSAVE (потеря данных).
- **Redis Master-Master**: в Open Source → бесконечный цикл, 100% CPU. Истинный multi-master только в Redis Enterprise (CRDT).
- **Поиск по неиндексированным атрибутам в LSM**: полный просмотр всех файлов — крайне медленно.
- **Конкурентное обновление в Cassandra**: часть изменений из одной транзакции, часть из другой — нет атомарности без Lightweight Transactions (Paxos).

## Сравнения и trade-offs

| Тип NoSQL | Модель данных | Плюсы | Минусы | Когда использовать |
|-----------|--------------|-------|--------|--------------------|
| Key-Value (Redis) | Ключ → значение (структуры данных) | In-memory скорость, атомарность, rich data types | Объём ограничен RAM, нет JOIN | Кэширование, сессии, очереди, счётчики |
| Document (MongoDB) | Документы BSON в коллекциях | Гибкая схема, вложенность | Дублирование имён полей, нет ACID между документами | Каталоги, CMS, профили |
| Wide-column (Cassandra) | Семейства колонок | Горизонтальное масштабирование, запись | Нет JOIN, ограниченная консистентность | Временные ряды, IoT, логи |

| Persistence Redis | Плюсы | Минусы |
|-------------------|-------|--------|
| RDB (снепшоты) | Компактный файл, быстрое восстановление | Потеря данных между снепшотами |
| AOF (журнал) | Потеря максимум 1 сек (everysec) | Больше файл, медленнее восстановление |

| Паттерн кэширования | Плюсы | Минусы |
|---------------------|-------|--------|
| Cache-aside | Простота, контроль на стороне приложения | Приложение отвечает за консистентность |
| Write-through | Кеш и БД всегда синхронны | Latency записи (синхронная) |
| Write-behind | Минимальная latency записи | Риск устаревших данных при прямом чтении из БД |

## Важно / подводные камни / best practices

- Размер ziplist: 500-2000 элементов, значение ≤ 128 байт (рекомендуется 64). Превышение → катастрофическое падение производительности.
- Использовать `appendfsync everysec` — компромисс скорости и надёжности (потеря макс. 1 сек).
- **Никогда** `appendfsync always` на SSD — write amplification убьёт диск.
- Lua-скрипты вместо WATCH/MULTI/EXEC в высококонкурентной среде: 1 RTT, нет race conditions, 4x+ производительность.
- Для больших кардинальностей в MongoDB — ссылки на родителя + вторичный индекс (не embedding).
- Антипаттерн: ожидание строгих ACID от NoSQL — используйте реляционную СУБД для транзакционной бизнес-логики.
- Антипаттерн: MongoDB без схемы в продакшне — использовать JSON Schema для валидации.
- Перед записью в Redis агрегировать данные локально (миллионы инкрементов → несколько ZADD).
- Redis Master-Master в Open Source = бесконечный цикл. Multi-master только через Redis Enterprise (CRDT).
