## Суть и контекст

Тема охватывает механизмы управления конкурентным доступом к данным в Redis: транзакции через MULTI/EXEC с оптимистичной блокировкой WATCH, конвейеризацию команд (pipeline) для снижения сетевых задержек, а также Lua scripting как способ атомарного выполнения произвольной логики на стороне сервера. Эти инструменты решают проблему race condition при параллельном доступе нескольких клиентов к общим ключам.

---

## Ключевые идеи

- Redis реализует оптимистичную блокировку через WATCH, а не пессимистичную (как SELECT FOR UPDATE в SQL)
- Транзакции MULTI/EXEC в Redis не поддерживают ROLLBACK — все команды выполняются до конца, даже если одна вернёт ошибку
- Pipeline (конвейеризация) буферизирует команды на клиенте и отправляет их единым пакетом, снижая число сетевых round-trips
- Pipeline может быть транзакционным (с MULTI/EXEC) и не-транзакционным
- Lua-скрипты выполняются атомарно — никакие другие команды от других клиентов в это время не обрабатываются
- Lua-скрипты функционально идентичны хранимым процедурам (stored procedures) в SQL-базах
- При высокой конкуренции Lua-скрипты радикально превосходят WATCH/MULTI/EXEC по производительности: 0 повторов и задержка <1 мс против 161 000 повторов и задержки до 498 мс
- WATCH в Redis аналогичен CAS-операциям в процессорах и базах данных

---

## Определения и термины

**Оптимистичная блокировка (Optimistic locking)** — метод управления конкурентным доступом, при котором база данных не блокирует данные (в отличие от pessimistic locking, как SELECT FOR UPDATE), а уведомляет клиента в случае модификации данных другим процессом. В Redis реализуется через команду WATCH. Клиент, получивший уведомление об изменении, должен повторить транзакцию или прервать её.

**Race condition (Состояние гонки)** — ситуация параллельного выполнения команд без синхронизации, приводящая к непредсказуемому чередованию (interleaving) операций. Например, параллельные команды инкремента и декремента счётчика без транзакций могут привести к бесконтрольному росту значения счётчика.

**CAS (Compare-and-set / Compare-and-swap)** — операция процессора, сравнивающая значение перед его установкой; в контексте баз данных — условное обновление, сравнивающее штамп версии (version stamp) или отслеживающее изменения состояния через WATCH. Если версия/состояние изменились, операция обновления отклоняется.

**Pipeline (Конвейеризация)** — механизм буферизации команд на стороне клиента для отправки их единым пакетом в Redis и получения всех ответов разом. Может быть транзакционным (с обёрткой MULTI/EXEC) и не-транзакционным. Снижает количество сетевых задержек (round-trips).

**DISCARD** — команда, сбрасывающая текущее соединение путём очистки очереди команд транзакции, отменяющая WATCH. Вызывается, если клиент начал транзакцию через MULTI, но решил не выполнять EXEC.

## Как устроено

### Механика MULTI/EXEC/WATCH

1. Клиент вызывает WATCH для целевого ключа (ключей)
2. Клиент считывает текущие данные (GET, HGET) для принятия логических решений
3. Клиент вызывает команду MULTI
4. Клиент передаёт команды модификации. Redis помещает их в очередь без немедленного выполнения
5. Клиент вызывает EXEC
6. Если отслеживаемые ключи не были изменены сторонним клиентом, Redis выполняет очередь команд последовательно, без прерывания на команды других клиентов
7. Если отслеживаемые ключи были изменены, EXEC отклоняется, в клиенте (Python) генерируется WatchError

### Не-транзакционные pipelines

1. Вызывается инициализация `pipe = conn.pipeline(False)` (в Python)
2. В очередь локального клиента складываются команды (hset, zadd, zincrby)
3. Вызывается `pipe.execute()`, отправляя весь пакет по сети

Снижает время выполнения 5 команд с 3-10 мс до 1-2 мс за счёт устранения 4 сетевых round-trips.

### Lua scripting — жизненный цикл скрипта

1. Клиент отправляет команду SCRIPT LOAD, содержащую код на Lua
2. Redis компилирует скрипт, кеширует его и возвращает SHA1-хеш скрипта
3. Клиент сохраняет SHA1-хеш в локальной переменной
4. При необходимости выполнения клиент вызывает EVALSHA, передавая хеш, число ключей, массив KEYS и массив ARGV
5. Redis выполняет скрипт атомарно (как единую сложную команду). Никакие другие команды от других клиентов в это время не обрабатываются
6. Если EVALSHA возвращает ошибку NOSCRIPT (после сброса кеша SCRIPT FLUSH или рестарта сервера), клиент перехватывает ошибку и вызывает EVAL напрямую с полным текстом скрипта

## Детали и нюансы

### Типы данных Lua при возврате

При возврате значений из Lua-скрипта в Redis происходит преобразование типов:
- `true` -> `1`
- `false` -> `None` (nil)
- Числа с плавающей точкой (1.5) усекаются до целых (1) — дробная часть полностью отбрасывается

### Блокирующие команды в транзакции

Блокирующие команды (BLPOP, BRPOP) внутри транзакции MULTI/EXEC: Redis не может блокировать соединение. Блокирующие вызовы тихо деградируют и выполняются как неблокирующие аналоги (LPOP/RPOP).

### Lua-скрипт с записью превышает lua-time-limit

Если скрипт Lua выполняется дольше lua-time-limit и выполняет операции модификации (write), Redis блокирует обработку других команд и не может быть прерван стандартной командой SCRIPT KILL. SCRIPT KILL сработает только для read-only скриптов. Если была произведена запись, администратор обязан вызвать SHUTDOWN NOSAVE.

### Передача ключей в Lua через ARGV вместо KEYS

В архитектуре Redis Cluster валидация принадлежности ключей к одному шарду происходит перед запуском скрипта путём чтения массива KEYS. Передача ключей через ARGV нарушает эту валидацию.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| WATCH/MULTI/EXEC | Простота реализации; стандартный механизм Redis | При высокой конкуренции — массовые повторы (до 161 000 на 600 операций), задержка до 498 мс | Низкая конкуренция за ключи |
| Lua scripting | Атомарность; 0 повторов; задержка <1 мс при высокой конкуренции | Блокирует весь Redis на время выполнения; нет ROLLBACK; усечение float | Высокая конкуренция; сложная логика на стороне сервера |
| Не-транзакционный pipeline | Снижение round-trips (5 команд: 1-2 мс вместо 3-10 мс) | Нет гарантии атомарности; нет согласованности чтения | Массовые вставки без необходимости в согласованности чтения |

| Аспект | Redis транзакции (MULTI/EXEC) | SQL транзакции |
|--------|-------------------------------|----------------|
| Откат | Нет ROLLBACK — все команды выполняются до конца, даже если одна вернёт ошибку | Есть ROLLBACK |
| Блокировка | Оптимистичная (WATCH) | Пессимистичная (SELECT FOR UPDATE) и оптимистичная |
| Аналогия | WATCH аналогичен CAS-операциям в процессорах | - |
| Хранимые процедуры | Lua-скрипты функционально идентичны stored procedures | Stored procedures |

## Примеры

### Покупка предмета с WATCH/MULTI/EXEC (Python)

```python
def purchase_item(conn, buyerid, itemid, sellerid, lprice):
    buyer = "users:%s"%buyerid
    seller = "users:%s"%sellerid
    item = "%s.%s"%(itemid, sellerid)
    inventory = "inventory:%s"%buyerid
    end = time.time() + 10
    pipe = conn.pipeline()

    while time.time() < end:
        try:
            pipe.watch("market:", buyer)
            price = pipe.zscore("market:", item)
            funds = int(pipe.hget(buyer, "funds"))
            if price != lprice or price > funds:
                pipe.unwatch()
                return None

            pipe.multi()
            pipe.hincrby(seller, "funds", int(price))
            pipe.hincrby(buyer, "funds", int(-price))
            pipe.sadd(inventory, itemid)
            pipe.zrem("market:", item)
            pipe.execute()
            return True
        except redis.exceptions.WatchError:
            pass
    return False
```

### Обёртка-загрузчик Lua (SCRIPT LOAD + EVALSHA / EVAL) (Python)

```python
def script_load(script):
    sha = [None]
    def call(conn, keys=[], args=[], force_eval=False):
        if not force_eval:
            if not sha:
                sha = conn.execute_command("SCRIPT", "LOAD", script, parse="LOAD")
            try:
                return conn.execute_command("EVALSHA", sha, len(keys), *(keys+args))
            except redis.exceptions.ResponseError as msg:
                if not msg.args.startswith("NOSCRIPT"):
                    raise
        return conn.execute_command("EVAL", script, len(keys), *(keys+args))
    return call
```

### Lua-скрипт покупки (замена WATCH/MULTI/EXEC)

```lua
local price = tonumber(redis.call('zscore', KEYS[1], ARGV[1]))
local funds = tonumber(redis.call('hget', KEYS[2], 'funds'))
if price and funds and funds >= price then
    redis.call('hincrby', KEYS[3], 'funds', price)
    redis.call('hincrby', KEYS[2], 'funds', -price)
    redis.call('sadd', KEYS[4], ARGV[2])
    redis.call('zrem', KEYS[1], ARGV[1])
    return true
end
```

Весь скрипт выполняется атомарно, отпадает необходимость в WATCH.

## Важно / подводные камни / best practices

- **Lua вместо WATCH при высокой конкуренции.** При нагрузке 5 процессов на чтение и 5 на запись транзакции WATCH вызывают до 161 000 повторов на 600 операций покупки, задержка до 498 мс. Lua даёт 0 повторов, задержку <1 мс
- **Не выполнять длительных вычислений в Lua.** Из-за однопоточной модели выполнение Lua-скрипта замораживает обработку всех входящих запросов
- **Использовать не-транзакционные pipeline(False) для массовых вставок** без необходимости в согласованности чтения
- **Передавать ключи в Lua через KEYS, а не ARGV.** В Redis Cluster валидация принадлежности ключей к одному шарду происходит через массив KEYS перед запуском скрипта
- **SCRIPT KILL не работает для скриптов с записью.** Если Lua-скрипт превысил lua-time-limit и выполнил write-операцию, единственный выход — SHUTDOWN NOSAVE
- **BLPOP/BRPOP в MULTI/EXEC тихо деградируют** до неблокирующих LPOP/RPOP
- **Float из Lua усекается до целого** — 1.5 превращается в 1
- **В Redis нет ROLLBACK** — все команды в транзакции выполняются до конца, даже если одна вернёт ошибку
