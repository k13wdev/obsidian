## Суть и контекст

Pub/Sub (Publish/Subscribe) — паттерн доставки сообщений в Redis, реализующий модель Push messaging. Издатели отправляют сообщения в каналы, подписчики получают их в реальном времени. Ключевая особенность — сообщения нигде не сохраняются (fire-and-forget): доставка происходит только тем клиентам, которые подключены и слушают канал в момент отправки.

Аналогия из источников: механизм Pub/Sub работает по принципу **радиовещания** (radio station) — сообщения доходят только до тех, кто настроил приёмник на нужную волну в момент передачи.

---

## Ключевые идеи

- Pub/Sub — это Push messaging: отправитель берёт на себя доставку сообщения всем подключённым получателям.
- Сообщения не записываются ни в память базы данных, ни на диск; состояние канала существует только в момент передачи.
- Подписчик может подписываться как на конкретные каналы (`SUBSCRIBE`), так и на каналы по шаблону (`PSUBSCRIBE`).
- Семантика Pub/Sub не предусматривает долговременного хранения (durability) или подтверждения обработки.
- Если подписчик не успевает обрабатывать сообщения, Redis принудительно его отключает для защиты сервера.
- Для надёжной доставки рекомендуется Pull messaging (через LIST или ZSET) вместо Pub/Sub.

---

## Определения и термины

**Pub/Sub (Publish/Subscribe)** — паттерн доставки сообщений (Push messaging), при котором слушатели (подписчики) подписываются на каналы, а издатели отправляют бинарные строковые сообщения в эти каналы. Подключенные подписчики получают все сообщения, отправленные в канал, но сообщения нигде не сохраняются.

**Push messaging** — метод, при котором отправитель берёт на себя отправку сообщения, следя за тем, чтобы все подключённые получатели его получили. Реализуется через PUBLISH/SUBSCRIBE.

**Паттерн-подписка (Pattern subscription)** — подписка на сообщения, транслируемые в каналы, имена которых соответствуют заданному шаблону (паттерну). Реализуется через PSUBSCRIBE и PUNSUBSCRIBE.

**Stream (Поток)** — один из поддерживаемых типов данных Redis. Детальное описание команд и механизмов (consumer groups, dead letter queue, backpressure, at-least-once) в предоставленных источниках не найдено.

## Как устроено

Механика Pub/Sub — пошаговый процесс:

1. Клиент отправляет команду `SUBSCRIBE` или `PSUBSCRIBE` для прослушивания каналов.
2. Издатель использует команду `PUBLISH` для отправки бинарного строкового сообщения в канал.
3. Redis мгновенно ретранслирует сообщение всем клиентам, которые в данный момент подключены и слушают указанный канал (fire-and-forget).
4. Если подписчик не успевает обрабатывать входящие сообщения (медленный клиент), исходящий буфер сервера (outgoing buffer) начинает заполняться.
5. Для предотвращения падения сервера Redis принудительно отключает медленных клиентов, если их исходящий буфер превышает лимит `client-output-buffer-limit pubsub`.

### Команды

```
SUBSCRIBE channel [channel ...]
UNSUBSCRIBE [channel [channel ...]]
PUBLISH channel message
PSUBSCRIBE pattern [pattern ...]
PUNSUBSCRIBE [pattern [pattern ...]]
```

- `SUBSCRIBE` — подписка на один или несколько каналов по имени.
- `UNSUBSCRIBE` — отписка от каналов.
- `PUBLISH` — отправка бинарного строкового сообщения в указанный канал.
- `PSUBSCRIBE` — подписка на каналы по шаблону (паттерну).
- `PUNSUBSCRIBE` — отписка от каналов по шаблону.

## Детали и нюансы

### Исходящий буфер и медленные клиенты

Если подписчик считывает сообщения слишком медленно, исходящий буфер переполняется:

- **В старых версиях Redis** — исчерпание RAM, замедление, краш Redis или OOM Killer.
- **В новых версиях Redis** — принудительное отключение клиента на основе `client-output-buffer-limit pubsub`.

### Разрыв соединения

При разрыве сетевого соединения клиента-подписчика сообщение ему не доставляется. Из-за отсутствия хранения (fire-and-forget) после переподключения клиент **никогда** не получит пропущенные сообщения.

### Формат получаемых сообщений (Python)

Сообщения приходят в виде словарей:
```
{'pattern': None, 'type': 'message', 'channel': 'channel', 'data': 'текст'}
```

Возврат из `subscribe` генерирует системное сообщение с типом `subscribe`.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| **Push messaging** (Pub/Sub) | Мгновенная доставка всем подключённым подписчикам; простая модель «издатель → канал → подписчик» | Нет хранения сообщений; нет гарантии доставки; при разрыве связи сообщения теряются; медленные клиенты могут быть отключены | Уведомления в реальном времени, когда потеря отдельных сообщений допустима |
| **Pull messaging** (LIST / ZSET) | Инициатива на стороне подписчика; сообщения хранятся до момента извлечения; надёжная доставка | Подписчик сам должен регулярно забирать сообщения | Системы, где требуется надёжная доставка и нельзя терять данные |

В Push messaging инициатива и риск лежат на моменте отправки. В Pull messaging инициатива лежит на подписчике, который сам регулярно забирает (pull) сообщения из хранилища.

## Примеры

### Python — подписка и получение сообщений

```python
pubsub = conn.pubsub()
pubsub.subscribe(['channel'])
for item in pubsub.listen():
    print item
```

Создаётся объект `pubsub`, оформляется подписка на `channel`. Цикл `listen()` блокирует поток, извлекая новые сообщения в виде словарей: `{'pattern': None, 'type': 'message', 'channel': 'channel', 'data': 'текст'}`. Возврат из `subscribe` генерирует системное сообщение с типом `subscribe`.

## Важно / подводные камни / best practices

- **НЕ использовать** PUBLISH/SUBSCRIBE для передачи критически важных данных без резервных путей восстановления. Семантика Pub/Sub не предусматривает долговременного хранения (durability) или подтверждения обработки.
- Использовать **Pull messaging** (через LIST или ZSET) вместо Pub/Sub в системах, где требуется надёжная доставка. В реалиях сетевой архитектуры нужно исходить из предположения о возможных обрывах связи.
- Контролировать параметр `client-output-buffer-limit pubsub` для защиты сервера от медленных клиентов.
- При разрыве соединения клиент **никогда** не получит пропущенные сообщения — необходимо проектировать систему с учётом этого ограничения.
