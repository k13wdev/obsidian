## Суть и контекст

Тема охватывает типовые архитектурные паттерны использования Redis для решения прикладных задач: кэширование сессий, корзины покупок, real-time аналитика, геолокация, таргетинг рекламы, лидерборды, автодополнение, подсчёт уникальных посетителей. Каждый паттерн описывает конкретную схему хранения данных в структурах Redis (HASH, SET, ZSET) и набор команд для работы с ними. Отдельно рассматриваются антипаттерны — ошибки при проектировании ключей и структур, приводящие к избыточному потреблению памяти и деградации производительности. Паттерны применимы при построении высоконагруженных веб-приложений, рекламных систем, социальных сетей и аналитических платформ.

---

## Ключевые идеи

- Redis-паттерны строятся на комбинации базовых структур данных (HASH, SET, ZSET, STRING) для решения конкретных прикладных задач.
- ZSET — наиболее универсальная структура: используется для лидербордов, автодополнения, геолокации, аналитики, таргетинга и инвертированных индексов.
- Инвертированный индекс на SET и ZSET — аналог механизмов Apache Lucene для полнотекстового поиска.
- Шардирование больших структур (HASH, SET) позволяет экономить до 70% памяти за счёт использования внутренних compact-кодировок (ziplist, intset).
- Оптимизация UUID (извлечение 56 бит из 128-битного UUID) экономит до 83% памяти при хранении в intset вместо hashtable.
- Фоновые процессы (cleanup, рескейлинг) — неотъемлемая часть паттернов, обеспечивающая контроль над ростом данных.
- Операции над множествами (SINTER, SUNION, SDIFF, ZINTERSTORE, ZUNIONSTORE) — основа для поисковых и матчинговых паттернов.

---

## Определения и термины

**Inverted index (Обратный индекс)** — структура данных (в Redis реализуемая через SET и ZSET), сопоставляющая слова (токены) или свойства с идентификаторами документов.

**Sharding (Шардирование)** — метод горизонтального масштабирования, при котором данные разбиваются на небольшие части на основе детерминистических правил. В контексте оптимизации памяти Redis — разбиение больших HASH, SET для хранения в ziplist или intset.

## Как устроено

### Паттерн: Кэширование сессий (Login cookies)

Схема хранения:
- `HASH login:` — маппинг "токен -> ID пользователя"
- `ZSET recent:` — токены с timestamp в качестве score

Алгоритм очистки (фоновый процесс):
1. `ZCARD recent:` — проверить количество токенов
2. Если количество > лимита (10 000 000) — выполнить `ZRANGE recent: 0 99` (получить 100 самых старых токенов)
3. Для каждого старого токена: `HDEL` из login:, `ZREM` из recent:, `DEL viewed:<token>`, `DEL cart:<token>`

---

### Паттерн: Корзины покупок

Схема хранения:
- `HASH cart:<session_token>` — поле = ID товара, значение = количество

Команды:
- `HSET` для добавления/обновления товара
- `HREM` при количестве <= 0 (удаление товара из корзины)

---

### Паттерн: Real-time analytics и счётчики

**Аналитика просмотров:**
- `ZSET viewed:` — member = ID товара, score = инвертированное количество просмотров
- `ZINCRBY viewed: -1 item_id` — инкремент просмотра (score уменьшается, поэтому наиболее просматриваемые элементы имеют наименьший score и оказываются в начале при сортировке)
- Фоновый процесс: `ZREMRANGEBYRANK viewed: 20000 -1` — удаление элементов за пределами топ-20000
- Рескейлинг: `ZINTERSTORE viewed: 1 viewed: WEIGHTS 0.5` — уменьшение всех score вдвое для предотвращения переполнения

**Счётчики временных срезов:**
- `ZSET known:` (с score 0) — имена активных счётчиков
- `HASH count:<precision>:<name>` — поле = метка времени, значение = количество
- Обновление: `ZADD known: hash_name 0` и `HINCRBY count:hash_name time_slice count`

---

### Паттерн: Геолокация (IP-to-city lookup)

Схема хранения:
- `ZSET ip2cityid:` — member = ID города, score = числовое представление IP
- `HASH cityid2city:` — ID города → JSON [city, region, country]

Формула конвертации IPv4 в score:
```
score = oct1 * 256^3 + oct2 * 256^2 + oct3 * 256 + oct4
```

Алгоритм поиска города по IP:
1. Конвертировать IP-адрес в числовой score
2. `ZREVRANGEBYSCORE ip2cityid: <IP-score> 0 LIMIT 0 1` — найти ближайший город с score <= запрошенному

---

### Паттерн: Таргетинг рекламы

Схема хранения:
- `SET idx:req:<location>` — ID объявлений по локации
- `ZSET ad:base_value:` — базовый eCPM объявлений

Алгоритм подбора рекламы:
1. `SUNION` всех множеств локаций → matched_ads
2. `ZINTERSTORE` между matched_ads и ad:value: — отфильтровать совпавшие объявления с их ценами
3. Для контентного бонуса: `ZINTERSTORE` для каждого слова страницы
4. `ZUNIONSTORE` с агрегатами MIN и MAX, весами 1, 0.5, 0.5 — объединение с учётом контентного релеванса

---

### Паттерн: Лидерборды и социальные графы

Схема хранения:
- `ZSET followers:<uid>` и `ZSET following:<uid>` — score = timestamp подписки
- `ZSET home:<uid>` — лента пользователя, member = ID статуса, score = timestamp публикации

Алгоритм формирования ленты при подписке:
1. `ZREVRANGE profile:<other_uid> 0 999` — получить последние 1000 статусов нового друга
2. `ZADD home:<uid>` — добавить эти статусы в ленту текущего пользователя

---

### Паттерн: Автодополнение (Autocomplete)

Схема хранения:
- `ZSET` с именами (score 0, лексикографическая сортировка)

Алгоритм поиска по префиксу (для префикса "abc"):
1. Сформировать маркерные строки: `"abbz...<uuid>"` (граница перед "abc") и `"abd<uuid>"` (граница после "abc")
2. `ZADD` маркеров в ZSET
3. `ZRANK` для обоих маркеров — определить позиции
4. `ZRANGE` между рангами маркеров — получить элементы в диапазоне
5. `ZREM` маркеров — удалить временные элементы

---

### Паттерн: Подсчёт уникальных посетителей

Оптимизация UUID:
- 128-битный UUID → извлечение 15 hex-символов (56 бит)

Шардирование:
- Шард = `hash(ID) // shard_size` (shard_size = 512 для intset)
- `SADD unique:<date>:<shard_id> ID`

Алгоритм подсчёта:
1. `SADD unique:<date>:<shard_id> ID`
2. Если результат SADD = 1 (новый элемент) → `INCR` глобального счётчика

---

### Паттерн: Кэширование веб-страниц

Схема хранения:
- `STRING cache:<hash_request>` — кэшированный контент страницы с TTL 300 секунд

Алгоритм:
1. Сформировать ключ `page_key = 'cache:' + hash_request(request)`
2. `GET page_key` — попытка получить кэш
3. Если кэша нет — вызвать callback, сохранить через `SETEX` с TTL 300 секунд
4. Вернуть контент

---

### Поиск вакансий через ZSET

- `ZINTERSTORE` с SET навыков (вес 1) и ZSET требований (вес -1)
- Score = 0 означает, что все навыки совпали с требованиями вакансии

## Детали и нюансы

### Инвертированный score в аналитике просмотров

Использование `ZINCRBY viewed: -1 item_id` (отрицательный инкремент) — приём, позволяющий хранить наиболее просматриваемые элементы с наименьшим score. Это обеспечивает эффективную выборку топ-N элементов через стандартные команды ZSET.

### Рескейлинг счётчиков

Команда `ZINTERSTORE viewed: 1 viewed: WEIGHTS 0.5` — Self-intersection с весом 0.5 уменьшает все score вдвое. Это необходимо для предотвращения неограниченного роста числовых значений при длительной работе.

### Маркеры в автодополнении

Маркерные строки содержат UUID для предотвращения коллизий при конкурентном доступе нескольких клиентов. Граница "abbz..." перед "abc" и "abd" после "abc" обеспечивает точный лексикографический диапазон.

### Шардирование для intset

Размер шарда 512 элементов выбран так, чтобы SET оставался в compact-кодировке intset, которая значительно экономичнее по памяти, чем hashtable.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Плоский HASH (320,000 ключей) | Простота реализации | 44 МБ памяти | Когда память не критична |
| Шардированный HASH (312 шардов по 1024) | 12 МБ (экономия 70%) | Сложность доступа к данным | Когда нужна экономия памяти |
| UUID как строка (36 байт) в SET hashtable | Простота, читаемость | 56 МБ на 1 млн записей | Небольшие объёмы данных |
| UUID 56 бит (15 hex) в intset | 9.5 МБ (экономия 83%) | Потеря части информации UUID | Подсчёт уникальных значений при большом объёме |
| SINTER/SUNION/SDIFF для инвертированного индекса | Аналогично Apache Lucene, нативная поддержка Redis | Ограничения по сложным запросам (в источниках не детализировано) | Полнотекстовый поиск и матчинг |
| ZINTERSTORE для поиска вакансий (score = навыки - требования) | Точное совпадение при score = 0 | В источниках не раскрыто | Матчинг по множеству критериев |

### Big keys: ziplist vs hashtable

| Параметр | Ziplist (большой, ~50,000 элементов) | Ziplist (оптимальный, 512-2048 элементов) |
|----------|--------------------------------------|-------------------------------------------|
| Производительность RPOPLPUSH | ~553 оп/сек | Значительно выше (в источниках не уточнено) |
| Рекомендация | Антипаттерн | Best practice |

## Примеры

### Lua-скрипт автодополнения

```lua
redis.call('zadd', KEYS[1], 0, ARGV[1], 0, ARGV[2])
local sindex = redis.call('zrank', KEYS[1], ARGV[1])
local eindex = redis.call('zrank', KEYS[1], ARGV[2])
eindex = math.min(sindex + 9, eindex - 2)
redis.call('zrem', KEYS[1], unpack(ARGV))
return redis.call('zrange', KEYS[1], sindex, eindex)
```

### Web page caching (Python)

```python
page_key = 'cache:' + hash_request(request)
content = conn.get(page_key)
if not content:
    content = callback(request)
    conn.setex(page_key, content, 300)
return content
```

## Важно / подводные камни / best practices

### Антипаттерн: Big keys

Ziplist до 50,000–100,000 элементов приводит к деградации производительности: RPOPLPUSH при ~50,000 элементах падает до 553 оп/сек.

**Best practice:** `max-ziplist-entries` 512-2048, `max-ziplist-value` 64-128 байт.

### Шардирование для экономии памяти

Плоский HASH на 320,000 ключей занимает 44 МБ. Разбиение на 312 мелких HASH (до 1024 элементов каждый) = 12 МБ — экономия 70%.

### Оптимизация UUID

128-битный UUID как строка занимает 36 байт (1 млн записей в SET hashtable = 56 МБ). Извлечение 56 бит (15 hex-символов) и хранение в intset = 9.5 МБ — экономия 83%.

### Фоновые процессы обязательны

Для паттернов с неограниченным ростом данных (сессии, аналитика) необходимы фоновые процессы очистки:
- Сессии: `ZRANGE` + `HDEL` + `ZREM` + `DEL` при превышении лимита
- Аналитика: `ZREMRANGEBYRANK` для обрезки до топ-N
- Рескейлинг: `ZINTERSTORE` с WEIGHTS 0.5
