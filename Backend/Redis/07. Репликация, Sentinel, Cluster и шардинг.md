## Суть и контекст

Тема охватывает механизмы обеспечения отказоустойчивости и горизонтального масштабирования Redis: репликацию данных между ведущим и подчинёнными узлами, автоматический failover через Redis Sentinel, кластеризацию через Redis Cluster с алгоритмическим шардингом на основе хеш-слотов, а также различия между client-side и proxy-based подходами к шардингу.

---

## Ключевые идеи

- Репликация в Redis работает по модели «ведущий — подчинённый» (master — slave). При первоначальном подключении подчинённый полностью сбрасывает свои данные и загружает снепшот ведущего.
- Инкрементальная репликация использует ID репликации и смещение репликации (replication offset) для определения, нужна ли полная или частичная синхронизация.
- Redis Sentinel — система мониторинга и автоматического failover. Процессы Sentinel голосуют при недоступности ведущего и, при достижении кворума, назначают нового ведущего из подчинённых.
- Redis Cluster использует 16384 хеш-слота для алгоритмического шардинга. Ключ хешируется и делится по модулю на 16384 для определения принадлежности к узлу.
- Узлы Redis Cluster обмениваются информацией через протокол Gossip для определения общего состояния кластера.
- В open source Redis кластеризация реализуется на стороне клиента через cluster-aware библиотеки. В Redis Enterprise — через прокси (proxy-based), что позволяет использовать стандартный клиент без доработок.

---

## Определения и термины

**SLAVEOF** — команда или параметр конфигурации, указывающий экземпляру Redis использовать заданный хост и порт в качестве ведущего узла для репликации. Команда `SLAVEOF no one` отключает узел от ведущего.

**Смещение репликации (Replication offset)** — инкрементируемый показатель, отражающий выполнение любого действия в ведущем узле Redis. Используется вместе с ID репликации для определения необходимости полной или частичной синхронизации.

**Кворум (Quorum)** — минимальное число голосов, которое нужно получить распределённой системе (Sentinel) для выполнения операций failover. Зависит от количества узлов, рекомендуется нечётное число.

**Хеш-слоты (Hash slots)** — 16384 виртуальных слота в механизме алгоритмического шардинга Redis Cluster. Ключ хешируется и делится по модулю на 16383 (16384 слота), чтобы определить принадлежность к конкретному узлу.

**Протокол Gossip** — протокол, используемый узлами Redis Cluster для постоянного обмена информацией друг с другом и определения общего состояния кластера. Позволяет шардам договориться о недоступности ведущего узла и повысить уровень подчинённого.

> Термины REPLICAOF, replication backlog, PSYNC, SENTINEL-команды, CLUSTER-команды, MOVED, ASK, consistent hashing — в предоставленных источниках не раскрыты.

## Как устроено

### Начальная синхронизация репликации

1. Подчинённый узел подключается к ведущему и отправляет команду `SYNC`.
2. Ведущий начинает `BGSAVE` и буферизирует все поступающие команды записи.
3. Ведущий завершает `BGSAVE` и передаёт снепшот подчинённому.
4. Подчинённый сбрасывает все старые данные в памяти и загружает снепшот.
5. Ведущий отправляет буферизованные команды записи.
6. Ведущий переходит в режим трансляции команд в реальном времени (live streaming).

При первоначальном подключении все данные подчинённого полностью уничтожаются.

### Инкрементальная репликация

1. Проверяется совпадение ID репликации ведущего и подчинённого.
2. Если ID совпадают, вычисляется разница в показателях смещения репликации.
3. Подчинённый получает только необработанные команды, соответствующие разнице смещений.

### Redis Sentinel

1. Процессы Sentinel постоянно мониторят доступность узлов.
2. При недоступности ведущего экземпляра Sentinel проводят голосование.
3. Если достигнут кворум, инициируется failover.
4. Один из подчинённых назначается новым ведущим.
5. Sentinel сообщает клиентам новые координаты (service discovery).

Механизм service discovery в Sentinel аналогичен Zookeeper и Consul.

### Redis Cluster и resharding

1. Вычисляется детерминистический хеш ключа.
2. Результат делится по модулю на 16384.
3. Для изменения топологии перемещаются хеш-слоты, а не ключи.

### Client-side vs proxy-based шардинг

В open source Redis кластеризация реализуется на стороне клиента через cluster-aware библиотеки. В Redis Enterprise — через прокси (proxy-based), что позволяет использовать стандартный клиент без доработок.

## Детали и нюансы

- Если несколько подчинённых узлов подключаются к одному ведущему до завершения `BGSAVE`, все они получат один и тот же дамп. Это может исчерпать пропускную способность сети.
- При чётном количестве узлов возможен split brain: голоса делятся поровну, кластер не может восстановиться. Данные в изолированном ведущем будут потеряны.
- Multi-key операции в Lua-скриптах в кластере требуют, чтобы все ключи находились на одном шарде.
- При Active-Active репликации возможен data lag: разные узлы содержат разные значения.
- **НЕ** настраивать Master-Master в open source Redis через взаимный `SLAVEOF` — это приведёт к загрузке CPU на 100% и потере данных. Active-Active возможен только через CRDT в Redis Enterprise.

---

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| Open Source Redis (cluster-aware client library) | Бесплатен, стандартный механизм | Требует cluster-aware библиотеку на стороне клиента | Open source проекты, когда допустима доработка клиента |
| Redis Enterprise (proxy-based) | Стандартный клиент без доработок, proxy protocol | Коммерческий продукт | Когда нужна простота интеграции и Active-Active через CRDT |

| Аспект | Полная синхронизация | Инкрементальная синхронизация |
|--------|---------------------|------------------------------|
| Условие | Первое подключение или несовпадение ID репликации | Совпадение ID репликации |
| Объём данных | Полный снепшот (BGSAVE) | Только разница смещений |
| Влияние на подчинённый | Полный сброс данных | Применение недостающих команд |

## Важно / подводные камни / best practices

- **Нечётное количество ведущих узлов** (3, 5) с минимум 2 подчинёнными у каждого — для избежания split brain.
- **Запускать Sentinel локально** на каждом сервере приложения.
- **Кворум**: для 3 серверов = 2, для 5 серверов = 3.
- **НЕ настраивать Master-Master** в open source Redis через взаимный `SLAVEOF` — приводит к CPU 100% и потере данных. Active-Active возможен только через CRDT в Redis Enterprise.
- При одновременном подключении нескольких подчинённых к ведущему до завершения `BGSAVE` — все получат один дамп, что может исчерпать пропускную способность сети.
- Multi-key операции в Lua в кластере: все ключи обязаны быть на одном шарде.
- Split brain при чётном количестве узлов: голоса делятся поровну, кластер не может восстановиться, данные в изолированном ведущем будут потеряны.
- Active-Active репликация сопряжена с data lag: разные узлы могут содержать разные значения.
