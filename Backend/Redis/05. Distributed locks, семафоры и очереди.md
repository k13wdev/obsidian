## Суть и контекст

Тема охватывает механизмы координации распределённых процессов в Redis: блокировки (distributed locks), считающие и честные семафоры (counting/fair semaphores), а также различные паттерны очередей (worker queue, приоритетные очереди, отложенные задачи, reliable queue). Эти примитивы решают задачу эксклюзивного или ограниченного доступа к ресурсам на разных машинах, используя Redis как центральный координатор, а также задачу надёжной передачи и выполнения задач между процессами.

---

## Ключевые идеи

- Distributed lock через SETNX заменяет оптимистичную блокировку WATCH и кардинально снижает количество конфликтов (retries)
- UUID (128-битный) используется как идентификатор владельца блокировки для защиты от удаления чужой блокировки
- Counting semaphore ограничивает количество одновременных процессов фиксированным числом, реализуется через ZSET
- Fair semaphore решает проблему рассинхронизации системных часов с помощью инкрементируемого счётчика
- Redis LIST используется как основа для worker queue с блокирующим чтением (BLPOP/BRPOP)
- Приоритетные очереди реализуются через несколько LIST с проверкой слева направо в BLPOP
- Отложенные задачи хранятся в ZSET с Unix timestamp в качестве score и переносятся в рабочую очередь поллером
- RPOPLPUSH / BRPOPLPUSH обеспечивает reliable queue — атомарное перемещение исключает потерю задачи при падении воркера
- Rate limiting через счётчики по временным срезам использует HINCRBY
- Fine-grained locking (блокировка конкретного объекта) снижает задержку с ~500мс до <3мс

## Определения и термины

**Distributed lock (Распределённая блокировка)** — механизм, при котором процесс получает эксклюзивный доступ к данным в области видимости, доступной всем клиентам на разных машинах (на сервере Redis), с использованием SETNX. Заменяет оптимистичную блокировку WATCH для снижения конфликтов.

**Counting semaphore (Считающий семафор)** — тип блокировки, ограничивающий количество процессов, которые могут параллельно получить доступ к ресурсу, заданным фиксированным числом. Реализуется через ZSET, где member — UUID, score — метка времени.

**Fair semaphore (Честный семафор)** — семафор, использующий инкрементируемый счётчик (counter) для определения очерёдности, что минимизирует влияние рассинхронизации системных часов.

**Delayed tasks (Отложенные задачи)** — задачи, помещаемые в ZSET с указанием времени выполнения (Unix timestamp) в качестве score, ожидая перевода в рабочую очередь (LIST). Redis не имеет встроенного блокирующего чтения по score из ZSET.

> Алгоритм Redlock, Fencing tokens, Token bucket, Sliding window, Fixed window counter — в источниках не найдено.

## Как устроено

### Захват блокировки (acquire lock)

1. Генерируется случайный 128-битный UUID
2. В цикле (пока не истечёт `acquire_timeout`) выполняется `SETNX lock:имя UUID`
3. Если SETNX вернул 1 — выполняется `EXPIRE lock:имя lock_timeout`. Возвращается UUID
4. Если SETNX вернул 0 — проверяется TTL. Если TTL отсутствует — устанавливается EXPIRE (защита от зависания блокировки без TTL)
5. Засыпание на 0.001 сек и повтор цикла

### Снятие блокировки (release lock)

1. `WATCH lock:имя`
2. `GET lock:имя`
3. Если значение совпадает с UUID — `MULTI`, `DEL lock:имя`, `EXEC`
4. Если не совпадает — блокировка утеряна (другой процесс уже перехватил)

### Fair semaphore (захват)

1. `ZREMRANGEBYSCORE semname '-inf' (now - timeout)` — удаление просроченных
2. `ZINTERSTORE semname:owner 2 semname:owner semname WEIGHTS 1 0` — синхронизация owner с актуальными записями
3. `INCR semname:counter` — инкремент глобального счётчика
4. `ZADD semname UUID now` — добавление метки времени
5. `ZADD semname:owner UUID counter` — добавление с counter в качестве score
6. `ZRANK semname:owner UUID` — получение позиции (ранга)
7. Если ранг < limit — семафор получен

### Worker Queue

1. В очередь `queue:имя` помещается JSON `['ИМЯ_ФУНКЦИИ', [АРГ1, АРГ2]]` через RPUSH
2. Worker вызывает `BLPOP queue:имя 30`, блокируясь до 30 секунд
3. Worker декодирует JSON и вызывает callback

### Приоритетные очереди

`BLPOP ['queue:high', 'queue:medium', 'queue:low'] 30` — Redis проверяет очереди слева направо. Задача из `queue:high` извлекается первой.

### Delayed Tasks (поллинг)

1. Задача сериализуется в JSON: `[UUID, queue, name, args]`
2. `ZADD delayed:` с `score = time.time() + delay`
3. Поллер вызывает `ZRANGE delayed: 0 0 withscores` — берёт элемент с минимальным score
4. Если score <= текущему времени, захватывает блокировку по UUID
5. `ZREM` из `delayed:` и `RPUSH` в рабочую очередь

### Reliable Queue

`RPOPLPUSH` и `BRPOPLPUSH` — атомарное перемещение элементов между списками. Исключает потерю задачи при падении воркера после LPOP.

### Rate Limiting (счётчики по временным срезам)

1. Текущее время `now` делится на точность среза `prec` и умножается на `prec`
2. `ZADD known: имя_счётчика 0`
3. `HINCRBY count:prec:имя_счётчика pnow count`

## Детали и нюансы

- **Превышение lock timeout**: процесс может превысить `lock_timeout`, и другой процесс захватит блокировку. UUID защищает от удаления чужой блокировки, но оба процесса могут одновременно изменять данные — это фундаментальное ограничение.
- **BLPOP/BRPOP внутри MULTI/EXEC**: деградирует до обычных LPOP/RPOP (теряется блокирующее поведение).
- **Рассинхронизация часов**: клиент с отстающими системными часами может «украсть» базовый counting semaphore. Решение — использовать fair semaphore со счётчиком вместо меток времени для определения очерёдности.
- **Fair semaphore на 32-битной платформе**: счётчик переполняется за несколько часов.
- **Redis не имеет встроенного блокирующего чтения по score из ZSET** — поэтому для delayed tasks необходим поллинг.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| SETNX (distributed lock) | 0 retries при >5 воркеров | Возможна одновременная работа двух процессов при превышении timeout | Эксклюзивный доступ к ресурсу в распределённой среде |
| WATCH (оптимистичная блокировка) | Встроенный механизм Redis | 161,000 retries при >5 воркеров | Низкая конкуренция за ресурс |
| Counting semaphore (ZSET + timestamp) | Простая реализация | Зависимость от синхронизации часов | Ограничение параллельного доступа при синхронных часах |
| Fair semaphore (ZSET + counter) | Не зависит от синхронизации часов | Переполнение счётчика на 32-битных платформах | Ограничение параллельного доступа при рассинхронизации часов |
| Redis LIST (очередь) | Без дополнительной инфраструктуры | Нет встроенных гарантий доставки | Простые задачи без строгих гарантий |
| RPOPLPUSH / BRPOPLPUSH (reliable queue) | Атомарное перемещение, защита от потери задач | Требуется дополнительный список | Задачи с требованием надёжной доставки |
| ZSET delayed tasks | Аналог cron job в оперативной памяти | Требуется поллинг | Отложенное выполнение задач |

Redis LIST очереди сравниваются с ActiveMQ, RabbitMQ, Gearman, Amazon SQS — Redis работает без дополнительной инфраструктуры.

## Примеры

### Захват блокировки с таймаутом

```python
def acquire_lock_with_timeout(conn, lockname, acquire_timeout=10, lock_timeout=10):
    identifier = str(uuid.uuid4())
    lockname = 'lock:' + lockname
    lock_timeout = int(math.ceil(lock_timeout))
    end = time.time() + acquire_timeout
    while time.time() < end:
        if conn.setnx(lockname, identifier):
            conn.expire(lockname, lock_timeout)
            return identifier
        elif not conn.ttl(lockname):
            conn.expire(lockname, lock_timeout)
        time.sleep(.001)
    return False
```

### Worker queue

```python
def worker_watch_queues(conn, queues, callbacks):
    while not QUIT:
        packed = conn.blpop(queues, 30)
        if not packed:
            continue
        name, args = json.loads(packed[1])
        if name not in callbacks:
            log_error("Unknown callback %s"%name)
            continue
        callbacks[name](*args)
```

### Поллинг отложенных задач

```python
def poll_queue(conn):
    while not QUIT:
        item = conn.zrange('delayed:', 0, 0, withscores=True)
        if not item or item[1] > time.time():
            time.sleep(.01)
            continue
        item = item
        identifier, queue, function, args = json.loads(item)
        locked = acquire_lock(conn, identifier)
        if not locked:
            continue
        if conn.zrem('delayed:', item):
            conn.rpush('queue:' + queue, item)
        release_lock(conn, identifier, locked)
```

## Важно / подводные камни / best practices

- **Fine-grained locking**: блокировать конкретный объект, а не весь ресурс. Переход от крупной блокировки к гранулярной снижает задержку с ~500мс до <3мс
- **1-2 процесса poll_queue()** достаточно для обработки отложенных задач
- **Не использовать LPUSH для приоритетных задач** — LIFO нарушает порядок. Использовать RPUSH в отдельные очереди
- **Сочетание блокировок и семафоров** для предотвращения race condition при захвате
- При захвате блокировки: если TTL отсутствует (например, из-за сбоя между SETNX и EXPIRE), необходимо установить EXPIRE — иначе блокировка зависнет навсегда
- UUID как идентификатор владельца предотвращает удаление чужой блокировки, но **не предотвращает** одновременную работу двух процессов при превышении timeout
