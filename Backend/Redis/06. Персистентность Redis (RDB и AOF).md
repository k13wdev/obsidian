## Суть и контекст

Redis хранит данные в оперативной памяти, что делает их уязвимыми при перезагрузке или сбое. Персистентность решает эту проблему двумя механизмами: RDB (снепшоты состояния на определённый момент времени) и AOF (непрерывный журнал операций записи). Оба механизма обеспечивают cache rehydration - воссоздание состояния кеша в оперативной памяти после перезагрузки или аварии. RDB аналогичен полному дампу SQL, AOF - потоку изменений (replication log), аналогичному redo log в Oracle.

---

## Ключевые идеи

- Redis предоставляет два механизма персистентности: RDB (снепшоты) и AOF (журнал операций)
- RDB создаёт максимально компактное представление данных для резервного копирования
- AOF фиксирует каждую операцию записи или модификации по мере поступления, обеспечивая более высокую гранулярность сохранности данных
- Оба механизма используют системный вызов fork и механизм Copy-on-Write для асинхронной работы без блокировки клиентов
- При наличии обоих файлов Redis по умолчанию восстанавливает данные из AOF, так как он содержит более полную версию данных
- AOF непрерывно растёт и требует периодической перезаписи (BGREWRITEAOF)
- Политика appendfsync определяет баланс между производительностью и гарантиями сохранности данных
- Для Copy-on-Write необходимо резервировать 30--45% RAM на сервере

## Определения и термины

**RDB (Redis Database / Snapshots)** - снепшот (снимок), представляющий собой асинхронную копию данных из памяти, сохранённую на диск и актуальную на определённый момент времени. Максимально компактное представление данных для резервного копирования.

**AOF (Append-Only File)** - журнал (лог-файл), в котором фиксируется каждая операция записи или модификации данных по мере её поступления на сервер. Журнал непрерывно растёт и содержит реальные команды в порядке их выполнения.

**Форк (Fork)** - процесс создания ядром ОС точной копии родительского процесса (процесса-потомка), которая получает доступ к той же памяти. В Unix/Linux память разделяется между родителем и потомком до момента изменения данных.

**Copy-on-Write (Копирование при записи)** - механизм совместного использования памяти, при котором ядро ОС отслеживает ссылки на каждую страницу памяти. Если на страницу появляется больше одной ссылки (происходит операция записи), изменения физически записываются в новые выделенные страницы памяти. Позволяет дочернему процессу сохранять неизменный снимок памяти, в то время как родительский процесс продолжает обрабатывать новые команды.

**fsync** - системный вызов ядра ОС, принудительно переносящий все модифицированные данные из кеша файловой системы на физический диск. Блокирует выполнение программы до завершения записи.

**Cache rehydration** - процесс воссоздания состояния кеша в оперативной памяти путём загрузки резервной копии из постоянного хранилища (AOF или RDB) после перезагрузки или сбоя системы.

## Как устроено

### Механика RDB Snapshotting

1. Триггером выступает команда `BGSAVE`, достижение условий конфигурации `save`, команда `SHUTDOWN` или получение запроса `SYNC` от подчинённого узла при репликации
2. Redis инициирует системный вызов fork
3. Дочерний процесс начинает запись данных из разделяемой памяти на диск в файл `dump.rdb`
4. Родительский процесс продолжает обслуживать клиентов. Если родитель вносит изменения в ключи, механизм Copy-on-Write выделяет новые страницы памяти для изменённых данных
5. Дочерний процесс завершает запись неизменного слепка и завершается

### Механика AOF

1. Команды изменения данных поступают на сервер Redis
2. Redis немедленно буферизует команды в конец файла журнала
3. Системный вызов fsync сбрасывает буферы ОС на диск согласно выбранной политике `appendfsync` (`always`, `everysec`, `no`)

### Механика BGREWRITEAOF

1. Срабатывает триггер (`auto-aof-rewrite-percentage` и `auto-aof-rewrite-min-size`) или ручная команда `BGREWRITEAOF`
2. Выполняется системный вызов fork
3. Дочерний процесс формирует новый файл лога на основе текущего состояния памяти в виде кратчайшей последовательности команд
4. После завершения перезаписи новый файл подменяет старый

### Восстановление данных

Если присутствуют одновременно RDB и AOF файлы, Redis по умолчанию использует для восстановления файл AOF, так как он содержит более полную версию данных.

## Детали и нюансы

### Edge cases

1. **Потеря данных при крэше в режиме RDB.** Теряются все операции записи между последним снепшотом и моментом аварии.

2. **BGSAVE на инстансах с большим объёмом RAM.** На bare metal fork занимает 10--20 мс на ГБ. На Xen (AWS EC2) - 200--300 мс на ГБ. Для базы 20 ГБ пауза может достигать 4--6 секунд.

3. **appendfsync always на SSD.** Каждая команда вызывает fsync. Возникает "Write Amplification", сокращающий срок службы SSD с нескольких лет до нескольких месяцев.

4. **appendfsync no.** Redis делегирует управление записью ОС. При отключении питания теряется непредсказуемый объём данных.

5. **Удаление старого AOF файла после BGREWRITEAOF.** При удалении файла в десятки ГБ система может зависнуть на несколько секунд.

### Контрольные суммы и утилиты проверки

- С версии 2.6 RDB включают контрольную сумму CRC64
- `redis-check-aof --fix` обрезает файл перед первой испорченной командой

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| RDB | Максимально компактное представление данных; загружается значительно быстрее | Теряются все операции между последним снепшотом и аварией | Резервное копирование; быстрое восстановление; аналог полного дампа SQL |
| AOF | Более высокая гранулярность сохранности данных; фиксирует каждую операцию | Журнал непрерывно растёт; требует перезаписи (BGREWRITEAOF) | Минимизация потерь данных; аналог redo log в Oracle |

### Политики appendfsync

| Политика | Поведение | Компромисс |
|----------|-----------|------------|
| `always` | fsync на каждую команду | Максимальная надёжность, но Write Amplification на SSD (срок службы с нескольких лет до нескольких месяцев) |
| `everysec` | fsync раз в секунду | Оптимальный компромисс; потеря максимум 1 секунда данных |
| `no` | ОС управляет записью | Максимальная производительность, но непредсказуемый объём потерь при отключении питания |

### BGSAVE vs SAVE для больших баз

| Команда | Поведение | Применение |
|---------|-----------|------------|
| `BGSAVE` | Асинхронно через fork; не блокирует клиентов | Штатная работа |
| `SAVE` | Блокирует все входящие команды | Снятие резервной копии баз 50+ ГБ при отключённых клиентах (3--5 минут вместо 15--20) |

## Примеры

### Конфигурация RDB

```
save 60 1000
stop-writes-on-bgsave-error no
rdbcompression yes
dbfilename dump.rdb
dir ./
```

### Конфигурация AOF

```
appendonly yes
appendfsync everysec
no-appendfsync-on-rewrite no
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

### Команды

```
BGSAVE
SAVE
SHUTDOWN
SHUTDOWN NOSAVE
```

- `BGSAVE` - асинхронно через fork
- `SAVE` - блокирует все входящие команды
- `SHUTDOWN` - блокирует клиентов, выполняет синхронный SAVE и завершает процесс
- `SHUTDOWN NOSAVE` - принудительно завершает без сохранения данных

### Утилиты проверки

```
redis-check-aof --fix <file.aof>
redis-check-dump <dump.rdb>
```

## Важно / подводные камни / best practices

- **appendfsync everysec** - оптимальный компромисс. Потеря максимум 1 секунда данных
- **Оставлять 30--45% RAM свободными** на сервере для Copy-on-Write
- **Использование синхронной SAVE вместо BGSAVE** для снятия резервной копии баз 50+ ГБ при отключённых клиентах (3--5 минут вместо 15--20)
- **BGSAVE на Xen (AWS EC2):** fork занимает 200--300 мс на ГБ (против 10--20 мс на bare metal), для базы 20 ГБ пауза может достигать 4--6 секунд
- **appendfsync always на SSD** вызывает Write Amplification, сокращая срок службы SSD с нескольких лет до нескольких месяцев
- **appendfsync no** - непредсказуемый объём потерь данных при отключении питания
- **Удаление старого AOF после BGREWRITEAOF** - файл в десятки ГБ может вызвать зависание на несколько секунд
- При наличии обоих файлов Redis восстанавливает данные из **AOF**, так как он содержит более полную версию данных
