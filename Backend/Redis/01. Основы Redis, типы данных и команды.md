## Суть и контекст

Redis (Remote Dictionary Service) — опенсорсная база данных типа ключ-значение (сервер структур данных), размещаемая в оперативной памяти. Redis размывает границы между кешем и хранилищем данных за счет работы с данными на скорости оперативной памяти. Применяется для кеширования сессий, очередей задач, систем таргетинга рекламы, агрегации логов и сбора статистик реального времени.

---

## Ключевые идеи

- Redis работает в оперативной памяти — данные обрабатываются на скорости RAM, что размывает границу между кешем и полноценным хранилищем
- Однопоточная архитектура обеспечивает атомарность выполнения команд без необходимости блокировок и синхронизации
- Redis поддерживает богатый набор структур данных: строки, списки, множества, упорядоченные множества, хеши, битовые массивы
- Компактные представления (ziplist, intset) значительно снижают расход памяти при малом количестве элементов, но автоматически конвертируются в стандартные структуры при превышении порогов
- Шардирование крупных структур на множество мелких позволяет задействовать компактные кодировки и экономить до 70% памяти

## Определения и термины

**Redis (Remote Dictionary Service)** — опенсорсная база данных типа ключ-значение (сервер структур данных), размещаемая в оперативной памяти. Размывает границы между кешем и хранилищем данных за счет работы с данными на скорости оперативной памяти.

**Однопоточная архитектура** — архитектура, при которой обработка данных осуществляется в один поток без использования многопоточности. Команды (включая скрипты Lua) выполняются атомарно, блокируя выполнение других команд чтения/записи до завершения текущей.

**Ziplist** — компактное неструктурированное сериализованное представление структур (LIST, HASH, ZSET) в памяти. Устраняет накладные расходы указателей, требуя перекодирования при операциях записи.

**Intset** — компактное представление множеств (SET) в виде отсортированного массива целых чисел. Применяется только в том случае, если элементы SET являются целыми числами по основанию 10 в пределах диапазона signed long целевой платформы.

**Event loop** — в предоставленных источниках не раскрыто.

## Как устроено

### Типы данных

**Строки (Strings)** хранят байтовые строки, целые числа (signed 32-bit или 64-bit) или числа с плавающей точкой стандарта IEEE 754 double.

**Списки (Lists)** представляют собой связанные списки строк с возможностью добавления и извлечения элементов с обоих концов через операции LPUSH и RPOP.

**Множества (Sets)** хранят неупорядоченные коллекции уникальных строк на базе хеш-таблиц с поддержкой пересечения, объединения и разности.

**Упорядоченные множества (ZSET)** связывают уникальные строковые элементы (members) с числами с плавающей точкой (scores) для обеспечения строгой сортировки.

**Хеши (Hashes)** хранят неупорядоченные таблицы маппинга ключей на значения.

**Битовые массивы (Bitmaps)** реализованы поверх структуры данных String. Операции считывания и записи (GETBIT, SETBIT) работают со смещениями битов от старших разрядов к младшим.

> В источниках упомянуто частично: HyperLogLog, Geospatial, Streams, Bitfields — детальное описание внутренних механизмов отсутствует, присутствует только упоминание о поддержке данных типов в Redis.

### Memory layout

В классическом виде узел связанного списка требует **21 байт** накладных расходов на 32-битных системах (3 указателя, 2 целых числа длин, строка и null-байт). Компактный ziplist хранит данные как `length, length, string`, сжимая накладные расходы до **~2 байт на элемент**.

Автоматическая конвертация ziplist или intset в стандартные хеш-таблицы или списки происходит при превышении заданных лимитов по количеству элементов или их длине.

### Основные команды

| Команда | Описание |
|---------|----------|
| `GET key` | Извлекает данные по ключу |
| `SET key value` | Устанавливает значение ключа |
| `DEL key` | Удаляет ключ и возвращает количество успешно удаленных ключей |
| `EXPIRE key seconds` | Задает таймаут удаления ключа в секундах |
| `TTL key` | Возвращает оставшееся время жизни ключа до удаления |
| `PERSIST key` | Отменяет таймер удаления |
| `MGET key1 key2 ...` | Атомарно получает значения множества ключей за один сетевой вызов |
| `MSET key1 val1 key2 val2 ...` | Атомарно устанавливает значения множества ключей за один сетевой вызов |

> В источниках упомянуто частично: команды KEYS, SCAN, EXISTS, TYPE детально не определены. Описана только команда HEXISTS для хешей.

### Настройка порогов компактных кодировок

Конфигурация определяет максимальное число элементов и максимальный размер значения (в байтах), при которых Redis применяет сверхкомпактное размещение структур в памяти:

```
list-max-ziplist-entries 512
list-max-ziplist-value 64
hash-max-ziplist-entries 512
hash-max-ziplist-value 64
zset-max-ziplist-entries 128
zset-max-ziplist-value 64
set-max-intset-entries 512
```

## Детали и нюансы

### Edge case: деградация производительности ziplist

Разрастание длины ziplist до 50 000 - 100 000 элементов приводит к катастрофическому снижению пропускной способности операций чтения/записи. Скорость выполнения операции RPOPLPUSH падает с ~50 000 операций в секунду до 553 операций в секунду. Экстремальная экономия оперативной памяти достигается за счет критической деградации производительности процессора при превышении лимитов.

### Edge case: потеря точности в ZSET score

Redis конвертирует score в структуру данных IEEE 754 double precision floating-point. Без потерь точности могут быть представлены только числа до 53 бит. При попытке использования 64-битных целых чисел в score может произойти потеря младших разрядов и возникнут коллизии при сортировке.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| **Redis** | Богатые структуры данных, снепшоты, репликация, механизм «издатель/подписчик» | Однопоточная архитектура | Когда нужны сложные структуры данных, persistence, pub/sub |
| **Memcached** | Многопоточное хранилище | Только политика вытеснения LRU, нет сложных структур данных, снепшотов, репликации и механизма «издатель/подписчик» | Когда достаточно простого кеширования строк |

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| **Ziplist (компактная кодировка)** | Сверхкомпактное хранение, ~2 байта накладных расходов на элемент | Требует перекодирования при записи; деградация на 50 000+ элементах | Малые структуры в пределах заданных лимитов |
| **Стандартные структуры (хеш-таблицы, списки)** | Высокая производительность чтения/записи на больших объемах | 21 байт накладных расходов на узел (32-bit) | Структуры с большим количеством элементов или длинными значениями |

## Важно / подводные камни / best practices

- **Короткие имена ключей.** Использование максимально коротких имен ключей, например `user:joe` вместо `username:joe`. На миллионах или миллиардах записей каждый лишний символ в имени ключа потребляет дополнительные гигабайты ценной оперативной памяти.
- **Шардирование HASH-таблиц.** Разбиение одной огромной HASH-таблицы на множество маленьких HASH-таблиц по 512--1024 элемента. Это позволяет Redis применить компактную кодировку ziplist к каждой отдельной HASH-таблице, что снижает потребление оперативной памяти на 70% (с 44 МБ до 12 МБ в тесте).
- **Контроль размера ziplist.** Не допускать разрастания ziplist до десятков тысяч элементов — падение производительности RPOPLPUSH с ~50 000 до 553 операций в секунду.
- **Осторожность с 64-битными score в ZSET.** IEEE 754 double precision представляет без потерь только числа до 53 бит — при использовании 64-битных целых возможна потеря младших разрядов и коллизии при сортировке.
- **Атомарность команд.** Однопоточная архитектура гарантирует атомарное выполнение команд (включая скрипты Lua), блокируя выполнение других команд до завершения текущей.

### Варианты использования (из источников)

- Кеширование сессий пользователей, корзин покупок и HTML-страниц (паттерн cache-aside)
- Размещение очередей задач (FIFO) с применением блокирующего чтения из списков (BLPOP)
- Системы таргетинга рекламы с использованием пересечений и объединений структур SET и ZSET (команды SINTERSTORE, ZUNIONSTORE)
- Агрегация логов и сбор статистик реального времени (через INCR и ZINCRBY)
