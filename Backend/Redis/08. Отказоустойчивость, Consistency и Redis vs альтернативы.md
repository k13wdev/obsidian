## Суть и контекст

Конспект посвящён механизмам обеспечения отказоустойчивости Redis (Sentinel, ручной failover), моделям согласованности данных (strong consistency, eventual consistency, strong eventual consistency), граничным случаям при сбоях в распределённых системах и сравнению Redis с Memcached. Тема охватывает CAP-теорему в контексте Redis, паттерны кэширования с точки зрения consistency/coherency, а также практические рекомендации по развёртыванию отказоустойчивых конфигураций.

---

## Ключевые идеи

- Redis Sentinel обеспечивает автоматический failover с голосованием и кворумом, а также service discovery для клиентов.
- CAP-теорема накладывает фундаментальное ограничение: из Consistency, Availability и Partition tolerance распределённая система может одновременно обеспечить только два свойства.
- Strong consistency достигается формулой R + W > N, а в Master-Slave архитектуре — направлением всех операций чтения и записи исключительно на Master-узел.
- Eventual consistency означает, что система станет полностью согласованной только после распространения всех операций записи по всем узлам в фоновом режиме.
- Strong eventual consistency в Redis Enterprise реализуется через Active-Active Geo-Distribution (Multi-master) с двунаправленной репликацией и разрешением конфликтов через CRDT.
- Cache consistency и cache coherency — разные понятия: coherency является более строгим термином.
- Сбой ведущего при активной записи приводит к потере данных, не успевших реплицироваться. Длительность окна потери зависит от сетевой задержки.
- Split brain при чётном количестве узлов блокирует failover, а данные в изолированном ведущем будут уничтожены при восстановлении.
- Master-Master через взаимный SLAVEOF в Open Source Redis приводит к бесконечному циклу, CPU 100% и потере данных.
- Redis не следует использовать для 100% ACID без потери единой транзакции.
- Redis и Memcached обе обеспечивают задержки менее миллисекунды и поддерживают секционирование, но Redis значительно богаче по функциональности.

## Определения и термины

**INFO** — команда, предоставляющая информацию о текущем статусе сервера Redis: использовании памяти, количестве подключенных клиентов, количестве ключей в базах данных и количестве выполненных команд после последнего снепшота.

**Redis Sentinel** — процесс, который отслеживает поведение и состояние ведущих и подчинённых узлов, и автоматически обрабатывает failover. Включает service discovery.

**Кворум (Quorum)** — минимальное число голосов узлов для разрешения выполнения операций failover. Рекомендуется нечётное количество узлов.

**CAP-теорема** — из трёх свойств — Consistency, Availability, Partition tolerance — распределённая система может одновременно обеспечить только два. Доступность = любой узел способен читать и писать.

**Eventual consistency** — система станет полностью согласованной только после распространения всех операций записи по всем узлам в фоновом режиме.

**Cache consistency** — мера соответствия данных в кэше данным в базовом хранилище.

**Cache coherency** — поведение системы при чтении/записи одного ключа, при котором система функционирует так, будто кэша не существует. Более строгий термин, чем cache consistency.

**Split brain** — разделение кластера коммуникационным разрывом на изолированные разделы.

**CRDT (Conflict-free Replicated Data Types)** — механизм разрешения конфликтов, используемый в Redis Enterprise для Strong Eventual Consistency при Active-Active Geo-Distribution.

## Как устроено

### Механика Sentinel Failover

1. Кластер Sentinel постоянно опрашивает (PING) доступность узлов
2. При недоступности ведущего — голосование
3. Если кворум достигнут — failover
4. Один из подчинённых назначается новым ведущим
5. Sentinel переключает остальные подчинённые на нового ведущего
6. Sentinel сообщает клиентам новые координаты

### Ручной Failover

1. Выполнить `SAVE` на доступном Slave (machine-b)
2. Скопировать `dump.rdb` через `scp` на резервный сервер (machine-c)
3. Запустить Redis на machine-c
4. Выполнить `SLAVEOF machine-c.vpn 6379` на machine-b

### Strong Consistency

Формула: **R + W > N**, где R — число узлов для чтения, W — число узлов для записи, N — фактор репликации.

В архитектуре Master-Slave: все операции чтения и записи направляются исключительно на Master-узел.

### Strong Eventual Consistency (Redis Enterprise)

Реализуется через Active-Active Geo-Distribution (Multi-master):
- Несколько ведущих экземпляров в разных дата-центрах
- Двунаправленная репликация
- Конфликты разрешаются через CRDT (Conflict-free Replicated Data Types)

## Детали и нюансы

### Сбой ведущего при активной записи

Все операции, не успевшие реплицироваться, утеряны. Длительность окна потери зависит от сетевой задержки.

### Split brain при чётном количестве узлов Sentinel/Master

Голоса распределяются поровну. Failover блокируется. Данные в изолированном ведущем будут уничтожены при восстановлении.

### Master-Master через взаимный SLAVEOF в Open Source Redis

Приводит к бесконечному циклу, CPU 100%, потере данных. Master-Master возможен только через CRDT в Redis Enterprise.

### Мониторинг, TLS, ACL, тюнинг сетевого стека

> В предоставленных источниках не раскрыто.

## Сравнения и trade-offs

### Redis vs Memcached

| Характеристика | Redis | Memcached |
|----------------|-------|-----------|
| Потоки | Однопоточная | Многопоточная |
| Структуры данных | List, Set, Hash, Zset, Geo, HyperLogLog, Stream | Только строки |
| Eviction-политики | Множество | Только LRU |
| Персистентность | AOF / RDB | Нет снепшотов |
| Репликация | Да | Нет |
| Транзакции | Да | Нет |
| Lua-скрипты | Да | Нет |
| Pub/Sub | Да | Нет |
| Задержки | Менее миллисекунды | Менее миллисекунды |
| Секционирование | Да | Да |

### Cache consistency vs Cache coherency

Cache consistency — мера соответствия данных в кэше данным в базовом хранилище. Cache coherency — более строгий термин: поведение системы при чтении/записи одного ключа, при котором система функционирует так, будто кэша не существует.

### Модели согласованности

| Модель | Суть | Как достигается |
|--------|------|-----------------|
| Strong consistency | R + W > N | Все чтение и запись на Master-узел |
| Eventual consistency | Согласованность после распространения записей | Фоновая репликация по всем узлам |
| Strong eventual consistency | Multi-master с разрешением конфликтов | Active-Active Geo-Distribution + CRDT (Redis Enterprise) |

## Примеры

### Ручной failover

```bash
root@machine-b ~:$ redis-cli
redis 127.0.0.1:6379> SAVE
redis 127.0.0.1:6379> QUIT
root@machine-b ~:$ scp /var/local/redis/dump.rdb machine-c.vpn:/var/local/redis/dump.rdb
root@machine-b ~:$ ssh machine-c.vpn
root@machine-c ~:$ sudo /etc/init.d/redis-server start
root@machine-c ~:$ exit
root@machine-b ~:$ redis-cli
redis 127.0.0.1:6379> SLAVEOF machine-c.vpn 6379
```

## Важно / подводные камни / best practices

- **Sentinel локально на каждом сервере приложения.** Sentinel должен быть развёрнут вместе с приложением.
- **Нечётное количество узлов** (3, 5, 7) с минимум 2 подчинёнными. Чётное количество приводит к split brain.
- **Кворум:** 3 узла — кворум 2, 5 узлов — кворум 3.
- **Не использовать Redis для 100% ACID** без потери единой транзакции.
- **Master-Master в Open Source Redis невозможен** — взаимный SLAVEOF приводит к бесконечному циклу, CPU 100% и потере данных. Для multi-master необходим Redis Enterprise с CRDT.
- **Окно потери данных** при сбое ведущего определяется сетевой задержкой между Master и Slave.
