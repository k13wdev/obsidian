# Kafka — основы, архитектура и partitioning

## Суть и контекст

Apache Kafka — распределённый, реплицируемый сервис журнала фиксации изменений (commit log), работающий по модели Pub-Sub. В отличие от классических брокеров сообщений, Kafka сохраняет сообщения на диск, упорядочивает их внутри партиций и не удаляет после прочтения (работают политики хранения). Kafka работает по модели «тупой брокер, умный потребитель» — брокер только пишет лог на диск, а потребители самостоятельно запрашивают (pull) данные и следят за своим курсором чтения (offset).

---

## Ключевые идеи

- Event streaming — надёжная, упорядоченная, неограниченная последовательность неизменяемых записей событий
- Kafka — commit log с Pub-Sub моделью, сообщения не удаляются после прочтения
- Topic — логическое объединение партиций (аналог таблицы в БД)
- Partition — единица масштабирования, строго упорядоченный append-only лог
- Segment — физический фрагмент партиции (.log, .index, .timeindex)
- Consumer Group — набор потребителей с общим group.id, разделяющих нагрузку
- Одна партиция внутри группы = максимум один потребитель
- Offset — 64-битное целое, логический порядковый номер записи в партиции
- Rebalance — переназначение партиций между потребителями (stop-the-world при Eager)
- ZooKeeper → KRaft — переход от внешнего сервиса метаданных к Raft-based контроллеру
- Leader обслуживает все чтения и записи, Follower пассивно реплицирует

---

## Определения и термины

**Event streaming (Потоковая передача событий)** — надёжная, полностью упорядоченная, неограниченная последовательность неизменяемых записей событий, доставляемая подписчикам как минимум один раз. Позволяет обрабатывать данные по мере поступления, в отличие от пакетной обработки.

**Apache Kafka** — распределённый, реплицируемый сервис журнала фиксации изменений (commit log), работающий по модели Pub-Sub. Сообщения сохраняются на диск, упорядочиваются внутри партиций и не удаляются после прочтения.

**Broker (Брокер)** — отдельный узел (сервер) Kafka, обрабатывающий запросы клиентов, назначающий смещения (offsets) и записывающий сообщения в локальное хранилище на диске.

**Topic (Топик)** — логическое объединение партиций; именованный канал для передачи сообщений определённого типа. Аналог таблицы в базе данных. Не является физическим файлом, в отличие от партиции.

**Partition (Партиция)** — строго упорядоченная, неизменяемая последовательность записей в виде журнала (log). Единица масштабирования и распараллеливания в Kafka. Физически представляет собой директорию на диске брокера.

**Segment (Сегмент)** — фрагмент файла партиции. Состоит из файла данных (.log) и файлов индексов (.index, .timeindex). По умолчанию закрывается при достижении 1 ГБ или по истечении 7 дней. Активен для записи всегда только один сегмент.

**Producer (Продюсер)** — клиентское приложение, формирующее и отправляющее сообщения в топик Kafka.

**Consumer (Потребитель)** — клиентское приложение, подписывающееся на один или несколько топиков и вытягивающее (pull) из них сообщения.

**Consumer Group (Группа потребителей)** — набор потребителей (определяемый общим `group.id`), разделяющих между собой нагрузку по чтению партиций топика. Одна партиция внутри одной группы может быть назначена максимум одному потребителю одновременно.

**Offset (Смещение)** — 64-битное целое число со знаком. Логический порядковый номер записи внутри конкретной партиции.

**Rebalance (Ребалансировка)** — процесс переназначения партиций между активными потребителями внутри Consumer Group.

**ZooKeeper** — централизованный сервис для поддержания метаданных кластера, списков брокеров, ACL и выбора контроллера.

**KRaft** — Raft-based контроллерный кворум, заменяющий ZooKeeper. Хранит метаданные кластера в виде внутреннего лог-топика Kafka.

**Leader Replica (Лидер-реплика)** — единственная реплика партиции, обслуживающая все запросы на чтение и запись от клиентов (по умолчанию).

**Follower Replica (Ведомая реплика)** — реплика, пассивно дублирующая данные с лидера для обеспечения отказоустойчивости.

## Как устроено

### Partitioning — распределение данных продюсером

1. Если партиция указана в `ProducerRecord` явно — используется она
2. Если ключ задан (не null) — хэшируется алгоритмом murmur2. 31 старший бит маскируется, затем остаток от деления на количество партиций: `Math.abs(Utils.murmur2(keyBytes)) % numPartitions`
3. Если ключ null — алгоритм Sticky Round-Robin. Батч заполняется для случайной партиции, затем переключение на следующую для минимизации запросов

### Rebalance — ребалансировка Consumer Group

1. Потребитель отправляет `JoinGroupRequest` узлу GroupCoordinator (брокеру)
2. Координатор ждёт присоединения других потребителей (задержка `group.initial.rebalance.delay.ms`, по умолчанию 3000 мс)
3. Координатор выбирает Group Leader (первый присоединившийся) и отправляет ему список участников через `JoinGroupResponse`
4. Group Leader выполняет локальное распределение партиций (PartitionAssignor: Range, RoundRobin, Sticky, CooperativeSticky)
5. Leader отправляет карту назначений координатору в `SyncGroupRequest`
6. Координатор рассылает `SyncGroupResponse` каждому потребителю с его списком партиций

### Commit Offset — фиксация смещения

1. Consumer считывает батч сообщений
2. При `enable.auto.commit` или ручном `commitSync`/`commitAsync` — клиент отправляет `OffsetCommitRequest` брокеру-координатору группы
3. Координатор сохраняет смещение в системный топик `__consumer_offsets` в виде сообщения: ключ (Group, Topic, Partition), значение (Offset)

### Файловая система партиции

| Файл | Назначение |
|------|-----------|
| `[base_offset].log` | Бинарные данные сообщений, zero-copy передача из page cache |
| `[base_offset].index` | mmap-файл, offset (4 байта) → позиция в .log (4 байта), шаг: `index.interval.bytes` |
| `[base_offset].timeindex` | Маппинг UNIX timestamp → позиция в .log |
| `leader-epoch-checkpoint` | Контроль эпох для восстановления после сбоев |

Каталог: `/tmp/kafka-logs/topic_name-partition_number/`

## Детали и нюансы

### Потребителей больше, чем партиций

Лишние потребители в Consumer Group находятся в режиме ожидания (idle) и не получают сообщений. Максимальный параллелизм в рамках одной группы строго равен количеству партиций.

### Изменение количества партиций

Функция `hash(key) % numPartitions` меняет результат. Существующие сообщения остаются в старых партициях, новые с тем же ключом летят в другие. Нарушается гарантия строгого упорядочивания сообщений для конкретного ключа. Невозможно уменьшить количество партиций без удаления топика.

### Eager Rebalancing — stop-the-world

Все потребители отзывают свои партиции (`onPartitionsRevoked`), прекращают чтение, синхронизируются и получают назначения заново. Ни один потребитель не читает сообщения в процессе перебалансировки. Решается переходом на Cooperative Rebalancing.

### Zombie Consumer

Coordinator не получает heartbeat в течение `session.timeout.ms` и инициирует Rebalance. Зависший потребитель просыпается и пытается закоммитить offset для партиции, которая ему уже не принадлежит. Решение: Transactional IDs и эпохи (epoch) для изоляции зомби-процессов (Zombie Fencing).

### Unclean Leader Election

Если `unclean.leader.election.enable=false`, при отсутствии ISR партиция недоступна для чтения и записи. Если `true` — лидером назначается отставшая реплика. Trade-off: согласованность данных vs доступность системы (потеря последних сообщений в обмен на возобновление работы).

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|-------------------|
| Много партиций | Высокий параллелизм | Больше файловых дескрипторов, перегрузка Controller при ребалансировках | Высокая нагрузка, много консьюмеров |
| Мало партиций | Простота, меньше ресурсов | Ограниченный параллелизм | Низкий трафик, строгий порядок |
| Eager Rebalance | Простота реализации | Stop-the-world пауза | Простые сценарии |
| Cooperative Rebalance | Инкрементальный, без полной остановки | Сложнее, требует поддержки клиентом | Production-системы |

| Аналогия Kafka ↔ РСУБД |
|-------------------------|
| Topic = Таблица |
| Partition = Шард (Shard) |
| Offset = Идентификатор строки |

## Примеры

### Создание топика

```bash
kafka-topics.sh --create --topic my-topic \
  --partitions 8 --replication-factor 2 \
  --bootstrap-server localhost:9092
```

Создаёт топик `my-topic` из 8 партиций, каждая с 2 копиями (Leader + 1 Follower).

### Описание состояния топика

```bash
kafka-topics.sh --describe --topic my-topic \
  --bootstrap-server localhost:9092
```

Выводит список партиций, ID лидера, Replicas и ISR (In-Sync Replicas).

### Кастомный Partitioner (Java)

```java
public class CustomPartitioner implements Partitioner {
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();
        if (key.equals("CRITICAL")) {
            return numPartitions - 1; // dedicated partition
        }
        return Math.abs(Utils.murmur2(keyBytes)) % (numPartitions - 1);
    }
}
```

Перенаправляет критические сообщения в выделенную (последнюю) партицию, остальные хэшируются по оставшимся.

## Важно / подводные камни / best practices

- **Максимальный параллелизм = количество партиций** — лишние консьюмеры в группе будут idle
- **Не меняйте количество партиций для ключевого роутинга** — нарушается ordering для конкретных ключей
- **Нельзя уменьшить число партиций** — только удалить и пересоздать топик
- **Коммитьте offset только после успешной обработки** — `enable.auto.commit=true` при медленной обработке приводит к потере данных (at-most-once)
- **Не превышайте max.poll.interval.ms** — иначе Coordinator запустит Rebalance, что может вызвать бесконечный цикл (livelock)
- **min.insync.replicas = RF-1, acks=all** — рекомендация для production (RF=3, min.insync.replicas=2)
- **Используйте Cooperative Rebalancing** — для избежания stop-the-world пауз
