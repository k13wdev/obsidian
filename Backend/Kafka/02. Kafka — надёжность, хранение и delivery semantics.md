# Kafka — надёжность, хранение и delivery semantics

## Суть и контекст

Надёжность Kafka строится на репликации данных между брокерами (Leader/Follower, ISR), настройке уровня подтверждения записи (acks) и политиках хранения данных. Kafka хранит сообщения в append-only логе на диске, используя page cache ОС и zero-copy для высокой производительности. Delivery semantics варьируются от at-most-once до exactly-once, последняя реализуется через idempotent producer и Kafka Transactions.

---

## Ключевые идеи

- Replication Factor — количество копий каждой партиции (рекомендуется 3 для production)
- Leader обслуживает чтения/записи, Follower пассивно реплицирует через Fetch-запросы
- ISR (In-Sync Replicas) — динамический набор реплик, синхронизированных с лидером
- acks=0/1/all определяет уровень гарантий на стороне продюсера
- min.insync.replicas задаёт минимум ISR для подтверждения при acks=all
- Unclean Leader Election — выбор out-of-sync реплики как лидера (потеря данных vs доступность)
- Append-only log + sequential I/O + page cache + zero-copy = высокая производительность
- Log compaction — для каждого ключа сохраняется только последнее значение
- Idempotent Producer — PID + sequence ID предотвращают дубликаты при ретраях
- Kafka Transactions (2PC) — атомарная запись в несколько топиков + коммит offset

## Определения и термины

**Replication Factor** — количество копий данных (реплик) для каждой партиции топика в кластере. Значение 3 является рекомендуемым для production, обеспечивая баланс между надёжностью и затратами на хранение.

**Leader Replica** — единственная реплика партиции, через которую проходят все запросы на чтение и запись. При падении лидера его место занимает одна из ISR реплик. В версиях с KIP-392 консьюмеры могут читать из фолловеров для снижения latency.

**Follower Replica** — реплика, пассивно реплицирующая сообщения от лидера. Фолловеры отправляют Fetch-запросы лидеру с указанием offset, что позволяет лидеру вычислять отставание каждой реплики.

**In-Sync Replica (ISR)** — динамический набор реплик (включая лидера), отстающих от лидера не более чем на заданное время (по умолчанию 10–30 секунд). Только реплики из ISR могут быть выбраны новым лидером при штатном сбое.

**min.insync.replicas** — минимально допустимое количество ISR (включая лидера), необходимое для подтверждения записи при `acks=all`. Задаётся на уровне топика или брокера. При падении ISR ниже значения — продюсеры получают `NotEnoughReplicasException`.

**Unclean Leader Election** — выбор новой лидер-реплики из числа out-of-sync реплик при недоступности всех ISR. Контролируется `unclean.leader.election.enable`. Если включено, восстанавливает доступность ценой безвозвратной потери данных.

**Append-only log** — неизменяемая структура данных на диске, в конец которой последовательно добавляются новые сообщения. Высокая скорость записи за счёт sequential I/O.

**Segments** — фрагменты файла партиции. Активный сегмент один, остальные закрыты. По умолчанию закрывается при 1 ГБ или 7 дней.

**Page Cache** — кэш ОС, используемый Kafka для хранения часто запрашиваемых сообщений в RAM. Избавляет от кэширования в JVM heap, снижая накладные расходы на GC.

**Zero-copy** — оптимизация ОС (sendfile), при которой данные передаются напрямую из page cache в сетевой сокет, минуя user-space. Снижает нагрузку на CPU. Отключается при включённой перепаковке/компрессии на брокере.

**Log Compaction** — политика очистки журнала, при которой для каждого уникального ключа сохраняется только последнее значение, старые записи удаляются. Для удаления ключа — tombstone-сообщение (value=null).

**Idempotent Producer** — продюсер, снабжающий каждое сообщение PID (producer ID) и sequence ID. Брокер отбрасывает дубликаты при сетевых ретраях. Включается `enable.idempotence=true`. Гарантирует строгий порядок при `max.in.flight.requests.per.connection` до 5.

**Kafka Transactions** — механизм 2-Phase Commit (2PC), обеспечивающий атомарную запись в несколько топиков и партиций, а также атомарный коммит оффсетов консьюмера. Опирается на системный топик `__transaction_state` и Transaction Coordinator.

## Как устроено

### Репликация и обработка сбоев

1. Controller назначает leader и follower для каждой партиции
2. Producer отправляет сообщения в leader-реплику
3. Leader записывает в локальный лог
4. Follower-реплики отправляют асинхронные Fetch-запросы лидеру (со следующего нужного offset)
5. Leader фиксирует (commit) сообщение и отвечает продюсеру (при `acks=all`) только после подтверждения всеми ISR
6. Если follower не успевает за лидером дольше `replica.lag.time.max.ms` (например, 30 сек), leader удаляет его из ISR
7. При сбое лидера — Controller назначает нового из ISR (или out-of-sync при `unclean.leader.election.enable=true`)

### Уровни acks продюсера

| acks | Поведение | Надёжность | Производительность |
|------|----------|-----------|-------------------|
| 0 | Fire and forget, не ждёт ответа | Потеря возможна | Максимальная |
| 1 | Ждёт записи только в leader | Потеря при падении leader до репликации | Высокая |
| all (-1) | Ждёт записи во все ISR | Потеря только при катастрофе | Ниже (зависит от ISR) |

### Хранение на диске

Партиция разбита на сегменты. На каждый сегмент — минимум 3 файла в `/tmp/kafka-logs/[topic]-[partition]/`:

- **[base_offset].log** — бинарные данные сообщений
- **[base_offset].index** — mmap-файл, offset (4 байта) → позиция в .log (4 байта). Записи с шагом `index.interval.bytes` (по умолчанию 4096 байт)
- **[base_offset].timeindex** — маппинг timestamp → позиция в .log

Запись: в Page Cache, fsync на диск асинхронно ОС (Kafka не ждёт физического сброса).
Чтение: zero-copy (sendfile) из Page Cache в сетевой сокет, CPU не участвует.

### Log Compaction

1. Фоновые cleaner threads сканируют закрытые «грязные» (dirty) сегменты
2. Формируется in-memory хэш-таблица: Ключ → Максимальный Offset
3. Cleaner читает старые «чистые» (clean) сегменты, сверяет с хэш-таблицей
4. Если offset сообщения < offset в хэш-таблице для того же ключа — сообщение отбрасывается
5. Tombstone-маркер (payload=null) сохраняется на время `delete.retention.ms`, затем полностью удаляется

### Kafka Transactions (Consume-Transform-Produce)

1. Producer вызывает `initTransactions()`, получая `transactional.id` и epoch (предотвращает zombie-продюсеров)
2. `beginTransaction()` — Transaction Coordinator записывает старт в `__transaction_state`
3. Producer отправляет сообщения в целевые партиции + `sendOffsetsToTransaction()` для оффсетов
4. `commitTransaction()` — Coordinator логирует намерение коммита
5. Coordinator асинхронно записывает Control Markers (type=1 для коммита) во все затронутые партиции
6. Консьюмер с `isolation.level=read_committed` буферизирует чтение до LSO (Last Stable Offset), не отдавая приложению до маркера коммита

## Детали и нюансы

### Потеря данных при acks=1

Лидер записывает, отправляет ACK и падает до репликации. Follower-реплики не получили данные. Данные безвозвратно теряются, хотя продюсер считает их отправленными.

### Unclean Leader Election + восстановление ISR

Отставшая реплика становится лидером, принимает новые сообщения. Когда старые ISR оживают, они обрезают логи (truncate) до уровня нового лидера. Все сообщения, отсутствовавшие на отставшем лидере, навсегда удаляются.

### Разрыв связи с ZooKeeper

Брокер теряет сессию с ZooKeeper (срабатывает `zookeeper.session.timeout.ms`). Controller исключает брокера, удаляя из ISR, и инициирует выборы нового лидера.

### Zombie Consumer и max.poll.interval.ms

Coordinator считает долго обрабатывающего консьюмера мёртвым, запускает Rebalance. Зомби-консьюмер просыпается и получает ошибку `IllegalGeneration` / `CommitFailedException`. Настройка должна балансировать между быстрым обнаружением сбоев и предотвращением случайных ребалансировок.

## Сравнения и trade-offs

| Настройка | Надёжность | Производительность | Когда использовать |
|----------|-----------|-------------------|-------------------|
| acks=0 | Минимальная | Максимальная | Метрики, логи |
| acks=1 | Средняя (потеря при падении leader) | Высокая | Некритичные данные |
| acks=all + min.insync.replicas=2 | Высокая | Средняя | Критичные данные (production) |

| Retention Policy | Описание | Когда использовать |
|-----------------|----------|-------------------|
| delete (time-based) | Удаление сегментов старше retention.ms | Классический event log |
| delete (size-based) | Удаление при превышении retention.bytes | Ограничение дискового пространства |
| compact | Сохранение только последнего значения для ключа | Состояние (таблицы, конфигурация) |

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| Idempotent Producer | Нет дубликатов, строгий порядок | Только в рамках одной партиции |
| Kafka Transactions | Атомарность между топиками и оффсетами | Overhead, transactional.id утечка памяти |

## Примеры

### Создание топика с compaction

```bash
kafka-topics.sh --create --topic compact-topic \
  --partitions 2 --replication-factor 2 \
  --config cleanup.policy=compact \
  --bootstrap-server localhost:9092
```

### Дамп сегментов лога

```bash
kafka-run-class.sh kafka.tools.DumpLogSegments \
  --files /tmp/kafka-logs/my-topic-0/00000000000000000000.log \
  --print-data-log
```

Декодирует бинарный файл сегмента, выводит offset, crc, timestamp, размер ключа/значения и payload.

### Idempotent Producer (Java)

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("enable.idempotence", "true");
props.put("acks", "all");
props.put("retries", Integer.MAX_VALUE);
props.put("max.in.flight.requests.per.connection", "5");
```

Включает защиту от дубликатов. Требует `acks=all` и `retries > 0`. `in-flight <= 5` поддерживает строгий порядок.

### Kafka Transactions (Consume-Transform-Produce)

```java
producer.initTransactions();
try {
    producer.beginTransaction();
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        producer.send(new ProducerRecord<>("output-topic", transform(record)));
    }
    producer.sendOffsetsToTransaction(offsets, consumerGroupId);
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
}
```

Связывает чтение, обработку, публикацию и фиксацию offset в единую атомарную операцию.

## Важно / подводные камни / best practices

- **RF=3, min.insync.replicas=2, acks=all** — golden standard для production. Сообщение на двух серверах до ACK
- **Не используйте enable.auto.commit=true при exactly-once** — offset коммитится по таймеру, при crash — потеря данных (at-most-once)
- **Не создавайте слишком много партиций** — каждая = директория + 3 файловых дескриптора. Тысячи партиций перегружают Controller и Page Cache
- **unclean.leader.election.enable=false** для критичных данных — иначе потеря сообщений при восстановлении out-of-sync реплики
- **Tombstone при log compaction** — для удаления ключа отправьте value=null, он сохраняется на `delete.retention.ms`
- **transactional.id утечка памяти** — 3 новых ID/сек за неделю = 5 ГБ ОЗУ. Переиспользуйте ID
