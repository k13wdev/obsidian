# TDD и BDD

## Суть и контекст

TDD (Test-Driven Development) и BDD (Behavior-Driven Development) — методологии разработки ПО, определяющие порядок написания тестов и кода. В Go TDD реализуется стандартными средствами (`testing`, Testify), BDD — фреймворком Ginkgo с библиотекой матчеров Gomega. BDD дополняет TDD, не заменяет его.

---

## Ключевые идеи

- TDD: сначала тест, потом код; цикл red-green-refactor
- BDD: тесты описывают поведение в читаемых терминах (`Describe` / `Context` / `It`)
- BDD дополняет TDD — используются совместно, а не вместо
- Ginkgo — фреймворк BDD для Go; Gomega — библиотека матчеров для Ginkgo
- Принцип Given-When-Then — структурирует тест через предусловия, действие и результат
- `BeforeEach`/`AfterEach` — мутируемое состояние должно инициализироваться только внутри них
- Для простых функций (утилиты, парсеры) table-driven tests эффективнее BDD

---

## Определения и термины

**TDD (Test-Driven Development)** — методология разработки ПО, при которой написание тестов предшествует написанию кода.

**BDD (Behavior-Driven Development)** — общая модель разработки для обеспечения того, чтобы приложение делало то, что оно должно делать на каждом шаге. Процессы и ожидаемые результаты описываются в понятных человеку (human-friendly) терминах. Дополняет TDD и классическое модульное тестирование.

**Ginkgo** — фреймворк для тестирования в Go, использующий модель BDD. Устанавливается командой `go get github.com/onsi/ginkgo/ginkgo`. Формализует тестирование в виде читаемых блоков.

**Gomega** — библиотека-зависимость для Ginkgo, предоставляющая инструменты утверждений (matchers).

**Принцип Given-When-Then (Дано-Когда-Тогда)** — способ написания тестов, описывающий предварительные условия (preconditions — given), действия (actions — when) и фактический результат (actual outcome — then).

## Как устроено

### Цикл TDD: Red-Green-Refactor

1. **Red** — написать тест, который падает (логика ещё не реализована)
2. **Green** — написать минимальный код, достаточный для прохождения теста
3. **Refactor** — улучшить код при сохранении зелёного статуса тестов

### Иерархия блоков Ginkgo

```
Describe("компонент или операция")
  └── Context("условие или состояние")
        └── It("ожидаемое поведение")
```

| Блок | Назначение |
|------|-----------|
| `Describe("...", func() {...})` | Верхнеуровневое описание тестируемого объекта или операции |
| `Context("...", func() {...})` | Группирует тесты по конкретному сценарию или состоянию |
| `It("...", func() {...})` | Содержит сам тест и утверждения (assertions) |
| `BeforeEach(func() {...})` | Выполняется перед каждым `It` в текущем блоке |
| `AfterEach(func() {...})` | Выполняется после каждого `It` в текущем блоке |

### Жизненный цикл хуков Ginkgo

1. Выполняется `BeforeEach` текущего `Describe`
2. Выполняется `BeforeEach` вложенного `Context`
3. Выполняется логика внутри `It` (и проверки `Expect`)
4. Выполняется `AfterEach` в обратном порядке (от внутренних к внешним)

## Примеры

### Базовая структура BDD-теста (Ginkgo)

```go
import (
    . "github.com/onsi/ginkgo/v2"
    . "github.com/onsi/gomega"
)

var _ = Describe("receive new remote TCP connection", func() {
    Context("user enters a number", func() {
        It("should be an integer", func() {
            Expect(parseInput("42")).To(Equal(42))
        })
    })

    Context("user enters a string", func() {
        It("should return an error", func() {
            _, err := parseInput("abc")
            Expect(err).ShouldNot(BeNil())
        })
    })
})
```

### Хуки и setup/teardown

```go
var _ = Describe("UserService", func() {
    var (
        mockRepo *MockRepository
        svc      *UserService
    )

    BeforeEach(func() {
        mockRepo = new(MockRepository)
        svc = NewUserService(mockRepo)
    })

    AfterEach(func() {
        mockRepo.AssertExpectations(GinkgoT())
    })

    Context("when user exists", func() {
        It("should return the user", func() {
            mockRepo.On("FindByID", 1).Return(&User{Name: "John"}, nil)

            user, err := svc.GetUser(1)

            Expect(err).To(BeNil())
            Expect(user.Name).To(Equal("John"))
        })
    })
})
```

### Точка входа для Ginkgo

```go
// Обязательно в пакете _test
func TestSuite(t *testing.T) {
    RegisterFailHandler(Fail)
    RunSpecs(t, "My Suite")
}
```

### Принцип Given-When-Then в обычных тестах

```go
func TestUserService_GetUser(t *testing.T) {
    // Given
    mockRepo := new(MockRepository)
    svc := NewUserService(mockRepo)
    mockRepo.On("FindByID", 1).Return(&User{Name: "John"}, nil)

    // When
    user, err := svc.GetUser(1)

    // Then
    assert.NoError(t, err)
    assert.Equal(t, "John", user.Name)
}
```

## Сравнения и trade-offs

| | TDD | BDD (Ginkgo) |
|--|-----|-------------|
| **Когда использовать** | Любой код; особенно утилиты, алгоритмы, бизнес-логика | Поведение систем с точки зрения пользователя/другой системы |
| **Стиль тестов** | Функциональный (`TestXxx`) | Декларативный (`Describe`/`Context`/`It`) |
| **Читаемость** | Технически точная | Близка к бизнес-требованиям |
| **Объём кода** | Меньше | Больше шаблонного кода |
| **Для простых функций** | Эффективнее | Избыточно |

BDD дополняет TDD: TDD — про реализацию, BDD — про поведение с точки зрения пользователя.

## Важно / подводные камни / best practices

**Утечка состояния между `It`:** Если инициализировать переменные в теле `Describe`/`Context` (не внутри `BeforeEach`), их изменение в одном `It` повлияет на другие. Мутируемое состояние — только в `BeforeEach`/`AfterEach`.

**Expect вне `It`:** Код с `Expect()`, оказавшийся в `Describe`/`Context` за пределами `It` или `BeforeEach`, выполнится на этапе построения дерева тестов и вызовет panic: `Expectation outside of It/BeforeEach`.

**BDD ≠ замена TDD:** BDD не заменяет модульные тесты. Для математических функций и утилит table-driven tests лаконичнее и эффективнее.
