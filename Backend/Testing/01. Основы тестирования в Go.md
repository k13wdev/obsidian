# Основы тестирования в Go

## Суть и контекст

Стандартный пакет `testing` — встроенный инструментарий Go для написания модульных, интеграционных и параллельных тестов. Тесты живут рядом с кодом в файлах `_test.go` и запускаются утилитой `go test`.

---

## Ключевые идеи

- Тестовая функция обязана начинаться с префикса `Test` и принимать `t *testing.T`
- Файл теста обязан иметь суффикс `_test.go`
- `t.Error`/`t.Errorf` — фиксируют сбой, но продолжают выполнение
- `t.Fatal`/`t.Fatalf` — фиксируют сбой и немедленно останавливают тест
- `t.Run` запускает подтесты (subtests) с изолированным контекстом
- Table-driven tests — паттерн с `[]struct` для многократной проверки одной логики
- `t.Parallel()` переводит тест в режим параллельного выполнения
- `t.Cleanup()` регистрирует teardown, выполняемый после теста и всех подтестов
- Флаг `-short` + `testing.Short()` позволяет пропускать медленные тесты
- `GOCACHE=off` не работает в Go 1.12+; для обхода кэша используется `-count=1`

---

## Определения и термины

**Unit test (Модульный тест)** — способ тестирования небольших участков кода, таких как функции и методы. Размещаются в том же пакете, что и тестируемая функция, в файлах `_test.go`.

**Table-driven tests (Табличные тесты)** — подход к организации тестового случая в виде таблицы, содержащей входные данные и ожидаемые результаты. Применяется для многократного использования одной и той же логики проверок. Реализуется через срез структур и цикл `for`, внутри которого вызывается подтест.

**Subtest (Подтест)** — изолированный тест, который запускается внутри родительской тестовой функции с помощью `t.Run()`. При выполнении формирует имя в формате `[ИмяФункции]/[ИмяПодтеста]`.

**Build tags** — теги сборки для скрытия или запуска определённых групп тестов. Позволяют прятать интеграционные тесты от выполнения при обычном прогоне юнит-тестов.

---

## Как устроено

### Жизненный цикл `go test`

1. Инструмент сканирует директорию на наличие файлов с суффиксом `_test.go`
2. Выполняется поиск и запуск функций с префиксом `Test`, принимающих `t *testing.T`
3. Для табличных тестов в цикле инициируются подтесты через `t.Run()`
4. При `t.Error()`/`t.Errorf()` — сбой фиксируется, выполнение продолжается
5. При `t.Fatal()`/`t.Fatalf()` — выполнение функции теста немедленно прерывается
6. После теста и всех его подтестов вызываются функции, зарегистрированные через `t.Cleanup()`

### Параллельное выполнение (`t.Parallel`)

По умолчанию тесты исполняются последовательно. При вызове `t.Parallel()`:
- Тест переводится в состояние **PAUSE**
- После завершения всех непараллельных тестов «приостановленные» тесты возобновляются (**CONT**) и выполняются параллельно
- Максимальное число одновременно работающих тестов задаётся `GOMAXPROCS` (по умолчанию = числу ядер CPU)

## Детали и нюансы

### t.Error vs t.Fatal

| Метод | Поведение |
|-------|-----------|
| `t.Error` / `t.Errorf` | Фиксирует сбой, сохраняет лог, продолжает выполнение функции |
| `t.Fatal` / `t.Fatalf` | Фиксирует сбой, немедленно прерывает выполнение функции теста |

Если после проваленной проверки дальнейшее выполнение вызовет панику (например, доступ к `nil`-указателю) — строго использовать `t.Fatal`.

### t.Helper

`t.Helper()` игнорирует номер строки самой вспомогательной функции в логах, перенаправляя стек ошибки на строку оригинального теста, где был произведён вызов.

### t.TempDir

`t.TempDir()` создаёт временную директорию и автоматически удаляет её по завершении теста.

### Таймаут

При превышении `-timeout` (например, `15s`) `go test` жёстко прерывает процесс с сообщением:
```
panic: test timed out after 15s
```
и выдаёт трассировку стека, включая файл `_testmain.go`.

### Изоляция медленных тестов

Флаг `-short` устанавливает `testing.Short()` в `true`. На основе этого условия тест вызывает `t.Skip()`, предотвращая выполнение ресурсоёмких тестов. Альтернатива — build tags.

## Сравнения и trade-offs

| Подход | Плюсы | Минусы | Когда использовать |
|--------|-------|--------|--------------------|
| `t.Error` | Накапливает все ошибки в одном прогоне | Может вызвать панику, если продолжение некорректно | Независимые проверки |
| `t.Fatal` | Немедленная остановка, чистый стектрейс | Пропускает оставшиеся проверки | Когда после сбоя нет смысла продолжать |
| Table-driven | DRY, легко добавлять кейсы, чёткая идентификация | Требует больше шаблонного кода | Функции с множеством входных комбинаций |
| `t.Parallel` | Ускоряет прогон тестов | Требует thread-safe кода, сложнее отлаживать | Независимые тесты без общего состояния |
| `t.Cleanup` | Явный teardown после всех подтестов | — | Замена `defer` в тестах с вложенными `t.Run` |

## Примеры

### Базовые методы логирования

```go
func TestFooer(t *testing.T) {
    t.Logf("Testing Fooer with input %d", 0)     // просто лог
    t.Errorf("got %q, want %q", "bar", "Foo")    // сбой, но продолжает
    t.Fatalf("critical: got %q, want %q", "bar", "Foo") // сбой + стоп
    t.Error("эта строка уже не выполнится")
}
```

### Table-driven tests с подтестами

```go
func TestFooer(t *testing.T) {
    tests := []struct {
        name     string
        input    int
        expected string
    }{
        {"divisible by 3", 3, "Foo"},
        {"not divisible", 1, "1"},
        {"zero", 0, "Foo"},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            result := Fooer(tt.input)
            if result != tt.expected {
                t.Errorf("got %q, want %q", result, tt.expected)
            }
        })
    }
}
```

### Параллельный подтест

```go
t.Run(tt.name, func(t *testing.T) {
    t.Parallel()
    // тело теста
})
```

### Cleanup и TempDir

```go
func TestWithDB(t *testing.T) {
    db := setupDB(t)
    t.Cleanup(func() {
        db.Close()
    })
    // тесты...
}

func TestWithFiles(t *testing.T) {
    dir := t.TempDir() // автоматически удалится после теста
    // тесты...
}
```

### Пропуск медленных тестов

```go
func TestIntegration(t *testing.T) {
    if testing.Short() {
        t.Skip("пропуск интеграционного теста в режиме -short")
    }
    // медленный тест...
}
```

## Ключевые флаги `go test`

| Флаг | Описание |
|------|----------|
| `-v` | Подробный вывод (имена тестов, статус, логи) |
| `-run <regexp>` | Запускает только тесты, имена которых соответствуют regexp |
| `-count=N` | Запускает тест N раз; `-count=1` отключает кеширование |
| `-timeout <d>` | Максимальное время выполнения (например, `15s`, `2m`) |
| `-short` | Устанавливает `testing.Short()` в `true` |
| `-parallel N` | Максимальное число параллельных тестов |

## Важно / подводные камни / best practices

- Тестовая функция без суффикса `_test.go` или без префикса `Test` будет молча проигнорирована `go test`
- `GOCACHE=off` вызывает ошибку в Go 1.12+; используй `-count=1`
- Без `-timeout` зависший тест (dead-lock, бесконечный цикл) будет ждать вечно
- `t.Cleanup` предпочтительнее `defer` в тестах с вложенными `t.Run`, так как срабатывает после всех подтестов
- `t.Helper()` в helper-функциях делает лог ошибки читаемым — показывает строку вызова, а не строку внутри helper
