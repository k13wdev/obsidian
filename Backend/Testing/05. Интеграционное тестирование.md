# Интеграционное тестирование

## Суть и контекст

Интеграционное тестирование в Go позволяет проверять работу репозиториев и SQL-запросов на реальной базе данных — это даёт уверенность в корректности обработки данных, синтаксиса запросов и возвращаемых ошибок. В экосистеме Go принято поднимать тестовую БД прямо из кода теста — без ручного управления внешними сервисами.

---

## Ключевые идеи

- Тестовая БД поднимается программно: через `embedded-postgres` или `testcontainers-go`
- `TestMain(m *testing.M)` — точка входа для setup/teardown на уровне всего тестового пакета
- `t.Cleanup()` гарантирует teardown даже при падении теста
- Изоляция тестов достигается через транзакции (`tx.Rollback()`) или отдельные PostgreSQL-схемы
- Флаг `-short` + `testing.Short()` позволяет пропускать медленные интеграционные тесты в CI/CD
- Паттерн Repository позволяет переиспользовать тесты при смене СУБД
- Важно закрывать соединения с БД (`defer db.Close()`) для предотвращения утечки пула подключений

---

## Определения и термины

**embedded-postgres** — пакет с открытым исходным кодом, который позволяет программно запускать экземпляры PostgreSQL внутри Go-приложений. Позволяет задать имя пользователя, пароль, версию БД, порт и путь выполнения. База запускается методом `start` и останавливается функцией `cleanup` после завершения тестов.

**testcontainers-go** — пакет для развёртывания Docker-контейнеров (например, с образом PostgreSQL) непосредственно из кода интеграционного теста.

**TestMain(m *testing.M)** — глобальная точка входа для всех тестов в пакете. Позволяет выполнить тяжёлый setup один раз (поднять контейнер, накатить миграции) и один раз сделать teardown после `m.Run()`.

## Как устроено

### Паттерн TestMain

```go
func TestMain(m *testing.M) {
    // 1. Поднять БД/контейнер
    db, cleanup := setupDatabase()
    defer cleanup()

    // 2. Накатить миграции
    applyMigrations(db)

    // 3. Запустить все тесты пакета
    os.Exit(m.Run())
}
```

Поднимать контейнер для каждого теста дорого — `TestMain` позволяет сделать это один раз на весь пакет.

### embedded-postgres

```go
func setupDB(t *testing.T) *sql.DB {
    pg := embeddedpostgres.NewDatabase(
        embeddedpostgres.DefaultConfig().
            Username("user").
            Password("pass").
            Database("testdb").
            Port(5432),
    )
    err := pg.Start()
    if err != nil {
        t.Fatal(err)
    }
    t.Cleanup(func() { pg.Stop() })

    db, err := sql.Open("postgres", "host=localhost port=5432 user=user password=pass dbname=testdb sslmode=disable")
    if err != nil {
        t.Fatal(err)
    }
    return db
}
```

### Ручное создание схемы

```go
func create_table() {
    connStr := "user=postgres dbname=s2 sslmode=disable"
    db, err := sql.Open("postgres", connStr)
    // ... проверка ошибки
    const query = `
        CREATE TABLE IF NOT EXISTS users (
            id SERIAL PRIMARY KEY,
            first_name TEXT,
            last_name TEXT
    )`
    _, err = db.Exec(query)
    db.Close()
}
```

### Полный цикл интеграционного теста

```go
func Test_count(t *testing.T) {
    var count int
    create_table() // подготовка схемы
    insert_record("INSERT INTO users (first_name, last_name) VALUES ('Epifanios', 'Doe')") // подготовка состояния

    connStr := "user=postgres dbname=s2 sslmode=disable"
    db, err := sql.Open("postgres", connStr)
    if err != nil {
        fmt.Println(err)
        return
    }

    // Выполнение тестируемого запроса
    row := db.QueryRow("SELECT COUNT(*) FROM users")
    err = row.Scan(&count)
    db.Close()

    // Сверка результата
    if count != 1 {
        t.Errorf("Select query returned %d", count)
    }

    // Очистка состояния
    drop_table()
}
```

## Детали и нюансы

### docker-compose подход

Альтернатива `testcontainers-go`. Контейнер поднимается вне Go-кода:

```bash
docker-compose -f docker-compose.test.yml up -d db_test
go test ./...
docker-compose -f docker-compose.test.yml down
```

**Плюсы:** Более быстрый старт — контейнер инициализируется один раз до старта `go test`.

**Минусы:** Тесты перестают быть автономными и кроссплатформенными «из коробки»; требуется Docker CLI на хостовой машине и соблюдение правильной последовательности команд в CI/CD.

### Миграции в тестах

Ручной подход (как в источниках): `CREATE TABLE IF NOT EXISTS` перед тестом, `DROP TABLE IF EXISTS` после.

В реальных проектах: запуск системы миграций (`golang-migrate/migrate` или `pressly/goose`) сразу после старта контейнера в `TestMain`. Структура тестовой БД гарантированно соответствует production.

## Сравнения и trade-offs

| Подход к изоляции | Механизм | Плюсы | Минусы |
|-------------------|----------|-------|--------|
| Транзакционная изоляция | `tx.Begin()` → тест → `tx.Rollback()` | Быстро, нет побочных данных | Только для тестов без DDL |
| Изоляция на уровне схем | `CREATE SCHEMA test_<uuid>` → тест → `DROP SCHEMA CASCADE` | Полная изоляция | Медленнее |
| Отдельная БД на тест | Поднять и уничтожить БД | Полная чистота | Очень медленно |
| Ручная очистка (`DROP TABLE`) | `create_table()` / `drop_table()` | Просто | Легко забыть, медленно |

| Подход к БД в тестах | Плюсы | Минусы |
|---------------------|-------|--------|
| `embedded-postgres` | Нет зависимости от Docker; чистая in-memory БД | Только PostgreSQL |
| `testcontainers-go` | Поддерживает любой образ Docker; production-близко | Требует Docker; slower startup |
| `docker-compose` | Быстрый старт (один раз) | Тесты не автономны; сложнее CI/CD |

## Важно / подводные камни / best practices

**Скорость:** Интеграционные тесты медленнее. Использовать `-short` + `testing.Short()` или build tags для их пропуска при быстрых прогонах.

**Утечки ресурсов:** Всегда закрывать соединения с БД (`defer db.Close()`) — иначе истощится пул подключений PostgreSQL.

**t.Cleanup vs defer:** `t.Cleanup` предпочтительнее — гарантирует вызов cleanup даже при панике, в том числе после всех подтестов.

**Паттерн Repository:** Если тестируемый код принимает интерфейс репозитория — те же тесты легко применить к другой СУБД, передав новую реализацию.

**TestMain + os.Exit:** `os.Exit(m.Run())` обязателен — без него defer-вызовы перед `m.Run()` не сработают при ненулевом коде завершения.

?
1. В начале теста открыть транзакцию: `tx, _ := db.Begin()`
2. Передать `tx` в тестируемый репозиторий вместо `db`
3. В `t.Cleanup()` вызвать `tx.Rollback()` — все изменения откатятся

1. Создать случайную схему: `CREATE SCHEMA test_<uuid>`

3. Запустить тест
4. Удалить схему: `DROP SCHEMA test_<uuid> CASCADE`
