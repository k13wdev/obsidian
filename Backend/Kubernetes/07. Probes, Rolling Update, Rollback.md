# 07. Probes, Rolling Update, Rollback

## Суть и контекст

Health Probes — механизм Kubernetes для мониторинга состояния контейнеров: Liveness перезапускает зависшие процессы, Readiness управляет маршрутизацией трафика, Startup защищает медленно стартующие приложения. Rolling Update обеспечивает zero-downtime деплой через управление двумя ReplicaSet одновременно.

---

## Ключевые идеи

- Все пробы выполняются Kubelet на Worker Node — Control Plane в них не участвует
- Liveness провал → контейнер перезапускается; Readiness провал → IP удаляется из Endpoints (трафик не идёт), контейнер не перезапускается
- Startup Probe блокирует Liveness и Readiness до успешного завершения — для медленно стартующих приложений
- Rolling Update: шаг обновления блокируется пока новый под не станет Ready — Readiness Probe — предохранитель обновления
- minReadySeconds: новый под должен быть Ready N секунд без сбоев прежде чем контроллер удалит старый
- maxSurge и maxUnavailable задают математический темп Rolling Update (абсолютные значения или проценты)
- Старый ReplicaSet сохраняется (масштабируется до 0) как история ревизий для rollback
- `kubectl rollout undo` на paused Deployment — не работает; сначала нужен `resume`
- Recreate вызывает полный downtime; только для legacy-приложений без поддержки двух параллельных версий

---

## Определения и термины

**Liveness Probe (Проба работоспособности)** — механизм Kubernetes, который непрерывно выполняется на протяжении всего жизненного цикла контейнера для выявления зависших или неисправных процессов. При провале пробы контейнер автоматически принудительно завершается и перезапускается компонентом Kubelet.

**Readiness Probe (Проба готовности)** — механизм, периодически проверяющий, готов ли контейнер обрабатывать клиентские запросы. При провале пробы контейнер **не перезапускается**, а его IP-адрес немедленно удаляется из объекта Endpoints соответствующего сервиса.

**Startup Probe (Проба запуска)** — механизм, предоставляющий дополнительное время для инициализации медленно запускающихся приложений. Пока эта проба не завершится успешно, Liveness и Readiness пробы заблокированы (не выполняются). При провале Startup пробы контейнер перезапускается.

**RollingUpdate (Плавающее обновление)** — стратегия развёртывания (Deployment) по умолчанию. Постепенно удаляет старые поды и создаёт новые, гарантируя отсутствие простоев (zero downtime) и сохранение пропускной способности приложения во время обновления.

**Recreate (Пересоздание)** — стратегия развёртывания, при которой все существующие старые поды удаляются одновременно перед началом создания новых подов. Приводит к периоду полной недоступности приложения.

**maxSurge (Максимальное превышение)** — свойство стратегии RollingUpdate. Определяет максимальное количество подов (в процентах или абсолютных значениях), которое может быть создано сверх желаемого (desired) количества реплик во время обновления.

**maxUnavailable (Максимальная недоступность)** — свойство стратегии RollingUpdate. Определяет максимальное количество подов, которое может быть недоступно (относительно желаемого количества реплик) в процессе обновления.

## Как устроено

### Типы механизмов проверок

| Тип | Механизм | Критерий успеха |
|-----|---------|----------------|
| HTTP GET | Kubelet отправляет HTTP-запрос на порт и path | HTTP-код от 200 до 399 |
| TCP Socket | Kubelet пытается установить TCP-соединение с портом | Соединение успешно установлено |
| Exec | Kubelet выполняет команду/бинарник внутри контейнера | Exit Code 0 |

### Параметры проб

| Параметр | Назначение |
|---------|-----------|
| `initialDelaySeconds` | Ожидание после старта контейнера до первой проверки |
| `periodSeconds` | Интервал между проверками |
| `failureThreshold` | Число последовательных неудач до действия |
| `successThreshold` | Число последовательных успехов для восстановления статуса |
| `timeoutSeconds` | Таймаут одной проверки |

### Liveness vs Readiness vs Startup

| Probe | Действие при провале | Действие при успехе | Когда использовать |
|-------|--------------------|--------------------|-------------------|
| Liveness | Контейнер убивается и перезапускается | Ничего | Обнаружение deadlock / зависания |
| Readiness | IP удаляется из Endpoints | IP добавляется в Endpoints | Управление трафиком (загрузка данных, warmup) |
| Startup | Контейнер убивается (перезапускается) | Liveness + Readiness разблокируются | Медленный старт JVM, legacy-приложения |

> Liveness Probe — хирург (убивает процесс). Readiness Probe — терапевт (изолирует от трафика, ждёт выздоровления). Startup Probe — охранник (не пускает Liveness пока приложение не запустилось).

### Startup Probe: алгоритм

1. Выполняется первой (Liveness и Readiness заблокированы)
2. Если успешно → навсегда отключается для этого контейнера → активируются Liveness и Readiness
3. Если `failureThreshold` превышен → контейнер убивается и перезапускается

### Rolling Update: алгоритм

1. Изменение шаблона пода в Deployment → Deployment-контроллер создаёт новый пустой ReplicaSet
2. Постепенно увеличивает реплики в новом RS (scale up), параллельно уменьшает в старом (scale down)
3. Темп контролируется `maxSurge` и `maxUnavailable`
4. Шаг не продолжается, пока новый под не станет `Ready` (Readiness Probe успешна)
5. Если задан `minReadySeconds` — ждёт N секунд после Ready без сбоев
6. По завершении: старый RS масштабируется до 0, **но не удаляется** (история ревизий)

**Пример (replicas=3, maxSurge=1, maxUnavailable=0):**
```
→ Создаётся 1 новый под (итого 4)
→ Ждём Ready нового пода
→ Удаляется 1 старый под (итого 3)
→ ... повторяем ...
В любой момент: строго ≥ 3 пода готовы
```

### Rollback

При `kubectl rollout undo`: Deployment-контроллер находит предыдущий ReplicaSet в истории и запускает обратный Rolling Update, восстанавливая шаблон пода из него.

---

## Детали и нюансы

- **Liveness без initialDelaySeconds для медленного приложения**: Kubelet сразу опрашивает не готовое приложение → failureThreshold превышен → контейнер убивается → бесконечный CrashLoopBackOff. Решение: `initialDelaySeconds` или `startupProbe`
- **Readiness Probe + Rolling Update = предохранитель**: новый под с ошибкой (HTTP 500) не пройдёт Readiness → не добавится в Endpoints → Rolling Update зависнет → старые поды не удалятся → production защищён
- **kubectl rollout undo на paused Deployment**: не работает. Нужен сначала `kubectl rollout resume`
- **Удаление пода + Readiness Probe**: при получении `deletionTimestamp` контроллер Endpoints **автоматически** удаляет под из Endpoints — переводить Readiness в failed вручную через SIGTERM излишне
- **Exit Codes**: Liveness провал → SIGKILL → Exit 137; SIGTERM (graceful stop) → Exit 143
- **minReadySeconds**: без него под считается Available в момент первого успеха Readiness Probe; если приложение упадёт на 2-й секунде — старый под уже удалён

---

## Сравнения и trade-offs

| Стратегия | Downtime | Параллельные версии | Когда использовать |
|-----------|---------|--------------------|--------------------|
| RollingUpdate | Нет (zero downtime) | Временно: v1 + v2 | Большинство приложений |
| Recreate | Полный | Нет | Legacy (несовместимые версии, RWO блокировки) |

| Параметр | maxSurge=1, maxUnavailable=0 | maxSurge=0, maxUnavailable=1 |
|---------|------------------------------|------------------------------|
| Мин. доступных подов | Всегда = desired | Может быть desired-1 |
| Доп. ресурсы | Нужен 1 доп. под | Не нужны |
| Скорость | Шаг за шагом | Шаг за шагом |

## Примеры

```yaml
# Liveness (HTTP) + Readiness (HTTP) проба
spec:
  containers:
  - name: nginx
    image: nginx
    livenessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 10
      periodSeconds: 5
      failureThreshold: 3
    readinessProbe:
      httpGet:
        path: /ready
        port: 80
```

```yaml
# Startup Probe (Exec) для медленно стартующего приложения
    startupProbe:
      exec:
        command: ["/bin/sh", "-c", "cat /app/initialized"]
      initialDelaySeconds: 30
      periodSeconds: 10
      failureThreshold: 60
# Приложение получает до 10 мин (30 + 10*60 сек) на инициализацию
```

```yaml
# RollingUpdate с minReadySeconds для защиты от мгновенного сбоя
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  minReadySeconds: 10
```

```bash
# Управление rolling update из CLI
kubectl rollout status deployment kubia
kubectl rollout pause deployment kubia
kubectl rollout resume deployment kubia
kubectl rollout undo deployment kubia --to-revision=1
```

## Важно / подводные камни / best practices

- **Всегда определять Readiness Probe**: без неё Kubelet считает контейнер готовым немедленно после старта PID 1 → трафик идёт в ещё не инициализированный контейнер → Connection refused у клиентов
- **Никогда не использовать Liveness без initialDelaySeconds** для медленных приложений — бесконечный CrashLoopBackOff
- **Startup Probe для legacy-приложений** вместо большого initialDelaySeconds — даёт гибкость настройки (failureThreshold × periodSeconds)
- **minReadySeconds (10–30 сек)** при Rolling Update: защита от сбоя сразу после успешного Readiness
- **Не добавлять retry-loops в Exec пробы**: Kubernetes сам обрабатывает failureThreshold — дублирование увеличивает CPU overhead
- **Recreate только для incompatible-версий**: миграции схем БД, монопольные блокировки томов
