# 05. ConfigMap, Secret, Конфигурация

## Суть и контекст

ConfigMap и Secret — основные механизмы передачи конфигурации и чувствительных данных в контейнеры Kubernetes без встраивания в образ. ConfigMap — для неконфиденциальных данных, Secret — для паролей, токенов, ключей. Оба объекта хранятся в etcd и могут монтироваться как volumes или передаваться через переменные окружения.

---

## Ключевые идеи

- ConfigMap и Secret структурно идентичны (карты ключ-значение), но Secret использует Base64-кодирование — не шифрование
- Secret хранятся в etcd; до K8s 1.7 — открытый JSON, с 1.7 — опциональное шифрование at-rest, но ключи шифрования хранятся на API-сервере
- Secret при монтировании как volume → tmpfs (in-memory): данные не записываются на диск узла
- ENV-переменные фиксируются при старте пода — изменения ConfigMap/Secret не применяются без рестарта
- Volume-монтирование ConfigMap (без subPath) обновляется на лету через атомарный механизм символических ссылок
- subPath — исключение: файлы через subPath никогда не обновляются автоматически
- Любой пользователь с правом get secret в namespace может декодировать Secret из Base64; любой root-процесс на узле — читать Secret через kubelet
- Нативный Secret K8s не поддерживает ротацию ключей и гранулярный аудит — для production рекомендованы сторонние инструменты (Vault, AWS Secrets Manager)

---

## Определения и термины

**ConfigMap** — объект (карта «ключ-значение»), применяемый для хранения неконфиденциальных данных конфигурации приложений и их передачи контейнерам в виде файлов или переменных окружения.

**Secret (Секрет)** — объект Kubernetes, используемый для хранения чувствительных данных (паролей, ключей, токенов аутентификации). Подобен ConfigMap, но предназначен для конфиденциальной информации.

**Base64-кодирование (в контексте Secret)** — формат кодирования данных. В Kubernetes по умолчанию данные в объектах Secret хранятся в кодировке Base64, что практически равнозначно открытому тексту (plaintext) и не является алгоритмом криптографического шифрования.

**EncryptionConfig** — конфигурация API-сервера K8s (с версии 1.7) для шифрования объектов Secret в хранилище etcd with-rest с использованием алгоритмов `aesgcm` или `aescbc`; ключи шифрования хранятся в открытом виде на API-сервере.

**Third-Party Secret Management Tools** — сторонние инструменты управления секретами для production: HashiCorp Vault, CyberArk Conjur, AWS Secrets Manager, Google Cloud Platform KMS, Azure Key Vault. Предоставляют автоматическую ротацию ключей, гранулярный zero-trust контроль и аудит версионирования.

## Как устроено

### Хранение и безопасность

- Все объекты ConfigMap и Secret хранятся в etcd
- До K8s 1.7: Secret в etcd — открытый (незашифрованный) JSON-манифест
- С K8s 1.7: опциональное шифрование at-rest через `EncryptionConfig` (алгоритмы `aesgcm`/`aescbc`), но ключи шифрования хранятся в открытом виде на API-сервере
- RBAC: доступ к Secret определяется политиками на уровне Namespace
- Любой пользователь с правом создавать поды в namespace может подключить любой Secret и прочитать содержимое
- Любой процесс с правами root на Worker Node может читать любые Secret (kubelet получает их без принципа «need to know»)

### Монтирование как Volume vs ENV

**Volume (ConfigMap):**
- Kubernetes записывает содержимое ConfigMap в файловую систему хоста
- Монтирует директорию с хоста в контейнер

**Volume (Secret):**
- Kubernetes использует файловую систему in-memory (tmpfs)
- Данные хранятся исключительно в оперативной памяти — никогда не записываются на физический диск
- При удалении контейнера/пода → tmpfs уничтожается

**ENV:**
- Конфигурация считывается из etcd при старте пода
- Kubernetes инжектирует пары «ключ-значение» в процесс при запуске контейнера
- **Заморожены на момент старта**: изменение ConfigMap/Secret не изменит ENV в работающем контейнере

### Атомарное обновление ConfigMap через symlinks

При обновлении ConfigMap через API (`kubectl edit configmap`) без рестарта пода:

```
/etc/nginx/conf.d/
├── ..4984_09_04_12_15_06.865837643/   ← новая директория с реальными файлами
│     └── my-nginx-config.conf
├── ..data → ..4984_09_04_12_15_06.865837643  ← симлинк (атомарно переключается)
└── my-nginx-config.conf → ..data/my-nginx-config.conf  ← симлинк (файл приложения)
```

1. Kubernetes создаёт новую директорию с именем-таймстемпом и записывает в неё обновлённые файлы
2. Атомарно перезаписывает симлинк `..data` на новую директорию
3. Все файлы обновляются для приложения мгновенно и одновременно

**Ограничения автообновления:**
- Процесс должен сам обнаруживать изменения файлов и перезагружать конфигурацию (например, `nginx -s reload`)
- Kubernetes не посылает процессу сигналы (SIGHUP) об обновлении
- Обновление файлов на узлах происходит **асинхронно** и может занимать **до 1 минуты**
- Файлы, смонтированные через **`subPath`**, **никогда не обновляются** автоматически

---

## Детали и нюансы

- **Base64 ≠ шифрование**: любой пользователь с `get secret` может декодировать: `echo "LS0t..." | base64 --decode`
- **Максимальный размер Secret — 1 МБ**
- **subPath + обновление**: при монтировании одного файла через `subPath` симлинки не применяются → автообновление не работает
- **Ключ с дефисом в ConfigMap + envFrom**: имена переменных окружения в Linux не могут содержать дефисы → ключ `FOO-BAR` игнорируется; API фиксирует ошибку (event), загружаются только валидные ключи
- **Несуществующий ConfigMap/Secret**: планировщик назначит под на узел, но контейнер перейдёт в `CreateContainerConfigError`. Исключение: `configMapKeyRef.optional: true` — контейнер запустится без этих переменных
- **`stringData` — write-only**: при отправке манифеста API-сервер автоматически кодирует значения в Base64 и перемещает в поле `data`

---

## Сравнения и trade-offs

| Параметр | ConfigMap | Secret |
|----------|-----------|--------|
| Назначение | Неконфиденциальные данные | Пароли, токены, ключи |
| Хранение в etcd | Открытый текст | Base64 (опционально aesgcm/aescbc) |
| Монтирование как volume | Файловая система хоста (диск) | tmpfs (in-memory, не на диск) |
| Права доступа файлов | Стандартные | Настраивается через `defaultMode` |

| Способ передачи | Обновление без рестарта | Утечка через логи/дампы | Наследование дочерними процессами |
|-----------------|------------------------|------------------------|----------------------------------|
| Volume mount | Да (через symlinks, ~1 мин) | Нет | Нет |
| ENV переменная | Нет (заморожено при старте) | Да (часто попадает в error reports) | Да |

## Примеры

```bash
# Создание ConfigMap из литерала и директории файлов
kubectl create configmap fortune-config --from-literal=sleep-interval=25 --from-file=configmap-files
```

```yaml
# Secret: stringData (write-only) автоматически кодируется в Base64
kind: Secret
apiVersion: v1
stringData:
  foo: plain text
data:
  https.cert: LS0tLS1CRUdJ...
```

```yaml
# envFrom: все ключи ConfigMap с префиксом
spec:
  containers:
  - image: some-image
    envFrom:
    - prefix: CONFIG_
      configMapRef:
        name: my-config-map
```

```yaml
# Secret как volume (tmpfs) с правами 0640
  volumes:
  - name: certs
    secret:
      secretName: fortune-https
      defaultMode: 0640
```

```yaml
# EncryptionConfig: шифрование Secret at-rest в etcd
kind: EncryptionConfig
apiVersion: v1
resources:
  - resources:
    - secrets
    providers:
    - aesgcm:
        keys:
        - name: key1
          secret: c2VjcmV0IGlzIHNlY3VyZQ==
```

## Важно / подводные камни / best practices

- **Никогда не хранить секреты в ENV**: значения попадают в error reports, логи приложения, наследуются дочерними процессами; невозможно безопасно стереть из памяти после использования
- **Никогда не запекать секреты в Dockerfile (ENV API_KEY=...)**: жёстко записываются в метаданные слоёв образа; читаются через `docker inspect`
- **Монтировать Secret только через volume (tmpfs)** — данные не записываются на диск
- **Для production — сторонние инструменты**: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault для ротации ключей и гранулярного аудита (нативный K8s Secret этого не поддерживает)
- **`optional: true`** для необязательных ConfigMap/Secret — контейнер запустится без них вместо `CreateContainerConfigError`
- **Не использовать subPath** если нужно автообновление конфигурации
