# 01. Архитектура Kubernetes, Control Plane, etcd

## Суть и контекст

Kubernetes — система автоматизации развёртывания, масштабирования и управления контейнеризированными приложениями. Абстрагирует инфраструктуру и предоставляет вычислительные мощности дата-центра как единый ресурс. Название происходит от греческого κυβερνήτης — «кормчий».

---

## Ключевые идеи

- Архитектура: Control Plane (управляет) + Worker Nodes (выполняют)
- Все компоненты взаимодействуют только через kube-apiserver — никакого прямого доступа к etcd
- etcd — единственное хранилище состояния кластера; использует алгоритм консенсуса Raft
- kube-scheduler и kube-controller-manager работают по принципу leader election (один активный + standby)
- Только kube-apiserver и etcd могут работать параллельно в HA-кластере
- Парадигма: серверы не «питомцы» (pets), а «скот» (cattle) — неисправные узлы заменяются

---

## Определения и термины

**Kubernetes (K8s)** — система с открытым исходным кодом, автоматизирующая развёртывание, масштабирование и управление контейнеризированными приложениями. Абстрагирует аппаратную инфраструктуру и предоставляет вычислительные мощности дата-центра как единый ресурс.

**Control Plane (панель управления)** — набор компонентов (API-сервер, etcd, планировщик, диспетчер контроллеров), управляющих кластером, хранящих его состояние и контролирующих планирование подов. Не запускает пользовательские приложения.

**Worker Node (рабочий узел)** — физическая или виртуальная машина, на которой выполняется фактическая работа и запускаются контейнеризированные приложения (поды).

**kube-apiserver** — главный шлюз (frontend) для Control Plane; предоставляет RESTful API; единственный компонент, напрямую взаимодействующий с etcd.

**etcd** — распределённое, быстрое и консистентное хранилище «ключ-значение», содержащее всё состояние кластера. Реализует алгоритм консенсуса Raft.

**kube-scheduler** — назначает созданные поды на доступные рабочие узлы на основе требований к ресурсам, аппаратных ограничений и политик affinity/anti-affinity.

**kube-controller-manager** — процесс, непрерывно наблюдающий за текущим состоянием и приводящий его к desired state. Включает: Node controller, Replication controller, Endpoints controller, Service Account controllers.

**cloud-controller-manager** — интерфейс (cloud-provider) для интеграции K8s с API облачных провайдеров (управление узлами, маршрутизацией, томами).

**kubelet** — первичный агент узла; взаимодействует с API-сервером и управляет контейнерами через Container Runtime. Принимает спецификации PodSpecs и гарантирует здоровье контейнеров.

**kube-proxy** — сетевой прокси на каждом узле; реализует концепцию Service; управляет правилами iptables для маршрутизации трафика к подам.

**Container Runtime** — исполняемая среда (Docker, rkt, CRI-O, containerd), запускающая и останавливающая контейнеры.

**CRI (Container Runtime Interface)** — gRPC API (ImageService + RuntimeService), позволяющий среде выполнения контейнеров интегрироваться с kubelet без перекомпиляции ядра K8s.

**Optimistic concurrency control** — метод, при котором данные включают номер версии (resourceVersion); если версия изменилась с момента чтения — обновление отклоняется.

## Как устроено

### Архитектура кластера

```
kubectl → kube-apiserver ← все компоненты
                ↓ (единственный клиент etcd)
              etcd (кластер, Raft)

Control Plane:
  kube-apiserver
  etcd
  kube-scheduler        (leader election)
  kube-controller-manager (leader election)
  cloud-controller-manager

Worker Node:
  kubelet
  kube-proxy
  Container Runtime (containerd/CRI-O)
```

Компоненты Control Plane **не общаются напрямую** — только через kube-apiserver.

### Алгоритм создания пода

1. Пользователь через `kubectl` отправляет манифест в API-сервер (HTTP POST)
2. API-сервер валидирует спецификацию, сохраняет в etcd, возвращает ответ
3. kube-scheduler опрашивает API-сервер, обнаруживает под без привязки к узлу; вычисляет лучший узел → запрос на обновление объекта пода
4. kubelet на целевом узле получает уведомление
5. kubelet через CRI вызывает Container Runtime для загрузки образа и запуска контейнера
6. kubelet непрерывно сообщает статус API-серверу

### kube-proxy (iptables режим)

kube-proxy конфигурирует правила iptables в ядре узла: трафик на виртуальный IP Service → напрямую в случайный Backend Pod (без userspace proxying).

### etcd: консистентность через Raft

- Кластер etcd: несколько членов, каждый запускает бинарный файл etcd
- Алгоритм Raft: строгая консистентность через голосование; требует кворума (простое большинство) для каждой записи
- Оптимистичная блокировка: API-сервер — единственный клиент etcd; при конкурентных изменениях проверяется `resourceVersion`; несовпадение → отклонение, предотвращение race conditions

### Leader election для scheduler и controller-manager

Механизм использует ресурс Endpoints в API-сервере с метаданными `holderIdentity` (ID текущего лидера) и `leaseDurationSeconds`. Одновременно активен только один экземпляр; остальные — standby.

### etcd backups через etcd-operator

1. Администратор развёртывает etcd-operator (CoreOS) в K8s через Helm
2. Оператор автоматизирует: создание, удаление, изменение размера, отказоустойчивость, плавающие обновления
3. Резервное копирование и автоматическое восстановление при катастрофических сбоях

---

## Детали и нюансы

- Чётное число узлов etcd (напр. 2) не улучшает отказоустойчивость, но снижает производительность: при отказе одного узла кворум недостижим → все записи блокируются
- Несколько активных scheduler/controller-manager в HA = хаос; нужен leader election
- Прямой доступ к etcd (как в Google Omega) → race conditions; в K8s этого не допускает архитектура с kube-apiserver как «бутылочным горлышком»

---

## Сравнения и trade-offs

| Размер кластера etcd | Отказоустойчивость | Когда использовать |
|---------------------|-------------------|--------------------|
| 1 узел | 0 отказов | Dev/тест |
| 3 узла | 1 отказ | Стандартный production |
| 5 узлов | 2 отказа | Критичные production кластеры |
| 7 узлов | 3 отказа | Максимальная отказоустойчивость |
| >7 узлов | — | Снижение производительности (консенсус Raft) |

| Компонент | Параллельные инстансы в HA | Механизм |
|-----------|--------------------------|----------|
| kube-apiserver | Да | Load Balancer |
| etcd | Да (Raft кворум) | Raft consensus |
| kube-scheduler | Нет (1 активный) | Leader election |
| kube-controller-manager | Нет (1 активный) | Leader election |

## Примеры

```bash
# Статус системных компонентов Control Plane
kubectl get componentstatuses

# Здоровье кластера etcd через etcdctl
kubectl exec etcd-cluster-0000 etcdctl cluster-health

# Запись/чтение ключей в etcd (отладка)
kubectl exec etcd-cluster-0000 etcdctl set test "Yeah, it works"
kubectl exec etcd-cluster-0000 etcdctl get test
```

```yaml
# Запуск kube-scheduler с leader election (HA)
command:
- /bin/sh
- -c
- /usr/local/bin/kube-scheduler --master=127.0.0.1:8080 --v=2 --leader-elect=true 1>>/var/log/kube-scheduler.log 2>&1
# --leader-elect=true: только один активный инстанс
```

```proto
// CRI RuntimeService — интерфейс для интеграции Container Runtime с kubelet
service RuntimeService {
    rpc Version(VersionRequest) returns (VersionResponse) {}
    rpc RunPodSandbox(RunPodSandboxRequest) returns (RunPodSandboxResponse) {}
    rpc StopPodSandbox(StopPodSandboxRequest) returns (StopPodSandboxResponse) {}
    rpc RemovePodSandbox(RemovePodSandboxRequest) returns (RemovePodSandboxResponse) {}
    rpc CreateContainer(CreateContainerRequest) returns (CreateContainerResponse) {}
    rpc StartContainer(StartContainerRequest) returns (StartContainerResponse) {}
}
```

## Важно / подводные камни / best practices

- **Нечётное число узлов etcd** (3, 5, 7); >7 — снижение производительности
- **Никогда не обращаться к etcd напрямую** — только через kube-apiserver
- **Leader election обязателен** для scheduler и controller-manager в HA
- **etcd-operator** для автоматизации backup/restore
- **«Cattle, not pets»** — узлы не имеют уникальных имён/привязанности; неисправные заменяются
