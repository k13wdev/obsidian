# 04. Networking: Service, Ingress, DNS, CNI

## Суть и контекст

Kubernetes реализует «плоскую» (flat) сеть: каждый под получает маршрутизируемый IP-адрес, и все поды могут обращаться друг к другу по этим адресам без NAT. Service — абстракция для стабильной точки входа к группе подов. Ingress — Layer 7 маршрутизация входящего HTTP/HTTPS-трафика. CNI-плагины отвечают за физическую реализацию сетевой модели на узлах.

---

## Ключевые идеи

- Flat network: каждый под имеет уникальный IP, маршрутизируемый во всём кластере без NAT — IP-адреса источника и назначения не изменяются при передаче между узлами
- kube-proxy на каждом узле конфигурирует правила iptables: трафик на virtual IP сервиса → DNAT → случайный pod из Endpoints
- ClusterIP — виртуальный IP, не назначенный ни одному интерфейсу: ping не работает, только TCP/UDP через iptables
- Ingress Controller обходит kube-proxy: получает IP из Endpoints напрямую и отправляет запрос в под, минуя Service virtual IP
- CoreDNS автоматически настраивается в /etc/resolv.conf каждого контейнера kubelet-ом
- Headless Service (clusterIP: None) → DNS возвращает A-записи подов напрямую, а не virtual IP
- ExternalName реализован исключительно через DNS CNAME, без ClusterIP и без kube-proxy

---

## Определения и термины

**Flat network (Плоская сеть)** — единое совместно используемое пространство сетевых адресов, в котором каждый под получает маршрутизируемый IP-адрес, и все поды могут обращаться друг к другу по этим адресам без использования трансляции сетевых адресов (NAT).

**CNI (Container Network Interface)** — спецификация и набор библиотек для написания дополнений, которые конфигурируют сетевые интерфейсы в Linux-контейнерах, назначают им IP-адреса и управляют маршрутизацией.

**Calico** — комплексное сетевое решение для организации сети между контейнерами на третьем сетевом уровне (Layer 3 routing), использующее маршрутизацию на основе IP-адресов без инкапсуляции (в базовом варианте) и реализующее применение сетевых политик.

**Flannel** — виртуальная сеть, выделяющая каждому узлу подсеть и использующая инкапсуляцию пакетов (например, в UDP-датаграммы через VXLAN), чтобы отправлять их от хоста к хосту.

**Cilium** — сетевое дополнение CNI, использующее технологии BPF и XDP для контейнеров.

**Service (Сервис)** — ресурс, предоставляющий единую постоянную точку входа (IP-адрес и порт) к группе подов, предоставляющих одинаковую службу.

**ClusterIP** — тип сервиса по умолчанию, предоставляющий отслеживание выбранных подов, кластерный IP-адрес, маршрутизируемый только внутри кластера, и известное имя во внутреннем DNS кластера.

**NodePort** — расширение типа ClusterIP, при котором каждый узел кластера открывает на себе порт (node port) и перенаправляет трафик, полученный на этот порт, соответствующему сервису.

**LoadBalancer** — расширение типа NodePort, использующее базовую облачную инфраструктуру для получения IP-адреса, доступного извне кластера.

**ExternalName** — тип сервиса, создающий алиас (запись CNAME в DNS) известного имени сервиса внутри кластера на внешнее DNS-имя, реализуемый исключительно на уровне DNS без выделения cluster IP.

**Headless Service (Сервис без IP)** — сервис, в спецификации которого поле clusterIP установлено в None; ему не назначается виртуальный IP-адрес, а DNS-сервер кластера возвращает IP-адреса отдельных подов вместо одного IP-адреса сервиса.

**Ingress** — ресурс, работающий на прикладном уровне сетевого стека (HTTP/HTTPS, Layer 7), предоставляющий единую точку входа в кластер и маршрутизирующий внешний трафик к внутренним сервисам на основе запрашиваемого хоста и пути.

**Ingress Controller (Контроллер Ingress)** — процесс (часто обратный прокси-сервер, например Nginx), который отслеживает ресурсы Ingress, Service и Endpoints в кластере и динамически обновляет свою конфигурацию для маршрутизации трафика напрямую к подам.

## Как устроено

### Flat network: Pod-to-Pod без NAT

```
Node 1: Pod A (IP: 10.1.1.1)    Node 2: Pod B (IP: 10.1.2.1)
     ↓                                  ↓
  [src: 10.1.1.1 → dst: 10.1.2.1]  No NAT (IPs preserved)
     ↓                                  ↓
  [src: 10.1.1.1 → dst: 10.1.2.1] arrives at Pod B
```

- Инфраструктурный контейнер `pause` инициализирует и удерживает сетевое пространство имён пода
- Все контейнеры внутри пода делят этот сетевой стек (один IP, один набор портов)

### Pod-to-Pod: внутри узла

1. Пакет отправляется из интерфейса `eth0` внутри контейнера
2. Пакет проходит через veth-пару и выходит через ответный интерфейс в корневом пространстве имён хоста
3. Пакет попадает на сетевой мост (bridge, например `cni0`), который маршрутизирует его на интерфейс другого пода, подключённого к этому же мосту

> Сетевой мост работает как Ethernet-коммутатор (switch), маршрутизирующий трафик на основе MAC-адресов интерфейсов.

### Pod-to-Pod: между узлами

**Layer 3 (Calico, BGP):** пакет направляется на шлюз узла → маршрутизаторы (или таблицы маршрутизации Linux, настроенные по протоколу BGP) перенаправляют пакет на целевой узел → доставляется целевому поду. IP-адреса источника и назначения не изменяются.

**Инкапсуляция (Flannel, WeaveNet):** пакет оборачивается в UDP-датаграмму (VXLAN) с IP-адресами хостов → отправляется по физической сети на целевой узел → распаковывается → передаётся в целевой под.

### Service и kube-proxy (iptables режим)

1. API-сервер уведомляет агенты kube-proxy о создании нового Service
2. kube-proxy настраивает правила iptables в ядре узла
3. Пакет, отправленный на virtual IP/порт сервиса, перехватывается ядром до того, как покинет узел
4. Правила iptables выполняют DNAT: заменяют целевой IP и порт сервиса на IP и порт случайно выбранного пода из объекта Endpoints
5. Пакет маршрутизируется к поду сетью CNI

### Ingress: алгоритм прохождения запроса

```
Client → DNS (kubia.example.com → IP Ingress Controller)
       → HTTP GET с заголовком Host: kubia.example.com
       → Ingress Controller (анализирует Host + path)
       → Endpoints сервиса (получает IP подов)
       → Напрямую к Pod (минуя virtual IP Service)
```

**Важно:** Ingress Controller не отправляет запрос на IP-адрес сервиса. Он обращается к объекту Endpoints для получения списка IP-адресов подов и перенаправляет HTTP-запрос **напрямую на IP одного из подов**, минуя kube-proxy и virtual IP.

### DNS и Service Discovery (CoreDNS)

- Надстройка kube-dns (или CoreDNS) запущена как Deployment в пространстве имён `kube-system`
- Kubelet конфигурирует `/etc/resolv.conf` в каждом контейнере: указывает IP сервиса kube-dns как `nameserver` и добавляет домены поиска: `default.svc.cluster.local`, `svc.cluster.local`, `cluster.local`
- CoreDNS отслеживает изменения Services и Endpoints через API-сервер
- Для обычного сервиса (ClusterIP) → возвращает виртуальный IP
- Для Headless-сервиса (clusterIP: None) → возвращает список A-записей IP-адресов подов

## Детали и нюансы

- **ping на ClusterIP не работает**: IP-адрес сервиса является виртуальным, не назначенным ни одному сетевому интерфейсу. iptables перехватывает только TCP/UDP для конкретных портов — ICMP-пакеты не перехватываются
- **NodePort + SNAT**: когда пакет приходит на порт узла и целевой под находится на **другом** узле, kube-proxy выполняет SNAT, подставляя IP принимающего узла. Под не видит оригинальный IP клиента
- **externalTrafficPolicy: Local**: трафик направляется только на поды на том же узле (без SNAT → клиентский IP сохраняется). Ограничение: неравномерная нагрузка если количество подов на узлах различается; если на узле нет локальных подов — трафик отбрасывается
- **StatefulSet без Headless Service**: обычный сервис балансирует на случайный под — для кластеризованных БД требуется Headless Service (DNS SRV-записи для peer discovery)

## Сравнения и trade-offs

| Тип Service | Доступность | Механизм | Когда использовать |
|-------------|-------------|----------|-------------------|
| ClusterIP | Только внутри кластера | Virtual IP + iptables DNAT | Внутренние микросервисы |
| NodePort | Внешний + внутренний | ClusterIP + порт на каждом узле | Dev/тест, без облачного LB |
| LoadBalancer | Внешний IP от облака | NodePort + облачный LB | Production внешний трафик |
| ExternalName | Внутри кластера → внешний DNS | DNS CNAME, без kube-proxy | Внешние API как внутренний сервис |
| Headless (clusterIP: None) | DNS-based прямой доступ | A-записи IP подов | StatefulSet, peer discovery |

| CNI-плагин | Механизм | Особенности |
|-----------|----------|-------------|
| Calico | Layer 3, BGP, без инкапсуляции | Производительность, NetworkPolicy |
| Flannel | VXLAN инкапсуляция | Простота настройки |
| Cilium | eBPF + XDP | L7 policy, observability |

| Routing | L4 (Service/kube-proxy) | L7 (Ingress) |
|---------|------------------------|--------------|
| Протокол | TCP/UDP | HTTP/HTTPS |
| Маршрутизация | По IP:порт через DNAT | По Host + path header |
| IP-адрес | Один LB IP на сервис | Один IP на все сервисы |
| Стоимость | Дорого для каждого сервиса | Один Ingress Controller |

## Примеры

```yaml
# Service типа NodePort
apiVersion: v1
kind: Service
metadata:
  name: kubia-nodeport
spec:
  type: NodePort
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30123
  selector:
    app: kubia
```

```yaml
# Service типа LoadBalancer
apiVersion: v1
kind: Service
metadata:
  name: kubia-loadbalancer
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 8080
  selector:
    app: kubia
```

```yaml
# Service типа ExternalName
apiVersion: v1
kind: Service
metadata:
  name: external-service
spec:
  type: ExternalName
  externalName: someapi.somecompany.com
  ports:
  - port: 80
```

```yaml
# Ingress: несколько сервисов через один IP по хосту и пути
apiVersion: extensions/v1beta1
kind: Ingress
metadata:
  name: test
spec:
  rules:
  - host: foo.example.com
    http:
      paths:
      - path: /
        backend:
          serviceName: foo
          servicePort: 80
  - host: bar.example.com
    http:
      paths:
      - path: /
        backend:
          serviceName: bar
          servicePort: 80
```

```bash
# DNS-запрос внутри пода для Headless Service
kubectl exec dnsutils nslookup kubia-headless
# Возвращает несколько A-записей с IP-адресами подов, а не один IP балансировщика
```

## Важно / подводные камни / best practices

- **Не использовать LoadBalancer для каждого микросервиса**: каждый Service LoadBalancer — отдельный внешний IP и физический LB у облачного провайдера. Вместо этого — единый Ingress Controller для всех HTTP-сервисов
- **Headless Service для stateful-приложений**: кластеризованные БД (Cassandra, Kafka) требуют peer discovery через DNS SRV-записи — только Headless Service обеспечивает прямой доступ к конкретным подам
- **ping ClusterIP не работает** — не признак сломанного сервиса; проверять через curl/telnet
- **externalTrafficPolicy: Local для сохранения клиентского IP** при использовании NodePort/LoadBalancer, но учитывать риск неравномерной нагрузки
- **Короткое DNS-имя** работает только в текущем namespace; для межнеймспейсного обращения — FQDN: `<svc>.<ns>.svc.cluster.local`

**CNI (Container Network Interface)**:::Спецификация и библиотеки для плагинов, конфигурирующих сетевые интерфейсы в контейнерах: назначают IP, управляют маршрутизацией

**Calico**:::CNI Layer 3, маршрутизация BGP без инкапсуляции; поддерживает NetworkPolicy

**Flannel**:::CNI с инкапсуляцией пакетов через VXLAN; выделяет каждому узлу подсеть

**Cilium**:::CNI на базе eBPF и XDP; поддерживает L7 политики и расширенную observability

**Service (K8s)**:::Ресурс с постоянным virtual IP и портом для группы подов; маршрутизация через kube-proxy + iptables DNAT

**ClusterIP**:::Тип Service по умолчанию; virtual IP доступен только внутри кластера; регистрируется в DNS кластера

**NodePort**:::Расширение ClusterIP; открывает порт (30000–32767) на каждом узле и перенаправляет трафик в Service

**LoadBalancer**:::Расширение NodePort; запрашивает внешний IP у облачного провайдера

**ExternalName**:::Тип Service без ClusterIP; создаёт DNS CNAME на внешнее имя; без kube-proxy

**Headless Service**:::clusterIP: None; DNS возвращает A-записи IP-адресов подов напрямую; используется для peer discovery в StatefulSet

**Ingress**:::L7 ресурс для маршрутизации HTTP/HTTPS по Host + path; единая точка входа для множества сервисов через один IP

**Ingress Controller**:::Процесс (напр. Nginx), который читает Endpoints сервисов и направляет HTTP-запросы напрямую в поды, минуя kube-proxy

kube-proxy: алгоритм маршрутизации к Service
?
1. API-сервер уведомляет kube-proxy о новом Service
2. kube-proxy настраивает правила iptables на узле
3. Пакет на virtual IP перехватывается ядром
4. DNAT: целевой IP:порт заменяется на IP:порт случайного пода из Endpoints
5. CNI маршрутизирует пакет к поду

Ingress Controller: как попадает запрос в под
?
1. Клиент → DNS → IP Ingress Controller
2. HTTP GET с заголовком Host
3. Ingress Controller читает правила Ingress и Endpoints сервиса
4. Запрос отправляется напрямую на IP пода (минуя virtual IP Service и kube-proxy)

Pod-to-Pod (внутри узла) алгоритм
?
1. eth0 контейнера → veth-пара → корневой namespace хоста
2. Сетевой мост (cni0) коммутирует трафик к veth второго пода
3. Пакет доставляется в целевой контейнер

Как DNS resolv.conf настраивается в поде K8s?::Kubelet записывает IP сервиса kube-dns как nameserver и добавляет домены поиска: default.svc.cluster.local, svc.cluster.local, cluster.local

Почему ping на ClusterIP не работает?::ClusterIP — виртуальный IP, не назначенный ни одному интерфейсу; iptables перехватывает только TCP/UDP по конкретным портам; ICMP не перехватывается

Edge case: NodePort + SNAT?::При перенаправлении трафика на под другого узла kube-proxy выполняет SNAT (заменяет источник на IP узла) — под не видит реальный IP клиента. Решение: externalTrafficPolicy: Local

Edge case: externalTrafficPolicy: Local?::Трафик только на поды локального узла (без SNAT → реальный IP клиента виден). Риск: неравномерная нагрузка; если локальных подов нет — трафик отбрасывается

Edge case: StatefulSet без Headless Service?::Обычный Service балансирует на случайный под; для кластеризованных БД (peer discovery) нужен Headless Service с DNS SRV-записями для прямого доступа к конкретным подам

DNS FQDN формат сервиса в K8s?::`<service-name>.<namespace>.svc.cluster.local`; короткое имя работает только в текущем namespace

Calico vs Flannel: ключевое отличие?::Calico — Layer 3 BGP без инкапсуляции (выше производительность, поддержка NetworkPolicy); Flannel — VXLAN инкапсуляция (проще в настройке, выше overhead)

Когда использовать Ingress вместо LoadBalancer?::Ingress — один IP + Ingress Controller для десятков HTTP-сервисов по Host/path. LoadBalancer на каждый Service — отдельный IP и физический LB у провайдера (дорого)
