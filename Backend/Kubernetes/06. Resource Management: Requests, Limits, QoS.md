# 06. Resource Management: Requests, Limits, QoS

## Суть и контекст

Kubernetes управляет вычислительными ресурсами (CPU, Memory) через механизм requests/limits. Requests используются планировщиком для выбора узла; limits — ядром Linux (cgroups) для принудительного ограничения потребления во время выполнения. QoS-классы определяют приоритет подов при нехватке ресурсов на узле.

---

## Ключевые идеи

- Scheduler учитывает только requests, не limits и не фактическое потребление — overcommit по limits допустим
- CPU — сжимаемый (compressible) ресурс: превышение limit → троттлинг, процесс не убивается
- Memory — несжимаемый (incompressible) ресурс: превышение limit → OOM Killer немедленно убивает процесс (Exit 137)
- QoS-класс определяется автоматически по конфигурации requests/limits; влияет на oom_score_adj
- BestEffort → убивается первым; Guaranteed (requests == limits) → убивается последним (oom_score_adj = -997)
- LimitRange задаёт дефолты и min/max на уровне Namespace (Admission Control при создании)
- ResourceQuota ограничивает суммарные ресурсы и количество объектов в Namespace
- LimitRange + ResourceQuota: без LimitRange при включённой ResourceQuota нельзя создавать поды без явных ресурсов

---

## Определения и термины

**Requests (Запросы ресурсов)** — минимальное количество вычислительных ресурсов (CPU/Memory), необходимое контейнеру для работы. Используется планировщиком (Scheduler) для выбора подходящего узла.

**Limits (Лимиты ресурсов)** — жёсткое максимальное ограничение на потребление ресурсов контейнером. Применяется на уровне среды выполнения (runtime) с помощью механизмов ядра ОС (cgroups).

**Quality of Service (QoS) classes** — классы качества обслуживания (BestEffort, Burstable, Guaranteed), назначаемые поду на основе конфигурации его requests и limits, определяющие приоритет распределения процессорного времени и приоритет принудительного завершения процессов (OOMKilled/Eviction) при нехватке памяти на узле.

**Guaranteed (QoS)** — высший класс приоритета. Присваивается поду, в котором для всех контейнеров установлены и requests, и limits для CPU и памяти, и их значения равны.

**Burstable (QoS)** — средний класс приоритета. Присваивается поду, у которого хотя бы один контейнер имеет requests или limits, и спецификация не попадает под критерии Guaranteed.

**BestEffort (QoS)** — низший класс приоритета. Присваивается поду, у которого ни для одного контейнера не установлены ни requests, ни limits.

**OOMKilled** — статус завершения контейнера (Exit Code 137 = 128 + 9/SIGKILL), когда процесс пытается выделить больше памяти, чем указано в его limits, или когда система исчерпала память и OOM Killer ядра Linux принудительно завершает процесс.

**Eviction (Вытеснение)** — процесс удаления подов с узла компонентом Kubelet при возникновении острой нехватки системных ресурсов на узле (memory pressure, disk pressure) до того, как сработает системный OOM Killer.

**LimitRange** — ресурс Kubernetes, определяющий минимальные, максимальные и значения по умолчанию для requests и limits отдельных подов/контейнеров/PVC в рамках одного пространства имён (Namespace).

**ResourceQuota** — ресурс Kubernetes, ограничивающий суммарное (совокупное) потребление вычислительных ресурсов (CPU, Memory, Storage) и максимальное количество создаваемых объектов (Pods, Services, Secrets) внутри одного пространства имён.

## Как устроено

### Requests и планирование

1. Scheduler вычисляет сумму requests всех контейнеров в поде
2. Сравнивает с **Allocatable** ресурсами (не Capacity) на каждом узле
3. Отфильтровывает узлы, где `Unallocated < Pod Requests`
4. Применяет функции `LeastRequestedPriority` или `MostRequestedPriority` для выбора оптимального узла

> Scheduler никогда не смотрит на limits и фактическое потребление — только на requests.

### CPU vs Memory в cgroups

**CPU (сжимаемый ресурс):**
- `requests.cpu` → `cpu.shares` (пропорция: 1 ядро = 1024)
- `limits.cpu` → квоты `cpu.cfs_period_us` / `cpu.cfs_quota_us` планировщика CFS
- При превышении limit: процесс **не убивается** → CPU-троттлинг (замедление выполнения)

**Memory (несжимаемый ресурс):**
- `limits.memory` → `memory.limit_in_bytes` в cgroup
- При превышении limit: ядро Linux **немедленно убивает процесс** через OOM Killer → Exit Code 137

### QoS и oom_score_adj

| QoS класс | Условие | oom_score_adj | cgroup slice | Поведение при OOM |
|-----------|---------|--------------|--------------|------------------|
| BestEffort | Нет requests и limits | 1000 | kubepods-besteffort.slice | Убивается первым |
| Burstable | Есть хотя бы один request/limit, но не Guaranteed | 2–999 (динамически) | kubepods-burstable.slice | Убивается по превышению requests |
| Guaranteed | requests == limits для всех контейнеров и ресурсов | -997 | kubepods.slice (корневой) | Убивается последним |

Kubelet записывает `oom_score_adj` в `/proc/<pid>/oom_score_adj` каждого процесса.

**Порядок OOM Eviction:** BestEffort → Burstable (потребляющие > своих requests) → Burstable (до requests) → Guaranteed.

### Overcommit

K8s допускает `сумма limits > 100% Capacity узла` (overcommit) — ориентируясь при планировании только на requests. Если все поды одновременно потребляют до своих limits → нехватка памяти → Kubelet начинает Eviction или срабатывает OOM Killer.

### LimitRange и ResourceQuota (Admission Control)

1. API-сервер получает запрос на создание пода
2. Плагин **LimitRanger**: применяет `defaultRequest` и `default` (limits) если они отсутствуют в спецификации; проверяет `min`/`max` — при нарушении → HTTP Forbidden
3. Плагин **ResourceQuota**: суммирует ресурсы нового пода с уже потреблёнными в Namespace; если превышает `hard` → манифест отклоняется

> Admission Control применяется только к новым манифестам — существующие поды не пересчитываются.

---

## Детали и нюансы

- **JVM + Memory Limits**: JVM читает системные параметры хоста (32 GB), игнорируя cgroups limit контейнера → выделяет большой heap → мгновенный OOMKilled. Решение: явный флаг `-Xmx` или новые версии Java с поддержкой `UseCGroupMemoryLimitForHeap`
- **ResourceQuota без LimitRange**: при включённой ResourceQuota нельзя создать под без явных limits/requests → API вернёт `Forbidden: failed quota: must specify limits.cpu`. Решение: создать LimitRange с дефолтами
- **Обновление LimitRange**: не применяется к существующим подам — только к новым манифестам
- **OOMKilled Exit Code**: 137 = 128 + 9 (SIGKILL), связан с POSIX-сигналами ОС

---

## Сравнения и trade-offs

| | CPU | Memory |
|-|-----|--------|
| Тип ресурса | Compressible (сжимаемый) | Incompressible (несжимаемый) |
| Превышение limit | Троттлинг (замедление) | OOM Killer → Exit 137 |
| cgroup параметр | cpu.shares / cfs_quota | memory.limit_in_bytes |
| Единицы | m (millicpu), 1 = 1000m | Mi, Gi |

| QoS стратегия | Requests | Limits | Когда использовать |
|---------------|---------|--------|--------------------|
| Guaranteed | Заданы | = Requests | Критичные компоненты (БД, infra) |
| Burstable | Заданы (baseline) | Выше requests (worst-case) | Переменная нагрузка, экономия ресурсов |
| BestEffort | Не заданы | Не заданы | Batch-задачи, dev/тест |

## Примеры

```yaml
# Requests и Limits для контейнера (QoS: Burstable)
spec:
  containers:
  - name: main
    image: busybox
    resources:
      requests:
        cpu: 200m       # 200 милликоров (1/5 ядра)
        memory: 10Mi    # 10 мебибайт
      limits:
        cpu: 1          # 1 целое ядро
        memory: 20Mi    # 20 мебибайт
```

```yaml
# LimitRange: дефолты и min/max для Namespace
apiVersion: v1
kind: LimitRange
metadata:
  name: example
spec:
  limits:
  - type: Container
    defaultRequest:
      cpu: 100m
      memory: 10Mi
    default:
      cpu: 200m
      memory: 100Mi
    min:
      cpu: 50m
      memory: 5Mi
    max:
      cpu: 1
      memory: 1Gi
```

```yaml
# ResourceQuota: суммарные ресурсы и количество объектов
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
spec:
  hard:
    pods: "10"
    requests.cpu: "1"
    requests.memory: 20Mi
    limits.cpu: "2"
    limits.memory: 2Gi
```

```bash
# Узнать QoS класс пода
kubectl get po <pod_name> -o json | jq .status.qosClass

# Мониторинг фактического потребления
kubectl top pods --all-namespaces
```

## Важно / подводные камни / best practices

- **Критичные компоненты — только Guaranteed** (requests == limits): защита от noisy neighbors, минимальный OOM score
- **Production нагрузки — не BestEffort**: без ресурсов под получает cpu.shares = 2 (минимум) и убивается первым
- **Burstable для переменной нагрузки**: requests = baseline, limits = worst-case peak → эффективный overcommit
- **JVM приложения**: всегда задавать `-Xmx` явно или использовать JVM с поддержкой cgroups
- **LimitRange обязателен при включённой ResourceQuota** — иначе невозможно создать под без явных ресурсов
- **Обновление LimitRange не влияет на работающие поды** — только на новые манифесты

**Limits (K8s)**:::Жёсткий максимум потребления ресурсов; применяется cgroups на runtime; превышение CPU → троттлинг; превышение Memory → OOMKilled (Exit 137)

**Guaranteed (QoS)**:::requests == limits для всех контейнеров и ресурсов; oom_score_adj = -997; убивается последним при OOM

**Eviction**:::Удаление подов Kubelet при memory/disk pressure до срабатывания системного OOM Killer; порядок: BestEffort → Burstable → Guaranteed

**ResourceQuota**:::Ресурс K8s; ограничивает суммарные ресурсы (CPU/Memory) и количество объектов (Pods/Services/Secrets) в Namespace

?
1. BestEffort (нет requests/limits, oom_score_adj=1000)
2. Burstable (потребляют > своих requests)
3. Burstable (потребляют ≤ своих requests)
4. Guaranteed (oom_score_adj=-997) — последними
