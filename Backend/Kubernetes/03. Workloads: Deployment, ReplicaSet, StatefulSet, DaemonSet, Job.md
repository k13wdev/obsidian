# 03. Workloads: Deployment, ReplicaSet, StatefulSet, DaemonSet, Job

## Суть и контекст

Kubernetes предоставляет несколько типов контроллеров для управления подами: Deployment (stateless приложения с rolling updates), StatefulSet (stateful с уникальной идентичностью), DaemonSet (по одному поду на узел), Job/CronJob (batch-задачи).

---

## Ключевые идеи

- Deployment управляет ReplicaSet, который управляет подами; Rolling Update — через два ReplicaSet одновременно
- Старый ReplicaSet не удаляется после обновления — хранится как история ревизий для rollback
- StatefulSet: поды создаются/удаляются строго последовательно, имеют stable identity (pod-0, pod-1)
- StatefulSet: при пересоздании пода его PVC переподключается — данные сохраняются
- DaemonSet: ровно один под на каждом узле; автоматически добавляется на новые узлы
- Job: batch-задача с гарантией успешного завершения; restartPolicy ≠ Always
- CronJob: создаёт Job по расписанию Unix cron; задачи должны быть идемпотентными

---

## Определения и термины

**Deployment** — ресурс для декларативного развёртывания и обновления приложений. Управляет ReplicaSet, который управляет подами.

**ReplicaSet** — гарантирует заданное количество идентичных запущенных экземпляров подов. Строительный блок для Deployment.

**StatefulSet** — контроллер для stateful-приложений; упорядоченное развёртывание, стабильная сетевая идентичность, выделенное постоянное хранилище для каждого экземпляра.

**DaemonSet** — гарантирует запуск ровно одной копии пода на каждом узле (или подмножестве узлов).

**Job** — запускает поды для выполнения однократной конечной задачи (batch); гарантирует успешное завершение заданного числа раз.

**CronJob** — создаёт объекты Job по расписанию Unix cron.

**Headless Service** — Service с `clusterIP: None`; используется со StatefulSet для прямого обнаружения каждого пода через DNS без балансировки нагрузки.

**Stable Identity** — поды StatefulSet получают предсказуемые имена (`kubia-0`, `kubia-1`), стабильный hostname и неизменную привязку к PVC — сохраняется даже при удалении/перепланировании.

**Rolling Update** — стратегия обновления по умолчанию для Deployment; удаляет старые поды по одному, добавляя новые; сохраняет доступность приложения.

## Как устроено

### Deployment → ReplicaSet → Pod: иерархия

```
Deployment
  ├── ReplicaSet v2 (3 реплики — текущая)
  └── ReplicaSet v1 (0 реплик — история)
        └── Pod pod-v1-xxx (удалён)
```

**Rolling Update алгоритм:**
1. Deployment Controller создаёт новый ReplicaSet v2
2. Постепенно увеличивает реплики в v2 (`maxSurge`) и уменьшает в v1 (`maxUnavailable`)
3. По завершении v1 масштабируется до 0 реплик, но **не удаляется** (история ревизий)

**Зачем ReplicaSet отдельно от Deployment:**
- ReplicaSet отвечает только за поддержание N копий
- Смена шаблона в ReplicaSet не затрагивает уже запущенные поды
- Deployment оркестрирует «танец» между старым и новым ReplicaSet

### StatefulSet: алгоритм

1. Поды создаются **строго последовательно** от индекса 0 до N-1: следующий под не создаётся, пока предыдущий не `Ready`
2. К каждому поду привязывается уникальный PVC (из `volumeClaimTemplates`)
3. При падении пода `pod-1` → StatefulSet Controller создаёт новый под **строго с тем же именем** `pod-1` и **переподключает тот же PVC**; данные сохраняются
4. CoreDNS создаёт A-записи для каждого пода через Headless Service: `kubia-0.kubia.default.svc.cluster.local`

### DaemonSet

- Контроллер DaemonSet создаёт по одному поду на каждом узле
- При добавлении нового Worker Node → под автоматически разворачивается
- При удалении узла → под удаляется сборщиком мусора

**Типичные use cases:** Fluentd (логи), Prometheus Node Exporter (метрики), kube-proxy, Calico/Cilium (сеть)

### Job: параллелизм

- `completions` — сколько раз задача должна завершиться успешно
- `parallelism` — сколько подов выполняются одновременно
- Пример: `completions=5`, `parallelism=2` → запускает 2 пода, ждёт завершения, запускает следующие; так до 5 успехов

### CronJob

- По расписанию CronJob Controller создаёт ресурс Job
- `startingDeadlineSeconds` — если задача не запустилась в течение N секунд от назначенного времени — пропускается

---

## Детали и нюансы

- **Network Partition в StatefulSet** → под переходит в Unknown; StatefulSet **не пересоздаёт под автоматически** (принцип At-Most-One) для защиты от Split-Brain; нужно ручное принудительное удаление: `kubectl delete pod <name> --force`
- **Сбой мастера во время CronJob** → K8s может пропустить запуск или создать два Job одновременно → задачи Job/CronJob **обязаны быть идемпотентными**
- **`revisionHistoryLimit`** в Deployment — ограничивает количество хранимых ReplicaSet (история ревизий); при частых деплоях без лимита захламляет etcd
- **`restartPolicy: Always` в Job** → под будет перезапускаться бесконечно после успешного завершения → нужно `OnFailure` или `Never`

---

## Сравнения и trade-offs

| Контроллер | Идентичность подов | Хранилище | Порядок создания | Когда использовать |
|-----------|-------------------|-----------|-----------------|-------------------|
| Deployment | Случайная (cattle) | Общий / без PVC | Параллельно | Stateless web/API |
| StatefulSet | Стабильная (pets) | Уникальный PVC | Строго последовательно | БД, кэш, Kafka, Zookeeper |
| DaemonSet | По одному на узел | — | По мере добавления узлов | Агенты логирования/мониторинга |
| Job | Временные | — | По задаче | Batch-вычисления |
| CronJob | Временные | — | По расписанию | Периодические задачи |

| Rolling Update параметр | Значение | Эффект |
|------------------------|----------|--------|
| `maxSurge: 1` | +1 под поверх желаемого | Доступно N+1 подов во время обновления |
| `maxUnavailable: 0` | 0 подов недоступно | Zero downtime (при наличии readinessProbe) |

## Примеры

```yaml
# Deployment с Rolling Update
apiVersion: apps/v1
kind: Deployment
metadata:
  name: kubia
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: kubia
  template:
    metadata:
      labels:
        app: kubia
    spec:
      containers:
      - image: luksa/kubia:v1
        name: nodejs
```

```bash
# Откат на конкретную ревизию
kubectl rollout undo deployment kubia --to-revision=1
```

```yaml
# StatefulSet с Headless Service и volumeClaimTemplates
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: sleep
spec:
  serviceName: sleep
  replicas: 2
  selector:
    matchLabels:
      app: sleep
  template:
    metadata:
      labels:
        app: sleep
    spec:
      containers:
      - name: sleep
        image: busybox
        volumeMounts:
        - name: sleep-volume
          mountPath: /storagedir
  volumeClaimTemplates:
  - metadata:
      name: sleep-volume
    spec:
      accessModes: [ "ReadWriteOnce" ]
      resources:
        requests:
          storage: 10Mi
# serviceName привязывает к Headless Service
# volumeClaimTemplates: создаёт уникальный PVC для каждого пода
```

```yaml
# Job с completions и parallelism
apiVersion: batch/v1
kind: Job
metadata:
  name: multi-completion-batch-job
spec:
  completions: 5
  parallelism: 2
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: main
        image: luksa/batch-job
```

```yaml
# CronJob по расписанию
apiVersion: batch/v1beta1
kind: CronJob
metadata:
  name: batch-job-every-fifteen-minutes
spec:
  schedule: "0,15,30,45 * * * *"
  startingDeadlineSeconds: 15
  jobTemplate:
    spec: ...  # описание Job
```

## Важно / подводные камни / best practices

- **Deployment вместо ReplicaSet напрямую**: декларативные обновления без риска человеческих ошибок
- **StatefulSet для БД**, не Deployment: уникальные PVC через `volumeClaimTemplates`
- **`revisionHistoryLimit`** ограничивать разумным числом (10-20) при частых деплоях
- **Job задачи — идемпотентны**: возможен повторный или параллельный запуск
- **`restartPolicy: OnFailure` или `Never` в Job**, не `Always`
- **Network Partition + StatefulSet**: принудительное удаление пода вручную если узел недоступен

**ReplicaSet**:::Гарантирует N запущенных копий подов; не умеет сам обновлять запущенные поды при смене шаблона; строительный блок Deployment

**StatefulSet**:::Контроллер для stateful-приложений: стабильные имена подов, уникальный PVC на каждый экземпляр, последовательное создание/удаление; гарантия At-Most-One

**DaemonSet**:::Ровно один под на каждом узле; автоматически добавляется на новые узлы; use cases: Fluentd, Node Exporter, CNI-плагины

**Headless Service**:::clusterIP: None; используется со StatefulSet для DNS A-записей каждого пода (pod-0.svc.ns.svc.cluster.local) без балансировки нагрузки

?

2. v2 постепенно масштабируется вверх (maxSurge), v1 — вниз (maxUnavailable)
3. По завершении v1 масштабируется до 0 (но НЕ удаляется — история ревизий)
