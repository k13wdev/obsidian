# 09. Pod Lifecycle, Debugging, Graceful Shutdown

## Суть и контекст

Понимание детального жизненного цикла пода и инструментов отладки критично для диагностики и решения проблем в production. Graceful shutdown — ключевой механизм защиты от потери запросов и повреждения данных при удалении подов.

---

## Ключевые идеи

- CrashLoopBackOff: экспоненциальная задержка от 10 с до 5 мин между перезапусками контейнера
- ImagePullBackOff: экспоненциальная задержка при неудачных попытках загрузки образа
- Graceful shutdown запускает два параллельных процесса: остановку контейнеров (Kubelet) и обновление iptables (Endpoints controller + kube-proxy)
- preStop hook + `sleep 5` необходим: iptables-правила обновляются асинхронно дольше, чем SIGTERM достигает контейнера
- terminationGracePeriodSeconds (default 30) включает время работы preStop hook
- Shell form в Dockerfile (CMD /binary) → SIGTERM перехватывается shell PID 1, не передаётся процессу → нет graceful shutdown
- `kubectl delete pod --force --grace-period=0` для StatefulSet → риск split-brain (два пода на один PV)
- Exit codes: OOMKilled=137, SIGTERM=143, SIGKILL=137, SIGINT=130, Segfault=139

---

## Определения и термины

**ContainerCreating** — статус пода, означающий, что узел назначен, и контейнерная среда в данный момент загружает образ и создаёт контейнеры.

**ImagePullBackOff** — статус ошибки, при котором Kubernetes не может загрузить образ контейнера из реестра. Применяется алгоритм экспоненциальной задержки (back-off) для повторных попыток загрузки.

**CrashLoopBackOff** — статус, указывающий на то, что контейнер завершился с ошибкой (crash), и система пытается его перезапустить, увеличивая паузу между перезапусками по экспоненте (от 10 секунд до 5 минут).

**OOMKilled** — статус (Exit Code 137), означающий, что процесс попытался выделить больше памяти, чем установлено в его limits, или узел исчерпал память, и ядро Linux принудительно убило процесс.

**Terminating** — состояние пода после установки поля `deletionTimestamp` API-сервером. Означает запуск процедуры корректного завершения работы (graceful shutdown).

**Evicted (Eviction)** — состояние принудительного удаления пода с узла компонентом Kubelet из-за нехватки системных ресурсов на узле (memory/disk pressure) или вытеснения подом с более высоким приоритетом.

**terminationGracePeriodSeconds** — время в секундах (по умолчанию 30), предоставляемое контейнерам пода для корректного завершения работы после отправки сигнала SIGTERM, прежде чем они будут убиты сигналом SIGKILL. Включает время выполнения preStop hook.

**preStop hook** — хук жизненного цикла, выполняемый Kubelet непосредственно перед отправкой сигнала SIGTERM. Используется для инициации graceful shutdown или выполнения произвольных операций перед остановкой.

**SIGTERM** — POSIX-сигнал (Exit Code 143 = 128 + 15), отправляемый главному процессу контейнера для требования корректного завершения работы.

## Как устроено

### Проблемы при запуске: алгоритмы

**FailedScheduling (Pending):**
- Планировщик не находит узел, удовлетворяющий requests (CPU/Memory) или nodeAffinity/selectors
- Под остаётся в `Pending` с причиной `FailedScheduling` в Events

**ImagePullBackOff:**
- Kubelet на узле пытается загрузить образ
- При ошибке (неверное имя, нет доступа, недоступен registry): переход в `ImagePullBackOff`
- Повторные попытки с экспоненциальной задержкой: до 5 минут между попытками

**CrashLoopBackOff:**
- Контейнер скачан и запущен, но процесс завершается с ошибкой
- Kubelet перезапускает с задержкой: 10s → 20s → 40s → 80s → 160s → **300s (max)**

### Graceful Shutdown: алгоритм

```
API сервер получает запрос на удаление
       ↓
Устанавливает deletionTimestamp → статус Terminating
       ↓
Параллельно запускаются два процесса:
│
├── Процесс A (Kubelet):
│     1. Выполняет preStop hook → ждёт завершения
│     2. Отправляет SIGTERM главному процессу
│     3. Ждёт terminationGracePeriodSeconds
│     4. Если процесс жив → SIGKILL
│     5. Уведомляет API-сервер → Pod физически удаляется
│
└── Процесс B (Endpoints Controller + kube-proxy):
      1. Endpoints controller удаляет IP из Endpoints
      2. Уведомляет kube-proxy
      3. kube-proxy обновляет iptables на всех узлах
      (МЕДЛЕННЕЕ, чем Процесс A!)
```

> **Критично:** путь от API-сервера до обновления iptables на всех узлах значительно длиннее пути до SIGTERM. Без `preStop sleep` запросы ещё идут по старым iptables-правилам на уже завершающийся контейнер → Connection Refused.

### Exit Codes (Linux стандарт: 128 + №сигнала)

| Exit Code | Сигнал | Причина |
|-----------|--------|---------|
| 130 | SIGINT (2) | Ctrl+C / принудительное прерывание |
| 137 | SIGKILL (9) | OOMKilled / --force / таймаут |
| 139 | SIGSEGV (11) | Segmentation fault |
| 143 | SIGTERM (15) | Graceful shutdown request |

## Детали и нюансы

- **Shell form в Dockerfile (`CMD /mybinary`)**: SIGTERM идёт на shell (PID 1), которая не передаёт его дочернему процессу → нет graceful shutdown → SIGKILL по таймауту. Решение: exec form `CMD ["/mybinary"]` или использовать dumb-init/tini
- **`kubectl delete pod --force --grace-period=0` + StatefulSet**: API-сервер немедленно удаляет ресурс → StatefulSet Controller создаёт замену → два пода с одним именем на один PV → split-brain / повреждение данных. Применять только если узел гарантированно мёртв
- **Ошибка preStop hook**: генерируется `FailedPreStopHook` в Events, но терминация продолжается — pod всё равно будет убит
- **Multi-Attach error**: PV с `ReadWriteOnce` ещё занят старым подом на другом узле → новый под зависает в `ContainerCreating`
- **`kubectl top pod` после запуска**: метрики могут возвращать ошибку — агрегация через Metrics Server занимает несколько минут
- **Глубокие данные через `/dev/termination-log`**: приложение записывает сообщение об ошибке → Kubelet включает его в `kubectl describe pod` в блоке `Last State: Message`

---

## Сравнения и trade-offs

| Статус | Что происходит | Действие |
|--------|---------------|---------|
| ContainerCreating | Загрузка образа, создание контейнера | Подождать; проверить образ |
| ImagePullBackOff | Ошибка загрузки образа (back-off) | `kubectl describe pod` → Events |
| CrashLoopBackOff | Процесс падает, экспоненциальный restart | `kubectl logs --previous` |
| OOMKilled (Exit 137) | Превышен limits.memory | Увеличить limit или оптимизировать |
| Terminating | Graceful shutdown запущен | Ждать `terminationGracePeriodSeconds` |
| Evicted | Нехватка ресурсов на узле | Проверить memory/disk pressure |

## Примеры

```bash
# Просмотр событий и причин ошибок
kubectl describe pod <pod_name>

# Логи упавшего контейнера (предыдущий экземпляр)
kubectl logs <pod_name> --previous

# Интерактивный shell внутри пода
kubectl exec -it <pod_name> -- /bin/sh

# Получить exit code контейнера
kubectl get pod <pod_name> -o json | jq '.status.containerStatuses[].lastState.terminated.exitCode'

# Проброс порта для отладки (минуя Service)
kubectl port-forward <pod_name> 8080:80
```

```yaml
# preStop hook: задержка для обновления iptables
lifecycle:
  preStop:
    exec:
      command:
      - sh
      - -c
      - "sleep 5"
```

## Важно / подводные камни / best practices

- **preStop hook с `sleep 5`**: дать время на обновление iptables перед SIGTERM — иначе Connection Refused на запросы в flight
- **Exec form для CMD/ENTRYPOINT** (`CMD ["/binary"]`): SIGTERM идёт прямо в процесс, а не в shell
- **terminationGracePeriodSeconds** настраивать под требования приложения (не всегда 30 сек)
- **Не размещать критичные миграции в preStop/SIGTERM**: платформа не гарантирует завершение (SIGKILL по таймауту, сбой узла)
- **`/dev/termination-log`**: удобный способ получить причину сбоя без просмотра логов
- **`--force --grace-period=0` только для гарантированно мёртвых узлов** в StatefulSet — риск split-brain

## Карточки для повторения

#flashcards/k8s/lifecycle

**ContainerCreating**:::Узел назначен; Kubelet загружает образ и создаёт контейнеры. Нормальный временный статус

**ImagePullBackOff**:::Kubernetes не может загрузить образ; алгоритм экспоненциальной задержки между попытками (до 5 мин). Причины: неверное имя, нет доступа, недоступен registry

**CrashLoopBackOff**:::Контейнер падает и перезапускается с экспоненциальной задержкой: 10→20→40→80→160→300 сек. `kubectl logs --previous` для анализа

**OOMKilled**:::Exit Code 137; процесс превысил limits.memory → ядро Linux немедленно убивает через OOM Killer (SIGKILL)

**Terminating**:::deletionTimestamp установлен API-сервером; запущен graceful shutdown (preStop hook + SIGTERM + таймер)

**Eviction**:::Kubelet принудительно удаляет поды при memory/disk pressure; порядок: BestEffort → Burstable → Guaranteed

**terminationGracePeriodSeconds**:::Время (default 30 сек) после SIGTERM до принудительного SIGKILL; включает время preStop hook

**preStop hook**:::Выполняется Kubelet перед SIGTERM; используется для sleep (ожидание обновления iptables) или graceful закрытия соединений

Graceful shutdown: полный алгоритм
?
1. API: deletionTimestamp → статус Terminating
2. Параллельно:
   A. Kubelet: preStop hook → SIGTERM → ждёт → SIGKILL (если таймаут)
   B. Endpoints Controller → kube-proxy → обновление iptables (медленнее!)
3. Pod физически удаляется из API

Почему нужен preStop sleep?::Обновление iptables на всех узлах (через Endpoints → kube-proxy) происходит МЕДЛЕННЕЕ, чем SIGTERM достигает контейнера. Без sleep новые запросы по старым iptables идут на уже завершающийся контейнер → Connection Refused

Edge case: shell form CMD + SIGTERM?::Shell перехватывает SIGTERM, не передаёт дочернему процессу → нет graceful shutdown → SIGKILL по таймауту. Решение: exec form `CMD ["/binary"]` или dumb-init

Edge case: kubectl delete pod --force + StatefulSet?::Немедленное удаление ресурса → StatefulSet создаёт замену → два пода на один PV → split-brain/повреждение данных. Только при гарантированно мёртвом узле

Edge case: ошибка preStop hook?::Генерируется FailedPreStopHook в Events, но терминация продолжается — pod всё равно будет убит по плану

Edge case: Multi-Attach error?::PV с ReadWriteOnce занят старым подом → новый под зависает в ContainerCreating. Нужно принудительно удалить старый pod

Exit Code 137::OOMKilled или SIGKILL (принудительное убийство) = 128 + 9

Exit Code 143::Graceful SIGTERM (корректное завершение по запросу) = 128 + 15

Exit Code 139::Segmentation fault (SIGSEGV) = 128 + 11

kubectl: как посмотреть логи упавшего контейнера?::`kubectl logs <pod> --previous` — логи предыдущего (упавшего) экземпляра

kubectl: как посмотреть причину ошибки пода?::`kubectl describe pod <name>` → блок Events (FailedScheduling, Failed to pull image и т.д.)
