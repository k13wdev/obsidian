# 06. Артефакты, Реестры, Версионирование

## Суть и контекст

Артефакт — результат работы pipeline, передаваемый между стадиями и хранимый в специализированном реестре. Бинарный код (JAR, Docker образ, Helm chart, Lambda zip) не хранится в Git — только в Artifact Repository. Ключевые принципы: Immutable Infrastructure (не обновляем на месте, пересоздаём), Semantic Versioning (читаемые версии), Git Commit ID как идентификатор артефакта.

---

## Ключевые идеи

- Файлы без `artifacts: paths:` удаляются сразу после job — каждая job работает в изолированном окружении
- Cache ≠ Artifacts: cache — зависимости между запусками pipeline; artifacts — жёсткая передача данных между стадиями
- Git Commit ID как тег Docker-образа/zip: значимое имя + точный rollback к конкретному коммиту
- Immutable Infrastructure: пересоздание сервера вместо обновления на месте — нет конфигурационного дрейфа
- Semantic Versioning: PATCH/MINOR/MAJOR сигнализируют тип изменений потребителям артефакта
- Tag immutability в ECR/Nexus: запрет перезаписи тегов → детерминированные деплои
- Инъекция Build ID + Commit ID в Helm аннотации → точная диагностика работающего кластера
- Promotion (dev→staging→prod) в GitOps: через PR в environment-репозиторий; в Lambda — смена алиаса

## Определения и термины

**Артефакт (Artifact)** — файлы, которые создаются job и затем сохраняются. Файлы, не объявленные через ключевое слово `artifacts`, немедленно удаляются после завершения job. В контексте Helm — упакованный архив (например, `movies-marketplace-1.0.0.tgz`). В Docker — образ контейнера.

**Artifact Repository (Хранилище бинарных файлов)** — система хранения бинарного кода: сборки, пакеты, сторонние плагины. Git/SVN хранят код, не бинарные файлы. Примеры: Nexus Repository OSS, JFrog Artifactory, GitLab Package Registry, Amazon ECR.

**GitLab Package Registry** — функция GitLab для хранения пакетов (Generic, Maven, npm, NuGet, PyPI). Позволяет версионировать завершённые сборки рядом с исходным кодом.

**Immutable Infrastructure (Неизменяемая инфраструктура)** — инфраструктура, основанная на неизменяемых компонентах: при обновлении пересоздаются и заменяются, а не обновляются на месте. Новые серверы инициализируются с заранее собранного образа, старые уничтожаются. Сокращает вероятность дрейфа конфигураций.

**Semantic Versioning (Семантическое версионирование)** — подход к версионированию: `MAJOR.MINOR.PATCH`. PATCH = исправления багов, MINOR = новые функции, MAJOR = несовместимые (breaking) изменения. Потребители артефакта понимают характер изменений без чтения кода.

**Helm Chart** — пакет предварительно сконфигурированных ресурсов Kubernetes. Включает:
- `Chart.yaml` — метаданные о версии
- `values.yaml` — значения по умолчанию
- `templates/` — манифесты Kubernetes

Аналог `yum`/`apt` для Kubernetes: клиент формирует манифесты на лету, подставляя переменные из `values.yaml` в шаблоны; создаёт инстанс — **Release**.

**Promotion (Продвижение)** — процесс передачи приложения из одной среды в другую (staging → production). В K8s/Jenkins X — через PR в Git-репозиторий целевого окружения. В бессерверных приложениях — сменой указателя Lambda-алиаса на конкретную версию функции.

---

## Как устроено

### Promotion Helm-артефакта через Jenkins X + GitOps

1. Выполняется `jx promote --app jx-movies-store --version 0.0.3 --env production`
2. Jenkins X автоматически генерирует PR в целевой репозиторий окружения (`environment-watchlist-production`)
3. В PR обновляется `env/requirements.yaml` — новая версия Helm-чарта (например, 0.0.1 → 0.0.3)
4. После валидации PR: pipeline загружает чарт из ChartMuseum + образы из Docker Registry
5. Kubernetes создаёт ресурсы (pod, service, ingress) в целевом кластере

### Promotion AWS Lambda через Aliases

1. Код + зависимости архивируются в `.zip` (имя файла = git commit ID) → Amazon S3
2. `aws lambda update-function-code` → деплой из S3 в Lambda-функцию
3. `aws lambda publish-version` → новая immutable версия из `$LATEST`
4. Jenkins парсит JSON-ответ через `jq` → извлекает номер версии (`Version`)
5. `aws lambda update-alias` → алиас (`sandbox`/`staging`/`production`) указывает на новую версию

### Настройка Nexus как Docker Registry

1. Nexus разворачивается через Packer AMI + Shell-скрипт
2. POST к Nexus REST API (`/service/rest/v1/script`) с Groovy-пейлоадом: `repository.createDockerHosted` с портом 5000
3. В интерфейсе Nexus создаётся кастомная роль `ManagedDockerPrivateRegistry` с правами `nx-repository-view-docker-docker-registry-*`
4. Создаётся пользователь `jenkins` с этой ролью
5. Для HTTPS без валидного сертификата: флаг `--insecure-registry` в конфигурации Docker Engine

## Детали и нюансы

- **Cache vs Artifacts — не взаимозаменяемы**: cache хранится на runner (или S3), может не передаваться между проектами и protected/unprotected ветками; artifacts всегда загружаются на инстанс GitLab/S3; artifacts expire (по умолчанию 30 дней)
- **Недетерминированные сборки**: транзитивные зависимости могут обновиться между CI и финальной сборкой → скрытые баги; решение: явная привязка версий + sha256-хэши пакетов (если хэш изменился — сборка падает)
- **Helm аналогия с пакетными менеджерами**: Helm Chart = rpm/deb пакет; Helm Repository = yum/apt репозиторий; `helm install` = `yum install`
- **Tag immutability**: без защиты `latest` перезаписывается с каждым деплоем → нет трассировки; ECR с `Tag immutability: Enabled` возвращает ошибку при попытке push на существующий тег

---

## Сравнения и trade-offs

| | Cache | Artifacts |
|-|-------|----------|
| Назначение | Зависимости между запусками pipeline | Передача данных между стадиями |
| Хранение | На runner (или S3) | На GitLab instance (или S3) |
| Срок жизни | До очистки runner | По умолчанию 30 дней |
| Надёжность | Удалён → pipeline работает (перескачает) | Удалён → следующая стадия упадёт |
| Между проектами | Нет | Только в рамках одного pipeline |

| Инфраструктура | Подход | Конфигурационный дрейф | Время деплоя |
|----------------|--------|----------------------|-------------|
| Mutable (обновление на месте) | apt upgrade / ansible update | Высокий риск | Меньше |
| Immutable (пересоздание) | Packer → новый образ → destroy old | Минимальный | Больше (но предсказуемо) |

| Реестр | Тип | Форматы |
|--------|-----|---------|
| Nexus OSS | Self-hosted | Docker, Maven, npm, PyPI, Helm, raw |
| JFrog Artifactory | Self-hosted / Cloud | Docker, Maven, npm, PyPI, Helm, raw |
| GitLab Package Registry | Встроен в GitLab | Generic, Maven, npm, NuGet, PyPI |
| Amazon ECR | Cloud (AWS) | Docker, OCI |

## Примеры

```bash
# Публикация Generic-пакета в GitLab Package Registry
curl --user "hfc-generic-build:<deploy token password>" \
  --upload-file my_app.tar.gz \
  "${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/packages/generic/stable_releases/0.0.1/$PACKAGE_FILE"
```

> `CI_API_V4_URL` и `CI_PROJECT_ID` — предопределённые переменные GitLab. Пакет помещается в группу `stable_releases` версии `0.0.1`.

```groovy
// Jenkins: тегирование и push Docker-артефакта в приватный реестр
stage('Push'){
    docker.withRegistry(registry, 'registry') {
        docker.image(imageName).push(commitID())
        if (env.BRANCH_NAME == 'develop') {
            docker.image(imageName).push('develop')
        }
        if (env.BRANCH_NAME == 'preprod') {
            docker.image(imageName).push('preprod')
        }
        if (env.BRANCH_NAME == 'master') {
            docker.image(imageName).push('latest')
        }
    }
}
```

> `push(commitID())` — тег = хэш коммита (иммутабельный). Алиасные теги (`develop`, `preprod`, `latest`) добавляются по ветке.

```bash
# Аутентификация Jenkins в Amazon ECR
aws ecr get-login-password --region REGION | docker login \
  --username AWS --password-stdin \
  ACCOUNT_ID.dkr.ecr.REGION.amazonaws.com/mlabouardy/movies-loader
```

```bash
# Создание и упаковка Helm-чарта
helm create watchlist
helm package chart --app-version ${appVersion} --version ${chartVersion}
# Создаст: movies-marketplace-1.0.0.tgz
```

```bash
# Публикация Helm-чарта в GitHub Pages как Helm-репозиторий
sh 'helm repo index --url https://mlabouardy.github.io/watchlist-charts/ .'
sh 'git add index.yaml movies-marketplace-1.0.0.tgz && git commit -m "movies-marketplace" && git push origin master'
```

> `helm repo index` генерирует `index.yaml` с метаданными чартов. GitHub Pages служит HTTP-сервером Helm-репозитория.

## Важно / подводные камни / best practices

- **Git Commit ID как тег**: `BUILD_ID` Jenkins неудобен; commit ID = значимое имя + точный rollback к конкретному состоянию кода
- **Immutable Infrastructure**: обновление Jenkins v1 → v2 на месте = риск простоя; Packer AMI → новый сервер → destroy old = без дрейфа
- **Инъекция Build ID + Commit ID в Helm аннотации**: `git/commitId: {{ .Values.metadata.git.commitId | quote }}` — точная диагностика что работает в кластере
- **Антипаттерн: Cache вместо Artifacts**: если cache удалён → следующая стадия упадёт (нет скомпилированного кода); artifacts — жёсткая зависимость; cache — мягкая оптимизация
- **Явные версии + sha256 для зависимостей**: транзитивные зависимости без pinning могут измениться между CI и финальной сборкой → скрытые баги в production
- **Hardcoded credentials в .gitlab-ci.yml**: навсегда в git истории; использовать CI Variables с Protected + Masked
