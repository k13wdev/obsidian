# 03. Тестирование в CI/CD

## Суть и контекст

Тестирование в CI/CD — система автоматических проверок, встроенных в pipeline и выполняемых при каждом коммите. Цель — максимально раннее обнаружение ошибок (Shift-left), пока изменения ещё не слиты в основную ветку. Правильно выстроенный уровень тестирования: сначала быстрые (unit), потом интеграционные, последними — медленные E2E. Неверная конструкция (Ice Cream Cone) — основание из медленных UI-тестов — убивает скорость обратной связи.

---

## Ключевые идеи

- Test Pyramid: unit (тысячи, быстрые) → integration (меньше) → E2E/UI (минимум, медленные)
- Fail-fast: если упал unit-тест (1 мин), pipeline блокируется немедленно — не нужно ждать E2E (1 час)
- Flaky tests — маскируют реальные баги; retry до SUCCESS уничтожает информативность тестов
- Test sharding: N шардов параллельно; ограничен длительностью самого долгого теста
- Параллелизм unit-тестов — «запах кода»: если нужно распараллеливать unit — они написаны неверно
- Test coverage как quality gate: ниже порога → pipeline останавливается
- Shift-left: тесты и проверки как можно раньше в жизненном цикле; удешевляет исправление ошибок
- Изоляция SUT: каждый тестовый запуск — своя in-memory БД, Docker-контейнер; нет влияния внешнего окружения
- Review environments (GitLab): временные среды для каждой ветки, уничтожаются после слияния
- always()/finally-логика: очистка SUT гарантируется даже при падении тестов

---

## Определения и термины

**Unit tests (Юнит-тесты)** — тесты, проверяющие отдельную единицу кода (одну функцию или метод) в изоляции. Должны выполняться очень быстро (менее 1 минуты на весь набор). Составляют широкое основание пирамиды тестов.

**Integration tests (Интеграционные тесты)** — тесты, проверяющие, как группы функций или сервисов взаимодействуют друг с другом (например, взаимодействие приложения с базой данных). Оперируют на более высоком уровне абстракции, чем юнит-тесты, и требуют подключения зависимостей. Выполняются дольше модульных.

**End-to-End tests (E2E / User tests / Acceptance tests / Сквозные тесты)** — тесты, симулирующие поведение пользователя, проверяющие приложение в целом через GUI или REST API. Самые медленные и хрупкие тесты. Доказывают, что приложение делает то, что ожидает клиент.

**Smoke tests (Смоук-тесты)** — тесты системного уровня, выполняющие базовую проверку функционала на развёрнутом сервере, чтобы убедиться, что система работает и конфигурации корректны. В некоторых компаниях смоук-тесты представляют собой полный набор приёмочных испытаний. Запускаются, например, через cURL.

**Regression tests (Тесты регрессии)** — тестирование для подтверждения того, что внесённые изменения не повредили функции, ранее работавшие правильно. Входят в блок функционального тестирования конвейера.

**Performance tests (Тесты производительности)** — тесты, определяющие, насколько быстро программа выполняет одну операцию (одна транзакция, конвертация). Занимают много времени, склонны к нестабильным результатам. Включают:
- **Load tests** — нагрузка множеством пользователей
- **Soak tests** — длительные тесты на утечки памяти и ресурсов

**Contract tests (Consumer-Driven Contract Testing, Pact)** — В предоставленных источниках не раскрыто.

**Test pyramid (Пирамида тестов)** — визуализация идеального распределения тестов: подавляющее большинство составляют модульные тесты (основание), меньше — интеграционных (середина), и совсем немного — сквозных/браузерных тестов (вершина).

**Test trophy** — В предоставленных источниках не раскрыто.

**Ice Cream Cone (антипаттерн)** — перевёрнутая пирамида: разработчики полагаются на массивы медленных интеграционных или ручных тестов; юнит-тестов мало; замедляет обратную связь на порядки.

**Flaky tests (Нестабильные тесты / флейки)** — тесты, которые непоследовательны: иногда проходят успешно, а иногда проваливаются при одних и тех же условиях. Повышают «шум» в pipeline, маскируя реальные баги. Могут быть вызваны как ошибками в самом тесте (race conditions), так и проблемами в тестируемом коде.

**Test parallelization (Распараллеливание тестов)** — одновременный запуск нескольких тестов (по одному на каждое ядро процессора или на разных машинах) для уменьшения общего времени выполнения pipeline. Тесты не должны зависеть друг от друга, иметь строгий порядок или разделять общую память.

**Test sharding (Шардинг тестов)** — метод распараллеливания, при котором общее количество тестов распределяется (дробится) на подмножества для одновременного выполнения на нескольких машинах (шардах).

**Test coverage (Покрытие тестами)** — оценка того, насколько эффективно тесты используют тестируемый код. Показывает (строка за строкой), какой код выполнялся во время тестов (statement coverage, condition coverage). Используется как «шлюз качества»: если покрытие падает ниже заданного порога — pipeline прерывается.

**Shift-left testing (Сдвиг влево)** — выполнение задач по сборке, проверке (тестированию) и защите как можно раньше в жизненном цикле разработки (для каждой порции нового кода). Ускоряет получение обратной связи, удешевляет исправление ошибок, пока изменения не слиты в основную ветку.

**Test environments / SUT (System Under Test)** — инфраструктура, куда развёртывается код для проведения тестирования. В GitLab используются «review environments» — временные среды, создаваемые для каждой ветки и уничтожаемые после слияния.

**Fuzz testing (Фазз-тестирование)** — метод тестирования безопасности и надёжности: CI job генерирует случайные или мутированные входные данные и передаёт их тестируемой функции; при возникновении исключения фиксируется байт-последовательность, вызвавшая краш.

## Как устроено

### Пирамида тестов (Test Pyramid)

```
          ┌─────────────┐
          │  User/E2E   │  ← мало тестов, тестируют UI, медленно
          │  tests      │
        ┌─┴─────────────┴─┐
        │ Integration     │  ← среднее число, проверка совместной работы
        │ tests           │
      ┌─┴─────────────────┴─┐
      │   Unit tests         │  ← много тестов, тестируют функции, быстро
      └─────────────────────┘
```

Порядок запуска в pipeline: **Linter/Unit → Integration → E2E**. Fail-fast: если упал unit, pipeline блокируется немедленно.

### Шардинг E2E тестов (GitHub Actions matrix)

1. CI конфигурация определяет стратегию `matrix` с массивом индексов (например, 0–5) и общим числом шардов (6)
2. Запускаются 6 изолированных окружений параллельно
3. Библиотека (например, `pytest-shard`) принимает аргументы `--shard-id` и `--num-shards`
4. Каждый инстанс вычисляет и выполняет только свою порцию тестов (например, 9 из 50)
5. Время выполнения сокращается до длительности самого долгого шарда

### Fuzz testing (GitLab Coverage-guided)

1. CI/CD job загружает бинарный файл (`gitlab-cov-fuzz`) и фазз-движок
2. Фазз-движок генерирует случайные или мутированные (coverage-guided) байты и передаёт в фазз-мишень (fuzz target)
3. Фазз-мишень трансформирует сырые байты в ожидаемый тип данных (строку, массив) и передаёт тестируемой функции
4. При возникновении исключения (например, `IndexError`) движок возвращает статус сбоя и байт-последовательность (sample), вызвавшую краш

### UI тестирование с Chrome Headless (Angular / Karma)

1. Karma и Jasmine конфигурируются для работы с браузером без UI
2. Добавляются флаги: `--headless`, `--disable-gpu`, `--no-sandbox`
3. Контейнер CI (без графической оболочки) вызывает тесты, которые рендерят HTML в памяти и выполняют проверки (например, наличие текста в `span`)

### Performance testing (k6 API load testing)

1. Скрипт k6 определяет стадии нагрузки: разгон (`duration: 5m, target: 100 пользователей`), удержание (`10m, 100`), спад
2. Конфигурируются пороги `thresholds` (например, 99% HTTP-запросов < 1500 мс)
3. Выполняются GET-запросы с валидацией структуры ответа; при нарушении порога — pipeline получает сигнал об ошибке

## Детали и нюансы

- **Coverage аналогичен Linting**: оба анализируют код на ранней фазе сборки (до запуска ресурсоёмких сред) и выступают жёсткими «шлюзами» (gates), блокируя интеграцию несоответствующего кода
- **IaC тестируется теми же инструментами**: Terraform, Ansible проходят linting и проверки безопасности в том же CI/CD pipeline, что и код приложения — до раскатки инфраструктуры
- **Периодический CI для флейков**: инструменты CI на PR редко отлавливают флейки из-за их редкости (1 из 500 запусков). Решение — настройка периодического CI (например, каждый час), чтобы накопить статистику без блокировки конкретного PR
- **Единая БД для параллельных тестов**: конфликты при одновременном запуске в шардах. Решение: разворачивать локальные in-memory БД (`H2`, `mongo-unit`) при старте каждого теста
- **Параллелизм unit-тестов = «запах кода»**: unit-тесты должны проверять код за доли секунды; если их приходится распараллеливать на уровне инфраструктуры — они написаны неверно (слишком тяжёлые или зависят от I/O)

## Сравнения и trade-offs

| Тип теста | Скорость | Хрупкость | Что проверяет | Место в pyramid |
|-----------|---------|----------|--------------|----------------|
| Unit | Очень быстро (<1 мин весь набор) | Низкая | Одна функция в изоляции | Основание (много) |
| Integration | Медленнее unit | Средняя | Взаимодействие компонентов + БД | Середина |
| E2E / User | Медленно | Высокая | Всё приложение через GUI/API | Вершина (мало) |
| Smoke | Быстро | Низкая | Базовая работоспособность среды | После деплоя |
| Performance | Очень медленно | Высокая (нестабильные результаты) | Скорость, нагрузка, утечки | Отдельный stage |

| Подход | Обратная связь | Проблема |
|--------|---------------|---------|
| Test Pyramid (правильно) | Быстрая (fail-fast на unit) | Требует дисциплины в написании unit |
| Ice Cream Cone (антипаттерн) | Медленная (UI-тесты как основа) | Часы ожидания, дорогое исправление |

| Стратегия распараллеливания | Сложность | Ограничение |
|-----------------------------|----------|-------------|
| По ядрам CPU (1 тест = 1 ядро) | Низкая | Тесты не должны шарить память |
| Test sharding (N машин) | Средняя | Ограничен самым долгим тестом; нужна независимость тестов |

## Примеры

```python
# Блокировка pipeline при снижении тестового покрытия
if coverage < prev_coverage:
  fail('coverage reduced from {} to {}'.format(prev_coverage, coverage))
```

```bash
# Запуск шардинга (распараллеливания) тестов через Pytest CLI
pytest --shard-id={} --num-shards={} {}
```

```bash
# Генерация HTML-отчёта о покрытии тестами в Go
go test -coverprofile=cover/cover.cov
go tool cover -html=cover/coverage.cov -o coverage.html
```

```groovy
// Запуск тестов параллельно в Jenkins Declarative Pipeline
stage('Pre-integration Tests'){
    parallel(
        'Quality Tests': {
            imageTest.inside{ sh 'golint' }
        },
        'Unit Tests': {
            imageTest.inside{ sh 'go test' }
        }
    )
}
```

```javascript
// Конфигурация Headless Chrome для Karma в Angular
customLaunchers: {
  ChromeHeadlessCI: {
    base: 'Chrome',
    flags: ['--headless', '--disable-gpu', '--no-sandbox', '--remote-debugging-port=9222']
  }
}
```

```yaml
# Finally-логика в GitHub Actions для гарантированной очистки тестовой среды (SUT)
clean-up-sut:
  if: ${{ always() }}
  needs: [setup-sut, end-to-end-tests]
```

> `always()` гарантирует запуск очистки даже если предыдущие jobs завершились с ошибкой — не оставляет висящую инфраструктуру.

## Важно / подводные камни / best practices

- **Запускать первыми самые быстрые тесты**: Linter/Unit → Integration → E2E; fail-fast экономит ресурсы и время ожидания
- **Изоляция SUT**: integration-тесты внутри Docker-контейнеров + in-memory БД; исключает влияние внешнего окружения и конфликты при параллельном запуске
- **Строгий порог coverage для новых проектов (например, 80%)**: автоматическая остановка pipeline при снижении заставляет поддерживать качество с первого дня
- **always()/finally для SUT cleanup**: без этого упавшие E2E-тесты оставляют висящую инфраструктуру → исчерпание ресурсов
- **Антипаттерн — retry упавших тестов до SUCCESS**: маскирует flaky тесты; баг проникает в production; решение: периодический CI для накопления статистики по флейкам
- **Антипаттерн — единая сборка для тестов и production**: E2E проходят на образах, не идентичных production → ложное чувство безопасности; единые Dockerfile для CI и CD
- **Антипаттерн — Ice Cream Cone**: большинство тестов — медленные UI/ручные; unit-тесты отсутствуют; обратная связь задерживается на порядки

**Integration tests**:::Тесты взаимодействия групп функций/сервисов (приложение + БД); выполняются дольше unit; требуют подключения зависимостей

**E2E / User / Acceptance tests**:::Симулируют поведение пользователя через GUI или REST API; самые медленные и хрупкие; доказывают соответствие ожиданиям клиента

**Smoke tests**:::Базовая проверка работоспособности на развёрнутом сервере; конфигурации корректны, система работает; запуск через cURL; могут = полный набор приёмочных тестов

**Regression tests**:::Подтверждение что изменения не сломали ранее работающие функции; блок функционального тестирования pipeline

**Performance tests**:::Скорость одной операции; медленные + нестабильные; включают: Load tests (нагрузка пользователями) и Soak tests (утечки памяти/ресурсов)

**Flaky tests**:::Непоследовательные тесты: иногда pass, иногда fail при одних условиях; маскируют реальные баги; причины: race conditions, ошибки в тесте или коде

**Test sharding**:::Разбивка тестов на подмножества для параллельного выполнения на N машинах; ограничен длительностью самого долгого теста; нужна независимость тестов

**Test coverage**:::Строка за строкой: какой код выполнялся (statement/condition coverage); quality gate — pipeline останавливается при падении ниже порога

**Shift-left testing**:::Выполнение тестирования как можно раньше в жизненном цикле; ускоряет обратную связь, удешевляет исправление ошибок до слияния

**SUT (System Under Test)**:::Инфраструктура для тестирования; GitLab «review environments» — временные среды для каждой ветки, уничтожаются после слияния
