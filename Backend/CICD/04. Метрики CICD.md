# 04. Метрики CI/CD

## Суть и контекст

Метрики CI/CD позволяют измерять эффективность инженерных команд и качество процессов доставки кода. Ключевой фреймворк — DORA (DevOps Research and Assessment): четыре показателя, разбивающие команды на четыре категории от низкоэффективных до элитных. Метрики делятся на два типа: **скорость** (Deployment Frequency, Lead Time for Changes) и **стабильность** (MTTR, Change Failure Rate). Важно: технические метрики (CPU, memory) — «vanity metrics»; ценность имеют только метрики, связанные с бизнес-целями.

---

## Ключевые идеи

- DORA: 4 метрики → скорость (частота + Lead Time) и стабильность (MTTR + Change Failure Rate)
- «Элитные» команды: деплой несколько раз в день, Lead Time < 1 часа, MTTR < 1 часа, Change Failure Rate 0–15%
- Редкие деплои «ради стабильности» → Change Failure Rate 100% (большие пакеты изменений)
- Оптимизировать MTTR, а не MTBF: 100% MTBF в распределённых системах недостижим
- Coverage как quality gate: ниже порога → pipeline stop; риск «бесполезных тестов» ради лимита
- Flaky rate: периодический CI (раз в час на main) — единственный способ накопить статистику
- SLO pipeline stages: <1 мин unit, <5 мин integration, <30 мин весь pipeline
- Только технические метрики (CPU/memory) — «vanity metrics»; нужна связь с бизнес-KPI

---

## Определения и термины

**DORA metrics** — четыре ключевых показателя, измеряющих эффективность команд разработчиков. Разбивают команды на четыре категории: низкоэффективные, среднеэффективные, высокоэффективные и суперэффективные (элитные). Оценивают скорость (Deployment Frequency, Lead Time) и стабильность (MTTR, Change Failure Rate).

**Deployment Frequency (Частота развёртывания)** — метрика скорости; показывает, как часто компания успешно выпускает релизы в production. Для «элитных» команд — несколько раз в день.

**Lead Time for Changes (Время доставки изменений)** — количество времени, которое требуется, чтобы коммит попал на production. Для «элитных» команд — менее одного часа.

**MTTR (Mean Time To Recovery — Среднее время восстановления)** — метрика стабильности; сколько времени требуется для вывода сервиса из нерабочего состояния после сбоя. «Элитные» компании восстанавливаются менее чем за один час.

**Change Failure Rate (Частота сбоев при изменениях)** — метрика стабильности; какой процент деплоев в production приводит к сбоям сервиса. В «элитных» командах — 0–15%.

**Test Coverage (Покрытие тестами)** — оценка того, насколько эффективно тесты используют тестируемый код. Показывает строка за строкой, какой код выполнялся во время тестов. Типы: statement coverage, branch coverage, functions, lines. Используется как «шлюз качества»: ниже порога → pipeline stop.

**%C/A (Percent Complete and Accurate — Доля завершённой и правильной работы)** — показатель качества выполнения на каждом этапе потока создания ценности. Как часто клиенты или следующие по потоку команды получают продукт, пригодный к использованию сразу без исправлений. Используется вместе с Lead Time для оценки hand-offs между стадиями.

**Flaky tests (Нестабильные тесты)** — тесты, работающие непоследовательно: иногда проходят, иногда нет при одних условиях. Маскируют реальные ошибки в системе, снижают уровень доверия к CI.

## Как устроено

### DORA метрики для не-SaaS проектов

| DORA метрика | SaaS | Не-SaaS (библиотеки / бинарные файлы) |
|-------------|------|--------------------------------------|
| Deployment Frequency | Частота деплоев в prod | Частота релизов (доступность для пользователей) |
| Lead Time for Changes | Коммит → prod | Коммит → публичный релиз |
| Change Failure Rate | % деплоев с инцидентом | % релизов с критической ошибкой → экстренный патч |
| MTTR | Время до восстановления сервиса | Время на выпуск исправляющего патча с момента сообщения |

### Jenkins + Prometheus + Grafana

1. Устанавливается плагин **Prometheus Metrics** в Jenkins → открывается HTTP endpoint `/prometheus`
2. Сервер Prometheus регулярно GET-запросами собирает метрики: длительность сборки (`default_jenkins_builds_stage_duration_milliseconds_summary`), размер очереди, статус
3. Grafana подключается к Prometheus как Data Source
4. Создаются дашборды с PromQL-запросами: build time, нагрузка на воркеры, % успешных/упавших сборок, CPU, JVM memory, executor free

### SonarQube в Jenkins pipeline

1. Запускаются линтеры (golint, eslint, Code Climate)
2. Генерируется файл отчёта покрытия (go test -coverprofile, npm run coverage)
3. `sonar-scanner` читает исходный код и отчёты покрытия
4. Данные загружаются на сервер SonarQube; сервер сверяет с правилами Quality Profiles (баги, уязвимости, code smells)
5. `waitForQualityGate()` ожидает статус (timeout 5 мин); если статус != `OK` → `error` останавливает сборку

## Детали и нюансы

- **DORA: взаимосвязь метрик**: редкие деплои → большой пакет изменений → Change Failure Rate 100%; быстрый откат (rollback) снижает MTTR; Canary/Blue-Green переводят MTTR из дней в минуты
- **Lead Time + DevSecOps**: если DAST/SAST — ручной аудит в конце цикла, Lead Time возрастает до месяцев; встраивание сканеров (OWASP Dependency-Check) в pipeline удерживает метрику в «элитном» диапазоне
- **Test Pyramid + build time**: чем больше тестов в основании пирамиды (unit), тем меньше общая длительность CI/CD pipeline
- **Jenkins плагины vs Prometheus**: плагины (JavaMelody) дают готовые HTML-отчёты без внешней инфраструктуры; Prometheus+Grafana — неограниченная агрегация, алерты (Slack/OpsGenie), независимое хранение от перезагрузок Jenkins-мастера
- **SonarQube Quality Gate**: отображает Bugs, Vulnerabilities, Security Hotspots, Code Smells, Duplications, покрытие нового кода — каждой метрике присвоен рейтинг (A–E)

## Сравнения и trade-offs

| Метрика | Что измеряет | Тип | «Элита» |
|---------|-------------|-----|--------|
| Deployment Frequency | Скорость доставки | Скорость | Несколько раз в день |
| Lead Time for Changes | Время от коммита до prod | Скорость | < 1 часа |
| MTTR | Время восстановления | Стабильность | < 1 часа |
| Change Failure Rate | % деплоев с инцидентами | Стабильность | 0–15% |

| Мониторинг pipeline | Плюсы | Минусы |
|--------------------|-------|--------|
| Jenkins плагины (JavaMelody) | Готово «из коробки», нет инфраструктуры | Ограниченные возможности агрегации, нет алертов |
| Prometheus + Grafana | Неограниченная агрегация, алерты, история | Требует внешней инфраструктуры |

| Подход | Описание | Trade-off |
|--------|---------|----------|
| Оптимизация MTBF | Предотвращать сбои | Невозможно в сложных системах (100% MTBF недостижимо) |
| Оптимизация MTTR | Быстро восстанавливаться | Требует инвестиций в rollback и мониторинг |

## Примеры

```python
# Блокировка pipeline при снижении тестового покрытия (Python/GitLab CI)
if coverage < prev_coverage:
  fail('coverage reduced from {} to {}'.format(prev_coverage, coverage))
```

```groovy
// Интеграция SonarQube + Quality Gate в Jenkins Declarative Pipeline
stage('Static Code Analysis'){
    withSonarQubeEnv('sonarqube') {
        sh 'sonar-scanner'
    }
}
stage("Quality Gate"){
    timeout(time: 5, unit: 'MINUTES') {
        def qg = waitForQualityGate()
        if (qg.status != 'OK') {
            error "Pipeline aborted due to quality gate failure: ${qg.status}"
        }
    }
}
```

> `withSonarQubeEnv('sonarqube')` передаёт токены серверу. `waitForQualityGate()` ждёт статус до 5 минут. Если статус != `OK` — `error` останавливает сборку.

```sql
-- PromQL / Grafana: длительность выполнения контейнеров пайплайна (Jenkins)
SELECT mean("uptime_ns") FROM "docker_container_status_docker"
WHERE ("hostname" = 'Jenkins') AND $timeFilter
GROUP BY time($__interval), "container_name" fill(null)
```

> Запрашивает метрику `uptime_ns` (в наносекундах) для хоста Jenkins. Результаты агрегируются по `container_name` — позволяет выводить графики длительности шагов CI/CD.

## Важно / подводные камни / best practices

- **Оптимизировать MTTR, а не MTBF**: в сложных распределённых системах 100% MTBF недостижимо; инвестируй в быстрый rollback, Canary, Blue-Green — переведи MTTR из дней в минуты
- **Разделять deployment и release**: инженеры измеряют MTTR и стабильность; владельцы продукта — бизнес-успешность фичи независимо
- **SLO для pipeline stages**: unit < 1 мин, integration < 5 мин, весь pipeline < 30 мин; медленный пайплайн откладывается разработчиками
- **Только метрики, связанные с бизнес-целями**: CPU/memory — «vanity metrics»; нужны конверсия, доход, успешные транзакции
- **Антипаттерн: редкие деплои «ради стабильности»**: каждый редкий деплой агрегирует огромный пакет → Change Failure Rate 100%; MTTR растёт; увеличение частоты деплоя снижает риски
- **Coverage gate vs Coverage metric**: жёсткий порог прост в автоматизации, но риск «бесполезных тестов» ради лимита (79.5% → пишут тесты не ради качества, а ради числа)
- **Flaky rate на PR не измерить**: разработчик перезапускает → ошибка в main; периодический CI (раз в час на main) — единственный способ накопить статистику
