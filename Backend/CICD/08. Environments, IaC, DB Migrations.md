# 08. Environments, IaC, DB Migrations

## Суть и контекст

Управление средами, инфраструктурой и миграциями БД — ключевые аспекты зрелого CI/CD. Среды (dev/staging/production) и эфемерные Review Environments автоматически создаются и уничтожаются pipeline-ом. Infrastructure as Code (Terraform/Ansible) делает инфраструктуру воспроизводимой и откатываемой. Expand-Contract — обязательный паттерн для zero-downtime миграций БД.

---

## Ключевые идеи

- Environments: test → staging/preprod → production; Review Environments — временные, на каждую ветку
- IaC: инфраструктура тестируемая, повторяемая, идемпотентная; Terraform = провижининг; Ansible = конфигурация
- Mutable Infrastructure → конфигурационный дрейф → невоспроизводимые ошибки → откат невозможен
- Immutable Infrastructure: новый образ → новый сервер → уничтожить старый; нет дрейфа
- Expand-Contract: нельзя переименовывать/удалять колонки без двухфазного процесса (expand → deploy → contract)
- Rollback немедленно: не ждать fix forward; анализ в рабочее время
- Approval Gate: timeout + input step в Jenkins; диалог «Approve Deploy?»
- Andone cord: при красной сборке — запрет новых коммитов до исправления
- Standard changes: надёжный pipeline переводит большинство релизов в предварительно одобренную категорию

---

## Определения и термины

**Environments (Среды выполнения)** — машины или инфраструктура, на которых развёртывается и выполняется программный продукт. Стандартная структура: тестовые среды (test) → промежуточные (staging/preprod) → рабочие (production).

**Review Environments (Эфемерные среды)** — временные динамические тестовые среды, автоматически создаваемые для каждой новой ветки разработки и удаляемые после слияния ветки. Позволяют QA-команде взаимодействовать с приложением в изолированной песочнице до объединения кода.

**Infrastructure as Code (IaC)** — подход, позволяющий управлять инфраструктурой с использованием конфигурационных файлов. Делает инфраструктуру **тестируемой, повторяемой, самовосстанавливающейся и идемпотентной**. Инструменты: Terraform, Ansible.

**Rollback Strategy (Стратегия отката)** — автоматизированный, задокументированный и протестированный процесс возврата системы к предыдущей стабильной версии при сбое развёртывания. Откат выполняется немедленно; анализ бага производится в рабочее время.

**Expand-Contract Pattern (Расширение-сжатие)** — метод изменения БД при zero-downtime деплое: сначала база **расширяется** (добавляются новые объекты), потом **сжимается** (удаляются старые неиспользуемые). БД временно поддерживает обе версии приложения.

**Standard changes (Стандартные изменения)** — высокопредсказуемые изменения с низким уровнем риска, для которых существует установленный и предварительно одобренный процесс внедрения. При надёжном pipeline большинство релизов переводятся в эту категорию.

**GitOps** — практика применения итеративной модели управления изменениями (как для исходного кода) к операционной инфраструктуре. Инфраструктура в Git; изменения через коммиты запускают CI/CD.

## Как устроено

### Review App (Ephemeral Environment) в GitLab

1. Разработчик создаёт новую ветку в репозитории
2. Pipeline запускает job `deploy_review` с блоком `environment`
3. Имя среды (`review/$CI_COMMIT_REF_NAME`) и URL (`$CI_ENVIRONMENT_SLUG.example.com`) генерируются из предопределённых переменных CI
4. GitLab отображает кнопку «View app» в интерфейсе Merge Request
5. После слияния ветки → среда уничтожается

### Настройка инфраструктуры через Terraform в CI

1. Описание ресурсов в `.tf` файле (HCL: HashiCorp Configuration Language)
2. Terraform генерирует `terraform.tfstate` — текущее состояние управляемой инфраструктуры
3. CI инициирует инициализацию → Terraform загружает плагин провайдера (AWS SDK и др.)
4. Diffing: желаемое состояние (код) vs текущее (tfstate)
5. Terraform отправляет API-запросы провайдеру для применения изменений

### Развёртывание конфигурации через Ansible

1. YAML-файл playbook: `hosts` → целевые машины; `tasks` → декларативные состояния пакетов и сервисов
2. CI job запускает `ansible-playbook` с SSH-ключом из переменной `$REMOTE_HOST_CREDENTIALS`
3. Ansible: подключается к хостам → проверяет текущее состояние → вносит только необходимые изменения

### Approval Gates (ручное подтверждение деплоя)

1. Pipeline завершает стадию сборки и тестирования
2. Input Step приостанавливает pipeline → диалог «Approve Deploy?» (timeout 2 ч)
3. Ответственный нажимает «Yes» → pipeline возобновляется
4. Запускается downstream job деплоя в production
5. При успехе — уведомление в Slack

### Zero-downtime DB migration (Expand-Contract)

1. Изменения БД **отделяются** от изменений кода приложения
2. Скрипт миграции: **расширение** — добавить новые таблицы/столбцы; приложение работает со старыми И новыми объектами
3. Деплой новой версии приложения (использует обновлённую структуру)
4. Скрипт **сжатия** — удалить устаревшие объекты БД (отдельный релиз, когда старая структура больше не нужна)

## Детали и нюансы

- **Terraform vs Ansible**: Terraform — декларативный провижининг ресурсов через API (сети, серверы, бакеты S3); Ansible — конфигурационное управление уже запущенными серверами (установка пакетов, настройка служб)
- **Config as Code**: Jenkinsfile / `.gitlab-ci.yml` в репозитории → audit, code review, откат конфигурации pipeline вместе с версией приложения
- **Andon cord (дёрганье шнура)**: при красной сборке — приоритет команды — вернуть в «зелёное» состояние; запрет новых коммитов до исправления; предотвращает накопление дефектов
- **Immutable Infrastructure**: при сбое IaC в середине → сервер в неопределённом состоянии; решение: destroy + rebuild из эталонного образа (не «лечить» живой сервер)

## Сравнения и trade-offs

| | Terraform | Ansible |
|-|-----------|---------|
| Назначение | Провижининг инфраструктуры | Конфигурационное управление |
| Подход | Декларативный (API вызовы к IaaS) | Декларативный (SSH к серверам) |
| Когда | Создать сеть, сервер, S3 бакет | Установить Apache, настроить службу |
| State | terraform.tfstate | Нет явного state-файла |

| Инфраструктура | Дрейф конфигурации | Откат при сбое | Сложность |
|----------------|-------------------|----------------|---------|
| Mutable (обновление на месте) | Высокий | Сложно / невозможно | Низкая |
| Immutable (пересоздание) | Минимальный | Destroy + rebuild | Выше |

| DB Migration подход | Zero-downtime | Rollback возможен |
|--------------------|---------------|------------------|
| Прямое изменение схемы (DDL) | Нет | Нет (старый код не работает) |
| Expand-Contract | Да | Да (обе версии кода совместимы) |

## Примеры

```yaml
# GitLab CI: динамическая Review App (Ephemeral Environment)
deploy_review:
  stage: deploy
  script:
    - echo "Deploy a review app"
  environment:
    name: review/$CI_COMMIT_REF_NAME
    url: https://$CI_ENVIRONMENT_SLUG.example.com
  only:
    - branches
  except:
    - main
```

```groovy
// Jenkins: Approval Gate перед деплоем в Production
stage('Deploy'){
  if(env.BRANCH_NAME == 'master'){
     timeout(time: 2, unit: "HOURS") {
        input message: "Approve Deploy?", ok: "Yes"
     }
     build job: "watchlist-deployment/master"
  }
}
```

```hcl
# Terraform HCL: ресурс AWS S3 бакета
resource "aws_s3_bucket" "coreSite" {
  bucket = "travelBlog"
  acl = "private"
}
```

```yaml
# Ansible playbook: установка и запуск Apache
---
- hosts: web
  become: yes
  tasks:
  - name: ensure Apache is installed
    yum:
      name: apache2
      state: latest
  - name: ensure Apache is enabled and started
    service:
      name: apache2
      enabled: yes
      state: started
```

```yaml
# GitLab CI: запуск Ansible playbook через CI job
ansible-run:
  stage: run
  tags:
    - ansible
  script:
    - ansible-playbook apache-playbook.yml --key-file $REMOTE_HOST_CREDENTIALS
```

## Важно / подводные камни / best practices

- **Expand-Contract обязателен**: переименование/удаление колонки одновременно с деплоем = невозможность rollback; старый код упадёт на новой схеме
- **Hardcoded env config в образе = mutable artifact**: `--db-host=10.10.10.10` делает образ непригодным для тест-среды; решение: шаблонизация (Helm `{{ .Values.db.host }}`, переменные окружения)
- **Immutable Infrastructure**: при сбое IaC посередине — не «лечи» сервер (mutable); destroy + rebuild из образа
- **Config as Code (Jenkinsfile/`.gitlab-ci.yml` в Git)**: аудит, откат конфигурации pipeline вместе с приложением, code review CI-логики
- **GitOps для инфраструктуры**: Pull Request → автоматически применяется; история изменений инфраструктуры в Git
- **Andone cord**: красная сборка = СТОП; приоритет — зелёный pipeline; нельзя коммитить поверх сломанного
- **Cold start (Lambda)**: первый запрос к новой версии — задержка; учитывать при performance тестировании serverless

Edge case: DB migration — переименование колонки + деплой + сбой?::Rollback невозможен: старый код обращается к переименованной колонке → потеря транзакций/отказ. Решение: Expand-Contract — сначала добавить новую колонку, потом удалить старую отдельным релизом

Edge case: сбой IaC в середине применения?::Сервер в неопределённом состоянии. Решение: Immutable Infrastructure — destroy сломанного + rebuild из эталонного образа; не пытаться «починить» живой сервер

Edge case: Cold start в Lambda после деплоя?::Первый запрос к новой версии обрабатывается с задержкой (инициализация контейнера); врождённое ограничение динамического выделения ресурсов; учитывать при тестировании производительности

Terraform vs Ansible — в чём разница?::Terraform: декларативный провижининг ресурсов через API IaaS (создать сервер, сеть, S3). Ansible: конфигурационное управление уже запущенными серверами через SSH (установить пакеты, настроить службы)

Mutable vs Immutable Infrastructure?::Mutable: обновление на месте → конфигурационный дрейф → невоспроизводимые ошибки. Immutable: новый образ → новый сервер → destroy old → нет дрейфа, предсказуемый откат
