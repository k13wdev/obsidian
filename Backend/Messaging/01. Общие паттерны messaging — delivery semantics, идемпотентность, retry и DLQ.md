# Общие паттерны messaging — delivery semantics, идемпотентность, retry и DLQ

## Суть и контекст

Конспект охватывает фундаментальные паттерны, общие для систем обмена сообщениями (RabbitMQ, Kafka и др.): гарантии доставки (delivery semantics), идемпотентность обработки, стратегии повторных попыток (retry), Dead Letter Queue и упорядочивание сообщений (ordering). Эти паттерны определяют надёжность, корректность и отказоустойчивость распределённых систем, построенных на брокерах сообщений.

---

## Ключевые идеи

- Три уровня гарантий доставки: at-most-once, at-least-once, exactly-once — каждый следующий сложнее и дороже в реализации
- Exactly-once в чистом виде по сети невозможен — достигается как at-least-once + идемпотентность
- Идемпотентность нужна и на стороне продюсера (Kafka PID + Sequence Number), и на стороне консьюмера (Transactional Inbox)
- Retry storm — лавинообразная перегрузка при массовых повторных попытках
- DLQ — «карантин» для ядовитых сообщений; в RabbitMQ встроенный (DLX), в Kafka — программная реализация
- Ordering: RabbitMQ — global ordering в одной очереди (но нарушается при competing consumers), Kafka — partition ordering (строгий порядок только внутри партиции)
- Causal ordering (partial order) vs total order — практический компромисс между порядком и масштабируемостью
- Eventual consistency — модель, при которой все узлы в конечном счёте приходят к единому состоянию

---

## Определения и термины

**At-most-once (не более одного раза)** — семантика доставки, при которой сообщение отправляется по принципу «fire-and-forget» без ожидания подтверждения (ACK). Сообщения никогда не дублируются, но могут быть утеряны при сбоях. Обеспечивает самую высокую пропускную способность за счёт отказа от гарантий.

**At-least-once (хотя бы один раз)** — семантика доставки, гарантирующая сохранение и доставку сообщения за счёт механизма повторных отправок (retries) при неполучении подтверждения. Сообщения не теряются, но возможны дубликаты. Требует обработки дубликатов на стороне консьюмера.

**Exactly-once (строго один раз)** — максимальная гарантия обработки, при которой каждое сообщение обрабатывается ровно один раз, без потерь и дубликатов. В Kafka достигается комбинацией идемпотентного продюсера и транзакций. В чистом виде «exactly-once» доставка по сети невозможна без тесного взаимодействия с бизнес-логикой приложения, так как брокер не контролирует побочные эффекты (side-effects) внешней среды.

**Eventual consistency (в конечном счёте согласованность)** — модель согласованности, при которой обновления не обязаны быть видны всем узлам мгновенно, но при отсутствии новых обновлений система гарантирует, что все узлы придут к одинаковому состоянию. В Kafka: фолловер-реплики догоняют лидера с задержкой, консьюмеры читают данные с разными смещениями и обновляют проекции асинхронно.

**Идемпотентность (Idempotence)** — свойство системы, при котором выполнение одной и той же операции несколько раз приводит к такому же результату, как и выполнение её один раз. Критически важно для реализации exactly-once.

**Idempotency key (ключ идемпотентности)** — уникальный идентификатор, передаваемый вместе с сообщением, позволяющий получателю или брокеру распознать и отбросить дубликат.

**Transactional Inbox** — паттерн на стороне консьюмера, при котором сохранение результата обработки сообщения и фиксация ключа идемпотентности происходят в рамках одной атомарной транзакции во внешней базе данных.

**Retry (повторная попытка)** — механизм повторной отправки сообщения продюсером при неполучении подтверждения (ACK) от брокера.

**retry.backoff.ms** — интервал ожидания в миллисекундах перед каждой последующей попыткой повторной отправки сообщения в Kafka. По умолчанию 100 мс.

**retries** — конфигурационный параметр продюсера Kafka, определяющий количество повторных попыток. По умолчанию Integer.MAX_VALUE.

**Retry storm (шторм повторных попыток)** — лавинообразное увеличение количества запросов к брокеру, вызванное массовыми повторными попытками клиентов при ошибках, приводящее к исчерпанию ресурсов (denial-of-service).

**connection.failed.authentication.delay.ms** — параметр конфигурации брокера Kafka для задержки ответа об ошибке аутентификации. Снижает частоту повторных попыток клиентов, защищая брокер от перегрузки.

**Dead Letter Queue / Exchange (DLQ / DLX)** — отдельная очередь (в RabbitMQ) или топик (в Kafka), куда направляются сообщения, которые не удалось обработать.

**Global ordering (глобальное упорядочивание)** — гарантия сохранения хронологического порядка сообщений в рамках всей очереди или топика. Поддерживается в RabbitMQ для одной очереди, но нарушается при параллельной обработке конкурирующими консьюмерами. В Kafka отсутствует на уровне топика.

**Partition ordering (упорядочивание на уровне партиции)** — гарантия строгого порядка сообщений только внутри отдельной партиции Kafka.

**Total order (полный порядок)** — модель, при которой все события выстраиваются в одну строго упорядоченную последовательность (аналогично одной партиции).

**Causal ordering / Partial order (причинно-следственное упорядочивание)** — разбиение событий на множество связанных последовательностей, где события внутри одной последовательности строго упорядочены друг относительно друга, но не связаны с событиями из других последовательностей.

**Consistent Hashing Exchange** — тип обменника RabbitMQ (x-consistent-hash), который хэширует routing key и закрепляет результат за конкретной очередью, обеспечивая попадание сообщений с одинаковым ключом строго в одну очередь.

## Как устроено

### Delivery semantics — механизм возникновения дубликатов (At-least-once)

1. Продюсер отправляет сообщение брокеру
2. Брокер успешно сохраняет сообщение на диск (и реплицирует)
3. Брокер отправляет ACK обратно продюсеру
4. Сетевой сбой уничтожает пакет ACK по пути к продюсеру
5. У продюсера срабатывает `request.timeout.ms`
6. Продюсер повторно отправляет то же самое сообщение (retry)
7. Брокер получает сообщение, считает его новым и записывает второй раз — возникает **дубликат**
8. Консьюмер получает оба экземпляра

### At-most-once

- **Kafka `acks=0`**: продюсер помещает сообщение в сетевой буфер сокета и считает его отправленным. Брокер не отправляет ACK. `RecordMetadata.hasOffset()` возвращает `false`, offset равен `-1`
- **RabbitMQ `no_ack`**: брокер немедленно удаляет сообщение из очереди при отправке в TCP-сокет консьюмера, не дожидаясь подтверждения. Если консьюмер падает до обработки — данные в TCP-буфере ОС уничтожаются

### At-least-once

- **Kafka `acks=1`**: лидер записывает сообщение локально и отвечает ACK. Если лидер падает до репликации на фолловеры — данные теряются, хотя продюсер получил ACK
- **Kafka `acks=all` (-1)**: лидер сохраняет сообщение, ожидает подтверждения записи от всех In-Sync Replicas (ISR) и только затем отправляет ACK продюсеру
- **RabbitMQ Publisher Confirms**: продюсер отправляет `Confirm.Select`. RabbitMQ асинхронно возвращает `Basic.Ack` (успех) или `Basic.Nack` (ошибка) для каждого сообщения, используя порядковый номер

### Exactly-once (Kafka Transactions)

1. Продюсер вызывает `initTransactions()`, регистрируя `transactional.id` и инкрементируя epoch для изоляции зомби-процессов
2. Продюсер вызывает `beginTransaction()`
3. Транзакционный координатор логирует старт в `__transaction_state`
4. Продюсер пишет данные в целевые партиции
5. Продюсер отправляет смещения консьюмера координатору через `sendOffsetsToTransaction()`
6. При вызове `commitTransaction()` координатор записывает контрольные маркеры (Control Markers) во все затронутые партиции
7. Консьюмеры с `read_committed` не выдают приложению сообщения с оффсетами выше Last Stable Offset (LSO). Сообщения из прерванных транзакций молча игнорируются

### Идемпотентность на стороне продюсера (Kafka)

1. При старте продюсеру назначается уникальный Producer ID (PID)
2. Каждому сообщению присваивается монотонно возрастающий Sequence Number
3. Брокер (leader) хранит в оперативной памяти последние 5 Sequence Numbers для каждого PID на каждую партицию
4. При получении дубликата (тот же PID и Sequence Number) брокер отклоняет его — возвращает `DuplicateSequenceException`, которая игнорируется продюсером без генерации ошибки
5. Если Sequence Number превышает ожидаемый более чем на 1 — ошибка `Out of order sequence number` (потенциальная потеря данных)

### Идемпотентность на стороне консьюмера (Transactional Inbox)

1. Консьюмер считывает сообщение из Kafka
2. В рамках одной SQL-транзакции записывает результат бизнес-логики в таблицу данных
3. В рамках той же транзакции сохраняет ключ идемпотентности в таблицу дедупликации с UNIQUE constraint
4. Если сообщение доставляется повторно — БД отвергает транзакцию из-за нарушения UNIQUE constraint
5. Оффсет фиксируется в БД, а не в Kafka

### Dead Letter Exchange (DLX) в RabbitMQ

Причины попадания сообщения в DLX:
1. Консьюмер возвращает `Basic.Reject` или `Basic.Nack` с `requeue=false`
2. Истекает TTL сообщения (`x-message-ttl` или `expiration`)
3. Очередь достигла максимальной длины (`x-max-length`) — сообщение выталкивается с начала

Брокер пересылает отброшенное сообщение в указанный Exchange (`x-dead-letter-exchange`), откуда оно попадает в DLQ.

### Dead Letter Queue (DLQ) в Kafka — программная реализация

1. Консьюмер получает пакет сообщений и пытается обработать
2. Возникает неисправимая ошибка (сбой десериализации, инвалидный формат)
3. Консьюмер перехватывает исключение в коде, не прерывая цикл `poll()`
4. Консьюмер (используя встроенный `KafkaProducer`) отправляет «ядовитое» сообщение в специально созданный DLQ-топик
5. Консьюмер фиксирует смещение оригинального сообщения (commit) и продолжает чтение партиции

### Ordering — Consistent Hashing Exchange в RabbitMQ

1. Создаётся обменник типа `x-consistent-hash`
2. К нему привязываются несколько очередей с указанием веса (weight)
3. Обменник хэширует routing key сообщения и закрепляет хэш за конкретной очередью
4. Сообщения с одинаковым ключом (например, ID заказа) всегда попадают строго в одну очередь и обрабатываются строго одним консьюмером — порядок сохраняется

### Ordering — Partition ordering в Kafka

- Строгий порядок гарантируется только в пределах партиции
- Продюсер передаёт ключ сообщения: `hash(key) % numPartitions` → номер партиции
- Все события для одного ключа выстраиваются в причинно-следственную цепочку (causal ordering) внутри одной партиции
- Консьюмер из Consumer Group монопольно читает партицию, обрабатывая события ключа строго по порядку

## Детали и нюансы

- **acks=1 — скрытая потеря данных**: лидер записывает сообщение, отправляет ACK продюсеру и падает до репликации на фолловеры. Происходит безвозвратная потеря данных, хотя продюсер считает сообщение доставленным
- **enable.auto.commit=true — антипаттерн**: смещения фиксируются по таймеру в фоне. При сбое приложения смещение может быть уже закоммичено. При перезапуске приложение пропустит недообработанные сообщения — at-least-once превращается в at-most-once
- **Kafka Transactions не охватывают внешние системы**: если сервис обновляет БД и публикует в Kafka, транзакции не гарантируют атомарность между ними. Применяется Transactional Outbox
- **Зависание в транзакции (Zombie Producer)**: если продюсер не шлёт commit/abort, консьюмеры с `read_committed` блокируются по LSO до истечения `transaction.timeout.ms` (по умолчанию 15 минут)
- **Идемпотентный продюсер не защищает от ошибок бизнес-логики**: если приложение вызовет `producer.send()` дважды с одним и тем же объектом, брокер запишет два разных сообщения с разными Sequence Numbers
- **Перезапуск продюсера — новый PID**: при рестарте генерируется новый PID, и брокер не распознает дубликат от старого экземпляра
- **Изменение числа партиций ломает ordering**: `hash(key) % numPartitions` меняет распределение. Новые сообщения для ключа попадут в другую партицию, нарушая causal ordering. Уменьшение числа партиций вообще не поддерживается без пересоздания топика
- **DLX-свойства очереди в RabbitMQ неизменяемы**: `x-dead-letter-exchange` нельзя изменить после создания очереди. RabbitMQ выдаст ошибку и закроет канал. Очередь нужно удалить и пересоздать
- **DLX + TTL = отложенная очередь**: помещая сообщения в очередь с заданным TTL и привязанным DLX, RabbitMQ эмулирует delayed queue — по истечении TTL сообщение перемещается в DLQ для фактической обработки

## Сравнения и trade-offs

| Семантика | Потери | Дубликаты | Сложность | Когда использовать |
|-----------|--------|-----------|-----------|--------------------|
| At-most-once | Возможны | Нет | Минимальная | Телеметрия, метрики, клики — где допустима потеря |
| At-least-once | Нет | Возможны | Средняя | Большинство бизнес-задач с идемпотентной обработкой |
| Exactly-once | Нет | Нет | Максимальная | Финансовые операции, строгая консистентность |

| Аспект | RabbitMQ DLX | Kafka DLQ |
|--------|-------------|-----------|
| Реализация | Встроенная (x-dead-letter-exchange) | Программная (консьюмер сам пишет в DLQ-топик) |
| Причины попадания | reject, TTL, max-length | Любая ошибка, перехваченная в коде |
| Настройка | Аргументы очереди | Код консьюмера + отдельный топик |

| Ordering | Total order | Causal / Partial order |
|----------|-------------|----------------------|
| Определение | Все события в одной последовательности | События упорядочены только внутри своей цепочки |
| Реализация | 1 партиция (Kafka) / 1 очередь + 1 консьюмер (RabbitMQ) | N партиций с ключами (Kafka) / Consistent Hashing Exchange (RabbitMQ) |
| Масштабируемость | Не масштабируется | Масштабируется горизонтально |

## Примеры

Аналогия SQL и идемпотентности из источников:
- `UPDATE t SET x=18 WHERE y=5` — **идемпотентен** (многократное выполнение не меняет результат)
- `UPDATE t SET x=x+1 WHERE y=5` — **неидемпотентен** (каждое выполнение увеличивает значение)

По умолчанию продюсеры Kafka ведут себя неидемпотентно (запись в конец лога), поэтому для безопасных retries необходимо включать PID + Sequence Number.

### Аналогии

- **Kafka Transactions** аналогичны двухфазному коммиту (2-Phase Commit) с координатором — реализация основана на алгоритме снимков Чанди-Лампорта (Chandy-Lamport snapshots), где контрольные маркеры вставляются в поток коммуникации для фиксации согласованного состояния
- **RabbitMQ Publisher Confirms** действуют аналогично коллбэкам в неблокирующем вводе-выводе (Node.js) — продюсер отсылает данные потоком, а брокер возвращает статусы `Basic.Ack` / `Basic.Nack` по мере сохранения

## Важно / подводные камни / best practices

- **delivery.timeout.ms вместо retries**: рекомендуется оставлять `retries` по умолчанию (Integer.MAX_VALUE) и контролировать время параметром `delivery.timeout.ms` (например, 120000 мс). Причина: выборы нового лидера занимают ~30 сек, и жёсткое число ретраев (retries=3) исчерпается слишком рано
- **Transactional Inbox для консьюмера**: сохранение результата бизнес-логики и ключа идемпотентности в одной SQL-транзакции — надёжный способ обеспечить exactly-once на стороне потребителя
- **Poison Messages → DLQ**: бесконечный цикл десериализации битого сообщения — антипаттерн. Нужно перехватить ошибку, отправить сообщение в DLQ, зафиксировать смещение и продолжить
- **Kafka Streams processing.guarantee = exactly_once**: абстрагирует сложность ручного управления транзакционными ID, эпохами и маркерами
- **connection.failed.authentication.delay.ms**: параметр брокера для защиты от retry storm при сбоях аутентификации

Transactional Inbox:::Паттерн консьюмера: результат обработки и ключ идемпотентности сохраняются в одной SQL-транзакции с UNIQUE constraint для дедупликации

Retry storm (шторм повторных попыток):::Лавинообразная перегрузка брокера из-за массовых retry клиентов при ошибках, приводящая к denial-of-service

connection.failed.authentication.delay.ms:::Параметр брокера Kafka для задержки ответа об ошибке аутентификации, защищающий от retry storm

Dead Letter Queue / Exchange (DLQ / DLX):::Отдельная очередь или топик для сообщений, которые не удалось обработать (reject, TTL, max-length, ошибки парсинга)

Global ordering:::Хронологический порядок сообщений в рамках всей очереди/топика. В RabbitMQ — одна очередь, нарушается при competing consumers. В Kafka — отсутствует на уровне топика

Partition ordering:::Строгий порядок сообщений только внутри одной партиции Kafka

Total order (полный порядок):::Все события в одной строго упорядоченной последовательности — одна партиция, не масштабируется

Causal ordering / Partial order:::Разбиение событий на связанные последовательности, упорядоченные внутри цепочки, но не связанные между собой

Consistent Hashing Exchange:::Тип обменника RabbitMQ (x-consistent-hash), хэширующий routing key для закрепления сообщений с одинаковым ключом за одной очередью

Как возникает дубликат при at-least-once?::Брокер сохраняет сообщение, но ACK теряется в сети → продюсер ретраит → брокер записывает повторно

Почему exactly-once невозможен в чистом виде?::Брокер не контролирует side-effects внешней среды (email, запись в стороннюю БД). Нужна связка at-least-once + идемпотентность

Как Kafka idempotent producer предотвращает дубликаты?
?
1. Продюсеру назначается PID
2. Каждому сообщению — Sequence Number
3. Брокер хранит последние 5 Sequence Numbers для PID
4. При совпадении — DuplicateSequenceException, дубликат отбрасывается

Алгоритм Transactional Inbox (консьюмер)
?
1. Считать сообщение из Kafka
2. В одной SQL-транзакции: записать результат + ключ идемпотентности (UNIQUE)
3. При повторной доставке — БД отвергнет из-за нарушения constraint
4. Оффсет фиксируется в БД, а не в Kafka

Причины попадания сообщения в DLX (RabbitMQ)
?
1. Basic.Reject / Basic.Nack с requeue=false
2. Истечение TTL (x-message-ttl / expiration)
3. Превышение максимальной длины очереди (x-max-length)

Алгоритм DLQ в Kafka (программная реализация)
?
1. Консьюмер получает пакет, ловит ошибку
2. Перехватывает исключение, не прерывая poll()
3. Отправляет «ядовитое» сообщение в DLQ-топик через KafkaProducer
4. Коммитит смещение и продолжает чтение

Как Consistent Hashing Exchange решает проблему ordering в RabbitMQ?::Хэширует routing key и закрепляет за конкретной очередью → сообщения с одинаковым ключом попадают в одну очередь к одному консьюмеру

Чем опасно изменение числа партиций в Kafka?::hash(key) % numPartitions меняется → новые сообщения для того же ключа попадают в другую партицию → causal ordering нарушается

Почему retries лучше контролировать через delivery.timeout.ms?::Выборы нового лидера занимают ~30 сек. При retries=3 попытки исчерпаются слишком рано. delivery.timeout.ms=120000 даёт брокеру шанс восстановиться

retry.backoff.ms по умолчанию — ==100 мс==

retries по умолчанию в Kafka — ==Integer.MAX_VALUE==

transaction.timeout.ms по умолчанию — ==15 минут==

Kafka acks=0 означает ==fire-and-forget, offset равен -1==

Брокер хранит последние ==5== Sequence Numbers для каждого PID

Чем DLX в RabbitMQ отличается от DLQ в Kafka?
?
- RabbitMQ: встроенный механизм через x-dead-letter-exchange (reject, TTL, max-length)
- Kafka: программная реализация — консьюмер сам пишет в DLQ-топик

Что происходит при зависании Kafka-продюсера в открытой транзакции?::Консьюмеры с read_committed блокируются по LSO до истечения transaction.timeout.ms (15 минут), затем координатор принудительно прерывает транзакцию

Идемпотентный продюсер защищает от дубликатов при ==retry брокера==, но не от ошибок ==бизнес-логики отправляющего приложения==
