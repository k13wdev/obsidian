# LRU Cache

## Суть и контекст

LRU Cache (Least Recently Used Cache) — специализированная структура данных для хранения ограниченного объёма информации. Автоматически удаляет элементы, **не использовавшиеся дольше всего**, при достижении максимального размера. Применяется как промежуточный слой между приложением и хранилищем данных для обеспечения мгновенных результатов.

## Ключевые идеи

- LRU = HashMap + Doubly Linked List
- Обе операции (get и put) работают за O(1)
- HashMap обеспечивает O(1) поиск; Linked List — порядок использования
- Самые свежие элементы — в голове списка; давно не используемые — в хвосте
- Вытеснение происходит только при достижении MAX_SIZE
- Узлы списка хранят **и ключ, и значение** — критически важная деталь

## Определения и термины

**LRU (Least Recently Used)** — политика вытеснения: при переполнении удаляется элемент, который не использовался дольше всего.

**Eviction Policy (политика вытеснения)** — правило, определяющее, какой элемент удалять при переполнении кэша.

**MAX_SIZE** — максимальная ёмкость кэша. Вытеснение срабатывает только при её достижении.

**moveToFront()** — вспомогательный метод: перемещает узел в начало двусвязного списка (отмечает как «свежий»).

**removeFromLinkedList()** — вспомогательный метод: удаляет узел из текущей позиции в списке через манипуляции с `next` и `prev`.

## Как устроено

### Комбинация структур данных

Две дополняющие друг друга структуры:

**HashMap** (ключ → указатель на узел в списке):
- O(1) поиск по ключу
- Самостоятельно не позволяет определить, какой элемент использовался давно

**Doubly Linked List** (двусвязный список):
- Поддерживает порядок использования
- Голова (head) — самый свежий; хвост (tail) — наиболее давний
- Именно двусвязный, а не односвязный: позволяет удалить узел из середины за O(1) при наличии указателя

### Операция get(key) — O(1)

1. Искать ключ в HashMap
2. Если не найден → вернуть -1 (cache miss)
3. Если найден:
   - Извлечь узел из текущей позиции в списке
   - Переместить узел в голову (отметить как свежий)
   - Вернуть значение из узла

### Операция put(key, value) — O(1)

1. Если ключ уже существует:
   - Обновить значение в узле
   - Переместить узел в голову
   - **Вернуть** (не создавать новый узел, не проверять размер)
2. Если кэш заполнен (размер == MAX_SIZE):
   - **Политика вытеснения**: удалить последний узел из хвоста
   - Удалить соответствующую запись из HashMap
3. Создать новый узел с ключом и значением
4. Вставить в голову списка
5. Добавить в HashMap

### removeFromLinkedList — детали

При удалении узла из середины двусвязного списка:
1. Проверить, не является ли узел `listHead` — если да, обновить `listHead = node.next`
2. Проверить, не является ли узел `listTail` — если да, обновить `listTail = node.prev`
3. Соединить соседей: если `node.prev != null` → `node.prev.next = node.next`
4. Соединить соседей: если `node.next != null` → `node.next.prev = node.prev`

(В реализации с dummy head/tail граничные случаи head/tail обрабатываются автоматически через dummy-узлы — шаги 1-2 не нужны.)

### insertAtFrontOfLinkedList — детали

При вставке узла в голову:
1. Если список пуст (`listHead == null`): `listHead = node; listTail = node; node.next = null; node.prev = null`
2. Иначе: `node.next = listHead; listHead.prev = node; listHead = node; node.prev = null`

(В реализации с dummy head/tail: `node.next = head.next; node.prev = head; head.next.prev = node; head.next = node`.)

### Почему узел хранит и ключ?

При удалении хвостового узла (вытеснении) нужно удалить запись из HashMap. Для этого необходимо знать ключ удаляемого элемента. Поэтому каждый узел хранит и ключ, и значение.

## Детали и нюансы

**Вытеснение только при MAX_SIZE**: политика срабатывает исключительно при достижении максимальной ёмкости — не при каждой вставке.

**Двусвязный список обязателен**: при наличии указателя на узел перемещение в голову (moveToFront) требует O(1) операций с `prev` и `next`. В односвязном пришлось бы искать предшественника за O(n).

**HashMap хранит указатели на узлы**: значениями в HashMap являются ссылки на узлы списка, а не сами значения. Это позволяет мгновенно добраться до узла и его соседей.

## Сравнения и trade-offs

| Структура | Без LRU | LRU Cache |
|-----------|---------|-----------|
| get | O(1) HashMap | O(1) |
| put | O(1) HashMap | O(1) |
| Вытеснение | Нет | O(1) (удаление хвоста) |
| Память | HashMap only | HashMap + Doubly Linked List |
| Определение «старейшего» | Невозможно | O(1) — хвост списка |

## Варианты использования

**Веб-серверы**: хранение результатов дорогостоящих поисковых запросов или отрендеренных страниц (например, страниц «О нас» или «Условия использования»).

**Оптимизация поиска**: кэширование недавно посещённых документов, когда весь набор данных (терабайты) не помещается в оперативной памяти.

**Системное проектирование**: кэш как промежуточный слой между приложением и базой данных.

**Проверка дубликатов**: кэш хранит наиболее вероятные совпадения из огромных наборов данных.

## Примеры

```go
type LRUNode struct {
    Key, Val   int
    Prev, Next *LRUNode
}

type LRUCache struct {
    cap        int
    cache      map[int]*LRUNode
    head, tail *LRUNode // dummy nodes
}

func NewLRUCache(capacity int) *LRUCache {
    head := &LRUNode{}
    tail := &LRUNode{}
    head.Next = tail
    tail.Prev = head
    return &LRUCache{
        cap:   capacity,
        cache: make(map[int]*LRUNode),
        head:  head,
        tail:  tail,
    }
}

func (c *LRUCache) removeFromList(node *LRUNode) {
    node.Prev.Next = node.Next
    node.Next.Prev = node.Prev
}

func (c *LRUCache) moveToFront(node *LRUNode) {
    node.Next = c.head.Next
    node.Prev = c.head
    c.head.Next.Prev = node
    c.head.Next = node
}

func (c *LRUCache) Get(key int) int {
    node, ok := c.cache[key]
    if !ok {
        return -1
    }
    c.removeFromList(node)
    c.moveToFront(node)
    return node.Val
}

func (c *LRUCache) Put(key, value int) {
    if node, ok := c.cache[key]; ok {
        node.Val = value
        c.removeFromList(node)
        c.moveToFront(node)
        return
    }
    if len(c.cache) == c.cap {
        // Вытеснение: удаляем хвостовой узел (LRU)
        lru := c.tail.Prev
        c.removeFromList(lru)
        delete(c.cache, lru.Key) // нужен ключ в узле!
    }
    node := &LRUNode{Key: key, Val: value}
    c.cache[key] = node
    c.moveToFront(node)
}
```

## Связи с другими темами

- [[HashMap]] — основная структура для O(1) поиска по ключу
- [[Linked List]] — двусвязный список управляет порядком использования
- [[Big-O Notation]] — комбинация двух структур даёт O(1) для всех операций

## Важно / подводные камни / best practices

- **Узел обязан хранить ключ**: без ключа нельзя удалить запись из HashMap при вытеснении
- **Двусвязный, не односвязный**: только двусвязный даёт O(1) удаление из середины
- **Dummy head и tail**: упрощают реализацию, убирая граничные случаи при работе с пустым списком
- **Вытеснение только при cap == len**: не при каждой операции put

## Карточки для повторения

#flashcards/algorithms/lru-cache

Что такое LRU Cache?::Структура данных с ограниченной ёмкостью, которая при переполнении удаляет элемент, не использовавшийся дольше всего (Least Recently Used).

Какие две структуры данных лежат в основе LRU?::HashMap (O(1) поиск по ключу) + Doubly Linked List (порядок использования: голова = свежий, хвост = давний).

Почему именно двусвязный список, а не односвязный?::Двусвязный позволяет удалить узел из середины за O(1) при наличии указателя — есть доступ к prev и next. В односвязном нужен поиск предшественника.

Алгоритм операции get(key)?::1) Найти в HashMap; 2) Cache miss → -1; 3) Cache hit → удалить из текущей позиции, переместить в голову, вернуть значение.

Алгоритм операции put(key, value)?::1) Ключ есть → обновить и переместить в голову; 2) Кэш полон → удалить хвост + запись из HashMap; 3) Создать узел, вставить в голову, добавить в HashMap.

Почему узел LRU должен хранить ключ?::При вытеснении хвостового узла нужно удалить запись из HashMap. Без ключа в узле нельзя найти, что удалять из HashMap.

Когда срабатывает политика вытеснения?::Только при достижении MAX_SIZE (cap == len). При каждой вставке, если кэш полон.

Сложность операций LRU Cache?::Get — O(1), Put — O(1), включая вытеснение. Оба благодаря комбинации HashMap и Doubly Linked List.

Типичные use cases LRU Cache?::Кэширование результатов запросов к БД, рендеринг веб-страниц, CDN-кэши, кэш ОС для файлов.
