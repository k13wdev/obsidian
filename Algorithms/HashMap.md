# HashMap

## Суть и контекст

HashMap (Hash Table, хэш-таблица) — структура данных для хранения пар «ключ-значение», обеспечивающая высокоэффективный поиск, вставку и удаление. Является одной из самых полезных сложных структур данных: позволяет моделировать отношения между объектами, устранять дубликаты и кэшировать данные.

## Ключевые идеи

- В основе — массив с логикой определения места хранения через хэш-функцию
- Среднее время операций O(1), худший случай O(n)
- При коллизиях используются: метод цепочек или открытая адресация
- При превышении load factor выполняется ресайзинг (рехэширование)
- Ключевой инструмент оптимизации bottlenecks в алгоритмах

## Определения и термины

**Hash Table / HashMap** — структура данных: массив + хэш-функция, вычисляющая индекс по ключу.

**Hash Function (хэш-функция)** — функция, преобразующая ключ в числовой индекс массива.

**Collision (коллизия)** — ситуация, когда два разных ключа хэшируются в один и тот же индекс массива.

**Load Factor (коэффициент заполнения)** — соотношение количества элементов (n) к числу слотов (m): α = n/m.

**Rehashing (рехэширование)** — процесс создания нового массива (обычно вдвое большего) и повторной вставки всех элементов с использованием хэш-функции для нового размера.

**Chaining (метод цепочек)** — разрешение коллизий через хранение связного списка в каждом слоте.

**Open Addressing (открытая адресация)** — разрешение коллизий путём поиска свободного слота внутри самого массива.

**Linear Probing (линейное исследование)** — при занятом слоте проверяется следующий по порядку.

**Double Hashing (двойное хэширование)** — шаг исследования при открытой адресации определяется второй хэш-функцией.

**Primary Clustering (первичная кластеризация)** — образование длинных цепочек занятых ячеек при линейном исследовании.

**Universal Hashing (универсальное хэширование)** — хэш-функция выбирается случайно из семейства функций для предотвращения worst case, вызванного злоумышленником.

## Как устроено

### Хэш-функция

Преобразует ключ (например, строку) в число — индекс в массиве. Требования к хорошей хэш-функции:

1. **Последовательность**: один и тот же ключ всегда → один и тот же индекс
2. **Равномерность**: ключи распределяются по массиву равномерно, минимизируется кластеризация
3. **Корректность**: возвращает только действительные индексы (в пределах размера массива)

Методы создания:
- **Метод деления**: `k mod m`
- **Метод умножения**
- **Универсальное хэширование**: случайный выбор функции из семейства

### Разрешение коллизий

#### Метод цепочек (Chaining)
- В каждом слоте хранится связный список элементов
- Коллизия → новый элемент добавляется в список по этому индексу
- Вместо списков можно использовать сбалансированные BST → худшее время O(log n) вместо O(n)

#### Открытая адресация (Open Addressing)
Все элементы хранятся в самом массиве. Коэффициент заполнения α никогда не превышает 1:
- **Линейное исследование**: `h(k, i) = (h₀(k) + i) mod m`. Занятый слот → проверяем следующий. Вызывает primary clustering — если свободная ячейка предшествует i занятым, вероятность её заполнения увеличивается до (i+1)/m
- **Квадратичное исследование**: `h(k, i) = (h₀(k) + c₁·i + c₂·i²) mod m`. Расстояние между пробами растёт квадратично
- **Двойное хэширование**: `h(k, i) = (h₁(k) + i·h₂(k)) mod m`. Шаг пробы определяет вторая хэш-функция. Один из лучших методов

**Удаление при Open Addressing**: нельзя просто записать NIL в ячейку — поиск других ключей (чей probe sequence проходил через эту ячейку) оборвётся. Требуется записывать специальный маркер DELETED. Из-за этого время поиска перестаёт зависеть от load factor. Для систем с частым удалением предпочтителен Chaining.

### Коэффициент заполнения и ресайзинг

**Load Factor** = n / m

При превышении порога (обычно 0.7):
1. Создаётся новый массив (обычно вдвое больше)
2. Все элементы повторно вставляются с хэш-функцией для нового размера

Амортизированная сложность вставки с редкими ресайзингами — O(1).

### Сложность операций

| Операция | Средний случай | Худший случай |
|----------|----------------|---------------|
| Поиск | O(1) | O(n) |
| Вставка | O(1) | O(n) |
| Удаление | O(1) | O(n) |

**Примечание**: среднее O(1) предполагает хорошую хэш-функцию. Для строк поиск технически O(K), где K — длина строки (нужно прочитать все символы для вычисления хэша).

## Детали и нюансы

**Caching**: результаты дорогостоящих запросов кэшируются в HashMap для мгновенного доступа.

**Амортизированная вставка**: хотя отдельный ресайзинг дорог, в среднем по всем операциям — O(1) амортизированно.

**Chaining vs BST**: замена связного списка на сбалансированное дерево в методе цепочек снижает worst case с O(n) до O(log n) — применяется в Java HashMap (treeification при длинных цепочках).

## Сравнения и trade-offs

| Метод коллизий | Преимущества | Недостатки |
|----------------|--------------|------------|
| Chaining | Простая реализация, работает при α > 1 | Доп. память на указатели |
| Linear Probing | Хорошая cache-locality | Primary clustering |
| Quadratic Probing | Меньше clustering | Secondary clustering |
| Double Hashing | Лучшее распределение | Сложнее реализовать |

## Паттерны использования в алгоритмах

**Исключение дубликатов**: проверка, встречался ли объект ранее (например, список из 10 миллиардов URL — встречался ли URL раньше?).

**Кэширование**: хранение результатов дорогих поисковых запросов или отрендеренных страниц.

**Подсчёт частоты**: определение сколько раз каждое слово встречается в тексте.

**Поиск пар**: нахождение двух чисел, сумма/разность которых равна k. Оптимизирует brute-force с O(n²) до O(n).

**Инвертированный индекс**: сопоставление слов с местами (страницами), где они встречаются — основа поисковых систем.

**Группировка**: объединение слов-анаграмм.

**Оптимизация узких мест (Bottlenecks)**: HashMap — первый инструмент при поиске ускорений в алгоритмах.

## Примеры

```go
// Базовое использование map в Go
func twoSum(nums []int, target int) []int {
    seen := make(map[int]int) // value → index
    for i, num := range nums {
        complement := target - num
        if j, ok := seen[complement]; ok {
            return []int{j, i}
        }
        seen[num] = i
    }
    return nil
}

// Подсчёт частоты слов
func wordFrequency(words []string) map[string]int {
    freq := make(map[string]int)
    for _, w := range words {
        freq[w]++
    }
    return freq
}

// Поиск дубликатов
func containsDuplicate(nums []int) bool {
    seen := make(map[int]bool)
    for _, n := range nums {
        if seen[n] {
            return true
        }
        seen[n] = true
    }
    return false
}

// Группировка анаграмм
func groupAnagrams(strs []string) [][]string {
    groups := make(map[[26]int][]string)
    for _, s := range strs {
        var key [26]int
        for _, c := range s {
            key[c-'a']++
        }
        groups[key] = append(groups[key], s)
    }
    result := make([][]string, 0, len(groups))
    for _, g := range groups {
        result = append(result, g)
    }
    return result
}

// Поиск пары с заданной разностью
func findPairWithDiff(arr []int, diff int) (int, int) {
    seen := make(map[int]bool)
    for _, n := range arr {
        seen[n] = true
    }
    for _, n := range arr {
        if seen[n+diff] {
            return n, n + diff
        }
    }
    return 0, 0
}

// Инвертированный индекс
func buildInvertedIndex(docs map[int][]string) map[string][]int {
    index := make(map[string][]int)
    for docID, words := range docs {
        for _, word := range words {
            index[word] = append(index[word], docID)
        }
    }
    return index
}

// Простая реализация хэш-таблицы с цепочками
type entry struct {
    key   string
    value int
}

type HashTable struct {
    buckets [][]entry
    size    int
}

func NewHashTable(size int) *HashTable {
    return &HashTable{
        buckets: make([][]entry, size),
        size:    size,
    }
}

func (ht *HashTable) hash(key string) int {
    h := 0
    for _, c := range key {
        h = (h*31 + int(c)) % ht.size
    }
    return h
}

func (ht *HashTable) Set(key string, value int) {
    idx := ht.hash(key)
    for i, e := range ht.buckets[idx] {
        if e.key == key {
            ht.buckets[idx][i].value = value
            return
        }
    }
    ht.buckets[idx] = append(ht.buckets[idx], entry{key, value})
}

func (ht *HashTable) Get(key string) (int, bool) {
    idx := ht.hash(key)
    for _, e := range ht.buckets[idx] {
        if e.key == key {
            return e.value, true
        }
    }
    return 0, false
}
```

## Связи с другими темами

- [[LRU Cache]] — LRU использует HashMap для O(1) поиска + двусвязный список для порядка
- [[Linked List]] — метод цепочек использует связный список в каждом bucket
- [[Big-O Notation]] — HashMap — ключевой инструмент снижения сложности с O(n²) до O(n)
- [[Sliding Window]] — HashMap используется для хранения частот символов в окне

## Важно / подводные камни / best practices

- **Load factor**: следи за соотношением n/m — при > 0.7 производительность деградирует
- **Качество хэш-функции**: плохая функция → кластеризация → деградация до O(n)
- **Строки**: поиск строки технически O(K) из-за вычисления хэша
- **Универсальное хэширование**: защита от adversarial inputs, которые намеренно вызывают коллизии
- **HashMap — первый инструмент оптимизации**: при bottleneck в O(n²) задаёмся вопросом «что можно закэшировать в HashMap?»

## Карточки для повторения

#flashcards/algorithms/hash-map

Что такое HashMap?::Структура данных для хранения пар ключ-значение. Основана на массиве + хэш-функции. Средние O(1) для поиска, вставки, удаления.

Что делает хэш-функция?::Преобразует ключ в числовой индекс массива. Требования: последовательность (одинаковый ввод → одинаковый индекс), равномерность, корректность (в пределах размера).

Что такое коллизия и два метода разрешения?::Коллизия — два ключа дают один индекс. Методы: 1) Chaining — связный список в слоте; 2) Open Addressing — поиск свободного слота в массиве.

Load Factor: что это и порог?::α = n/m (элементов / слотов). При α > 0.7 — ресайзинг: создаётся массив вдвое больше, все элементы рехэшируются.

Сложность HashMap в среднем и худшем случае?::Средний: O(1) для всех операций. Худший: O(n) при плохой хэш-функции и массе коллизий.

Почему поиск строки в HashMap не строго O(1)?::Вычисление хэша строки требует прочтения всех K символов → O(K) по времени.

Методы открытой адресации и их проблемы::Linear Probing → primary clustering (длинные цепочки). Quadratic → secondary clustering. Double Hashing — лучший метод, шаг определяет вторая функция.

Паттерн оптимизации O(n²) → O(n) через HashMap?::Вместо двух вложенных циклов (brute force) — один цикл + HashMap для мгновенного поиска (pair sum, pair diff, duplicates).

Зачем нужно универсальное хэширование?::Случайный выбор функции из семейства предотвращает worst case от adversarial inputs, специально подобранных для вызова коллизий.

Амортизированная сложность вставки с ресайзингом?::O(1) амортизированно — хотя ресайзинг дорог, он происходит редко, и в среднем каждая вставка остаётся O(1).

Формулы пробирования в Open Addressing?
?
1. Linear: h(k,i) = (h₀(k) + i) mod m
2. Quadratic: h(k,i) = (h₀(k) + c₁·i + c₂·i²) mod m
3. Double Hashing: h(k,i) = (h₁(k) + i·h₂(k)) mod m

Почему при Open Addressing нельзя записать NIL при удалении?::Поиск других ключей, чей probe sequence проходил через эту ячейку, оборвётся. Нужен маркер DELETED. Для частых удалений предпочтительнее Chaining.
