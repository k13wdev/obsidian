# Использование NotebookLM MCP

Этот skill определяет, как использовать интеграцию `notebooklm-mcp-cli` через MCP внутри Claude Code.

Он предназначен для preload в subagent'ы, которым нужен доступ к NotebookLM, особенно:
- `notebooklm-study-notes`
- любые будущие research/synthesis агенты, которым нужен grounded-доступ к notebook'ам NotebookLM

## Область ответственности

Этот skill отвечает именно за **то, как использовать MCP-интеграцию**.

Он не определяет формат самих конспектов. Вызывающий subagent по-прежнему отвечает за:
- структуру результата
- стиль конспекта
- правила синтеза
- anti-hallucination правила

Этот skill определяет только:
- как разрешать целевой notebook
- как получать данные из NotebookLM
- как реагировать на проблемы с auth и resolution notebook
- как предпочитать MCP вместо ad-hoc shell-использования

## Основное правило

Если NotebookLM MCP доступен, предпочитай доступ к NotebookLM через MCP вместо догадок и вместо опоры на память.

Если NotebookLM MCP недоступен, не делай вид, что он доступен. В таком случае можно опираться только на:
- выдержки, переданные пользователем
- уже полученные результаты из NotebookLM
- локальные файлы репозитория, если вызывающий агент явно это разрешает

## Разрешение notebook

Перед любой значимой операцией с NotebookLM прочитай:
- `.claude/notebooklm-config.toml`

Используй этот порядок resolution строго:
1. `notebook_alias`
2. `notebook_id`
3. `notebook_title`
4. `tag_query`, если `allow_tag_fallback = true`

Если ничего не настроено, явно сообщи:

`Автопривязка NotebookLM notebook не настроена. Нужен alias, id, exact title или tag query.`

Если notebook не удаётся найти, прямо укажи, что именно не сработало:
- alias не найден
- notebook id невалиден
- exact title не найден
- tag query не вернул подходящий notebook

Не переключайся молча на другой notebook, если пользователь явно не просил cross-notebook поведение.

## Single-notebook mode

Если в `.claude/notebooklm-config.toml` задано:
- `single_notebook_mode = true`

то после разрешения notebook оставайся внутри него до конца задачи, если только пользователь явно не просит:
- сравнение нескольких notebook'ов
- более широкий research
- поиск подходящего notebook

## Предпочтительная стратегия получения данных

Выбирай самый лёгкий способ получения данных, который всё ещё даёт grounded-результат.

### Используй notebook query, когда:
- пользователь задаёт прямой вопрос
- пользователю нужен краткий grounded-ответ
- нужен короткий синтез по многим источникам внутри одного notebook

### Используй source list + source content, когда:
- пользователь хочет подробный study guide
- пользователь просит exhaustive extraction
- пользователь просит примеры, код, edge cases, trade-offs или полный coverage источников
- notebook query слишком сжатый и может скрыть важные детали

### Практическое правило

Для генерации конспекта:
1. разреши notebook
2. при необходимости посмотри доступные sources
3. вытяни релевантный source content или достаточный набор выдержек
4. только потом делай синтез

Не опирайся на один поверхностный notebook query, если задача требует исчерпывающего покрытия.

## Правила grounding

Считай результаты NotebookLM исходным материалом, а не разрешением на импровизацию.

Разрешено:
- суммировать возвращённый NotebookLM content
- цитировать возвращённый NotebookLM content
- реорганизовывать возвращённый NotebookLM content
- выявлять противоречия между возвращёнными фрагментами

Запрещено:
- добавлять неуказанные факты
- заполнять пробелы общими знаниями
- склеивать конфликтующие утверждения без указания на конфликт
- заявлять точную формулировку, если она не была возвращена

Если данных не хватает, пиши:

> В предоставленных источниках не раскрыто.

Если данные частичные, пиши:

> В источниках упомянуто частично: ...

## MCP-first поведение

Если доступны и NotebookLM MCP, и shell-доступ к `nlm`:
- предпочитай MCP для операций с NotebookLM внутри Claude Code;
- используй shell `nlm` только как fallback, если MCP недоступен, сломан или пользователь явно просит CLI-режим.

Не запускай интерактивные CLI REPL-сценарии.

Избегай:
- `nlm chat start`

Предпочитай one-shot или структурированные операции.

## Восстановление после проблем с авторизацией

Авторизация NotebookLM может протухать.

Если вызовы к NotebookLM падают из-за auth/session проблем:
1. явно скажи, что авторизация NotebookLM, вероятно, истекла;
2. попроси пользователя при необходимости обновить login через `nlm login`;
3. после обновления продолжай работу с тем же notebook из конфига;
4. не продолжай работу так, будто данные из NotebookLM были успешно получены.

Если проблема только в transport availability:
- прямо скажи, что MCP недоступен или не подключён;
- не выдумывай доступ к notebook.

## Рекомендованное поведение при выборе sources

Если в notebook много sources:
- предпочитай те, что напрямую относятся к теме запроса пользователя;
- если задача требует полноты, укажи, какие именно sources были использованы;
- если coverage источников частичный, скажи это явно.

Когда важна точная provenance:
- сохраняй title source или source identifier, который вернул NotebookLM;
- включай эту provenance в ответ, если вызывающий subagent требует указания источников.

## Output hygiene для учебных workflows

Для study-note агентов:
- сначала собирай данные
- затем parse/extract/structure
- только потом пиши итоговый документ

Не смешивай технические шаги выбора notebook с финальным учебным документом.

Если вызывающий subagent генерирует Obsidian-ready Markdown:
- не включай технические детали работы NotebookLM в финальный документ, если пользователь не просил этого явно
- держи их в процессе работы, а не в финальном конспекте

## Ожидаемые локальные файлы

Этот skill предполагает, что в проекте могут существовать файлы:
- `.claude/notebooklm-config.toml`
- `.mcp.json`
- `.claude/agents/notebooklm-study-notes.md`

Читай только то, что действительно нужно.

## Минимальный operating checklist

Перед ответом, grounded на NotebookLM:
1. Прочитай `.claude/notebooklm-config.toml`
2. Разреши notebook через alias/id/title/tag fallback
3. Убедись, что notebook действительно найден
4. Получи достаточно NotebookLM content для выполнения задачи
5. Синтезируй без добавления внешних фактов
6. Явно пометь отсутствие данных или частичное покрытие

Если проваливаются шаги 1-4, не имитируй шаг 5.
